<p><b>Software architecture</b> refers to the fundamental structures of a <a href="/wiki/Software_system" title="Software system">software system</a>, the discipline of creating such structures, and the documentation of these structures. These structures are needed to reason about the software system. Each structure comprises software elements, relations among them, and properties of both elements and relations,<sup class="reference" id="cite_ref-DSA2_1-0"><a href="#cite_note-DSA2-1">[1]</a></sup> along with <a href="/wiki/Design_rationale" title="Design rationale">rationale</a> for the introduction and configuration of each element. The <i>architecture</i> of a software system is a metaphor, analogous to the <a href="/wiki/Architecture" title="Architecture">architecture</a> of a building.<sup class="reference" id="cite_ref-PERRY1992_2-0"><a href="#cite_note-PERRY1992-2">[2]</a></sup></p>
<p>Software architecture is about making fundamental structural choices which are costly to change once implemented. Software architecture choices, also called <a class="new" href="/w/index.php?title=Architectural_Decision&amp;action=edit&amp;redlink=1" title="Architectural Decision (page does not exist)">architectural decisions</a>, include specific structural options from possibilities in the design of software. For example, the systems that controlled the <a class="mw-redirect" href="/wiki/Space_shuttle" title="Space shuttle">space shuttle</a> launch vehicle had the requirement of being very fast and very reliable. Therefore, an appropriate <a href="/wiki/Real-time_computing" title="Real-time computing">real-time computing</a> language would need to be chosen. Additionally, to satisfy the need for reliability the choice could be made to have multiple redundant and independently produced copies of the program, and to run these copies on independent hardware while cross-checking results.</p>
<p>Documenting software architecture facilitates communication between <a href="/wiki/Stakeholder_(corporate)#In_management" title="Stakeholder (corporate)">stakeholders</a>, captures decisions about the architecture design, and allows reuse of design components between projects.<sup class="reference" id="cite_ref-SAP2_3-0"><a href="#cite_note-SAP2-3">[3]</a></sup><sup class="reference" style="white-space:nowrap;">:pp.29–35</sup></p>

<h2><span class="mw-headline" id="Scope">Scope</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Software_architecture&amp;action=edit&amp;section=1" title="Edit section: Scope">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Opinions vary as to the scope of software architectures:<sup class="reference" id="cite_ref-4"><a href="#cite_note-4">[4]</a></sup></p>
<ul>
<li><i>Overall, macroscopic system structure</i>;<sup class="reference" id="cite_ref-5"><a href="#cite_note-5">[5]</a></sup> this refers to architecture as a higher level <a class="mw-redirect" href="/wiki/Abstraction_(computer_science)" title="Abstraction (computer science)">abstraction</a> of a software system that consists of a collection of computational <i>components</i> together with <i>connectors</i> that describe the interaction between these components.</li>
<li><i>The important stuff—whatever that is</i>;<sup class="reference" id="cite_ref-FOWL2003_6-0"><a href="#cite_note-FOWL2003-6">[6]</a></sup> this refers to the fact that software architects should concern themselves with those decisions that have high impact on the system and its stakeholders.</li>
<li><i>That which is fundamental to understanding a system in its environment</i>;<sup class="reference" id="cite_ref-7"><a href="#cite_note-7">[7]</a></sup> in this definition, the environment is characterized by <i>stakeholder concerns</i>, technical constraints, and various dimensions of project context.<sup class="reference" id="cite_ref-TORR2014_8-0"><a href="#cite_note-TORR2014-8">[8]</a></sup></li>
<li><i>Things that people perceive as hard to change</i>;<sup class="reference" id="cite_ref-FOWL2003_6-1"><a href="#cite_note-FOWL2003-6">[6]</a></sup> since designing the architecture often takes place at the beginning of a software system's lifecycle, the architect should focus on decisions that "have to" be right the first time. Following this line of thought, architectural design issues may become non-architectural once their irreversibility can be overcome (see "Software architecture and agile development" below).</li>
<li><i>A set of architectural design decisions</i>;<sup class="reference" id="cite_ref-jansen05_9-0"><a href="#cite_note-jansen05-9">[9]</a></sup> software architecture should not be considered merely a set of models or structures, but should include the decisions that lead to these particular structures, and the rationale behind them (e.g., justifications, answers to "why" questions)). This insight has led to substantial research into software architecture knowledge management.<sup class="reference" id="cite_ref-AKM_10-0"><a href="#cite_note-AKM-10">[10]</a></sup></li>
</ul>
<p>There is no sharp distinction between software architecture versus design and requirements engineering (see <a href="#Related_fields">Related fields</a> below). They are all part of a "chain of intentionality" from high-level intentions to low-level details.<sup class="reference" id="cite_ref-FAIRBANKS2010_11-0"><a href="#cite_note-FAIRBANKS2010-11">[11]</a></sup><sup class="reference" style="white-space:nowrap;">(p18)</sup> This duality is also referred to as the "twin peaks" of software engineering.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (August 2016)">citation needed</span></a></i>]</sup></p>
<h2><span class="mw-headline" id="Characteristics">Characteristics</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Software_architecture&amp;action=edit&amp;section=2" title="Edit section: Characteristics">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Software architecture exhibits the following:</p>
<p><b>Multitude of stakeholders:</b> software systems have to cater to a variety of stakeholders such as business managers, application owners, developers, end users and infrastructure operators. These stakeholders all have their own concerns with respect to the system. Balancing these concerns and demonstrating how they are addressed is part of designing the system.<sup class="reference" id="cite_ref-SAP2_3-1"><a href="#cite_note-SAP2-3">[3]</a></sup><sup class="reference" style="white-space:nowrap;">:pp.29–31</sup> This implies that architecture involves dealing with a broad variety of concerns and stakeholders, and has a multidisciplinary nature. Software architect require non-technicals skills such as communication and negotiation competencies.</p>
<p><b>Separation of concerns:</b> the established way for architects to reduce complexity is to separate the concerns that drive the design. Architecture documentation shows that all stakeholder concerns are addressed by modeling and describing the architecture from separate points of view associated with the various stakeholder concerns.<sup class="reference" id="cite_ref-ISO42010_12-0"><a href="#cite_note-ISO42010-12">[12]</a></sup> These separate descriptions are called architectural views (see for example the <a class="mw-redirect" href="/wiki/4%2B1_Architectural_View_Model" title="4+1 Architectural View Model">4+1 Architectural View Model</a>).</p>
<p><b>Quality-driven:</b> classic <a href="/wiki/Software_design" title="Software design">software design</a> approaches (e.g. <a class="mw-redirect" href="/wiki/Jackson_Structured_Programming" title="Jackson Structured Programming">Jackson Structured Programming</a>) were driven by required functionality and the flow of data through the system, but the current insight<sup class="reference" id="cite_ref-SAP2_3-2"><a href="#cite_note-SAP2-3">[3]</a></sup><sup class="reference" style="white-space:nowrap;">:pp.26–28</sup> is that the architecture of a software system is more closely related to its <a class="mw-redirect" href="/wiki/Quality_attributes" title="Quality attributes">quality attributes</a> such as <a class="mw-redirect" href="/wiki/Fault-tolerance" title="Fault-tolerance">fault-tolerance</a>, <a href="/wiki/Backward_compatibility" title="Backward compatibility">backward compatibility</a>, <a href="/wiki/Extensibility" title="Extensibility">extensibility</a>, <a class="mw-redirect" href="/wiki/Reliability_(engineering)" title="Reliability (engineering)">reliability</a>, <a href="/wiki/Maintainability" title="Maintainability">maintainability</a>, <a href="/wiki/Availability" title="Availability">availability</a>, security, usability, and other such –<a class="mw-redirect" href="/wiki/Ilities" title="Ilities">ilities</a>. Stakeholder concerns often translate into <a class="mw-redirect" href="/wiki/Requirements" title="Requirements">requirements</a> and constraints on these quality attributes, which are variously called <a class="mw-redirect" href="/wiki/Non-functional_requirements" title="Non-functional requirements">non-functional requirements</a>, extra-functional requirements, behavioral requirements, or quality attribute requirements.</p>
<p><b>Recurring styles:</b> like building architecture, the software architecture discipline has developed standard ways to address recurring concerns. These "standard ways" are called by various names at various levels of abstraction. Common terms for recurring solutions are architectural style, principle,<sup class="reference" id="cite_ref-FAIRBANKS2010_11-1"><a href="#cite_note-FAIRBANKS2010-11">[11]</a></sup><sup class="reference" style="white-space:nowrap;">:pp.273–277</sup> tactic,<sup class="reference" id="cite_ref-SAP2_3-3"><a href="#cite_note-SAP2-3">[3]</a></sup><sup class="reference" style="white-space:nowrap;">:pp.70–72</sup> <a href="/wiki/Reference_architecture" title="Reference architecture">reference architecture</a><sup class="reference" id="cite_ref-REFARCHPRIMER_13-0"><a href="#cite_note-REFARCHPRIMER-13">[13]</a></sup><sup class="reference" id="cite_ref-REFARCHCLASS_14-0"><a href="#cite_note-REFARCHCLASS-14">[14]</a></sup> and <a href="/wiki/Architectural_pattern" title="Architectural pattern">architectural pattern</a>.<sup class="reference" id="cite_ref-SAP2_3-4"><a href="#cite_note-SAP2-3">[3]</a></sup><sup class="reference" style="white-space:nowrap;">:pp.203–205</sup></p>
<p><b>Conceptual integrity:</b> a term introduced by Fred Brooks in <a href="/wiki/The_Mythical_Man-Month" title="The Mythical Man-Month">The Mythical Man-Month</a> to denote the idea that the architecture of a software system represents an overall vision of what it should do and how it should do it. This vision should be separated from its implementation. The architect assumes the role of "keeper of the vision", making sure that additions to the system are in line with the architecture, hence preserving <a href="/wiki/The_Mythical_Man-Month#Conceptual_integrity" title="The Mythical Man-Month">conceptual integrity</a>.<sup class="reference" id="cite_ref-BROOKS_15-0"><a href="#cite_note-BROOKS-15">[15]</a></sup><sup class="reference" style="white-space:nowrap;">:pp.41–50</sup></p>
<h2><span class="mw-headline" id="Motivation">Motivation</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Software_architecture&amp;action=edit&amp;section=3" title="Edit section: Motivation">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Software architecture is an "intellectually graspable" abstraction of a complex system.<sup class="reference" id="cite_ref-SAP2_3-5"><a href="#cite_note-SAP2-3">[3]</a></sup><sup class="reference" style="white-space:nowrap;">:pp.5–6</sup> This abstraction provides a number of benefits:</p>
<ul>
<li><i>It gives a basis for analysis of software systems' behavior before the system has been built.</i><sup class="reference" id="cite_ref-PERRY1992_2-1"><a href="#cite_note-PERRY1992-2">[2]</a></sup> The ability to verify that a future software system fulfills its stakeholders' needs without actually having to build it represents substantial cost-saving and risk-mitigation.<sup class="reference" id="cite_ref-SARA_16-0"><a href="#cite_note-SARA-16">[16]</a></sup> A number of techniques have been developed in academia and practice to perform such analyses, for instance <a class="mw-redirect" href="/wiki/ATAM" title="ATAM">ATAM</a>, ARID and TARA.</li>
<li><i>It provides a basis for re-use of elements and decisions.</i><sup class="reference" id="cite_ref-PERRY1992_2-2"><a href="#cite_note-PERRY1992-2">[2]</a></sup><sup class="reference" id="cite_ref-SAP2_3-6"><a href="#cite_note-SAP2-3">[3]</a></sup><sup class="reference" style="white-space:nowrap;">:p.35</sup> A complete software architecture or parts of it, like individual architectural strategies and decisions, can be re-used across multiple systems whose stakeholders require similar quality attributes or functionality, saving design costs and mitigating the risk of design mistakes (assuming that the project contexts<sup class="reference" id="cite_ref-17"><a href="#cite_note-17">[17]</a></sup> match).</li>
<li><i>It supports early design decisions that impact a system's development, deployment, and maintenance life.</i><sup class="reference" id="cite_ref-SAP2_3-7"><a href="#cite_note-SAP2-3">[3]</a></sup><sup class="reference" style="white-space:nowrap;">:p.31</sup> Getting the early, high-impact decisions right is important to prevent schedule and <a href="/wiki/Cost_overrun" title="Cost overrun">budget overruns</a>. On the other hand, a principle of <a href="/wiki/Lean_software_development" title="Lean software development">lean software development</a> is to defer decisions until the last responsible moment (M. and T. Poppendieck); however, it is not always clear when the this moment for a particular subset of decisions has come.</li>
<li><i>It facilitates communication with stakeholders, contributing to a system that better fulfills their needs.</i><sup class="reference" id="cite_ref-SAP2_3-8"><a href="#cite_note-SAP2-3">[3]</a></sup><sup class="reference" style="white-space:nowrap;">:p.29–31</sup> Communicating about complex systems from the point of view of stakeholders helps them understand the consequences of their stated requirements and the design decisions based on them. Architecture gives the ability to communicate about design decisions before the system is implemented, when they are still relatively easy to adapt.</li>
<li><i>It helps in risk management.</i> Software architecture helps to reduce risks and chance of failure.<sup class="reference" id="cite_ref-FAIRBANKS2010_11-2"><a href="#cite_note-FAIRBANKS2010-11">[11]</a></sup><sup class="reference" style="white-space:nowrap;">(p18)</sup></li>
<li><i>It enables cost reduction.</i> Software architecture is a means to manage risk and costs in complex IT projects.<sup class="reference" id="cite_ref-RCDA_18-0"><a href="#cite_note-RCDA-18">[18]</a></sup></li>
</ul>
<h2><span class="mw-headline" id="History">History</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Software_architecture&amp;action=edit&amp;section=4" title="Edit section: History">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The comparison between software design and (civil) architecture was first drawn in the late 1960s ,<sup class="reference" id="cite_ref-19"><a href="#cite_note-19">[19]</a></sup> but the term <i>software architecture</i> became prevalent only in the 1990s.<sup class="reference" id="cite_ref-20"><a href="#cite_note-20">[20]</a></sup> The field of <a href="/wiki/Computer_science" title="Computer science">computer science</a> had encountered problems associated with complexity since its formation.<sup class="reference" id="cite_ref-21"><a href="#cite_note-21">[21]</a></sup> Earlier problems of complexity were solved by developers by choosing the right <a href="/wiki/Data_structure" title="Data structure">data structures</a>, developing <a href="/wiki/Algorithm" title="Algorithm">algorithms</a>, and by applying the concept of <a href="/wiki/Separation_of_concerns" title="Separation of concerns">separation of concerns</a>. Although the term "software architecture" is relatively new to the industry, the fundamental principles of the field have been applied sporadically by <a href="/wiki/Software_engineering" title="Software engineering">software engineering</a> pioneers since the mid-1980s. Early attempts to capture and explain software architecture of a system were imprecise and disorganized, often characterized by a set of box-and-line <a href="/wiki/Diagram" title="Diagram">diagrams</a>. <sup class="reference" id="cite_ref-22"><a href="#cite_note-22">[22]</a></sup></p>
<p>Software architecture as a concept has its origins in the research of <a class="mw-redirect" href="/wiki/Edsger_Dijkstra" title="Edsger Dijkstra">Edsger Dijkstra</a> in 1968 and <a href="/wiki/David_Parnas" title="David Parnas">David Parnas</a> in the early 1970s. These scientists emphasized that the structure of a software system matters and getting the structure right is critical. During the 1990s there was a concerted effort to define and codify fundamental aspects of the discipline, with research work concentrating on architectural styles (<a class="mw-redirect" href="/wiki/Patterns" title="Patterns">patterns</a>), <a href="/wiki/Architecture_description_language" title="Architecture description language">architecture description languages</a>, and <a href="/wiki/Software_documentation#Architecture.2FDesign_documentation" title="Software documentation">architecture documentation</a>.<sup class="reference" id="cite_ref-23"><a href="#cite_note-23">[23]</a></sup> Research institutions have played a prominent role in furthering software architecture as a discipline. For instance, <a href="/wiki/Mary_Shaw_(computer_scientist)" title="Mary Shaw (computer scientist)">Mary Shaw</a> and David Garlan of <a class="mw-redirect" href="/wiki/Carnegie_Mellon" title="Carnegie Mellon">Carnegie Mellon</a> wrote a book titled <i>Software Architecture: Perspectives on an Emerging Discipline</i> in 1996, which promoted software architecture concepts such as <a class="mw-redirect" href="/wiki/Software_component" title="Software component">components</a>, connectors, and styles.</p>
<p><a href="/wiki/IEEE_1471" title="IEEE 1471">IEEE 1471</a>-2000, <i>Recommended Practice for Architecture Description of Software-Intensive Systems</i>, was the first formal standard in the area of software architecture. It was adopted in 2007 by ISO as <a href="/wiki/IEEE_1471" title="IEEE 1471">ISO/IEC 42010:2007</a>. In November 2011, IEEE 1471–2000 was superseded by <a href="/wiki/ISO/IEC_42010" title="ISO/IEC 42010">ISO/IEC/IEEE 42010:2011</a>, <i>Systems and software engineering — Architecture description</i> (jointly published by IEEE and ISO).<sup class="reference" id="cite_ref-ISO42010_12-1"><a href="#cite_note-ISO42010-12">[12]</a></sup> While in <a href="/wiki/IEEE_1471" title="IEEE 1471">IEEE 1471</a>, software architecture was about the architecture of "software-intensive systems", defined as "any system where software contributes essential influences to the design, construction, deployment, and evolution of the system as a whole", the 2011 edition goes a step further by including the <a href="/wiki/ISO/IEC_15288" title="ISO/IEC 15288">ISO/IEC 15288</a> and <a href="/wiki/ISO/IEC_12207" title="ISO/IEC 12207">ISO/IEC 12207</a> definitions of a system, which embrace not only hardware and software, but also "humans, processes, procedures, facilities, materials and naturally occurring entities".</p>
<h2><span class="mw-headline" id="Architecture_activities">Architecture activities</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Software_architecture&amp;action=edit&amp;section=5" title="Edit section: Architecture activities">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>There are many activities that a software architect performs. A software architect typically works with project managers, discusses <a href="/wiki/Architecturally_Significant_Requirements" title="Architecturally Significant Requirements">architecturally significant requirements</a> with stakeholders, designs a software architecture, evaluates a design, communicates with designers and stakeholders, documents the architectural design and more.<sup class="reference" id="cite_ref-Kruchten_2008_24-0"><a href="#cite_note-Kruchten_2008-24">[24]</a></sup> There are four core activities in software architecture design.<sup class="reference" id="cite_ref-hofmeister07_25-0"><a href="#cite_note-hofmeister07-25">[25]</a></sup> These core architecture activities are performed iteratively and at different stages of the initial software development life-cycle, as well as over the evolution of a system.</p>
<p><b>Architectural Analysis</b> is the process of understanding the environment in which a proposed system or systems will operate and determining the requirements for the system. The input or requirements to the analysis activity can come from any number of stakeholders and include items such as:</p>
<ul>
<li>what the system will do when operational (the functional requirements)</li>
<li>how well the system will perform runtime non-functional requirements such as reliability, operability, performance efficiency, security, compatibility defined in <a class="mw-redirect" href="/wiki/ISO/IEC_25010" title="ISO/IEC 25010">ISO/IEC 25010</a>:2011 standard <sup class="reference" id="cite_ref-ISO25010_26-0"><a href="#cite_note-ISO25010-26">[26]</a></sup></li>
<li>development-time non-functional requirements such as maintainability and transferability defined in ISO 25010:2011 standard <sup class="reference" id="cite_ref-ISO25010_26-1"><a href="#cite_note-ISO25010-26">[26]</a></sup></li>
<li>business requirements and environmental contexts of a system that may change over time, such as legal, social, financial, competitive, and technology concerns <sup class="reference" id="cite_ref-27"><a href="#cite_note-27">[27]</a></sup></li>
</ul>
<p>The outputs of the analysis activity are those requirements that have a measurable impact on a software system’s architecture, called <a href="/wiki/Architecturally_Significant_Requirements" title="Architecturally Significant Requirements">architecturally significant requirements</a>.<sup class="reference" id="cite_ref-ASR_Chen_28-0"><a href="#cite_note-ASR_Chen-28">[28]</a></sup></p>
<p><b>Architectural Synthesis</b> or design is the process of creating an architecture. Given the <a href="/wiki/Architecturally_Significant_Requirements" title="Architecturally Significant Requirements">architecturally significant requirements</a> determined by the analysis, the current state of the design and the results of any evaluation activities, the design is created and improved. See <sup class="reference" id="cite_ref-SAP2_3-9"><a href="#cite_note-SAP2-3">[3]</a></sup><sup class="reference" style="white-space:nowrap;">:pp.311–326</sup><sup class="reference" id="cite_ref-hofmeister07_25-1"><a href="#cite_note-hofmeister07-25">[25]</a></sup> for a discussion of various techniques for improving a current design.</p>
<p><b>Architecture Evaluation</b> is the process of determining how well the current design or a portion of it satisfies the requirements derived during analysis. An evaluation can occur whenever an architect is considering a design decision, it can occur after some portion of the design has been completed, it can occur after the final design has been completed or it can occur after the system has been constructed. Some of the available software architecture evaluation techniques include <a href="/wiki/Architecture_tradeoff_analysis_method" title="Architecture tradeoff analysis method">Architecture Tradeoff Analysis Method (ATAM)</a> and TARA.<sup class="reference" id="cite_ref-woods2012_29-0"><a href="#cite_note-woods2012-29">[29]</a></sup> Frameworks for comparing the techniques are discussed in <sup class="reference" id="cite_ref-SARA_16-1"><a href="#cite_note-SARA-16">[16]</a></sup> and.<sup class="reference" id="cite_ref-30"><a href="#cite_note-30">[30]</a></sup></p>
<p><b>Architecture Evolution</b> is the process of maintaining and adapting an existing software architecture to meet requirement and environmental changes. As software architecture provides a fundamental structure of a software system, its evolution and maintenance would necessarily impact its fundamental structure. As such, architecture evolution is concerned with adding new functionality as well as maintaining existing functionality and system behaviour, for instance, via architectural refactoring.<sup class="reference" id="cite_ref-AR_OZ_31-0"><a href="#cite_note-AR_OZ-31">[31]</a></sup></p>
<p>Architecture requires critical supporting activities. These supporting activities take place throughout the core software architecture process. They include knowledge management and communication, design reasoning and decision making, and documentation.</p>
<h3><span class="mw-headline" id="Architecture_supporting_activities">Architecture supporting activities</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Software_architecture&amp;action=edit&amp;section=6" title="Edit section: Architecture supporting activities">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Software architecture supporting activities are carried out during core software architecture activities. These supporting activities assist a software architect to carry out analysis, synthesis, evaluation and evolution. For instance, an architect has to gather knowledge, make decisions and document during the analysis phase.</p>
<ul>
<li><b>Knowledge Management and Communication</b> is the activity of exploring and managing knowledge that is essential to designing a software architecture. A software architect does not work in isolation. They get inputs, functional and non-functional requirements and design contexts, from various stakeholders; and provides outputs to stakeholders. Software architecture knowledge is often tacit and is retained in the heads of stakeholders. Software Architecture <a class="mw-redirect" href="/wiki/Knowledge_Management" title="Knowledge Management">Knowledge Management</a> (AKM) is about finding, communicating, and retaining knowledge. As software architecture design issues are intricate and interdependent, a knowledge gap in design reasoning can lead to incorrect software architecture design.<sup class="reference" id="cite_ref-Kruchten_2008_24-1"><a href="#cite_note-Kruchten_2008-24">[24]</a></sup><sup class="reference" id="cite_ref-SAKM_32-0"><a href="#cite_note-SAKM-32">[32]</a></sup> Examples of AKM and communication activities include searching for design patterns, prototyping, asking experienced developers and architects, evaluating the designs of similar systems, sharing knowledge with other designers and stakeholders.</li>
<li><b>Design Reasoning and Decision Making</b> is the activity of evaluating design decisions. This activity is fundamental to all three core software architecture activities.<sup class="reference" id="cite_ref-jansen05_9-1"><a href="#cite_note-jansen05-9">[9]</a></sup><sup class="reference" id="cite_ref-tang09_33-0"><a href="#cite_note-tang09-33">[33]</a></sup> It entails gathering and associating decision contexts, formulating design decision problems, finding solution options and evaluating tradeoffs before making decisions. This process occurs at different levels of decision granularity, while evaluating significant architectural requirements and software architecture decisions, and software architecture analysis, synthesis, and evaluation. Examples of reasoning activities include understanding the impacts of a requirement or a design on quality attributes, questioning the issues that a design might cause, assessing possible solution options, and evaluating the tradeoffs between solutions.</li>
<li><b>Documentation</b> is the activity of recording the design generated during the software architecture process. A system design is described using several views that frequently include a static view showing the code structure of the system, a dynamic view showing the actions of the system during execution, and a deployment view showing how a system is placed on hardware for execution. Kruchten's 4+1 view suggests a description of commonly used views for documenting software architecture;<sup class="reference" id="cite_ref-Kru95_34-0"><a href="#cite_note-Kru95-34">[34]</a></sup> Documenting Software Architectures: Views and Beyond has descriptions of the kinds of notations that could be used within the view description.<sup class="reference" id="cite_ref-DSA2_1-1"><a href="#cite_note-DSA2-1">[1]</a></sup> Examples of documentation activities are writing a specification, recording a system design model, documenting a design rationale, developing a viewpoint, documenting views. Software engineering methods such as the <a href="/wiki/OpenUP" title="OpenUP">OpenUP</a> and architecture design methods such as The Process of Software Architecting (P. Eeles, P. Cripps) suggest artifact (a.k.a. work product) types and templates for these documentation activities; ISO/IEC/IEEE 42010:2011 is accompanied by a documentation template as well (<a class="external free" href="http://www.iso-architecture.org/ieee-1471/templates/" rel="nofollow">http://www.iso-architecture.org/ieee-1471/templates/</a>).</li>
</ul>
<h2><span class="mw-headline" id="Software_architecture_topics">Software architecture topics</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Software_architecture&amp;action=edit&amp;section=7" title="Edit section: Software architecture topics">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<h3><span class="mw-headline" id="Software_architecture_description">Software architecture description</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Software_architecture&amp;action=edit&amp;section=8" title="Edit section: Software architecture description">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="hatnote" role="note">Main article: <a href="/wiki/Software_architecture_description" title="Software architecture description">Software architecture description</a></div>
<p>Software architecture description involves the principles and practices of modeling and representing architectures, using mechanisms such as: architecture description languages, architecture viewpoints, and architecture frameworks.</p>
<h4><span class="mw-headline" id="Architecture_description_languages">Architecture description languages</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Software_architecture&amp;action=edit&amp;section=9" title="Edit section: Architecture description languages">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<div class="hatnote" role="note">Main article: <a href="/wiki/Architecture_description_language" title="Architecture description language">Architecture description language</a></div>
<p>An architecture description language (ADL) is any means of expression used to describe a software architecture (<a href="/wiki/ISO/IEC_42010" title="ISO/IEC 42010">ISO/IEC/IEEE 42010</a>). Many special-purpose ADLs have been developed since the 1990s, including ArchiMate, <a class="mw-redirect" href="/wiki/Architecture_analysis_and_design_language" title="Architecture analysis and design language">AADL</a> (SAE standard), <a href="/wiki/Wright_(ADL)" title="Wright (ADL)">Wright</a> (developed by Carnegie Mellon), <a class="new" href="/w/index.php?title=Acme_(ADL)&amp;action=edit&amp;redlink=1" title="Acme (ADL) (page does not exist)">Acme</a> (developed by Carnegie Mellon), xADL (developed by UCI), <a href="/wiki/Darwin_(ADL)" title="Darwin (ADL)">Darwin</a> (developed by <a href="/wiki/Imperial_College_London" title="Imperial College London">Imperial College London</a>), DAOP-ADL (developed by University of Málaga), SBC-ADL (developed by <a class="mw-redirect" href="/wiki/National_Sun_Yat-Sen_University" title="National Sun Yat-Sen University">National Sun Yat-Sen University</a>), and <a class="new" href="/w/index.php?title=ByADL_(ADL)&amp;action=edit&amp;redlink=1" title="ByADL (ADL) (page does not exist)">ByADL</a> (University of L'Aquila, Italy).</p>
<p>ADLs have not yet succeeded on a broad scale in practice; UML, often profiled, and Informal Rich Pictures (IPRs) a.k.a. box-and-line diagrams dominate. Usage of UML has been criticized by some thought leaders, but successes have also been reported. Simon Brown's Context, Containers, Components, Classes (C4) model is a recent adiditon to the architect's notation toolbox: <a class="external free" href="https://www.voxxed.com/blog/2014/10/simple-sketches-for-diagramming-your-software-architecture/" rel="nofollow">https://www.voxxed.com/blog/2014/10/simple-sketches-for-diagramming-your-software-architecture/</a>.</p>
<p>According to Gregor Hohpe, architects should stop drawing diagrams, in whatever notation, and start communicating: <a class="external free" href="http://www.enterpriseintegrationpatterns.com/ramblings/94_37things.html" rel="nofollow">http://www.enterpriseintegrationpatterns.com/ramblings/94_37things.html</a>.</p>

</div>
<p>Software architecture descriptions are commonly organized into <a href="/wiki/View_model" title="View model">views</a>, which are analogous to the different types of <a href="/wiki/Blueprint" title="Blueprint">blueprints</a> made in building <a href="/wiki/Architecture" title="Architecture">architecture</a>. Each view addresses a set of system concerns, following the conventions of its <i>viewpoint</i>, where a viewpoint is a specification that describes the notations, modeling and analysis techniques to use in a view that express the architecture in question from the perspective of a given set of stakeholders and their concerns (<a href="/wiki/ISO/IEC_42010" title="ISO/IEC 42010">ISO/IEC/IEEE 42010</a>). The viewpoint specifies not only the concerns framed (i.e., to be addressed) but the presentation, model kinds used, conventions used and any consistency (correspondence) rules to keep a view consistent with other views.</p>
<p>Popular viewpoint models include the 4+1 views on software architecture, the viewpoints and perspectices catalog by Nick Rozanski and Eoin Woods, and the IBM ADS viewpoint model by Phlippe Spaas et al.</p>
<h4><span class="mw-headline" id="Architecture_frameworks">Architecture frameworks</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Software_architecture&amp;action=edit&amp;section=11" title="Edit section: Architecture frameworks">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<div class="hatnote" role="note">Main article: <a href="/wiki/Architecture_framework" title="Architecture framework">Architecture framework</a></div>
<p>An architecture framework captures the "conventions, principles and practices for the description of architectures established within a specific domain of application and/or community of stakeholders" (<a href="/wiki/ISO/IEC_42010" title="ISO/IEC 42010">ISO/IEC/IEEE 42010</a>). A framework is usually implemented in terms of one or more viewpoints or ADLs.</p>
<h3><span class="mw-headline" id="Architecture_design_methods">Architecture design methods</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Software_architecture&amp;action=edit&amp;section=12" title="Edit section: Architecture design methods">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Methods define process models (activities performed by roles) and specify artifacts to be created and delivered; they may also suggest technqiues and practices that assist practitioners when performing the activities and producing the artifacts defined in the method. Five such methods are compared and consolidated in.<sup class="reference" id="cite_ref-hofmeister07_25-2"><a href="#cite_note-hofmeister07-25">[25]</a></sup></p>
<h4><span class="mw-headline" id="Software_architecture_and_agile_development">Software architecture and agile development</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Software_architecture&amp;action=edit&amp;section=13" title="Edit section: Software architecture and agile development">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<div class="hatnote" role="note">Main article: <a class="mw-redirect" href="/wiki/Agile_development" title="Agile development">Agile development</a></div>
<p>The importance of architecture was stated in the early works on Agile; for instance, Ken Schwaber's original Scrum paper from OOPSLA '97 has the notion of a pregame, in which a high-level system architecture is established (<a class="external free" href="http://www.jeffsutherland.org/oopsla/schwapub.pdf" rel="nofollow">http://www.jeffsutherland.org/oopsla/schwapub.pdf</a>). However, there are also concerns that software architecture leads to too much <a href="/wiki/Big_Design_Up_Front" title="Big Design Up Front">Big Design Up Front</a>, especially among proponents of <a href="/wiki/Agile_software_development" title="Agile software development">Agile software development</a>. A number of methods have been developed to balance the trade-offs of up-front design and agility,<sup class="reference" id="cite_ref-Boehm2004_35-0"><a href="#cite_note-Boehm2004-35">[35]</a></sup> including the agile method <a href="/wiki/Dynamic_systems_development_method" title="Dynamic systems development method">DSDM</a> which mandates a "Foundations" phase during which "just enough" architectural foundations are laid. <a href="/wiki/IEEE_Software" title="IEEE Software">IEEE Software</a> devoted a special issue<sup class="reference" id="cite_ref-36"><a href="#cite_note-36">[36]</a></sup> to the interaction between agility and architecture. <a href="/wiki/Philippe_Kruchten" title="Philippe Kruchten">P. Kruchten</a>, one of the creators of the Unified Process (UP) and the original 4+1 views on software architecture, summarizes the synergetic relationship in a December 2013 blog post called "Agile architecture".<sup class="reference" id="cite_ref-37"><a href="#cite_note-37">[37]</a></sup></p>
<h3><span class="mw-headline" id="Architectural_styles_and_patterns">Architectural styles and patterns</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Software_architecture&amp;action=edit&amp;section=14" title="Edit section: Architectural styles and patterns">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="hatnote" role="note">Main article: <a class="mw-redirect" href="/wiki/Software_Architecture_styles_and_patterns" title="Software Architecture styles and patterns">Software Architecture styles and patterns</a></div>
<p>An <a href="/wiki/Architectural_pattern" title="Architectural pattern">architectural pattern</a> is a general, reusable solution to a commonly occurring problem in software architecture within a given context. Architectural patterns are often documented as software <a class="mw-redirect" href="/wiki/Design_pattern_(computer_science)" title="Design pattern (computer science)">design patterns</a>.</p>
<p>Following traditional building architecture, a 'software architectural style' is a specific method of construction, characterized by the features that make it notable" (<a href="/wiki/Architectural_style" title="Architectural style">Architectural style</a>). "An architectural style defines: a family of systems in terms of a pattern of structural organization; a vocabulary of components and connectors, with constraints on how they can be combined."<sup class="reference" id="cite_ref-SG_38-0"><a href="#cite_note-SG-38">[38]</a></sup> "Architectural styles are reusable 'packages' of design decisions and constraints that are applied to an architecture to induce chosen desirable qualities."<sup class="reference" id="cite_ref-39"><a href="#cite_note-39">[39]</a></sup></p>
<p>There are many recognized architectural patterns and styles, among them:</p>
<ul>
<li><a class="mw-redirect" href="/wiki/Blackboard_(computing)" title="Blackboard (computing)">Blackboard</a></li>
<li><a href="/wiki/Client%E2%80%93server_model" title="Client–server model">Client-server</a> (2-tier, <a class="mw-redirect" href="/wiki/Three-tier_(computing)" title="Three-tier (computing)">3-tier</a>, <a class="mw-redirect" href="/wiki/N-tier" title="N-tier">n-tier</a>, <a href="/wiki/Cloud_computing" title="Cloud computing">cloud computing</a> exhibit this style)</li>
<li><a class="mw-redirect" href="/wiki/Software_componentry" title="Software componentry">Component-based</a></li>
<li><a href="/wiki/Database-centric_architecture" title="Database-centric architecture">Data-centric</a></li>
<li><a href="/wiki/Event-driven_architecture" title="Event-driven architecture">Event-driven</a> (or <a href="/wiki/Implicit_invocation" title="Implicit invocation">Implicit invocation</a>)</li>
<li><a class="mw-redirect" href="/wiki/Abstraction_(computer_science)#Layered_architecture" title="Abstraction (computer science)">Layered</a> (or <a href="/wiki/Multilayered_architecture" title="Multilayered architecture">Multilayered architecture</a>)</li>
<li><a href="/wiki/Monolithic_application" title="Monolithic application">Monolithic application</a></li>
<li><a href="/wiki/Peer-to-peer" title="Peer-to-peer">Peer-to-peer</a> (P2P)</li>
<li><a class="mw-redirect" href="/wiki/Pipes_and_filters" title="Pipes and filters">Pipes and filters</a></li>
<li><a href="/wiki/Plug-in_(computing)" title="Plug-in (computing)">Plug-ins</a></li>
<li><a href="/wiki/Representational_state_transfer" title="Representational state transfer">Representational state transfer</a> (REST)</li>
<li><a href="/wiki/Rule-based_system" title="Rule-based system">Rule-based</a></li>
<li><a href="/wiki/Service-oriented_architecture" title="Service-oriented architecture">Service-oriented architecture</a> and <a href="/wiki/Microservices" title="Microservices">microservices</a> as its implementation approach</li>
<li><a href="/wiki/Shared_nothing_architecture" title="Shared nothing architecture">Shared nothing architecture</a></li>
<li><a href="/wiki/Space-based_architecture" title="Space-based architecture">Space-based architecture</a></li>
</ul>
<p>Some software architects treat architectural patterns and architectural styles as the same,<sup class="reference" id="cite_ref-MSDN_40-0"><a href="#cite_note-MSDN-40">[40]</a></sup> Others treat styles as compositions of patterns combined with architectural principles that jointly address a particular <i>intent</i>. These two positions have in common that both patterns and styles are idioms for architects to use; they "provide a common language"<sup class="reference" id="cite_ref-MSDN_40-1"><a href="#cite_note-MSDN-40">[40]</a></sup> or "vocabulary"<sup class="reference" id="cite_ref-SG_38-1"><a href="#cite_note-SG-38">[38]</a></sup> with which to describe classes of systems.</p>
<h3><span class="mw-headline" id="Software_architecture_erosion">Software architecture erosion</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Software_architecture&amp;action=edit&amp;section=15" title="Edit section: Software architecture erosion">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Software architecture erosion (or "decay") refers to the gap observed between the planned and actual architecture of a software system as realized in its implementation.<sup class="reference" id="cite_ref-41"><a href="#cite_note-41">[41]</a></sup> Software architecture erosion occurs when implementation decisions either do not fully achieve the architecture-as-planned or otherwise violate constraints or principles of that architecture.<sup class="reference" id="cite_ref-PERRY1992_2-3"><a href="#cite_note-PERRY1992-2">[2]</a></sup> The gap between planned and actual architectures is sometimes understood in terms of the notion of <a href="/wiki/Technical_debt" title="Technical debt">technical debt</a>.</p>
<p>As an example, consider a strictly <a class="mw-redirect" href="/wiki/Abstraction_(computer_science)#Layered_architecture" title="Abstraction (computer science)">layered</a> system, where each layer can only use services provided by the layer immediately below it. Any source code component that does not observe this constraint represents an architecture violation. If not corrected, such violations can transform the architecture into a monolithic block, with adverse effects on understandability, maintainability, and evolvability.</p>
<p>Various approaches have been proposed to address erosion. "These approaches, which include tools, techniques and processes, are primarily classified into three generic categories that attempt to minimise, prevent and repair architecture erosion. Within these broad categories, each approach is further broken down reflecting the high-level strategies adopted to tackle erosion. These are: process-oriented architecture conformance, architecture evolution management, architecture design enforcement, architecture to implementation linkage, self-adaptation and architecture restoration techniques consisting of recovery, discovery and reconciliation."<sup class="reference" id="cite_ref-42"><a href="#cite_note-42">[42]</a></sup></p>
<p>There are two major techniques to detect architectural violations: reflexion models and domain-specific languages. Reflexion model (RM) techniques compare a high-level model provided by the system's architects with the source code implementation. Examples of commercial RM-based tools include the <a class="external text" href="http://www.axivion.com" rel="nofollow">Bauhaus Suite</a> (developed by Axivion), <a class="external text" href="http://www.iese.fraunhofer.de/en/competencies/architecture/tools_architecture.html#contentPar_textblockwithpics" rel="nofollow">SAVE</a> (developed by Fraunhofer IESE) and <a class="external text" href="http://structure101.com/" rel="nofollow">Structure-101</a> (developed by Headway Software). There are also <a href="/wiki/Domain-specific_language" title="Domain-specific language">domain-specific languages</a> with focus on specifying and checking architectural constraints, including .QL (developed by Semmle Limited) and <a class="external text" href="http://aserg.labsoft.dcc.ufmg.br/dclsuite/" rel="nofollow">DCL</a> (from Federal University of Minas Gerais).</p>
<h3><span class="mw-headline" id="Software_architecture_recovery">Software architecture recovery</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Software_architecture&amp;action=edit&amp;section=16" title="Edit section: Software architecture recovery">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="hatnote" role="note">Main article: <a href="/wiki/Software_architecture_recovery" title="Software architecture recovery">Software architecture recovery</a></div>
<p>Software architecture recovery (or reconstruction, or <a href="/wiki/Reverse_engineering" title="Reverse engineering">reverse engineering</a>) includes the methods, techniques and processes to uncover a software system's architecture from available information, including its implementation and documentation. Architecture recovery is often necessary to make informed decisions in the face of obsolete or out-of-date documentation and <a href="/wiki/Software_architecture#Software_architecture_erosion" title="Software architecture">architecture erosion</a>: implementation and maintenance decisions diverging from the envisioned architecture.<sup class="reference" id="cite_ref-43"><a href="#cite_note-43">[43]</a></sup></p>
<h2><span class="mw-headline" id="Related_fields">Related fields</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Software_architecture&amp;action=edit&amp;section=17" title="Edit section: Related fields">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<h3><span class="mw-headline" id="Design">Design</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Software_architecture&amp;action=edit&amp;section=18" title="Edit section: Design">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="hatnote" role="note">Main article: <a href="/wiki/Software_design" title="Software design">Software design</a></div>
<p>Architecture is <a href="/wiki/Software_design" title="Software design">design</a> but not all design is architectural.<sup class="reference" id="cite_ref-DSA2_1-2"><a href="#cite_note-DSA2-1">[1]</a></sup> In practice, the architect is the one who draws the line between software architecture (architectural design) and detailed design (non-architectural design). There are no rules or guidelines that fit all cases, although there have been attempts to formalize the distinction. According to the <i>Intension/Locality Hypothesis</i>,<sup class="reference" id="cite_ref-edenkazman_44-0"><a href="#cite_note-edenkazman-44">[44]</a></sup> the distinction between architectural and detailed design is defined by the <i>Locality Criterion</i>,<sup class="reference" id="cite_ref-edenkazman_44-1"><a href="#cite_note-edenkazman-44">[44]</a></sup> according to which a statement about software design is non-local (architectural) if and only if a program that satisfies it can be expanded into a program that does not. For example, the <a class="mw-redirect" href="/wiki/Client%E2%80%93server" title="Client–server">client–server</a> style is architectural (strategic) because a program that is built on this principle can be expanded into a program that is not client–server—for example, by adding <a href="/wiki/Peer-to-peer" title="Peer-to-peer">peer-to-peer</a> nodes.</p>
<h3><span class="mw-headline" id="Requirements_Engineering">Requirements Engineering</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Software_architecture&amp;action=edit&amp;section=19" title="Edit section: Requirements Engineering">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="hatnote" role="note">Main article: <a href="/wiki/Requirements_engineering" title="Requirements engineering">Requirements engineering</a></div>
<p><a href="/wiki/Requirements_engineering" title="Requirements engineering">Requirements engineering</a> and software architecture can be seen as complementary approaches: while software architecture targets the '<a class="mw-redirect" href="/wiki/Solution_space" title="Solution space">solution space</a>' or the 'how', requirements engineering addresses the '<a class="mw-redirect" href="/wiki/Problem" title="Problem">problem space</a>' or the 'what'.<sup class="reference" id="cite_ref-shekaran94_45-0"><a href="#cite_note-shekaran94-45">[45]</a></sup> Requirements engineering entails the <a href="/wiki/Requirements_elicitation" title="Requirements elicitation">elicitation</a>, <a href="/wiki/Requirements_analysis" title="Requirements analysis">negotiation</a>, <a class="mw-redirect" href="/wiki/Software_Requirements_Specification" title="Software Requirements Specification">specification</a>, <a href="/wiki/Data_validation" title="Data validation">validation</a>, <a href="/wiki/Requirements_traceability" title="Requirements traceability">documentation</a> and <a href="/wiki/Requirements_management" title="Requirements management">management</a> of <a href="/wiki/Requirement" title="Requirement">requirements</a>. Both requirements engineering and software architecture revolve around <a href="/wiki/Stakeholder_(corporate)" title="Stakeholder (corporate)">stakeholder</a> concerns, needs and wishes.</p>
<p>There is considerable overlap between requirements engineering and software architecture, as evidenced for example by a study into five industrial software architecture methods that concludes that <i>"the inputs (goals, constrains, etc.) are usually ill-defined, and only get discovered or better understood as the architecture starts to emerge"</i> and that while <i>"most architectural concerns are expressed as requirements on the system, they can also include mandated design decisions"</i>.<sup class="reference" id="cite_ref-hofmeister07_25-3"><a href="#cite_note-hofmeister07-25">[25]</a></sup> In short, the choice of required behavior given a particular problem impacts the architecture of the solution that addresses that problem, while at the same time the architectural design may impact the problem and introduce new requirements.<sup class="reference" id="cite_ref-boer09_46-0"><a href="#cite_note-boer09-46">[46]</a></sup> Approaches such as the Twin Peaks model <sup class="reference" id="cite_ref-twinpeaks_47-0"><a href="#cite_note-twinpeaks-47">[47]</a></sup> aim to exploit the <a href="/wiki/Synergy" title="Synergy">synergistic</a> relation between requirements and architecture.</p>
<h3><span class="mw-headline" id="Other_types_of_.27architecture.27">Other types of 'architecture'</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Software_architecture&amp;action=edit&amp;section=20" title="Edit section: Other types of 'architecture'">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="hatnote" role="note">Main articles: <a href="/wiki/Computer_architecture" title="Computer architecture">Computer architecture</a>, <a href="/wiki/Systems_architecture" title="Systems architecture">Systems architecture</a>, and <a href="/wiki/Enterprise_architecture" title="Enterprise architecture">Enterprise architecture</a></div>
<dl>
<dt>Computer architecture</dt>
<dd><a href="/wiki/Computer_architecture" title="Computer architecture">Computer architecture</a> targets the internal structure of a computer system, in terms of collaborating hardware components such as the <a href="/wiki/Central_processing_unit" title="Central processing unit">CPU</a> – or processor – the <a href="/wiki/Bus_(computing)" title="Bus (computing)">bus</a> and the <a href="/wiki/Computer_memory" title="Computer memory">memory</a>.</dd>
</dl>
<dl>
<dt>Systems architecture</dt>
<dd>The term <a href="/wiki/Systems_architecture" title="Systems architecture">systems architecture</a> has originally been applied to the architecture of <a href="/wiki/System" title="System">systems</a> that consists of both hardware and <a href="/wiki/Software" title="Software">software</a>. The main concern addressed by the systems architecture, also known as IT architecture, is then the integration of software and hardware in a complete, correctly working device. In another common – much broader – meaning, the term applies to the architecture of any complex system which may be of technical, <a href="/wiki/Sociotechnical_system" title="Sociotechnical system">sociotechnical</a> or social nature.</dd>
</dl>
<dl>
<dt>Enterprise architecture</dt>
<dd>The goal of <a href="/wiki/Enterprise_architecture" title="Enterprise architecture">enterprise architecture</a> is to "translate business vision and strategy into effective enterprise".<sup class="reference" id="cite_ref-48"><a href="#cite_note-48">[48]</a></sup> Enterprise architecture <a href="/wiki/Architecture_framework" title="Architecture framework">frameworks</a>, such as <a class="mw-redirect" href="/wiki/TOGAF" title="TOGAF">TOGAF</a> and the <a href="/wiki/Zachman_Framework" title="Zachman Framework">Zachman Framework</a>, usually distinguish between different enterprise architecture layers. Although terminology differs from framework to framework, many include at least a distinction between a <i><a href="/wiki/Business" title="Business">business</a> layer</i>, an <i><a href="/wiki/Application_software" title="Application software">application</a></i> (or <i><a href="/wiki/Information" title="Information">information</a></i>) <i>layer</i>, and a <i><a href="/wiki/Technology" title="Technology">technology</a> layer</i>. Enterprise architecture addresses among others the alignment between these layers, usually in a top-down approach. Continuing the building architecture metaphor for software architecture, enterprise architecture is analogous to city-level planning.</dd>
</dl>

<h2><span class="mw-headline" id="Further_reading">Further reading</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Software_architecture&amp;action=edit&amp;section=23" title="Edit section: Further reading">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<ul>
<li>Paul Clements, Felix Bachmann, Len Bass, David Garlan, James Ivers, Reed Little, Paulo Merson, Robert Nord, Judith Stafford: <i>Documenting Software Architectures: Views and Beyond, Second Edition</i>. Addison-Wesley, 2010, <a class="internal mw-magiclink-isbn" href="/wiki/Special:BookSources/0321552687">ISBN 0-321-55268-7</a>. This book describes what is software architecture and shows how to document it in multiple views, using UML and other notations. It also explains how to complement the architecture views with behavior, software interface, and rationale documentation. Accompanying the book is a <a class="external text" href="https://wiki.sei.cmu.edu/sad/index.php/The_Adventure_Builder_SAD" rel="nofollow">wiki that contains an example of software architecture documentation</a>.</li>
<li>Len Bass, Paul Clements, Rick Kazman: <i>Software Architecture in Practice, Third Edition</i>. Addison Wesley, 2012, <a class="internal mw-magiclink-isbn" href="/wiki/Special:BookSources/0321815734">ISBN 0-321-81573-4</a> (This book, now in third edition, eloquently covers the fundamental concepts of the discipline. The theme is centered on achieving quality attributes of a system.)</li>
<li>Philippe Kruchten: <i>Architectural Blueprints – the 4+1 View Model of Software Architecture</i>. In: IEEE Software. 12 (6) November 1995, pp. 42–50 (also available online at the <a class="external text" href="http://www3.software.ibm.com/ibmdl/pub/software/rational/web/whitepapers/2003/Pbk4p1.pdf" rel="nofollow">Rational website</a>(PDF))</li>
<li>Martin Fowler (with Ralph Johnson) <a class="external text" href="http://martinfowler.com/ieeeSoftware/whoNeedsArchitect.pdf" rel="nofollow">Who Needs an Architect?</a> IEEE Software, Jul./Aug. 2003</li>
<li>George Fairbanks: <i>Just Enough Software Architecture</i>, proposes a risk-driven approach to design method tailoring and modeling.</li>
<li>Nick Rozanski and Eoin Woods, <i>Software Systems Architecture (2nd edition)</i> introduces core architecture concepts, as well as a viewpoint- and perspective-based architecture framework</li>
<li>Peter Cripps and Peter Eeles: <i>The Process of Software Architecting</i>, a comprehensive collection of architecting tasks, work products harvested from several industrial architecture design methods.</li>
</ul>
