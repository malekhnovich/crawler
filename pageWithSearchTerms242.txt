<p><b>Software deployment</b> is all of the activities that make a <a href="/wiki/Software_system" title="Software system">software system</a> available for use.</p>
<p>The general deployment process consists of several interrelated activities with possible transitions between them. These activities can occur at the <a href="/wiki/Software_developer" title="Software developer">producer</a> side or at the <a href="/wiki/User_(computing)" title="User (computing)">consumer</a> side or both. Because every <a href="/wiki/Software_system" title="Software system">software system</a> is unique, the precise <a href="/wiki/Process_(computing)" title="Process (computing)">processes</a> or <a href="/wiki/Algorithm" title="Algorithm">procedures</a> within each activity can hardly be defined. Therefore, "deployment" should be interpreted as a <i>general process</i> that has to be customized according to specific requirements or characteristics. A brief description of each activity will be presented later.</p>

<h2><span class="mw-headline" id="Deployment_activities">Deployment activities</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Software_deployment&amp;action=edit&amp;section=1" title="Edit section: Deployment activities">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<dl>
<dt>Release </dt>
<dd>The <a class="mw-redirect" href="/wiki/Software_release" title="Software release">release</a> activity follows from the completed <a href="/wiki/Software_development_process" title="Software development process">development</a> process. It includes all the operations to prepare a system for <a href="/wiki/Compiler" title="Compiler">assembly</a> and transfer to the customer site. Therefore, it must determine the <a class="mw-redirect" href="/wiki/Resource_(computer_science)" title="Resource (computer science)">resources</a> required to operate at the customer site and collect information for carrying out subsequent activities of deployment process.</dd>
</dl>
<dl>
<dt>Install and activate </dt>
<dd>Activation is the activity of starting up the <a href="/wiki/Executable" title="Executable">executable</a> component of software. For simple systems, it involves establishing some form of <a href="/wiki/Command_(computing)" title="Command (computing)">command</a> for execution. For complex systems, it should make all the supporting systems ready to use. (Not to be confused with the common use of the term <i>activation</i> concerning a software license, which is a function of <a class="mw-redirect" href="/wiki/Digital_Rights_Management#Limited_install_activations" title="Digital Rights Management">Digital Rights Management</a> systems.)</dd>
</dl>
<dl>
<dd>In larger software deployments, the working copy of the <a href="/wiki/Software" title="Software">software</a> might be installed on a production server in a production environment. Other versions of the deployed software may be installed in a <a class="mw-redirect" href="/wiki/Test_environment" title="Test environment">test environment</a>, <a class="mw-redirect" href="/wiki/Development_environment_(software_development_process)" title="Development environment (software development process)">development environment</a> and disaster recovery environment.</dd>
</dl>
<div class="hatnote" role="note">Further information: <a href="/wiki/Installation_(computer_programs)" title="Installation (computer programs)">Installation (computer programs)</a></div>
<dl>
<dt>Deactivate </dt>
<dd>Deactivation is the inverse of activation, and refers to shutting down any executing components of a system. Deactivation is often required to perform other deployment activities, e.g., a software system may need to be deactivated before an update can be performed. The practice of removing infrequently used or obsolete systems from service is often referred to as <a href="/wiki/Application_retirement" title="Application retirement">application retirement</a> or application decommissioning.</dd>
</dl>
<dl>
<dt>Adapt </dt>
<dd>The <a href="/wiki/Adaptation_(computer_science)" title="Adaptation (computer science)">adaptation</a> activity is also a process to modify a software system that has been previously installed. It differs from updating in that adaptations are initiated by local events such as changing the <a class="mw-redirect" href="/wiki/Runtime_environment" title="Runtime environment">environment</a> of customer site, while updating is mostly started from remote software producer.</dd>
</dl>
<dl>
<dt>Update </dt>
<dd>The update process replaces an earlier version of all or part of a software system with a newer release.</dd>
</dl>
<dl>
<dt>Built-In </dt>
<dd>Mechanisms for installing updates are built into some software systems. Automation of these update processes ranges from fully automatic to user initiated and controlled. <a href="/wiki/Norton_Internet_Security" title="Norton Internet Security">Norton Internet Security</a> is an example of a system with a semi-automatic method for retrieving and installing updates to both the antivirus definitions and other components of the system. Other software products provide query mechanisms for determining when updates are available.</dd>
</dl>
<dl>
<dt>Version tracking </dt>
<dd>Version tracking systems help the user find and install updates to software systems installed on PCs and local networks.
<ul>
<li>Web based version tracking systems notify the user when updates are available for software systems installed on a local system. For example: <a href="/wiki/VersionTracker" title="VersionTracker">VersionTracker Pro</a> checks software versions on a user's computer and then queries its database to see if any updates are available.</li>
<li>Local version tracking system notifies the user when updates are available for software systems installed on a local system. For example: Software Catalog stores version and other information for each software package installed on a local system. One click of a button launches a browser window to the upgrade web page for the application, including auto-filling of the user name and password for sites that require a login.</li>
<li>Browser based version tracking systems notify the user when updates are available for software packages installed on a local system. For example: wfx-Versions is a Firefox extension which helps the user find the current version number of any program listed on the web.</li>
</ul>
</dd>
</dl>
<dl>
<dt>Uninstall </dt>
<dd>Uninstallation is the inverse of installation. It is the removal of a system that is no longer required. It also involves some reconfiguration of other software systems in order to remove the uninstalled system’s <a href="/wiki/Computer_file" title="Computer file">files</a> and <a class="mw-redirect" href="/wiki/Coupling_(computer_science)" title="Coupling (computer science)">dependencies</a>.</dd>
</dl>
<dl>
<dt>Retire </dt>
<dd>Ultimately, a software system is marked as <a class="mw-redirect" href="/wiki/Obsolete" title="Obsolete">obsolete</a> and <a href="/wiki/Technical_support" title="Technical support">support</a> by the producers is withdrawn. It is the end of the <a class="mw-redirect" href="/wiki/Product_life_cycle_management" title="Product life cycle management">life cycle of a software product</a>.</dd>
</dl>
<h2><span class="mw-headline" id="Deployment_roles">Deployment roles</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Software_deployment&amp;action=edit&amp;section=2" title="Edit section: Deployment roles">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<div class="thumb tright">
<div class="thumbinner" style="width:222px;"><a class="image" href="/wiki/File:Value_added_Technology_Deployment.jpg"><img alt="" class="thumbimage" data-file-height="575" data-file-width="800" height="158" src="//upload.wikimedia.org/wikipedia/commons/thumb/e/ee/Value_added_Technology_Deployment.jpg/220px-Value_added_Technology_Deployment.jpg" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/e/ee/Value_added_Technology_Deployment.jpg/330px-Value_added_Technology_Deployment.jpg 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/e/ee/Value_added_Technology_Deployment.jpg/440px-Value_added_Technology_Deployment.jpg 2x" width="220"/></a>

<p>The complexity and variability of software products has fostered the emergence of specialized roles for coordinating and engineering the deployment process. For desktop systems, end-users frequently also become the "software deployers" when they install a software package on their machine. The deployment of <a href="/wiki/Enterprise_software" title="Enterprise software">enterprise software</a> involves many more roles, and those roles typically change as the application progresses from test (pre-production) to production environments. Typical roles involved in software deployments for enterprise applications may include:</p>
<ul>
<li>in pre-production environments:
<ul>
