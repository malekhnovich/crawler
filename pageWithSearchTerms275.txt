<h1 class="firstHeading" id="firstHeading" lang="en">View model</h1>
<div class="mw-body-content" id="bodyContent">
<div id="siteSub">From Wikipedia, the free encyclopedia</div>
<div id="contentSub"></div>
<div class="mw-jump" id="jump-to-nav">

<p>A <b>view model</b> or <b>viewpoints framework</b> in <a href="/wiki/Systems_engineering" title="Systems engineering">systems engineering</a>, <a href="/wiki/Software_engineering" title="Software engineering">software engineering</a>, and <a href="/wiki/Enterprise_engineering" title="Enterprise engineering">enterprise engineering</a> is a framework which defines a coherent set of <i>views</i> to be used in the construction of a <a class="mw-redirect" href="/wiki/System_architecture" title="System architecture">system architecture</a>, <a href="/wiki/Software_architecture" title="Software architecture">software architecture</a>, or <a href="/wiki/Enterprise_architecture" title="Enterprise architecture">enterprise architecture</a>. A <i>view</i> is a representation of a whole system from the perspective of a related set of concerns.<sup class="reference" id="cite_ref-ISO42010_1-0"><a href="#cite_note-ISO42010-1">[1]</a></sup><sup class="reference" id="cite_ref-2"><a href="#cite_note-2">[2]</a></sup></p>
<p>Since the early 1990s there have been a number of efforts to prescribe approaches for describing and analyzing system architectures. These recent efforts define a set of views (or viewpoints). They are sometimes referred to as <a href="/wiki/Architecture_framework" title="Architecture framework">architecture frameworks</a> or <a href="/wiki/Enterprise_architecture_framework" title="Enterprise architecture framework">enterprise architecture frameworks</a>, but are not usually called "view models".</p>
<p>Usually a <i>view</i> is a work product that presents specific architecture data for a given system. However, the same term is sometimes used to refer to a view <i>definition</i>, including the particular viewpoint and the corresponding guidance that defines each concrete view. The term <i>view model</i> is related to view definitions.</p>

<h2><span class="mw-headline" id="Overview">Overview</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=View_model&amp;action=edit&amp;section=1" title="Edit section: Overview">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The purpose of views and viewpoints is to enable humans to comprehend very <a href="/wiki/Complex_system" title="Complex system">complex systems</a>, to organize the elements of the problem and the solution around domains of <a class="mw-redirect" href="/wiki/Expertise" title="Expertise">expertise</a> and to <a href="/wiki/Separation_of_concerns" title="Separation of concerns">separate concerns</a>. In the <a href="/wiki/Engineering" title="Engineering">engineering</a> of physically intensive systems, viewpoints often correspond to capabilities and responsibilities within the engineering organization.<sup class="reference" id="cite_ref-NIST2003_3-0"><a href="#cite_note-NIST2003-3">[3]</a></sup></p>
<p>Most complex system specifications are so extensive that no single individual can fully comprehend all aspects of the specifications. Furthermore, we all have different interests in a given system and different reasons for examining the <a href="/wiki/System" title="System">system</a>'s <a class="mw-redirect" href="/wiki/Specification" title="Specification">specifications</a>. A <a href="/wiki/Business" title="Business">business</a> executive will ask different questions of a system make-up than would a system implementer. The concept of viewpoints framework, therefore, is to provide separate viewpoints into the specification of a given complex system in order to facilitate communication with the stakeholders. Each viewpoint satisfies an audience with interest in a particular set of aspects of the system. Each viewpoint may use a specific <i>viewpoint language</i> that optimizes the vocabulary and presentation for the audience of that viewpoint. Viewpoint modeling has become an effective approach for dealing with the inherent complexity of large distributed systems.</p>
<p>Architecture description practices, as described in <a href="/wiki/IEEE_1471" title="IEEE 1471">IEEE Std 1471-2000</a>, utilize multiple views to address several areas of concerns, each one focusing on a specific aspect of the system. Examples of <a href="/wiki/Architecture_framework" title="Architecture framework">architecture frameworks</a> using multiple views include Kruchten's <a class="mw-redirect" href="/wiki/4%2B1" title="4+1">"4+1" view model</a>, the <a href="/wiki/Zachman_Framework" title="Zachman Framework">Zachman Framework</a>, <a class="mw-redirect" href="/wiki/TOGAF" title="TOGAF">TOGAF</a>, <a class="mw-redirect" href="/wiki/DoDAF" title="DoDAF">DoDAF</a>, <a href="/wiki/RM-ODP" title="RM-ODP">RM-ODP</a>, and Hamdaqa's <a class="mw-redirect" href="/wiki/5%2B1" title="5+1">"5+1" view model</a>.<sup class="reference" id="cite_ref-4"><a href="#cite_note-4">[4]</a></sup></p>
<h2><span class="mw-headline" id="History">History</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=View_model&amp;action=edit&amp;section=2" title="Edit section: History">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>In the 1970s, methods began to appear in software engineering for modeling with multiple views. <a href="/wiki/Douglas_T._Ross" title="Douglas T. Ross">Douglas T. Ross</a> and K.E. Schoman in 1977 introduce the constructs context, viewpoint, and vantage point to organize the modeling process in systems requirements definition.<sup class="reference" id="cite_ref-Ross-Schoman1977_5-0"><a href="#cite_note-Ross-Schoman1977-5">[5]</a></sup> According to Ross and Schoman, a viewpoint "makes clear what aspects are considered relevant to achieving ... the overall purpose [of the model]" and determines <i>How do we look at [a subject being modelled]?</i></p>
<p>As examples of viewpoints, the paper offers: Technical, Operational and Economic viewpoints. In 1992, <a href="/wiki/Anthony_Finkelstein" title="Anthony Finkelstein">Anthony Finkelstein</a> and others published a very important paper on viewpoints.<sup class="reference" id="cite_ref-Finkelstein.2B1992_6-0"><a href="#cite_note-Finkelstein.2B1992-6">[6]</a></sup> In that work: "A viewpoint can be thought of as a combination of the idea of a “actor”, “knowledge source”, “role” or “agent” in the development process and the idea of a “view” or “perspective” which an actor maintains." An important idea in this paper was to distinguish "a <i>representation style</i>, the scheme and notation by which the viewpoint expresses what it can see" and "a <i>specification</i>, the statements expressed in the viewpoint's style describing particular domains". Subsequent work, such as <a href="/wiki/IEEE_1471" title="IEEE 1471">IEEE 1471</a>, preserved this distinction by utilizing two separate terms: viewpoint and view, respectively.</p>
<p>Since the early 1990s there have been a number of efforts to codify approaches for describing and analyzing system architectures. These are often terms <a href="/wiki/Architecture_framework" title="Architecture framework">architecture frameworks</a> or sometimes <i>viewpoint sets</i>. Many of these have been funded by the <a href="/wiki/United_States_Department_of_Defense" title="United States Department of Defense">United States Department of Defense</a>, but some have sprung from international or national efforts in <a href="/wiki/International_Organization_for_Standardization" title="International Organization for Standardization">ISO</a> or the <a class="mw-redirect" href="/wiki/IEEE" title="IEEE">IEEE</a>. Among these, the IEEE Recommended Practice for Architectural Description of Software-Intensive Systems (<a href="/wiki/IEEE_1471" title="IEEE 1471">IEEE Std 1471-2000</a>) established useful definitions of view, viewpoint, stakeholder and concern and guidelines for documenting a <a class="mw-redirect" href="/wiki/System_architecture" title="System architecture">system architecture</a> through the use of multiple views by applying viewpoints to address <a href="/wiki/Stakeholder_analysis" title="Stakeholder analysis">stakeholder concerns</a>.<sup class="reference" id="cite_ref-ShSk_7-0"><a href="#cite_note-ShSk-7">[7]</a></sup></p>
<p><a href="/wiki/IEEE_1471" title="IEEE 1471">IEEE 1471</a> (now <a href="/wiki/ISO/IEC_42010" title="ISO/IEC 42010">ISO/IEC/IEEE 42010:2011</a>, <i>Systems and software engineering — Architecture description</i>) prescribes the contents of architecture descriptions and describes their creation and use under a number of scenarios, including precedented and unprecedented design, evolutionary design, and capture of design of existing systems. In all of these scenarios the overall process is the same: identify <a href="/wiki/Project_stakeholder" title="Project stakeholder">stakeholders</a>, elicit concerns, identify a set of viewpoints to be used, and then apply these viewpoint specifications to develop the set of views relevant to the system of interest. Rather than define a particular set of viewpoints, the standard provides uniform mechanisms and requirements for architects and organizations to define their own viewpoints. In 1996 the ISO Reference Model for Open Distributed Processing (<a href="/wiki/RM-ODP" title="RM-ODP">RM-ODP</a>) was published to provide a useful framework for describing the architecture and design of large-scale distributed systems.</p>
<h2><span class="mw-headline" id="View_model_topics">View model topics</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=View_model&amp;action=edit&amp;section=3" title="Edit section: View model topics">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<h3><span class="mw-headline" id="View">View</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=View_model&amp;action=edit&amp;section=4" title="Edit section: View">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>A view of a system is a representation of the system from the perspective of a viewpoint. This viewpoint on a system involves a perspective focusing on specific concerns regarding the system, which suppresses details to provide a simplified model having only those elements related to the concerns of the viewpoint. For example, a security viewpoint focuses on security concerns and a security viewpoint model contains those elements that are related to security from a more general model of a system.<sup class="reference" id="cite_ref-8"><a href="#cite_note-8">[8]</a></sup></p>
<p>A <a href="/wiki/Graphical_projection" title="Graphical projection">view</a> allows a user to examine a portion of a particular interest area. For example, an Information View may present all functions, organizations, technology, etc. that use a particular piece of information, while the Organizational View may present all functions, technology, and information of concern to a particular organization. In the <a href="/wiki/Zachman_Framework" title="Zachman Framework">Zachman Framework</a> views comprise a group of work products whose development requires a particular analytical and technical expertise because they focus on either the “what,” “how,” “who,” “where,” “when,” or “why” of the enterprise. For example, Functional View work products answer the question “how is the mission carried out?” They are most easily developed by experts in <a href="/wiki/Functional_decomposition" title="Functional decomposition">functional decomposition</a> using process and activity modeling. They show the enterprise from the point of view of functions. They also may show organizational and information components, but only as they relate to functions.<sup class="reference" id="cite_ref-TEAF00_9-0"><a href="#cite_note-TEAF00-9">[9]</a></sup></p>
<h3><span class="mw-headline" id="Viewpoints">Viewpoints</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=View_model&amp;action=edit&amp;section=5" title="Edit section: Viewpoints">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>In systems engineering, a viewpoint is a partitioning or restriction of concerns in a system. Adoption of a viewpoint is usable so that issues in those aspects can be addressed separately. A good selection of viewpoints also partitions the design of the system into specific areas of expertise.<sup class="reference" id="cite_ref-NIST2003_3-1"><a href="#cite_note-NIST2003-3">[3]</a></sup></p>
<p>Viewpoints provide the conventions, rules, and languages for constructing, presenting and analysing views. In ISO/IEC 42010:2007 (<a href="/wiki/IEEE_1471" title="IEEE 1471">IEEE-Std-1471-2000</a>) a viewpoint is a specification for an individual view. A view is a representation of a whole system from the perspective of a viewpoint. A view may consist of one or more <a class="mw-redirect" href="/wiki/Software_Architectural_Model" title="Software Architectural Model">architectural models</a>.<sup class="reference" id="cite_ref-10"><a href="#cite_note-10">[10]</a></sup> Each such architectural model is developed using the methods established by its associated architectural system, as well as for the system as a whole. <sup class="reference" id="cite_ref-ShSk_7-1"><a href="#cite_note-ShSk-7">[7]</a></sup></p>
<h3><span class="mw-headline" id="Modeling_perspectives">Modeling perspectives</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=View_model&amp;action=edit&amp;section=6" title="Edit section: Modeling perspectives">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p><a class="mw-redirect" href="/wiki/Modeling_perspectives" title="Modeling perspectives">Modeling perspectives</a> is a set of different ways to represent pre-selected aspects of a system. Each <a href="/wiki/Perspective_(visual)" title="Perspective (visual)">perspective</a> has a different focus, conceptualization, dedication and visualization of what the <a href="/wiki/Scientific_modelling" title="Scientific modelling">model</a> is representing.</p>
<p>In <a href="/wiki/Information_system" title="Information system">information systems</a>, the traditional way to divide modeling perspectives is to distinguish the structural, functional and behavioral/processual perspectives. This together with rule, object, communication and actor and role perspectives is one way of classifying modeling approaches <sup class="reference" id="cite_ref-11"><a href="#cite_note-11">[11]</a></sup></p>
<h3><span class="mw-headline" id="Viewpoint_model">Viewpoint model</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=View_model&amp;action=edit&amp;section=7" title="Edit section: Viewpoint model">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>In any given viewpoint, it is possible to make a model of the system that contains only the objects that are visible from that viewpoint, but also captures all of the objects, relationships and constraints that are present in the system and relevant to that viewpoint. Such a model is said to be a viewpoint model, or a view of the system from that viewpoint.<sup class="reference" id="cite_ref-NIST2003_3-2"><a href="#cite_note-NIST2003-3">[3]</a></sup></p>
<p>A given view is a specification for the system at a particular level of abstraction from a given viewpoint. Different levels of abstraction contain different levels of detail. Higher-level views allow the engineer to fashion and comprehend the whole design and identify and resolve problems in the large. Lower-level views allow the engineer to concentrate on a part of the design and develop the detailed specifications.<sup class="reference" id="cite_ref-NIST2003_3-3"><a href="#cite_note-NIST2003-3">[3]</a></sup></p>
<div class="thumb tright">
<div class="thumbinner" style="width:202px;"><a class="image" href="/wiki/File:ArchitectureFrameworkStructure.png"><img alt="" class="thumbimage" data-file-height="310" data-file-width="324" height="191" src="//upload.wikimedia.org/wikipedia/en/thumb/1/1c/ArchitectureFrameworkStructure.png/200px-ArchitectureFrameworkStructure.png" srcset="//upload.wikimedia.org/wikipedia/en/thumb/1/1c/ArchitectureFrameworkStructure.png/300px-ArchitectureFrameworkStructure.png 1.5x, //upload.wikimedia.org/wikipedia/en/1/1c/ArchitectureFrameworkStructure.png 2x" width="200"/></a>
<div class="thumbcaption">
<div class="magnify"><a class="internal" href="/wiki/File:ArchitectureFrameworkStructure.png" title="Enlarge"></a></div>
Illustration of the views, products and data in Architecture Framework.</div>
</div>
</div>
<p>In the system itself, however, all of the specifications appearing in the various viewpoint models must be addressed in the realized components of the system. And the specifications for any given component may be drawn from many different viewpoints. On the other hand, the specifications induced by the distribution of functions over specific components and component interactions will typically reflect a different partitioning of concerns than that reflected in the original viewpoints. Thus additional viewpoints, addressing the concerns of the individual components and the bottom-up synthesis of the system, may also be useful.<sup class="reference" id="cite_ref-NIST2003_3-4"><a href="#cite_note-NIST2003-3">[3]</a></sup></p>
<h3><span class="mw-headline" id="Architecture_description">Architecture description</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=View_model&amp;action=edit&amp;section=8" title="Edit section: Architecture description">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>An <a href="/wiki/Software_architecture_description" title="Software architecture description">architecture description</a> is a representation of a system architecture, at any time, in terms of its component parts, how those parts function, the rules and constraints under which those parts function, and how those parts relate to each other and to the environment. In an architecture description the <i>architecture data</i> is shared across several views and products.</p>
<p>At the data layer are the architecture data elements and their defining attributes and relationships. At the presentation layer are the products and views that support a visual means to communicate and understand the purpose of the architecture, what it describes, and the various architectural analyses performed. Products provide a way for visualizing architecture data as graphical, tabular, or textual representations. Views provide the ability to visualize architecture data that stem across products, logically organizing the data for a specific or holistic perspective of the architecture.</p>
<h2><span class="mw-headline" id="Types_of_System_View_Models">Types of System View Models</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=View_model&amp;action=edit&amp;section=9" title="Edit section: Types of System View Models">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<h3><span class="mw-headline" id="Three_schema_approach">Three schema approach</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=View_model&amp;action=edit&amp;section=10" title="Edit section: Three schema approach">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="thumb tright">
<div class="thumbinner" style="width:322px;"><a class="image" href="/wiki/File:4-2_ANSI-SPARC_three_level_architecture.svg"><img alt="" class="thumbimage" data-file-height="431" data-file-width="653" height="211" src="//upload.wikimedia.org/wikipedia/commons/thumb/8/84/4-2_ANSI-SPARC_three_level_architecture.svg/320px-4-2_ANSI-SPARC_three_level_architecture.svg.png" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/8/84/4-2_ANSI-SPARC_three_level_architecture.svg/480px-4-2_ANSI-SPARC_three_level_architecture.svg.png 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/8/84/4-2_ANSI-SPARC_three_level_architecture.svg/640px-4-2_ANSI-SPARC_three_level_architecture.svg.png 2x" width="320"/></a>
<div class="thumbcaption">
<div class="magnify"><a class="internal" href="/wiki/File:4-2_ANSI-SPARC_three_level_architecture.svg" title="Enlarge"></a></div>
The notion of a three-schema model was first introduced in 1977 by the <a href="/wiki/ANSI-SPARC_Architecture" title="ANSI-SPARC Architecture">ANSI/X3/SPARC three level architecture</a>, which determined three levels to model data.<sup class="reference" id="cite_ref-MW99_12-0"><a href="#cite_note-MW99-12">[12]</a></sup></div>
</div>
</div>
<p>The <a class="mw-redirect" href="/wiki/Three_schema_approach" title="Three schema approach">Three schema approach</a> for data modeling, introduced in 1977, can be considered one of the first view models. It is an approach to building information systems and systems information management, that promotes the <a href="/wiki/Conceptual_model" title="Conceptual model">conceptual model</a> as the key to achieving <a href="/wiki/Data_integration" title="Data integration">data integration</a>.<sup class="reference" id="cite_ref-STRAP08_13-0"><a href="#cite_note-STRAP08-13">[13]</a></sup> The Three schema approach defines three schemas and views:</p>
<ul>
<li>External schema for user views</li>
<li><a href="/wiki/Conceptual_schema" title="Conceptual schema">Conceptual schema</a> integrates external schemata</li>
<li>Internal schema that defines physical storage structures</li>
</ul>
<p>At the center, the conceptual schema defines the <a href="/wiki/Ontology" title="Ontology">ontology</a> of the <a href="/wiki/Concept" title="Concept">concepts</a> as the <a href="/wiki/User_(computing)" title="User (computing)">users</a> think of them and talk about them. The physical schema describes the internal formats of the <a href="/wiki/Data" title="Data">data</a> stored in the <a href="/wiki/Database" title="Database">database</a>, and the external schema defines the view of the data presented to the <a class="mw-redirect" href="/wiki/Application_program" title="Application program">application programs</a>.<sup class="reference" id="cite_ref-JFS04_14-0"><a href="#cite_note-JFS04-14">[14]</a></sup> The framework attempted to permit multiple data models to be used for external schemata.<sup class="reference" id="cite_ref-15"><a href="#cite_note-15">[15]</a></sup></p>
<p>Over the years, the skill and interest in building information systems has grown tremendously. However, for the most part, the traditional approach to building systems has only focused on defining <a href="/wiki/Data" title="Data">data</a> from two distinct views, the "user view" and the "computer view". From the user view, which will be referred to as the “external schema,” the definition of data is in the context of reports and screens designed to aid individuals in doing their specific jobs. The required structure of data from a usage view changes with the business environment and the individual preferences of the user. From the computer view, which will be referred to as the “internal schema,” data is defined in terms of file structures for storage and retrieval. The required structure of data for <a class="mw-redirect" href="/wiki/Computer_storage" title="Computer storage">computer storage</a> depends upon the specific computer technology employed and the need for efficient processing of data.<sup class="reference" id="cite_ref-ITL93_16-0"><a href="#cite_note-ITL93-16">[16]</a></sup></p>
<h3><span class="mw-headline" id="4.2B1_view_model_of_architecture">4+1 view model of architecture</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=View_model&amp;action=edit&amp;section=11" title="Edit section: 4+1 view model of architecture">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="thumb tright">
<div class="thumbinner" style="width:356px;"><a class="image" href="/wiki/File:4%2B1_Architectural_View_Model.svg"><img alt="" class="thumbimage" data-file-height="464" data-file-width="708" height="232" src="//upload.wikimedia.org/wikipedia/commons/thumb/e/e6/4%2B1_Architectural_View_Model.svg/354px-4%2B1_Architectural_View_Model.svg.png" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/e/e6/4%2B1_Architectural_View_Model.svg/531px-4%2B1_Architectural_View_Model.svg.png 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/e/e6/4%2B1_Architectural_View_Model.svg/708px-4%2B1_Architectural_View_Model.svg.png 2x" width="354"/></a>
<div class="thumbcaption">
<div class="magnify"><a class="internal" href="/wiki/File:4%2B1_Architectural_View_Model.svg" title="Enlarge"></a></div>
Illustration of the <a class="mw-redirect" href="/wiki/4%2B1" title="4+1">4+1</a> view model or architecture.</div>
</div>
</div>
<p><a class="mw-redirect" href="/wiki/4%2B1" title="4+1">4+1</a> is a view model designed by <a href="/wiki/Philippe_Kruchten" title="Philippe Kruchten">Philippe Kruchten</a> in 1995 for describing the architecture of software-intensive systems, based on the use of multiple, concurrent views.<sup class="reference" id="cite_ref-Kru95_17-0"><a href="#cite_note-Kru95-17">[17]</a></sup> The views are used to describe the system in the viewpoint of different stakeholders, such as end-users, developers and project managers. The four views of the model are logical, development, process and physical view:</p>
<p>The four views of the model are concerned with :</p>
<ul>
<li><i>Logical view</i> : is concerned with the functionality that the system provides to end-users.</li>
<li><i>Development view</i> : illustrates a system from a programmers perspective and is concerned with software management.</li>
<li><i>Process view</i> : deals with the dynamic aspect of the system, explains the system processes and how they communicate, and focuses on the runtime behavior of the system.</li>
<li><i>Physical view</i> : depicts the system from a system engineer's point of view. It is concerned with the topology of software components on the physical layer, as well as communication between these components.</li>
</ul>
<p>In addition selected <a href="/wiki/Use_case" title="Use case">use cases</a> or scenarios are utilized to illustrate the architecture. Hence the model contains 4+1 views.<sup class="reference" id="cite_ref-Kru95_17-1"><a href="#cite_note-Kru95-17">[17]</a></sup></p>
<h2><span class="mw-headline" id="Types_of_Enterprise_Architecture_View">Types of Enterprise Architecture View</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=View_model&amp;action=edit&amp;section=12" title="Edit section: Types of Enterprise Architecture View">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p><a class="mw-redirect" href="/wiki/Enterprise_Architecture_framework" title="Enterprise Architecture framework">Enterprise Architecture framework</a> defines how to organize the structure and views associated with an <a class="mw-redirect" href="/wiki/Enterprise_Architecture" title="Enterprise Architecture">Enterprise Architecture</a>. Because the discipline of Enterprise Architecture and Engineering is so broad, and because enterprises can be large and complex, the models associated with the discipline also tend to be large and complex. To manage this scale and complexity, an Architecture Framework provides tools and methods that can bring the task into focus and allow valuable artifacts to be produced when they are most needed.</p>
<p>Architecture Frameworks are commonly used in <a href="/wiki/Information_technology" title="Information technology">Information technology</a> and <a href="/wiki/Information_system" title="Information system">Information system</a> governance. An organization may wish to mandate that certain models be produced before a <a class="mw-redirect" href="/wiki/System_design" title="System design">system design</a> can be approved. Similarly, they may wish to specify certain views be used in the documentation of procured systems - the <a href="/wiki/United_States_Department_of_Defense" title="United States Department of Defense">U.S. Department of Defense</a> stipulates that specific DoDAF views be provided by equipment suppliers for capital project above a certain value.</p>
<h3><span class="mw-headline" id="Zachman_Framework">Zachman Framework</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=View_model&amp;action=edit&amp;section=13" title="Edit section: Zachman Framework">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="thumb tright">
<div class="thumbinner" style="width:482px;"><a class="image" href="/wiki/File:Simplification_Zachman_Enterprise_Framework.jpg"><img alt="" class="thumbimage" data-file-height="420" data-file-width="892" height="226" src="//upload.wikimedia.org/wikipedia/commons/thumb/3/3e/Simplification_Zachman_Enterprise_Framework.jpg/480px-Simplification_Zachman_Enterprise_Framework.jpg" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/3/3e/Simplification_Zachman_Enterprise_Framework.jpg/720px-Simplification_Zachman_Enterprise_Framework.jpg 1.5x, //upload.wikimedia.org/wikipedia/commons/3/3e/Simplification_Zachman_Enterprise_Framework.jpg 2x" width="480"/></a>
<div class="thumbcaption">
<div class="magnify"><a class="internal" href="/wiki/File:Simplification_Zachman_Enterprise_Framework.jpg" title="Enlarge"></a></div>
Simplified illustration of the <a href="/wiki/Zachman_Framework" title="Zachman Framework">Zachman Framework</a> with an explanation of the rows.<sup class="reference" id="cite_ref-18"><a href="#cite_note-18">[18]</a></sup> The original framework is more advanced, see for an example <a class="external text" href="http://zachmaninternational.com/index.php/ea-articles/26-articles/13-the-zachman-framework-a-concise-definition" rel="nofollow"><i>here</i></a>.</div>
</div>
</div>
<p>The <a href="/wiki/Zachman_Framework" title="Zachman Framework">Zachman Framework</a>, originally conceived by <a href="/wiki/John_Zachman" title="John Zachman">John Zachman</a> at IBM in the 1987, is a framework for enterprise architecture, which provides a formal and highly structured way of viewing and defining an enterprise.</p>
<p>The Framework is used for organizing architectural "artifacts" in a way that takes into account both who the artifact targets (for example, business owner and builder) and what particular issue (for example, data and functionality) is being addressed. These artifacts may include design documents, specifications, and models.<sup class="reference" id="cite_ref-19"><a href="#cite_note-19">[19]</a></sup></p>
<p>The Zachman Framework is often referenced as a standard approach for expressing the basic elements of <a href="/wiki/Enterprise_architecture" title="Enterprise architecture">enterprise architecture</a>. The Zachman Framework has been recognized by the U.S. Federal Government as having "... received worldwide acceptance as an integrated framework for managing change in enterprises and the systems that support them."<sup class="reference" id="cite_ref-20"><a href="#cite_note-20">[20]</a></sup></p>
<h3><span class="mw-headline" id="RM-ODP_views">RM-ODP views</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=View_model&amp;action=edit&amp;section=14" title="Edit section: RM-ODP views">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="thumb tright">
<div class="thumbinner" style="width:322px;"><a class="image" href="/wiki/File:RM-ODP_viewpoints.jpg"><img alt="" class="thumbimage" data-file-height="360" data-file-width="570" height="202" src="//upload.wikimedia.org/wikipedia/commons/thumb/7/7b/RM-ODP_viewpoints.jpg/320px-RM-ODP_viewpoints.jpg" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/7/7b/RM-ODP_viewpoints.jpg/480px-RM-ODP_viewpoints.jpg 1.5x, //upload.wikimedia.org/wikipedia/commons/7/7b/RM-ODP_viewpoints.jpg 2x" width="320"/></a>
<div class="thumbcaption">
<div class="magnify"><a class="internal" href="/wiki/File:RM-ODP_viewpoints.jpg" title="Enlarge"></a></div>
The <a href="/wiki/RM-ODP" title="RM-ODP">RM-ODP</a> view model, which provides five generic and complementary viewpoints on the system and its environment.</div>
</div>
</div>
<p>The International Organization for Standardization (ISO) Reference Model for Open Distributed Processing (<a href="/wiki/RM-ODP" title="RM-ODP">RM-ODP</a>) <sup class="reference" id="cite_ref-21"><a href="#cite_note-21">[21]</a></sup> specifies a set of viewpoints for partitioning the design of a distributed software/hardware system. Since most integration problems arise in the design of such systems or in very analogous situations, these viewpoints may prove useful in separating integration concerns. The RMODP viewpoints are:<sup class="reference" id="cite_ref-NIST2003_3-5"><a href="#cite_note-NIST2003-3">[3]</a></sup></p>
<ul>
<li>the <i>enterprise viewpoint</i>, which is concerned with the purpose and behaviors of the system as it relates to the business objective and the business processes of the organization</li>
<li>the <i>information viewpoint</i>, which is concerned with the nature of the information handled by the system and constraints on the use and interpretation of that information</li>
<li>the <i>computational viewpoint</i>, which is concerned with the functional decomposition of the system into a set of components that exhibit specific behaviors and interact at interfaces</li>
<li>the <i>engineering viewpoint</i>, which is concerned with the mechanisms and functions required to support the interactions of the computational components</li>
<li>the <i>technology viewpoint</i>, which is concerned with the explicit choice of technologies for the implementation of the system, and particularly for the communications among the components</li>
</ul>
<p>RMODP further defines a requirement for a design to contain specifications of consistency between viewpoints, including:<sup class="reference" id="cite_ref-NIST2003_3-6"><a href="#cite_note-NIST2003-3">[3]</a></sup></p>
<ul>
<li>the use of enterprise objects and processes in defining information units</li>
<li>the use of enterprise objects and behaviors in specifying the behaviors of computational components, and use of the information units in defining computational interfaces</li>
<li>the association of engineering choices with computational interfaces and behavior requirements</li>
<li>the satisfaction of information, computational and engineering requirements in the chosen technologies</li>
</ul>
<h3><span class="mw-headline" id="DoDAF_views">DoDAF views</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=View_model&amp;action=edit&amp;section=15" title="Edit section: DoDAF views">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>The <a href="/wiki/Department_of_Defense_Architecture_Framework" title="Department of Defense Architecture Framework">Department of Defense Architecture Framework</a> (DoDAF) defines a standard way to organize an <a href="/wiki/Enterprise_architecture" title="Enterprise architecture">enterprise architecture</a> (EA) or <a href="/wiki/Systems_architecture" title="Systems architecture">systems architecture</a> into complementary and consistent views. It is especially suited to large systems with complex integration and interoperability challenges, and is apparently unique in its use of "<a class="mw-redirect" href="/wiki/Operational_view" title="Operational view">operational views</a>" detailing the external customer's operating domain in which the developing system will operate.</p>
<div class="thumb tright">
<div class="thumbinner" style="width:362px;"><a class="image" href="/wiki/File:DoDAF_Linkages_Among_Views.jpg"><img alt="" class="thumbimage" data-file-height="597" data-file-width="1147" height="187" src="//upload.wikimedia.org/wikipedia/commons/thumb/5/54/DoDAF_Linkages_Among_Views.jpg/360px-DoDAF_Linkages_Among_Views.jpg" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/5/54/DoDAF_Linkages_Among_Views.jpg/540px-DoDAF_Linkages_Among_Views.jpg 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/5/54/DoDAF_Linkages_Among_Views.jpg/720px-DoDAF_Linkages_Among_Views.jpg 2x" width="360"/></a>
<div class="thumbcaption">
<div class="magnify"><a class="internal" href="/wiki/File:DoDAF_Linkages_Among_Views.jpg" title="Enlarge"></a></div>
<a class="mw-redirect" href="/wiki/DoDAF" title="DoDAF">DoDAF</a> linkages among views.<sup class="reference" id="cite_ref-DoD07_22-0"><a href="#cite_note-DoD07-22">[22]</a></sup></div>
</div>
</div>
<p>The DoDAF defines a set of products that act as mechanisms for visualizing, understanding, and assimilating the broad scope and complexities of an architecture description through graphic, tabular, or textual means. These products are organized under four views:</p>
<ul>
<li>Overarching All View (AV),</li>
<li><a href="/wiki/Operational_View" title="Operational View">Operational View</a> (OV),</li>
<li>Systems View (SV), and the</li>
<li>Technical Standards View (TV).</li>
</ul>
<p>Each view depicts certain perspectives of an architecture as described below. Only a subset of the full DoDAF viewset is usually created for each system development. The figure represents the information that links the <a class="mw-redirect" href="/wiki/Operational_view" title="Operational view">operational view</a>, systems and services view, and technical standards view. The three views and their interrelationships driven – by common architecture data elements – provide the basis for deriving measures such as interoperability or performance, and for measuring the impact of the values of these metrics on operational mission and task effectiveness.<sup class="reference" id="cite_ref-DoD07_22-1"><a href="#cite_note-DoD07-22">[22]</a></sup></p>
<h3><span class="mw-headline" id="Federal_Enterprise_Architecture_views">Federal Enterprise Architecture views</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=View_model&amp;action=edit&amp;section=16" title="Edit section: Federal Enterprise Architecture views">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>In the US <a class="mw-redirect" href="/wiki/Federal_Enterprise_Architecture" title="Federal Enterprise Architecture">Federal Enterprise Architecture</a> enterprise, segment, and solution architecture provide different business perspectives by varying the level of detail and addressing related but distinct concerns. Just as enterprises are themselves hierarchically organized, so are the different views provided by each type of architecture. The Federal Enterprise Architecture Practice Guidance (2006) has defined three types of architecture:<sup class="reference" id="cite_ref-FEAPMO06_23-0"><a href="#cite_note-FEAPMO06-23">[23]</a></sup></p>
<div class="thumb tright">
<div class="thumbinner" style="width:422px;"><a class="image" href="/wiki/File:Architectural_Levels_and_Attributes.jpg"><img alt="" class="thumbimage" data-file-height="489" data-file-width="1091" height="188" src="//upload.wikimedia.org/wikipedia/commons/thumb/1/14/Architectural_Levels_and_Attributes.jpg/420px-Architectural_Levels_and_Attributes.jpg" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/1/14/Architectural_Levels_and_Attributes.jpg/630px-Architectural_Levels_and_Attributes.jpg 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/1/14/Architectural_Levels_and_Attributes.jpg/840px-Architectural_Levels_and_Attributes.jpg 2x" width="420"/></a>
<div class="thumbcaption">
<div class="magnify"><a class="internal" href="/wiki/File:Architectural_Levels_and_Attributes.jpg" title="Enlarge"></a></div>
<a class="mw-redirect" href="/wiki/Federal_Enterprise_Architecture" title="Federal Enterprise Architecture">Federal Enterprise Architecture</a> levels and attributes<sup class="reference" id="cite_ref-FEAPMO06_23-1"><a href="#cite_note-FEAPMO06-23">[23]</a></sup></div>
</div>
</div>
<ul>
<li>Enterprise architecture,</li>
<li>Segment architecture, and</li>
<li>Solution architecture.</li>
</ul>
<p>By definition, Enterprise Architecture (EA) is fundamentally concerned with identifying common or shared assets – whether they are strategies, business processes, investments, data, systems, or technologies. EA is driven by strategy; it helps an agency identify whether its resources are properly aligned to the agency mission and strategic goals and objectives. From an investment perspective, EA is used to drive decisions about the IT investment portfolio as a whole. Consequently, the primary stakeholders of the EA are the senior managers and executives tasked with ensuring the agency fulfills its mission as effectively and efficiently as possible.<sup class="reference" id="cite_ref-FEAPMO06_23-2"><a href="#cite_note-FEAPMO06-23">[23]</a></sup></p>
<p>By contrast, segment architecture defines a simple roadmap for a core mission area, business service, or enterprise service. Segment architecture is driven by business management and delivers products that improve the delivery of services to citizens and agency staff. From an investment perspective, segment architecture drives decisions for a business case or group of business cases supporting a core mission area or common or shared service. The primary stakeholders for segment architecture are business owners and managers. Segment architecture is related to EA through three principles: structure, reuse, and alignment. First, segment architecture inherits the framework used by the EA, although it may be extended and specialized to meet the specific needs of a core mission area or common or shared service. Second, segment architecture reuses important assets defined at the enterprise level including: data; common business processes and investments; and applications and technologies. Third, segment architecture aligns with elements defined at the enterprise level, such as business strategies, mandates, standards, and performance measures.<sup class="reference" id="cite_ref-FEAPMO06_23-3"><a href="#cite_note-FEAPMO06-23">[23]</a></sup></p>
<h3><span class="mw-headline" id="Nominal_set_of_views">Nominal set of views</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=View_model&amp;action=edit&amp;section=17" title="Edit section: Nominal set of views">edit</a><span class="mw-editsection-bracket">]</span></span></h3>

</div>
</div>
<p>This "set of views", as described below, is a listing of possible modeling viewpoints. Not all of these views may be used for any one project and other views may be defined as necessary. Note that for some analyses elements from multiple viewpoints may be combined into a new view, possibly using a layered representation.</p>
<p>In a latter presentation this nominal set of views was presented as an Extended RASDS Semantic Information Model Derivation.<sup class="reference" id="cite_ref-ShSk06_24-1"><a href="#cite_note-ShSk06-24">[24]</a></sup> Hereby RASDS stands for Reference Architecture for Space Data Systems. see second image.</p>
<dl>
<dt>Enterprise Viewpoint<sup class="reference" id="cite_ref-ShSk_7-3"><a href="#cite_note-ShSk-7">[7]</a></sup></dt>
</dl>
<ul>
<li>Organization view – Includes <a href="/wiki/Organization" title="Organization">organizational</a> elements and their structures and relationships. May include agreements, contracts, policies and organizational interactions.</li>
<li>Requirements view – Describes the <a href="/wiki/Requirement" title="Requirement">requirements</a>, goals, and objectives that drive the system. Says what the system must be able to do.</li>
<li>Scenario view – Describes the way that the system is intended to be used, see <a href="/wiki/Scenario_planning" title="Scenario planning">scenario planning</a>. Includes user views and descriptions of how the system is expected to behave.</li>
</ul>
<dl>
<dt>Information viewpoint<sup class="reference" id="cite_ref-ShSk_7-4"><a href="#cite_note-ShSk-7">[7]</a></sup></dt>
</dl>
<ul>
<li>Metamodel view – An abstract view that defines <a href="/wiki/Information_model" title="Information model">information model</a> elements and their structures and relationships. Defines the classes of data that are created and managed by the system and the data architecture.</li>
<li>Information view – Describes the actual <a href="/wiki/Data" title="Data">data</a> and <a href="/wiki/Information" title="Information">information</a> as it is realized and manipulated within the system. Data elements are defined by the metamodel view and they are referred to by functional objects in other views.</li>
</ul>

Reference Architecture for Space Data Systems.<sup class="reference" id="cite_ref-ShSk06_24-2"><a href="#cite_note-ShSk06-24">[24]</a></sup></div>
</div>
</div>
<dl>
<dt>Functional viewpoint<sup class="reference" id="cite_ref-ShSk_7-5"><a href="#cite_note-ShSk-7">[7]</a></sup></dt>
</dl>
<ul>
<li>Functional Dataflow view – An abstract view that describes the <a href="/wiki/Function_model" title="Function model">functional elements</a> in the system, their interactions, behavior, provided services, constraints and data flows among them. Defines which functions the system is capable of performing, regardless of how these functions are actually implemented.</li>
<li>Functional Control view – Describes the <a href="/wiki/Control_flow" title="Control flow">control flows</a> and interactions among functional elements within the system. Includes overall system control interactions, interactions between control elements and sensor / effector elements and management interactions.</li>
</ul>
<dl>
<dt>Physical viewpoint<sup class="reference" id="cite_ref-ShSk_7-6"><a href="#cite_note-ShSk-7">[7]</a></sup></dt>
</dl>
<ul>
<li>Data System view – Describes instruments, computers, and data storage components, their data system attributes and the communications connectors (busses, networks, point to point links) that are used in the system.</li>
<li>Telecomm view – Describes the telecomm components (antenna, transceiver), their attributes and their connectors (RF or optical links).</li>
<li>Navigation view – Describes the motion of the major elements within the system (trajectory, path, orbit), including their interaction with external elements and forces that are outside of the control of the system, but that must be modeled with it to understand system behavior (planets, asteroids, solar pressure, gravity)</li>
<li>Structural view – Describes the structural components in the system (s/c bus, struts, panels, articulation), their physical attributes and connectors, along with the relevant structural aspects of other components (mass, stiffness, attachment)</li>
<li>Thermal view – Describes the active and passive thermal components in the system (radiators, coolers, vents) and their connectors (physical and free space radiation) and attributes, along with the thermal properties of other components (i.e. antenna as sun shade)</li>
<li>Power view – Describes the active and passive power components in the system (solar panels, batteries, RTGs) within the system and their connectors, along with the power properties of other components (data system and propulsion elements as power sinks and structural panels as grounding plane)</li>
<li>Propulsion view – Describes the active and passive propulsion components in the system (thrusters, gyros, motors, wheels) within the system and their connectors, along with the propulsive properties of other components</li>
</ul>

<div class="thumbinner" style="width:362px;"><a class="image" href="/wiki/File:MBED_Top_Level_Ontology.jpg"><img alt="" class="thumbimage" data-file-height="608" data-file-width="697" height="314" src="//upload.wikimedia.org/wikipedia/commons/thumb/b/b1/MBED_Top_Level_Ontology.jpg/360px-MBED_Top_Level_Ontology.jpg" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/b/b1/MBED_Top_Level_Ontology.jpg/540px-MBED_Top_Level_Ontology.jpg 1.5x, //upload.wikimedia.org/wikipedia/commons/b/b1/MBED_Top_Level_Ontology.jpg 2x" width="360"/></a>

</div>
</div>
<dl>
<dt>Engineering viewpoint<sup class="reference" id="cite_ref-ShSk_7-8"><a href="#cite_note-ShSk-7">[7]</a></sup></dt>
</dl>
<ul>
<li>Allocation view – Describes the allocation of functional objects to engineered physical and computational components within the system, permits analysis of performance and used to verify satisfaction of requirements</li>
<li>Software view - Describes the software engineering aspects of the system, software design and implementation of functionality within software components, select languages and libraries to be used, define APIs, do the engineering of abstract functional objects into tangible software elements. Some functional elements, described using a software language, may actually be implemented as hardware (FPGA, ASIC)</li>
<li>Hardware views – Describes the hardware engineering aspects of the system, hardware design, selection and implementation of all of the physical components to be assembled into the system. There may be many of these views, each specific to a different engineering discipline.</li>
<li>Communications Protocol view – Describes the end to end design of the communications protocols and related data transport and data management services, shows the protocol stacks as they are implemented on each of the physical components of the system.</li>
<li>Risk view – Describes the risks associated with the system design, processes, and technologies, assigns additional risk assessment attributes to other elements described in the architecture</li>
<li>Control Engineering view - Analyzes system from the perspective of its controllability, allocation of elements into system under control and control system</li>
<li>Integration and Test view – Looks at the system from the perspective of what must be done to assemble, integrate and test system and sub-systems, and assemblies. Includes verification of proper functionality, driven by scenarios, in satisfaction of requirements.</li>
<li>IV&amp;V view – independent validation and verification of functionality and proper operation of the system in satisfaction of requirements. Does system as designed and developed meet goals and objectives.</li>
</ul>
<dl>
<dt>Technology viewpoint<sup class="reference" id="cite_ref-ShSk_7-9"><a href="#cite_note-ShSk-7">[7]</a></sup></dt>
</dl>
<ul>
<li>Standards view – Defines the standards to be adopted during design of the system (e.g. communication protocols, radiation tolerance, soldering). These are essentially constraints on the design and implementation processes.</li>
<li>Infrastructure view – Defines the infrastructure elements that are to support the engineering, design, and fabrication process. May include data system elements (design repositories, frameworks, tools, networks) and hardware elements (chip fabrication, thermal vacuum facility, machine shop, RF testing lab)</li>
<li>Technology Development &amp; Assessment view – Includes description of technology development programs designed to produce algorithms or components that may be included in a system development project. Includes evaluation of properties of selected hardware and software components to determine if they are at a sufficient state of maturity to be adopted for the mission being designed.</li>
</ul>
<p>In contrast to the previous listed view models, this "nominal set of views" lists a whole range of views, possible to develop powerful and extensible approaches for describing a general class of software intensive system architectures.<sup class="reference" id="cite_ref-ShSk_7-10"><a href="#cite_note-ShSk-7">[7]</a></sup></p>
