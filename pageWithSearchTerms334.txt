<h1 class="firstHeading" id="firstHeading" lang="en">Database</h1>
<div class="mw-body-content" id="bodyContent">
<div id="siteSub">From Wikipedia, the free encyclopedia</div>
<div id="contentSub"><span class="mw-redirectedfrom">  (Redirected from <a class="mw-redirect" href="/w/index.php?title=Database_management_system&amp;redirect=no" title="Database management system">Database management system</a>)</span></div>
<div class="mw-jump" id="jump-to-nav">
					Jump to:					<a href="#mw-head">navigation</a>, 					<a href="#p-search">search</a>
</div>
<div class="mw-content-ltr" dir="ltr" id="mw-content-text" lang="en"><div class="hatnote" role="note">"Database Software" redirects here. For the computer program, see <a href="/wiki/Europress" title="Europress">Europress</a>.</div>
<p>A <b>database</b> is an organized collection of <a href="/wiki/Data_(computing)" title="Data (computing)">data</a>.<sup class="reference" id="cite_ref-1"><a href="#cite_note-1">[1]</a></sup> It is the collection of <a href="/wiki/Database_schema" title="Database schema">schemas</a>, <a href="/wiki/Table_(database)" title="Table (database)">tables</a>, <a href="/wiki/Query_language" title="Query language">queries</a>, reports, <a href="/wiki/View_(SQL)" title="View (SQL)">views</a>, and other objects. The data are typically organized to model aspects of reality in a way that supports <a href="/wiki/Process_(computing)" title="Process (computing)">processes</a> requiring information, such as modelling the availability of rooms in hotels in a way that supports finding a hotel with vacancies.</p>
<p>A <b>database management system</b> (<b>DBMS</b>) is a <a class="mw-redirect" href="/wiki/Computer_software" title="Computer software">computer software</a> application that interacts with the user, other applications, and the database itself to capture and analyze data. A general-purpose DBMS is designed to allow the definition, creation, querying, update, and administration of databases. Well-known DBMSs include <a href="/wiki/MySQL" title="MySQL">MySQL</a>, <a href="/wiki/PostgreSQL" title="PostgreSQL">PostgreSQL</a>, <a href="/wiki/MongoDB" title="MongoDB">MongoDB</a>, <a href="/wiki/Microsoft_SQL_Server" title="Microsoft SQL Server">Microsoft SQL Server</a>, <a href="/wiki/Oracle_Database" title="Oracle Database">Oracle</a>, <a href="/wiki/Sybase" title="Sybase">Sybase</a>, <a href="/wiki/SAP_HANA" title="SAP HANA">SAP HANA</a>, and <a href="/wiki/IBM_DB2" title="IBM DB2">IBM DB2</a>. A database is not generally <a href="/wiki/Software_portability" title="Software portability">portable</a> across different DBMSs, but different DBMS can interoperate by using <a href="/wiki/Technical_standard" title="Technical standard">standards</a> such as <a href="/wiki/SQL" title="SQL">SQL</a> and <a class="mw-redirect" href="/wiki/ODBC" title="ODBC">ODBC</a> or <a class="mw-redirect" href="/wiki/JDBC" title="JDBC">JDBC</a> to allow a single application to work with more than one DBMS. Database management systems are often classified according to the <a href="/wiki/Database_model" title="Database model">database model</a> that they support; the most popular database systems since the 1980s have all supported the <a href="/wiki/Relational_model" title="Relational model">relational model</a> as represented by the <a href="/wiki/SQL" title="SQL">SQL</a> language.<sup class="noprint Inline-Template" style="white-space:nowrap;">[<i><a class="mw-redirect" href="/wiki/Wikipedia:Disputed_statement" title="Wikipedia:Disputed statement"><span title="This claim has reliable sources with contradicting facts (June 2015)">disputed</span></a> <span class="metadata">– <a href="/wiki/Talk:Database#All_SQL.3F" title="Talk:Database">discuss</a></span></i>]</sup> Sometimes a DBMS is loosely referred to as a 'database'.</p>

<h2><span class="mw-headline" id="Terminology_and_overview">Terminology and overview</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Database&amp;action=edit&amp;section=1" title="Edit section: Terminology and overview">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Formally, a "database" refers to a set of related data and the way it is organized. Access to these data is usually provided by a "database management system" (DBMS) consisting of an integrated set of computer software that allows <a href="/wiki/User_(computing)" title="User (computing)">users</a> to interact with one or more databases and provides access to all of the data contained in the database (although restrictions may exist that limit access to particular data). The DBMS provides various functions that allow entry, storage and retrieval of large quantities of information and provides ways to manage how that information is organized.</p>
<p>Because of the close relationship between them, the term "database" is often used casually to refer to both a database and the DBMS used to manipulate it.</p>
<p>Outside the world of professional <a href="/wiki/Information_technology" title="Information technology">information technology</a>, the term <i>database</i> is often used to refer to any collection of related data (such as a <a href="/wiki/Spreadsheet" title="Spreadsheet">spreadsheet</a> or a card index). This article is concerned only with databases where the size and usage requirements necessitate use of a database management system.<sup class="reference" id="cite_ref-FOOTNOTEUllmanWidom19971_2-0"><a href="#cite_note-FOOTNOTEUllmanWidom19971-2">[2]</a></sup></p>
<p>Existing DBMSs provide various functions that allow management of a database and its data which can be classified into four main functional groups:</p>
<ul>
<li>Data definition – Creation, modification and removal of definitions that define the organization of the data.</li>
<li>Update – Insertion, modification, and deletion of the actual data.<sup class="reference" id="cite_ref-3"><a href="#cite_note-3">[3]</a></sup></li>
<li>Retrieval – Providing information in a form directly usable or for further processing by other applications. The retrieved data may be made available in a form basically the same as it is stored in the database or in a new form obtained by altering or combining existing data from the database.<sup class="reference" id="cite_ref-4"><a href="#cite_note-4">[4]</a></sup></li>
<li>Administration – Registering and monitoring users, enforcing data security, monitoring performance, maintaining data integrity, dealing with concurrency control, and recovering information that has been corrupted by some event such as an unexpected system failure.<sup class="reference" id="cite_ref-5"><a href="#cite_note-5">[5]</a></sup></li>
</ul>
<p>Both a database and its DBMS conform to the principles of a particular <a href="/wiki/Database_model" title="Database model">database model</a>.<sup class="reference" id="cite_ref-FOOTNOTETsitchizrisLochovsky1982_6-0"><a href="#cite_note-FOOTNOTETsitchizrisLochovsky1982-6">[6]</a></sup> "Database system" refers collectively to the database model, database management system, and database.<sup class="reference" id="cite_ref-FOOTNOTEBeynon.E2.80.93Davies2004_7-0"><a href="#cite_note-FOOTNOTEBeynon.E2.80.93Davies2004-7">[7]</a></sup></p>
<p>Physically, database <a href="/wiki/Server_(computing)" title="Server (computing)">servers</a> are dedicated computers that hold the actual databases and run only the DBMS and related software. Database servers are usually <a class="mw-redirect" href="/wiki/Multiprocessor" title="Multiprocessor">multiprocessor</a> computers, with generous memory and <a class="mw-redirect" href="/wiki/Redundant_array_of_independent_disks" title="Redundant array of independent disks">RAID</a> disk arrays used for stable storage. RAID is used for recovery of data if any of the disks fail. Hardware database accelerators, connected to one or more servers via a high-speed channel, are also used in large volume transaction processing environments. DBMSs are found at the heart of most <a href="/wiki/Database_application" title="Database application">database applications</a>. DBMSs may be built around a custom <a href="/wiki/Computer_multitasking" title="Computer multitasking">multitasking</a> <a class="mw-redirect" href="/wiki/Kernel_(computing)" title="Kernel (computing)">kernel</a> with built-in <a href="/wiki/Computer_network" title="Computer network">networking</a> support, but modern DBMSs typically rely on a standard <a href="/wiki/Operating_system" title="Operating system">operating system</a> to provide these functions from databases before the inception of Structured Query Language (SQL). The data recovered was disparate, redundant and disorderly, since there was no proper method to fetch it and arrange it in a concrete structure.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (April 2016)">citation needed</span></a></i>]</sup></p>
<p>Since DBMSs comprise a significant <a href="/wiki/Economy" title="Economy">economical</a> <a href="/wiki/Market_(economics)" title="Market (economics)">market</a>, computer and storage vendors often take into account DBMS requirements in their own development plans.<sup class="reference" id="cite_ref-FOOTNOTENelsonNelson2001_8-0"><a href="#cite_note-FOOTNOTENelsonNelson2001-8">[8]</a></sup></p>
<p>Databases and DBMSs can be categorized according to the database model(s) that they support (such as relational or XML), the type(s) of computer they run on (from a server cluster to a mobile phone), the <a href="/wiki/Query_language" title="Query language">query language</a>(s) used to access the database (such as SQL or <a href="/wiki/XQuery" title="XQuery">XQuery</a>), and their internal engineering, which affects performance, <a href="/wiki/Scalability" title="Scalability">scalability</a>, resilience, and security.</p>

<p>Databases are used to support internal operations of organizations and to underpin online interactions with customers and suppliers (see <a href="/wiki/Enterprise_software" title="Enterprise software">Enterprise software</a>).</p>
<p>Databases are used to hold administrative information and more specialized data, such as engineering data or economic models. Examples of database applications include computerized <a href="/wiki/Library" title="Library">library</a> systems, <a class="mw-redirect" href="/wiki/Flight_reservation_system" title="Flight reservation system">flight reservation systems</a>, computerized <a class="mw-redirect" href="/wiki/Parts_inventory_system" title="Parts inventory system">parts inventory systems</a>, and many <a href="/wiki/Content_management_system" title="Content management system">content management systems</a> that store <a href="/wiki/Website" title="Website">websites</a> as collections of webpages in a database.</p>
<h2><span class="mw-headline" id="General-purpose_and_special-purpose_DBMSs">General-purpose and special-purpose DBMSs</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Database&amp;action=edit&amp;section=3" title="Edit section: General-purpose and special-purpose DBMSs">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>A DBMS has evolved into a complex software system and its development typically requires thousands of human years of development effort.<sup class="reference" id="cite_ref-9"><a href="#cite_note-9">[a]</a></sup> Some general-purpose DBMSs such as <a class="mw-redirect" href="/wiki/Adabas" title="Adabas">Adabas</a>, <a href="/wiki/Oracle_Corporation" title="Oracle Corporation">Oracle</a> and DB2 have been undergoing upgrades since the 1970s. General-purpose DBMSs aim to meet the needs of as many applications as possible, which adds to the complexity. However, the fact that their development cost can be spread over a large number of users means that they are often the most cost-effective approach. However, a general-purpose DBMS is not always the optimal solution: in some cases a general-purpose DBMS may introduce unnecessary overhead. Therefore, there are many examples of systems that use special-purpose databases. A common example is an <a href="/wiki/Email" title="Email">email</a> system that performs many of the functions of a general-purpose DBMS such as the insertion and deletion of messages composed of various items of data or associating messages with a particular email address; but these functions are limited to what is required to handle email and don't provide the user with all of the functionality that would be available using a general-purpose DBMS.</p>
<p>Many other databases have <a href="/wiki/Application_software" title="Application software">application software</a> that accesses the database on behalf of end-users, without exposing the DBMS interface directly. Application programmers may use a <a href="/wiki/Wire_protocol" title="Wire protocol">wire protocol</a> directly, or more likely through an <a href="/wiki/Application_programming_interface" title="Application programming interface">application programming interface</a>. Database designers and database administrators interact with the DBMS through dedicated interfaces to build and maintain the applications' databases, and thus need some more knowledge and understanding about how DBMSs operate and the DBMSs' external interfaces and tuning parameters.</p>
<h2><span class="mw-headline" id="History">History</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Database&amp;action=edit&amp;section=4" title="Edit section: History">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Following the <a href="/wiki/Technology" title="Technology">technology</a> progress in the areas of <a class="mw-redirect" href="/wiki/Processors" title="Processors">processors</a>, <a href="/wiki/Computer_memory" title="Computer memory">computer memory</a>, <a class="mw-redirect" href="/wiki/Computer_storage" title="Computer storage">computer storage</a>, and <a class="mw-redirect" href="/wiki/Computer_networks" title="Computer networks">computer networks</a>, the sizes, capabilities, and performance of databases and their respective DBMSs have grown in orders of magnitude. The development of database technology can be divided into three eras based on data model or structure: <a href="/wiki/Navigational_database" title="Navigational database">navigational</a>,<sup class="reference" id="cite_ref-FOOTNOTEBachman1973_10-0"><a href="#cite_note-FOOTNOTEBachman1973-10">[9]</a></sup> SQL/<a href="/wiki/Relational_database" title="Relational database">relational</a>, and post-relational.</p>
<p>The two main early navigational data models were the <a href="/wiki/Hierarchical_database_model" title="Hierarchical database model">hierarchical model</a>, epitomized by IBM's IMS system, and the <a href="/wiki/CODASYL" title="CODASYL">CODASYL</a> model (<a href="/wiki/Network_model" title="Network model">network model</a>), implemented in a number of products such as <a href="/wiki/IDMS" title="IDMS">IDMS</a>.</p>
<p>The <a href="/wiki/Relational_model" title="Relational model">relational model</a>, first proposed in 1970 by <a href="/wiki/Edgar_F._Codd" title="Edgar F. Codd">Edgar F. Codd</a>, departed from this tradition by insisting that applications should search for data by content, rather than by following links. The relational model employs sets of ledger-style tables, each used for a different type of entity. Only in the mid-1980s did computing hardware become powerful enough to allow the wide deployment of relational systems (DBMSs plus applications). By the early 1990s, however, relational systems dominated in all large-scale <a href="/wiki/Data_processing" title="Data processing">data processing</a> applications, and as of 2015<sup class="plainlinks noprint asof-tag update" style="display:none;"><a class="external text" href="//en.wikipedia.org/w/index.php?title=Database&amp;action=edit">[update]</a></sup> they remain dominant : <a href="/wiki/IBM_DB2" title="IBM DB2">IBM DB2</a>, <a class="mw-redirect" href="/wiki/Oracle_database" title="Oracle database">Oracle</a>, <a href="/wiki/MySQL" title="MySQL">MySQL</a>, and <a href="/wiki/Microsoft_SQL_Server" title="Microsoft SQL Server">Microsoft SQL Server</a> are the top <a class="mw-redirect" href="/wiki/DBMS" title="DBMS">DBMS</a>.<sup class="reference" id="cite_ref-11"><a href="#cite_note-11">[10]</a></sup> The dominant database language, standardised SQL for the relational model, has influenced database languages for other data models.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (March 2013)">citation needed</span></a></i>]</sup></p>
<p><a href="/wiki/Object_database" title="Object database">Object databases</a> were developed in the 1980s to overcome the inconvenience of <a href="/wiki/Object-relational_impedance_mismatch" title="Object-relational impedance mismatch">object-relational impedance mismatch</a>, which led to the coining of the term "post-relational" and also the development of hybrid <a href="/wiki/Object-relational_database" title="Object-relational database">object-relational databases</a>.</p>
<p>The next generation of post-relational databases in the late 2000s became known as <a href="/wiki/NoSQL" title="NoSQL">NoSQL</a> databases, introducing fast <a class="mw-redirect" href="/wiki/Key-value_store" title="Key-value store">key-value stores</a> and <a href="/wiki/Document-oriented_database" title="Document-oriented database">document-oriented databases</a>. A competing "next generation" known as <a href="/wiki/NewSQL" title="NewSQL">NewSQL</a> databases attempted new implementations that retained the relational/SQL model while aiming to match the high performance of NoSQL compared to commercially available relational DBMSs.</p>
<h3><span class="mw-headline" id="1960s.2C_navigational_DBMS">1960s, navigational DBMS</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Database&amp;action=edit&amp;section=5" title="Edit section: 1960s, navigational DBMS">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="hatnote" role="note">Further information: <a href="/wiki/Navigational_database" title="Navigational database">Navigational database</a></div>
<div class="thumb tright">
<div class="thumbinner" style="width:282px;"><a class="image" href="/wiki/File:CodasylB.png"><img alt="" class="thumbimage" data-file-height="1405" data-file-width="850" height="463" src="//upload.wikimedia.org/wikipedia/commons/thumb/d/d6/CodasylB.png/280px-CodasylB.png" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/d/d6/CodasylB.png/420px-CodasylB.png 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/d/d6/CodasylB.png/560px-CodasylB.png 2x" width="280"/></a>
<div class="thumbcaption">
<div class="magnify"><a class="internal" href="/wiki/File:CodasylB.png" title="Enlarge"></a></div>
Basic structure of navigational <a href="/wiki/CODASYL" title="CODASYL">CODASYL</a> database model</div>
</div>
</div>
<p>The introduction of the term <i>database</i> coincided with the availability of direct-access storage (disks and drums) from the mid-1960s onwards. The term represented a contrast with the tape-based systems of the past, allowing shared interactive use rather than daily <a href="/wiki/Batch_processing" title="Batch processing">batch processing</a>. The <a href="/wiki/Oxford_English_Dictionary" title="Oxford English Dictionary">Oxford English Dictionary</a> cites<sup class="reference" id="cite_ref-12"><a href="#cite_note-12">[11]</a></sup> a 1962 report by the System Development Corporation of California as the first to use the term "data-base" in a specific technical sense.</p>
<p>As computers grew in speed and capability, a number of general-purpose database systems emerged; by the mid-1960s a number of such systems had come into commercial use. Interest in a standard began to grow, and <a href="/wiki/Charles_Bachman" title="Charles Bachman">Charles Bachman</a>, author of one such product, the <a href="/wiki/Integrated_Data_Store" title="Integrated Data Store">Integrated Data Store</a> (IDS), founded the "Database Task Group" within <a href="/wiki/CODASYL" title="CODASYL">CODASYL</a>, the group responsible for the creation and standardization of <a href="/wiki/COBOL" title="COBOL">COBOL</a>. In 1971, the Database Task Group delivered their standard, which generally became known as the "CODASYL approach", and soon a number of commercial products based on this approach entered the market.</p>
<p>The CODASYL approach relied on the "manual" navigation of a linked data set which was formed into a large network. Applications could find records by one of three methods:</p>
<ol>
<li>Use of a primary key (known as a CALC key, typically implemented by <a href="/wiki/Hash_function" title="Hash function">hashing</a>)</li>
<li>Navigating relationships (called sets) from one record to another</li>
<li>Scanning all the records in a sequential order</li>
</ol>
<p>Later systems added <a href="/wiki/B-tree" title="B-tree">B-trees</a> to provide alternate access paths. Many CODASYL databases also added a very straightforward query language. However, in the final tally, CODASYL was very complex and required significant training and effort to produce useful applications.</p>
<p><a class="mw-redirect" href="/wiki/International_Business_Machines" title="International Business Machines">IBM</a> also had their own DBMS in 1966, known as <a href="/wiki/IBM_Information_Management_System" title="IBM Information Management System">Information Management System</a> (IMS). IMS was a development of software written for the <a href="/wiki/Apollo_program" title="Apollo program">Apollo program</a> on the <a class="mw-redirect" href="/wiki/System/360" title="System/360">System/360</a>. IMS was generally similar in concept to CODASYL, but used a strict hierarchy for its model of data navigation instead of CODASYL's network model. Both concepts later became known as navigational databases due to the way data was accessed, and Bachman's 1973 <a href="/wiki/Turing_Award" title="Turing Award">Turing Award</a> presentation was <i>The Programmer as Navigator</i>. IMS is classified<sup class="noprint Inline-Template" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Manual_of_Style/Words_to_watch#Unsupported_attributions" title="Wikipedia:Manual of Style/Words to watch"><span title="The material near this tag may use weasel words or too-vague attribution. (March 2014)">by whom?</span></a></i>]</sup> as a <a class="mw-redirect" href="/wiki/Hierarchical_database" title="Hierarchical database">hierarchical database</a>. IDMS and <a href="/wiki/Cincom_Systems" title="Cincom Systems">Cincom Systems</a>' <a href="/wiki/Cincom_Systems#1970s" title="Cincom Systems">TOTAL</a> database are classified as network databases. IMS remains in use as of 2014<sup class="plainlinks noprint asof-tag update" style="display:none;"><a class="external text" href="//en.wikipedia.org/w/index.php?title=Database&amp;action=edit">[update]</a></sup>.<sup class="reference" id="cite_ref-13"><a href="#cite_note-13">[12]</a></sup></p>
<h3><span class="mw-headline" id="1970s.2C_relational_DBMS">1970s, relational DBMS</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Database&amp;action=edit&amp;section=6" title="Edit section: 1970s, relational DBMS">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Edgar Codd worked at IBM in <a href="/wiki/San_Jose,_California" title="San Jose, California">San Jose, California</a>, in one of their offshoot offices that was primarily involved in the development of <a class="mw-redirect" href="/wiki/Hard_disk" title="Hard disk">hard disk</a> systems. He was unhappy with the navigational model of the CODASYL approach, notably the lack of a "search" facility. In 1970, he wrote a number of papers that outlined a new approach to database construction that eventually culminated in the groundbreaking <i>A Relational Model of Data for Large Shared Data Banks</i>.<sup class="reference" id="cite_ref-FOOTNOTECodd1970_14-0"><a href="#cite_note-FOOTNOTECodd1970-14">[13]</a></sup></p>
<p>In this paper, he described a new system for storing and working with large databases. Instead of records being stored in some sort of <a href="/wiki/Linked_list" title="Linked list">linked list</a> of free-form records as in CODASYL, Codd's idea was to use a "<a href="/wiki/Table_(database)" title="Table (database)">table</a>" of fixed-length records, with each table used for a different type of entity. A linked-list system would be very inefficient when storing "sparse" databases where some of the data for any one record could be left empty. The relational model solved this by splitting the data into a series of normalized tables (or <i>relations</i>), with optional elements being moved out of the main table to where they would take up room only if needed. Data may be freely inserted, deleted and edited in these tables, with the DBMS doing whatever maintenance needed to present a table view to the application/user.</p>
<div class="thumb tright">
<div class="thumbinner" style="width:302px;"><a class="image" href="/wiki/File:Relational_key_SVG.svg"><img alt="" class="thumbimage" data-file-height="258" data-file-width="300" height="258" src="//upload.wikimedia.org/wikipedia/commons/thumb/4/4c/Relational_key_SVG.svg/300px-Relational_key_SVG.svg.png" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/4/4c/Relational_key_SVG.svg/450px-Relational_key_SVG.svg.png 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/4/4c/Relational_key_SVG.svg/600px-Relational_key_SVG.svg.png 2x" width="300"/></a>
<div class="thumbcaption">
<div class="magnify"><a class="internal" href="/wiki/File:Relational_key_SVG.svg" title="Enlarge"></a></div>
In the <a href="/wiki/Relational_model" title="Relational model">relational model</a>, records are "linked" using virtual keys not stored in the database but defined as needed between the data contained in the records.</div>
</div>
</div>
<p>The relational model also allowed the content of the database to evolve without constant rewriting of links and pointers. The relational part comes from entities referencing other entities in what is known as one-to-many relationship, like a traditional hierarchical model, and many-to-many relationship, like a navigational (network) model. Thus, a relational model can express both hierarchical and navigational models, as well as its native tabular model, allowing for pure or combined modeling in terms of these three models, as the application requires.</p>
<p>For instance, a common use of a database system is to track information about users, their name, login information, various addresses and phone numbers. In the navigational approach, all of this data would be placed in a single record, and unused items would simply not be placed in the database. In the relational approach, the data would be <i>normalized</i> into a user table, an address table and a phone number table (for instance). Records would be created in these optional tables only if the address or phone numbers were actually provided.</p>
<p>Linking the information back together is the key to this system. In the relational model, some bit of information was used as a "<a class="mw-redirect" href="/wiki/Primary_key" title="Primary key">key</a>", uniquely defining a particular record. When information was being collected about a user, information stored in the optional tables would be found by searching for this key. For instance, if the login name of a user is unique, addresses and phone numbers for that user would be recorded with the login name as its key. This simple "re-linking" of related data back into a single collection is something that traditional computer languages are not designed for.</p>
<p>Just as the navigational approach would require programs to loop in order to collect records, the relational approach would require loops to collect information about any <i>one</i> record. Codd's solution to the necessary looping was a set-oriented language, a suggestion that would later spawn the ubiquitous SQL. Using a branch of mathematics known as <a class="mw-redirect" href="/wiki/Tuple_calculus" title="Tuple calculus">tuple calculus</a>, he demonstrated that such a system could support all the operations of normal databases (inserting, updating etc.) as well as providing a simple system for finding and returning <i>sets</i> of data in a single operation.</p>
<p>Codd's paper was picked up by two people at Berkeley, Eugene Wong and <a href="/wiki/Michael_Stonebraker" title="Michael Stonebraker">Michael Stonebraker</a>. They started a project known as <a class="mw-redirect" href="/wiki/INGRES" title="INGRES">INGRES</a> using funding that had already been allocated for a geographical database project and student programmers to produce code. Beginning in 1973, INGRES delivered its first test products which were generally ready for widespread use in 1979. INGRES was similar to <a href="/wiki/IBM_System_R" title="IBM System R">System R</a> in a number of ways, including the use of a "language" for <a href="/wiki/Data_access" title="Data access">data access</a>, known as <a href="/wiki/QUEL_query_languages" title="QUEL query languages">QUEL</a>. Over time, INGRES moved to the emerging SQL standard.</p>
<p>IBM itself did one test implementation of the relational model, <a class="mw-redirect" href="/wiki/PRTV" title="PRTV">PRTV</a>, and a production one, <a class="mw-redirect" href="/wiki/Business_System_12" title="Business System 12">Business System 12</a>, both now discontinued. <a href="/wiki/Honeywell" title="Honeywell">Honeywell</a> wrote <a href="/wiki/Multics_Relational_Data_Store" title="Multics Relational Data Store">MRDS</a> for <a href="/wiki/Multics" title="Multics">Multics</a>, and now there are two new implementations: <a href="/wiki/Dataphor" title="Dataphor">Alphora Dataphor</a> and <a href="/wiki/Rel_(DBMS)" title="Rel (DBMS)">Rel</a>. Most other DBMS implementations usually called <i>relational</i> are actually SQL DBMSs.</p>
<p>In 1970, the University of Michigan began development of the <a href="/wiki/MICRO_Relational_Database_Management_System" title="MICRO Relational Database Management System">MICRO Information Management System</a><sup class="reference" id="cite_ref-FOOTNOTEHersheyEasthope1972_15-0"><a href="#cite_note-FOOTNOTEHersheyEasthope1972-15">[14]</a></sup> based on D.L. Childs' Set-Theoretic Data model. <sup class="reference" id="cite_ref-FOOTNOTENorth2010_16-0"><a href="#cite_note-FOOTNOTENorth2010-16">[15]</a></sup><sup class="reference" id="cite_ref-FOOTNOTEChilds1968a_17-0"><a href="#cite_note-FOOTNOTEChilds1968a-17">[16]</a></sup><sup class="reference" id="cite_ref-FOOTNOTEChilds1968b_18-0"><a href="#cite_note-FOOTNOTEChilds1968b-18">[17]</a></sup> MICRO was used to manage very large data sets by the <a class="mw-redirect" href="/wiki/US_Department_of_Labor" title="US Department of Labor">US Department of Labor</a>, the <a class="mw-redirect" href="/wiki/U.S._Environmental_Protection_Agency" title="U.S. Environmental Protection Agency">U.S. Environmental Protection Agency</a>, and researchers from the <a href="/wiki/University_of_Alberta" title="University of Alberta">University of Alberta</a>, the <a href="/wiki/University_of_Michigan" title="University of Michigan">University of Michigan</a>, and <a href="/wiki/Wayne_State_University" title="Wayne State University">Wayne State University</a>. It ran on IBM mainframe computers using the <a href="/wiki/Michigan_Terminal_System" title="Michigan Terminal System">Michigan Terminal System</a>.<sup class="reference" id="cite_ref-MICROManual1977_19-0"><a href="#cite_note-MICROManual1977-19">[18]</a></sup> The system remained in production until 1998.</p>
<h3><span class="mw-headline" id="Integrated_approach">Integrated approach</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Database&amp;action=edit&amp;section=7" title="Edit section: Integrated approach">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="hatnote" role="note">Main article: <a href="/wiki/Database_machine" title="Database machine">Database machine</a></div>
<p>In the 1970s and 1980s, attempts were made to build database systems with integrated hardware and software. The underlying philosophy was that such integration would provide higher performance at lower cost. Examples were IBM <a class="mw-redirect" href="/wiki/System/38" title="System/38">System/38</a>, the early offering of <a href="/wiki/Teradata" title="Teradata">Teradata</a>, and the <a href="/wiki/Britton_Lee,_Inc." title="Britton Lee, Inc.">Britton Lee, Inc.</a> database machine.</p>
<p>Another approach to hardware support for database management was <a href="/wiki/International_Computers_Limited" title="International Computers Limited">ICL</a>'s <a href="/wiki/Content_Addressable_File_Store" title="Content Addressable File Store">CAFS</a> accelerator, a hardware disk controller with programmable search capabilities. In the long term, these efforts were generally unsuccessful because specialized database machines could not keep pace with the rapid development and progress of general-purpose computers. Thus most database systems nowadays are software systems running on general-purpose hardware, using general-purpose computer data storage. However this idea is still pursued for certain applications by some companies like <a href="/wiki/Netezza" title="Netezza">Netezza</a> and Oracle (<a class="mw-redirect" href="/wiki/Exadata" title="Exadata">Exadata</a>).</p>
<h3><span class="mw-headline" id="Late_1970s.2C_SQL_DBMS">Late 1970s, SQL DBMS</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Database&amp;action=edit&amp;section=8" title="Edit section: Late 1970s, SQL DBMS">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>IBM started working on a prototype system loosely based on Codd's concepts as <i>System R</i> in the early 1970s. The first version was ready in 1974/5, and work then started on multi-table systems in which the data could be split so that all of the data for a record (some of which is optional) did not have to be stored in a single large "chunk". Subsequent multi-user versions were tested by customers in 1978 and 1979, by which time a standardized <a href="/wiki/Query_language" title="Query language">query language</a> – SQL<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="First version of SQL standard was SQL-86 adopted in 1986 (May 2012)">citation needed</span></a></i>]</sup> – had been added. Codd's ideas were establishing themselves as both workable and superior to CODASYL, pushing IBM to develop a true production version of System R, known as <i>SQL/DS</i>, and, later, <i>Database 2</i> (DB2).</p>
<p><a href="/wiki/Larry_Ellison" title="Larry Ellison">Larry Ellison</a>'s Oracle started from a different chain, based on IBM's papers on System R, and beat IBM to market when the first version was released in 1978.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (March 2013)">citation needed</span></a></i>]</sup></p>
<p>Stonebraker went on to apply the lessons from INGRES to develop a new database, Postgres, which is now known as PostgreSQL. PostgreSQL is often used for global mission critical applications (the .org and .info domain name registries use it as their primary <a href="/wiki/Data_store" title="Data store">data store</a>, as do many large companies and financial institutions).</p>
<p>In Sweden, Codd's paper was also read and <a href="/wiki/Mimer_SQL" title="Mimer SQL">Mimer SQL</a> was developed from the mid-1970s at <a href="/wiki/Uppsala_University" title="Uppsala University">Uppsala University</a>. In 1984, this project was consolidated into an independent enterprise. In the early 1980s, Mimer introduced transaction handling for high robustness in applications, an idea that was subsequently implemented on most other DBMSs.</p>
<p>Another data model, the <a href="/wiki/Entity%E2%80%93relationship_model" title="Entity–relationship model">entity–relationship model</a>, emerged in 1976 and gained popularity for <a href="/wiki/Database_design" title="Database design">database design</a> as it emphasized a more familiar description than the earlier relational model. Later on, entity–relationship constructs were retrofitted as a data modeling construct for the relational model, and the difference between the two have become irrelevant.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (March 2013)">citation needed</span></a></i>]</sup></p>
<h3><span class="mw-headline" id="1980s.2C_on_the_desktop">1980s, on the desktop</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Database&amp;action=edit&amp;section=9" title="Edit section: 1980s, on the desktop">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>The 1980s ushered in the age of <a class="mw-redirect" href="/wiki/Desktop_Computer" title="Desktop Computer">desktop computing</a>. The new computers empowered their users with spreadsheets like <a href="/wiki/Lotus_1-2-3" title="Lotus 1-2-3">Lotus 1-2-3</a> and database software like <a class="mw-redirect" href="/wiki/DBASE" title="DBASE">dBASE</a>. The dBASE product was lightweight and easy for any computer user to understand out of the box. <a class="mw-redirect" href="/wiki/C._Wayne_Ratliff" title="C. Wayne Ratliff">C. Wayne Ratliff</a> the creator of dBASE stated: "dBASE was different from programs like BASIC, C, FORTRAN, and COBOL in that a lot of the dirty work had already been done. The data manipulation is done by dBASE instead of by the user, so the user can concentrate on what he is doing, rather than having to mess with the dirty details of opening, reading, and closing files, and managing space allocation."<sup class="reference" id="cite_ref-20"><a href="#cite_note-20">[19]</a></sup> dBASE was one of the top selling software titles in the 1980s and early 1990s.</p>
<h3><span class="mw-headline" id="1990s.2C_object-oriented">1990s, object-oriented</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Database&amp;action=edit&amp;section=10" title="Edit section: 1990s, object-oriented">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>The 1990s, along with a rise in <a href="/wiki/Object-oriented_programming" title="Object-oriented programming">object-oriented programming</a>, saw a growth in how data in various databases were handled. Programmers and designers began to treat the data in their databases as objects. That is to say that if a person's data were in a database, that person's attributes, such as their address, phone number, and age, were now considered to belong to that person instead of being extraneous data. This allows for relations between data to be relations to objects and their attributes and not to individual fields.<sup class="reference" id="cite_ref-21"><a href="#cite_note-21">[20]</a></sup> The term "<a href="/wiki/Object-relational_impedance_mismatch" title="Object-relational impedance mismatch">object-relational impedance mismatch</a>" described the inconvenience of translating between programmed objects and database tables. <a href="/wiki/Object_database" title="Object database">Object databases</a> and <a href="/wiki/Object-relational_database" title="Object-relational database">object-relational databases</a> attempt to solve this problem by providing an object-oriented language (sometimes as extensions to SQL) that programmers can use as alternative to purely relational SQL. On the programming side, libraries known as <a href="/wiki/Object-relational_mapping" title="Object-relational mapping">object-relational mappings</a> (ORMs) attempt to solve the same problem.</p>
<h3><span class="mw-headline" id="2000s.2C_NoSQL_and_NewSQL">2000s, NoSQL and NewSQL</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Database&amp;action=edit&amp;section=11" title="Edit section: 2000s, NoSQL and NewSQL">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="hatnote" role="note">Main articles: <a href="/wiki/NoSQL" title="NoSQL">NoSQL</a> and <a href="/wiki/NewSQL" title="NewSQL">NewSQL</a></div>
<p><a class="mw-redirect" href="/wiki/XML_databases" title="XML databases">XML databases</a> are a type of structured document-oriented database that allows querying based on <a href="/wiki/XML" title="XML">XML</a> document attributes. XML databases are mostly used in <a class="mw-redirect" href="/wiki/Enterprise_database_management" title="Enterprise database management">enterprise database management</a>, where XML is being used as the machine-to-machine data interoperability standard. XML database management systems include <a href="/wiki/Commercial_software" title="Commercial software">commercial software</a> <a href="/wiki/MarkLogic" title="MarkLogic">MarkLogic</a> and <a href="/wiki/Oracle_Database" title="Oracle Database">Oracle Berkeley DB XML</a>, and a free use software <a href="/wiki/Clusterpoint" title="Clusterpoint">Clusterpoint Distributed XML/JSON Database</a>. All are <a href="/wiki/Enterprise_software" title="Enterprise software">enterprise software</a> database platforms and support industry standard <a href="/wiki/ACID" title="ACID">ACID</a>-compliant <a href="/wiki/Transaction_processing" title="Transaction processing">transaction processing</a> with strong database consistency characteristics and high level of database security.<sup class="reference" id="cite_ref-22"><a href="#cite_note-22">[21]</a></sup><sup class="reference" id="cite_ref-23"><a href="#cite_note-23">[22]</a></sup><sup class="reference" id="cite_ref-24"><a href="#cite_note-24">[23]</a></sup></p>
<p>NoSQL databases are often very fast, do not require fixed table schemas, avoid join operations by storing <a href="/wiki/Denormalization" title="Denormalization">denormalized</a> data, and are designed to <a class="mw-redirect" href="/wiki/Horizontal_scaling" title="Horizontal scaling">scale horizontally</a>. The most popular NoSQL systems include <a href="/wiki/MongoDB" title="MongoDB">MongoDB</a>, <a class="mw-redirect" href="/wiki/Couchbase" title="Couchbase">Couchbase</a>, <a href="/wiki/Riak" title="Riak">Riak</a>, <a href="/wiki/Memcached" title="Memcached">Memcached</a>, <a href="/wiki/Redis" title="Redis">Redis</a>, <a href="/wiki/CouchDB" title="CouchDB">CouchDB</a>, <a href="/wiki/Hazelcast" title="Hazelcast">Hazelcast</a>, <a href="/wiki/Apache_Cassandra" title="Apache Cassandra">Apache Cassandra</a>, and <a class="mw-redirect" href="/wiki/HBase" title="HBase">HBase</a>,<sup class="reference" id="cite_ref-25"><a href="#cite_note-25">[24]</a></sup> which are all <a href="/wiki/Open-source_software" title="Open-source software">open-source software</a> products.</p>
<p>In recent years, there was a high demand for massively distributed databases with high partition tolerance but according to the <a href="/wiki/CAP_theorem" title="CAP theorem">CAP theorem</a> it is impossible for a <a href="/wiki/Distributed_computing" title="Distributed computing">distributed system</a> to simultaneously provide <a href="/wiki/Consistency_model" title="Consistency model">consistency</a>, availability, and partition tolerance guarantees. A distributed system can satisfy any two of these guarantees at the same time, but not all three. For that reason, many NoSQL databases are using what is called <a href="/wiki/Eventual_consistency" title="Eventual consistency">eventual consistency</a> to provide both availability and partition tolerance guarantees with a reduced level of data consistency.</p>
<p>NewSQL is a class of modern relational databases that aims to provide the same scalable performance of NoSQL systems for online transaction processing (read-write) workloads while still using SQL and maintaining the ACID guarantees of a traditional database system. Such databases include <a href="/wiki/ScaleBase" title="ScaleBase">ScaleBase</a>, <a href="/wiki/Clustrix" title="Clustrix">Clustrix</a>, <a href="/wiki/EnterpriseDB" title="EnterpriseDB">EnterpriseDB</a>, <a href="/wiki/MemSQL" title="MemSQL">MemSQL</a>, <a href="/wiki/NuoDB" title="NuoDB">NuoDB</a>,<sup class="reference" id="cite_ref-FOOTNOTEProctor2013_26-0"><a href="#cite_note-FOOTNOTEProctor2013-26">[25]</a></sup> and <a href="/wiki/VoltDB" title="VoltDB">VoltDB</a>.</p>
<h2><span class="mw-headline" id="Research">Research</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Database&amp;action=edit&amp;section=12" title="Edit section: Research">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Database technology has been an active research topic since the 1960s, both in <a href="/wiki/Academia" title="Academia">academia</a> and in the research and development groups of companies (for example <a href="/wiki/IBM_Research" title="IBM Research">IBM Research</a>). Research activity includes <a href="/wiki/Database_theory" title="Database theory">theory</a> and development of <a href="/wiki/Prototype" title="Prototype">prototypes</a>. Notable research topics have included <a href="/wiki/Data_model" title="Data model">models</a>, the atomic transaction concept, and related <a href="/wiki/Concurrency_control" title="Concurrency control">concurrency control</a> techniques, query languages and <a href="/wiki/Query_optimization" title="Query optimization">query optimization</a> methods, <a href="/wiki/RAID" title="RAID">RAID</a>, and more.</p>
<p>The database research area has several dedicated <a href="/wiki/Academic_journal" title="Academic journal">academic journals</a> (for example, <i><a href="/wiki/ACM_Transactions_on_Database_Systems" title="ACM Transactions on Database Systems">ACM Transactions on Database Systems</a></i>-TODS, <i><a href="/wiki/Data_and_Knowledge_Engineering" title="Data and Knowledge Engineering">Data and Knowledge Engineering</a></i>-DKE) and annual <a href="/wiki/Academic_conference" title="Academic conference">conferences</a> (e.g., <a href="/wiki/Association_for_Computing_Machinery" title="Association for Computing Machinery">ACM</a> <a href="/wiki/SIGMOD" title="SIGMOD">SIGMOD</a>, ACM <a href="/wiki/Symposium_on_Principles_of_Database_Systems" title="Symposium on Principles of Database Systems">PODS</a>, <a href="/wiki/VLDB" title="VLDB">VLDB</a>, <a class="mw-redirect" href="/wiki/IEEE" title="IEEE">IEEE</a> ICDE).</p>
<h2><span class="mw-headline" id="Examples">Examples</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Database&amp;action=edit&amp;section=13" title="Edit section: Examples">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>One way to classify databases involves the type of their contents, for example: <a href="/wiki/Bibliographic_database" title="Bibliographic database">bibliographic</a>, document-text, statistical, or multimedia objects. Another way is by their application area, for example: accounting, music compositions, movies, banking, manufacturing, or insurance. A third way is by some technical aspect, such as the database structure or interface type. This section lists a few of the adjectives used to characterize different kinds of databases.</p>
<ul>
<li>An <a href="/wiki/In-memory_database" title="In-memory database">in-memory database</a> is a database that primarily resides in <a class="mw-redirect" href="/wiki/Main_memory" title="Main memory">main memory</a>, but is typically backed-up by non-volatile computer data storage. Main memory databases are faster than disk databases, and so are often used where response time is critical, such as in telecommunications network equipment.<sup class="reference" id="cite_ref-27"><a href="#cite_note-27">[26]</a></sup> <a href="/wiki/SAP_HANA" title="SAP HANA">SAP HANA</a> platform is a very hot topic for in-memory database. By May 2012, HANA was able to run on servers with 100TB main memory powered by IBM. The co founder of the company claimed that the system was big enough to run the 8 largest SAP customers.</li>
<li>An <a href="/wiki/Active_database" title="Active database">active database</a> includes an event-driven architecture which can respond to conditions both inside and outside the database. Possible uses include security monitoring, alerting, statistics gathering and authorization. Many databases provide active database features in the form of <a href="/wiki/Database_trigger" title="Database trigger">database triggers</a>.</li>
<li>A <a href="/wiki/Cloud_database" title="Cloud database">cloud database</a> relies on <a href="/wiki/Cloud_computing" title="Cloud computing">cloud technology</a>. Both the database and most of its DBMS reside remotely, "in the cloud", while its applications are both developed by programmers and later maintained and utilized by (application's) end-users through a <a href="/wiki/Web_browser" title="Web browser">web browser</a> and <a href="/wiki/Open_API" title="Open API">Open APIs</a>.</li>
<li><a href="/wiki/Data_warehouse" title="Data warehouse">Data warehouses</a> archive data from operational databases and often from external sources such as market research firms. The warehouse becomes the central source of data for use by managers and other end-users who may not have access to operational data. For example, sales data might be aggregated to weekly totals and converted from internal product codes to use <a href="/wiki/Universal_Product_Code" title="Universal Product Code">UPCs</a> so that they can be compared with <a class="mw-redirect" href="/wiki/ACNielsen" title="ACNielsen">ACNielsen</a> data. Some basic and essential components of data warehousing include extracting, analyzing, and <a href="/wiki/Data_mining" title="Data mining">mining</a> data, transforming, loading, and managing data so as to make them available for further use.</li>
<li>A <a href="/wiki/Deductive_database" title="Deductive database">deductive database</a> combines <a href="/wiki/Logic_programming" title="Logic programming">logic programming</a> with a relational database, for example by using the <a href="/wiki/Datalog" title="Datalog">Datalog</a> language.</li>
<li>A <a href="/wiki/Distributed_database" title="Distributed database">distributed database</a> is one in which both the data and the DBMS span multiple computers.</li>
<li>A document-oriented database is designed for storing, retrieving, and managing document-oriented, or semi structured data, information. Document-oriented databases are one of the main categories of NoSQL databases.</li>
<li>An <a href="/wiki/Embedded_database" title="Embedded database">embedded database</a> system is a DBMS which is tightly integrated with an application software that requires access to stored data in such a way that the DBMS is hidden from the application’s end-users and requires little or no ongoing maintenance.<sup class="reference" id="cite_ref-28"><a href="#cite_note-28">[27]</a></sup></li>
<li><b>End-user databases</b> consist of data developed by individual end-users. Examples of these are collections of documents, spreadsheets, presentations, multimedia, and other files. Several products exist to support such databases. Some of them are much simpler than full-fledged DBMSs, with more elementary DBMS functionality.</li>
<li>A <a href="/wiki/Federated_database_system" title="Federated database system">federated database system</a> comprises several distinct databases, each with its own DBMS. It is handled as a single database by a federated database management system (FDBMS), which transparently integrates multiple autonomous DBMSs, possibly of different types (in which case it would also be a <a href="/wiki/Heterogeneous_database_system" title="Heterogeneous database system">heterogeneous database system</a>), and provides them with an integrated conceptual view.</li>
<li>Sometimes the term <i>multi-database</i> is used as a synonym to federated database, though it may refer to a less integrated (e.g., without an FDBMS and a managed integrated schema) group of databases that cooperate in a single application. In this case, typically <a href="/wiki/Middleware_(distributed_applications)" title="Middleware (distributed applications)">middleware</a> is used for distribution, which typically includes an atomic commit protocol (ACP), e.g., the <a href="/wiki/Two-phase_commit_protocol" title="Two-phase commit protocol">two-phase commit protocol</a>, to allow <a href="/wiki/Distributed_transaction" title="Distributed transaction">distributed (global) transactions</a> across the participating databases.</li>
<li>A <a href="/wiki/Graph_database" title="Graph database">graph database</a> is a kind of NoSQL database that uses <a class="mw-redirect" href="/wiki/Graph_(data_structure)" title="Graph (data structure)">graph structures</a> with nodes, edges, and properties to represent and store information. General graph databases that can store any graph are distinct from specialized graph databases such as <a href="/wiki/Triplestore" title="Triplestore">triplestores</a> and <a class="mw-redirect" href="/wiki/Network_database_model" title="Network database model">network databases</a>.</li>
<li>An <a href="/wiki/Array_DBMS" title="Array DBMS">array DBMS</a> is a kind of NoSQL DBMS that allows to model, store, and retrieve (usually large) multi-dimensional <a href="/wiki/Array_data_structure" title="Array data structure">arrays</a> such as satellite images and climate simulation output.</li>
<li>In a <a href="/wiki/Hypertext" title="Hypertext">hypertext</a> or <a href="/wiki/Hypermedia" title="Hypermedia">hypermedia</a> database, any word or a piece of text representing an object, e.g., another piece of text, an article, a picture, or a film, can be <a href="/wiki/Hyperlink" title="Hyperlink">hyperlinked</a> to that object. Hypertext databases are particularly useful for organizing large amounts of disparate information. For example, they are useful for organizing <a href="/wiki/Online_encyclopedia" title="Online encyclopedia">online encyclopedias</a>, where users can conveniently jump around the text. The <a href="/wiki/World_Wide_Web" title="World Wide Web">World Wide Web</a> is thus a large distributed hypertext database.</li>
<li>A <a href="/wiki/Knowledge_base" title="Knowledge base">knowledge base</a> (abbreviated <b>KB</b>, <b>kb</b> or Δ<sup class="reference" id="cite_ref-29"><a href="#cite_note-29">[28]</a></sup><sup class="reference" id="cite_ref-30"><a href="#cite_note-30">[29]</a></sup>) is a special kind of database for <a href="/wiki/Knowledge_management" title="Knowledge management">knowledge management</a>, providing the means for the computerized collection, organization, and <a href="/wiki/Information_retrieval" title="Information retrieval">retrieval</a> of <a href="/wiki/Knowledge" title="Knowledge">knowledge</a>. Also a collection of data representing problems with their solutions and related experiences.</li>
</ul>
<ul>
<li>A <a href="/wiki/Mobile_database" title="Mobile database">mobile database</a> can be carried on or synchronized from a mobile computing device.</li>
<li><a href="/wiki/Operational_database" title="Operational database">Operational databases</a> store detailed data about the operations of an organization. They typically process relatively high volumes of updates using <a class="mw-redirect" href="/wiki/Transaction_(database)" title="Transaction (database)">transactions</a>. Examples include <a href="/wiki/Customer_relationship_management" title="Customer relationship management">customer databases</a> that record contact, credit, and demographic information about a business' customers, personnel databases that hold information such as salary, benefits, skills data about employees, <a href="/wiki/Enterprise_resource_planning" title="Enterprise resource planning">enterprise resource planning</a> systems that record details about product components, parts inventory, and financial databases that keep track of the organization's money, accounting and financial dealings.</li>
<li>A <a href="/wiki/Parallel_database" title="Parallel database">parallel database</a> seeks to improve performance through <a href="/wiki/Parallel_computing" title="Parallel computing">parallelization</a> for tasks such as loading data, building indexes and evaluating queries.</li>
</ul>
<dl>
<dd>
<dl>
<dd>The major parallel DBMS architectures which are induced by the underlying <a href="/wiki/Computer_hardware" title="Computer hardware">hardware</a> architecture are:
<ul>
<li><b><a class="mw-redirect" href="/wiki/Shared_memory_architecture" title="Shared memory architecture">Shared memory architecture</a></b>, where multiple processors share the main memory space, as well as other data storage.</li>
<li><b>Shared disk architecture</b>, where each processing unit (typically consisting of multiple processors) has its own main memory, but all units share the other storage.</li>
<li><b><a href="/wiki/Shared_nothing_architecture" title="Shared nothing architecture">Shared nothing architecture</a></b>, where each processing unit has its own main memory and other storage.</li>
</ul>
</dd>
</dl>
</dd>
</dl>
<ul>
<li><a href="/wiki/Probabilistic_database" title="Probabilistic database">Probabilistic databases</a> employ <a href="/wiki/Fuzzy_logic" title="Fuzzy logic">fuzzy logic</a> to draw inferences from imprecise data.</li>
<li><a href="/wiki/Real-time_database" title="Real-time database">Real-time databases</a> process transactions fast enough for the result to come back and be acted on right away.</li>
<li>A <a href="/wiki/Spatial_database" title="Spatial database">spatial database</a> can store the data with multidimensional features. The queries on such data include location-based queries, like "Where is the closest hotel in my area?".</li>
<li>A <a href="/wiki/Temporal_database" title="Temporal database">temporal database</a> has built-in time aspects, for example a temporal data model and a temporal version of SQL. More specifically the temporal aspects usually include valid-time and transaction-time.</li>
<li>A <a href="/wiki/Terminology-oriented_database" title="Terminology-oriented database">terminology-oriented database</a> builds upon an <a class="mw-redirect" href="/wiki/Object-oriented_database" title="Object-oriented database">object-oriented database</a>, often customized for a specific field.</li>
<li>An <a href="/wiki/Unstructured_data" title="Unstructured data">unstructured data</a> database is intended to store in a manageable and protected way diverse objects that do not fit naturally and conveniently in common databases. It may include email messages, documents, journals, multimedia objects, etc. The name may be misleading since some objects can be highly structured. However, the entire possible object collection does not fit into a predefined structured framework. Most established DBMSs now support unstructured data in various ways, and new dedicated DBMSs are emerging.</li>
</ul>
<h2><span class="mw-headline" id="Design_and_modeling">Design and modeling</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Database&amp;action=edit&amp;section=14" title="Edit section: Design and modeling">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<div class="hatnote" role="note">Main article: <a href="/wiki/Database_design" title="Database design">Database design</a></div>
<p>The first task of a database designer is to produce a <a class="mw-redirect" href="/wiki/Conceptual_data_model" title="Conceptual data model">conceptual data model</a> that reflects the structure of the information to be held in the database. A common approach to this is to develop an entity-relationship model, often with the aid of drawing tools. Another popular approach is the <a href="/wiki/Unified_Modeling_Language" title="Unified Modeling Language">Unified Modeling Language</a>. A successful data model will accurately reflect the possible state of the external world being modeled: for example, if people can have more than one phone number, it will allow this information to be captured. Designing a good conceptual data model requires a good understanding of the application domain; it typically involves asking deep questions about the things of interest to an organisation, like "can a customer also be a supplier?", or "if a product is sold with two different forms of packaging, are those the same product or different products?", or "if a plane flies from New York to Dubai via Frankfurt, is that one flight or two (or maybe even three)?". The answers to these questions establish definitions of the terminology used for entities (customers, products, flights, flight segments) and their relationships and attributes.</p>
<p>Producing the conceptual data model sometimes involves input from <a href="/wiki/Business_process_modeling" title="Business process modeling">business processes</a>, or the analysis of <a href="/wiki/Workflow" title="Workflow">workflow</a> in the organization. This can help to establish what information is needed in the database, and what can be left out. For example, it can help when deciding whether the database needs to hold historic data as well as current data.</p>
<p>Having produced a conceptual data model that users are happy with, the next stage is to translate this into a <a href="/wiki/Database_schema" title="Database schema">schema</a> that implements the relevant data structures within the database. This process is often called logical database design, and the output is a <a href="/wiki/Logical_data_model" title="Logical data model">logical data model</a> expressed in the form of a schema. Whereas the conceptual data model is (in theory at least) independent of the choice of database technology, the logical data model will be expressed in terms of a particular database model supported by the chosen DBMS. (The terms <i>data model</i> and <i>database model</i> are often used interchangeably, but in this article we use <i>data model</i> for the design of a specific database, and <i>database model</i> for the modelling notation used to express that design.)</p>
<p>The most popular database model for general-purpose databases is the relational model, or more precisely, the relational model as represented by the SQL language. The process of creating a logical database design using this model uses a methodical approach known as <a href="/wiki/Database_normalization" title="Database normalization">normalization</a>. The goal of normalization is to ensure that each elementary "fact" is only recorded in one place, so that insertions, updates, and deletions automatically maintain consistency.</p>
<p>The final stage of database design is to make the decisions that affect performance, scalability, recovery, security, and the like. This is often called <i>physical database design</i>. A key goal during this stage is <a href="/wiki/Data_independence" title="Data independence">data independence</a>, meaning that the decisions made for performance optimization purposes should be invisible to end-users and applications. Physical design is driven mainly by performance requirements, and requires a good knowledge of the expected workload and access patterns, and a deep understanding of the features offered by the chosen DBMS.</p>
<p>Another aspect of physical database design is security. It involves both defining <a href="/wiki/Access_control" title="Access control">access control</a> to database objects as well as defining security levels and methods for the data itself.</p>

</div>
</div>
<p>A database management system provides three views of the database data:</p>
<ul>
<li>The <b>external level</b> defines how each group of end-users sees the organization of data in the database. A single database can have any number of views at the external level.</li>
<li>The <b>conceptual level</b> unifies the various external views into a compatible global view.<sup class="reference" id="cite_ref-FOOTNOTEDate200331.E2.80.9332_32-0"><a href="#cite_note-FOOTNOTEDate200331.E2.80.9332-32">[31]</a></sup> It provides the synthesis of all the external views. It is out of the scope of the various database end-users, and is rather of interest to database application developers and database administrators.</li>
<li>The <b>internal level</b> (or <i>physical level</i>) is the internal organization of data inside a DBMS. It is concerned with cost, performance, scalability and other operational matters. It deals with storage layout of the data, using storage structures such as <a class="mw-redirect" href="/wiki/Index_(database)" title="Index (database)">indexes</a> to enhance performance. Occasionally it stores data of individual views (<a href="/wiki/Materialized_view" title="Materialized view">materialized views</a>), computed from generic data, if performance justification exists for such redundancy. It balances all the external views' performance requirements, possibly conflicting, in an attempt to optimize overall performance across all activities.</li>
</ul>
<p>While there is typically only one conceptual (or logical) and physical (or internal) view of the data, there can be any number of different external views. This allows users to see database information in a more business-related way rather than from a technical, processing viewpoint. For example, a financial department of a company needs the payment details of all employees as part of the company's expenses, but does not need details about employees that are the interest of the <a href="/wiki/Human_resources" title="Human resources">human resources</a> department. Thus different departments need different <i>views</i> of the company's database.</p>
<p>The three-level database architecture relates to the concept of <i>data independence</i> which was one of the major initial driving forces of the relational model. The idea is that changes made at a certain level do not affect the view at a higher level. For example, changes in the internal level do not affect application programs written using conceptual level interfaces, which reduces the impact of making physical changes to improve performance.</p>
<p>The conceptual view provides a level of indirection between internal and external. On one hand it provides a common view of the database, independent of different external view structures, and on the other hand it abstracts away details of how the data are stored or managed (internal level). In principle every level, and even every external view, can be presented by a different data model. In practice usually a given DBMS uses the same data model for both the external and the conceptual levels (e.g., relational model). The internal level, which is hidden inside the DBMS and depends on its implementation, requires a different level of detail and uses its own types of data structure types.</p>
<p>Separating the <i>external</i>, <i>conceptual</i> and <i>internal</i> levels was a major feature of the relational database model implementations that dominate 21st century databases.<sup class="reference" id="cite_ref-FOOTNOTEDate200331.E2.80.9332_32-1"><a href="#cite_note-FOOTNOTEDate200331.E2.80.9332-32">[31]</a></sup></p>
<h2><span class="mw-headline" id="Languages">Languages</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Database&amp;action=edit&amp;section=17" title="Edit section: Languages">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Database languages are special-purpose languages, which do one or more of the following:</p>
<ul>
<li><a href="/wiki/Data_definition_language" title="Data definition language">Data definition language</a> – defines data types and the relationships among them</li>
<li><a href="/wiki/Data_manipulation_language" title="Data manipulation language">Data manipulation language</a> – performs tasks such as inserting, updating, or deleting data occurrences</li>
<li><a href="/wiki/Query_language" title="Query language">Query language</a> – allows searching for information and computing derived information</li>
</ul>
<p>Database languages are specific to a particular data model. Notable examples include:</p>
<ul>
<li>SQL combines the roles of data definition, data manipulation, and query in a single language. It was one of the first commercial languages for the relational model, although it departs in some respects from <a href="/wiki/Codd%27s_12_rules" title="Codd's 12 rules">the relational model as described by Codd</a> (for example, the rows and columns of a table can be ordered). SQL became a standard of the <a href="/wiki/American_National_Standards_Institute" title="American National Standards Institute">American National Standards Institute</a> (ANSI) in 1986, and of the <a href="/wiki/International_Organization_for_Standardization" title="International Organization for Standardization">International Organization for Standardization</a> (ISO) in 1987. The standards have been regularly enhanced since and is supported (with varying degrees of conformance) by all mainstream commercial relational DBMSs.<sup class="reference" id="cite_ref-FOOTNOTEChapple2005_33-0"><a href="#cite_note-FOOTNOTEChapple2005-33">[32]</a></sup><sup class="reference" id="cite_ref-IBM-sql_34-0"><a href="#cite_note-IBM-sql-34">[33]</a></sup></li>
<li><a class="mw-redirect" href="/wiki/OQL" title="OQL">OQL</a> is an object model language standard (from the <a href="/wiki/Object_Data_Management_Group" title="Object Data Management Group">Object Data Management Group</a>). It has influenced the design of some of the newer query languages like <a class="new" href="/w/index.php?title=JDOQL&amp;action=edit&amp;redlink=1" title="JDOQL (page does not exist)">JDOQL</a> and <a href="/wiki/EJB_QL" title="EJB QL">EJB QL</a>.</li>
<li><a href="/wiki/XQuery" title="XQuery">XQuery</a> is a standard XML query language implemented by XML database systems such as <a href="/wiki/MarkLogic" title="MarkLogic">MarkLogic</a> and <a href="/wiki/EXist" title="EXist">eXist</a>, by relational databases with XML capability such as Oracle and DB2, and also by in-memory XML processors such as <a href="/wiki/Saxon_XSLT" title="Saxon XSLT">Saxon</a>.</li>
<li><a href="/wiki/SQL/XML" title="SQL/XML">SQL/XML</a> combines <a href="/wiki/XQuery" title="XQuery">XQuery</a> with SQL.<sup class="reference" id="cite_ref-FOOTNOTEWagner2010_35-0"><a href="#cite_note-FOOTNOTEWagner2010-35">[34]</a></sup></li>
</ul>
<p>A database language may also incorporate features like:</p>
<ul>
<li>DBMS-specific Configuration and storage engine management</li>
<li>Computations to modify query results, like counting, summing, averaging, sorting, grouping, and cross-referencing</li>
<li>Constraint enforcement (e.g. in an automotive database, only allowing one engine type per car)</li>
<li>Application programming interface version of the query language, for programmer convenience</li>
</ul>
<h2><span class="mw-headline" id="Performance.2C_security.2C_and_availability">Performance, security, and availability</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Database&amp;action=edit&amp;section=18" title="Edit section: Performance, security, and availability">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Because of the critical importance of database technology to the smooth running of an enterprise, database systems include complex mechanisms to deliver the required performance, security, and availability, and allow database administrators to control the use of these features.</p>
<h3><span class="mw-headline" id="Storage">Storage</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Database&amp;action=edit&amp;section=19" title="Edit section: Storage">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="hatnote" role="note">Main articles: <a href="/wiki/Computer_data_storage" title="Computer data storage">Computer data storage</a> and <a href="/wiki/Database_engine" title="Database engine">Database engine</a></div>
<p>Database storage is the container of the physical materialization of a database. It comprises the <i>internal</i> (physical) <i>level</i> in the database architecture. It also contains all the information needed (e.g., <a href="/wiki/Metadata" title="Metadata">metadata</a>, "data about the data", and internal <a href="/wiki/Data_structure" title="Data structure">data structures</a>) to reconstruct the <i>conceptual level</i> and <i>external level</i> from the internal level when needed. Putting data into permanent storage is generally the responsibility of the <a href="/wiki/Database_engine" title="Database engine">database engine</a> a.k.a. "storage engine". Though typically accessed by a DBMS through the underlying operating system (and often utilizing the operating systems' <a href="/wiki/File_system" title="File system">file systems</a> as intermediates for storage layout), storage properties and configuration setting are extremely important for the efficient operation of the DBMS, and thus are closely maintained by database administrators. A DBMS, while in operation, always has its database residing in several types of storage (e.g., memory and external storage). The database data and the additional needed information, possibly in very large amounts, are coded into bits. Data typically reside in the storage in structures that look completely different from the way the data look in the conceptual and external levels, but in ways that attempt to optimize (the best possible) these levels' reconstruction when needed by users and programs, as well as for computing additional types of needed information from the data (e.g., when querying the database).</p>
<p>Some DBMSs support specifying which <a href="/wiki/Character_encoding" title="Character encoding">character encoding</a> was used to store data, so multiple encodings can be used in the same database.</p>
<p>Various low-level <a href="/wiki/Database_storage_structures" title="Database storage structures">database storage structures</a> are used by the storage engine to serialize the data model so it can be written to the medium of choice. Techniques such as indexing may be used to improve performance. Conventional storage is row-oriented, but there are also <a href="/wiki/Column-oriented_DBMS" title="Column-oriented DBMS">column-oriented</a> and <a href="/wiki/Correlation_database" title="Correlation database">correlation databases</a>.</p>
<h4><span class="mw-headline" id="Materialized_views">Materialized views</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Database&amp;action=edit&amp;section=20" title="Edit section: Materialized views">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<div class="hatnote" role="note">Main article: <a href="/wiki/Materialized_view" title="Materialized view">Materialized view</a></div>
<p>Often storage redundancy is employed to increase performance. A common example is storing <i>materialized views</i>, which consist of frequently needed <i>external views</i> or query results. Storing such views saves the expensive computing of them each time they are needed. The downsides of materialized views are the overhead incurred when updating them to keep them synchronized with their original updated database data, and the cost of storage redundancy.</p>
<h4><span class="mw-headline" id="Replication">Replication</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Database&amp;action=edit&amp;section=21" title="Edit section: Replication">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<div class="hatnote" role="note">Main article: <a class="mw-redirect" href="/wiki/Database_replication" title="Database replication">Database replication</a></div>
<p>Occasionally a database employs storage redundancy by database objects replication (with one or more copies) to increase data availability (both to improve performance of simultaneous multiple end-user accesses to a same database object, and to provide resiliency in a case of partial failure of a distributed database). Updates of a replicated object need to be synchronized across the object copies. In many cases, the entire database is replicated.</p>

<div class="hatnote" role="note">Main article: <a href="/wiki/Database_security" title="Database security">Database security</a></div>
<p><a href="/wiki/Database_security" title="Database security">Database security</a> deals with all various aspects of protecting the database content, its owners, and its users. It ranges from protection from intentional unauthorized database uses to unintentional database accesses by unauthorized entities (e.g., a person or a computer program).</p>
<p>Database access control deals with controlling who (a person or a certain computer program) is allowed to access what information in the database. The information may comprise specific database objects (e.g., record types, specific records, data structures), certain computations over certain objects (e.g., query types, or specific queries), or utilizing specific access paths to the former (e.g., using specific indexes or other data structures to access information). Database access controls are set by special authorized (by the database owner) personnel that uses dedicated protected security DBMS interfaces.</p>
<p>This may be managed directly on an individual basis, or by the assignment of individuals and <a class="mw-redirect" href="/wiki/Privilege_(Computing)" title="Privilege (Computing)">privileges</a> to groups, or (in the most elaborate models) through the assignment of individuals and groups to roles which are then granted entitlements. Data security prevents unauthorized users from viewing or updating the database. Using passwords, users are allowed access to the entire database or subsets of it called "subschemas". For example, an employee database can contain all the data about an individual employee, but one group of users may be authorized to view only payroll data, while others are allowed access to only work history and medical data. If the DBMS provides a way to interactively enter and update the database, as well as interrogate it, this capability allows for managing personal databases.</p>
<p><a href="/wiki/Data_security" title="Data security">Data security</a> in general deals with protecting specific chunks of data, both physically (i.e., from corruption, or destruction, or removal; e.g., see <a href="/wiki/Physical_security" title="Physical security">physical security</a>), or the interpretation of them, or parts of them to meaningful information (e.g., by looking at the strings of bits that they comprise, concluding specific valid credit-card numbers; e.g., see <a class="mw-redirect" href="/wiki/Data_encryption" title="Data encryption">data encryption</a>).</p>
<p>Change and access logging records who accessed which attributes, what was changed, and when it was changed. Logging services allow for a forensic <a href="/wiki/Database_audit" title="Database audit">database audit</a> later by keeping a record of access occurrences and changes. Sometimes application-level code is used to record changes rather than leaving this to the database. Monitoring can be set up to attempt to detect security breaches.</p>
<h3><span class="mw-headline" id="Transactions_and_concurrency">Transactions and concurrency</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Database&amp;action=edit&amp;section=23" title="Edit section: Transactions and concurrency">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="hatnote" role="note">Further information: <a href="/wiki/Concurrency_control" title="Concurrency control">Concurrency control</a></div>
<p><a class="mw-redirect" href="/wiki/Database_transactions" title="Database transactions">Database transactions</a> can be used to introduce some level of <a href="/wiki/Fault_tolerance" title="Fault tolerance">fault tolerance</a> and <a href="/wiki/Data_integrity" title="Data integrity">data integrity</a> after recovery from a <a href="/wiki/Crash_(computing)" title="Crash (computing)">crash</a>. A database transaction is a unit of work, typically encapsulating a number of operations over a database (e.g., reading a database object, writing, acquiring <a href="/wiki/Lock_(database)" title="Lock (database)">lock</a>, etc.), an abstraction supported in database and also other systems. Each transaction has well defined boundaries in terms of which program/code executions are included in that transaction (determined by the transaction's programmer via special transaction commands).</p>
<p>The acronym <a href="/wiki/ACID" title="ACID">ACID</a> describes some ideal properties of a database transaction: <a href="/wiki/Atomicity_(database_systems)" title="Atomicity (database systems)">Atomicity</a>, <a href="/wiki/Consistency_(database_systems)" title="Consistency (database systems)">Consistency</a>, <a href="/wiki/Isolation_(database_systems)" title="Isolation (database systems)">Isolation</a>, and <a href="/wiki/Durability_(database_systems)" title="Durability (database systems)">Durability</a>.</p>
<h3><span class="mw-headline" id="Migration">Migration</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Database&amp;action=edit&amp;section=24" title="Edit section: Migration">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<dl>
<dd><i>See also section <a href="/wiki/Data_migration#Database_migration" title="Data migration">Database migration</a> in article <a href="/wiki/Data_migration" title="Data migration">Data migration</a></i></dd>
</dl>
<p>A database built with one DBMS is not portable to another DBMS (i.e., the other DBMS cannot run it). However, in some situations, it is desirable to move, migrate a database from one DBMS to another. The reasons are primarily economical (different DBMSs may have different <a href="/wiki/Total_cost_of_ownership" title="Total cost of ownership">total costs of ownership</a> or TCOs), functional, and operational (different DBMSs may have different capabilities). The migration involves the database's transformation from one DBMS type to another. The transformation should maintain (if possible) the database related application (i.e., all related application programs) intact. Thus, the database's conceptual and external architectural levels should be maintained in the transformation. It may be desired that also some aspects of the architecture internal level are maintained. A complex or large database migration may be a complicated and costly (one-time) project by itself, which should be factored into the decision to migrate. This in spite of the fact that tools may exist to help migration between specific DBMSs. Typically, a DBMS vendor provides tools to help importing databases from other popular DBMSs.</p>
<h3><span class="mw-headline" id="Building.2C_maintaining.2C_and_tuning">Building, maintaining, and tuning</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Database&amp;action=edit&amp;section=25" title="Edit section: Building, maintaining, and tuning">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="hatnote" role="note">Main article: <a href="/wiki/Database_tuning" title="Database tuning">Database tuning</a></div>
<p>After designing a database for an application, the next stage is building the database. Typically, an appropriate <a class="mw-redirect" href="/wiki/General-purpose_DBMS" title="General-purpose DBMS">general-purpose DBMS</a> can be selected to be utilized for this purpose. A DBMS provides the needed <a href="/wiki/User_interface" title="User interface">user interfaces</a> to be utilized by database administrators to define the needed application's data structures within the DBMS's respective data model. Other user interfaces are used to select needed DBMS parameters (like security related, storage allocation parameters, etc.).</p>
<p>When the database is ready (all its data structures and other needed components are defined), it is typically populated with initial application's data (database initialization, which is typically a distinct project; in many cases using specialized DBMS interfaces that support bulk insertion) before making it operational. In some cases, the database becomes operational while empty of application data, and data are accumulated during its operation.</p>
<p>After the database is created, initialised and populated it needs to be maintained. Various database parameters may need changing and the database may need to be tuned (<a href="/wiki/Database_tuning" title="Database tuning">tuning</a>) for better performance; application's data structures may be changed or added, new related application programs may be written to add to the application's functionality, etc.</p>
<h3><span class="mw-headline" id="Backup_and_restore">Backup and restore</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Database&amp;action=edit&amp;section=26" title="Edit section: Backup and restore">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="hatnote" role="note">Main article: <a href="/wiki/Backup" title="Backup">Backup</a></div>
<p>Sometimes it is desired to bring a database back to a previous state (for many reasons, e.g., cases when the database is found corrupted due to a software error, or if it has been updated with erroneous data). To achieve this, a <b>backup</b> operation is done occasionally or continuously, where each desired database state (i.e., the values of its data and their embedding in database's data structures) is kept within dedicated backup files (many techniques exist to do this effectively). When this state is needed, i.e., when it is decided by a database administrator to bring the database back to this state (e.g., by specifying this state by a desired point in time when the database was in this state), these files are utilized to <b>restore</b> that state.</p>
<h3><span class="mw-headline" id="Static_analysis">Static analysis</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Database&amp;action=edit&amp;section=27" title="Edit section: Static analysis">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Static analysis techniques for software verification can be applied also in the scenario of query languages. In particular, the *<a href="/wiki/Abstract_interpretation" title="Abstract interpretation">Abstract interpretation</a> framework has been extended to the field of query languages for relational databases as a way to support sound approximation techniques.<sup class="reference" id="cite_ref-FOOTNOTEHalderCortesi2012_36-0"><a href="#cite_note-FOOTNOTEHalderCortesi2012-36">[35]</a></sup> The semantics of query languages can be tuned according to suitable abstractions of the concrete domain of data. The abstraction of relational database system has many interesting applications, in particular, for security purposes, such as fine grained access control, watermarking, etc.</p>
<h3><span class="mw-headline" id="Other">Other</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Database&amp;action=edit&amp;section=28" title="Edit section: Other">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Other DBMS features might include:</p>
<ul>
<li><a class="mw-redirect" href="/wiki/Database_log" title="Database log">Database logs</a></li>
<li>Graphics component for producing graphs and charts, especially in a data warehouse system</li>
<li><b><a class="mw-redirect" href="/wiki/Query_optimizer" title="Query optimizer">Query optimizer</a></b> – Performs <a href="/wiki/Query_optimization" title="Query optimization">query optimization</a> on every query to choose for it the most efficient <i><a href="/wiki/Query_plan" title="Query plan">query plan</a></i> (a partial order (tree) of operations) to be executed to compute the query result. May be specific to a particular storage engine.</li>
<li>Tools or hooks for database design, application programming, application program maintenance, database performance analysis and monitoring, database configuration monitoring, DBMS hardware configuration (a DBMS and related database may span computers, networks, and storage units) and related database mapping (especially for a distributed DBMS), storage allocation and database layout monitoring, storage migration, etc.</li>
<li>Increasingly, there are calls for a single systems and methodology that incorporates all of these core functionalities into the same build, test, and deployment framework for database management and source control.<sup class="reference" id="cite_ref-37"><a href="#cite_note-37">[36]</a></sup>Borrowing from other developments in the software industry, some are labeling such offerings "<a href="/wiki/DBMaestro" title="DBMaestro">DevOps for Database</a>". Packaged thusly, these database management solutions are supposed to be stable, secure, backed up, compliant, testable, and consistent between environments.</li>
</ul>
