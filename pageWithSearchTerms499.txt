<h1 class="firstHeading" id="firstHeading" lang="en">Database design</h1>
<div class="mw-body-content" id="bodyContent">
<div id="siteSub">From Wikipedia, the free encyclopedia</div>
<div id="contentSub"></div>
<div class="mw-jump" id="jump-to-nav">
					Jump to:					<a href="#mw-head">navigation</a>, 					<a href="#p-search">search</a>
</div>
<div class="mw-content-ltr" dir="ltr" id="mw-content-text" lang="en"><p><b>Database design</b> is the process of producing a detailed <a href="/wiki/Data_model" title="Data model">data model</a> of <a href="/wiki/Database" title="Database">database</a>. This <a href="/wiki/Data_model" title="Data model">data model</a> contains all the needed logical and physical design choices and physical storage parameters needed to generate a design in a <a href="/wiki/Data_definition_language" title="Data definition language">data definition language</a>, which can then be used to create a database. A fully attributed data model contains detailed attributes for each entity.</p>
<p>The term database design can be used to describe many different parts of the design of an overall <a class="mw-redirect" href="/wiki/Database_system" title="Database system">database system</a>. Principally, and most correctly, it can be thought of as the logical design of the base data structures used to store the data. In the <a href="/wiki/Relational_model" title="Relational model">relational model</a> these are the <a class="mw-redirect" href="/wiki/Database_table" title="Database table">tables</a> and <a class="mw-redirect" href="/wiki/Database_view" title="Database view">views</a>. In an <a href="/wiki/Object_database" title="Object database">object database</a> the entities and relationships map directly to object classes and named relationships. However, the term database design could also be used to apply to the overall process of designing, not just the base data structures, but also the forms and queries used as part of the overall database application within the <a class="mw-redirect" href="/wiki/Database_management_system" title="Database management system">database management system</a> (DBMS).<sup class="reference" id="cite_ref-1"><a href="#cite_note-1">[1]</a></sup></p>
<p>The process of doing database design generally consists of a number of steps which will be carried out by the database designer. Usually, the designer must:</p>
<ul>
<li>Determine the data to be stored in the database.</li>
<li>Determine the relationships between the different data elements.</li>
<li>Superimpose a logical structure upon the data on the basis of these relationships.<sup class="reference" id="cite_ref-Teorey.2C_T.J._2009_2-0"><a href="#cite_note-Teorey.2C_T.J._2009-2">[2]</a></sup></li>
</ul>
<p>Within the <a href="/wiki/Relational_model" title="Relational model">relational model</a> the final step above can generally be broken down into two further steps, that of determining the grouping of information within the system, generally determining what are the basic objects about which information is being stored, and then determining the relationships between these groups of information, or objects. This step is not necessary with an <a href="/wiki/Object_database" title="Object database">Object database</a>.<sup class="reference" id="cite_ref-Teorey.2C_T.J._2009_2-1"><a href="#cite_note-Teorey.2C_T.J._2009-2">[2]</a></sup></p>
<p></p>

<h2><span class="mw-headline" id="Determining_data_to_be_stored">Determining data to be stored</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Database_design&amp;action=edit&amp;section=1" title="Edit section: Determining data to be stored">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>In a majority of cases, a person who is doing the design of a database is a person with expertise in the area of database design, rather than expertise in the domain from which the data to be stored is drawn e.g. financial information, biological information etc. Therefore, the data to be stored in the database must be determined in cooperation with a person who does have expertise in that domain, and who is aware of what data must be stored within the system.</p>
<p>This process is one which is generally considered part of <a href="/wiki/Requirements_analysis" title="Requirements analysis">requirements analysis</a>, and requires skill on the part of the database designer to elicit the needed information from those with the domain knowledge. This is because those with the necessary domain knowledge frequently cannot express clearly what their system requirements for the database are as they are unaccustomed to thinking in terms of the discrete data elements which must be stored. Data to be stored can be determined by Requirement Specification.<sup class="reference" id="cite_ref-3"><a href="#cite_note-3">[3]</a></sup></p>
<h2><span class="mw-headline" id="Determining_data_relationships">Determining data relationships</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Database_design&amp;action=edit&amp;section=2" title="Edit section: Determining data relationships">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Once a database designer is aware of the data which is to be stored within the database, they must then determine where dependency is within the data. Sometimes when data is changed you can be changing other data that is not visible. For example, in a list of names and addresses, assuming a situation where multiple people can have the same address, but one person cannot have more than one address, the address is dependent upon the name. When provided a name and the list the address can be uniquely determined; however, the inverse does not hold - when given an address and the list, a name cannot be uniquely determined because multiple people can reside at an address. Because an address is determined by a name, an address is considered dependent on a name.</p>
<p>(NOTE: A common misconception is that the <a href="/wiki/Relational_model" title="Relational model">relational model</a> is so called because of the stating of relationships between data elements therein. This is not true. The relational model is so named because it is based upon the mathematical structures known as <a class="mw-redirect" href="/wiki/Relation_(mathematics)" title="Relation (mathematics)">relations</a>.)</p>
<h2><span class="mw-headline" id="Logically_structuring_data">Logically structuring data</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Database_design&amp;action=edit&amp;section=3" title="Edit section: Logically structuring data">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<div class="hatnote" role="note">Main article: <a class="mw-redirect" href="/wiki/Logical_schema" title="Logical schema">Logical schema</a></div>
<p>Once the relationships and dependencies amongst the various pieces of information have been determined, it is possible to arrange the data into a logical structure which can then be mapped into the storage objects supported by the <a class="mw-redirect" href="/wiki/Database_management_system" title="Database management system">database management system</a>. In the case of <a class="mw-redirect" href="/wiki/Relational_databases" title="Relational databases">relational databases</a> the storage objects are <a class="mw-redirect" href="/wiki/Database_table" title="Database table">tables</a> which store data in rows and columns. In an <a href="/wiki/Object_database" title="Object database">Object database</a> the storage objects correspond directly to the objects used by the <a class="mw-redirect" href="/wiki/Object-oriented_programming_language" title="Object-oriented programming language">Object-oriented programming language</a> used to write the applications that will manage and access the data. The relationships may be defined as attributes of the object classes involved or as methods that operate on the object classes.</p>
<p>The way this mapping is generally performed is such that each set of related data which depends upon a single object, whether real or abstract, is placed in a table. Relationships between these dependent objects is then stored as links between the various objects.</p>
<p>Each table may represent an implementation of either a logical object or a relationship joining one or more instances of one or more logical objects. Relationships between tables may then be stored as links connecting child tables with parents. Since complex logical relationships are themselves tables they will probably have links to more than one parent.</p>

<p>Database designs also include ER (<a class="mw-redirect" href="/wiki/Entity-relationship_model" title="Entity-relationship model">entity-relationship model</a>) diagrams. An ER diagram is a diagram that helps to design databases in an efficient way.</p>
<p>Attributes in ER diagrams are usually modeled as an oval with the name of the attribute, linked to the entity or relationship that contains the attribute.</p>
<h2><span class="mw-headline" id="A_design_process_suggestion_for_Microsoft_Access.5B4.5D">A design process suggestion for Microsoft Access<sup class="reference" id="cite_ref-4"><a href="#cite_note-4">[4]</a></sup></span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Database_design&amp;action=edit&amp;section=5" title="Edit section: A design process suggestion for Microsoft Access[4]">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<ol>
<li><b>Determine the purpose of the database</b> - This helps prepare for the remaining steps.</li>
<li><b>Find and organize the information required</b> - Gather all of the types of information to record in the database, such as product name and order number.</li>
<li><b>Divide the information into tables</b> - Divide information items into major entities or subjects, such as Products or Orders. Each subject then becomes a table.</li>
<li><b>Turn information items into columns</b> - Decide what information needs to be stored in each table. Each item becomes a field, and is displayed as a column in the table. For example, an Employees table might include fields such as Last Name and Hire Date.</li>
<li><b>Specify primary keys</b> - Choose each table’s primary key. The primary key is a column, or a set of columns, that is used to uniquely identify each row. An example might be Product ID or Order ID.</li>
<li><b>Set up the table relationships</b> - Look at each table and decide how the data in one table is related to the data in other tables. Add fields to tables or create new tables to clarify the relationships, as necessary.</li>
<li><b>Refine the design</b> - Analyze the design for errors. Create tables and add a few records of sample data. Check if results come from the tables as expected. Make adjustments to the design, as needed.</li>
<li><b>Apply the <a href="/wiki/Database_normalization" title="Database normalization">normalization rules</a></b> - Apply the data normalization rules to see if tables are structured correctly. Make adjustments to the tables</li>
</ol>
<h2><span class="mw-headline" id="Normalization">Normalization</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Database_design&amp;action=edit&amp;section=6" title="Edit section: Normalization">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<div class="hatnote" role="note">Main article: <a href="/wiki/Database_normalization" title="Database normalization">Database normalization</a></div>
<p>In the field of <a href="/wiki/Relational_database" title="Relational database">relational database</a> design, <i>normalization</i> is a systematic way of ensuring that a database structure is suitable for general-purpose querying and free of certain undesirable characteristics—insertion, update, and deletion anomalies that could lead to loss of <a href="/wiki/Data_integrity" title="Data integrity">data integrity</a>.</p>
<p>A standard piece of database design guidance is that the designer should create a fully normalized design; selective <a href="/wiki/Denormalization" title="Denormalization">denormalization</a> can subsequently be performed, but only for <a href="/wiki/Computer_performance" title="Computer performance">performance</a> reasons. However, some modeling disciplines, such as the <a href="/wiki/Dimensional_modeling" title="Dimensional modeling">dimensional modeling</a> approach to <a href="/wiki/Data_warehouse" title="Data warehouse">data warehouse</a> design, explicitly recommend non-normalized designs, i.e. designs that in large part do not adhere to 3NF. Normalization consists of normal forms that are 1NF,2NF,3NF,BOYCE-CODD NF (3.5NF),4NF and 5NF</p>

<div class="hatnote" role="note">Main article: <a href="/wiki/Conceptual_schema" title="Conceptual schema">Conceptual schema</a></div>
<h3><span class="mw-headline" id="Schema_refinement">Schema refinement</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Database_design&amp;action=edit&amp;section=8" title="Edit section: Schema refinement">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Schema refinement of the database specifies that the data is normalized to reduce data insufficiency and conflicts.</p>
<h3><span class="mw-headline" id="Physical_design">Physical design</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Database_design&amp;action=edit&amp;section=9" title="Edit section: Physical design">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="hatnote" role="note">Main article: <a class="mw-redirect" href="/wiki/Physical_schema" title="Physical schema">Physical schema</a></div>
<p>The physical design of the database specifies the physical configuration of the database on the storage media. This includes detailed specification of <a href="/wiki/Data_element" title="Data element">data elements</a>, <a href="/wiki/Data_type" title="Data type">data types</a>, <a class="mw-redirect" href="/wiki/Index_(database)" title="Index (database)">indexing</a> options and other parameters residing in the DBMS <a href="/wiki/Data_dictionary" title="Data dictionary">data dictionary</a>. It is the detailed design of a system that includes modules &amp; the database's hardware &amp; software specifications of the system.</p>
