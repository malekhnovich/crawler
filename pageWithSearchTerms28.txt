<h1 class="firstHeading" id="firstHeading" lang="en">Creational pattern</h1>
<div class="mw-body-content" id="bodyContent">
<div id="siteSub">From Wikipedia, the free encyclopedia</div>
<div id="contentSub"></div>
<div class="mw-jump" id="jump-to-nav">
					Jump to:					<a href="#mw-head">navigation</a>, 					<a href="#p-search">search</a>
</div>
<div class="mw-content-ltr" dir="ltr" id="mw-content-text" lang="en"><p>In <a href="/wiki/Software_engineering" title="Software engineering">software engineering</a>, <b>creational design patterns</b> are <a class="mw-redirect" href="/wiki/Design_pattern_(computer_science)" title="Design pattern (computer science)">design patterns</a> that deal with <a href="/wiki/Object_lifetime" title="Object lifetime">object creation</a> mechanisms, trying to create objects in a manner suitable to the situation. The basic form of object creation could result in design problems or in added complexity to the design. Creational design patterns solve this problem by somehow controlling this object creation.</p>
<p>Creational design patterns are composed of two dominant ideas. One is encapsulating knowledge about which concrete classes the system uses. Another is hiding how instances of these concrete classes are created and combined.<sup class="reference" id="cite_ref-1"><a href="#cite_note-1">[1]</a></sup></p>
<p>Creational design patterns are further categorized into Object-creational patterns and Class-creational patterns, where Object-creational patterns deal with Object creation and Class-creational patterns deal with Class-instantiation. In greater details, Object-creational patterns defer part of its object creation to another object, while Class-creational patterns defer its object creation to subclasses.<sup class="reference" id="cite_ref-2"><a href="#cite_note-2">[2]</a></sup></p>
<p>Five well-known design patterns that are parts of creational patterns are the</p>
<ul>
<li><a href="/wiki/Abstract_factory_pattern" title="Abstract factory pattern">Abstract factory pattern</a>, which provides an interface for creating related or dependent objects without specifying the objects' concrete classes.<sup class="reference" id="cite_ref-3"><a href="#cite_note-3">[3]</a></sup></li>
<li><a href="/wiki/Builder_pattern" title="Builder pattern">Builder pattern</a>, which separates the construction of a complex object from its representation so that the same construction process can create different representations.</li>
<li><a href="/wiki/Factory_method_pattern" title="Factory method pattern">Factory method pattern</a>, which allows a class to defer instantiation to subclasses.<sup class="reference" id="cite_ref-4"><a href="#cite_note-4">[4]</a></sup></li>
<li><a href="/wiki/Prototype_pattern" title="Prototype pattern">Prototype pattern</a>, which specifies the kind of object to create using a prototypical instance, and creates new objects by cloning this prototype.</li>
<li><a href="/wiki/Singleton_pattern" title="Singleton pattern">Singleton pattern</a>, which ensures that a class only has one instance, and provides a global point of access to it.<sup class="reference" id="cite_ref-5"><a href="#cite_note-5">[5]</a></sup></li>
</ul>
<p></p>

<h2><span class="mw-headline" id="Definition">Definition</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Creational_pattern&amp;action=edit&amp;section=1" title="Edit section: Definition">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The creational patterns aim to separate a system from how its objects are created, composed, and represented. They increase the system's flexibility in terms of the what, who, how, and when of object creation.<sup class="reference" id="cite_ref-CS3DP_6-0"><a href="#cite_note-CS3DP-6">[6]</a></sup></p>
<h2><span class="mw-headline" id="Usage">Usage</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Creational_pattern&amp;action=edit&amp;section=2" title="Edit section: Usage">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>As modern software engineering depends more on object composition than class inheritance, emphasis shifts away from hard-coding behaviors toward defining a smaller set of basic behaviors that can be composed into more complex ones.<sup class="reference" id="cite_ref-7"><a href="#cite_note-7">[7]</a></sup> Hard-coding behaviors are inflexible because they require overriding or re-implementing the whole thing in order to change parts of the design. Additionally, hard-coding does not promote reuse and is hard to keep track of errors. For these reasons, creational patterns are more useful than hard-coding behaviors. Creational patterns make design become more flexible. They provide different ways to remove explicit references in the concrete classes from the code that needs to instantiate them.<sup class="reference" id="cite_ref-8"><a href="#cite_note-8">[8]</a></sup> In other words, they create independency for objects and classes.</p>
<p>Consider applying creational patterns when:</p>
<ul>
<li>A system should be independent of how its objects and products are created.</li>
<li>A set of related objects is designed to be used together.</li>
<li>Hiding the implementations of a class library or product, revealing only their interfaces.</li>
<li>Constructing different representation of independent complex objects.</li>
<li>A class wants its subclass to implement the object it creates.</li>
<li>The class instantiations are specified at run-time.</li>
<li>There must be a single instance and client can access this instance at all times.</li>
<li>Instance should be extensible without being modified.</li>
</ul>

<p><b>Participants</b>:</p>
<ul>
<li><b>Creator</b>: Declares object interface. Returns object.</li>
<li><b>ConcreteCreator</b>: Implements object's interface.</li>
</ul>
<h2><span class="mw-headline" id="Examples">Examples</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Creational_pattern&amp;action=edit&amp;section=4" title="Edit section: Examples">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Some examples of creational design patterns include:</p>
<ul>
<li><a href="/wiki/Abstract_factory_pattern" title="Abstract factory pattern">Abstract factory pattern</a>: centralize decision of what <a href="/wiki/Factory_method_pattern" title="Factory method pattern">factory</a> to instantiate</li>
<li><a href="/wiki/Factory_method_pattern" title="Factory method pattern">Factory method pattern</a>: centralize creation of an object of a specific type choosing one of several implementations</li>
<li><a href="/wiki/Builder_pattern" title="Builder pattern">Builder pattern</a>: separate the construction of a complex object from its representation so that the same construction process can create different representations</li>
<li><a class="mw-redirect" href="/wiki/Lazy_initialization_pattern" title="Lazy initialization pattern">Lazy initialization pattern</a>: tactic of delaying the creation of an object, the calculation of a value, or some other expensive process until the first time it is needed</li>
<li><a href="/wiki/Object_pool_pattern" title="Object pool pattern">Object pool pattern</a>: avoid expensive acquisition and release of resources by recycling objects that are no longer in use</li>
<li><a href="/wiki/Prototype_pattern" title="Prototype pattern">Prototype pattern</a>: used when the type of objects to create is determined by a prototypical instance, which is cloned to produce new objects</li>
<li><a href="/wiki/Singleton_pattern" title="Singleton pattern">Singleton pattern</a>: restrict instantiation of a class to one object</li>
</ul>
