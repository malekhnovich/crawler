<h1 class="firstHeading" id="firstHeading" lang="en">Dependency injection</h1>
<div class="mw-body-content" id="bodyContent">
<div id="siteSub">From Wikipedia, the free encyclopedia</div>
<div id="contentSub"></div>
<div class="mw-jump" id="jump-to-nav">
					Jump to:					<a href="#mw-head">navigation</a>, 					<a href="#p-search">search</a>
</div>
<div class="mw-content-ltr" dir="ltr" id="mw-content-text" lang="en"><p>In <a href="/wiki/Software_engineering" title="Software engineering">software engineering</a>, <b>dependency injection</b> is a <a href="/wiki/Software_design_pattern" title="Software design pattern">software design pattern</a> that implements <a href="/wiki/Inversion_of_control" title="Inversion of control">inversion of control</a> for resolving dependencies. A dependency is an <a href="/wiki/Object_(computer_science)" title="Object (computer science)">object</a> that can be used (a <a href="/wiki/Service_(systems_architecture)" title="Service (systems architecture)">service</a>). An injection is the passing of a dependency to a dependent object (a <a href="/wiki/Client_(computing)" title="Client (computing)">client</a>) that would use it. The service is made part of the client's <a href="/wiki/State_(computer_science)" title="State (computer science)">state</a>.<sup class="reference" id="cite_ref-JamesShore_1-0"><a href="#cite_note-JamesShore-1">[1]</a></sup> Passing the service to the client, rather than allowing a client to build or <a href="/wiki/Service_locator_pattern" title="Service locator pattern">find the service</a>, is the fundamental requirement of the pattern.</p>
<p>Dependency injection allows a program design to follow the <a href="/wiki/Dependency_inversion_principle" title="Dependency inversion principle">dependency inversion principle</a>. The client delegates the responsibility of providing its dependencies to external code (the injector) . The client is not allowed to call the injector code.<sup class="reference" id="cite_ref-HollywoodPrinciple.c2_2-0"><a href="#cite_note-HollywoodPrinciple.c2-2">[2]</a></sup> It is the injecting code that constructs the services and calls the client to inject them. This means the client code does not need to know about the injecting code. The client does not need to know how to construct the services. The client does not need to know which actual services it is using. The client only needs to know about the intrinsic interfaces of the services because these define how the client may use the services. This separates the responsibilities of use and construction.</p>
<p>There are three common means for a client to accept a dependency injection: <a href="/wiki/Mutator_method" title="Mutator method">setter</a>-, <a class="mw-redirect" href="/wiki/Interface_(object-oriented_programming)" title="Interface (object-oriented programming)">interface</a>- and <a href="/wiki/Constructor_(object-oriented_programming)" title="Constructor (object-oriented programming)">constructor</a>-based injection. Setter and constructor injection differ mainly by when they can be used. Interface injection differs in that the dependency is given a chance to control its own injection. All require that separate construction code (the injector) take responsibility for introducing a client and its dependencies to each other.<sup class="reference" id="cite_ref-3"><a href="#cite_note-3">[3]</a></sup></p>
<p></p>

<h2><span class="mw-headline" id="Overview">Overview</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Dependency_injection&amp;action=edit&amp;section=1" title="Edit section: Overview">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<div class="quotebox" style="float: right; clear: right; margin: 0.5em 0 0.8em 1.4em; width:33%; padding: 6px; border: 1px solid #aaa; font-size: 88%; background-color: #F9F9F9;">
<div class="" style="position: relative; text-align: left;">
<div>
<p><b>Dependency injection for five-year-olds</b></p>
<p>When you go and get things out of the refrigerator for yourself, you can cause problems. You might leave the door open, you might get something Mommy or Daddy doesn't want you to have. You might even be looking for something we don't even have or which has expired.</p>
<p>What you should be doing is stating a need, "I need something to drink with lunch," and then we will make sure you have something when you sit down to eat.</p>
</div>
</div>
<cite style="display: block; text-align: left;">John Munsch, 28 October 2009.<sup class="reference" id="cite_ref-MarkSeeman2011P4_4-0"><a href="#cite_note-MarkSeeman2011P4-4">[4]</a></sup><sup class="reference" id="cite_ref-5"><a href="#cite_note-5">[5]</a></sup><sup class="reference" id="cite_ref-6"><a href="#cite_note-6">[6]</a></sup></cite></div>
<p>Dependency injection separates the creation of a client's dependencies from the client's behavior, which allows program designs to be <a href="/wiki/Loose_coupling" title="Loose coupling">loosely</a> <a href="/wiki/Coupling_(computer_programming)" title="Coupling (computer programming)">coupled</a><sup class="reference" id="cite_ref-7"><a href="#cite_note-7">[7]</a></sup> and to follow the <a href="/wiki/Dependency_inversion_principle" title="Dependency inversion principle">dependency inversion</a> and <a href="/wiki/Single_responsibility_principle" title="Single responsibility principle">single responsibility principles</a>.<sup class="reference" id="cite_ref-MarkSeeman2011P4_4-1"><a href="#cite_note-MarkSeeman2011P4-4">[4]</a></sup><sup class="reference" id="cite_ref-8"><a href="#cite_note-8">[8]</a></sup> It directly contrasts with the <a href="/wiki/Service_locator_pattern" title="Service locator pattern">service locator pattern</a>, which allows clients to know about the system they use to find dependencies.</p>
<p>An injection, the basic unit of dependency injection, is not a new or a custom mechanism. It works in the same way that "<a href="/wiki/Parameter_(computer_programming)" title="Parameter (computer programming)">parameter passing</a>" works.<sup class="reference" id="cite_ref-9"><a href="#cite_note-9">[9]</a></sup> Referring to "parameter passing" as an injection carries the added implication that it's being done to isolate the client from details.</p>
<p>An injection is also about what is in control of the passing (never the client) and is independent of how the passing is accomplished, whether by passing a reference or a value.</p>
<p>Dependency injection involves four roles:</p>
<ul>
<li>the <b>service</b> object(s) to be used</li>
<li>the <b>client</b> object that is depending on the services it uses</li>
<li>the <a class="mw-redirect" href="/wiki/Interface_(object-oriented_programming)" title="Interface (object-oriented programming)"><b>interfaces</b></a> that define how the client may use the services</li>
<li>the <b>injector</b>, which is responsible for constructing the services and injecting them into the client</li>
</ul>
<p>Any object that may be used can be considered a <b>service</b>. Any object that uses other objects can be considered a <b>client</b>. The names have nothing to do with what the objects are for and everything to do with the role the objects play in any one injection.</p>
<p>The <b>interfaces</b> are the types the client expects its dependencies to be. At issue is what they make accessible. They may truly be interface types implemented by the services but also may be abstract classes or even the <a href="/wiki/Class_(computer_programming)#Abstract_and_Concrete" title="Class (computer programming)">concrete</a> services themselves, though this last would violate <a href="/wiki/Dependency_inversion_principle" title="Dependency inversion principle">DIP</a><sup class="reference" id="cite_ref-10"><a href="#cite_note-10">[10]</a></sup> and sacrifice the dynamic decoupling that enables testing. It's only required that the client does not know which they are and therefore never treats them as concrete, say by constructing or extending them.</p>
<p>Given that the client has no concrete knowledge, so long as the what the client uses, the interfaces name and <a href="/wiki/Application_programming_interface" title="Application programming interface">API</a>, then the client won't need to change even if what is behind the interface changes. However, if the interface is <a href="/wiki/Code_refactoring" title="Code refactoring">refactored</a> from being a class to an interface type (or vice versa) the client will need to be recompiled.<sup class="reference" id="cite_ref-11"><a href="#cite_note-11">[11]</a></sup> This is significant if the client and services are published separately. This unfortunate coupling is one that dependency injection cannot resolve.</p>
<p>The <b>injector</b> introduces the services into the client. Often, it also constructs the client. An injector may connect together a very complex object graph by treating an object like a client and later as a service for another client. The injector may actually be many objects working together but may not be the client. The injector may be referred to by other names such as: assembler, provider, container, factory, builder, spring, construction code, or main.</p>
<p>Dependency injection can be applied as a discipline, one that asks that all objects separate construction and behavior. Relying on a DI framework to perform construction can lead to forbidding the use of the <b>new</b> keyword, or, less strictly, only allow direct construction of <a href="/wiki/Value_object" title="Value object">value objects</a>.<sup class="reference" id="cite_ref-12"><a href="#cite_note-12">[12]</a></sup><sup class="reference" id="cite_ref-13"><a href="#cite_note-13">[13]</a></sup><sup class="reference" id="cite_ref-14"><a href="#cite_note-14">[14]</a></sup><sup class="reference" id="cite_ref-15"><a href="#cite_note-15">[15]</a></sup></p>
<h3><span class="mw-headline" id="Taxonomy">Taxonomy</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Dependency_injection&amp;action=edit&amp;section=2" title="Edit section: Taxonomy">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p><a href="/wiki/Inversion_of_control" title="Inversion of control">Inversion of control</a> (IoC) is more general than DI. Put simply, IoC means letting other code call you rather than insisting on doing the calling. An example of IoC without DI is the <a href="/wiki/Template_method_pattern" title="Template method pattern">template method pattern</a>. Here <a href="/wiki/Polymorphism_(computer_science)" title="Polymorphism (computer science)">polymorphism</a> is achieved through <a href="/wiki/Inheritance_(object-oriented_programming)#Subclasses_and_superclasses" title="Inheritance (object-oriented programming)">subclassing</a>, that is, <a href="/wiki/Inheritance_(object-oriented_programming)" title="Inheritance (object-oriented programming)">inheritance</a>.<sup class="reference" id="cite_ref-16"><a href="#cite_note-16">[16]</a></sup></p>
<p>Dependency injection implements IoC through <a href="/wiki/Object_composition" title="Object composition">composition</a> so is often identical to that of the <a href="/wiki/Strategy_pattern" title="Strategy pattern">strategy pattern</a>, but while the strategy pattern is intended for dependencies to be interchangeable throughout an object's <a href="/wiki/Object_lifetime" title="Object lifetime">lifetime</a>, in dependency injection it may be that only a single instance of a dependency is used.<sup class="reference" id="cite_ref-17"><a href="#cite_note-17">[17]</a></sup> This still achieves polymorphism, but through <a href="/wiki/Delegation_pattern" title="Delegation pattern">delegation</a> and <a href="/wiki/Object_composition" title="Object composition">composition</a>.</p>
<h3><span class="mw-headline" id="Dependency_injection_frameworks">Dependency injection frameworks</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Dependency_injection&amp;action=edit&amp;section=3" title="Edit section: Dependency injection frameworks">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p><a href="/wiki/Application_framework" title="Application framework">Application frameworks</a> such as <a href="/wiki/Spring_Framework" title="Spring Framework">Spring</a>, <a class="mw-redirect" href="/wiki/Guice" title="Guice">Guice</a>, <a class="mw-redirect" href="/wiki/Play_framework" title="Play framework">Play framework</a>, <a class="external text" href="http://ruediste.github.io/salta/" rel="nofollow">Salta</a>, <a href="/wiki/HK2_DI_Kernel" title="HK2 DI Kernel">Glassfish HK2</a>, and <a href="/wiki/Managed_Extensibility_Framework" title="Managed Extensibility Framework">Managed Extensibility Framework</a> (MEF) support dependency injection but are not required to do dependency injection.<sup class="reference" id="cite_ref-18"><a href="#cite_note-18">[18]</a></sup><sup class="reference" id="cite_ref-19"><a href="#cite_note-19">[19]</a></sup></p>
<h3><span class="mw-headline" id="Advantages">Advantages</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Dependency_injection&amp;action=edit&amp;section=4" title="Edit section: Advantages">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<ul>
<li>Dependency injection allows a client the flexibility of being configurable. Only the client's behavior is fixed. The client may act on anything that supports the intrinsic interface the client expects.</li>
<li>Dependency injection can be used to externalize a system's configuration details into configuration files allowing the system to be reconfigured without recompilation. Separate configurations can be written for different situations that require different implementations of components. This includes, but is not limited to, testing.</li>
<li>Because dependency injection doesn't require any change in code behavior it can be applied to legacy code as a <a href="/wiki/Code_refactoring" title="Code refactoring">refactoring</a>. The result is clients that are more independent and that are easier to <a href="/wiki/Unit_testing" title="Unit testing">unit test</a> in isolation using <a href="/wiki/Method_stub" title="Method stub">stubs</a> or <a href="/wiki/Mock_object" title="Mock object">mock objects</a> that simulate other objects not under test. This ease of testing is often the first benefit noticed when using dependency injection.</li>
<li>Dependency injection allows a client to remove all knowledge of a concrete implementation that it needs to use. This helps isolate the client from the impact of design changes and defects. It promotes reusability, testability and maintainability.<sup class="reference" id="cite_ref-JSR330_20-0"><a href="#cite_note-JSR330-20">[20]</a></sup></li>
<li>Reduction of <a href="/wiki/Boilerplate_code" title="Boilerplate code">boilerplate code</a> in the application objects since all work to initialize or set up dependencies is handled by a provider component.<sup class="reference" id="cite_ref-JSR330_20-1"><a href="#cite_note-JSR330-20">[20]</a></sup></li>
<li>Dependency injection allows concurrent or independent development. Two developers can independently develop <a href="/wiki/Class_(computer_programming)" title="Class (computer programming)">classes</a> that use each other, while only needing to know the interface the classes will communicate through. <a href="/wiki/Plug-in_(computing)" title="Plug-in (computing)">Plugins</a> are often developed by third party shops that never even talk to the developers who created the product that uses the plugins.</li>
<li>Dependency Injection decreases coupling between a class and its dependency.<sup class="reference" id="cite_ref-21"><a href="#cite_note-21">[21]</a></sup><sup class="reference" id="cite_ref-22"><a href="#cite_note-22">[22]</a></sup></li>
</ul>
<h3><span class="mw-headline" id="Disadvantages">Disadvantages</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Dependency_injection&amp;action=edit&amp;section=5" title="Edit section: Disadvantages">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<ul>
<li>Dependency injection creates clients that demand configuration details be supplied by construction code. This can be onerous when obvious defaults are available.</li>
<li>Dependency injection can make code difficult to trace (read) because it separates behavior from construction. This means developers must refer to more files to follow how a system performs.</li>
<li>Dependency injection typically requires more upfront development effort since one can not summon into being something right when and where it is needed but must ask that it be injected and then ensure that it has been injected.</li>
<li>Dependency injection can cause an explosion of types, especially in languages that have explicit interface types, like Java and C# <sup class="reference" id="cite_ref-23"><a href="#cite_note-23">[23]</a></sup></li>
<li>Dependency injection forces complexity to move out of classes and into the linkages between classes which might not always be desirable or easily managed.<sup class="reference" id="cite_ref-stackoverflow.com_24-0"><a href="#cite_note-stackoverflow.com-24">[24]</a></sup></li>
<li>Ironically, dependency injection can encourage dependence on a dependency injection framework.<sup class="reference" id="cite_ref-stackoverflow.com_24-1"><a href="#cite_note-stackoverflow.com-24">[24]</a></sup><sup class="reference" id="cite_ref-sites.google.com_25-0"><a href="#cite_note-sites.google.com-25">[25]</a></sup><sup class="reference" id="cite_ref-26"><a href="#cite_note-26">[26]</a></sup></li>
</ul>
<h2><span class="mw-headline" id="Examples">Examples</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Dependency_injection&amp;action=edit&amp;section=6" title="Edit section: Examples">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<h3><span class="mw-headline" id="Without_dependency_injection">Without dependency injection</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Dependency_injection&amp;action=edit&amp;section=7" title="Edit section: Without dependency injection">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>In the following <a href="/wiki/Java_(programming_language)" title="Java (programming language)">Java</a> example, the Client class contains a Service <a href="/wiki/Member_variable" title="Member variable">member variable</a> that is initialized by the Client <a href="/wiki/Constructor_(object-oriented_programming)" title="Constructor (object-oriented programming)">constructor</a>. The client controls which implementation of service is used and controls its construction. In this situation, the client is said to have a hard-coded dependency on ServiceExample.</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="c1">// An example without dependency injection</span>
<span class="kd">public</span> <span class="kd">class</span> <span class="nc">Client</span> <span class="o">{</span>
    <span class="c1">// Internal reference to the service used by this client</span>
    <span class="kd">private</span> <span class="n">Service</span> <span class="n">service</span><span class="o">;</span>

    <span class="c1">// Constructor</span>
    <span class="n">Client</span><span class="o">()</span> <span class="o">{</span>
        <span class="c1">// Specify a specific implementation in the constructor instead of using dependency injection</span>
        <span class="n">service</span> <span class="o">=</span> <span class="k">new</span> <span class="n">Service</span><span class="o">();</span>
    <span class="o">}</span>

    <span class="c1">// Method within this client that uses the services</span>
    <span class="kd">public</span> <span class="n">String</span> <span class="nf">greet</span><span class="o">()</span> <span class="o">{</span>
        <span class="k">return</span> <span class="s">"Hello "</span> <span class="o">+</span> <span class="n">service</span><span class="o">.</span><span class="na">getName</span><span class="o">();</span>
    <span class="o">}</span>
<span class="o">}</span>
</pre></div>
<p>Dependency injection is an alternative technique to initialize the member variable rather than explicitly creating a service object as shown above.</p>
<h3><span class="mw-headline" id="Three_types_of_dependency_injection">Three types of dependency injection</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Dependency_injection&amp;action=edit&amp;section=8" title="Edit section: Three types of dependency injection">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>There are at least three ways an object can receive a reference to an external module:<sup class="reference" id="cite_ref-27"><a href="#cite_note-27">[27]</a></sup></p>
<ul>
<li><i>constructor injection</i>: the dependencies are provided through a class constructor.</li>
<li><i>setter injection</i>: the client exposes a setter method that the injector uses to inject the dependency.</li>
<li><i>interface injection</i>: the dependency provides an injector method that will inject the dependency into any client passed to it. Clients must implement an interface that exposes a <a class="mw-redirect" href="/wiki/Setter_method" title="Setter method">setter method</a> that accepts the dependency.</li>
</ul>
<h4><span class="mw-headline" id="Other_types">Other types</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Dependency_injection&amp;action=edit&amp;section=9" title="Edit section: Other types">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>It is possible for DI frameworks to have other types of <i>injection</i> beyond those presented above.<sup class="reference" id="cite_ref-28"><a href="#cite_note-28">[28]</a></sup></p>
<p>Testing frameworks may also use other types. Some modern testing frameworks do not even require that clients actively accept dependency injection thus making legacy code testable. In particular, in the Java language it is possible to use reflection to make private attributes public when testing and thus accept injections by assignment.<sup class="reference" id="cite_ref-29"><a href="#cite_note-29">[29]</a></sup></p>
<p>Some attempts at Inversion of Control do not provide full removal of dependency but instead simply substitute one form of dependency for another. As a rule of thumb, if a programmer can look at nothing but the client code and tell what framework is being used, then the client has a hard-coded dependency on the framework.</p>
<h4><span class="mw-headline" id="Constructor_injection">Constructor injection</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Dependency_injection&amp;action=edit&amp;section=10" title="Edit section: Constructor injection">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>This method requires the client to provide a parameter in a <a href="/wiki/Constructor_(object-oriented_programming)" title="Constructor (object-oriented programming)">constructor</a> for the dependency.</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="c1">// Constructor</span>
<span class="n">Client</span><span class="o">(</span><span class="n">Service</span> <span class="n">service</span><span class="o">)</span> <span class="o">{</span>
    <span class="c1">// Save the reference to the passed-in service inside this client</span>
    <span class="k">this</span><span class="o">.</span><span class="na">service</span> <span class="o">=</span> <span class="n">service</span><span class="o">;</span>
<span class="o">}</span>
</pre></div>
<h4><span class="mw-headline" id="Setter_injection">Setter injection</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Dependency_injection&amp;action=edit&amp;section=11" title="Edit section: Setter injection">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>This method requires the client to provide a <a class="mw-redirect" href="/wiki/Setter_method" title="Setter method">setter method</a> for each dependency.</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="c1">// Setter method</span>
<span class="kd">public</span> <span class="kt">void</span> <span class="nf">setService</span><span class="o">(</span><span class="n">Service</span> <span class="n">service</span><span class="o">)</span> <span class="o">{</span>
    <span class="c1">// Save the reference to the passed-in service inside this client</span>
    <span class="k">this</span><span class="o">.</span><span class="na">service</span> <span class="o">=</span> <span class="n">service</span><span class="o">;</span>
<span class="o">}</span>
</pre></div>
<h4><span class="mw-headline" id="Interface_injection">Interface injection</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Dependency_injection&amp;action=edit&amp;section=12" title="Edit section: Interface injection">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>This is simply the client publishing a role interface to the setter methods of the client's dependencies. It can be used to establish how the injector should talk to the client when injecting dependencies.</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="c1">// Service setter interface.</span>
<span class="kd">public</span> <span class="kd">interface</span> <span class="nc">ServiceSetter</span> <span class="o">{</span>
    <span class="kd">public</span> <span class="kt">void</span> <span class="nf">setService</span><span class="o">(</span><span class="n">Service</span> <span class="n">service</span><span class="o">);</span>
<span class="o">}</span>

<span class="c1">// Client class</span>
<span class="kd">public</span> <span class="kd">class</span> <span class="nc">Client</span> <span class="kd">implements</span> <span class="n">ServiceSetter</span> <span class="o">{</span>
    <span class="c1">// Internal reference to the service used by this client.</span>
    <span class="kd">private</span> <span class="n">Service</span> <span class="n">service</span><span class="o">;</span>

    <span class="c1">// Set the service that this client is to use.</span>
    <span class="nd">@Override</span>
    <span class="kd">public</span> <span class="kt">void</span> <span class="nf">setService</span><span class="o">(</span><span class="n">Service</span> <span class="n">service</span><span class="o">)</span> <span class="o">{</span>
        <span class="k">this</span><span class="o">.</span><span class="na">service</span> <span class="o">=</span> <span class="n">service</span><span class="o">;</span>
    <span class="o">}</span>
<span class="o">}</span>
</pre></div>
<h4><span class="mw-headline" id="Constructor_injection_comparison">Constructor injection comparison</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Dependency_injection&amp;action=edit&amp;section=13" title="Edit section: Constructor injection comparison">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>Preferred when all dependencies can be constructed first because it can be used to ensure the client object is always in a valid state, as opposed to having some of its dependency references be null (not be set). However, on its own, it lacks the flexibility to have its dependencies changed later. This can be a first step towards making the client <a href="/wiki/Immutable_object" title="Immutable object">immutable</a> and therefore <a href="/wiki/Thread_safety" title="Thread safety">thread safe</a>.</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="c1">// Constructor</span>
<span class="n">Client</span><span class="o">(</span><span class="n">Service</span> <span class="n">service</span><span class="o">,</span> <span class="n">Service</span> <span class="n">otherService</span><span class="o">)</span> <span class="o">{</span>
    <span class="k">if</span> <span class="o">(</span><span class="n">service</span> <span class="o">==</span> <span class="kc">null</span><span class="o">)</span> <span class="o">{</span>
        <span class="k">throw</span> <span class="k">new</span> <span class="n">InvalidParameterException</span><span class="o">(</span><span class="s">"service must not be null"</span><span class="o">);</span>
    <span class="o">}</span>
    <span class="k">if</span> <span class="o">(</span><span class="n">otherService</span> <span class="o">==</span> <span class="kc">null</span><span class="o">)</span> <span class="o">{</span>
        <span class="k">throw</span> <span class="k">new</span> <span class="n">InvalidParameterException</span><span class="o">(</span><span class="s">"otherService must not be null"</span><span class="o">);</span>
    <span class="o">}</span>

    <span class="c1">// Save the service references inside this client</span>
    <span class="k">this</span><span class="o">.</span><span class="na">service</span> <span class="o">=</span> <span class="n">service</span><span class="o">;</span>
    <span class="k">this</span><span class="o">.</span><span class="na">otherService</span> <span class="o">=</span> <span class="n">otherService</span><span class="o">;</span>
<span class="o">}</span>
</pre></div>
<h4><span class="mw-headline" id="Setter_injection_comparison">Setter injection comparison</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Dependency_injection&amp;action=edit&amp;section=14" title="Edit section: Setter injection comparison">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>Requires the client to provide a setter method for each dependency. This gives the freedom to manipulate the state of the dependency references at any time. This offers flexibility, but if there is more than one dependency to be injected, it is difficult for the client to ensure that all dependencies are injected before the client could be provided for use.</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="c1">// Set the service to be used by this client</span>
<span class="kd">public</span> <span class="kt">void</span> <span class="nf">setService</span><span class="o">(</span><span class="n">Service</span> <span class="n">service</span><span class="o">)</span> <span class="o">{</span>
    <span class="k">if</span> <span class="o">(</span><span class="n">service</span> <span class="o">==</span> <span class="kc">null</span><span class="o">)</span> <span class="o">{</span>
        <span class="k">throw</span> <span class="k">new</span> <span class="n">InvalidParameterException</span><span class="o">(</span><span class="s">"service must not be null"</span><span class="o">);</span>
    <span class="o">}</span>
    <span class="k">this</span><span class="o">.</span><span class="na">service</span> <span class="o">=</span> <span class="n">service</span><span class="o">;</span>
<span class="o">}</span>

<span class="c1">// Set the other service to be used by this client</span>
<span class="kd">public</span> <span class="kt">void</span> <span class="nf">setOtherService</span><span class="o">(</span><span class="n">Service</span> <span class="n">otherService</span><span class="o">)</span> <span class="o">{</span>
    <span class="k">if</span> <span class="o">(</span><span class="n">otherService</span> <span class="o">==</span> <span class="kc">null</span><span class="o">)</span> <span class="o">{</span>
        <span class="k">throw</span> <span class="k">new</span> <span class="n">InvalidParameterException</span><span class="o">(</span><span class="s">"otherService must not be null"</span><span class="o">);</span>
    <span class="o">}</span>
    <span class="k">this</span><span class="o">.</span><span class="na">otherService</span> <span class="o">=</span> <span class="n">otherService</span><span class="o">;</span>
<span class="o">}</span>
</pre></div>
<p>Because these injections happen independently there is no way to tell when the injector is finished wiring the client. A dependency can be left null simply by the injector failing to call its setter. This forces the check that injection was completed from when the client is assembled to whenever it is used.</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="c1">// Set the service to be used by this client</span>
<span class="kd">public</span> <span class="kt">void</span> <span class="nf">setService</span><span class="o">(</span><span class="n">Service</span> <span class="n">service</span><span class="o">)</span> <span class="o">{</span>
    <span class="k">this</span><span class="o">.</span><span class="na">service</span> <span class="o">=</span> <span class="n">service</span><span class="o">;</span>
<span class="o">}</span>

<span class="c1">// Set the other service to be used by this client</span>
<span class="kd">public</span> <span class="kt">void</span> <span class="nf">setOtherService</span><span class="o">(</span><span class="n">Service</span> <span class="n">otherService</span><span class="o">)</span> <span class="o">{</span>
    <span class="k">this</span><span class="o">.</span><span class="na">otherService</span> <span class="o">=</span> <span class="n">otherService</span><span class="o">;</span>
<span class="o">}</span>

<span class="c1">// Check the service references of this client</span>
<span class="kd">private</span> <span class="kt">void</span> <span class="nf">validateState</span><span class="o">()</span> <span class="o">{</span>
    <span class="k">if</span> <span class="o">(</span><span class="n">service</span> <span class="o">==</span> <span class="kc">null</span><span class="o">)</span> <span class="o">{</span>
        <span class="k">throw</span> <span class="k">new</span> <span class="n">IllegalStateException</span><span class="o">(</span><span class="s">"service must not be null"</span><span class="o">);</span>
    <span class="o">}</span>
    <span class="k">if</span> <span class="o">(</span><span class="n">otherService</span> <span class="o">==</span> <span class="kc">null</span><span class="o">)</span> <span class="o">{</span>
        <span class="k">throw</span> <span class="k">new</span> <span class="n">IllegalStateException</span><span class="o">(</span><span class="s">"otherService must not be null"</span><span class="o">);</span>
    <span class="o">}</span>
<span class="o">}</span>

<span class="c1">// Method that uses the service references</span>
<span class="kd">public</span> <span class="kt">void</span> <span class="nf">doSomething</span><span class="o">()</span> <span class="o">{</span>
    <span class="n">validateState</span><span class="o">();</span>
    <span class="n">service</span><span class="o">.</span><span class="na">doYourThing</span><span class="o">();</span>
    <span class="n">otherService</span><span class="o">.</span><span class="na">doYourThing</span><span class="o">();</span>
<span class="o">}</span>
</pre></div>
<h4><span class="mw-headline" id="Interface_injection_comparison">Interface injection comparison</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Dependency_injection&amp;action=edit&amp;section=15" title="Edit section: Interface injection comparison">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>The advantage of interface injection is that dependencies can be completely ignorant of their clients yet can still receive a reference to a new client and, using it, send a reference-to-self back to the client. In this way, the dependencies become injectors. The key is that the injecting method (which could just be a classic setter method) is provided through an interface.</p>
<p>An assembler is still needed to introduce the client and its dependencies. The assembler would take a reference to the client, cast it to the setter interface that sets that dependency, and pass it to that dependency object which would turn around and pass a reference-to-self back to the client.</p>
<p>For interface injection to have value, the dependency must do something in addition to simply passing back a reference to itself. This could be acting as a factory or sub-assembler to resolve other dependencies, thus abstracting some details from the main assembler. It could be reference-counting so that the dependency knows how many clients are using it. If the dependency maintains a collection of clients, it could later inject them all with a different instance of itself.</p>
<p><sup class="noprint Inline-Template" style="white-space:nowrap;">[<i><a class="mw-redirect" href="/wiki/Wikipedia:AUDIENCE" title="Wikipedia:AUDIENCE"><span title="Wikipedia:AUDIENCE">example needed</span></a></i>]</sup></p>
<h3><span class="mw-headline" id="Assembling_examples">Assembling examples</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Dependency_injection&amp;action=edit&amp;section=16" title="Edit section: Assembling examples">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Manually assembling in main by hand is one way of implementing dependency injection.</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="kd">public</span> <span class="kd">class</span> <span class="nc">Injector</span> <span class="o">{</span>
    <span class="kd">public</span> <span class="kd">static</span> <span class="kt">void</span> <span class="nf">main</span><span class="o">(</span><span class="n">String</span><span class="o">[]</span> <span class="n">args</span><span class="o">)</span> <span class="o">{</span>
        <span class="c1">// Build the dependencies first</span>
        <span class="n">Service</span> <span class="n">service</span> <span class="o">=</span> <span class="k">new</span> <span class="n">ServiceExample</span><span class="o">();</span>

        <span class="c1">// Inject the service, constructor style</span>
        <span class="n">Client</span> <span class="n">client</span> <span class="o">=</span> <span class="k">new</span> <span class="n">Client</span><span class="o">(</span><span class="n">service</span><span class="o">);</span>

        <span class="c1">// Use the objects</span>
        <span class="n">System</span><span class="o">.</span><span class="na">out</span><span class="o">.</span><span class="na">println</span><span class="o">(</span><span class="n">client</span><span class="o">.</span><span class="na">greet</span><span class="o">());</span>
    <span class="o">}</span>	
<span class="o">}</span>
</pre></div>
<p>The example above constructs the object graph manually and then invokes it at one point to start it working. Important to note is that this injector is not pure. It uses one of the objects it constructs. It has a purely construction-only relationship with ServiceExample but mixes construction and using of Client. This should not be common. It is, however, unavoidable. Just like object oriented software needs a non-object oriented static method like main() to get started, a dependency injected object graph needs at least one (preferably only one) entry point to get the using started.</p>
<p>Manual construction in the main method may not be this straight forward and may involve calling <a href="/wiki/Builder_pattern" title="Builder pattern">builders</a>, <a href="/wiki/Factory_(object-oriented_programming)" title="Factory (object-oriented programming)">factories</a>, or other <a href="/wiki/Creational_pattern" title="Creational pattern">construction patterns</a> as well. This can be fairly advanced and abstract. The line is crossed from manual dependency injection to <a href="/wiki/Software_framework" title="Software framework">framework</a> dependency injection once the constructing code is no longer custom to the application and is instead universal.<sup class="reference" id="cite_ref-30"><a href="#cite_note-30">[30]</a></sup></p>
<p>Frameworks like <a href="/wiki/Spring_Framework" title="Spring Framework">Spring</a> can construct these same objects and wire them together before returning a reference to client. Notice that all mention of ServiceExample has been removed from the code.</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="kn">import</span> <span class="nn">org.springframework.beans.factory.BeanFactory</span><span class="o">;</span>
<span class="kn">import</span> <span class="nn">org.springframework.context.ApplicationContext</span><span class="o">;</span>
<span class="kn">import</span> <span class="nn">org.springframework.context.support.ClassPathXmlApplicationContext</span><span class="o">;</span>

<span class="kd">public</span> <span class="kd">class</span> <span class="nc">Injector</span> <span class="o">{</span>

	<span class="kd">public</span> <span class="kd">static</span> <span class="kt">void</span> <span class="nf">main</span><span class="o">(</span><span class="n">String</span><span class="o">[]</span> <span class="n">args</span><span class="o">)</span> <span class="o">{</span>

		<span class="c1">// -- Assembling objects -- //</span>

        	<span class="n">BeanFactory</span> <span class="n">beanfactory</span><span class="o">=</span><span class="k">new</span> <span class="n">ClassPathXmlApplicationContext</span><span class="o">(</span><span class="s">"Beans.xml"</span><span class="o">);</span>

	        <span class="n">Client</span> <span class="n">client</span><span class="o">=(</span><span class="n">Client</span><span class="o">)</span><span class="n">beanfactory</span><span class="o">.</span><span class="na">getBean</span><span class="o">(</span><span class="s">"client"</span><span class="o">);</span>

		<span class="c1">// -- Using objects -- //</span>
		
		<span class="n">System</span><span class="o">.</span><span class="na">out</span><span class="o">.</span><span class="na">println</span><span class="o">(</span> <span class="n">client</span><span class="o">.</span><span class="na">greet</span><span class="o">()</span> <span class="o">);</span>
	<span class="o">}</span>
<span class="o">}</span>
</pre></div>
<p>Frameworks like Spring allow assembly details to be externalized in configuration files. This code (above) constructs objects and wires them together according to Beans.xml (below). ServiceExample is still constructed even though it's only mentioned below. A long and complex object graph can be defined this way and the only class mentioned in code would be the one with the entry point method, which in this case is greet().</p>

  <span class="na">xsi:schemaLocation=</span><span class="s">"http://www.springframework.org/schema/beans</span>
<span class="s">  http://www.springframework.org/schema/beans/spring-beans-3.0.xsd"</span><span class="nt">&gt;</span>

    <span class="nt">&lt;bean</span> <span class="na">id=</span><span class="s">"service"</span> <span class="na">class=</span><span class="s">"ServiceExample"</span><span class="nt">&gt;</span>
    <span class="nt">&lt;/bean&gt;</span>

    <span class="nt">&lt;bean</span> <span class="na">id=</span><span class="s">"client"</span> <span class="na">class=</span><span class="s">"Client"</span><span class="nt">&gt;</span>
        <span class="nt">&lt;constructor-arg</span> <span class="na">value=</span><span class="s">"service"</span> <span class="nt">/&gt;</span>        
    <span class="nt">&lt;/bean&gt;</span>
<span class="nt">&lt;/beans&gt;</span>
</pre></div>
<p>In the example above Client and Service have not had to undergo any changes to be provided by spring. They are allowed to remain simple <a href="/wiki/Plain_Old_Java_Object" title="Plain Old Java Object">POJOs</a>.<sup class="reference" id="cite_ref-31"><a href="#cite_note-31">[31]</a></sup><sup class="reference" id="cite_ref-32"><a href="#cite_note-32">[32]</a></sup><sup class="reference" id="cite_ref-33"><a href="#cite_note-33">[33]</a></sup> This shows how spring can connect services and clients that are completely ignorant of its existence. This could not be said if spring annotations are added to the classes. By keeping spring specific annotations and calls from spreading out among many classes, the system stays only loosely dependent on spring.<sup class="reference" id="cite_ref-sites.google.com_25-1"><a href="#cite_note-sites.google.com-25">[25]</a></sup> This can be important if the system intends to outlive spring.</p>
<p>The choice to keep POJOs pure doesn't come without cost. Rather than spending the effort to develop and maintain complex configuration files it is possible to simply use annotations to mark classes and let spring do the rest of the work. Resolving dependencies can be simple if they follow a convention such as matching by type or by name. This is choosing <a href="/wiki/Convention_over_configuration" title="Convention over configuration">convention over configuration</a>.<sup class="reference" id="cite_ref-34"><a href="#cite_note-34">[34]</a></sup> It is also arguable that, when refactoring to another framework, removing framework specific annotations would be a trivial part of the task<sup class="reference" id="cite_ref-35"><a href="#cite_note-35">[35]</a></sup> and many injection annotations are now standardized.<sup class="reference" id="cite_ref-36"><a href="#cite_note-36">[36]</a></sup><sup class="reference" id="cite_ref-37"><a href="#cite_note-37">[37]</a></sup></p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="kn">import</span> <span class="nn">org.springframework.beans.factory.BeanFactory</span><span class="o">;</span>
<span class="kn">import</span> <span class="nn">org.springframework.context.ApplicationContext</span><span class="o">;</span>
<span class="kn">import</span> <span class="nn">org.springframework.context.annotation.AnnotationConfigApplicationContext</span><span class="o">;</span>

<span class="kd">public</span> <span class="kd">class</span> <span class="nc">Injector</span> <span class="o">{</span>
    <span class="kd">public</span> <span class="kd">static</span> <span class="kt">void</span> <span class="nf">main</span><span class="o">(</span><span class="n">String</span><span class="o">[]</span> <span class="n">args</span><span class="o">)</span> <span class="o">{</span>
        <span class="c1">// Assemble the objects</span>
        <span class="n">BeanFactory</span> <span class="n">beanfactory</span> <span class="o">=</span> <span class="k">new</span> <span class="n">AnnotationConfigApplicationContext</span><span class="o">(</span><span class="n">MyConfiguration</span><span class="o">.</span><span class="na">class</span><span class="o">);</span>
        <span class="n">Client</span> <span class="n">client</span> <span class="o">=</span> <span class="n">beanfactory</span><span class="o">.</span><span class="na">getBean</span><span class="o">(</span><span class="n">Client</span><span class="o">.</span><span class="na">class</span><span class="o">);</span>

        <span class="c1">// Use the objects</span>
        <span class="n">System</span><span class="o">.</span><span class="na">out</span><span class="o">.</span><span class="na">println</span><span class="o">(</span><span class="n">client</span><span class="o">.</span><span class="na">greet</span><span class="o">());</span>
    <span class="o">}</span>
<span class="o">}</span>
</pre></div>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="kn">import</span> <span class="nn">org.springframework.context.annotation.Bean</span><span class="o">;</span>
<span class="kn">import</span> <span class="nn">org.springframework.context.annotation.ComponentScan</span><span class="o">;</span>
<span class="kn">import</span> <span class="nn">org.springframework.context.annotation.Configuration</span><span class="o">;</span>

<span class="nd">@ComponentScan</span>
<span class="kd">static</span> <span class="kd">class</span> <span class="nc">MyConfiguration</span> <span class="o">{</span>
    <span class="nd">@Bean</span>
    <span class="kd">public</span> <span class="n">Client</span> <span class="nf">client</span><span class="o">(</span><span class="n">ServiceExample</span> <span class="n">service</span><span class="o">)</span> <span class="o">{</span>
        <span class="k">return</span> <span class="k">new</span> <span class="n">Client</span><span class="o">(</span><span class="n">service</span><span class="o">);</span>
    <span class="o">}</span>
<span class="o">}</span>
</pre></div>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="nd">@Component</span>
<span class="kd">public</span> <span class="kd">class</span> <span class="nc">ServiceExample</span> <span class="o">{</span>
    <span class="kd">public</span> <span class="n">String</span> <span class="nf">getName</span><span class="o">()</span> <span class="o">{</span>
        <span class="k">return</span> <span class="s">"World!"</span><span class="o">;</span>
    <span class="o">}</span>
<span class="o">}</span>
</pre></div>
<h3><span class="mw-headline" id="Assembly_comparison">Assembly comparison</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Dependency_injection&amp;action=edit&amp;section=17" title="Edit section: Assembly comparison">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>The different injector implementations (factories, <a href="/wiki/Service_locator_pattern" title="Service locator pattern">service locators</a>, and dependency injection containers) are not that different as far as dependency injection is concerned. What makes all the difference is where they are allowed to be used. Move calls to a factory or a service locator out of the client and into main and suddenly main makes a fairly good dependency injection container.</p>
<p>By moving all knowledge of the injector out, a clean client, free of knowledge of the outside world, is left behind. However, any object that uses other objects can be considered a client. The object that contains main is no exception. This main object is not using dependency injection. It's actually using the service locator pattern. This can't be avoided because the choice of service implementations must be made somewhere.</p>
<p>Externalizing the dependencies into configuration files doesn't change this fact. What makes this reality part of a good design is that the service locator is not spread throughout the code base. It's confined to one place per application. This leaves the rest of the code base free to use dependency injection to make clean clients.</p>
<h3><span class="mw-headline" id="AngularJS_example">AngularJS example</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Dependency_injection&amp;action=edit&amp;section=18" title="Edit section: AngularJS example">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>In the <a href="/wiki/AngularJS" title="AngularJS">AngularJS</a> framework, there are only three ways a component (object or function) can directly access its dependencies:</p>
<ol>
<li>The component can create the dependency, typically using the <code>new</code> operator.</li>
<li>The component can look up the dependency, by referring to a global variable.</li>
<li>The component can have the dependency passed to it where it is needed.</li>
</ol>
<p>The first two options of creating or looking up dependencies are not optimal because they hard code the dependency to the component. This makes it difficult, if not impossible, to modify the dependencies. This is especially problematic in tests, where it is often desirable to provide mock dependencies for test isolation.</p>
<p>The third option is the most viable, since it removes the responsibility of locating the dependency from the component. The dependency is simply handed to the component.</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="kd">function</span> <span class="nx">SomeClass</span><span class="p">(</span><span class="nx">greeter</span><span class="p">)</span> <span class="p">{</span>
  <span class="k">this</span><span class="p">.</span><span class="nx">greeter</span> <span class="o">=</span> <span class="nx">greeter</span><span class="p">;</span>
<span class="p">}</span>

<span class="nx">SomeClass</span><span class="p">.</span><span class="nx">prototype</span><span class="p">.</span><span class="nx">doSomething</span> <span class="o">=</span> <span class="kd">function</span><span class="p">(</span><span class="nx">name</span><span class="p">)</span> <span class="p">{</span>
  <span class="k">this</span><span class="p">.</span><span class="nx">greeter</span><span class="p">.</span><span class="nx">greet</span><span class="p">(</span><span class="nx">name</span><span class="p">);</span>
<span class="p">}</span>
</pre></div>
<p>In the above example <code>SomeClass</code> is not concerned with creating or locating the greeter dependency, it is simply handed the greeter when it is instantiated.</p>
<p>This is desirable, but it puts the responsibility of getting hold of the dependency on the code that constructs <code>SomeClass</code>.</p>
<p>To manage the responsibility of dependency creation, each AngularJS application has an injector. The injector is a service locator that is responsible for construction and look-up of dependencies.</p>
<p>Here is an example of using the injector service:</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="c1">// Provide the wiring information in a module</span>
<span class="kd">var</span> <span class="nx">myModule</span> <span class="o">=</span> <span class="nx">angular</span><span class="p">.</span><span class="nx">module</span><span class="p">(</span><span class="s1">'myModule'</span><span class="p">,</span> <span class="p">[]);</span>

<span class="c1">// Teach the injector how to build a greeter service. </span>
<span class="c1">// Notice that greeter is dependent on the $window service. </span>
<span class="c1">// The greeter service is an object that</span>
<span class="c1">// contains a greet method.</span>
<span class="nx">myModule</span><span class="p">.</span><span class="nx">factory</span><span class="p">(</span><span class="s1">'greeter'</span><span class="p">,</span> <span class="kd">function</span><span class="p">(</span><span class="nx">$window</span><span class="p">)</span> <span class="p">{</span>
  <span class="k">return</span> <span class="p">{</span>
    <span class="nx">greet</span><span class="o">:</span> <span class="kd">function</span><span class="p">(</span><span class="nx">text</span><span class="p">)</span> <span class="p">{</span>
      <span class="nx">$window</span><span class="p">.</span><span class="nx">alert</span><span class="p">(</span><span class="nx">text</span><span class="p">);</span>
    <span class="p">}</span>
  <span class="p">};</span>
<span class="p">});</span>
</pre></div>
<p>Create a new injector that can provide components defined in the <code>myModule</code> module and request our greeter service from the injector. (This is usually done automatically by the AngularJS bootstrap).</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="kd">var</span> <span class="nx">injector</span> <span class="o">=</span> <span class="nx">angular</span><span class="p">.</span><span class="nx">injector</span><span class="p">([</span><span class="s1">'myModule'</span><span class="p">,</span> <span class="s1">'ng'</span><span class="p">]);</span>
<span class="kd">var</span> <span class="nx">greeter</span> <span class="o">=</span> <span class="nx">injector</span><span class="p">.</span><span class="nx">get</span><span class="p">(</span><span class="s1">'greeter'</span><span class="p">);</span>
</pre></div>
<p>Asking for dependencies solves the issue of hard coding, but it also means that the injector needs to be passed throughout the application. Passing the injector breaks the <a href="/wiki/Law_of_Demeter" title="Law of Demeter">Law of Demeter</a>. To remedy this, we use a declarative notation in our HTML templates, to hand the responsibility of creating components over to the injector, as in this example:</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="o">&lt;</span><span class="nx">div</span> <span class="nx">ng</span><span class="o">-</span><span class="nx">controller</span><span class="o">=</span><span class="s2">"MyController"</span><span class="o">&gt;</span>
  <span class="o">&lt;</span><span class="nx">button</span> <span class="nx">ng</span><span class="o">-</span><span class="nx">click</span><span class="o">=</span><span class="s2">"sayHello()"</span><span class="o">&gt;</span><span class="nx">Hello</span><span class="o">&lt;</span><span class="err">/button&gt;</span>
<span class="o">&lt;</span><span class="err">/div&gt;</span>
<span class="kd">function</span> <span class="nx">MyController</span><span class="p">(</span><span class="nx">$scope</span><span class="p">,</span> <span class="nx">greeter</span><span class="p">)</span> <span class="p">{</span>
  <span class="nx">$scope</span><span class="p">.</span><span class="nx">sayHello</span> <span class="o">=</span> <span class="kd">function</span><span class="p">()</span> <span class="p">{</span>
    <span class="nx">greeter</span><span class="p">.</span><span class="nx">greet</span><span class="p">(</span><span class="s1">'Hello World'</span><span class="p">);</span>
  <span class="p">};</span>
<span class="p">}</span>
</pre></div>
<p>When AngularJS compiles the HTML, it processes the <code>ng-controller</code> directive, which in turn asks the injector to create an instance of the controller and its dependencies.</p>
<pre>
injector.instantiate(MyController);
</pre>
<p>This is all done behind the scenes. Notice that by having the <code>ng-controller</code> ask the injector to instantiate the class, it can satisfy all of the dependencies of <code>MyController</code> without the controller ever knowing about the injector. This is the best outcome. The application code simply declares the dependencies it needs, without having to deal with the injector. This setup does not break the Law of Demeter.</p>
