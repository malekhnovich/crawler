<h1 class="firstHeading" id="firstHeading" lang="en">Thread-local storage</h1>
<div class="mw-body-content" id="bodyContent">

<p><b>Thread-local storage</b> (<b>TLS</b>) is a computer programming method that uses <a class="mw-redirect" href="/wiki/Static_memory_allocation" title="Static memory allocation">static</a> or global <a class="mw-redirect" href="/wiki/Computer_storage" title="Computer storage">memory</a> local to a <a href="/wiki/Thread_(computing)" title="Thread (computing)">thread</a>.</p>
<p>TLS is used in some places where ordinary, single-threaded programs would use <a class="mw-redirect" href="/wiki/Global_variables" title="Global variables">global variables</a> but where this would be inappropriate in multithreaded cases. An example of such situations is where functions use a global variable to set an error condition (for example the global variable <code><a class="mw-redirect" href="/wiki/Errno" title="Errno">errno</a></code> used by many functions of the C library). If <code><a class="mw-redirect" href="/wiki/Errno" title="Errno">errno</a></code> were a global variable, a call of a system function on one thread may overwrite the value previously set by a call of a system function on a different thread, possibly before following code on that different thread could check for the error condition. The solution is to have <code><a class="mw-redirect" href="/wiki/Errno" title="Errno">errno</a></code> be a variable that looks like it is global, but in fact exists once per thread—i.e., it lives in thread-local storage. A second use case would be multiple threads accumulating information into a global variable. To avoid a <a href="/wiki/Race_condition" title="Race condition">race condition</a>, every access to this global variable would have to be protected by a mutex. Alternatively, each thread might accumulate into a thread-local variable (that, by definition, can not be read from or written to from other threads, implying that there can be no race conditions). Threads then only have to synchronise a final accumulation from their own thread-local variable into a single, truly global variable.</p>
<p>Many systems impose restrictions on the size of the thread-local memory block, in fact often rather tight limits. On the other hand, if a system can provide at least a <a href="/wiki/Memory_address" title="Memory address">memory address</a> (pointer) sized variable thread-local, then this allows the use of arbitrarily sized memory blocks in a thread-local manner, by allocating such a memory block dynamically and storing the <a href="/wiki/Memory_address" title="Memory address">memory address</a> of that block in the thread-local variable.</p>

<h2><span class="mw-headline" id="Windows_implementation">Windows implementation</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Thread-local_storage&amp;action=edit&amp;section=1" title="Edit section: Windows implementation">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The <a href="/wiki/Application_programming_interface" title="Application programming interface">application programming interface</a> (API) function <code>TlsAlloc</code> can be used to obtain an unused <i>TLS slot index</i>; the <i>TLS slot index</i> will then be considered ‘used’.</p>
<p>The <code>TlsGetValue</code> and <code>TlsSetValue</code> functions are then used to read and write a memory address to a thread-local variable identified by the <i>TLS slot index</i>. <code>TlsSetValue</code> only affects the variable for the current thread. The <code>TlsFree</code> function can be called to release the <i>TLS slot index</i>.</p>
<p>There is a <a href="/wiki/Win32_Thread_Information_Block" title="Win32 Thread Information Block">Win32 Thread Information Block</a> for each thread. One of the entries in this block is the thread-local storage table for that thread.<sup class="reference" id="cite_ref-1"><a href="#cite_note-1">[1]</a></sup> TlsAlloc returns an index to this table, unique per address space, for each call. Each thread has its own copy of the thread-local storage table. Hence, each thread can independently use TlsSetValue(index) and obtain the specified value via TlsGetValue(index), because these set and look up an entry in the thread's own table.</p>
<p>Apart from TlsXxx function family, Windows executables can define a section which is mapped to a different page for each thread of the executing process. Unlike TlsXxx values, these pages can contain arbitrary and valid addresses. These addresses, however, are different for each executing thread and therefore should not be passed to asynchronous functions (which may execute in a different thread) or otherwise passed to code which assume that a virtual address is unique within the whole process. TLS sections are managed using <a href="/wiki/Paging" title="Paging">memory paging</a> and its size is quantized to a page size (4kB on x86 machines). Such sections may only be defined inside a main executable of a program - <a href="/wiki/Dynamic-link_library" title="Dynamic-link library">DLLs</a> should not contain such sections, because they are not correctly initialized when loading with LoadLibrary.</p>
<h2><span class="mw-headline" id="Pthreads_implementation">Pthreads implementation</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Thread-local_storage&amp;action=edit&amp;section=2" title="Edit section: Pthreads implementation">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>In the <a class="mw-redirect" href="/wiki/Pthreads" title="Pthreads">Pthreads</a> API, memory local to a thread is designated with the term Thread-specific data.</p>
<p>The functions <code>pthread_key_create</code> and <code>pthread_key_delete</code> are used respectively to create and delete a key for thread-specific data. The type of the key is explicitly left opaque and is referred to as <code>pthread_key_t</code>. This key can be seen by all threads. In each thread, the key can be associated with thread-specific data via <code>pthread_setspecific</code>. The data can later be retrieved using <code>pthread_getspecific</code>.</p>
<p>In addition <code>pthread_key_create</code> can optionally accept a destructor function that will automatically be called at thread exit, if the thread-specific data is not <i>NULL</i>. The destructor receives the value associated with the key as parameter so it can perform cleanup actions (close connections, free memory, etc.). Even when a destructor is specified, the program must still call <code>pthread_key_delete</code> to free the thread-specific data at process level (the destructor only frees the data local to the thread).</p>
<p>It should be noted pthreads isn't the only way Unix multi-threaded and multi-cpu can share memory; far from it. ie. MIT-SHM is an older yet capable standard. Note pthreads has undergone many incompatible version changes, also that (linux) libc stopped using /lib/tls/libs, that pthreads is not "tls" and is not limited to sharing memory between threads in a particular manner structured as a "standard tls". Moreso that in linux gas(1) and ld(1) have been continually modifying how pthreads gets implemented at compile time. tls is important for libc efficiency when multi-thread applications link against it but is rarely a concern for non-OS-lib libs and apps and should perhaps be avoided (due to rapid incompatibility of code).</p>
<h2><span class="mw-headline" id="Language-specific_implementation">Language-specific implementation</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Thread-local_storage&amp;action=edit&amp;section=3" title="Edit section: Language-specific implementation">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Apart from relying on programmers to call the appropriate API functions, it is also possible to extend the programming language to support TLS.</p>
<h3><span class="mw-headline" id="C_and_C.2B.2B">C and C++</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Thread-local_storage&amp;action=edit&amp;section=4" title="Edit section: C and C++">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>In <a href="/wiki/C11_(C_standard_revision)" title="C11 (C standard revision)">C11</a>, the keyword <code>_Thread_local</code> is used to define thread-local variables. The header <code>&lt;threads.h&gt;</code>, if supported, defines <code>thread_local</code> as a synonym for that keyword. Example usage:</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="cp">#include</span> <span class="cpf">&lt;threads.h&gt;</span>
<span class="n">thread_local</span> <span class="kt">int</span> <span class="n">foo</span> <span class="o">=</span> <span class="mi">0</span><span class="p">;</span>
</pre></div>
<p><a href="/wiki/C%2B%2B11" title="C++11">C++11</a> introduces the <code>thread_local</code><sup class="reference" id="cite_ref-2"><a href="#cite_note-2">[2]</a></sup> keyword which can be used in the following cases</p>
<ul>
<li>Namespace level (global) variables</li>
<li>File static variables</li>
<li>Function static variables</li>
<li>Static member variables</li>
</ul>
<p>Aside from that, various C++ compiler implementations provide specific ways to declare thread-local variables:</p>
<ul>
<li><a class="mw-redirect" href="/wiki/Oracle_Solaris_Studio" title="Oracle Solaris Studio">Solaris Studio</a> C/C++, IBM XL C/C++,<sup class="reference" id="cite_ref-3"><a href="#cite_note-3">[3]</a></sup> <a href="/wiki/GNU_Compiler_Collection" title="GNU Compiler Collection">GNU C</a>,<sup class="reference" id="cite_ref-4"><a href="#cite_note-4">[4]</a></sup> <a href="/wiki/Clang" title="Clang">Clang</a><sup class="reference" id="cite_ref-5"><a href="#cite_note-5">[5]</a></sup> and <a href="/wiki/Intel_C%2B%2B_Compiler" title="Intel C++ Compiler">Intel C++ Compiler</a> (Linux systems)<sup class="reference" id="cite_ref-6"><a href="#cite_note-6">[6]</a></sup> use the syntax:
<dl>
<dd><code>__thread int number;</code></dd>
</dl>
</li>
<li><a href="/wiki/Visual_C%2B%2B" title="Visual C++">Visual C++</a>,<sup class="reference" id="cite_ref-7"><a href="#cite_note-7">[7]</a></sup> Intel C/C++ (Windows systems),<sup class="reference" id="cite_ref-8"><a href="#cite_note-8">[8]</a></sup> <a href="/wiki/C%2B%2BBuilder" title="C++Builder">C++Builder</a>, and Digital Mars C++ use the syntax:
<dl>
<dd><code>__declspec(thread) int number;</code></dd>
</dl>
</li>
<li>C++Builder also supports the syntax:
<dl>
<dd><code>int __thread number;</code></dd>
</dl>
</li>
</ul>
<p>On Windows versions before Vista and Server 2008, <code>__declspec(thread)</code> works in DLLs only when those DLLs are bound to the executable, and will <i>not</i> work for those loaded with <i>LoadLibrary()</i> (a protection fault or data corruption may occur).<sup class="reference" id="cite_ref-9"><a href="#cite_note-9">[9]</a></sup></p>
<h3><span class="mw-headline" id="Common_Lisp_.28and_maybe_other_dialects.29">Common Lisp (and maybe other dialects)</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Thread-local_storage&amp;action=edit&amp;section=5" title="Edit section: Common Lisp (and maybe other dialects)">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p><a href="/wiki/Common_Lisp" title="Common Lisp">Common Lisp</a> provides a feature called <a href="/wiki/Scope_(computer_science)#Dynamic_scoping" title="Scope (computer science)">dynamically scoped</a> variables.</p>
<p>Dynamic variables have a binding which is private to the invocation of a function and all of the children called by that function.</p>
<p>This abstraction naturally maps to thread-specific storage, and Lisp implementations that provide threads do this. Common Lisp has numerous standard dynamic variables, and so threads cannot be sensibly added to an implementation of the language without these variables having thread-local semantics in dynamic binding.</p>
<p>For instance the standard variable <code>*print-base*</code> determines the default radix in which integers are printed. If this variable is overridden, then all enclosing code will print integers in an alternate radix:</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="c1">;;; function foo and its children will print</span>
<span class="c1">;; in hexadecimal:</span>
<span class="p">(</span><span class="k">let</span> <span class="p">((</span><span class="vg">*print-base*</span> <span class="mi">16</span><span class="p">))</span> <span class="p">(</span><span class="nv">foo</span><span class="p">))</span>
</pre></div>
<p>If functions can execute concurrently on different threads, this binding has to be properly thread-local, otherwise each thread will fight over who controls a global printing radix.</p>
<h3><span class="mw-headline" id="D">D</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Thread-local_storage&amp;action=edit&amp;section=6" title="Edit section: D">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>In <a href="/wiki/D_(programming_language)" title="D (programming language)">D</a> version 2, all static and global variables are thread-local by default and are declared with syntax similar to "normal" global and static variables in other languages. Global variables must be explicitly requested using the <i>shared</i> keyword:</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="kt">int</span> <span class="n">threadLocal</span><span class="p">;</span>  <span class="c1">// This is a thread-local variable.</span>
<span class="k">shared</span> <span class="kt">int</span> <span class="n">global</span><span class="p">;</span>  <span class="c1">// This is a global variable shared with all threads.</span>
</pre></div>
<p>The <i>shared</i> keyword works both as the storage class, and as a <a href="/wiki/Type_qualifier" title="Type qualifier">type qualifier</a> – <i>shared</i> variables are subject to some restrictions which statically enforce data integrity.<sup class="reference" id="cite_ref-10"><a href="#cite_note-10">[10]</a></sup> To declare a "classic" global variable without these restrictions, the unsafe <i>__gshared</i> keyword must be used:<sup class="reference" id="cite_ref-11"><a href="#cite_note-11">[11]</a></sup></p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="k">__gshared</span> <span class="kt">int</span> <span class="n">global</span><span class="p">;</span>  <span class="c1">// This is a plain old global variable.</span>
</pre></div>
<h3><span class="mw-headline" id="Java">Java</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Thread-local_storage&amp;action=edit&amp;section=7" title="Edit section: Java">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>In <a href="/wiki/Java_(programming_language)" title="Java (programming language)">Java</a>, thread-local variables are implemented by the <code><a class="external text" href="//docs.oracle.com/javase/8/docs/api/java/lang/ThreadLocal.html" rel="nofollow">ThreadLocal</a></code> <a class="mw-redirect" href="/wiki/Class_(computer_science)" title="Class (computer science)">class</a> object. ThreadLocal holds variable of type T, which is accessible via get/set methods. For example ThreadLocal variable holding Integer value looks like this:</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="kd">private</span> <span class="kd">static</span> <span class="kd">final</span> <span class="n">ThreadLocal</span><span class="o">&lt;</span><span class="n">Integer</span><span class="o">&gt;</span> <span class="n">myThreadLocalInteger</span> <span class="o">=</span> <span class="k">new</span> <span class="n">ThreadLocal</span><span class="o">&lt;</span><span class="n">Integer</span><span class="o">&gt;();</span>
</pre></div>
<p>At least for Oracle/OpenJDK, this does not use native thread-local storage in spite of OS threads being used for other aspects of Java threading. Instead, each Thread object stores a (non-thread-safe) map of ThreadLocal objects to their values (as opposed to each ThreadLocal having a map of Thread objects to values and incurring a performance overhead).<sup class="reference" id="cite_ref-12"><a href="#cite_note-12">[12]</a></sup></p>
<h3><span class="mw-headline" id=".NET_languages:_C.23_and_others">.NET languages: C# and others</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Thread-local_storage&amp;action=edit&amp;section=8" title="Edit section: .NET languages: C# and others">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>In <a href="/wiki/.NET_Framework" title=".NET Framework">.NET Framework</a> languages such as <a href="/wiki/C_Sharp_(programming_language)" title="C Sharp (programming language)">C#</a>, static fields can be marked with the <a class="external text" href="http://msdn2.microsoft.com/en-us/library/system.threadstaticattribute(vs.80).aspx" rel="nofollow">ThreadStatic attribute</a>:</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="k">class</span> <span class="nc">FooBar</span> <span class="p">{</span>
<span class="na">   [ThreadStatic]</span> <span class="k">static</span> <span class="kt">int</span> <span class="n">foo</span><span class="p">;</span>
<span class="p">}</span>
</pre></div>
<p>In .NET 4.0 the <a class="external text" href="http://msdn.microsoft.com/en-us/library/dd642243.aspx" rel="nofollow">System.Threading.ThreadLocal&lt;T&gt;</a> class is available for allocating and lazily loading thread-local variables.</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="k">class</span> <span class="nc">FooBar</span> <span class="p">{</span>
   <span class="k">private</span> <span class="k">static</span> <span class="n">System</span><span class="p">.</span><span class="n">Threading</span><span class="p">.</span><span class="n">ThreadLocal</span><span class="p">&lt;</span><span class="kt">int</span><span class="p">&gt;</span> <span class="n">foo</span><span class="p">;</span>
<span class="p">}</span>
</pre></div>
<p>Also <a class="external text" href="http://msdn2.microsoft.com/en-us/library/system.threading.thread.getnameddataslot(vs.80).aspx" rel="nofollow">an API</a> is available for dynamically allocating thread-local variables.</p>
<h3><span class="mw-headline" id="Object_Pascal">Object Pascal</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Thread-local_storage&amp;action=edit&amp;section=9" title="Edit section: Object Pascal">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>In <a href="/wiki/Object_Pascal" title="Object Pascal">Object Pascal</a> (Delphi) or <a href="/wiki/Free_Pascal" title="Free Pascal">Free Pascal</a> the <i>threadvar</i> reserved keyword can be used instead of 'var' to declare variables using the thread-local storage.</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="k">var</span>
   <span class="n">mydata_process</span><span class="o">:</span> <span class="kt">integer</span><span class="o">;</span>
<span class="k">threadvar</span>
   <span class="n">mydata_threadlocal</span><span class="o">:</span> <span class="kt">integer</span><span class="o">;</span>
</pre></div>
<h3><span class="mw-headline" id="Objective-C">Objective-C</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Thread-local_storage&amp;action=edit&amp;section=10" title="Edit section: Objective-C">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>In <a href="/wiki/Cocoa_(API)" title="Cocoa (API)">Cocoa</a>, <a href="/wiki/GNUstep" title="GNUstep">GNUstep</a>, and <a href="/wiki/OpenStep" title="OpenStep">OpenStep</a>, each <tt>NSThread</tt> object has a thread-local dictionary that can be accessed through the thread's <tt>threadDictionary</tt> method.</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="bp">NSMutableDictionary</span> <span class="o">*</span><span class="n">dict</span> <span class="o">=</span> <span class="p">[[</span><span class="bp">NSThread</span> <span class="n">currentThread</span><span class="p">]</span> <span class="n">threadDictionary</span><span class="p">];</span>
<span class="n">dict</span><span class="p">[</span><span class="s">@"A key"</span><span class="p">]</span> <span class="o">=</span> <span class="s">@"Some data"</span><span class="p">;</span>
</pre></div>
<h3><span class="mw-headline" id="Perl">Perl</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Thread-local_storage&amp;action=edit&amp;section=11" title="Edit section: Perl">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>In <a href="/wiki/Perl" title="Perl">Perl</a> threads were added late in the evolution of the language, after a large body of extant code was already present on the <a class="mw-redirect" href="/wiki/Comprehensive_Perl_Archive_Network" title="Comprehensive Perl Archive Network">Comprehensive Perl Archive Network</a> (CPAN). Thus, threads in Perl by default take their own local storage for all variables, to minimise the impact of threads on extant non-thread-aware code. In Perl, a thread-shared variable can be created using an attribute:</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="k">use</span> <span class="n">threads</span><span class="p">;</span>
<span class="k">use</span> <span class="nn">threads::</span><span class="n">shared</span><span class="p">;</span>

<span class="k">my</span> <span class="nv">$localvar</span><span class="p">;</span>
<span class="k">my</span> <span class="nv">$sharedvar</span> <span class="p">:</span><span class="n">shared</span><span class="p">;</span>
</pre></div>
<h3><span class="mw-headline" id="Python">Python</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Thread-local_storage&amp;action=edit&amp;section=12" title="Edit section: Python">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>In <a href="/wiki/Python_(programming_language)" title="Python (programming language)">Python</a> version 2.4 or later, <i>local</i> class in <i>threading</i> module can be used to create thread-local storage.</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="kn">import</span> <span class="nn">threading</span>
<span class="n">mydata</span> <span class="o">=</span> <span class="n">threading</span><span class="o">.</span><span class="n">local</span><span class="p">()</span>
<span class="n">mydata</span><span class="o">.</span><span class="n">x</span> <span class="o">=</span> <span class="mi">1</span>
</pre></div>
<h3><span class="mw-headline" id="Ruby">Ruby</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Thread-local_storage&amp;action=edit&amp;section=13" title="Edit section: Ruby">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p><a href="/wiki/Ruby_(programming_language)" title="Ruby (programming language)">Ruby</a> can create/access thread-local variables using []=/[] methods:</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="no">Thread</span><span class="o">.</span><span class="n">current</span><span class="o">[</span><span class="ss">:user_id</span><span class="o">]</span> <span class="o">=</span> <span class="mi">1</span>
</pre></div>
