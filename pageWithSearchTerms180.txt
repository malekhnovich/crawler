<p>Em <a href="/wiki/Engenharia_de_software" title="Engenharia de software">engenharia de software</a>, um <span class="no-conversion"><b><span lang="pt-pt" xml:lang="pt-pt">padrão de desenho</span></b> <sup>(<a href="/wiki/Portugu%C3%AAs_europeu" title="Português europeu">português europeu</a>)</sup> ou <b><span lang="pt-br" xml:lang="pt-br">padrão de projeto</span></b> <sup>(<a href="/wiki/Portugu%C3%AAs_brasileiro" title="Português brasileiro">português brasileiro</a>)</sup></span> (do inglês <b>design pattern</b>) é uma solução geral para um problema que ocorre com frequência dentro de um determinado contexto no <a href="/wiki/Projeto_de_software" title="Projeto de software">projeto de software</a>. Um padrão de projeto não é um projeto finalizado que pode ser diretamente transformado em código <a class="mw-redirect" href="/wiki/C%C3%B3digo_fonte" title="Código fonte">fonte</a> ou de <a href="/wiki/C%C3%B3digo_de_m%C3%A1quina" title="Código de máquina">máquina</a>, ele é uma descrição ou modelo (<i>template</i>) de como resolver um problema que pode ser usado em muitas situações diferentes. Padrões são <a href="/wiki/Boas_pr%C3%A1ticas" title="Boas práticas">melhores práticas</a> formalizadas que o programador pode usar para resolver problemas comuns quando projetar uma aplicação ou sistema. Padrões de projeto <a class="mw-redirect" href="/wiki/Programa%C3%A7%C3%A3o_orientada_a_objetos" title="Programação orientada a objetos">orientados a objeto</a> normalmente mostram relacionamentos e <a href="/wiki/Intera%C3%A7%C3%A3o" title="Interação">interações</a> entre <a href="/wiki/Classe_(programa%C3%A7%C3%A3o)" title="Classe (programação)">classes</a> ou <a href="/wiki/Objeto_(ci%C3%AAncia_da_computa%C3%A7%C3%A3o)" title="Objeto (ciência da computação)">objetos</a>, sem especificar as classes ou objetos da aplicação final que estão envolvidas. Padrões que implicam <a class="mw-redirect" href="/wiki/Programa%C3%A7%C3%A3o_orientada_a_objetos" title="Programação orientada a objetos">orientação a objetos</a> ou estado mutável mais geral, não são tão aplicáveis em linguagens de <a href="/wiki/Programa%C3%A7%C3%A3o_funcional" title="Programação funcional">programação funcional</a>.</p>
<p>Padrões de projeto residem no domínio de módulos e interconexões. Em um nível mais alto há <a class="new" href="/w/index.php?title=Padr%C3%B5es_arquiteturais&amp;action=edit&amp;redlink=1" title="Padrões arquiteturais (página não existe)">padrões arquiteturais</a> que são maiores em escopo, usualmente descrevendo um padrão global seguido por um sistema inteiro.<sup class="reference" id="cite_ref-R.C.Martin_1-0"><a href="#cite_note-R.C.Martin-1"><span>[</span>1<span>]</span></a></sup></p>
<p>Um padrão de projeto define :</p>
<ul>
<li>seu nome,</li>
<li>o problema,</li>
<li>quando aplicar esta solução e</li>
<li>suas consequências.</li>
</ul>
<p>Os padrões de projeto :</p>
<ul>
<li>visam facilitar a reutilização de soluções de desenho - isto é, soluções na fase de projeto do software - e</li>
<li>estabelecem um vocabulário comum de desenho, facilitando comunicação, documentação e aprendizado dos sistemas de <i>software</i>.</li>
</ul>
<p></p>

<h2><span class="mw-headline" id="Hist.C3.B3ria">História</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a class="mw-editsection-visualeditor" href="/w/index.php?title=Padr%C3%A3o_de_projeto_de_software&amp;veaction=edit&amp;vesection=1" title="Editar secção: História">editar</a><span class="mw-editsection-divider"> | </span><a href="/w/index.php?title=Padr%C3%A3o_de_projeto_de_software&amp;action=edit&amp;section=1" title="Editar secção: História">editar código-fonte</a><span class="mw-editsection-bracket">]</span></span></h2>
<p><b>Christopher Alexander</b><sup class="reference" id="cite_ref-2"><a href="#cite_note-2"><span>[</span>2<span>]</span></a></sup><sup class="reference" id="cite_ref-3"><a href="#cite_note-3"><span>[</span>3<span>]</span></a></sup><sup class="reference" id="cite_ref-4"><a href="#cite_note-4"><span>[</span>4<span>]</span></a></sup> em seus livros (1977/1979) <i>Notes on the Synthesis of Form</i>, <i>The Timeless Way of Building</i> e <i>A Pattern Language</i>, estabelece que um padrão deve ter, idealmente, as seguintes características:</p>
<ul>
<li><b>Encapsulamento:</b> um padrão encapsula um problema ou solução bem definida. Ele deve ser independente, específico e formulado de maneira a ficar claro onde ele se aplica.</li>
<li><b>Generalidade:</b> todo padrão deve permitir a construção de outras realizações a partir deste padrão.</li>
<li><b>Equilíbrio:</b> quando um padrão é utilizado em uma aplicação, o equilíbrio dá a razão, relacionada com cada uma das restrições envolvidas, para cada passo do projeto. Uma análise racional que envolva uma abstração de dados empíricos, uma observação da aplicação de padrões em artefatos tradicionais, uma série convincente de exemplos e uma análise de soluções ruins ou fracassadas pode ser a forma de encontrar este equilíbrio.</li>
<li><b>Abstração:</b> os padrões representam abstrações da <a href="/wiki/Empirismo" title="Empirismo">experiência empírica</a> ou do conhecimento cotidiano.</li>
<li><b>Abertura:</b> um padrão deve permitir a sua extensão para níveis mais baixos de detalhe.</li>
<li><b>Combinatoriedade:</b> os padrões são relacionados hierarquicamente. Padrões de alto nível podem ser compostos ou relacionados com padrões que endereçam problemas de nível mais baixo.</li>
</ul>
<p>Além da definição das características de um padrão, Alexander definiu o formato que a descrição de um padrão deve ter. Ele estabeleceu que um padrão deve ser descrito em cinco partes:</p>
<ul>
<li><b>Nome:</b> uma descrição da solução, mais do que do problema ou do contexto.</li>
<li><b>Exemplo:</b> uma ou mais figuras, diagramas ou descrições que ilustrem um protótipo de aplicação.</li>
<li><b>Contexto:</b> a descrição das situações sob as quais o padrão se aplica.</li>
<li><b>Problema:</b> uma descrição das forças e restrições envolvidos e como elas interagem.</li>
<li><b>Solução:</b> relacionamentos estáticos e regras dinâmicas descrevendo como construir artefatos de acordo com o padrão, freqüentemente citando variações e formas de ajustar a solução segundo as circunstâncias. Inclui referências a outras soluções e o relacionamento com outros padrões de nível mais baixo ou mais alto.</li>
</ul>
<p>Em <a href="/wiki/1987" title="1987">1987</a>, a partir dos conceitos criados por Alexander, os <a href="/wiki/Programador" title="Programador">programadores</a> <a href="/wiki/Kent_Beck" title="Kent Beck">Kent Beck</a> e <a href="/wiki/Ward_Cunningham" title="Ward Cunningham">Ward Cunningham</a> propuseram os primeiros padrões de projeto para a área da <a href="/wiki/Ci%C3%AAncia_da_computa%C3%A7%C3%A3o" title="Ciência da computação">ciência da computação</a>, os patterns de Alexander procuravam prover uma fonte de ideias provadas para indivíduos e comunidades para serem usadas em construções, mostrando assim o quanto belo, confortável e flexível os ambientes podem ser construídos. Em um trabalho para a conferência <a class="new" href="/w/index.php?title=OOPSLA&amp;action=edit&amp;redlink=1" title="OOPSLA (página não existe)">OOPSLA</a>, eles apresentaram alguns padrões para a construção de janelas na <a href="/wiki/Smalltalk" title="Smalltalk">linguagem Smalltalk</a>.<sup class="reference" id="cite_ref-5"><a href="#cite_note-5"><span>[</span>5<span>]</span></a></sup> Nos anos seguintes Beck, Cunningham e outros seguiram com o desenvolvimento destas ideias.</p>
<p>O movimento ao redor de padrões de projeto ganhou popularidade com o livro <i><a class="new" href="/w/index.php?title=Design_Patterns:_Elements_of_Reusable_Object-Oriented_Software&amp;action=edit&amp;redlink=1" title="Design Patterns: Elements of Reusable Object-Oriented Software (página não existe)">Design Patterns: Elements of Reusable Object-Oriented Software</a></i>, publicado em <a href="/wiki/1995" title="1995">1995</a>. Os autores desse livro, <a href="/wiki/Erich_Gamma" title="Erich Gamma">Erich Gamma</a>, <a class="mw-redirect" href="/wiki/Richard_Helm" title="Richard Helm">Richard Helm</a>, <a href="/wiki/Ralph_Johnson" title="Ralph Johnson">Ralph Johnson</a> e <a href="/wiki/John_Vlissides" title="John Vlissides">John Vlissides</a>, são conhecidos como a "<a class="mw-disambig" href="/wiki/Gangue_dos_quatro" title="Gangue dos quatro">Gangue dos Quatro</a>" (Gang of Four) ou simplesmente "<b>GoF</b>".</p>
<p>Posteriormente, vários outros livros do estilo foram publicados, merecendo destaque <i>Applying UML and Patterns: An Introduction to Object-Oriented Analysis and Design and Iterative Development</i>, que introduziu um conjunto de padrões conhecidos como <b>GRASP</b> (<i>General Responsibility Assignment Software Patterns</i>).</p>
<h2><span class="mw-headline" id="Padr.C3.B5es_GoF_.28.27Gang_of_Four.27.29.3B">Padrões GoF ('Gang of Four');</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a class="mw-editsection-visualeditor" href="/w/index.php?title=Padr%C3%A3o_de_projeto_de_software&amp;veaction=edit&amp;vesection=2" title="Editar secção: Padrões GoF ('Gang of Four');">editar</a><span class="mw-editsection-divider"> | </span><a href="/w/index.php?title=Padr%C3%A3o_de_projeto_de_software&amp;action=edit&amp;section=2" title="Editar secção: Padrões GoF ('Gang of Four');">editar código-fonte</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Os padrões "GoF" são organizados em 3 famílias :</p>
<ul>
<li><b>Padrões de criação</b> : relacionados à criação de objetos</li>
<li><b>Padrões estruturais</b> : tratam das associações entre classes e objetos.</li>
<li><b>Padrões comportamentais</b> : tratam das interações e divisões de responsabilidades entre as classes ou objetos.</li>
</ul>

<p>Um padrão "GoF" também é classificado segundo o seu escopo em 2 outros grupos :</p>
<ul>
<li><b>Padrões com escopo de classe</b> : definido por relacionamentos de <a href="/wiki/Heran%C3%A7a" title="Herança">herança</a> e em <a href="/wiki/Tempo_de_compila%C3%A7%C3%A3o" title="Tempo de compilação">tempo de compilação</a>.</li>

<h2><span class="mw-headline" id="Padr.C3.B5es_GRASP">Padrões GRASP</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a class="mw-editsection-visualeditor" href="/w/index.php?title=Padr%C3%A3o_de_projeto_de_software&amp;veaction=edit&amp;vesection=6" title="Editar secção: Padrões GRASP">editar</a><span class="mw-editsection-divider"> | </span><a href="/w/index.php?title=Padr%C3%A3o_de_projeto_de_software&amp;action=edit&amp;section=6" title="Editar secção: Padrões GRASP">editar código-fonte</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Os padrões GRASP, sigla para <b>General Responsibility Assignment Software Patterns</b> (or <b>Principles</b>), consistem de um conjunto de práticas para atribuição de responsabilidades a classes e objetos em projetos orientados a objeto.</p>
<p>Os padrões utilizados pelo GRASP são: Controlador (Controller), Criador (Creator), Indireção (Indirection), Especialista na informação (Information expert), Alta coesão (High Cohesion), Baixo acoplamento (Loose coupling), Polimorfismo (Polymorphism), Variações protegidas (Protected variations), e Invenção pura (Pure fabrication).</p>
<p>Todos esses padrões servem para a resolução de problemas comuns e bastante típicos de desenvolvimento de software orientado a objeto. Portanto, tais técnicas apenas documentam e normatizam as práticas já consolidadas, testadas e conhecidas no mercado.</p>
<p>Os padrões GRASP estão mais como uma ferramenta mental ou uma filosofia de design, mas que ainda assim são úteis para o aprendizado e desenvolvimento de um bom design de software. Note que alguns padrões GoF implementam soluções correspondentes com padrões GRASP.</p>
<p>A principal obra sobre o assunto foi o livro "Utilizando UML e Padrões" de autoria de Craig Larman.<sup class="reference" id="cite_ref-6"><a href="#cite_note-6"><span>[</span>6<span>]</span></a></sup></p>

<h2><span class="mw-headline" id="Cr.C3.ADticas">Críticas</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a class="mw-editsection-visualeditor" href="/w/index.php?title=Padr%C3%A3o_de_projeto_de_software&amp;veaction=edit&amp;vesection=8" title="Editar secção: Críticas">editar</a><span class="mw-editsection-divider"> | </span><a href="/w/index.php?title=Padr%C3%A3o_de_projeto_de_software&amp;action=edit&amp;section=8" title="Editar secção: Críticas">editar código-fonte</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Segundo alguns usuários, alguns 'Padrões de Projeto' são apenas evidências de que alguns recursos estão ausentes em uma determinada linguagem de programação (<a href="/wiki/Java_(linguagem_de_programa%C3%A7%C3%A3o)" title="Java (linguagem de programação)">Java</a> ou <a href="/wiki/C%2B%2B" title="C++">C++</a> por exemplo). Nesta linha, Peter Norvig demonstra que 16 dos 23 padrões do livro 'Design Patterns' são simplificados ou eliminados nas linguagens Lisp ou Dylan, usando os recursos diretos destas linguagens.<sup class="reference" id="cite_ref-Norvig1998_7-0"><a href="#cite_note-Norvig1998-7"><span>[</span>7<span>]</span></a></sup></p>
<p>Segundo outros, excessos nas tentativas de fazer o código se conformar aos 'Padrões de Projeto' aumentam desnecessariamente a sua complexidade.<sup class="reference" id="cite_ref-CodeComplete2_8-0"><a href="#cite_note-CodeComplete2-8"><span>[</span>8<span>]</span></a></sup></p>
