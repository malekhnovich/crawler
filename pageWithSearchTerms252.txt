<p>In <a href="/wiki/Software_engineering" title="Software engineering">software engineering</a>, <b>structured analysis</b> (SA) and its allied technique, <b>structured design</b> (SD), are methods for analyzing business <a class="mw-redirect" href="/wiki/Requirements" title="Requirements">requirements</a> and developing <a class="mw-redirect" href="/wiki/Specification" title="Specification">specifications</a> for converting practices into <a href="/wiki/Computer_program" title="Computer program">computer programs</a>, hardware configurations, and related manual procedures.</p>
<p>Structured analysis and design techniques are fundamental tools of <a href="/wiki/Systems_analysis" title="Systems analysis">systems analysis</a>. They developed from classical systems analysis of the 1960s and 1970s.<sup class="reference" id="cite_ref-2"><a href="#cite_note-2">[2]</a></sup></p>

<h2><span class="mw-headline" id="Objectives_of_structured_analysis">Objectives of structured analysis</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Structured_analysis&amp;action=edit&amp;section=1" title="Edit section: Objectives of structured analysis">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Structured analysis became popular in the 1980s and is still used today.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (March 2016)">citation needed</span></a></i>]</sup> Structured analysis consists of interpreting the <a href="/wiki/System" title="System">system</a> concept (or real world situations) into data and control terminology represented by <a href="/wiki/Data_flow_diagram" title="Data flow diagram">data flow diagrams</a>. The flow of data and control from bubble to the data store to bubble can be difficult to track and the number of bubbles can increase.</p>
<p>One approach is to first define events from the outside world that require the system to react, then assign a bubble to that event. Bubbles that need to interact are then connected until the system is defined. Bubbles are usually grouped into higher level bubbles to decrease complexity. <a href="/wiki/Data_dictionary" title="Data dictionary">Data dictionaries</a> are needed to describe the data and command flows, and a process specification is needed to capture the transaction/transformation information.<sup class="reference" id="cite_ref-FAA00_3-0"><a href="#cite_note-FAA00-3">[3]</a></sup></p>
<p>SA and SD were accompanied by notation methods including <a href="/wiki/Structure_chart" title="Structure chart">structure charts</a>, <a href="/wiki/Data_flow_diagram" title="Data flow diagram">data flow diagrams</a> and <a class="mw-redirect" href="/wiki/Data_model_diagram" title="Data model diagram">data model diagrams</a>, of which there were many variations, including those developed by <a href="/wiki/Tom_DeMarco" title="Tom DeMarco">Tom DeMarco</a>, <a href="/wiki/Ken_Orr" title="Ken Orr">Ken Orr</a>, <a href="/wiki/Larry_Constantine" title="Larry Constantine">Larry Constantine</a>, <a href="/wiki/Vaughn_Frick" title="Vaughn Frick">Vaughn Frick</a>, <a class="mw-redirect" href="/wiki/Ed_Yourdon" title="Ed Yourdon">Ed Yourdon</a>, Steven Ward, <a href="/wiki/Peter_Chen" title="Peter Chen">Peter Chen</a>, and others.</p>
<p>These techniques were combined in various published <a href="/wiki/Software_development_process" title="Software development process">system development methodologies</a>, including <a href="/wiki/Structured_systems_analysis_and_design_method" title="Structured systems analysis and design method">structured systems analysis and design method</a>, profitable information by design (PRIDE), Nastec structured analysis &amp; design, SDM/70 and the Spectrum structured system development methodology.</p>
<h2><span class="mw-headline" id="History">History</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Structured_analysis&amp;action=edit&amp;section=2" title="Edit section: History">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Structured analysis is part of a series of structured methods that "represent a collection of analysis, design, and programming techniques that were developed in response to the problems facing the software world from the 1960s to the 1980s. In this timeframe most commercial programming was done in <a class="mw-redirect" href="/wiki/Cobol" title="Cobol">Cobol</a> and <a href="/wiki/Fortran" title="Fortran">Fortran</a>, then <a href="/wiki/C_(programming_language)" title="C (programming language)">C</a> and <a href="/wiki/BASIC" title="BASIC">BASIC</a>. There was little guidance on "good" design and programming techniques, and there were no standard techniques for documenting requirements and designs. Systems were getting larger and more complex, and the information system development became harder and harder to do so."<sup class="reference" id="cite_ref-DL00_4-0"><a href="#cite_note-DL00-4">[4]</a></sup></p>
<p>As a way to help manage large and complex software, the following structured methods emerged since the end of the 1960s :<sup class="reference" id="cite_ref-DL00_4-1"><a href="#cite_note-DL00-4">[4]</a></sup></p>
<ul>
<li><a href="/wiki/Structured_programming" title="Structured programming">Structured programming</a> in circa 1967 with <a class="mw-redirect" href="/wiki/Edsger_Dijkstra" title="Edsger Dijkstra">Edsger Dijkstra</a> - "Go To Statement Considered Harmful"</li>
<li><a href="/wiki/Niklaus_Wirth" title="Niklaus Wirth">Niklaus Wirth</a> Stepwise design in 1971</li>
<li><a href="/wiki/Nassi%E2%80%93Shneiderman_diagram" title="Nassi–Shneiderman diagram">Nassi–Shneiderman diagram</a> in 1972</li>
<li><a href="/wiki/Warnier/Orr_diagram" title="Warnier/Orr diagram">Warnier/Orr diagram</a> in 1974 - "Logical Construction of Programs"</li>
<li><a class="mw-redirect" href="/wiki/HIPO" title="HIPO">HIPO</a> in 1974 - IBM Hierarchy input-process-output (though this should really be output-input-process)</li>
<li><a class="mw-redirect" href="/wiki/Structured_design" title="Structured design">Structured design</a> around 1975 with <a href="/wiki/Larry_Constantine" title="Larry Constantine">Larry Constantine</a>, <a class="mw-redirect" href="/wiki/Ed_Yourdon" title="Ed Yourdon">Ed Yourdon</a> and <a href="/wiki/Wayne_Stevens_(software_engineer)" title="Wayne Stevens (software engineer)">Wayne Stevens</a>.<sup class="reference" id="cite_ref-FOOTNOTEStevensMyersConstantine1974_5-0"><a href="#cite_note-FOOTNOTEStevensMyersConstantine1974-5">[5]</a></sup><sup class="reference" id="cite_ref-FOOTNOTEYourdonConstantine1979_6-0"><a href="#cite_note-FOOTNOTEYourdonConstantine1979-6">[6]</a></sup></li>
<li><a href="/wiki/Jackson_structured_programming" title="Jackson structured programming">Jackson structured programming</a> in circa 1975 developed by <a href="/wiki/Michael_A._Jackson" title="Michael A. Jackson">Michael A. Jackson</a></li>
<li>Structured analysis in circa 1978 with <a href="/wiki/Tom_DeMarco" title="Tom DeMarco">Tom DeMarco</a>, Yourdon, Gane &amp; Sarson, McMenamin &amp; Palmer.</li>
<li><a class="mw-redirect" href="/wiki/Structured_Analysis_and_Design_Technique" title="Structured Analysis and Design Technique">Structured analysis and design technique</a> (SADT) developed by <a href="/wiki/Douglas_T._Ross" title="Douglas T. Ross">Douglas T. Ross</a></li>
<li><a href="/wiki/Edward_Yourdon#Yourdon_Structured_Method" title="Edward Yourdon">Yourdon structured method</a> developed by <a href="/wiki/Edward_Yourdon" title="Edward Yourdon">Edward Yourdon</a>.</li>
<li>Structured analysis and system specification published in 1979 by <a href="/wiki/Tom_DeMarco" title="Tom DeMarco">Tom DeMarco</a>.</li>
<li><a href="/wiki/Structured_systems_analysis_and_design_method" title="Structured systems analysis and design method">Structured systems analysis and design method</a> (SSADM) first presented in 1983 developed by the <a class="mw-redirect" href="/wiki/UK_government" title="UK government">UK</a> <a href="/wiki/Office_of_Government_Commerce" title="Office of Government Commerce">Office of Government Commerce</a>.</li>
<li><a href="/wiki/IDEF0" title="IDEF0">IDEF0</a> based on SADT, developed by <a href="/wiki/Douglas_T._Ross" title="Douglas T. Ross">Douglas T. Ross</a> in 1985.<sup class="reference" id="cite_ref-7"><a href="#cite_note-7">[7]</a></sup></li>
<li><a class="mw-redirect" href="/wiki/Hatley-Pirbhai_modeling" title="Hatley-Pirbhai modeling">Hatley-Pirbhai modeling</a>, defined in "Strategies for Real-Time System Specification" by Derek J. Hatley and Imtiaz A. Pirbhai in 1988.</li>
<li><a href="/wiki/Information_engineering" title="Information engineering">Information engineering</a> in circa 1990 with Finkelstein and popularised by <a href="/wiki/James_Martin_(author)" title="James Martin (author)">James Martin</a>.</li>
</ul>
<p>According to Hay(1999) "<a href="/wiki/Information_engineering" title="Information engineering">information engineering</a> was a logical extension of the structured techniques that were developed during the 1970's. Structured programming led to structured design, which in turn led to structured systems analysis. These techniques were characterized by their use of <a href="/wiki/Diagram" title="Diagram">diagrams</a>: structure charts for structured design, and data flow diagrams for structured analysis, both to aid in communication between users and developers, and to improve the analyst's and the designer's discipline. During the 1980's, tools began to appear which both automated the drawing of the diagrams, and kept track of the things drawn in a <a href="/wiki/Data_dictionary" title="Data dictionary">data dictionary</a>".<sup class="reference" id="cite_ref-8"><a href="#cite_note-8">[8]</a></sup> After the example of <a href="/wiki/Computer-aided_design" title="Computer-aided design">computer-aided design</a> and <a href="/wiki/Computer-aided_manufacturing" title="Computer-aided manufacturing">computer-aided manufacturing</a> (CAD/CAM), the use of these tools was named <a href="/wiki/Computer-aided_software_engineering" title="Computer-aided software engineering">computer-aided software engineering</a> (CASE).</p>
<h2><span class="mw-headline" id="Structured_analysis_topics">Structured analysis topics</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Structured_analysis&amp;action=edit&amp;section=3" title="Edit section: Structured analysis topics">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<h3><span class="mw-headline" id="Single_abstraction_mechanism">Single abstraction mechanism</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Structured_analysis&amp;action=edit&amp;section=4" title="Edit section: Single abstraction mechanism">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="thumb tright">
<div class="thumbinner" style="width:322px;"><a class="image" href="/wiki/File:Structured_Analysis_example.jpg"><img alt="" class="thumbimage" data-file-height="587" data-file-width="937" height="200" src="//upload.wikimedia.org/wikipedia/commons/thumb/5/5e/Structured_Analysis_example.jpg/320px-Structured_Analysis_example.jpg" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/5/5e/Structured_Analysis_example.jpg/480px-Structured_Analysis_example.jpg 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/5/5e/Structured_Analysis_example.jpg/640px-Structured_Analysis_example.jpg 2x" width="320"/></a>
<div class="thumbcaption">
<div class="magnify"><a class="internal" href="/wiki/File:Structured_Analysis_example.jpg" title="Enlarge"></a></div>
Structured analysis example.<sup class="reference" id="cite_ref-DoDAF_V2_9-0"><a href="#cite_note-DoDAF_V2-9">[9]</a></sup></div>
</div>
</div>
<p>Structured analysis typically creates a hierarchy employing a single abstraction mechanism. The structured analysis method can employ <a href="/wiki/IDEF" title="IDEF">IDEF</a> (see figure), is process driven, and starts with a purpose and a viewpoint. This method identifies the overall function and iteratively divides functions into smaller functions, preserving inputs, outputs, controls, and mechanisms necessary to optimize processes. Also known as a <a href="/wiki/Functional_decomposition" title="Functional decomposition">functional decomposition</a> approach, it focuses on cohesion within functions and coupling between functions leading to structured data.<sup class="reference" id="cite_ref-DoDAF_V2_9-1"><a href="#cite_note-DoDAF_V2-9">[9]</a></sup></p>
<p>The functional decomposition of the structured method describes the process without delineating system behavior and dictates system structure in the form of required functions. The method identifies inputs and outputs as related to the activities. One reason for the popularity of structured analysis is its intuitive ability to communicate high-level processes and concepts, whether in single system or enterprise levels. Discovering how objects might support functions for commercially prevalent <a class="mw-redirect" href="/wiki/Object-oriented" title="Object-oriented">object-oriented</a> development is unclear. In contrast to IDEF, the <a href="/wiki/Unified_Modeling_Language" title="Unified Modeling Language">UML</a> is interface driven with multiple abstraction mechanisms useful in describing <a href="/wiki/Service-orientation" title="Service-orientation">service-oriented</a> architectures (SOAs).<sup class="reference" id="cite_ref-DoDAF_V2_9-2"><a href="#cite_note-DoDAF_V2-9">[9]</a></sup></p>
<h3><span class="mw-headline" id="Approach">Approach</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Structured_analysis&amp;action=edit&amp;section=5" title="Edit section: Approach">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Structured Analysis views a system from the perspective of the data flowing through it. The function of the system is described by processes that transform the data flows. Structured analysis takes advantage of information hiding through successive decomposition (or top down) analysis. This allows attention to be focused on pertinent details and avoids confusion from looking at irrelevant details. As the level of detail increases, the breadth of information is reduced. The result of structured analysis is a set of related graphical diagrams, process descriptions, and data definitions. They describe the transformations that need to take place and the data required to meet a system's <a href="/wiki/Functional_requirement" title="Functional requirement">functional requirements</a>.<sup class="reference" id="cite_ref-HeSi86_10-0"><a href="#cite_note-HeSi86-10">[10]</a></sup></p>
<div class="thumb tright">
<div class="thumbinner" style="width:322px;"><a class="image" href="/wiki/File:Analysis_Model_Objects.jpg"><img alt="" class="thumbimage" data-file-height="613" data-file-width="1094" height="179" src="//upload.wikimedia.org/wikipedia/commons/thumb/d/db/Analysis_Model_Objects.jpg/320px-Analysis_Model_Objects.jpg" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/d/db/Analysis_Model_Objects.jpg/480px-Analysis_Model_Objects.jpg 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/d/db/Analysis_Model_Objects.jpg/640px-Analysis_Model_Objects.jpg 2x" width="320"/></a>
<div class="thumbcaption">
<div class="magnify"><a class="internal" href="/wiki/File:Analysis_Model_Objects.jpg" title="Enlarge"></a></div>
The structured analyse approach develops perspectives on both process objects and data objects.<sup class="reference" id="cite_ref-HeSi86_10-1"><a href="#cite_note-HeSi86-10">[10]</a></sup></div>
</div>
</div>
<p>De Marco's approach<sup class="reference" id="cite_ref-11"><a href="#cite_note-11">[11]</a></sup> consists of the following objects (see figure):<sup class="reference" id="cite_ref-HeSi86_10-2"><a href="#cite_note-HeSi86-10">[10]</a></sup></p>
<ul>
<li><a class="mw-redirect" href="/wiki/Context_diagram" title="Context diagram">Context diagram</a></li>
<li><a class="mw-redirect" href="/wiki/Dataflow_diagram" title="Dataflow diagram">Data flow diagram</a></li>
<li>Process specifications</li>
<li><a href="/wiki/Data_dictionary" title="Data dictionary">Data dictionary</a></li>
</ul>
<p>Hereby the <a href="/wiki/Data_flow_diagram" title="Data flow diagram">data flow diagrams</a> (DFDs) are directed graphs. The arcs represent <a href="/wiki/Data" title="Data">data</a>, and the nodes (circles or bubbles) represent processes that transform the data. A process can be further decomposed to a more detailed DFD which shows the subprocesses and data flows within it. The subprocesses can in turn be decomposed further with another set of DFDs until their functions can be easily understood. Functional primitives are processes which do not need to be decomposed further. Functional primitives are described by a process specification (or mini-spec). The process specification can consist of pseudo-code, <a href="/wiki/Flowchart" title="Flowchart">flowcharts</a>, or structured English. The DFDs model the structure of the system as a network of interconnected processes composed of functional primitives. The <a href="/wiki/Data_dictionary" title="Data dictionary">data dictionary</a> is a set of entries (definitions) of data flows, data elements, files, and databases. The data dictionary entries are partitioned in a top-down manner. They can be referenced in other data dictionary entries and in data flow diagrams.<sup class="reference" id="cite_ref-HeSi86_10-3"><a href="#cite_note-HeSi86-10">[10]</a></sup></p>
<h3><span class="mw-headline" id="Context_diagram">Context diagram</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Structured_analysis&amp;action=edit&amp;section=6" title="Edit section: Context diagram">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="thumb tright">
<div class="thumbinner" style="width:322px;"><a class="image" href="/wiki/File:NDE_Context_Diagram_(vector).svg"><img alt="" class="thumbimage" data-file-height="420" data-file-width="620" height="217" src="//upload.wikimedia.org/wikipedia/commons/thumb/b/bd/NDE_Context_Diagram_%28vector%29.svg/320px-NDE_Context_Diagram_%28vector%29.svg.png" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/b/bd/NDE_Context_Diagram_%28vector%29.svg/480px-NDE_Context_Diagram_%28vector%29.svg.png 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/b/bd/NDE_Context_Diagram_%28vector%29.svg/640px-NDE_Context_Diagram_%28vector%29.svg.png 2x" width="320"/></a>
<div class="thumbcaption">
<div class="magnify"><a class="internal" href="/wiki/File:NDE_Context_Diagram_(vector).svg" title="Enlarge"></a></div>
Example of a system context diagram.<sup class="reference" id="cite_ref-12"><a href="#cite_note-12">[12]</a></sup></div>
</div>
</div>
<p><a class="mw-redirect" href="/wiki/System_Context_Diagram" title="System Context Diagram">Context diagrams</a> are diagrams that represent the actors outside a system that could interact with that system.<sup class="reference" id="cite_ref-KoSw03_13-0"><a href="#cite_note-KoSw03-13">[13]</a></sup> This diagram is the highest level view of a <a href="/wiki/System" title="System">system</a>, similar to <a href="/wiki/Block_diagram" title="Block diagram">block diagram</a>, showing a, possibly <a href="/wiki/Software_system" title="Software system">software</a>-based, system as a whole and its inputs and outputs from/to external factors.</p>
<p>This type of diagram according to Kossiakoff (2003) usually "pictures the system at the center, with no details of its interior structure, surrounded by all its interacting systems, environment and activities. The objective of a system context diagram is to focus attention on external factors and events that should be considered in developing a complete set of system requirements and constraints".<sup class="reference" id="cite_ref-KoSw03_13-1"><a href="#cite_note-KoSw03-13">[13]</a></sup> System context diagrams are related to <a href="/wiki/Data_flow_diagram" title="Data flow diagram">data flow diagram</a>, and show the interactions between a system and other actors which the system is designed to face. System context diagrams can be helpful in understanding the context in which the system will be part of <a href="/wiki/Software_engineering" title="Software engineering">software engineering</a>.</p>
<h3><span class="mw-headline" id="Data_dictionary">Data dictionary</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Structured_analysis&amp;action=edit&amp;section=7" title="Edit section: Data dictionary">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="thumb tleft">
<div class="thumbinner" style="width:322px;"><a class="image" href="/wiki/File:Entity_Relationship_Diagram.jpg"><img alt="" class="thumbimage" data-file-height="336" data-file-width="597" height="180" src="//upload.wikimedia.org/wikipedia/commons/thumb/1/16/Entity_Relationship_Diagram.jpg/320px-Entity_Relationship_Diagram.jpg" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/1/16/Entity_Relationship_Diagram.jpg/480px-Entity_Relationship_Diagram.jpg 1.5x, //upload.wikimedia.org/wikipedia/commons/1/16/Entity_Relationship_Diagram.jpg 2x" width="320"/></a>
<div class="thumbcaption">
<div class="magnify"><a class="internal" href="/wiki/File:Entity_Relationship_Diagram.jpg" title="Enlarge"></a></div>
<a class="mw-redirect" href="/wiki/Entity_relationship_diagram" title="Entity relationship diagram">Entity relationship diagram</a>, essential for the design of database tables, extracts, and metadata.<sup class="reference" id="cite_ref-USDT01_14-0"><a href="#cite_note-USDT01-14">[14]</a></sup></div>
</div>
</div>
<p>A <a href="/wiki/Data_dictionary" title="Data dictionary">data dictionary</a> or <i>database dictionary</i> is a file that defines the basic organization of a <a href="/wiki/Database" title="Database">database</a>.<sup class="reference" id="cite_ref-USDT01_14-1"><a href="#cite_note-USDT01-14">[14]</a></sup> A database dictionary contains a list of all files in the database, the number of records in each file, and the names and types of each data field. Most <a class="mw-redirect" href="/wiki/Database_management_system" title="Database management system">database management systems</a> keep the data dictionary hidden from users to prevent them from accidentally destroying its contents. Data dictionaries do not contain any actual data from the database, only bookkeeping information for managing it. Without a data dictionary, however, a database management system cannot access data from the database.<sup class="reference" id="cite_ref-USDT01_14-2"><a href="#cite_note-USDT01-14">[14]</a></sup></p>
<p>Database users and <a href="/wiki/Application_software" title="Application software">application</a> developers can benefit from an authoritative data dictionary document that catalogs the organization, contents, and conventions of one or more databases.<sup class="reference" id="cite_ref-15"><a href="#cite_note-15">[15]</a></sup> This typically includes the names and descriptions of various <a href="/wiki/Table_(database)" title="Table (database)">tables</a> and <a href="/wiki/Column_(database)" title="Column (database)">fields</a> in each database, plus additional details, like the <a href="/wiki/Data_type" title="Data type">type</a> and length of each <a href="/wiki/Data_element" title="Data element">data element</a>. There is no universal standard as to the level of detail in such a document, but it is primarily a distillation of <a href="/wiki/Metadata" title="Metadata">metadata</a> about <a href="/wiki/Database_design" title="Database design">database structure</a>, not the data itself. A data dictionary document also may include further information describing how data elements are encoded. One of the advantages of well-designed data dictionary documentation is that it helps to establish consistency throughout a complex database, or across a large collection of <a class="mw-redirect" href="/wiki/Federated_database" title="Federated database">federated databases</a>.<sup class="reference" id="cite_ref-16"><a href="#cite_note-16">[16]</a></sup></p>
<h3><span class="mw-headline" id="Data_flow_diagrams">Data flow diagrams</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Structured_analysis&amp;action=edit&amp;section=8" title="Edit section: Data flow diagrams">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="thumb tright">
<div class="thumbinner" style="width:332px;"><a class="image" href="/wiki/File:Data_Flow_Diagram_Example.jpg"><img alt="" class="thumbimage" data-file-height="640" data-file-width="1169" height="181" src="//upload.wikimedia.org/wikipedia/commons/thumb/0/0f/Data_Flow_Diagram_Example.jpg/330px-Data_Flow_Diagram_Example.jpg" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/0/0f/Data_Flow_Diagram_Example.jpg/495px-Data_Flow_Diagram_Example.jpg 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/0/0f/Data_Flow_Diagram_Example.jpg/660px-Data_Flow_Diagram_Example.jpg 2x" width="330"/></a>
<div class="thumbcaption">
<div class="magnify"><a class="internal" href="/wiki/File:Data_Flow_Diagram_Example.jpg" title="Enlarge"></a></div>
Data flow diagram example.<sup class="reference" id="cite_ref-17"><a href="#cite_note-17">[17]</a></sup></div>
</div>
</div>
<p>A <a href="/wiki/Data_flow_diagram" title="Data flow diagram">data flow diagram</a> (DFD) is a graphical representation of the "flow" of data through an <a href="/wiki/Information_system" title="Information system">information system</a>. It differs from the system <a href="/wiki/Flowchart" title="Flowchart">flowchart</a> as it shows the flow of data through processes instead of <a href="/wiki/Computer_hardware" title="Computer hardware">computer hardware</a>. Data flow diagrams were invented by <a href="/wiki/Larry_Constantine" title="Larry Constantine">Larry Constantine</a>, developer of <a class="mw-redirect" href="/wiki/Structured_design" title="Structured design">structured design</a>, based on Martin and Estrin's "data flow graph" model of computation.<sup class="reference" id="cite_ref-18"><a href="#cite_note-18">[18]</a></sup></p>
<p>It is common practice to draw a <a href="/wiki/System_context_diagram" title="System context diagram">system context diagram</a> first which shows the interaction between the system and outside entities. The DFD is designed to show how a system is divided into smaller portions and to highlight the flow of data between those parts. This context-level data flow diagram is then "exploded" to show more detail of the system being modeled.</p>
<p>Data flow diagrams (DFDs) are one of the three essential perspectives of <a href="/wiki/Structured_systems_analysis_and_design_method" title="Structured systems analysis and design method">structured systems analysis and design method</a> (SSADM). The sponsor of a project and the end users will need to be briefed and consulted throughout all stages of a system's evolution. With a data flow diagram, users are able to visualize how the system will operate, what the system will accomplish, and how the system will be implemented. The old system's data flow diagrams can be drawn up and compared with the new system's data flow diagrams to draw comparisons to implement a more efficient system. Data flow diagrams can be used to provide the end user with a physical idea of where the data they input ultimately has an effect upon the structure of the whole system from order to dispatch to recook. How any system is developed can be determined through a data flow diagram.</p>
<h3><span class="mw-headline" id="Structure_chart">Structure chart</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Structured_analysis&amp;action=edit&amp;section=9" title="Edit section: Structure chart">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="thumb tright">
<div class="thumbinner" style="width:322px;"><a class="image" href="/wiki/File:Configuration_System_Structure_Chart.jpg"><img alt="" class="thumbimage" data-file-height="487" data-file-width="1042" height="150" src="//upload.wikimedia.org/wikipedia/commons/thumb/9/92/Configuration_System_Structure_Chart.jpg/320px-Configuration_System_Structure_Chart.jpg" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/9/92/Configuration_System_Structure_Chart.jpg/480px-Configuration_System_Structure_Chart.jpg 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/9/92/Configuration_System_Structure_Chart.jpg/640px-Configuration_System_Structure_Chart.jpg 2x" width="320"/></a>
<div class="thumbcaption">
<div class="magnify"><a class="internal" href="/wiki/File:Configuration_System_Structure_Chart.jpg" title="Enlarge"></a></div>
A configuration system structure chart.<sup class="reference" id="cite_ref-IRT_19-0"><a href="#cite_note-IRT-19">[19]</a></sup></div>
</div>
</div>
<p>A <a href="/wiki/Structure_chart" title="Structure chart">structure chart</a> (SC) is a chart that shows the breakdown of the <a class="mw-redirect" href="/wiki/Configuration_system" title="Configuration system">configuration system</a> to the lowest manageable levels.<sup class="reference" id="cite_ref-IRT_19-1"><a href="#cite_note-IRT-19">[19]</a></sup> This chart is used in <a href="/wiki/Structured_programming" title="Structured programming">structured programming</a> to arrange the program modules in a tree structure. Each module is represented by a box which contains the name of the modules. The tree structure visualizes the relationships between the modules.<sup class="reference" id="cite_ref-20"><a href="#cite_note-20">[20]</a></sup></p>
<p>Structure charts are used in structured analysis to specify the high-level design, or architecture, of a <a href="/wiki/Computer_program" title="Computer program">computer program</a>. As a design tool, they aid the programmer in dividing and conquering a large software problem, that is, recursively breaking a problem down into parts that are small enough to be understood by a human brain. The process is called <a class="mw-redirect" href="/wiki/Top-down_design" title="Top-down design">top-down design</a>, or <a href="/wiki/Functional_decomposition" title="Functional decomposition">functional decomposition</a>. Programmers use a structure chart to build a program in a manner similar to how an architect uses a blueprint to build a house. In the design stage, the chart is drawn and used as a way for the client and the various software designers to communicate. During the actual building of the program (implementation), the chart is continually referred to as the master-plan.<sup class="reference" id="cite_ref-Wol_21-0"><a href="#cite_note-Wol-21">[21]</a></sup></p>
<h3><span class="mw-headline" id="Structured_design">Structured design</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Structured_analysis&amp;action=edit&amp;section=10" title="Edit section: Structured design">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Structured design (SD) is concerned with the development of modules and the synthesis of these modules in a so-called "module hierarchy".<sup class="reference" id="cite_ref-FOOTNOTEPage-Jones1980_22-0"><a href="#cite_note-FOOTNOTEPage-Jones1980-22">[22]</a></sup> In order to design optimal module structure and interfaces two principles are crucial:</p>
<ul>
<li><i><a href="/wiki/Cohesion_(computer_science)" title="Cohesion (computer science)">Cohesion</a></i> which is "concerned with the grouping of functionally related processes into a particular module",<sup class="reference" id="cite_ref-HeSi86_10-4"><a href="#cite_note-HeSi86-10">[10]</a></sup> and</li>
<li><i><a class="mw-redirect" href="/wiki/Coupling_(computer_science)" title="Coupling (computer science)">Coupling</a></i> relates to "the flow of information or parameters passed between modules. Optimal coupling reduces the interfaces of modules and the resulting complexity of the software".<sup class="reference" id="cite_ref-HeSi86_10-5"><a href="#cite_note-HeSi86-10">[10]</a></sup></li>
</ul>
<p>Structured design was developed by <a href="/wiki/Larry_Constantine" title="Larry Constantine">Larry Constantine</a> in the late 1960s, then refined and published with collaborators in the 1970s;<sup class="reference" id="cite_ref-FOOTNOTEStevensMyersConstantine1974_5-1"><a href="#cite_note-FOOTNOTEStevensMyersConstantine1974-5">[5]</a></sup><sup class="reference" id="cite_ref-FOOTNOTEYourdonConstantine1979_6-1"><a href="#cite_note-FOOTNOTEYourdonConstantine1979-6">[6]</a></sup> see <a href="/wiki/Larry_Constantine#Structured_design" title="Larry Constantine">Larry Constantine: structured design</a> for details. <a href="#CITEREFPage-Jones1980">Page-Jones (1980)</a> has proposed his own approach which consists of three main objects :</p>
<ul>
<li>structure charts</li>
<li>module specifications</li>
<li>data dictionary.</li>
</ul>
<p>The <a href="/wiki/Structure_chart" title="Structure chart">structure chart</a> aims to show "the module hierarchy or calling sequence relationship of modules. There is a module specification for each module shown on the structure chart. The module specifications can be composed of pseudo-code or a program design language. The <a href="/wiki/Data_dictionary" title="Data dictionary">data dictionary</a> is like that of structured analysis. At this stage in the <a class="mw-redirect" href="/wiki/Software_development_lifecycle" title="Software development lifecycle">software development lifecycle</a>, after analysis and design have been performed, it is possible to automatically generate data type declarations",<sup class="reference" id="cite_ref-23"><a href="#cite_note-23">[23]</a></sup> and procedure or subroutine templates.<sup class="reference" id="cite_ref-HeSi86_10-6"><a href="#cite_note-HeSi86-10">[10]</a></sup></p>
<h3><span class="mw-headline" id="Structured_query_language">Structured query language</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Structured_analysis&amp;action=edit&amp;section=11" title="Edit section: Structured query language">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>The <a class="mw-redirect" href="/wiki/Structured_query_language" title="Structured query language">structured query language</a> (SQL) is a standardized language for querying information from a <a href="/wiki/Database" title="Database">database</a>. SQL was first introduced as a commercial database system in 1979 and has since been the favorite query language for database management systems running on minicomputers and mainframes. Increasingly, however, SQL is being supported by PC database systems because it supports distributed databases (see definition of distributed database). This enables several users on a computer network to access the same database simultaneously. Although there are different dialects of SQL, it is nevertheless the closest thing to a standard query language that currently exists.<sup class="reference" id="cite_ref-USDT01_14-3"><a href="#cite_note-USDT01-14">[14]</a></sup></p>

<h2><span class="mw-headline" id="Criticisms">Criticisms</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Structured_analysis&amp;action=edit&amp;section=13" title="Edit section: Criticisms">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Problems with data flow diagrams have included the following:<sup class="reference" id="cite_ref-FAA00_3-1"><a href="#cite_note-FAA00-3">[3]</a></sup></p>
<ol>
<li>Choosing bubbles appropriately</li>
<li>Partitioning bubbles in a meaningful and mutually agreed upon manner,</li>
<li>Documentation size needed to understand the Data Flows,</li>
<li>Data flow diagrams are strongly functional in nature and thus subject to frequent change</li>
<li>Though "data" flow is emphasized, "data" modeling is not, so there is little understanding the subject matter of the system</li>
<li>Customers have difficulty following how the concept is mapped into data flows and bubbles</li>
<li>Designers must shift the DFD organization into an implementable format</li>
</ol>
