<h1 class="firstHeading" id="firstHeading" lang="en">Model–view–controller</h1>
<div class="mw-body-content" id="bodyContent">

<p><b>Model–view–controller</b> (<b>MVC</b>) is a software <a href="/wiki/Architectural_pattern" title="Architectural pattern">architectural pattern</a> for implementing <a href="/wiki/User_interface" title="User interface">user interfaces</a> on computers. It divides a given software application into three interconnected parts, so as to separate internal representations of information from the ways that information is presented to or accepted from the user.<sup class="reference" id="cite_ref-1"><a href="#cite_note-1">[1]</a></sup><sup class="reference" id="cite_ref-2"><a href="#cite_note-2">[2]</a></sup></p>
<p>Traditionally used for desktop <a href="/wiki/Graphical_user_interface" title="Graphical user interface">graphical user interfaces</a> (GUIs), this architecture has become popular for designing <a href="/wiki/Web_application" title="Web application">web applications</a>.</p>

<h2><span class="mw-headline" id="Description">Description</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Model%E2%80%93view%E2%80%93controller&amp;action=edit&amp;section=1" title="Edit section: Description">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>As with other software patterns, MVC expresses the "core of the solution" to a problem while allowing it to be adapted for each system.<sup class="reference" id="cite_ref-gof_3-0"><a href="#cite_note-gof-3">[3]</a></sup> Particular MVC architectures can vary significantly from the traditional description here.<sup class="reference" id="cite_ref-4"><a href="#cite_note-4">[4]</a></sup></p>
<h3><span class="mw-headline" id="Components">Components</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Model%E2%80%93view%E2%80%93controller&amp;action=edit&amp;section=2" title="Edit section: Components">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="thumb tright">
<div class="thumbinner" style="width:202px;"><a class="image" href="/wiki/File:MVC-Process.svg"><img alt="" class="thumbimage" data-file-height="550" data-file-width="500" height="220" src="//upload.wikimedia.org/wikipedia/commons/thumb/a/a0/MVC-Process.svg/200px-MVC-Process.svg.png" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/a/a0/MVC-Process.svg/300px-MVC-Process.svg.png 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/a/a0/MVC-Process.svg/400px-MVC-Process.svg.png 2x" width="200"/></a>
<div class="thumbcaption">
<div class="magnify"><a class="internal" href="/wiki/File:MVC-Process.svg" title="Enlarge"></a></div>
A typical collaboration of the MVC components.</div>
</div>
</div>
<p>The central component of MVC, the <i>model</i>, captures the behavior of the application in terms of its <a href="/wiki/Problem_domain" title="Problem domain">problem domain</a>, independent of the user interface.<sup class="reference" id="cite_ref-5"><a href="#cite_note-5">[5]</a></sup></p>
<ul>
<li>The <i>model</i> directly manages the data, logic and rules of the application.</li>
<li>A <i>view</i> can be any output representation of information, such as a chart or a diagram. Multiple views of the same information are possible, such as a bar chart for management and a tabular view for accountants.</li>
<li>The third part, the <i>controller</i>, accepts input and converts it to commands for the model or view.<sup class="reference" id="cite_ref-6"><a href="#cite_note-6">[6]</a></sup></li>
</ul>
<h3><span class="mw-headline" id="Interactions">Interactions</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Model%E2%80%93view%E2%80%93controller&amp;action=edit&amp;section=3" title="Edit section: Interactions">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>In addition to dividing the application into three kinds of components, the model–view–controller design defines the interactions between them.<sup class="reference" id="cite_ref-posa_7-0"><a href="#cite_note-posa-7">[7]</a></sup></p>
<ul>
<li>A <i>model</i> stores data that is retrieved according to commands from the controller and displayed in the view.</li>
<li>A <i>view</i> generates new output to the user based on changes in the model.</li>
<li>A <i>controller</i> can send commands to the model to update the model's state (e.g. editing a document). It can also send commands to its associated view to change the view's presentation of the model (e.g. by scrolling through a document).</li>
</ul>
<h2><span class="mw-headline" id="History">History</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Model%E2%80%93view%E2%80%93controller&amp;action=edit&amp;section=4" title="Edit section: History">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>One of the seminal insights in the early development of graphical user interfaces, MVC became one of the first approaches to describe and implement software constructs in terms of their <a href="/wiki/Single_responsibility_principle" title="Single responsibility principle">responsibilities</a>.<sup class="reference" id="cite_ref-8"><a href="#cite_note-8">[8]</a></sup></p>
<p><a href="/wiki/Trygve_Reenskaug" title="Trygve Reenskaug">Trygve Reenskaug</a> introduced MVC into <a href="/wiki/Smalltalk" title="Smalltalk">Smalltalk</a>-76 while visiting the Xerox <a href="/wiki/PARC_(company)" title="PARC (company)">Palo Alto Research Center</a> (PARC)<sup class="reference" id="cite_ref-page_trygver_9-0"><a href="#cite_note-page_trygver-9">[9]</a></sup><sup class="reference" id="cite_ref-notes1_10-0"><a href="#cite_note-notes1-10">[10]</a></sup> in the 1970s. In the 1980s, Jim Althoff and others implemented a version of MVC for the Smalltalk-80 class library. Only later did a 1988 article in <i><a href="/wiki/The_Journal_of_Object_Technology" title="The Journal of Object Technology">The Journal of Object Technology</a></i> (JOT) express MVC as a general concept.<sup class="reference" id="cite_ref-art1_11-0"><a href="#cite_note-art1-11">[11]</a></sup></p>
<p>The MVC pattern has subsequently evolved,<sup class="reference" id="cite_ref-Fowler_12-0"><a href="#cite_note-Fowler-12">[12]</a></sup> giving rise to variants such as <a href="/wiki/Hierarchical_model%E2%80%93view%E2%80%93controller" title="Hierarchical model–view–controller">hierarchical model–view–controller</a> (HMVC), <a href="/wiki/Model%E2%80%93view%E2%80%93adapter" title="Model–view–adapter">model–view–adapter</a> (MVA), <a href="/wiki/Model%E2%80%93view%E2%80%93presenter" title="Model–view–presenter">model–view–presenter</a> (MVP), <a href="/wiki/Model%E2%80%93view%E2%80%93viewmodel" title="Model–view–viewmodel">model–view–viewmodel</a> (MVVM), and others that adapted MVC to different contexts.</p>
<p>The use of the MVC pattern in web applications exploded in popularity after the introduction<sup class="noprint Inline-Template" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Manual_of_Style/Dates_and_numbers#Chronological_items" title="Wikipedia:Manual of Style/Dates and numbers"><span title="The time period mentioned near this tag is ambiguous. (September 2016)">when?</span></a></i>]</sup> of <a href="/wiki/Apple_Inc." title="Apple Inc.">Apple</a>'s <a href="/wiki/WebObjects" title="WebObjects">WebObjects</a>, which was originally written in <a href="/wiki/Objective-C" title="Objective-C">Objective-C</a> (that borrowed heavily from Smalltalk) and helped enforce MVC principles. Later, the MVC pattern became popular with Java developers when WebObjects was ported to <a href="/wiki/Java_(programming_language)" title="Java (programming language)">Java</a>. Later frameworks for Java such as <a href="/wiki/Spring_Framework" title="Spring Framework">Spring</a> (released in 2002) continued the strong bond between Java and MVC. The introduction of the frameworks <a href="/wiki/Ruby_on_Rails" title="Ruby on Rails">Rails</a> (December 2005, for <a href="/wiki/Ruby_(programming_language)" title="Ruby (programming language)">Ruby</a>) and <a href="/wiki/Django_(web_framework)" title="Django (web framework)">Django</a> (July 2005, for <a href="/wiki/Python_(programming_language)" title="Python (programming language)">Python</a>), both of which had a strong emphasis on rapid deployment, increased MVC's popularity outside the traditional enterprise environment in which it has long been popular. MVC <a href="/wiki/Web_framework" title="Web framework">web frameworks</a> now hold large market-shares relative to non-MVC web <a href="/wiki/Wt_(web_toolkit)" title="Wt (web toolkit)">toolkits</a>.<sup class="reference" id="cite_ref-13"><a href="#cite_note-13">[13]</a></sup></p>
<h2><span class="mw-headline" id="Use_in_web_applications">Use in web applications</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Model%E2%80%93view%E2%80%93controller&amp;action=edit&amp;section=5" title="Edit section: Use in web applications">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Although originally developed for desktop computing, model–view–controller has been widely adopted as an architecture for <a href="/wiki/World_Wide_Web" title="World Wide Web">World Wide Web</a> applications in major <a href="/wiki/Programming_language" title="Programming language">programming languages</a>. Several commercial and noncommercial <a href="/wiki/Web_framework" title="Web framework">web frameworks</a> have been created that enforce the pattern. These <a href="/wiki/Software_framework" title="Software framework">software frameworks</a> vary in their interpretations, mainly in the way that the MVC responsibilities are divided between the <a href="/wiki/Client%E2%80%93server_model" title="Client–server model">client and server</a>.<sup class="reference" id="cite_ref-leff_14-0"><a href="#cite_note-leff-14">[14]</a></sup></p>
<p>Early web MVC frameworks took a <a href="/wiki/Thin_client" title="Thin client">thin client</a> approach that placed almost the entire model, view and controller logic on the server. This is still reflected in popular frameworks such as <a href="/wiki/Ruby_on_Rails" title="Ruby on Rails">Ruby on Rails</a>, <a href="/wiki/Django_(web_framework)" title="Django (web framework)">Django</a>, <a href="/wiki/ASP.NET_MVC" title="ASP.NET MVC">ASP.NET MVC</a>. In this approach, the client sends either <a href="/wiki/Hyperlink" title="Hyperlink">hyperlink</a> requests or <a class="mw-redirect" href="/wiki/Form_(web)" title="Form (web)">form</a> input to the controller and then receives a complete and updated web page (or other document) from the view; the model exists entirely on the server.<sup class="reference" id="cite_ref-leff_14-1"><a href="#cite_note-leff-14">[14]</a></sup> As client technologies have matured, frameworks such as <a href="/wiki/AngularJS" title="AngularJS">AngularJS</a>, <a href="/wiki/Ember.js" title="Ember.js">EmberJS</a>, <a href="/wiki/JavaScriptMVC" title="JavaScriptMVC">JavaScriptMVC</a> and <a href="/wiki/Backbone.js" title="Backbone.js">Backbone</a> have been created that allow the MVC components to execute partly on the client (also see <a href="/wiki/Ajax_(programming)" title="Ajax (programming)">Ajax</a>).</p>
