<h1 class="firstHeading" id="firstHeading" lang="en">Structural pattern</h1>
<div class="mw-body-content" id="bodyContent">
<div id="siteSub">From Wikipedia, the free encyclopedia</div>
<div id="contentSub"></div>
<div class="mw-jump" id="jump-to-nav">
					Jump to:					<a href="#mw-head">navigation</a>, 					<a href="#p-search">search</a>
</div>
<div class="mw-content-ltr" dir="ltr" id="mw-content-text" lang="en"><p>In <a href="/wiki/Software_engineering" title="Software engineering">software engineering</a>, <b>structural design patterns</b> are <a class="mw-redirect" href="/wiki/Design_pattern_(computer_science)" title="Design pattern (computer science)">design patterns</a> that ease the design by identifying a simple way to realize relationships between entities.</p>
<p>Examples of Structural Patterns include:</p>
<ul>
<li><a href="/wiki/Adapter_pattern" title="Adapter pattern">Adapter pattern</a>: 'adapts' one interface for a class into one that a client expects
<ul>
<li>Adapter pipeline: Use multiple adapters for debugging purposes.<sup class="reference" id="cite_ref-1"><a href="#cite_note-1">[1]</a></sup></li>
<li>Retrofit Interface Pattern:<sup class="reference" id="cite_ref-2"><a href="#cite_note-2">[2]</a></sup><sup class="reference" id="cite_ref-3"><a href="#cite_note-3">[3]</a></sup> An adapter used as a new interface for multiple classes at the same time.</li>
</ul>
</li>
<li><a href="/wiki/Aggregate_pattern" title="Aggregate pattern">Aggregate pattern</a>: a version of the <a href="/wiki/Composite_pattern" title="Composite pattern">Composite pattern</a> with methods for aggregation of children</li>
<li><a href="/wiki/Bridge_pattern" title="Bridge pattern">Bridge pattern</a>: decouple an abstraction from its implementation so that the two can vary independently
<ul>
<li>Tombstone: An intermediate "lookup" object contains the real location of an object.<sup class="reference" id="cite_ref-4"><a href="#cite_note-4">[4]</a></sup></li>
</ul>
</li>
<li><a href="/wiki/Composite_pattern" title="Composite pattern">Composite pattern</a>: a tree structure of objects where every object has the same interface</li>
<li><a href="/wiki/Decorator_pattern" title="Decorator pattern">Decorator pattern</a>: add additional functionality to a class at runtime where subclassing would result in an exponential rise of new classes</li>
<li><a class="mw-redirect" href="/wiki/Extensibility_pattern" title="Extensibility pattern">Extensibility pattern</a>: aka. Framework - hide complex code behind a simple interface</li>
<li><a href="/wiki/Facade_pattern" title="Facade pattern">Facade pattern</a>: create a simplified interface of an existing interface to ease usage for common tasks</li>
<li><a href="/wiki/Flyweight_pattern" title="Flyweight pattern">Flyweight pattern</a>: a large quantity of objects share a common properties object to save space</li>
<li><a href="/wiki/Marker_interface_pattern" title="Marker interface pattern">Marker pattern</a>: an empty interface to associate metadata with a class.</li>
<li><a class="mw-redirect" href="/wiki/Pipes_and_filters" title="Pipes and filters">Pipes and filters</a>: a chain of processes where the output of each process is the input of the next</li>
<li><a href="/wiki/Opaque_pointer" title="Opaque pointer">Opaque pointer</a>: a pointer to an undeclared or private type, to hide implementation details</li>
<li><a href="/wiki/Proxy_pattern" title="Proxy pattern">Proxy pattern</a>: a class functioning as an interface to another thing</li>
</ul>
