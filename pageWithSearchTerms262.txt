<h1 class="firstHeading" id="firstHeading" lang="en">Iterative and incremental development</h1>
<div class="mw-body-content" id="bodyContent">
<div id="siteSub">From Wikipedia, the free encyclopedia</div>
<div id="contentSub"></div>
<div class="mw-jump" id="jump-to-nav">
					Jump to:					<a href="#mw-head">navigation</a>, 					<a href="#p-search">search</a>
</div>
<div class="mw-content-ltr" dir="ltr" id="mw-content-text" lang="en"><p><b>Iterative and Incremental development</b> is any combination of both <a href="/wiki/Iterative_design" title="Iterative design">iterative design</a> or <a href="/wiki/Iterative_method" title="Iterative method">iterative method</a> and <a href="/wiki/Incremental_build_model" title="Incremental build model">incremental build model</a> for software development. The combination is of long standing <sup class="reference" id="cite_ref-1"><a href="#cite_note-1">[1]</a></sup> and has been widely suggested for large development efforts. For example, the 1985 DOD-STD-2167<sup class="reference" id="cite_ref-2"><a href="#cite_note-2">[2]</a></sup> mentions (in section 4.1.2): "During software development, more than one iteration of the software development cycle may be in progress at the same time." and "This process may be described as an 'evolutionary acquisition' or 'incremental build' approach." The relationship between iterations and increments is determined by the overall <a class="mw-redirect" href="/wiki/Software_development_methodology" title="Software development methodology">software development methodology</a> and <a href="/wiki/Software_development_process" title="Software development process">software development process</a>. The exact number and nature of the particular incremental builds and what is iterated will be specific to each individual development effort.</p>

<p>Iterative and incremental development are essential parts of the <a href="/wiki/Modified_waterfall_models" title="Modified waterfall models">Modified waterfall models</a>, <a href="/wiki/Rational_Unified_Process" title="Rational Unified Process">Rational Unified Process</a>, <a class="mw-redirect" href="/wiki/Extreme_Programming" title="Extreme Programming">Extreme Programming</a> and generally the various <a href="/wiki/Agile_software_development" title="Agile software development">agile software development</a> frameworks.</p>
<p>It follows a similar process to the <a href="/wiki/PDCA" title="PDCA">plan-do-check-act</a> cycle of business process improvement.</p>

<h2><span class="mw-headline" id="Overview">Overview</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Iterative_and_incremental_development&amp;action=edit&amp;section=1" title="Edit section: Overview">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<div class="thumb tright">
<div class="thumbinner" style="width:222px;"><a class="image" href="/wiki/File:Agile_Project_Management_by_Planbox.png"><img alt="" class="thumbimage" data-file-height="642" data-file-width="1048" height="135" src="//upload.wikimedia.org/wikipedia/commons/thumb/5/50/Agile_Project_Management_by_Planbox.png/220px-Agile_Project_Management_by_Planbox.png" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/5/50/Agile_Project_Management_by_Planbox.png/330px-Agile_Project_Management_by_Planbox.png 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/5/50/Agile_Project_Management_by_Planbox.png/440px-Agile_Project_Management_by_Planbox.png 2x" width="220"/></a>
<div class="thumbcaption">
<div class="magnify"><a class="internal" href="/wiki/File:Agile_Project_Management_by_Planbox.png" title="Enlarge"></a></div>
A simplified version of a typical iteration cycle in agile project management</div>
</div>
</div>
<p>The basic idea behind this method is to develop a system through repeated cycles (iterative) and in smaller portions at a time (incremental), allowing <a href="/wiki/Software_developer" title="Software developer">software developers</a> to take advantage of what was learned during development of earlier parts or versions of the system. Learning comes from both the development and use of the system, where possible key steps in the process start with a simple implementation of a subset of the software requirements and iteratively enhance the evolving versions until the full system is implemented. At each <a href="/wiki/Iteration" title="Iteration">iteration</a>, design modifications are made and new functional capabilities are added.</p>
<p>The procedure itself consists of the initialization step, the iteration step, and the Project Control List. The initialization step creates a base version of the system. The goal for this initial implementation is to create a product to which the user can react. It should offer a sampling of the key aspects of the problem and provide a solution that is simple enough to understand and implement easily. To guide the iteration process, a project control list is created that contains a record of all tasks that need to be performed. It includes such items as new features to be implemented and areas of redesign of the existing solution. The control list is constantly being revised as a result of the analysis phase.</p>
<p>The iteration involves the redesign and implementation of iteration is to be simple, straightforward, and modular, supporting redesign at that stage or as a task added to the project control list. The level of design detail is not dictated by the iterative approach. In a light-weight iterative project the code may represent the major source of <a href="/wiki/Software_documentation" title="Software documentation">documentation</a> of the system; however, in a critical iterative project a formal <a class="mw-redirect" href="/wiki/Software_Design_Document" title="Software Design Document">Software Design Document</a> may be used. The analysis of an iteration is based upon user feedback, and the program analysis facilities available. It involves analysis of the structure, modularity, <a href="/wiki/Usability" title="Usability">usability</a>, reliability, efficiency, &amp; achievement of goals. The project control list is modified in light of the analysis results.</p>

<h3><span class="mw-headline" id="Phases">Phases</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Iterative_and_incremental_development&amp;action=edit&amp;section=2" title="Edit section: Phases">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Incremental development slices the system functionality into increments (portions). In each increment, a slice of functionality is delivered through <a class="mw-redirect" href="/wiki/Cross_discipline" title="Cross discipline">cross-discipline</a> work, from the <a class="mw-redirect" href="/wiki/Requirements" title="Requirements">requirements</a> to the <a href="/wiki/Software_deployment" title="Software deployment">deployment</a>. The <a href="/wiki/Unified_Process" title="Unified Process">Unified Process</a> groups increments/iterations into phases: inception, elaboration, construction, and transition.</p>
<ul>
<li>Inception identifies project scope, requirements (functional and non-functional) and risks at a high level but in enough detail that work can be estimated.</li>
<li>Elaboration delivers a working architecture that mitigates the top risks and fulfills the non-functional requirements.</li>
<li>Construction incrementally fills-in the architecture with production-ready code produced from analysis, design, implementation, and testing of the functional requirements.</li>
<li>Transition delivers the system into the production operating environment.</li>
</ul>
<p>Each of the phases may be divided into 1 or more iterations, which are usually time-boxed rather than feature-boxed. Architects and analysts work one iteration ahead of developers and testers to keep their work-product backlog full.</p>
<h3><span class="mw-headline" id="Usage">Usage</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Iterative_and_incremental_development&amp;action=edit&amp;section=3" title="Edit section: Usage">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Many examples of early usage are provided in <a href="/wiki/Craig_Larman" title="Craig Larman">Craig Larman</a> and <a href="/wiki/Victor_Basili" title="Victor Basili">Victor Basili</a>'s article "Iterative and Incremental Development: A Brief History",<sup class="reference" id="cite_ref-3"><a href="#cite_note-3">[3]</a></sup> with one of the earliest being NASA's 1960s Project Mercury.</p>
<p>Another is an "early and striking example of a major IID success is the very heart of NASA’s space shuttle software—the primary avionics software system, which FSD built from 1977 to 1980. The team applied IID in a series of 17 iterations over 31 months, averaging around eight weeks per iteration. Their motivation for avoiding the waterfall life cycle was that the shuttle program’s requirements changed during the software development process".</p>
<p>Some organizations, such as the US Department of Defense, have a preference for iterative methodologies, starting with <a href="/wiki/MIL-STD-498" title="MIL-STD-498">MIL-STD-498</a> "clearly encouraging evolutionary acquisition and IID".</p>
<p>The current DoD Instruction 5000.2, released in 2000, states a clear preference for IID: "There are two approaches, evolutionary and single step [waterfall], to full capability. An evolutionary approach is preferred. … [In this] approach, the ultimate capability delivered to the user is divided into two or more blocks, with increasing increments of capability...software development shall follow an iterative spiral development process in which continually expanding software versions are based on learning from earlier development. It can also be done in phases .</p>

<p><a class="mw-redirect" href="/wiki/Waterfall_development" title="Waterfall development">Waterfall development</a> completes the project-wide work-products of each discipline in one step before moving on to the next discipline in the next step. Business value is delivered all at once, and only at the very end of the project. Backtracking is possible in an iterative approach.</p>
<h3><span class="mw-headline" id="Implementation_guidelines">Implementation guidelines</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Iterative_and_incremental_development&amp;action=edit&amp;section=5" title="Edit section: Implementation guidelines">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Guidelines that drive the implementation and analysis include:</p>
<ul>
<li>Any difficulty in design, coding and testing a modification should signal the need for redesign or re-coding.</li>
<li>Modifications should fit easily into isolated and easy-to-find modules. If they do not, some redesign is possibly needed.</li>
<li>Modifications to tables should be especially easy to make. If any table modification is not quickly and easily done, redesign is indicated.</li>
<li>Modifications should become easier to make as the iterations progress. If they are not, there is a basic problem such as a design flaw or a proliferation of <a href="/wiki/Patch_(computing)" title="Patch (computing)">patches</a>.</li>
<li>Patches should normally be allowed to exist for only one or two iterations. Patches may be necessary to avoid redesigning during an implementation phase.</li>
<li>The existing implementation should be analyzed frequently to determine how well it measures up to project goals.</li>
<li>Program analysis facilities should be used whenever available to aid in the analysis of partial implementations.</li>
<li>User reaction should be solicited and analyzed for indications of deficiencies in the current implementation.</li>
</ul>
<h2><span class="mw-headline" id="Use_in_hardware_and_embedded_systems">Use in hardware and embedded systems</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Iterative_and_incremental_development&amp;action=edit&amp;section=6" title="Edit section: Use in hardware and embedded systems">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>While the term <i>iterative and incremental</i> development got started in the software industry, many <a href="/wiki/Open-source_hardware" title="Open-source hardware">hardware</a> and <a href="/wiki/Embedded_software" title="Embedded software">embedded software</a> <a href="/wiki/New_product_development" title="New product development">development</a> efforts are using iterative and incremental techniques. For example, the large US <a href="/wiki/Launch_service_provider" title="Launch service provider">launch service provider</a> <a href="/wiki/United_Launch_Alliance" title="United Launch Alliance">United Launch Alliance</a> (ULA) has undertaken a decade-long project to restructure its launch business—reducing <a href="/wiki/Atlas_V" title="Atlas V">two lau</a><a href="/wiki/Delta_IV" title="Delta IV">nch vehicles</a> to <a class="mw-redirect" href="/wiki/Vulcan_launch_vehicle" title="Vulcan launch vehicle">one</a>—using an iterative and incremental approach to get to a <a class="mw-redirect" href="/wiki/Reusable_launch_vehicle" title="Reusable launch vehicle">partially-reusable</a> and much lower-cost launch system over the next decade.<sup class="reference" id="cite_ref-sn20150424_4-0"><a href="#cite_note-sn20150424-4">[4]</a></sup></p>
