<p>In <a href="/wiki/Software_engineering" title="Software engineering">software engineering</a>, <b>multitier architecture</b> (often referred to as <b><i>n</i>-tier architecture</b>) is a <a class="mw-redirect" href="/wiki/Client%E2%80%93server_architecture" title="Client–server architecture">client–server architecture</a> in which presentation, application processing, and data management functions are physically separated. The most widespread use of multitier architecture is the <b>three-tier architecture</b>.</p>
<p><i>N</i>-tier application architecture provides a model by which developers can create flexible and reusable applications. By segregating an application into tiers, developers acquire the option of modifying or adding a specific layer, instead of reworking the entire application. A three-tier architecture is typically composed of a <i>presentation</i> tier, a <i>domain logic</i> tier, and a <i>data storage</i> tier.</p>
<p>While the concepts of layer and tier are often used interchangeably, one fairly common point of view is that there is indeed a difference. This view holds that a <i>layer</i> is a logical structuring mechanism for the elements that make up the software solution, while a <i>tier</i> is a physical structuring mechanism for the system infrastructure.<sup class="reference" id="cite_ref-1"><a href="#cite_note-1">[1]</a></sup><sup class="reference" id="cite_ref-2"><a href="#cite_note-2">[2]</a></sup></p>
<p></p>

<h2><span class="mw-headline" id="Three-tier_architecture">Three-tier architecture</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Multitier_architecture&amp;action=edit&amp;section=1" title="Edit section: Three-tier architecture">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<div class="thumb tright">
<div class="thumbinner" style="width:402px;"><a class="image" href="/wiki/File:Overview_of_a_three-tier_application_vectorVersion.svg"><img alt="" class="thumbimage" data-file-height="530" data-file-width="593" height="358" src="//upload.wikimedia.org/wikipedia/commons/thumb/5/51/Overview_of_a_three-tier_application_vectorVersion.svg/400px-Overview_of_a_three-tier_application_vectorVersion.svg.png" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/5/51/Overview_of_a_three-tier_application_vectorVersion.svg/600px-Overview_of_a_three-tier_application_vectorVersion.svg.png 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/5/51/Overview_of_a_three-tier_application_vectorVersion.svg/800px-Overview_of_a_three-tier_application_vectorVersion.svg.png 2x" width="400"/></a>
<div class="thumbcaption">
<div class="magnify"><a class="internal" href="/wiki/File:Overview_of_a_three-tier_application_vectorVersion.svg" title="Enlarge"></a></div>
Overview of a three-tier application.</div>
</div>
</div>
<p>Three-tier architecture is a client–server <a class="mw-redirect" href="/wiki/Software_Architecture_styles_and_patterns" title="Software Architecture styles and patterns">software architecture pattern</a> in which the <a href="/wiki/User_interface" title="User interface">user interface</a> (presentation), <a class="mw-redirect" href="/wiki/Business_logic_layer" title="Business logic layer">functional process logic</a> ("business rules"), <a href="/wiki/Computer_data_storage" title="Computer data storage">computer data storage</a> and <a href="/wiki/Data_access" title="Data access">data access</a> are developed and maintained as independent <a href="/wiki/Modular_programming" title="Modular programming">modules</a>, most often on separate <a class="mw-redirect" href="/wiki/Platform_(computing)" title="Platform (computing)">platforms</a>.<sup class="reference" id="cite_ref-3"><a href="#cite_note-3">[3]</a></sup> It was developed by <a href="/wiki/John_J._Donovan" title="John J. Donovan">John J. Donovan</a> in Open Environment Corporation (OEC), a tools company he founded in <a class="mw-redirect" href="/wiki/Cambridge,_MA" title="Cambridge, MA">Cambridge, Massachusetts</a>.</p>
<p>Apart from the usual advantages of modular <a href="/wiki/Software" title="Software">software</a> with well-defined interfaces, the three-tier architecture is intended to allow any of the three tiers to be upgraded or replaced independently in response to changes in requirements or <a href="/wiki/Technology" title="Technology">technology</a>. For example, a change of <a href="/wiki/Operating_system" title="Operating system">operating system</a> in the <i>presentation tier</i> would only affect the user interface code.</p>
<p>Typically, the user interface runs on a desktop <a href="/wiki/Personal_computer" title="Personal computer">PC</a> or <a class="mw-redirect" href="/wiki/Workstation_(computer_hardware)" title="Workstation (computer hardware)">workstation</a> and uses a standard <a href="/wiki/Graphical_user_interface" title="Graphical user interface">graphical user interface</a>, functional process logic that may consist of one or more separate modules running on a workstation or <a href="/wiki/Application_server" title="Application server">application server</a>, and an <a class="mw-redirect" href="/wiki/RDBMS" title="RDBMS">RDBMS</a> on a <a href="/wiki/Database_server" title="Database server">database server</a> or <a href="/wiki/Mainframe_computer" title="Mainframe computer">mainframe</a> that contains the computer data storage logic. The middle tier may be multitiered itself (in which case the overall architecture is called an "<i>n</i>-tier architecture").</p>
<p>Three-tier architecture:</p>
<dl>
<dt>Presentation tier</dt>
<dd>This is the topmost level of the application. The presentation tier displays information related to such services as browsing merchandise, purchasing and shopping cart contents. It communicates with other tiers by which it puts out the results to the browser/client tier and all other tiers in the network. In simple terms, it is a layer which users can access directly (such as a web page, or an operating system's GUI).</dd>
<dt>Application tier (<a href="/wiki/Business_logic" title="Business logic">business logic</a>, logic tier, or middle tier)</dt>
<dd>The logical tier is pulled out from the presentation tier and, as its own layer, it controls an application’s functionality by performing detailed processing.</dd>
<dt>Data tier</dt>
<dd>The data tier includes the data persistence mechanisms (database servers, file shares, etc.) and the data access layer that encapsulates the persistence mechanisms and exposes the data. The data access layer should provide an <a href="/wiki/Application_programming_interface" title="Application programming interface">API</a> to the application tier that exposes methods of managing the stored data without exposing or creating dependencies on the data storage mechanisms. Avoiding dependencies on the storage mechanisms allows for updates or changes without the application tier clients being affected by or even aware of the change. As with the separation of any tier, there are costs for implementation and often costs to performance in exchange for improved scalability and maintainability.</dd>
</dl>
<h3><span class="mw-headline" id="Web_development_usage">Web development usage</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Multitier_architecture&amp;action=edit&amp;section=2" title="Edit section: Web development usage">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>In the <a href="/wiki/Web_development" title="Web development">web development</a> field, three-tier is often used to refer to <a href="/wiki/Website" title="Website">websites</a>, commonly <a class="mw-redirect" href="/wiki/Electronic_commerce" title="Electronic commerce">electronic commerce</a> websites, which are built using three tiers:</p>
<ol>
<li>A front-end <a href="/wiki/Web_server" title="Web server">web server</a> serving static content, and potentially some <a href="/wiki/Web_cache" title="Web cache">cached</a> dynamic content. In web based application, Front End is the content rendered by the browser. The content may be static or generated dynamically.</li>
<li>A middle dynamic content processing and generation level <a href="/wiki/Application_server" title="Application server">application server</a> (e.g., <a href="/wiki/ASP.NET" title="ASP.NET">ASP.NET</a>, <a href="/wiki/Ruby_on_Rails" title="Ruby on Rails">Ruby on Rails</a>, <a href="/wiki/Django_(web_framework)" title="Django (web framework)">Django (web framework)</a>, <a href="/wiki/Laravel" title="Laravel">Laravel</a>, <a href="/wiki/Spring_Framework" title="Spring Framework">Spring Framework</a>, <a href="/wiki/CodeIgniter" title="CodeIgniter">CodeIgniter</a>, <a href="/wiki/Symfony" title="Symfony">Symfony</a>, <a href="/wiki/Flask_(web_framework)" title="Flask (web framework)">Flask (web framework)</a>)</li>
<li>A back-end <a href="/wiki/Database" title="Database">database</a> or <a href="/wiki/Data_store" title="Data store">data store</a>, comprising both data sets and the <a class="mw-redirect" href="/wiki/Database_management_system" title="Database management system">database management system</a> software that manages and provides access to the data.</li>
</ol>
<h3><span class="mw-headline" id="Other_considerations">Other considerations</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Multitier_architecture&amp;action=edit&amp;section=3" title="Edit section: Other considerations">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Data transfer between tiers is part of the architecture. Protocols involved may include one or more of <a href="/wiki/Simple_Network_Management_Protocol" title="Simple Network Management Protocol">SNMP</a>, <a class="mw-redirect" href="/wiki/CORBA" title="CORBA">CORBA</a>, <a class="mw-redirect" href="/wiki/Java_RMI" title="Java RMI">Java RMI</a>, <a href="/wiki/.NET_Remoting" title=".NET Remoting">.NET Remoting</a>, <a href="/wiki/Windows_Communication_Foundation" title="Windows Communication Foundation">Windows Communication Foundation</a>, <a class="mw-redirect" href="/wiki/Internet_socket" title="Internet socket">sockets</a>, <a href="/wiki/User_Datagram_Protocol" title="User Datagram Protocol">UDP</a>, <a href="/wiki/Web_service" title="Web service">web services</a> or other standard or proprietary protocols. Often <a href="/wiki/Middleware_(distributed_applications)" title="Middleware (distributed applications)">middleware</a> is used to connect the separate tiers. Separate tiers often (but not necessarily) run on separate physical servers, and each tier may itself run on a <a class="mw-redirect" href="/wiki/Cluster_(computing)" title="Cluster (computing)">cluster</a>.</p>
<h2><span class="mw-headline" id="Traceability">Traceability</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Multitier_architecture&amp;action=edit&amp;section=4" title="Edit section: Traceability">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The end-to-end traceability of data flows through <i>n</i>-tier systems is a challenging task which becomes more important when systems increase in complexity. The <a href="/wiki/Application_Response_Measurement" title="Application Response Measurement">Application Response Measurement</a> defines concepts and <a href="/wiki/Application_programming_interface" title="Application programming interface">APIs</a> for measuring performance and correlating transactions between tiers. Generally, the term "tiers" is used to describe physical distribution of components of a system on separate servers, computers, or networks (processing nodes). A three-tier architecture then will have three processing nodes. The term "layers" refer to a logical grouping of components which may or may not be physically located on one processing node.</p>
