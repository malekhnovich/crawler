<h1 class="firstHeading" id="firstHeading" lang="en">Thread pool</h1>
<div class="mw-body-content" id="bodyContent">
<div id="siteSub">From Wikipedia, the free encyclopedia</div>
<div id="contentSub"><span class="mw-redirectedfrom">  (Redirected from <a class="mw-redirect" href="/w/index.php?title=Thread_pool_pattern&amp;redirect=no" title="Thread pool pattern">Thread pool pattern</a>)</span></div>

<p>In <a href="/wiki/Computer_programming" title="Computer programming">computer programming</a>, a <b>thread pool <a class="mw-redirect" href="/wiki/Design_pattern_(computer_science)" title="Design pattern (computer science)">pattern</a></b> (also <b>replicated workers</b> or <b>worker-crew model</b>)<sup class="reference" id="cite_ref-1"><a href="#cite_note-1">[1]</a></sup> consists of a number <span class="texhtml mvar" style="font-style:italic;">m</span> of <a class="mw-redirect" href="/wiki/Thread_(computer_science)" title="Thread (computer science)">threads</a>, created to perform a number <span class="texhtml mvar" style="font-style:italic;">n</span> of <a class="mw-redirect" href="/wiki/Task_(computers)" title="Task (computers)">tasks</a> <a href="/wiki/Concurrent_computing" title="Concurrent computing">concurrently</a>. Typically <span class="texhtml mvar" style="font-style:italic;">m</span> is not equal to <span class="texhtml mvar" style="font-style:italic;">n</span>; instead, the number of threads is tuned to the computing resources available to handle tasks in <a href="/wiki/Parallel_computing" title="Parallel computing">parallel</a> (<a href="/wiki/Central_processing_unit" title="Central processing unit">processors</a>, <a href="/wiki/Multi-core_processor" title="Multi-core processor">cores</a>, <a href="/wiki/Computer_memory" title="Computer memory">memory</a>) while the number of tasks depends on the problem and may not be known upfront.</p>
<p>Reasons for using a thread pool, rather than the obvious alternative of spawning one thread per task, are to prevent the time and memory overhead inherent in thread creation,<sup class="reference" id="cite_ref-2"><a href="#cite_note-2">[2]</a></sup> and to avoid running out of resources such as open files or network connections (of which operating systems allocate a limited number to running programs).<sup class="reference" id="cite_ref-3"><a href="#cite_note-3">[3]</a></sup> A common way of distributing the tasks to threads (<a href="/wiki/Scheduling_(computing)" title="Scheduling (computing)">scheduling</a> the tasks for execution) is by means of a <a href="/wiki/Synchronization_(computer_science)" title="Synchronization (computer science)">synchronized</a> <a class="mw-redirect" href="/wiki/Queue_(data_structure)" title="Queue (data structure)">queue</a> known as a <i><a class="mw-redirect" href="/wiki/Task_queue" title="Task queue">task queue</a></i>. The threads in the pool take tasks off the queue, perform them, then return to the queue for their next task.</p>

<h2><span class="mw-headline" id="Overview">Overview</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Thread_pool&amp;action=edit&amp;section=1" title="Edit section: Overview">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<h3><span class="mw-headline" id="Pool_size">Pool size</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Thread_pool&amp;action=edit&amp;section=2" title="Edit section: Pool size">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>The number of threads used is a parameter that can be tuned to provide the best performance.<sup class="reference" id="cite_ref-ACM.2C_Thread_pool_size_4-0"><a href="#cite_note-ACM.2C_Thread_pool_size-4">[4]</a></sup></p>
<p>A major advantage of using a thread pool over creating a new thread for each task is that thread creation and destruction overhead is negated, which may result in better <a href="/wiki/Performance_tuning" title="Performance tuning">performance</a> and better system <a href="/wiki/Stability_Model" title="Stability Model">stability</a>. Creating and destroying a thread and its associated resources is an expensive process in terms of time. An excessive number of threads will also waste memory, and context-switching between the runnable threads also damages performance. For example, a socket connection to another machine—which might take thousands (or even millions) of cycles to drop and re-establish—can be avoided by associating it with a thread which lives over the course of more than one transaction.</p>
<p>Typically, a thread pool executes on a single computer. However, thread pools are conceptually related to <a href="/wiki/Server_farm" title="Server farm">server farms</a> in which a master process, which might be a thread pool itself, distributes tasks to worker processes on different computers, in order to increase the overall throughput. <a href="/wiki/Embarrassingly_parallel" title="Embarrassingly parallel">Embarrassingly parallel</a> problems are highly amenable to this approach.</p>
<h3><span class="mw-headline" id="Dynamically_changing_the_pool_size">Dynamically changing the pool size</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Thread_pool&amp;action=edit&amp;section=3" title="Edit section: Dynamically changing the pool size">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Additionally, the number of threads can be dynamic based on the number of waiting tasks. For example, a <a href="/wiki/Web_server" title="Web server">web server</a> can add threads if numerous <a href="/wiki/Web_page" title="Web page">web page</a> requests come in and can remove threads when those requests taper down.<sup class="noprint Inline-Template" style="white-space:nowrap;">[<i><a class="mw-redirect" href="/wiki/Wikipedia:Disputed_statement" title="Wikipedia:Disputed statement"><span title="This claim has reliable sources with contradicting facts (December 2015)">disputed</span></a> <span class="metadata">– <a href="/wiki/Talk:Thread_pool" title="Talk:Thread pool">discuss</a></span></i>]</sup> The cost of having a larger thread pool is increased resource usage. The algorithm used to determine when to create or destroy threads will affect the overall performance:</p>
<ul>
<li>create too many threads, and resources are wasted and time also wasted creating any unused threads</li>
<li>destroy too many threads and more time will be spent later creating them again</li>
<li>creating threads too slowly might result in poor client performance (long wait times)</li>
<li>destroying threads too slowly may starve other processes of resources</li>
</ul>
<p>The algorithm chosen will depend on the problem and the expected usage patterns.</p>
