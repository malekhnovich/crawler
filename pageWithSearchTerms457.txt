<p>A <b>software design description</b> (aka <b>software design document</b> or <b>SDD</b>) is a written description of a <a href="/wiki/Software" title="Software">software</a> product, that a software designer writes in order to give a <a href="/wiki/Software_development" title="Software development">software development</a> team overall guidance to the architecture of the software project. An SDD usually accompanies an architecture diagram with pointers to detailed feature specifications of smaller pieces of the design. Practically, the description is required to coordinate a large team under a single vision, needs to be a stable reference, and outline all parts of the software and how they will work.</p>

<h2><span class="mw-headline" id="Composition">Composition</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Software_design_description&amp;action=edit&amp;section=1" title="Edit section: Composition">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The SDD usually contains the following information:</p>
<ol>
<li>The <i><a class="mw-redirect" href="/wiki/Data-driven_design" title="Data-driven design">data design</a></i> describes structures that reside within the software. Attributes and relationships between <a class="mw-redirect" href="/wiki/Data_object" title="Data object">data objects</a> dictate the choice of <a class="mw-redirect" href="/wiki/Data_structures" title="Data structures">data structures</a>.</li>
<li>The <i><a href="/wiki/Software_architecture" title="Software architecture">architecture design</a></i> uses information flowing characteristics, and maps them into the program structure. The transformation mapping method is applied to exhibit distinct boundaries between incoming and outgoing data. The data flow diagrams allocate control input, processing and output along three separate modules.</li>
<li>The <i><a class="mw-redirect" href="/wiki/Interface_design" title="Interface design">interface design</a></i> describes internal and external program interfaces, as well as the design of human interface. Internal and external interface designs are based on the information obtained from the analysis model.</li>
<li>The <i><a href="/wiki/Procedural_design" title="Procedural design">procedural design</a></i> describes structured programming concepts using graphical, tabular and textual notations. These design mediums enable the designer to represent procedural detail, that facilitates translation to code. This blueprint for implementation forms the basis for all subsequent software engineering work.</li>
</ol>
<h2><span class="mw-headline" id="IEEE_1016">IEEE 1016</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Software_design_description&amp;action=edit&amp;section=2" title="Edit section: IEEE 1016">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p><b>IEEE 1016-2009</b>, titled <i>IEEE Standard for Information Technology—Systems Design—Software Design Descriptions</i>,<sup class="reference" id="cite_ref-IEEE1016_1-0"><a href="#cite_note-IEEE1016-1">[1]</a></sup> is an <a class="mw-redirect" href="/wiki/IEEE" title="IEEE">IEEE</a> standard that specifies "the required information content and organization" for an SDD.<sup class="reference" id="cite_ref-2"><a href="#cite_note-2">[2]</a></sup> IEEE 1016 does not specify the medium of an SDD; it is "applicable to automated databases and design description languages but can be used for paper documents and other means of descriptions."<sup class="reference" id="cite_ref-3"><a href="#cite_note-3">[3]</a></sup></p>
<p>The 2009 edition was a major revision to IEEE 1016-1998, elevating it from recommended practice to full standard. This revision was modeled after <a href="/wiki/IEEE_1471" title="IEEE 1471">IEEE Std 1471-2000</a>, <i>Recommended Practice for Architectural Description of Software-intensive Systems</i>, extending the concepts of <a href="/wiki/IEEE_1471#IEEE_Conceptual_Framework_for_Architecture_Description" title="IEEE 1471">view, viewpoint, stakeholder, and concern</a> from architecture description to support documentation of high-level and detailed design and construction of software. [IEEE 1016, <i>Introduction</i>]</p>
<p>Following the IEEE 1016 conceptual model, an SDD is organized into one or more design views. Each design view follows the conventions of its design viewpoint. IEEE 1016 defines the following design viewpoints for use:<sup class="reference" id="cite_ref-4"><a href="#cite_note-4">[4]</a></sup></p>
<ul>
<li>Context viewpoint</li>
<li>Composition viewpoint</li>
<li>Logical viewpoint</li>
<li>Dependency viewpoint</li>
<li>Information viewpoint</li>
<li>Patterns use viewpoint</li>
<li>Interface viewpoint</li>
<li>Structure viewpoint</li>
<li>Interaction viewpoint</li>
<li>State dynamics viewpoint</li>
<li>Algorithm viewpoint</li>
<li>Resource viewpoint</li>
</ul>
<p>In addition, users of the standard are not limited to these viewpoints but may define their own.<sup class="reference" id="cite_ref-5"><a href="#cite_note-5">[5]</a></sup></p>
