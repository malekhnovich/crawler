<h1 class="firstHeading" id="firstHeading" lang="en">Introduction to Software Engineering/Architecture/Design Patterns</h1>

<div class="mw-content-ltr" dir="ltr" id="mw-content-text" lang="en"><h2><span class="mw-headline" id="Design_Patterns">Design Patterns</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Introduction_to_Software_Engineering/Architecture/Design_Patterns&amp;action=edit&amp;section=1" title="Edit section: Design Patterns">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>If you remember, software engineers speak a common language called UML. And if we use this analogy of language, then <i>design patterns</i> are the common stories our culture shares, like for instance fairy tales. They are stories about commonly occurring problems in software design and their solutions. And as young children learn about good and evil from fairy tales, beginning software engineers learn about good design (design patterns) and bad design (anti-patterns).</p>
<h3><span class="mw-headline" id="History">History</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Introduction_to_Software_Engineering/Architecture/Design_Patterns&amp;action=edit&amp;section=2" title="Edit section: History">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Patterns originated as an architectural concept by Christopher Alexander (1977/79). In 1987, Kent Beck and Ward Cunningham began experimenting with the idea of applying patterns to programming and presented their results at the OOPSLA conference that year.<sup class="reference" id="cite_ref-Smith1987_1-0"><a href="#cite_note-Smith1987-1">[1]</a></sup><sup class="reference" id="cite_ref-Beck1987_2-0"><a href="#cite_note-Beck1987-2">[2]</a></sup> In the following years, Beck, Cunningham and others followed up on this work.</p>
<p>Design patterns gained popularity in computer science after the book <i>Design Patterns: Elements of Reusable Object-Oriented Software</i> was published in 1994 by the so-called "Gang of Four" (Gamma et al.).<sup class="reference" id="cite_ref-GoF_3-0"><a href="#cite_note-GoF-3">[3]</a></sup> That same year, the first Pattern Languages of Programming Conference was held and the following year, the Portland Pattern Repository was set up for documentation of design patterns.</p>
<h3><span class="mw-headline" id="Definition_of_a_Design_Pattern">Definition of a Design Pattern</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Introduction_to_Software_Engineering/Architecture/Design_Patterns&amp;action=edit&amp;section=3" title="Edit section: Definition of a Design Pattern">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>In software engineering, a design pattern is a general reusable solution to a commonly occurring problem in software design. A design pattern is not a finished design that can be transformed directly into code. It is a description or template for how to solve a problem that can be used in many different situations. Object-oriented design patterns typically show relationships and interactions between classes or objects, without specifying the final application classes or objects that are involved.</p>
<p>Design patterns reside in the domain of modules and interconnections. At a higher level there are architectural patterns that are larger in scope, usually describing an overall pattern followed by an entire system.<sup class="reference" id="cite_ref-R.C.Martin_4-0"><a href="#cite_note-R.C.Martin-4">[4]</a></sup></p>
<p>There are many types of design patterns: Structural patterns address concerns related to the high level structure of an application being developed. Computational patterns address concerns related to the identification of key computations. Algorithm strategy patterns address concerns related to high level strategies that describe how to exploit application characteristic on a computation platform. Implementation strategy patterns address concerns related to the realization of the source code to support how the program itself is organized and the common data structures specific to parallel programming. Execution patterns address concerns related to the support of the execution of an application, including the strategies in executing streams of tasks and building blocks to support the synchronization between tasks.</p>
<h3><span class="mw-headline" id="Examples_of_Design_Patterns">Examples of Design Patterns</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Introduction_to_Software_Engineering/Architecture/Design_Patterns&amp;action=edit&amp;section=4" title="Edit section: Examples of Design Patterns">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Design patterns are easiest understood when looking at concrete examples. For beginners the following ten patterns may suffice. However, you should make it a habit to learn about some of the other patterns mentioned below. The more patterns you know, the better.</p>
<h4><span class="mw-headline" id="Factory_Method">Factory Method</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Introduction_to_Software_Engineering/Architecture/Design_Patterns&amp;action=edit&amp;section=5" title="Edit section: Factory Method">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>The Factory pattern creates an object from a set of similar classes, based on some parameter, usually a string. An example, is the creation of a MessageDigest object in Java:</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="n">MessageDigest</span> <span class="n">md</span> <span class="o">=</span> <span class="n">MessageDigest</span><span class="o">.</span><span class="na">getInstance</span><span class="o">(</span><span class="s">"SHA-1"</span><span class="o">);</span>
</pre></div>
<p>If one changes the parameter to "MD5" for instance, one gets an object that calculates the message digest based on the MD5 algorithm instead. The advantage of using a parameter is that changing the algorithm does not require us to re-compile our code. Other examples of this pattern are the loading of the database connection driver in Java using <code>Class.forName("jdbc.idbDriver")</code>, which admittedly is some very odd syntax, but the idea is the same.</p>
<h4><span class="mw-headline" id="Abstract_Factory">Abstract Factory</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Introduction_to_Software_Engineering/Architecture/Design_Patterns&amp;action=edit&amp;section=6" title="Edit section: Abstract Factory">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>Where the Factory pattern only affects one class, the Abstract Factory pattern affects a whole bunch of classes. A well known example from Java is the Swing Look-and-Feel:</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="n">UIManager</span><span class="o">.</span><span class="na">setLookAndFeel</span><span class="o">(</span><span class="s">"javax.swing.plaf.metal.MetalLookAndFeel"</span><span class="o">);</span>
</pre></div>
<p>This looks quite similar to the Factory pattern, but the difference is that now every Swing class that is being loaded is affected, not just one.</p>
<h4><span class="mw-headline" id="Singleton">Singleton</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Introduction_to_Software_Engineering/Architecture/Design_Patterns&amp;action=edit&amp;section=7" title="Edit section: Singleton">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>This is one of the most dangerous design patterns, when in doubt don't use it. Its main purpose is to guarantee that only one instance of a particular object exists. Possible applications are a printer manager or a database connection manager. It is useful when access to a limited resource needs to be controlled.</p>
<h4><span class="mw-headline" id="Iterator">Iterator</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Introduction_to_Software_Engineering/Architecture/Design_Patterns&amp;action=edit&amp;section=8" title="Edit section: Iterator">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>Nowadays, the Iterator pattern is trivial: it allows you to go through a list of objects, starting at the beginning, iterating through the list one element after the other, until reaching the end.</p>
<h4><span class="mw-headline" id="Template_Method">Template Method</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Introduction_to_Software_Engineering/Architecture/Design_Patterns&amp;action=edit&amp;section=9" title="Edit section: Template Method">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>Also the Template Method pattern is rather simple: as soon as you define an abstract class, that forces its subclasses to implement some method, you are using a simple form of the Template pattern.</p>
<h4><span class="mw-headline" id="Command">Command</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Introduction_to_Software_Engineering/Architecture/Design_Patterns&amp;action=edit&amp;section=10" title="Edit section: Command">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>To understand the idea behind the Command pattern consider the following restaurant example: A customer goes to a restaurant and orders some food. The waiter takes the order (command, in this case) and hands it to the cook in the kitchen. In the kitchen the command is executed, and depending on the command different food or drink is being prepared.</p>
<h4><span class="mw-headline" id="Observer">Observer</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Introduction_to_Software_Engineering/Architecture/Design_Patterns&amp;action=edit&amp;section=11" title="Edit section: Observer">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>The Observer pattern is one of the most popular patterns, and it has many variants. Assume you have a table in a spreadsheet. That data can be displayed in table form, but also in form of some graph or histogram. If the underlying data changes, not only the table view has to change, but you also expect the histogram to change. To communicate these changes you can use the Observer pattern: the underlying data is the <i>observable</i> and the table view as well as the histogram view are <i>observers</i> that observe the observable. Another example of the Observer pattern is a button in Swing for instance: here the JButton is the observable, and if something happens to the button (usually this means it was pressed by the user) then the listener (the observer) gets notified.</p>
<h4><span class="mw-headline" id="Composite">Composite</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Introduction_to_Software_Engineering/Architecture/Design_Patterns&amp;action=edit&amp;section=12" title="Edit section: Composite">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>The Composite pattern is very wide spread. Basically, it is a list that may contain objects, but also lists. A typical example is a file system, which may consist of directories and files. Here directories may contain files, but also may contain other directories. Other examples of the Composite pattern are menus that may contain other menus, or in user management one often has users and groups, where groups may contain users, but also other groups.</p>
<h4><span class="mw-headline" id="State">State</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Introduction_to_Software_Engineering/Architecture/Design_Patterns&amp;action=edit&amp;section=13" title="Edit section: State">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>In the State pattern, an internal state of the object influences its behavior. Assume you have some drawing program in which you want to be able to draw straight lines and dotted lines. Instead of creating different classes for lines, you have one Line class that has an internal state called 'dotted' or 'straight' and depending on this internal state either dotted lines or straight lines are drawn. This pattern is also implicitly used by Java, when setting the font via ’setFont()’ or the color via 'setColor()', for instance.</p>
<h4><span class="mw-headline" id="Proxy">Proxy</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Introduction_to_Software_Engineering/Architecture/Design_Patterns&amp;action=edit&amp;section=14" title="Edit section: Proxy">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>The idea behind the Proxy pattern is that we have some complex object and we need to make it simpler. One typical application is an object that exists on another machine, but you want to give the impression as if the user is dealing with a local object. Another application is when an object would take a long time to create (like loading a large image/video), but the actual object may never be needed. In this case a proxy represents the object until it is needed.</p>
<h3><span class="mw-headline" id="Patterns_in_Practice">Patterns in Practice</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Introduction_to_Software_Engineering/Architecture/Design_Patterns&amp;action=edit&amp;section=15" title="Edit section: Patterns in Practice">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Design patterns can speed up the development process by providing tested, proven development paradigms. Effective software design requires considering issues that may not become visible until later in the implementation. Reusing design patterns helps to prevent subtle issues that can cause major problems, and it also improves code readability for coders and architects who are familiar with the patterns. In addition to this, patterns allow developers to communicate using well-known, well understood names for software interactions.</p>
<p>In order to achieve flexibility, design patterns usually introduce additional levels of indirection, which in some cases may complicate the resulting designs and hurt application performance.</p>
<p>By definition, a pattern must be programmed anew into each application that uses it. Since some authors see this as a step backward from software reuse as provided by components, researchers have worked to turn patterns into components. Meyer and Arnout were able to provide full or partial componentization of two-thirds of the patterns they attempted.<sup class="reference" id="cite_ref-Meyer2006_5-0"><a href="#cite_note-Meyer2006-5">[5]</a></sup></p>
<h3><span class="mw-headline" id="Classification_and_List_of_Patterns">Classification and List of Patterns</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Introduction_to_Software_Engineering/Architecture/Design_Patterns&amp;action=edit&amp;section=16" title="Edit section: Classification and List of Patterns">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Design patterns were originally grouped into the categories: creational patterns, structural patterns, and behavioral patterns, and described using the concepts of delegation, aggregation, and consultation. Another classification has also introduced the notion of architectural design pattern that may be applied at the architecture level of the software such as the Model-View-Controller pattern. The following patterns are taken from <i>Design Patterns</i> <sup class="reference" id="cite_ref-GoF_3-1"><a href="#cite_note-GoF-3">[3]</a></sup> and <i>Code Complete</i>,<sup class="reference" id="cite_ref-McConnell2004_6-0"><a href="#cite_note-McConnell2004-6">[6]</a></sup> unless otherwise stated.</p>
<p><br/></p>
<h4><span class="mw-headline" id="Creational_patterns">Creational patterns</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Introduction_to_Software_Engineering/Architecture/Design_Patterns&amp;action=edit&amp;section=17" title="Edit section: Creational patterns">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<ul>
<li><b>Abstract factory:</b> Provide an interface for creating families of related or dependent objects without specifying their concrete classes.</li>
</ul>
<ul>
<li><b>Builder:</b> Separate the construction of a complex object from its representation allowing the same construction process to create various representations.</li>
</ul>
<ul>
<li><b>Factory method:</b> Define an interface for creating an object, but let subclasses decide which class to instantiate. Factory Method lets a class defer instantiation to subclasses.</li>
</ul>
<ul>
<li><b>Lazy initialization:</b> Tactic of delaying the creation of an object, the calculation of a value, or some other expensive process until the first time it is needed.<sup class="reference" id="cite_ref-PoEAA_7-0"><a href="#cite_note-PoEAA-7">[7]</a></sup></li>
</ul>
<ul>
<li><b>Multiton:</b> Ensure a class has only named instances, and provide global point of access to them.</li>
</ul>
<ul>
<li><b>Object pool:</b> Avoid expensive acquisition and release of resources by recycling objects that are no longer in use. Can be considered a generalisation of connection pool and thread pool patterns.</li>
</ul>
<ul>
<li><b>Prototype:</b> Specify the kinds of objects to create using a prototypical instance, and create new objects by copying this prototype.</li>
</ul>
<ul>
<li><b>Resource acquisition is initialization:</b> Ensure that resources are properly released by tying them to the lifespan of suitable objects.</li>
</ul>
<ul>
<li><b>Singleton:</b> Ensure a class has only one instance, and provide a global point of access to it.</li>
</ul>
<p><br/></p>
<h4><span class="mw-headline" id="Structural_Patterns">Structural Patterns</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Introduction_to_Software_Engineering/Architecture/Design_Patterns&amp;action=edit&amp;section=18" title="Edit section: Structural Patterns">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<ul>
<li><b>Adapter or Wrapper:</b> Convert the interface of a class into another interface clients expect. Adapter lets classes work together that could not otherwise because of incompatible interfaces.</li>
</ul>
<ul>
<li><b>Bridge:</b> Decouple an abstraction from its implementation allowing the two to vary independently.</li>
</ul>
<ul>
<li><b>Composite:</b> Compose objects into tree structures to represent part-whole hierarchies. Composite lets clients treat individual objects and compositions of objects uniformly.</li>
</ul>
<ul>
<li><b>Decorator:</b> Attach additional responsibilities to an object dynamically keeping the same interface. Decorators provide a flexible alternative to subclassing for extending functionality.</li>
</ul>
<ul>
<li><b>Facade:</b> Provide a unified interface to a set of interfaces in a subsystem. Facade defines a higher-level interface that makes the subsystem easier to use.</li>
</ul>
<ul>
<li><b>Front Controller:</b> Provide a unified interface to a set of interfaces in a subsystem. Front Controller defines a higher-level interface that makes the subsystem easier to use.</li>
</ul>
<ul>
<li><b>Flyweight:</b> Use sharing to support large numbers of fine-grained objects efficiently.</li>
</ul>
<ul>
<li><b>Proxy:</b> Provide a surrogate or placeholder for another object to control access to it.</li>
</ul>
<p><br/></p>
<h4><span class="mw-headline" id="Behavioral_Patterns">Behavioral Patterns</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Introduction_to_Software_Engineering/Architecture/Design_Patterns&amp;action=edit&amp;section=19" title="Edit section: Behavioral Patterns">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<ul>
<li><b>Blackboard:</b> Generalized observer, which allows multiple readers and writers. Communicates information system-wide.</li>
</ul>
<ul>
<li><b>Chain of responsibility:</b> Avoid coupling the sender of a request to its receiver by giving more than one object a chance to handle the request. Chain the receiving objects and pass the request along the chain until an object handles it.</li>
</ul>
<ul>
<li><b>Command:</b> Encapsulate a request as an object, thereby letting you parameterize clients with different requests, queue or log requests, and support undoable operations.</li>
</ul>
<ul>
<li><b>Interpreter:</b> Given a language, define a representation for its grammar along with an interpreter that uses the representation to interpret sentences in the language.</li>
</ul>
<ul>
<li><b>Iterator:</b> Provide a way to access the elements of an aggregate object sequentially without exposing its underlying representation.</li>
</ul>
<ul>
<li><b>Mediator:</b> Define an object that encapsulates how a set of objects interact. Mediator promotes loose coupling by keeping objects from referring to each other explicitly, and it lets you vary their interaction independently.</li>
</ul>
<ul>
<li><b>Memento:</b> Without violating encapsulation, capture and externalize an object's internal state allowing the object to be restored to this state later.</li>
</ul>
<ul>
<li><b>Null object:</b> Avoid null references by providing a default object.</li>
</ul>
<ul>
<li><b>Observer or Publish/subscribe:</b> Define a one-to-many dependency between objects where a state change in one object results with all its dependents being notified and updated automatically.</li>
</ul>
<ul>
<li><b>Servant:</b> Define common functionality for a group of classes</li>
</ul>
<ul>
<li><b>Specification:</b> Recombinable business logic in a boolean fashion</li>
</ul>
<ul>
<li><b>State:</b> Allow an object to alter its behavior when its internal state changes. The object will appear to change its class.</li>
</ul>
<ul>
<li><b>Strategy:</b> Define a family of algorithms, encapsulate each one, and make them interchangeable. Strategy lets the algorithm vary independently from clients that use it.</li>
</ul>
<ul>
<li><b>Template method:</b> Define the skeleton of an algorithm in an operation, deferring some steps to subclasses. Template Method lets subclasses redefine certain steps of an algorithm without changing the algorithm's structure.</li>
</ul>
<ul>
<li><b>Visitor:</b> Represent an operation to be performed on the elements of an object structure. Visitor lets you define a new operation without changing the classes of the elements on which it operates.</li>
</ul>
<p><br/></p>
<h4><span class="mw-headline" id="Concurrency_Patterns">Concurrency Patterns</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Introduction_to_Software_Engineering/Architecture/Design_Patterns&amp;action=edit&amp;section=20" title="Edit section: Concurrency Patterns">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>Most of the following concurrency patterns are taken from <i>POSA2</i><sup class="reference" id="cite_ref-POSA2_8-0"><a href="#cite_note-POSA2-8">[8]</a></sup></p>
<ul>
<li><b>Active Object:</b> Decouples method execution from method invocation that reside in their own thread of control. The goal is to introduce concurrency, by using asynchronous method invocation and a scheduler for handling requests.</li>
</ul>
<ul>
<li><b>Balking:</b> Only execute an action on an object when the object is in a particular state.</li>
</ul>
<ul>
<li><b>Binding Properties:</b> Combining multiple observers to force properties in different objects to be synchronized or coordinated in some way.<sup class="reference" id="cite_ref-9"><a href="#cite_note-9">[9]</a></sup></li>
</ul>
<ul>
<li><b>Messaging pattern:</b> The messaging design pattern (MDP) allows the interchange of information (i.e. messages) between components and applications.</li>
</ul>
<ul>
<li><b>Double-checked locking:</b> Reduce the overhead of acquiring a lock by first testing the locking criterion (the 'lock hint') in an unsafe manner; only if that succeeds does the actual lock proceed. Can be unsafe when implemented in some language/hardware combinations. It can therefore sometimes be considered an anti-pattern.</li>
</ul>
<ul>
<li><b>Event-based asynchronous:</b> Addresses problems with the Asynchronous pattern that occur in multithreaded programs.<sup class="reference" id="cite_ref-PC.232008_10-0"><a href="#cite_note-PC.232008-10">[10]</a></sup></li>
</ul>
<ul>
<li><b>Guarded suspension:</b> Manages operations that require both a lock to be acquired and a precondition to be satisfied before the operation can be executed.</li>
</ul>
<ul>
<li><b>Lock:</b> One thread puts a "lock" on a resource, preventing other threads from accessing or modifying it.<sup class="reference" id="cite_ref-11"><a href="#cite_note-11">[11]</a></sup><sup class="reference" id="cite_ref-PoEAA_7-1"><a href="#cite_note-PoEAA-7">[7]</a></sup></li>
</ul>
<ul>
<li><b>Monitor object:</b> An object whose methods are subject to mutual exclusion, thus preventing multiple objects from erroneously trying to use it at the same time.</li>
</ul>
<ul>
<li><b>Reactor:</b> A reactor object provides an asynchronous interface to resources that must be handled synchronously.</li>
</ul>
<ul>
<li><b>Read-write lock:</b> Allows concurrent read access to an object but requires exclusive access for write operations.</li>
</ul>
<ul>
<li><b>Scheduler:</b> Explicitly control when threads may execute single-threaded code.</li>
</ul>
<ul>
<li><b>Thread pool:</b> A number of threads are created to perform a number of tasks, which are usually organized in a queue. Typically, there are many more tasks than threads. Can be considered a special case of the object pool pattern.</li>
</ul>
<ul>
<li><b>Thread-specific storage:</b> Static or "global" memory local to a thread.</li>
</ul>
<h4><span class="mw-headline" id="Data_Access_Patterns">Data Access Patterns</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Introduction_to_Software_Engineering/Architecture/Design_Patterns&amp;action=edit&amp;section=21" title="Edit section: Data Access Patterns">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>Another interesting area where patterns have a wide application is the area of <i>data access patterns</i>. Clifton Nock <sup class="reference" id="cite_ref-12"><a href="#cite_note-12">[12]</a></sup> lists the following patterns:</p>
<ul>
<li><b>ORM Patterns:</b> Domain Object Factory, Object/Relational Map, Update Factory</li>
</ul>
<ul>
<li><b>Resource Management Patterns:</b> Resource Pool, Resource Timer, Retryer, Paging Iterator</li>
</ul>
<ul>
<li><b>Cache Patterns:</b> Cache Accessor, Demand Cache, Primed Cache, Cache Collector, Cache Replicator</li>
</ul>
<ul>
<li><b>Concurrency Patterns:</b> Transaction, Optimistic Lock, Pessimistic Lock</li>
</ul>
<p><br/></p>
<h4><span class="mw-headline" id="Enterprise_Patterns">Enterprise Patterns</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Introduction_to_Software_Engineering/Architecture/Design_Patterns&amp;action=edit&amp;section=22" title="Edit section: Enterprise Patterns">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>If you deal with J2EE or with .Net Enterprise applications, the problems that occur and the solutions to them are similar. These solutions are the Enterprise patterns. The book <i>Core J2EE Patterns</i> <sup class="reference" id="cite_ref-13"><a href="#cite_note-13">[13]</a></sup> lists these patterns:</p>
<ul>
<li><b>Presentation Tier Patterns:</b> Intercepting Filter, Front Controller, View Helper, Composite View, Service to Worker, Dispatcher View</li>
</ul>
<ul>
<li><b>Business Tier Patterns:</b> Business Delegate, Value Object, Session Facade, Composite Entity, Value Object Assembler, Value List Handler, Service Locator</li>
</ul>
<ul>
<li><b>Integration Tier Patterns:</b> Data Access Object, Service Activator</li>
</ul>
<p><br/></p>
<h4><span class="mw-headline" id="Real-Time_Patterns">Real-Time Patterns</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Introduction_to_Software_Engineering/Architecture/Design_Patterns&amp;action=edit&amp;section=23" title="Edit section: Real-Time Patterns">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>Finally, in the area of real-time and embedded software development a vast number of patterns have been identified.<sup class="reference" id="cite_ref-14"><a href="#cite_note-14">[14]</a></sup><sup class="reference" id="cite_ref-15"><a href="#cite_note-15">[15]</a></sup> In his book <i>Real-Time Design Patterns: Robust Scalable Architecture for Real-Time Systems</i><sup class="reference" id="cite_ref-16"><a href="#cite_note-16">[16]</a></sup><sup class="reference" id="cite_ref-17"><a href="#cite_note-17">[17]</a></sup> Bruce Powel Douglass lists some very intriguing patterns:</p>
<ul>
<li><b>Architecture Patterns:</b> Layered Pattern, Channel Architecture Pattern, Component-Based Architecture, Recursive Containment Pattern and Hierarchical Control Pattern, Microkernel Architecture Pattern, Virtual Machine Pattern</li>
</ul>
<ul>
<li><b>Concurrency Patterns:</b> Message Queuing Pattern, Interrupt Pattern, Guarded Call Pattern, Rendezvous Pattern, Cyclic Executive Pattern, Round Robin Pattern</li>
</ul>
<ul>
<li><b>Memory Patterns:</b> Static Allocation Pattern, Pool Allocation Pattern, Fixed Sized Buffer Pattern, Smart Pointer Pattern, Garbage Collection Pattern, Garbage Compactor Pattern</li>
</ul>
<ul>
<li><b>Resource Patterns:</b> Critical Section Pattern, Priority Inheritance Pattern, Priority Ceiling Pattern, Simultaneous Locking Pattern, Ordered Locking Pattern</li>
</ul>
<ul>
<li><b>Distribution Patterns:</b> Shared Memory Pattern, Remote Method Call Pattern, Observer Pattern, Data Bus Pattern, Proxy Pattern, Broker Pattern</li>
</ul>
<ul>
<li><b>Safety and Reliability Patterns:</b> Monitor-Actuator Pattern, Sanity Check Pattern, Watchdog Pattern, Safety Executive Pattern, Protected Single Channel Pattern, Homogeneous Redundancy Pattern, Triple Modular Redundancy Pattern, Heterogeneous Redundancy Pattern</li>
</ul>
<p><br/>
Efforts have also been made to codify design patterns in particular domains, including use of existing design patterns as well as domain specific design patterns. Examples include user interface design patterns,<sup class="reference" id="cite_ref-18"><a href="#cite_note-18">[18]</a></sup> information visualization <sup class="reference" id="cite_ref-19"><a href="#cite_note-19">[19]</a></sup>, secure design<sup class="reference" id="cite_ref-20"><a href="#cite_note-20">[20]</a></sup>, "secure usability"<sup class="reference" id="cite_ref-21"><a href="#cite_note-21">[21]</a></sup>, web design <sup class="reference" id="cite_ref-22"><a href="#cite_note-22">[22]</a></sup> and business model design.<sup class="reference" id="cite_ref-23"><a href="#cite_note-23">[23]</a></sup> The annual Pattern Languages of Programming Conference proceedings <sup class="reference" id="cite_ref-24"><a href="#cite_note-24">[24]</a></sup> include many examples of domain specific patterns.</p>
<h3><span class="mw-headline" id="Documenting_and_Describing_Patterns">Documenting and Describing Patterns</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Introduction_to_Software_Engineering/Architecture/Design_Patterns&amp;action=edit&amp;section=24" title="Edit section: Documenting and Describing Patterns">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Assume you discovered a new design pattern. Or your friend wants to explain to you this cool pattern she found in a pattern repository. How do we describe patterns? There is no single, standard format for documenting design patterns. Rather, a variety of different formats have been used by different pattern authors.<sup class="reference" id="cite_ref-GabrielHillside_25-0"><a href="#cite_note-GabrielHillside-25">[25]</a></sup> However, according to Martin Fowler certain pattern forms have become more well-known than others, and consequently become common starting points for new pattern writing efforts.<sup class="reference" id="cite_ref-Fowler2006_26-0"><a href="#cite_note-Fowler2006-26">[26]</a></sup> One example of a commonly used documentation format is the one used by the book <i>Design Patterns</i>.<sup class="reference" id="cite_ref-GoF_3-2"><a href="#cite_note-GoF-3">[3]</a></sup> It contains the following sections:</p>
<ul>
<li><b>Pattern Name and Classification:</b> A descriptive and unique name that helps in identifying and referring to the pattern.</li>
<li><b>Intent:</b> A description of the goal behind the pattern and the reason for using it.</li>
<li><b>Also Known As:</b> Other names for the pattern.</li>
<li><b>Motivation (Forces):</b> A scenario consisting of a problem and a context in which this pattern can be used.</li>
<li><b>Applicability:</b> Situations in which this pattern is usable; the context for the pattern.</li>
<li><b>Structure:</b> A graphical representation of the pattern. Class diagrams and Interaction diagrams may be used for this purpose.</li>
<li><b>Participants:</b> A listing of the classes and objects used in the pattern and their roles in the design.</li>
<li><b>Collaboration:</b> A description of how classes and objects used in the pattern interact with each other.</li>
<li><b>Consequences:</b> A description of the results, side effects, and trade offs caused by using the pattern.</li>
<li><b>Implementation:</b> A description of an implementation of the pattern; the solution part of the pattern.</li>
<li><b>Sample Code:</b> An illustration of how the pattern can be used in a programming language.</li>
<li><b>Known Uses:</b> Examples of real usages of the pattern.</li>
<li><b>Related Patterns:</b> Other patterns that have some relationship with the pattern; discussion of the differences between the pattern and similar patterns.</li>
</ul>
<p>Of particular interest are the Structure, Participants, and Collaboration sections. These sections describe a <i>design motif</i>: a prototypical <i>micro-architecture</i> that developers copy and adapt to their particular designs to solve the recurrent problem described by the design pattern. A micro-architecture is a set of program constituents (e.g., classes, methods...) and their relationships. Developers use the design pattern by introducing in their designs this prototypical micro-architecture, which means that micro-architectures in their designs will have structure and organization similar to the chosen design motif.</p>
