<h1 class="firstHeading" id="firstHeading" lang="en">Data model</h1>
<div class="mw-body-content" id="bodyContent">
<div id="siteSub">From Wikipedia, the free encyclopedia</div>
<div id="contentSub"></div>
<div class="mw-jump" id="jump-to-nav">
					Jump to:					<a href="#mw-head">navigation</a>, 					<a href="#p-search">search</a>
</div>
<div class="mw-content-ltr" dir="ltr" id="mw-content-text" lang="en"><p>A <b>Data Model</b> is an abstract model that organizes elements of <a href="/wiki/Data" title="Data">data</a> and standardizes how they relate to one another and to properties of the real world entities. For instance, a data model may specify that a data element representing a car comprise a number of other elements which in turn represent the color, size and owner of the car.</p>
<div class="thumb tright">
<div class="thumbinner" style="width:362px;"><a class="image" href="/wiki/File:Data_modeling_context.svg"><img alt="" class="thumbimage" data-file-height="507" data-file-width="638" height="286" src="//upload.wikimedia.org/wikipedia/commons/thumb/2/2b/Data_modeling_context.svg/360px-Data_modeling_context.svg.png" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/2/2b/Data_modeling_context.svg/540px-Data_modeling_context.svg.png 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/2/2b/Data_modeling_context.svg/720px-Data_modeling_context.svg.png 2x" width="360"/></a>
<div class="thumbcaption">
<div class="magnify"><a class="internal" href="/wiki/File:Data_modeling_context.svg" title="Enlarge"></a></div>
Overview of data modeling context: Data model is based on Data, Data relationship, Data semantic and Data constraint. A data model provides the details of <a href="/wiki/Information" title="Information">information</a> to be stored, and is of primary use when the final product is the generation of computer <a class="mw-redirect" href="/wiki/Software_code" title="Software code">software code</a> for an application or the preparation of a <a href="/wiki/Functional_specification" title="Functional specification">functional specification</a> to aid a <a class="mw-redirect" href="/wiki/Computer_software" title="Computer software">computer software</a> make-or-buy decision. The figure is an example of the interaction between <a href="/wiki/Business_process_modeling" title="Business process modeling">process</a> and data models.<sup class="reference" id="cite_ref-SS93_1-0"><a href="#cite_note-SS93-1">[1]</a></sup></div>
</div>
</div>
<p>A data model explicitly determines the structure of data. Data models are specified in a <a href="/wiki/Data_modeling" title="Data modeling">data modeling</a> notation, which is often graphical in form.<sup class="reference" id="cite_ref-MRM99_2-0"><a href="#cite_note-MRM99-2">[2]</a></sup></p>
<p>A data model can be sometimes referred to as a <a href="/wiki/Data_structure" title="Data structure">data structure</a>, especially in the context of <a href="/wiki/Programming_language" title="Programming language">programming languages</a>. Data models are often complemented by <a href="/wiki/Function_model" title="Function model">function models</a>, especially in the context of <a class="mw-redirect" href="/wiki/Enterprise_model" title="Enterprise model">enterprise models</a>.</p>
<p></p>

<h2><span class="mw-headline" id="Overview">Overview</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Data_model&amp;action=edit&amp;section=1" title="Edit section: Overview">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Managing large quantities of structured and <a href="/wiki/Unstructured_data" title="Unstructured data">unstructured data</a> is a primary function of <a href="/wiki/Information_system" title="Information system">information systems</a>. Data models describe the structure, manipulation and integrity aspects of the data stored in data management systems such as relational databases. They typically do not describe unstructured data, such as <a href="/wiki/Word_processor" title="Word processor">word processing</a> documents, <a href="/wiki/Email" title="Email">email messages</a>, pictures, digital audio, and video.</p>
<h3><span class="mw-headline" id="The_role_of_data_models">The role of data models</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Data_model&amp;action=edit&amp;section=2" title="Edit section: The role of data models">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="thumb tright">
<div class="thumbinner" style="width:322px;"><a class="image" href="/wiki/File:3-4_Data_model_roles.svg"><img alt="" class="thumbimage" data-file-height="477" data-file-width="741" height="206" src="//upload.wikimedia.org/wikipedia/commons/thumb/9/93/3-4_Data_model_roles.svg/320px-3-4_Data_model_roles.svg.png" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/9/93/3-4_Data_model_roles.svg/480px-3-4_Data_model_roles.svg.png 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/9/93/3-4_Data_model_roles.svg/640px-3-4_Data_model_roles.svg.png 2x" width="320"/></a>
<div class="thumbcaption">
<div class="magnify"><a class="internal" href="/wiki/File:3-4_Data_model_roles.svg" title="Enlarge"></a></div>
How data models deliver benefit<sup class="reference" id="cite_ref-MW99_3-0"><a href="#cite_note-MW99-3">[3]</a></sup></div>
</div>
</div>
<p>The main aim of data models is to support the development of <a href="/wiki/Information_system" title="Information system">information systems</a> by providing the definition and format of data. According to West and Fowler (1999) "if this is done consistently across systems then compatibility of data can be achieved. If the same data structures are used to store and access data then different applications can share data. The results of this are indicated above. However, systems and interfaces often cost more than they should, to build, operate, and maintain. They may also constrain the business rather than support it. A major cause is that the quality of the data models implemented in systems and interfaces is poor".<sup class="reference" id="cite_ref-MW99_3-1"><a href="#cite_note-MW99-3">[3]</a></sup></p>
<ul>
<li>"Business rules, specific to how things are done in a particular place, are often fixed in the structure of a data model. This means that small changes in the way business is conducted lead to large changes in computer systems and interfaces".<sup class="reference" id="cite_ref-MW99_3-2"><a href="#cite_note-MW99-3">[3]</a></sup></li>
<li>"Entity types are often not identified, or incorrectly identified. This can lead to replication of data, data structure, and functionality, together with the attendant costs of that duplication in development and maintenance".<sup class="reference" id="cite_ref-MW99_3-3"><a href="#cite_note-MW99-3">[3]</a></sup></li>
<li>"Data models for different systems are arbitrarily different. The result of this is that complex interfaces are required between systems that share data. These interfaces can account for between 25-70% of the cost of current systems".<sup class="reference" id="cite_ref-MW99_3-4"><a href="#cite_note-MW99-3">[3]</a></sup></li>
<li>"Data cannot be shared electronically with customers and suppliers, because the structure and meaning of data has not been standardized. For example, engineering design data and drawings for process plant are still sometimes exchanged on paper".<sup class="reference" id="cite_ref-MW99_3-5"><a href="#cite_note-MW99-3">[3]</a></sup></li>
</ul>
<p>The reason for these problems is a lack of standards that will ensure that data models will both meet business needs and be consistent.<sup class="reference" id="cite_ref-MW99_3-6"><a href="#cite_note-MW99-3">[3]</a></sup></p>
<p>A data model explicitly determines the structure of data or structured data. Typical applications of data models include database models, design of information systems, and enabling exchange of data. Usually data models are specified in a data modeling language.[3]</p>
<p>A data model can be sometimes referred to as a data structure, especially in the context of programming languages. Data models are often complemented by function models, especially in the context of enterprise models.</p>
<h3><span class="mw-headline" id="Three_perspectives">Three perspectives</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Data_model&amp;action=edit&amp;section=3" title="Edit section: Three perspectives">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="thumb tright">
<div class="thumbinner" style="width:322px;"><a class="image" href="/wiki/File:4-2_ANSI-SPARC_three_level_architecture.svg"><img alt="" class="thumbimage" data-file-height="431" data-file-width="653" height="211" src="//upload.wikimedia.org/wikipedia/commons/thumb/8/84/4-2_ANSI-SPARC_three_level_architecture.svg/320px-4-2_ANSI-SPARC_three_level_architecture.svg.png" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/8/84/4-2_ANSI-SPARC_three_level_architecture.svg/480px-4-2_ANSI-SPARC_three_level_architecture.svg.png 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/8/84/4-2_ANSI-SPARC_three_level_architecture.svg/640px-4-2_ANSI-SPARC_three_level_architecture.svg.png 2x" width="320"/></a>
<div class="thumbcaption">
<div class="magnify"><a class="internal" href="/wiki/File:4-2_ANSI-SPARC_three_level_architecture.svg" title="Enlarge"></a></div>
The ANSI/SPARC <a class="mw-redirect" href="/wiki/Three_schema_approach" title="Three schema approach">three level architecture</a>. This shows that a data model can be an external model (or view), a conceptual model, or a physical model. This is not the only way to look at data models, but it is a useful way, particularly when comparing models.<sup class="reference" id="cite_ref-MW99_3-7"><a href="#cite_note-MW99-3">[3]</a></sup></div>
</div>
</div>
<p>A data model <i>instance</i> may be one of three kinds according to <a class="mw-redirect" href="/wiki/ANSI" title="ANSI">ANSI</a> in 1975:<sup class="reference" id="cite_ref-4"><a href="#cite_note-4">[4]</a></sup></p>
<ul>
<li><a class="mw-redirect" href="/wiki/Conceptual_data_model" title="Conceptual data model">Conceptual data model</a> : describes the semantics of a domain, being the scope of the model. For example, it may be a model of the interest area of an organization or industry. This consists of entity classes, representing kinds of things of significance in the domain, and relationship assertions about associations between pairs of entity classes. A conceptual schema specifies the kinds of facts or propositions that can be expressed using the model. In that sense, it defines the allowed expressions in an artificial 'language' with a scope that is limited by the scope of the model.</li>
<li><a href="/wiki/Logical_data_model" title="Logical data model">Logical data model</a> : describes the semantics, as represented by a particular data manipulation technology. This consists of descriptions of tables and columns, object oriented classes, and XML tags, among other things.</li>
<li><a href="/wiki/Physical_data_model" title="Physical data model">Physical data model</a> : describes the physical means by which data are stored. This is concerned with partitions, CPUs, tablespaces, and the like.</li>
</ul>
<p>The significance of this approach, according to ANSI, is that it allows the three perspectives to be relatively independent of each other. Storage technology can change without affecting either the logical or the conceptual model. The table/column structure can change without (necessarily) affecting the conceptual model. In each case, of course, the structures must remain consistent with the other model. The table/column structure may be different from a direct translation of the entity classes and attributes, but it must ultimately carry out the objectives of the conceptual entity class structure. Early phases of many software development projects emphasize the design of a <a href="/wiki/Conceptual_schema" title="Conceptual schema">conceptual data model</a>. Such a design can be detailed into a <a href="/wiki/Logical_data_model" title="Logical data model">logical data model</a>. In later stages, this model may be translated into <a href="/wiki/Physical_data_model" title="Physical data model">physical data model</a>. However, it is also possible to implement a conceptual model directly.</p>
<h2><span class="mw-headline" id="History">History</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Data_model&amp;action=edit&amp;section=4" title="Edit section: History">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>One of the earliest pioneering works in modelling information systems was done by Young and Kent (1958),<sup class="reference" id="cite_ref-5"><a href="#cite_note-5">[5]</a></sup><sup class="reference" id="cite_ref-JAB07_6-0"><a href="#cite_note-JAB07-6">[6]</a></sup> who argued for "a precise and abstract way of specifying the informational and time characteristics of a <a href="/wiki/Data_processing" title="Data processing">data processing</a> problem". They wanted to create "a notation that should enable the <a href="/wiki/Systems_analyst" title="Systems analyst">analyst</a> to organize the problem around any piece of <a href="/wiki/Computer_hardware" title="Computer hardware">hardware</a>". Their work was a first effort to create an abstract specification and invariant basis for designing different alternative implementations using different hardware components. A next step in IS modelling was taken by <a href="/wiki/CODASYL" title="CODASYL">CODASYL</a>, an IT industry consortium formed in 1959, who essentially aimed at the same thing as Young and Kent: the development of "a proper structure for machine independent problem definition language, at the system level of data processing". This led to the development of a specific IS <a href="/wiki/Information_algebra" title="Information algebra">information algebra</a>.<sup class="reference" id="cite_ref-JAB07_6-1"><a href="#cite_note-JAB07-6">[6]</a></sup></p>
<p>In the 1960s data modeling gained more significance with the initiation of the <a href="/wiki/Management_information_system" title="Management information system">management information system</a> (MIS) concept. According to Leondes (2002), "during that time, the information system provided the data and information for management purposes. The first generation <a class="mw-redirect" href="/wiki/Database_system" title="Database system">database system</a>, called <a href="/wiki/Integrated_Data_Store" title="Integrated Data Store">Integrated Data Store</a> (IDS), was designed by <a href="/wiki/Charles_Bachman" title="Charles Bachman">Charles Bachman</a> at General Electric. Two famous database models, the <a class="mw-redirect" href="/wiki/Network_data_model" title="Network data model">network data model</a> and the <a class="mw-redirect" href="/wiki/Hierarchical_data_model" title="Hierarchical data model">hierarchical data model</a>, were proposed during this period of time".<sup class="reference" id="cite_ref-7"><a href="#cite_note-7">[7]</a></sup> Towards the end of the 1960s, <a href="/wiki/Edgar_F._Codd" title="Edgar F. Codd">Edgar F. Codd</a> worked out his theories of data arrangement, and proposed the <a href="/wiki/Relational_model" title="Relational model">relational model</a> for database management based on <a href="/wiki/First-order_logic" title="First-order logic">first-order predicate logic</a>.<sup class="reference" id="cite_ref-8"><a href="#cite_note-8">[8]</a></sup></p>
<p>In the 1970s <a class="mw-redirect" href="/wiki/Entity_relationship_model" title="Entity relationship model">entity relationship modeling</a> emerged as a new type of conceptual data modeling, originally proposed in 1976 by <a href="/wiki/Peter_Chen" title="Peter Chen">Peter Chen</a>. Entity relationship models were being used in the first stage of <a href="/wiki/Information_system" title="Information system">information system</a> design during the <a href="/wiki/Requirements_analysis" title="Requirements analysis">requirements analysis</a> to describe information needs or the type of <a href="/wiki/Information" title="Information">information</a> that is to be stored in a <a href="/wiki/Database" title="Database">database</a>. This technique can describe any <a class="mw-redirect" href="/wiki/Ontology_(computer_science)" title="Ontology (computer science)">ontology</a>, i.e., an overview and classification of concepts and their relationships, for a certain <a class="mw-redirect" href="/wiki/Universe_of_discourse" title="Universe of discourse">area of interest</a>.</p>
<p>In the 1970s <a href="/wiki/G.M._Nijssen" title="G.M. Nijssen">G.M. Nijssen</a> developed "Natural Language Information Analysis Method" (NIAM) method, and developed this in the 1980s in cooperation with <a href="/wiki/Terry_Halpin" title="Terry Halpin">Terry Halpin</a> into <a class="mw-redirect" href="/wiki/Object-Role_Modeling" title="Object-Role Modeling">Object-Role Modeling</a> (ORM).</p>
<p>Bill Kent, in his 1978 book <i>Data and Reality,</i><sup class="reference" id="cite_ref-9"><a href="#cite_note-9">[9]</a></sup> compared a data model to a map of a territory, emphasizing that in the real world, "highways are not painted red, rivers don't have county lines running down the middle, and you can't see contour lines on a mountain". In contrast to other researchers who tried to create models that were mathematically clean and elegant, Kent emphasized the essential messiness of the real world, and the task of the data modeller to create order out of chaos without excessively distorting the truth.</p>
<p>In the 1980s, according to Jan L. Harrington (2000), "the development of the <a class="mw-redirect" href="/wiki/Object-oriented" title="Object-oriented">object-oriented</a> paradigm brought about a fundamental change in the way we look at data and the procedures that operate on data. Traditionally, data and procedures have been stored separately: the data and their relationship in a database, the procedures in an application program. Object orientation, however, combined an entity's procedure with its data."<sup class="reference" id="cite_ref-JLH00_10-0"><a href="#cite_note-JLH00-10">[10]</a></sup></p>
<h2><span class="mw-headline" id="Types_of_data_models">Types of data models</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Data_model&amp;action=edit&amp;section=5" title="Edit section: Types of data models">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<h3><span class="mw-headline" id="Database_model">Database model</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Data_model&amp;action=edit&amp;section=6" title="Edit section: Database model">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="hatnote" role="note">Main article: <a href="/wiki/Database_model" title="Database model">Database model</a></div>
<p>A database model is a specification describing how a database is structured and used.</p>
<p>Several such models have been suggested. Common models include:</p>
<dl>
<dt><a href="/wiki/Flat_file_database" title="Flat file database">Flat model</a></dt>
<dd>This may not strictly qualify as a data model. The flat (or table) model consists of a single, two-dimensional array of data elements, where all members of a given column are assumed to be similar values, and all members of a row are assumed to be related to one another.</dd>
<dt><a class="mw-redirect" href="/wiki/Hierarchical_model" title="Hierarchical model">Hierarchical model</a></dt>
<dd>The hierarchical model is similar to the network model except that links in the hierarchical model form a tree structure,while the network model allows arbitrary graphs.</dd>
<dt><a href="/wiki/Network_model" title="Network model">Network model</a></dt>
<dd>This model organizes data using two fundamental constructs, called records and sets. Records contain fields, and sets define one-to-many relationships between records: one owner, many members.The network data model is an abstraction of the design concept used in the implementation of databases.</dd>
<dt><a href="/wiki/Relational_model" title="Relational model">Relational model</a></dt>
<dd>is a database model based on first-order predicate logic. Its core idea is to describe a database as a collection of predicates over a finite set of predicate variables, describing constraints on the possible values and combinations of values.The power of the relational data model lies in its mathematical foundations and a simple user-level paradigm.</dd>
<dt><a class="mw-redirect" href="/wiki/Object-relational_model" title="Object-relational model">Object-relational model</a></dt>
<dd>Similar to a relational database model, but objects, classes and inheritance are directly supported in <a href="/wiki/Database_schema" title="Database schema">database schemas</a> and in the query language.</dd>
<dt><a href="/wiki/Star_schema" title="Star schema">Star schema</a></dt>
<dd>The simplest style of data warehouse schema. The star schema consists of a few "fact tables" (possibly only one, justifying the name) referencing any number of "dimension tables". The star schema is considered an important special case of the <a href="/wiki/Snowflake_schema" title="Snowflake schema">snowflake schema</a>.</dd>
</dl>

<h3><span class="mw-headline" id="Data_structure_diagram">Data structure diagram</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Data_model&amp;action=edit&amp;section=7" title="Edit section: Data structure diagram">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="hatnote" role="note">Main article: <a href="/wiki/Data_structure_diagram" title="Data structure diagram">Data structure diagram</a></div>
<div class="thumb tright">
<div class="thumbinner" style="width:242px;"><a class="image" href="/wiki/File:Aggregate_Data_Structure_Diagram.jpg"><img alt="" class="thumbimage" data-file-height="1407" data-file-width="850" height="397" src="//upload.wikimedia.org/wikipedia/commons/thumb/c/c4/Aggregate_Data_Structure_Diagram.jpg/240px-Aggregate_Data_Structure_Diagram.jpg" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/c/c4/Aggregate_Data_Structure_Diagram.jpg/360px-Aggregate_Data_Structure_Diagram.jpg 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/c/c4/Aggregate_Data_Structure_Diagram.jpg/480px-Aggregate_Data_Structure_Diagram.jpg 2x" width="240"/></a>
<div class="thumbcaption">
<div class="magnify"><a class="internal" href="/wiki/File:Aggregate_Data_Structure_Diagram.jpg" title="Enlarge"></a></div>
Example of a Data Structure Diagram</div>
</div>
</div>
<p>A data structure diagram (DSD) is a <a href="/wiki/Diagram" title="Diagram">diagram</a> and data model used to describe <a href="/wiki/Conceptual_schema" title="Conceptual schema">conceptual data models</a> by providing graphical notations which document <a href="/wiki/Entity" title="Entity">entities</a> and their <a href="/wiki/Relational_model" title="Relational model">relationships</a>, and the <a class="mw-redirect" href="/wiki/Integrity_constraints" title="Integrity constraints">constraints</a> that bind them. The basic graphic elements of DSDs are <a href="/wiki/Box" title="Box">boxes</a>, representing entities, and <a href="/wiki/Arrow" title="Arrow">arrows</a>, representing relationships. Data structure diagrams are most useful for documenting complex data entities.</p>
<p>Data structure diagrams are an extension of the <a class="mw-redirect" href="/wiki/Entity-relationship_model" title="Entity-relationship model">entity-relationship model</a> (ER model). In DSDs, <a href="/wiki/Attribute_(computing)" title="Attribute (computing)">attributes</a> are specified inside the entity boxes rather than outside of them, while relationships are drawn as boxes composed of attributes which specify the constraints that bind entities together. The E-R model, while robust, doesn't provide a way to specify the constraints between relationships, and becomes visually cumbersome when representing entities with several attributes. DSDs differ from the ER model in that the ER model focuses on the relationships between different entities, whereas DSDs focus on the relationships of the elements within an entity and enable users to fully see the links and relationships between each entity.</p>
<p>There are several styles for representing data structure diagrams, with the notable difference in the manner of defining <a href="/wiki/Cardinality_(data_modeling)" title="Cardinality (data modeling)">cardinality</a>. The choices are between arrow heads, inverted arrow heads (<a class="mw-redirect" href="/wiki/Entity-relationship_model" title="Entity-relationship model">crow's feet</a>), or numerical representation of the cardinality.</p>

<h3><span class="mw-headline" id="Entity-relationship_model">Entity-relationship model</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Data_model&amp;action=edit&amp;section=8" title="Edit section: Entity-relationship model">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="hatnote" role="note">Main article: <a class="mw-redirect" href="/wiki/Entity-relationship_model" title="Entity-relationship model">Entity-relationship model</a></div>
<p>An entity-relationship model (ERM), sometimes referred to as an entity-relationship diagram (ERD), is an abstract <a href="/wiki/Conceptual_schema" title="Conceptual schema">conceptual data model</a> (or <a href="/wiki/Semantic_data_model" title="Semantic data model">semantic data model</a>) used in <a href="/wiki/Software_engineering" title="Software engineering">software engineering</a> to represent structured data. There are several notations used for ERMs.</p>
<h3><span class="mw-headline" id="Geographic_data_model">Geographic data model</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Data_model&amp;action=edit&amp;section=9" title="Edit section: Geographic data model">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="hatnote" role="note">Main article: <a href="/wiki/Data_model_(GIS)" title="Data model (GIS)">Data model (GIS)</a></div>
<p>A data model in <a href="/wiki/Geographic_information_system" title="Geographic information system">Geographic information systems</a> is a mathematical construct for representing geographic objects or surfaces as data. For example,</p>
<ul>
<li>the <a href="/wiki/Vector_graphics" title="Vector graphics">vector</a> data model represents geography as collections of points, lines, and polygons;</li>
<li>the <a href="/wiki/Raster_graphics" title="Raster graphics">raster</a> data model represent geography as cell matrixes that store numeric values;</li>
<li>and the <a href="/wiki/Triangulated_irregular_network" title="Triangulated irregular network">Triangulated irregular network</a> (TIN) data model represents geography as sets of contiguous, nonoverlapping triangles.<sup class="reference" id="cite_ref-12"><a href="#cite_note-12">[12]</a></sup></li>
</ul>

<h3><span class="mw-headline" id="Generic_data_model">Generic data model</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Data_model&amp;action=edit&amp;section=10" title="Edit section: Generic data model">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="hatnote" role="note">Main article: <a href="/wiki/Generic_data_model" title="Generic data model">Generic data model</a></div>
<p>Generic data models are generalizations of conventional data models. They define standardised general relation types, together with the kinds of things that may be related by such a relation type. Generic data models are developed as an approach to solve some shortcomings of conventional data models. For example, different modelers usually produce different conventional data models of the same domain. This can lead to difficulty in bringing the models of different people together and is an obstacle for data exchange and data integration. Invariably, however, this difference is attributable to different levels of abstraction in the models and differences in the kinds of facts that can be instantiated (the semantic expression capabilities of the models). The modelers need to communicate and agree on certain elements which are to be rendered more concretely, in order to make the differences less significant.</p>
<h3><span class="mw-headline" id="Semantic_data_model">Semantic data model</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Data_model&amp;action=edit&amp;section=11" title="Edit section: Semantic data model">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="hatnote" role="note">Main article: <a href="/wiki/Semantic_data_model" title="Semantic data model">Semantic data model</a></div>
<div class="thumb tright">
<div class="thumbinner" style="width:322px;"><a class="image" href="/wiki/File:A2_4_Semantic_Data_Models.svg"><img alt="" class="thumbimage" data-file-height="333" data-file-width="608" height="175" src="//upload.wikimedia.org/wikipedia/commons/thumb/5/58/A2_4_Semantic_Data_Models.svg/320px-A2_4_Semantic_Data_Models.svg.png" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/5/58/A2_4_Semantic_Data_Models.svg/480px-A2_4_Semantic_Data_Models.svg.png 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/5/58/A2_4_Semantic_Data_Models.svg/640px-A2_4_Semantic_Data_Models.svg.png 2x" width="320"/></a>
<div class="thumbcaption">
<div class="magnify"><a class="internal" href="/wiki/File:A2_4_Semantic_Data_Models.svg" title="Enlarge"></a></div>
Semantic data models<sup class="reference" id="cite_ref-FIPS184_11-1"><a href="#cite_note-FIPS184-11">[11]</a></sup></div>
</div>
</div>
<p>A semantic data model in software engineering is a technique to define the meaning of data within the context of its interrelationships with other data. A semantic data model is an abstraction which defines how the stored symbols relate to the real world.<sup class="reference" id="cite_ref-FIPS184_11-2"><a href="#cite_note-FIPS184-11">[11]</a></sup> A semantic data model is sometimes called a <a class="mw-redirect" href="/wiki/Conceptual_data_model" title="Conceptual data model">conceptual data model</a>.</p>
<p>The logical data structure of a <a class="mw-redirect" href="/wiki/Database_management_system" title="Database management system">database management system</a> (DBMS), whether <a class="mw-redirect" href="/wiki/Hierarchical_model" title="Hierarchical model">hierarchical</a>, <a href="/wiki/Network_model" title="Network model">network</a>, or <a href="/wiki/Relational_model" title="Relational model">relational</a>, cannot totally satisfy the <a href="/wiki/Requirements_analysis" title="Requirements analysis">requirements</a> for a conceptual definition of data because it is limited in scope and biased toward the implementation strategy employed by the DBMS. Therefore, the need to define data from a <a class="mw-redirect" href="/wiki/Three_schema_approach" title="Three schema approach">conceptual view</a> has led to the development of semantic data modeling techniques. That is, techniques to define the meaning of data within the context of its interrelationships with other data. As illustrated in the figure. The real world, in terms of resources, ideas, events, etc., are symbolically defined within physical data stores. A semantic data model is an abstraction which defines how the stored symbols relate to the real world. Thus, the model must be a true representation of the real world.<sup class="reference" id="cite_ref-FIPS184_11-3"><a href="#cite_note-FIPS184-11">[11]</a></sup></p>

<h3><span class="mw-headline" id="Data_architecture">Data architecture</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Data_model&amp;action=edit&amp;section=13" title="Edit section: Data architecture">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="hatnote" role="note">Main article: <a href="/wiki/Data_architecture" title="Data architecture">Data architecture</a></div>
<p>Data architecture is the design of data for use in defining the target state and the subsequent planning needed to hit the target state. It is usually one of several <a href="/wiki/Architecture_domain" title="Architecture domain">architecture domains</a> that form the pillars of an <a href="/wiki/Enterprise_architecture" title="Enterprise architecture">enterprise architecture</a> or <a href="/wiki/Solution_architecture" title="Solution architecture">solution architecture</a>.</p>
<p>A data architecture describes the data structures used by a business and/or its applications. There are descriptions of data in storage and data in motion; descriptions of data stores, data groups and data items; and mappings of those data artifacts to data qualities, applications, locations etc.</p>
<p>Essential to realizing the target state, Data architecture describes how data is processed, stored, and utilized in a given system. It provides criteria for data processing operations that make it possible to design data flows and also control the flow of data in the system.</p>
<h3><span class="mw-headline" id="Data_modeling">Data modeling</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Data_model&amp;action=edit&amp;section=14" title="Edit section: Data modeling">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="hatnote" role="note">Main article: <a href="/wiki/Data_modeling" title="Data modeling">Data modeling</a></div>
<div class="thumb tright">

<p>Data modeling in <a href="/wiki/Software_engineering" title="Software engineering">software engineering</a> is the process of creating a data model by applying formal data model descriptions using data modeling techniques. Data modeling is a technique for defining business <a href="/wiki/Requirement" title="Requirement">requirements</a> for a database. It is sometimes called <i>database modeling</i> because a data model is eventually implemented in a database.<sup class="reference" id="cite_ref-WBD04_14-0"><a href="#cite_note-WBD04-14">[14]</a></sup></p>
<p>The figure illustrates the way data models are developed and used today. A <a class="mw-redirect" href="/wiki/Conceptual_data_model" title="Conceptual data model">conceptual data model</a> is developed based on the data <a class="mw-redirect" href="/wiki/Requirements" title="Requirements">requirements</a> for the application that is being developed, perhaps in the context of an <a href="/wiki/Activity_diagram" title="Activity diagram">activity model</a>. The data model will normally consist of entity types, attributes, relationships, integrity rules, and the definitions of those objects. This is then used as the start point for interface or <a href="/wiki/Database_design" title="Database design">database design</a>.<sup class="reference" id="cite_ref-MW99_3-8"><a href="#cite_note-MW99-3">[3]</a></sup></p>
<h3><span class="mw-headline" id="Data_properties">Data properties</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Data_model&amp;action=edit&amp;section=15" title="Edit section: Data properties">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Some important properties of data for which requirements need to be met are:</p>
<ul>
<li>definition-related properties<sup class="reference" id="cite_ref-MW99_3-9"><a href="#cite_note-MW99-3">[3]</a></sup>
<ul>
<li><i>relevance</i>: the usefulness of the data in the context of your business.</li>
<li><i>clarity</i>: the availability of a clear and shared definition for the data.</li>
<li><i>consistency</i>: the compatibility of the same type of data from different sources.</li>
</ul>
</li>
</ul>

<li>content-related properties
<ul>

<h3><span class="mw-headline" id="Data_organization">Data organization</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Data_model&amp;action=edit&amp;section=16" title="Edit section: Data organization">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Another kind of data model describes how to organize data using a <a class="mw-redirect" href="/wiki/Database_management_system" title="Database management system">database management system</a> or other data management technology. It describes, for example, relational tables and columns or object-oriented classes and attributes. Such a data model is sometimes referred to as the <i><a href="/wiki/Physical_data_model" title="Physical data model">physical data model</a></i>, but in the original ANSI three schema architecture, it is called "logical". In that architecture, the physical model describes the storage media (cylinders, tracks, and tablespaces). Ideally, this model is derived from the more conceptual data model described above. It may differ, however, to account for constraints like processing capacity and usage patterns.</p>
<p>While <i>data analysis</i> is a common term for data modeling, the activity actually has more in common with the ideas and methods of <i><a class="extiw" href="https://en.wiktionary.org/wiki/synthesis" title="wiktionary:synthesis">synthesis</a></i> (inferring general concepts from particular instances) than it does with <i><a class="extiw" href="https://en.wiktionary.org/wiki/Analysis" title="wiktionary:Analysis">analysis</a></i> (identifying component concepts from more general ones). {<i>Presumably we call ourselves <a href="/wiki/Systems_analyst" title="Systems analyst">systems analysts</a> because no one can say <a href="/wiki/Systems_analyst" title="Systems analyst">systems synthesists</a>.</i>} Data modeling strives to bring the data structures of interest together into a cohesive, inseparable, whole by eliminating unnecessary data redundancies and by relating data structures with <a href="/wiki/Relational_model" title="Relational model">relationships</a>.</p>
<p>A different approach is to use <a href="/wiki/Adaptive_system" title="Adaptive system">adaptive systems</a> such as <a href="/wiki/Artificial_neural_network" title="Artificial neural network">artificial neural networks</a> that can autonomously create implicit models of data.</p>
<h3><span class="mw-headline" id="Data_structure">Data structure</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Data_model&amp;action=edit&amp;section=17" title="Edit section: Data structure">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="hatnote" role="note">Main article: <a href="/wiki/Data_structure" title="Data structure">Data structure</a></div>
<div class="thumb tright">
<div class="thumbinner" style="width:242px;"><a class="image" href="/wiki/File:Binary_tree.svg"><img alt="" class="thumbimage" data-file-height="250" data-file-width="300" height="200" src="//upload.wikimedia.org/wikipedia/commons/thumb/f/f7/Binary_tree.svg/240px-Binary_tree.svg.png" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/f/f7/Binary_tree.svg/360px-Binary_tree.svg.png 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/f/f7/Binary_tree.svg/480px-Binary_tree.svg.png 2x" width="240"/></a>
<div class="thumbcaption">
<div class="magnify"><a class="internal" href="/wiki/File:Binary_tree.svg" title="Enlarge"></a></div>

<p>A data structure is a way of storing data in a computer so that it can be used efficiently. It is an organization of mathematical and logical concepts of data. Often a carefully chosen data structure will allow the most <a href="/wiki/Algorithmic_efficiency" title="Algorithmic efficiency">efficient</a> <a href="/wiki/Algorithm" title="Algorithm">algorithm</a> to be used. The choice of the data structure often begins from the choice of an <a href="/wiki/Abstract_data_type" title="Abstract data type">abstract data type</a>.</p>
<p>A data model describes the structure of the data within a given domain and, by implication, the underlying structure of that domain itself. This means that a data model in fact specifies a dedicated <i>grammar</i> for a dedicated artificial language for that domain. A data model represents classes of entities (kinds of things) about which a company wishes to hold information, the attributes of that information, and relationships among those entities and (often implicit) relationships among those attributes. The model describes the organization of the data to some extent irrespective of how data might be represented in a computer system.</p>
<p>The entities represented by a data model can be the tangible entities, but models that include such concrete entity classes tend to change over time. Robust data models often identify <a href="/wiki/Abstraction" title="Abstraction">abstractions</a> of such entities. For example, a data model might include an entity class called "Person", representing all the people who interact with an organization. Such an <a class="mw-redirect" href="/wiki/Abstract_entity" title="Abstract entity">abstract entity</a> class is typically more appropriate than ones called "Vendor" or "Employee", which identify specific roles played by those people.</p>

<h3><span class="mw-headline" id="Data_model_theory">Data model theory</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Data_model&amp;action=edit&amp;section=18" title="Edit section: Data model theory">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>The term data model can have two meanings:<sup class="reference" id="cite_ref-Beynon-Davies_P._2004_15-0"><a href="#cite_note-Beynon-Davies_P._2004-15">[15]</a></sup></p>
<ol>
<li>A data model <i>theory</i>, i.e. a formal description of how data may be structured and accessed.</li>
<li>A data model <i>instance</i>, i.e. applying a data model <i>theory</i> to create a practical data model <i>instance</i> for some particular application.</li>
</ol>
<p>A data model theory has three main components:<sup class="reference" id="cite_ref-Beynon-Davies_P._2004_15-1"><a href="#cite_note-Beynon-Davies_P._2004-15">[15]</a></sup></p>
<ul>
<li>The structural part: a collection of data structures which are used to create databases representing the entities or objects modeled by the database.</li>
<li>The integrity part: a collection of rules governing the constraints placed on these data structures to ensure structural integrity.</li>
<li>The manipulation part: a collection of operators which can be applied to the data structures, to update and query the data contained in the database.</li>
</ul>
<p>For example, in the <a href="/wiki/Relational_model" title="Relational model">relational model</a>, the structural part is based on a modified concept of the <a class="mw-redirect" href="/wiki/Relation_(mathematics)" title="Relation (mathematics)">mathematical relation</a>; the integrity part is expressed in <a href="/wiki/First-order_logic" title="First-order logic">first-order logic</a> and the manipulation part is expressed using the <a href="/wiki/Relational_algebra" title="Relational algebra">relational algebra</a>, <a class="mw-redirect" href="/wiki/Tuple_calculus" title="Tuple calculus">tuple calculus</a> and <a class="mw-redirect" href="/wiki/Domain_calculus" title="Domain calculus">domain calculus</a>.</p>
<p>A data model instance is created by applying a data model theory. This is typically done to solve some business enterprise requirement. Business requirements are normally captured by a semantic <a href="/wiki/Logical_data_model" title="Logical data model">logical data model</a>. This is transformed into a physical data model instance from which is generated a physical database. For example, a data modeler may use a data modeling tool to create an <a class="mw-redirect" href="/wiki/Entity-relationship_model" title="Entity-relationship model">entity-relationship model</a> of the corporate data repository of some business enterprise. This model is transformed into a <a href="/wiki/Relational_model" title="Relational model">relational model</a>, which in turn generates a <a href="/wiki/Relational_database" title="Relational database">relational database</a>.</p>

<p>A data flow diagram (DFD) is a graphical representation of the "flow" of data through an <a href="/wiki/Information_system" title="Information system">information system</a>. It differs from the <a href="/wiki/Flowchart" title="Flowchart">flowchart</a> as it shows the <i>data</i> flow instead of the <i>control</i> flow of the program. A data flow diagram can also be used for the <a href="/wiki/Data_visualization" title="Data visualization">visualization</a> of <a href="/wiki/Data_processing" title="Data processing">data processing</a> (structured design). Data flow diagrams were invented by <a href="/wiki/Larry_Constantine" title="Larry Constantine">Larry Constantine</a>, the original developer of structured design,<sup class="reference" id="cite_ref-18"><a href="#cite_note-18">[18]</a></sup> based on Martin and Estrin's "data flow graph" model of computation.</p>
<p>It is common practice to draw a <a class="mw-redirect" href="/wiki/System_Context_Diagram" title="System Context Diagram">context-level Data flow diagram</a> first which shows the interaction between the system and outside entities. The <b>DFD</b> is designed to show how a system is divided into smaller portions and to highlight the flow of data between those parts. This context-level Data flow diagram is then "exploded" to show more detail of the system being modeled</p>
<h3><span class="mw-headline" id="Information_model">Information model</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Data_model&amp;action=edit&amp;section=22" title="Edit section: Information model">edit</a><span class="mw-editsection-bracket">]</span></span></h3>

Example of an <a href="/wiki/EXPRESS_(data_modeling_language)" title="EXPRESS (data modeling language)">EXPRESS G</a> <a href="/wiki/Information_model" title="Information model">Information model</a></div>

<p>An Information model is not a type of data model, but more or less an alternative model. Within the field of software engineering both a data model and an information model can be abstract, formal representations of entity types that includes their properties, relationships and the operations that can be performed on them. The entity types in the model may be kinds of real-world objects, such as devices in a network, or they may themselves be abstract, such as for the entities used in a billing system. Typically, they are used to model a constrained domain that can be described by a closed set of entity types, properties, relationships and operations.</p>
<p>According to Lee (1999)<sup class="reference" id="cite_ref-Lee99_19-0"><a href="#cite_note-Lee99-19">[19]</a></sup> an information model is a representation of concepts, relationships, constraints, rules, and <a href="/wiki/Operation_(mathematics)" title="Operation (mathematics)">operations</a> to specify <a href="/wiki/Semantic_data_model" title="Semantic data model">data semantics</a> for a chosen domain of discourse. It can provide sharable, stable, and organized structure of information requirements for the domain context.<sup class="reference" id="cite_ref-Lee99_19-1"><a href="#cite_note-Lee99-19">[19]</a></sup> More in general the term <i>information model</i> is used for models of individual things, such as facilities, buildings, process plants, etc. In those cases the concept is specialised to <a class="mw-redirect" href="/wiki/Facility_Information_Model" title="Facility Information Model">Facility Information Model</a>, <a href="/wiki/Building_information_modeling" title="Building information modeling">Building Information Model</a>, Plant Information Model, etc. Such an information model is an integration of a model of the facility with the data and documents about the facility.</p>
<p>An information model provides formalism to the description of a problem domain without constraining how that description is mapped to an actual implementation in software. There may be many mappings of the information model. Such mappings are called data models, irrespective of whether they are <a href="/wiki/Object_model" title="Object model">object models</a> (e.g. using <a href="/wiki/Unified_Modeling_Language" title="Unified Modeling Language">UML</a>), <a class="mw-redirect" href="/wiki/Entity-relationship_model" title="Entity-relationship model">entity relationship models</a> or <a href="/wiki/XML_schema" title="XML schema">XML schemas</a>.</p>

<h3><span class="mw-headline" id="Object_model">Object model</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Data_model&amp;action=edit&amp;section=23" title="Edit section: Object model">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="hatnote" role="note">Main article: <a href="/wiki/Object_model" title="Object model">Object model</a></div>
<p>An object model in computer science is a collection of objects or classes through which a program can examine and manipulate some specific parts of its world. In other words, the object-oriented interface to some service or system. Such an interface is said to be the <i>object model of</i> the represented service or system. For example, the <a href="/wiki/Document_Object_Model" title="Document Object Model">Document Object Model (DOM)</a> <a class="external autonumber" href="http://www.w3.org/DOM/" rel="nofollow">[1]</a> is a collection of objects that represent a <a href="/wiki/Web_page" title="Web page">page</a> in a <a href="/wiki/Web_browser" title="Web browser">web browser</a>, used by <a href="/wiki/Scripting_language" title="Scripting language">script</a> programs to examine and dynamically change the page. There is a <a href="/wiki/Microsoft_Excel" title="Microsoft Excel">Microsoft Excel</a> object model<sup class="reference" id="cite_ref-20"><a href="#cite_note-20">[20]</a></sup> for controlling Microsoft Excel from another program, and the <a href="/wiki/ASCOM_(standard)" title="ASCOM (standard)">ASCOM</a> Telescope Driver<sup class="reference" id="cite_ref-21"><a href="#cite_note-21">[21]</a></sup> is an object model for controlling an astronomical telescope.</p>
<p>In <a href="/wiki/Computing" title="Computing">computing</a> the term <i>object model</i> has a distinct second meaning of the general properties of <a href="/wiki/Object_(computer_science)" title="Object (computer science)">objects</a> in a specific computer <a href="/wiki/Programming_language" title="Programming language">programming language</a>, technology, notation or <a href="/wiki/Methodology" title="Methodology">methodology</a> that uses them. For example, the <i><a href="/wiki/Java_(programming_language)" title="Java (programming language)">Java</a> object model</i>, the <i><a href="/wiki/Component_Object_Model" title="Component Object Model">COM</a> object model</i>, or <i>the object model of <a href="/wiki/Object-modeling_technique" title="Object-modeling technique">OMT</a></i>. Such object models are usually defined using concepts such as <a class="mw-redirect" href="/wiki/Class_(computer_science)" title="Class (computer science)">class</a>, <a class="mw-redirect" href="/wiki/Message_(computer_science)" title="Message (computer science)">message</a>, <a class="mw-redirect" href="/wiki/Inheritance_(computer_science)" title="Inheritance (computer science)">inheritance</a>, <a class="mw-redirect" href="/wiki/Polymorphism_in_object-oriented_programming" title="Polymorphism in object-oriented programming">polymorphism</a>, and <a href="/wiki/Information_hiding" title="Information hiding">encapsulation</a>. There is an extensive literature on formalized object models as a subset of the <a class="mw-redirect" href="/wiki/Formal_semantics_of_programming_languages" title="Formal semantics of programming languages">formal semantics of programming languages</a>.</p>
<h3><span class="mw-headline" id="Object-Role_Model">Object-Role Model</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Data_model&amp;action=edit&amp;section=24" title="Edit section: Object-Role Model">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="hatnote" role="note">Main article: <a class="mw-redirect" href="/wiki/Object-Role_Modeling" title="Object-Role Modeling">Object-Role Modeling</a></div>
<div class="thumb tright">

<p>Object-Role Modeling (ORM) is a method for <a class="mw-redirect" href="/wiki/Conceptual_modeling" title="Conceptual modeling">conceptual modeling</a>, and can be used as a tool for information and rules analysis.<sup class="reference" id="cite_ref-23"><a href="#cite_note-23">[23]</a></sup></p>
<p>Object-Role Modeling is a fact-oriented method for performing <a href="/wiki/Systems_analysis" title="Systems analysis">systems analysis</a> at the conceptual level. The quality of a database application depends critically on its design. To help ensure correctness, clarity, adaptability and productivity, information systems are best specified first at the conceptual level, using concepts and language that people can readily understand.</p>
<p>The conceptual design may include data, process and behavioral perspectives, and the actual DBMS used to implement the design might be based on one of many logical data models (relational, hierarchic, network, object-oriented etc.).<sup class="reference" id="cite_ref-msd_24-0"><a href="#cite_note-msd-24">[24]</a></sup></p>
<h3><span class="mw-headline" id="Unified_Modeling_Language_models">Unified Modeling Language models</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Data_model&amp;action=edit&amp;section=25" title="Edit section: Unified Modeling Language models">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="hatnote" role="note">Main article: <a href="/wiki/Unified_Modeling_Language" title="Unified Modeling Language">Unified Modeling Language</a></div>
<p>The Unified Modeling Language (UML) is a standardized general-purpose <a href="/wiki/Modeling_language" title="Modeling language">modeling language</a> in the field of <a href="/wiki/Software_engineering" title="Software engineering">software engineering</a>. It is a <a href="/wiki/Visual_language" title="Visual language">graphical language</a> for visualizing, specifying, constructing, and documenting the <a href="/wiki/Artifact_(software_development)" title="Artifact (software development)">artifacts</a> of a software-intensive system. The Unified Modeling Language offers a standard way to write a system's blueprints, including:<sup class="reference" id="cite_ref-OMG00_25-0"><a href="#cite_note-OMG00-25">[25]</a></sup></p>
<ul>
<li>Conceptual things such as <a href="/wiki/Business_process" title="Business process">business processes</a> and system functions</li>
<li>Concrete things such as <a href="/wiki/Programming_language" title="Programming language">programming language</a> statements, database schemas, and</li>
<li>Reusable <a href="/wiki/Component-based_software_engineering" title="Component-based software engineering">software components</a>.</li>
</ul>
<p>UML offers a mix of <a class="mw-redirect" href="/wiki/Functional_model" title="Functional model">functional models</a>, data models, and <a href="/wiki/Database_model" title="Database model">database models</a>.</p>
