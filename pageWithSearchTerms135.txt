<h1 class="firstHeading" id="firstHeading" lang="en">Naked objects</h1>
<div class="mw-body-content" id="bodyContent">

<div class="mw-content-ltr" dir="ltr" id="mw-content-text" lang="en"><p><b>Naked objects</b> is an <a class="mw-redirect" href="/wiki/Architectural_pattern_(computer_science)" title="Architectural pattern (computer science)">architectural pattern</a> used in <a href="/wiki/Software_engineering" title="Software engineering">software engineering</a>.</p>

<h2><span class="mw-headline" id="Definition">Definition</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Naked_objects&amp;action=edit&amp;section=1" title="Edit section: Definition">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The naked objects pattern is defined by three principles:</p>
<ol>
<li><i>All</i> <a href="/wiki/Business_logic" title="Business logic">business logic</a> should be encapsulated onto the <a class="mw-redirect" href="/wiki/Business_object_(computer_science)" title="Business object (computer science)">domain objects</a>. This principle is not unique to naked objects: it is just a strong commitment to <a href="/wiki/Information_hiding" title="Information hiding">encapsulation</a>.</li>
<li>The <a href="/wiki/User_interface" title="User interface">user interface</a> should be a direct representation of the domain objects, with all user actions explicitly consist in the creating or the retrieving of domain objects and/or invoking <a class="mw-redirect" href="/wiki/Method_(computer_science)" title="Method (computer science)">methods</a> on those objects. This principle is also not unique to naked objects: it is just a specific interpretation of an <a href="/wiki/Object-oriented_user_interface" title="Object-oriented user interface">object-oriented user interface</a> (OOUI).<br/>
The original idea in the naked objects pattern arises from the combination of these two which form the third principle:</li>
<li>The user interface shall be 100% automatically created from the definition of the domain objects. This may be done using several different technologies, including <a class="mw-redirect" href="/wiki/Source_code_generation" title="Source code generation">source code generation</a>; implementations of the naked objects pattern to date have favoured the technology of <a class="mw-redirect" href="/wiki/Reflection_(computer_science)" title="Reflection (computer science)">reflection</a>.</li>
</ol>
<p>The naked objects pattern was first described formally in Richard Pawson's PhD thesis<sup class="reference" id="cite_ref-pawson_1-0"><a href="#cite_note-pawson-1">[1]</a></sup> which includes a thorough investigation of various antecedents and inspirations for the pattern including, for example, the <a href="/wiki/Morphic_(software)" title="Morphic (software)">Morphic</a> user interface.</p>
<p>Naked Objects is commonly contrasted with the <a href="/wiki/Model%E2%80%93view%E2%80%93controller" title="Model–view–controller">model–view–controller</a> pattern. However, the published version of Pawson's thesis (see References) contains a foreword by <a href="/wiki/Trygve_Reenskaug" title="Trygve Reenskaug">Trygve Reenskaug</a>, who first formulated the model–view–controller pattern, suggesting that naked objects is closer to the original intent of model–view–controller (MVC) than many of the subsequent interpretations and implementations.</p>
<h2><span class="mw-headline" id="Benefits">Benefits</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Naked_objects&amp;action=edit&amp;section=2" title="Edit section: Benefits">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Pawson's thesis claims four benefits for the pattern:</p>
<ul>
<li><b>A faster development cycle</b>, because there are fewer layers to develop. In a more conventional design, the developer must define and implement three or more separate layers: the domain object layer, the presentation layer, and the task or process scripts that connect the two. (If the naked objects pattern is combined with <a href="/wiki/Object-relational_mapping" title="Object-relational mapping">object-relational mapping</a> or an <a href="/wiki/Object_database" title="Object database">object database</a>, then it is possible to create <i>all</i> layers of the system from the domain object definitions alone; however, this does not form part of the naked objects pattern <i>per se</i>.) The thesis includes a case study comparing two different implementations of the same application: one based on a conventional '4-layer' implementation; the other using naked objects.</li>
<li><b>Greater agility</b>, referring to the ease with which an application may be altered to accommodate future changes in business requirements. In part this arises from the reduction in the number of developed layers that must be kept in synchronisation. However the claim is also made that the enforced 1:1 correspondence between the user presentation and the domain model, forces higher-quality <a href="/wiki/Object_model" title="Object model">object modelling</a>, which in turn improves the agility.</li>
<li><b>A more empowering style of user interface</b>. This benefit is really attributable to the resulting <a href="/wiki/Object-oriented_user_interface" title="Object-oriented user interface">object-oriented user interface</a> (OOUI), rather than to naked objects per se, although the argument is made that naked objects makes it much easier to conceive and to implement an OOUI.</li>
<li><b>Easier requirements analysis</b>. The argument here is that with the naked objects pattern, the domain objects form a common language between users and developers and that this common language facilitates the process of discussing requirements - because there are no other representations to discuss. Combined with the faster development cycle, it becomes possible to prototype functional applications in real time.</li>
</ul>
<h2><span class="mw-headline" id="Limitations">Limitations</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Naked_objects&amp;action=edit&amp;section=3" title="Edit section: Limitations">edit</a><span class="mw-editsection-bracket">]</span></span></h2>

<h2><span class="mw-headline" id="Practical_experience">Practical experience</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Naked_objects&amp;action=edit&amp;section=5" title="Edit section: Practical experience">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The <a href="/wiki/Department_of_Social_Protection" title="Department of Social Protection">Department of Social Protection</a> (DSP) (formerly known as the Department for Social and Family Affairs) in Ireland has built a suite of enterprise applications using the naked objects pattern. As part of its Service Delivery Modernisation (SDM) programme, the DSP designed a new enterprise architecture both to meet its planned new business requirements and to provide greater agility over the longer term. The naked objects pattern forms a key element of the SDM architecture.<sup class="reference" id="cite_ref-sdm_2-0"><a href="#cite_note-sdm-2">[2]</a></sup> In November 2002, the DSP went live with a new application to replace its existing system for the administration of child benefit. This is believed to be the first operational application of the naked objects pattern, anywhere. The DSP's experience in building this first application, including the reactions of user to the radical user interface is documented extensively in Pawson's thesis,<sup class="reference" id="cite_ref-pawson_1-1"><a href="#cite_note-pawson-1">[1]</a></sup> and more recently in a presentation at QCon London 2011.<sup class="reference" id="cite_ref-qcon_3-0"><a href="#cite_note-qcon-3">[3]</a></sup></p>
<p>One of the more striking aspects of the DSP experience was the way that the Naked Objects technique permitted re-use very actively. Once a domain object, such as a Customer, had been defined for one 'application' it could be (has been) readily adapted with the minimum of tweaking and addition for use elsewhere. This suggests that the approach could become a favourite in government circles, where re-use is seen as a powerful technique for breaking down siloed systems. The UK 'Transformational Government' policy is particularly keen to see re-use become a standard requirement of new government systems, both consuming other governmental system components and making new ones available for others to use. This re-use is often seen in terms of services, but objects could be an equally powerful approach.</p>
<p>The DSP's initial 'Naked Object Architecture' was developed by an external contractor,<sup class="reference" id="cite_ref-fujitsu_4-0"><a href="#cite_note-fujitsu-4">[4]</a></sup> but the architecture was subsequently redeveloped around the <a class="external text" href="http://www.nakedobjects.net" rel="nofollow">Naked Objects framework</a>, which now forms the basis for future application development, as confirmed in the request for tenders for a four-year programme of further applications to be built using naked objects.<sup class="reference" id="cite_ref-rft_5-0"><a href="#cite_note-rft-5">[5]</a></sup></p>
<h2><span class="mw-headline" id="Criticisms">Criticisms</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Naked_objects&amp;action=edit&amp;section=6" title="Edit section: Criticisms">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The naked objects pattern has attracted a fair amount of criticism since the first public demonstration of the idea at the <a class="external text" href="http://www.oopsla.org/2001/" rel="nofollow">OOPSLA 2001</a> conference under the banner of <a class="external text" href="http://www.informatik.uni-trier.de/~ley/db/journals/sigplan/sigplan36.html" rel="nofollow">Intriguing Technologies</a>. This criticism has typically focused on one of three main areas:</p>
<ul>
<li>The validity of aiming to encapsulate all business logic onto the domain objects. Arguments both for and against this idea may be found within the research literature for <a href="/wiki/Object-oriented_programming" title="Object-oriented programming">object-oriented programming</a> and <a href="/wiki/Domain-driven_design" title="Domain-driven design">domain-driven design</a>.</li>
<li>The applicability of object-oriented user interfaces.<sup class="reference" id="cite_ref-6"><a href="#cite_note-6">[6]</a></sup></li>
<li>The usability of generic user interfaces.</li>
</ul>
<p>None of these criticisms is unique to naked objects, but the fact that naked objects combines all three ideas gives it a higher profile.</p>
<h2><span class="mw-headline" id="Relationship_to_other_ideas">Relationship to other ideas</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Naked_objects&amp;action=edit&amp;section=7" title="Edit section: Relationship to other ideas">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The naked objects pattern has relevance to several other disciplines and/or trends, including:</p>
<dl>
<dt>Object storage mechanisms</dt>
<dd><a href="/wiki/Object-relational_mapping" title="Object-relational mapping">Object-relational mapping</a>, <a href="/wiki/Object_database" title="Object database">object databases</a>, and <a class="mw-redirect" href="/wiki/Object_persistence" title="Object persistence">object persistence</a> are all concerned with eliminating the need to write a conventional <a href="/wiki/Data_access_layer" title="Data access layer">Data access layer</a> underneath the domain objects. These patterns are complementary and potentially synergistic with the naked objects pattern, which is concerned with eliminating the need to write layers above the domain objects.</dd>
<dt><a href="/wiki/Agile_software_development" title="Agile software development">Agile software development</a></dt>
<dd>Naked objects is compatible with the trend towards agile development methodologies in many different ways, but especially to fine-grained iterative development. The DSP experience (described above) was probably also the largest application of agile software development techniques within a public-sector organisation, worldwide.<sup class="reference" id="cite_ref-PawsonWade2003_7-0"><a href="#cite_note-PawsonWade2003-7">[7]</a></sup></dd>
<dt><a href="/wiki/Domain-driven_design" title="Domain-driven design">Domain-driven design</a></dt>
<dd>Domain-driven design is the idea that an evolving domain (object) model should be used as a mechanism to help explore requirements rather than vice versa. The fact that a naked object system forces direct correspondence between the user interface and the domain model makes it easier to attempt domain-driven design, and makes the benefits more visible.<sup class="reference" id="cite_ref-book.haywood_8-0"><a href="#cite_note-book.haywood-8">[8]</a></sup></dd>
<dt><a href="/wiki/Model-driven_architecture" title="Model-driven architecture">Model-driven architecture</a> (MDA)</dt>
<dd>Although naked objects does not conform to the strict definition of MDA, it shares many of the same goals. Dan Haywood has argued that naked objects is a more effective approach to achieving those goals.<sup class="reference" id="cite_ref-mda_9-0"><a href="#cite_note-mda-9">[9]</a></sup></dd>
<dt><a href="/wiki/Restful_Objects" title="Restful Objects">Restful Objects</a></dt>
<dd>A standard for creating a <a class="mw-redirect" href="/wiki/REST" title="REST">RESTful</a> interface from a domain object model. Though the Restful Objects specification does not state that the interface must be generated reflective from the domain model, that possibility exists.</dd>
</dl>
