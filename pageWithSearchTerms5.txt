<h1 class="firstHeading" id="firstHeading" lang="en">Class (computer programming)</h1>
<div class="mw-body-content" id="bodyContent">
<div id="siteSub">From Wikipedia, the free encyclopedia</div>
<div id="contentSub"><span class="mw-redirectedfrom">  (Redirected from <a class="mw-redirect" href="/w/index.php?title=Class_(computer_science)&amp;redirect=no" title="Class (computer science)">Class (computer science)</a>)</span></div>

<p>In <a href="/wiki/Object-oriented_programming" title="Object-oriented programming">object-oriented programming</a>, a <b>class</b> is an extensible program-code-template for creating <a class="mw-redirect" href="/wiki/Object_(object-oriented_programming)" title="Object (object-oriented programming)">objects</a>, providing initial values for state (<a href="/wiki/Member_variable" title="Member variable">member variables</a>) and implementations of behavior (member functions or <a href="/wiki/Method_(computer_programming)" title="Method (computer programming)">methods</a>).<sup class="reference" id="cite_ref-FOOTNOTEGammaHelmJohnsonVlissides199514_1-0"><a href="#cite_note-FOOTNOTEGammaHelmJohnsonVlissides199514-1">[1]</a></sup><sup class="reference" id="cite_ref-FOOTNOTEBruce20022.1_Objects.2C_classes.2C_and_object_types.2C_.2F.2Fbooks.google.com.2Fbooks.3Fid.3D9NGWq3K1RwUC.26pg.3DPA18_2-0"><a href="#cite_note-FOOTNOTEBruce20022.1_Objects.2C_classes.2C_and_object_types.2C_.2F.2Fbooks.google.com.2Fbooks.3Fid.3D9NGWq3K1RwUC.26pg.3DPA18-2">[2]</a></sup> In many languages, the class name is used as the name for the class (the template itself), the name for the default <a href="/wiki/Constructor_(object-oriented_programming)" title="Constructor (object-oriented programming)">constructor</a> of the class (a <a href="/wiki/Subroutine" title="Subroutine">subroutine</a> that creates objects), and as the <a href="/wiki/Data_type" title="Data type">type</a> of objects generated by <a href="/wiki/Instance_(computer_science)" title="Instance (computer science)">instantiating</a> the class; these distinct concepts are easily conflated.<sup class="reference" id="cite_ref-FOOTNOTEBruce20022.1_Objects.2C_classes.2C_and_object_types.2C_.2F.2Fbooks.google.com.2Fbooks.3Fid.3D9NGWq3K1RwUC.26pg.3DPA18_2-1"><a href="#cite_note-FOOTNOTEBruce20022.1_Objects.2C_classes.2C_and_object_types.2C_.2F.2Fbooks.google.com.2Fbooks.3Fid.3D9NGWq3K1RwUC.26pg.3DPA18-2">[2]</a></sup></p>
<p>When an object is created by a constructor of the class, the resulting object is called an <a href="/wiki/Instance_(computer_science)" title="Instance (computer science)">instance</a> of the class, and the member variables specific to the object are called <a href="/wiki/Instance_variable" title="Instance variable">instance variables</a>, to contrast with the <a href="/wiki/Class_variable" title="Class variable">class variables</a> shared across the class.</p>
<p>In some languages, classes are only a compile-time feature (new classes cannot be declared at runtime), while in other languages classes are <a href="/wiki/First-class_citizen" title="First-class citizen">first-class citizens</a>, and are generally themselves objects (typically of type <code>Class</code> or similar). In these languages, a class that creates classes is called a <a href="/wiki/Metaclass" title="Metaclass">metaclass</a>.</p>

<h2><span class="mw-headline" id="Class_vs._type">Class vs. type</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Class_(computer_programming)&amp;action=edit&amp;section=1" title="Edit section: Class vs. type">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>In casual use, people often refer to the "class" of an object, but narrowly speaking objects have <i>type</i>: the interface, namely the types of member variables, the signatures of member functions (methods), and properties these satisfy. At the same time, a class has an implementation (specifically the implementation of the methods), and can create objects of a given type, with a given implementation.<sup class="reference" id="cite_ref-FOOTNOTEGammaHelmJohnsonVlissides199517_3-0"><a href="#cite_note-FOOTNOTEGammaHelmJohnsonVlissides199517-3">[3]</a></sup> In the terms of type theory, a class is an implementation—<wbr></wbr>​​a <i>concrete</i> <a href="/wiki/Data_structure" title="Data structure">data structure</a> and collection of subroutines—<wbr></wbr>​​while a type is an <a href="/wiki/Protocol_(object-oriented_programming)" title="Protocol (object-oriented programming)">interface</a>. Different (concrete) classes can produce objects of the same (abstract) type (depending on type system); for example, the type <code>Stack</code> might be implemented with two classes –  <code>SmallStack</code> (fast for small stacks, but scales poorly) and <code>ScalableStack</code> (scales well but high overhead for small stacks). Similarly, a given class may have several different constructors.</p>
<p>Types generally represent nouns, such as a person, place or thing, or something <a href="/wiki/Nominalization" title="Nominalization">nominalized</a>, and a class represents an implementation of these. For example, a <code>Banana</code> type might represent the properties and functionality of bananas in general, while the <code>ABCBanana</code> and <code>XYZBanana</code> classes would represent ways of producing bananas (say, banana suppliers or data structures and functions to represent and draw bananas in a video game). The <code>ABCBanana</code> class could then produce particular bananas: instances of the <code>ABCBanana</code> class would be objects of type <code>Banana</code>. Often only a single implementation of a type is given, in which case the class name is often identical with the type name.</p>
<h2><span class="mw-headline" id="Design_and_implementation">Design and implementation</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Class_(computer_programming)&amp;action=edit&amp;section=2" title="Edit section: Design and implementation">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Classes are composed from structural and behavioral constituents.<sup class="reference" id="cite_ref-FOOTNOTEGamma_et_al.199514_4-0"><a href="#cite_note-FOOTNOTEGamma_et_al.199514-4">[4]</a></sup> Programming languages that include classes as a programming construct offer support for various class-related features, and the syntax required to use these features varies greatly from one programming language to another.</p>
<h3><span class="mw-headline" id="Structure"><span id="PROPERTY"></span>Structure</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Class_(computer_programming)&amp;action=edit&amp;section=3" title="Edit section: Structure">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="thumb tright">
<div class="thumbinner" style="width:163px;"><a class="image" href="/wiki/File:Oop-uml-class-example.png"><img alt="" class="thumbimage" data-file-height="206" data-file-width="161" height="206" src="//upload.wikimedia.org/wikipedia/commons/1/18/Oop-uml-class-example.png" width="161"/></a>
<div class="thumbcaption"><a href="/wiki/Unified_Modeling_Language" title="Unified Modeling Language">UML</a> notation for classes</div>
</div>
</div>
<p>A class contains <a href="/wiki/Data_(computing)" title="Data (computing)">data</a> field descriptions (or <i>properties</i>, <i>fields</i>, <i>data members</i>, or <i>attributes</i>). These are usually field types and names that will be associated with state variables at program run time; these state variables either belong to the class or specific instances of the class. In most languages, the structure defined by the class determines the layout of the memory used by its instances. Other implementations are possible: for example, objects in <a href="/wiki/Python_(programming_language)" title="Python (programming language)">Python</a> use associative key-value containers.<sup class="reference" id="cite_ref-pythondata_model_5-0"><a href="#cite_note-pythondata_model-5">[5]</a></sup></p>
<p>Some programming languages support specification of <a href="/wiki/Class_invariant" title="Class invariant">invariants</a> as part of the definition of the class, and enforce them through the type system. <a class="mw-redirect" href="/wiki/Encapsulation_(object-oriented_programming)" title="Encapsulation (object-oriented programming)">Encapsulation</a> of state is necessary for being able to enforce the invariants of the class.</p>
<h3><span class="mw-headline" id="Behavior">Behavior</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Class_(computer_programming)&amp;action=edit&amp;section=4" title="Edit section: Behavior">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="hatnote" role="note">Main article: <a href="/wiki/Method_(computer_programming)" title="Method (computer programming)">Method (computer programming)</a></div>
<p>The behavior of class or its instances is defined using <a href="/wiki/Method_(computer_programming)" title="Method (computer programming)">methods</a>. Methods are <a href="/wiki/Subroutine" title="Subroutine">subroutines</a> with the ability to operate on objects or classes. These operations may alter the state of an object or simply provide ways of accessing it.<sup class="reference" id="cite_ref-FOOTNOTEBooch199486-88_6-0"><a href="#cite_note-FOOTNOTEBooch199486-88-6">[6]</a></sup> Many kinds of methods exist, but support for them varies across languages. Some types of methods are created and called by programmer code, while other special methods—such as constructors, destructors, and conversion operators—are created and called by compiler-generated code. A language may also allow the programmer to define and call these special methods.<sup class="reference" id="cite_ref-7"><a href="#cite_note-7">[7]</a></sup><sup class="reference" id="cite_ref-8"><a href="#cite_note-8">[8]</a></sup></p>
<h3><span class="mw-headline" id="The_concept_of_class_interface">The concept of class interface</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Class_(computer_programming)&amp;action=edit&amp;section=5" title="Edit section: The concept of class interface">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="hatnote" role="note">Main article: <a href="/wiki/Interface_(computing)" title="Interface (computing)">Interface (computing)</a></div>
<p>Every class <i>implements</i> (or <i>realizes</i>) an interface by providing <a href="#Structure">structure</a> and behavior. Structure consists of data and state, and behavior consists of code that specifies how methods are implemented.<sup class="reference" id="cite_ref-FOOTNOTEBooch1994105_9-0"><a href="#cite_note-FOOTNOTEBooch1994105-9">[9]</a></sup> There is a distinction between the definition of an interface and the implementation of that interface; however, this line is blurred in many programming languages because class declarations both define and implement an interface. Some languages, however, provide features that separate interface and implementation. For example, an <a href="#Abstract_and_Concrete">abstract class</a> can define an interface without providing implementation.</p>
<p>Languages that support class inheritance also allow classes to inherit interfaces from the classes that they are derived from.<sup class="noprint Inline-Template" style="white-space:nowrap;">[<i><a class="mw-redirect" href="/wiki/Wikipedia:AUDIENCE" title="Wikipedia:AUDIENCE"><span title="Wikipedia:AUDIENCE">example needed</span></a></i>]</sup> In languages that support <a href="#Information_hiding_and_encapsulation">access specifiers</a>, the interface of a class is considered to be the set of public members of the class, including both methods and attributes (via implicit <a href="/wiki/Mutator_method" title="Mutator method">getter and setter methods</a>); any private members or internal data structures are not intended to be depended on by external code and thus are not part of the interface.</p>
<p>Object-oriented programming methodology dictates that the operations of any interface of a class are to be independent of each other. It results in a layered design where clients of an interface use the methods declared in the interface. An interface places no requirements for clients to invoke the operations of one interface in any particular order. This approach has the benefit that client code can assume that the operations of an interface are available for use whenever the client has access to the object.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="Source needed for whole paragraph...this is a rather specific statement about OOP methodology. (April 2012)">citation needed</span></a></i>]</sup></p>
<h4><span class="mw-headline" id="Example">Example</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Class_(computer_programming)&amp;action=edit&amp;section=6" title="Edit section: Example">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>The buttons on the front of your television set are the interface between you and the electrical wiring on the other side of its plastic casing. You press the "power" button to toggle the television on and off. In this example, your particular television is the instance, each method is represented by a button, and all the buttons together comprise the interface. (Other television sets that are the same model as yours would have the same interface.) In its most common form, an interface is a specification of a group of related methods without any associated implementation of the methods.</p>
<p>A television set also has a myriad of <i>attributes</i>, such as size and whether it supports color, which together comprise its structure. A class represents the full description of a television, including its attributes (structure) and buttons (interface).</p>
<p>Getting the total number of televisions manufactured could be a <i>static method</i> of the television class. This method is clearly associated with the class, yet is outside the domain of each individual instance of the class. Another example would be a static method that finds a particular instance out of the set of all television objects.</p>
<h3><span class="mw-headline" id="Member_accessibility">Member accessibility</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Class_(computer_programming)&amp;action=edit&amp;section=7" title="Edit section: Member accessibility">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="hatnote" role="note">"Private member" redirects here. For other uses, see <a class="mw-redirect" href="/wiki/Private_members_club" title="Private members club">Private members club</a> and <a href="/wiki/Private_member%27s_bill" title="Private member's bill">Private member's bill</a>.</div>
<div class="hatnote" role="note">Further information: <a href="/wiki/Information_hiding" title="Information hiding">Information hiding</a></div>
<p>The following is a common set of <a class="mw-redirect" href="/wiki/Access_specifiers" title="Access specifiers">access specifiers</a>:<sup class="reference" id="cite_ref-JavaAccessControl_10-0"><a href="#cite_note-JavaAccessControl-10">[10]</a></sup></p>
<ul>
<li><i>Private</i> (or <i>class-private</i>) restricts the access to the class itself. Only methods that are part of the same class can access private members.</li>
<li><i>Protected</i> (or <i>class-protected</i>) allows the class itself and all its subclasses to access the member.</li>
<li><i>Public</i> means that any code can access the member by its name.</li>
</ul>
<p>Although many object-oriented languages support the above access specifiers, their semantics may differ.</p>
<p>Object-oriented design uses the access specifiers in conjunction with careful design of public method implementations to enforce class invariants—constraints on the state of the objects. A common usage of access specifiers is to separate the internal data of a class from its interface: the internal structure is made private, while public <a class="mw-redirect" href="/wiki/Accessor_method" title="Accessor method">accessor methods</a> can be used to inspect or alter such private data.</p>
<p>Access specifiers do not necessarily control <i>visibility</i>, in that even private members may be visible to client external code. In some languages, an inaccessible but visible member may be referred to at run-time (for example, by a pointer returned from a member function), but an attempt to use it by referring to the name of the member from client code will be prevented by the type checker.<sup class="reference" id="cite_ref-11"><a href="#cite_note-11">[11]</a></sup></p>
<p>The various object-oriented programming languages enforce member accessibility and visibility to various degrees, and depending on the language's <a href="/wiki/Type_system" title="Type system">type system</a> and compilation policies, enforced at either <a href="/wiki/Compile_time" title="Compile time">compile-time</a> or <a href="/wiki/Run_time_(program_lifecycle_phase)" title="Run time (program lifecycle phase)">run-time</a>. For example, the <a href="/wiki/Java_(programming_language)" title="Java (programming language)">Java</a> language does not allow client code that accesses the private data of a class to compile. <sup class="reference" id="cite_ref-12"><a href="#cite_note-12">[12]</a></sup> In the <a href="/wiki/C%2B%2B" title="C++">C++</a> language, private methods are visible, but not accessible in the interface; however, they may be made invisible by explicitly declaring fully abstract classes that represent the interfaces of the class.<sup class="reference" id="cite_ref-cppinterface_13-0"><a href="#cite_note-cppinterface-13">[13]</a></sup></p>
<p>Some languages feature other accessibility schemes:</p>
<ul>
<li><i>Instance vs. class accessibility</i>: <a href="/wiki/Ruby_(programming_language)" title="Ruby (programming language)">Ruby</a> supports <i>instance-private</i> and <i>instance-protected</i> access specifiers in lieu of class-private and class-protected, respectively. They differ in that they restrict access based on the instance itself, rather than the instance's class.<sup class="reference" id="cite_ref-14"><a href="#cite_note-14">[14]</a></sup></li>
<li><i>Friend</i>: C++ supports a mechanism where a function explicitly declared as a <a href="/wiki/Friend_function" title="Friend function">friend function</a> of the class may access the members designated as private or protected.<sup class="reference" id="cite_ref-15"><a href="#cite_note-15">[15]</a></sup></li>
<li><i>Path-based</i>: Java supports restricting access to a member within a <a href="/wiki/Java_syntax#Access_modifiers" title="Java syntax">Java package</a>, which is the logical path of the file. However, it is a common practice when extending a Java framework to implement classes in the same package as a framework class in order to access protected members. The source file may exist in a completely different location, and may be deployed to a different .jar file, yet still be in the same logical path as far as the JVM is concerned.<sup class="reference" id="cite_ref-JavaAccessControl_10-1"><a href="#cite_note-JavaAccessControl-10">[10]</a></sup></li>
</ul>
<h2><span class="mw-headline" id="Inter-class_relationships">Inter-class relationships</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Class_(computer_programming)&amp;action=edit&amp;section=8" title="Edit section: Inter-class relationships">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>In addition to the design of standalone classes, programming languages may support more advanced class design based upon relationships between classes. The inter-class relationship design capabilities commonly provided are <i>compositional</i> and <i>hierarchical</i>.</p>
<h3><span class="mw-headline" id="Compositional">Compositional</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Class_(computer_programming)&amp;action=edit&amp;section=9" title="Edit section: Compositional">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Classes can be composed of other classes, thereby establishing a compositional relationship between the enclosing class and its embedded classes. Compositional relationship between classes is also commonly known as a <i><a href="/wiki/Has-a" title="Has-a">has-a</a></i> relationship.<sup class="reference" id="cite_ref-FOOTNOTEBooch1994180_16-0"><a href="#cite_note-FOOTNOTEBooch1994180-16">[16]</a></sup> For example, a class "Car" could be composed of and contain a class "Engine". Therefore, a Car <i>has an</i> Engine. One aspect of composition is containment, which is the enclosure of component instances by the instance that has them. If an enclosing object contains component instances by value, the components and their enclosing object have a similar <a href="/wiki/Object_lifetime" title="Object lifetime">lifetime</a>. If the components are contained by reference, they may not have a similar lifetime.<sup class="reference" id="cite_ref-FOOTNOTEBooch1994128-129_17-0"><a href="#cite_note-FOOTNOTEBooch1994128-129-17">[17]</a></sup> For example, in Objective-C 2.0:</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="k">@interface</span> <span class="nc">Car</span> : <span class="bp">NSObject</span>

<span class="k">@property</span> <span class="bp">NSString</span> <span class="o">*</span><span class="n">name</span><span class="p">;</span>
<span class="k">@property</span> <span class="n">Engine</span> <span class="o">*</span><span class="n">engine</span>
<span class="k">@property</span> <span class="bp">NSArray</span> <span class="o">*</span><span class="n">tires</span><span class="p">;</span>

<span class="k">@end</span>
</pre></div>
<p>This <code>Car</code> class <i>has</i> an instance of <code>NSString</code> (a string object), <code>Engine</code>, and <code>NSArray</code> (an array object).</p>
<h3><span class="mw-headline" id="Hierarchical">Hierarchical</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Class_(computer_programming)&amp;action=edit&amp;section=10" title="Edit section: Hierarchical">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Classes can be <i>derived</i> from one or more existing classes, thereby establishing a hierarchical relationship between the derived-from classes (<i>base classes</i>, <i>parent classes</i> or <i>superclasses</i>) and the derived class (<i>child class</i> or <i>subclass</i>) . The relationship of the derived class to the derived-from classes is commonly known as an <b><a href="/wiki/Is-a" title="Is-a">is-a</a></b> relationship.<sup class="reference" id="cite_ref-FOOTNOTEBooch1994112_18-0"><a href="#cite_note-FOOTNOTEBooch1994112-18">[18]</a></sup> For example, a class 'Button' could be derived from a class 'Control'. Therefore, a Button <b>is a</b> Control. Structural and behavioral members of the parent classes are <i>inherited</i> by the child class. Derived classes can define additional structural members (data fields) and behavioral members (methods) in addition to those that they <i>inherit</i> and are therefore <i>specializations</i> of their superclasses. Also, derived classes can <a href="/wiki/Method_overriding" title="Method overriding">override</a> inherited methods if the language allows.</p>
<p>Not all languages support multiple inheritance. For example, Java allows a class to implement multiple interfaces, but only inherit from one class.<sup class="reference" id="cite_ref-javainterface_19-0"><a href="#cite_note-javainterface-19">[19]</a></sup> If multiple inheritance is allowed, the hierarchy is a <a href="/wiki/Directed_acyclic_graph" title="Directed acyclic graph">directed acyclic graph</a> (or DAG for short), otherwise it is a <a href="/wiki/Tree_(graph_theory)" title="Tree (graph theory)">tree</a>. The hierarchy has classes as nodes and inheritance relationships as links. Classes in the same level are more likely to be <a href="/wiki/Association_(object-oriented_programming)" title="Association (object-oriented programming)">associated</a> than classes in different levels. The levels of this hierarchy are called <a href="/wiki/Layer_(object-oriented_design)" title="Layer (object-oriented design)">layers</a> or <a class="mw-redirect" href="/wiki/Level_of_abstraction" title="Level of abstraction">levels of abstraction</a>.</p>
<p>Example (Simplified Objective-C 2.0 code, from iPhone SDK):</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="k">@interface</span> <span class="bp">UIResponder</span> : <span class="bp">NSObject</span> <span class="c1">//...</span>
<span class="k">@interface</span> <span class="bp">UIView</span> : <span class="bp">UIResponder</span> <span class="c1">//...</span>
<span class="k">@interface</span> <span class="bp">UIScrollView</span> : <span class="bp">UIView</span> <span class="c1">//...</span>
<span class="k">@interface</span> <span class="bp">UITableView</span> : <span class="bp">UIScrollView</span> <span class="c1">//...</span>
</pre></div>
<p>In this example, a UITableView <b>is a</b> UIScrollView <b>is a</b> UIView <b>is a</b> UIResponder <b>is an</b> NSObject.</p>
<h4><span class="mw-headline" id="Definitions_of_subclass">Definitions of subclass</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Class_(computer_programming)&amp;action=edit&amp;section=11" title="Edit section: Definitions of subclass">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<div class="hatnote" role="note">Main articles: <a href="/wiki/Inheritance_(object-oriented_programming)" title="Inheritance (object-oriented programming)">Inheritance (object-oriented programming)</a>, <a class="mw-redirect" href="/wiki/Superclass_(computer_science)" title="Superclass (computer science)">Superclass (computer science)</a>, and <a class="mw-redirect" href="/wiki/Subclass_(computer_science)" title="Subclass (computer science)">Subclass (computer science)</a></div>
<p>Conceptually, a superclass is a <a class="mw-redirect" href="/wiki/Superset" title="Superset">superset</a> of its subclasses. For example, a common class hierarchy would involve <code>GraphicObject</code> as a superclass of <code>Rectangle</code> and <code>Elipse</code>, while <code>Square</code> would be a subclass of <code>Rectangle</code>. These are all <a href="/wiki/Subset" title="Subset">subset relations</a> in set theory as well, i.e., all squares are rectangles but not all rectangles are squares.</p>
<p>A common conceptual error is to mistake a <i>part of</i> relation with a subclass. For example, a car and truck are both kinds of vehicles and it would be appropriate to model them as subclasses of a vehicle class. However, it would be an error to model the component parts of the car as subclass relations. For example, a car is composed of an engine and body, but it would not be appropriate to model engine or body as a subclass of car.</p>
<p>In object-oriented modeling these kinds of relations are typically modeled as object properties. In this example the Car class would have a property called <code>parts</code>. <code>parts</code> would be typed to hold a collection of objects such as instances of <code>Body, Engine, Tires,...</code>. Object modeling languages such as UML include capabilities to model various aspects of part of and other kinds of relations. Data such as the cardinality of the objects, constraints on input and output values, etc. This information can be utilized by developer tools to generate additional code beside the basic data definitions for the objects. Things such as error checking on get and set methods.<sup class="reference" id="cite_ref-20"><a href="#cite_note-20">[20]</a></sup></p>
<p>One important question when modeling and implementing a system of object classes is whether a class can have one or more superclasses. In the real world with actual sets it would be rare to find sets that didn't intersect with more than one other set. However, while some systems such as Flavors and CLOS provide a capability for more than one parent to do so at run time introduces complexity that many in the object-oriented community consider antithetical to the goals of using object classes in the first place. Understanding which class will be responsible for handling a message can get complex when dealing with more than one superclass. If used carelessly this feature can introduce some of the same system complexity and ambiguity classes were designed to avoid.<sup class="reference" id="cite_ref-21"><a href="#cite_note-21">[21]</a></sup></p>
<p>Most modern object-oriented languages such as Smalltalk and Java require single inheritance at run time. For these languages, multiple inheritance may be useful for modeling but not for an implementation.</p>
<p>However, <a class="mw-redirect" href="/wiki/Semantic_web" title="Semantic web">semantic web</a> application objects do have multiple superclasses. The volatility of the Internet requires this level of flexibility and the technology standards such as the <a href="/wiki/Web_Ontology_Language" title="Web Ontology Language">Web Ontology Language (OWL)</a> are designed to support it.</p>
<p>A similar issue is whether or not the class hierarchy can be modified at run time. Languages such as Flavors, CLOS, and Smalltalk all support this feature as part of their <a class="mw-redirect" href="/wiki/Meta-object_protocol" title="Meta-object protocol">meta-object protocols</a>. Since classes are themselves first-class objects, it is possible to have them dynamically alter their structure by sending them the appropriate messages. Other languages that focus more on strong typing such as Java and C++ do not allow the class hierarchy to be modified at run time. Semantic web objects have the capability for run time changes to classes. The rational is similar to the justification for allowing multiple superclasses, that the Internet is so dynamic and flexible that dynamic changes to the hierarchy are required to manage this volatility.<sup class="reference" id="cite_ref-22"><a href="#cite_note-22">[22]</a></sup></p>
<h3><span class="mw-headline" id="Orthogonality_of_the_class_concept_and_inheritance">Orthogonality of the class concept and inheritance</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Class_(computer_programming)&amp;action=edit&amp;section=12" title="Edit section: Orthogonality of the class concept and inheritance">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Although class-based languages are commonly assumed to support inheritance, inheritance is not an intrinsic aspect of the concept of classes. Some languages, often referred to as "<a class="mw-redirect" href="/wiki/Object-based_languages" title="Object-based languages">object-based languages</a>", support classes yet do not support inheritance. Examples of object-based languages include earlier versions of <a href="/wiki/Visual_Basic" title="Visual Basic">Visual Basic</a>.</p>
<h3><span class="mw-headline" id="Within_object-oriented_analysis">Within object-oriented analysis</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Class_(computer_programming)&amp;action=edit&amp;section=13" title="Edit section: Within object-oriented analysis">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="hatnote" role="note">Main article: <a href="/wiki/Association_(object-oriented_programming)" title="Association (object-oriented programming)">Association (object-oriented programming)</a></div>
<p>In <a class="mw-redirect" href="/wiki/Object-oriented_analysis" title="Object-oriented analysis">object-oriented analysis</a> and in <a class="mw-redirect" href="/wiki/Unified_Modelling_Language" title="Unified Modelling Language">UML</a>, an <b>association</b> between two classes represents a collaboration between the classes or their corresponding instances. Associations have direction; for example, a bi-directional association between two classes indicates that both of the classes are aware of their relationship.<sup class="reference" id="cite_ref-ibmuml_23-0"><a href="#cite_note-ibmuml-23">[23]</a></sup> Associations may be labeled according to their name or purpose.<sup class="reference" id="cite_ref-FOOTNOTEBooch1994179_24-0"><a href="#cite_note-FOOTNOTEBooch1994179-24">[24]</a></sup></p>
<p>An association role is given end of an association and describes the role of the corresponding class. For example, a "subscriber" role describes the way instances of the class "Person" participate in a "subscribes-to" association with the class "Magazine". Also, a "Magazine" has the "subscribed magazine" role in the same association. Association role multiplicity describes how many instances correspond to each instance of the other class of the association. Common multiplicities are "0..1", "1..1", "1..*" and "0..*", where the "*" specifies any number of instances.<sup class="reference" id="cite_ref-ibmuml_23-1"><a href="#cite_note-ibmuml-23">[23]</a></sup></p>
<h2><span class="mw-headline" id="Taxonomy_of_classes">Taxonomy of classes</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Class_(computer_programming)&amp;action=edit&amp;section=14" title="Edit section: Taxonomy of classes">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>There are many categories of classes, some of which overlap.</p>
<h3><span class="mw-headline" id="Abstract_and_concrete"><span id="Abstract_and_concrete_classes"></span><span id="Abstract_and_Concrete"></span>Abstract and concrete</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Class_(computer_programming)&amp;action=edit&amp;section=15" title="Edit section: Abstract and concrete">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="hatnote" role="note">Main article: <a href="/wiki/Abstract_type" title="Abstract type">Abstract type</a></div>
<p>In a language that supports inheritance, an <b>abstract class</b>, or <i>abstract base class</i> (ABC), is a class that cannot be instantiated because it is either labeled as abstract or it simply specifies <a class="mw-redirect" href="/wiki/Abstract_method" title="Abstract method">abstract methods</a> (or <i>virtual methods</i>). An abstract class may provide implementations of some methods, and may also specify virtual methods via <a href="/wiki/Type_signature" title="Type signature">signatures</a> that are to be implemented by direct or indirect descendants of the abstract class. Before a class derived from an abstract class can be instantiated, all abstract methods of its parent classes must be implemented by some class in the derivation chain.<sup class="reference" id="cite_ref-cpppoly_25-0"><a href="#cite_note-cpppoly-25">[25]</a></sup></p>
<p>Most object-oriented programming languages allow the programmer to specify which classes are considered abstract and will not allow these to be instantiated. For example, in <a href="/wiki/Java_(programming_language)" title="Java (programming language)">Java</a>, <a href="/wiki/C_Sharp_(programming_language)" title="C Sharp (programming language)">C#</a> and <a href="/wiki/PHP" title="PHP">PHP</a>, the keyword <i>abstract</i> is used.<sup class="reference" id="cite_ref-26"><a href="#cite_note-26">[26]</a></sup><sup class="reference" id="cite_ref-27"><a href="#cite_note-27">[27]</a></sup> In <a href="/wiki/C%2B%2B" title="C++">C++</a>, an abstract class is a class having at least one abstract method given by the appropriate syntax in that language (a pure virtual function in C++ parlance).<sup class="reference" id="cite_ref-cpppoly_25-1"><a href="#cite_note-cpppoly-25">[25]</a></sup></p>
<p>A class consisting of only virtual methods is called a Pure Abstract Base Class (or <i>Pure ABC</i>) in C++ and is also known as an <i>interface</i> by users of the language.<sup class="reference" id="cite_ref-cppinterface_13-1"><a href="#cite_note-cppinterface-13">[13]</a></sup> Other languages, notably Java and C#, support a variant of abstract classes called an <a href="/wiki/Interface_(Java)" title="Interface (Java)">interface</a> via a keyword in the language. In these languages, <a href="/wiki/Multiple_inheritance" title="Multiple inheritance">multiple inheritance</a> is not allowed, but a class can implement multiple interfaces. Such a class can only contain abstract publicly accessible methods.<sup class="reference" id="cite_ref-javainterface_19-1"><a href="#cite_note-javainterface-19">[19]</a></sup><sup class="reference" id="cite_ref-28"><a href="#cite_note-28">[28]</a></sup><sup class="reference" id="cite_ref-29"><a href="#cite_note-29">[29]</a></sup></p>
<p>A <b>concrete class</b> is a class that can be <a class="mw-redirect" href="/wiki/Instantiation_(computer_science)" title="Instantiation (computer science)">instantiated</a>, as opposed to abstract classes, which cannot.</p>
<h3><span class="mw-headline" id="Local_and_inner">Local and inner</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Class_(computer_programming)&amp;action=edit&amp;section=16" title="Edit section: Local and inner">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>In some languages, classes can be declared in <a class="mw-redirect" href="/wiki/Scope_(programming)" title="Scope (programming)">scopes</a> other than the global scope. There are various types of such classes.</p>
<p>An <b><a href="/wiki/Inner_class" title="Inner class">Inner class</a></b> is a class defined within another class. The relationship between an inner class and its containing class can also be treated as another type of class association. An inner class is typically neither associated with instances of the enclosing class nor instantiated along with its enclosing class. Depending on language, it may or may not be possible to refer to the class from outside the enclosing class. A related concept is <i>inner types</i>, also known as <i>inner data type</i> or <i>nested type</i>, which is a generalization of the concept of inner classes. <a href="/wiki/C%2B%2B" title="C++">C++</a> is an example of a language that supports both inner classes and inner types (via <i><a href="/wiki/Typedef" title="Typedef">typedef</a></i> declarations).<sup class="reference" id="cite_ref-30"><a href="#cite_note-30">[30]</a></sup><sup class="reference" id="cite_ref-31"><a href="#cite_note-31">[31]</a></sup></p>
<p>Another type is a <b>local class</b>, which is a class defined within a procedure or function. This limits references to the class name to within the scope where the class is declared. Depending on the semantic rules of the language, there may be additional restrictions on local classes compared to non-local ones. One common restriction is to disallow local class methods to access local variables of the enclosing function. For example, in C++, a local class may refer to static variables declared within its enclosing function, but may not access the function's automatic variables.<sup class="reference" id="cite_ref-32"><a href="#cite_note-32">[32]</a></sup></p>
<h3><span class="mw-headline" id="Metaclasses">Metaclasses</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Class_(computer_programming)&amp;action=edit&amp;section=17" title="Edit section: Metaclasses">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="hatnote" role="note">Main article: <a href="/wiki/Metaclass" title="Metaclass">Metaclass</a></div>
<p><b>Metaclasses</b> are classes whose instances are classes.<sup class="reference" id="cite_ref-FOOTNOTEBooch1994133-134_33-0"><a href="#cite_note-FOOTNOTEBooch1994133-134-33">[33]</a></sup> A metaclass describes a common structure of a collection of classes and can implement a <a class="mw-redirect" href="/wiki/Design_pattern_(computer_science)" title="Design pattern (computer science)">design pattern</a> or describe particular kinds of classes. Metaclasses are often used to describe <a href="/wiki/Software_framework" title="Software framework">frameworks</a>.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (May 2012)">citation needed</span></a></i>]</sup></p>
<p>In some languages, such as <a href="/wiki/Python_(programming_language)" title="Python (programming language)">Python</a>, <a href="/wiki/Ruby_(programming_language)" title="Ruby (programming language)">Ruby</a> or <a href="/wiki/Smalltalk" title="Smalltalk">Smalltalk</a>, a class is also an object; thus each class is an instance of a unique metaclass which is built into the language. <sup class="reference" id="cite_ref-pythondata_model_5-1"><a href="#cite_note-pythondata_model-5">[5]</a></sup> <sup class="reference" id="cite_ref-34"><a href="#cite_note-34">[34]</a></sup> <sup class="reference" id="cite_ref-FOOTNOTEBooch1994134_35-0"><a href="#cite_note-FOOTNOTEBooch1994134-35">[35]</a></sup> The <a href="/wiki/Common_Lisp_Object_System" title="Common Lisp Object System">Common Lisp Object System</a> (CLOS) provides <a href="/wiki/Metaobject" title="Metaobject">metaobject protocols</a> (MOPs) to implement those classes and metaclasses. <sup class="reference" id="cite_ref-36"><a href="#cite_note-36">[36]</a></sup></p>
<h3><span class="mw-headline" id="Non-subclassable">Non-subclassable</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Class_(computer_programming)&amp;action=edit&amp;section=18" title="Edit section: Non-subclassable">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Non-subclassable classes allow programmers to design classes and hierarchies of classes which at some level in the hierarchy, further derivation is prohibited. (A stand-alone class may be also designated as non-subclassable, preventing the formation of any hierarchy). Contrast this to <i>abstract</i> classes, which imply, encourage, and require derivation in order to be used at all. A non-subclassable class is implicitly <i>concrete</i>.</p>
<p>A non-subclassable class is created by declaring the class as <code>sealed</code> in C# or as <code>final</code> in Java or PHP.<sup class="reference" id="cite_ref-37"><a href="#cite_note-37">[37]</a></sup><sup class="reference" id="cite_ref-38"><a href="#cite_note-38">[38]</a></sup><sup class="reference" id="cite_ref-39"><a href="#cite_note-39">[39]</a></sup></p>
<p>For example, Java's <code class="mw-highlight" dir="ltr"><span class="n">String</span></code> class is designated as <i>final</i>.<sup class="reference" id="cite_ref-40"><a href="#cite_note-40">[40]</a></sup></p>
<p>Non-subclassable classes may allow a compiler (in compiled languages) to perform optimizations that are not available for subclassable classes.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (May 2012)">citation needed</span></a></i>]</sup></p>
<p><br/></p>
<h3><span class="mw-headline" id="Mixins">Mixins</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Class_(computer_programming)&amp;action=edit&amp;section=19" title="Edit section: Mixins">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Some languages have special support for <a href="/wiki/Mixin" title="Mixin">mixins</a>, though in any language with multiple inheritance a mixin is simply a class that does not represent an is-a-type-of relationship. Mixins are typically used to add the same methods to multiple classes; for example, a class <code>UnicodeConversionMixin</code> might provide a method called <code>unicode_to_ascii</code> when included in classes <code>FileReader</code> and <code>WebPageScraper</code> that do not share a common parent.</p>
<h3><span class="mw-headline" id="Partial">Partial</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Class_(computer_programming)&amp;action=edit&amp;section=20" title="Edit section: Partial">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<table class="metadata plainlinks ambox ambox-content ambox-Unreferenced" role="presentation">

<td class="mbox-text"><span class="mbox-text-span">This section <b>does not <a href="/wiki/Wikipedia:Citing_sources" title="Wikipedia:Citing sources">cite</a> any <a href="/wiki/Wikipedia:Verifiability" title="Wikipedia:Verifiability">sources</a></b>. <span class="hide-when-compact">Please help improve this section by <a href="/wiki/Help:Introduction_to_referencing_with_Wiki_Markup/1" title="Help:Introduction to referencing with Wiki Markup/1">adding citations to reliable sources</a>. <p>In languages supporting the feature, a <b>partial class</b> is a class whose definition may be split into multiple pieces, within a single <a href="/wiki/Source_code" title="Source code">source-code</a> file or across multiple files. The pieces are merged at compile-time, making compiler output the same as for a non-partial class.</p>
<p>The primary motivation for introduction of partial classes is to facilitate the implementation of <a class="mw-redirect" href="/wiki/Code_generator" title="Code generator">code generators</a>, such as <a class="mw-redirect" href="/wiki/Visual_designer" title="Visual designer">visual designers</a>. It is otherwise a challenge or compromise to develop code generators that can manage the generated code when it is interleaved within developer-written code. Using partial classes, a code generator can process a separate file or coarse-grained partial class within a file, and is thus alleviated from intricately interjecting generated code via extensive parsing, increasing compiler efficiency and eliminating the potential risk of corrupting developer code. In a simple implementation of partial classes, the compiler can perform a phase of <a class="new" href="/w/index.php?title=Precompilation&amp;action=edit&amp;redlink=1" title="Precompilation (page does not exist)">precompilation</a> where it "unifies" all the parts of a partial class. Then, compilation can proceed as usual.</p>
<p>Other benefits and effects of the partial class feature include:</p>
<ul>
<li>Enables separation of a class's interface and implementation code in a unique way.</li>
<li>Eases navigation through large classes within an <a href="/wiki/Source_code_editor" title="Source code editor">editor</a>.</li>
<li>Enables <a href="/wiki/Separation_of_concerns" title="Separation of concerns">separation of concerns</a>, in a way similar to <a href="/wiki/Aspect-oriented_programming" title="Aspect-oriented programming">aspect-oriented programming</a> but without using any extra tools.</li>
<li>Enables multiple developers to work on a single class concurrently without the need to merge individual code into one file at a later time.</li>
</ul>
<p>Partial classes have existed in <a href="/wiki/Smalltalk" title="Smalltalk">Smalltalk</a> under the name of <i>Class Extensions</i> for considerable time. With the arrival of the <a href="/wiki/.NET_Framework" title=".NET Framework">.NET framework 2</a>, <a href="/wiki/Microsoft" title="Microsoft">Microsoft</a> introduced partial classes, supported in both <a href="/wiki/C_Sharp_(programming_language)" title="C Sharp (programming language)">C#</a> 2.0 and <a href="/wiki/Visual_Basic_.NET" title="Visual Basic .NET">Visual Basic 2005</a>. <a class="mw-redirect" href="/wiki/WinRT" title="WinRT">WinRT</a> also supports partial classes.</p>
<h4><span class="mw-headline" id="Example_in_VB.NET">Example in VB.NET</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Class_(computer_programming)&amp;action=edit&amp;section=21" title="Edit section: Example in VB.NET">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>This simple example, written in <a href="/wiki/Visual_Basic_.NET" title="Visual Basic .NET">Visual Basic .NET</a>, shows how parts of the same class are defined in two different files.</p>
<dl>
<dt>file1.vb</dt>
</dl>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="k">Partial</span> <span class="k">Class</span> <span class="nc">MyClass</span>
    <span class="k">Private</span> <span class="n">_name</span> <span class="ow">As</span> <span class="kt">String</span>
<span class="k">End</span> <span class="k">Class</span>
</pre></div>
<dl>
<dt>file2.vb</dt>
</dl>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="k">Partial</span> <span class="k">Class</span> <span class="nc">MyClass</span>
    <span class="k">Public</span> <span class="k">Readonly</span> <span class="k">Property</span> <span class="nf">Name</span><span class="p">()</span> <span class="ow">As</span> <span class="kt">String</span>
         <span class="k">Get</span>
             <span class="k">Return</span> <span class="n">_name</span>
         <span class="k">End</span> <span class="k">Get</span>
    <span class="k">End</span> <span class="k">Property</span>
<span class="k">End</span> <span class="k">Class</span>
</pre></div>
<p>When compiled, the result is the same as if the two files were written as one, like this:</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="k">Class</span> <span class="nc">MyClass</span>
    <span class="k">Private</span> <span class="n">_name</span> <span class="ow">As</span> <span class="kt">String</span>
    <span class="k">Public</span> <span class="k">Readonly</span> <span class="k">Property</span> <span class="nf">Name</span><span class="p">()</span> <span class="ow">As</span> <span class="kt">String</span>
         <span class="k">Get</span>
             <span class="k">Return</span> <span class="n">_name</span>
         <span class="k">End</span> <span class="k">Get</span>
    <span class="k">End</span> <span class="k">Property</span>
<span class="k">End</span> <span class="k">Class</span>
</pre></div>
<h4><span class="mw-headline" id="Example_in_Objective-C">Example in Objective-C</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Class_(computer_programming)&amp;action=edit&amp;section=22" title="Edit section: Example in Objective-C">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>In <a href="/wiki/Objective-C" title="Objective-C">Objective-C</a>, partial classes, also known as <b>categories</b>, may even spread over multiple libraries and executables, like this example:</p>
<p>In Foundation, header file NSData.h:</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="k">@interface</span> <span class="bp">NSData</span> : <span class="bp">NSObject</span>

<span class="p">-</span> <span class="p">(</span><span class="kt">id</span><span class="p">)</span><span class="nf">initWithContentsOfURL:</span><span class="p">(</span><span class="bp">NSURL</span> <span class="o">*</span><span class="p">)</span><span class="nv">URL</span><span class="p">;</span>
<span class="c1">//...</span>

<span class="k">@end</span>
</pre></div>
<p>In user-supplied library, a separate binary from Foundation framework, header file NSData+base64.h:</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="cp">#import &lt;Foundation/Foundation.h&gt;</span>

<span class="k">@interface</span> <span class="bp">NSData</span> <span class="nl">(base64)</span>

<span class="p">-</span> <span class="p">(</span><span class="bp">NSString</span> <span class="o">*</span><span class="p">)</span><span class="nf">base64String</span><span class="p">;</span>
<span class="p">-</span> <span class="p">(</span><span class="kt">id</span><span class="p">)</span><span class="nf">initWithBase64String:</span><span class="p">(</span><span class="bp">NSString</span> <span class="o">*</span><span class="p">)</span><span class="nv">base64String</span><span class="p">;</span>

<span class="k">@end</span>
</pre></div>
<p>And in an app, yet another separate binary file, source code file main.m:</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="cp">#import &lt;Foundation/Foundation.h&gt;</span>
<span class="cp">#import "NSData+base64.h"</span>

<span class="kt">int</span> <span class="nf">main</span><span class="p">(</span><span class="kt">int</span> <span class="n">argc</span><span class="p">,</span> <span class="kt">char</span> <span class="o">*</span><span class="n">argv</span><span class="p">[])</span>
<span class="p">{</span>
    <span class="k">if</span> <span class="p">(</span><span class="n">argc</span> <span class="o">&lt;</span> <span class="mi">2</span><span class="p">)</span>
        <span class="k">return</span> <span class="n">EXIT_FAILURE</span><span class="p">;</span>
    <span class="bp">NSString</span> <span class="o">*</span><span class="n">sourceURLString</span> <span class="o">=</span> <span class="p">[</span><span class="bp">NSString</span> <span class="nl">stringWithCString</span><span class="p">:</span><span class="n">argv</span><span class="p">[</span><span class="mi">1</span><span class="p">]];</span>
    <span class="bp">NSData</span> <span class="o">*</span><span class="n">data</span> <span class="o">=</span> <span class="p">[[</span><span class="bp">NSData</span> <span class="n">alloc</span><span class="p">]</span> <span class="nl">initWithContentsOfURL</span><span class="p">:[</span><span class="bp">NSURL</span> <span class="nl">URLWithString</span><span class="p">:</span><span class="n">sourceURLString</span><span class="p">]];</span>
    <span class="n">NSLog</span><span class="p">(</span><span class="s">@"%@"</span><span class="p">,</span> <span class="p">[</span><span class="n">data</span> <span class="n">base64String</span><span class="p">]);</span>
    <span class="k">return</span> <span class="n">EXIT_SUCCESS</span><span class="p">;</span>
<span class="p">}</span>
</pre></div>
<p>The dispatcher will find both methods called over the NSData instance and invoke both of them correctly.</p>
<h3><span class="mw-headline" id="Uninstantiable">Uninstantiable</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Class_(computer_programming)&amp;action=edit&amp;section=23" title="Edit section: Uninstantiable">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p><b>Uninstantiable classes</b> allow programmers to group together per-class fields and methods that are accessible at runtime without an instance of the class. Indeed, instantiation is prohibited for this kind of class.</p>
<p>For example, in C#, a class marked "static" can not be instantiated, can only have static members (fields, methods, other), may not have <i>instance constructors</i>, and is <i>sealed</i>. <sup class="reference" id="cite_ref-41"><a href="#cite_note-41">[41]</a></sup></p>
<h3><span class="mw-headline" id="Unnamed">Unnamed</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Class_(computer_programming)&amp;action=edit&amp;section=24" title="Edit section: Unnamed">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>An <b>unnamed class</b> or <b>anonymous class</b> is a class which is not bound to a name or identifier upon definition. This is analogous to named versus <a href="/wiki/Anonymous_function" title="Anonymous function">unnamed functions</a>.</p>
<h2><span class="mw-headline" id="Benefits">Benefits</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Class_(computer_programming)&amp;action=edit&amp;section=25" title="Edit section: Benefits">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The benefits of organizing software into object classes fall into three categories:<sup class="reference" id="cite_ref-42"><a href="#cite_note-42">[42]</a></sup></p>
<ul>
<li>Rapid development</li>
<li>Ease of maintenance</li>
<li>Reuse of code and designs</li>
</ul>
<p>Object classes facilitate rapid development because they lessen the semantic gap between the code and the users. System analysts can talk to both developers and users using essentially the same vocabulary, talking about accounts, customers, bills, etc. Object classes often facilitate rapid development because most object-oriented environments come with powerful debugging and testing tools. Instances of classes can be inspected at run time to verify that the system is performing as expected. Also, rather than get dumps of core memory, most object-oriented environments have interpreted debugging capabilities so that the developer can analyze exactly where in the program the error occurred and can see which methods were called to which arguments and with what arguments.<sup class="reference" id="cite_ref-43"><a href="#cite_note-43">[43]</a></sup></p>
<p>Object classes facilitate ease of maintenance via encapsulation. When developers need to change the behavior of an object they can localize the change to just that object and its component parts. This reduces the potential for unwanted side effects from maintenance enhancements.</p>
<p>Software re-use is also a major benefit of using Object classes. Classes facilitate re-use via inheritance and interfaces. When a new behavior is required it can often be achieved by creating a new class and having that class inherit the default behaviors and data of its superclass and then tailor some aspect of the behavior or data accordingly. Re-use via interfaces (also known as methods) occurs when another object wants to invoke (rather than create a new kind of) some object class. This method for re-use removes many of the common errors that can make their way into software when one program re-uses code from another.<sup class="reference" id="cite_ref-44"><a href="#cite_note-44">[44]</a></sup></p>
<p>These benefits come with a cost of course. One of the most serious obstacles to using object classes has been performance. Interpreted environments that support languages such as Smalltalk and CLOS provided rapid development but the resulting code was not nearly as fast as what could be achieved in some procedural languages such as C. This has been partly addressed by the development of object-oriented languages that are not interpreted such as C++ and Java.<sup class="reference" id="cite_ref-45"><a href="#cite_note-45">[45]</a></sup> Also, due to <a href="/wiki/Moore%27s_law" title="Moore's law">Moore's law</a> the processing power of computers has increased to the point where efficient code is not as critical for most systems as it was in the past.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="Efficient code has become more important in today's age not less important (September 2016)">citation needed</span></a></i>]</sup>Still, no matter how well designed the language, there will always be an inevitable bit of required extra overhead to create a class rather than use procedural code and in some circumstances, especially where performance or memory are required to be optimal, that using object classes may not be the best approach. Also, getting the benefits of object classes requires that they be used appropriately and that requires training. Without the proper training developers may simply code procedural programs in an object-oriented environment and end up with the worst of both worlds.<sup class="reference" id="cite_ref-46"><a href="#cite_note-46">[46]</a></sup></p>
<h2><span class="mw-headline" id="Run-time_representation">Run-time representation</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Class_(computer_programming)&amp;action=edit&amp;section=26" title="Edit section: Run-time representation">edit</a><span class="mw-editsection-bracket">]</span></span></h2>

<p>As a data type, a class is usually considered as a compile-time construct. A language may also support <a href="/wiki/Prototype-based_programming" title="Prototype-based programming">prototype</a> or <a href="/wiki/Factory_method_pattern" title="Factory method pattern">factory</a> <a href="/wiki/Metaobject" title="Metaobject">metaobjects</a> that represent run-time information about classes, or even represent metadata that provides access to <a class="mw-redirect" href="/wiki/Reflection_(computer_science)" title="Reflection (computer science)">reflection</a> facilities and ability to manipulate data structure formats at run-time. Many languages distinguish this kind of <a href="/wiki/Run-time_type_information" title="Run-time type information">run-time type information</a> about classes from a class on the basis that the information is not needed at run-time. Some dynamic languages do not make strict distinctions between run-time and compile-time constructs, and therefore may not distinguish between metaobjects and classes.</p>
<p>For example, if Human is a <a href="/wiki/Metaobject" title="Metaobject">metaobject</a> representing the class Person, then instances of class Person can be created by using the facilities of the Human <a href="/wiki/Metaobject" title="Metaobject">metaobject</a>.</p>
