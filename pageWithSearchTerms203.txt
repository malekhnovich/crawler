<div class="hatnote" role="note">This article is about the computer programming concept. For the practice of hiding data in a message or file, see <a href="/wiki/Steganography" title="Steganography">Steganography</a>. For data encryption, see <a href="/wiki/Cryptography" title="Cryptography">Cryptography</a>.</div>
<p>In <a href="/wiki/Computer_science" title="Computer science">computer science</a>, <b>information hiding</b> is the principle of segregation of the <i>design decisions</i> in a <a href="/wiki/Computer_program" title="Computer program">computer program</a> that are most likely to change, thus protecting other parts of the program from extensive modification if the design decision is changed. The protection involves providing a stable <a class="mw-redirect" href="/wiki/Interface_(computer_science)" title="Interface (computer science)">interface</a> which protects the remainder of the program from the implementation (the details that are most likely to change).</p>
<p>Written another way, information hiding is the ability to prevent certain aspects of a <a class="mw-redirect" href="/wiki/Class_(computer_science)" title="Class (computer science)">class</a> or <a class="mw-redirect" href="/wiki/Software_component" title="Software component">software component</a> from being accessible to its <a href="/wiki/Client_(computing)" title="Client (computing)">clients</a>, using either programming language features (like private variables) or an explicit exporting policy.</p>

<h2><span class="mw-headline" id="Overview">Overview</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Information_hiding&amp;action=edit&amp;section=1" title="Edit section: Overview">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The term <i>encapsulation</i> is often used interchangeably with information hiding. Not all agree on the distinctions between the two though; one may think of information hiding as being the principle and encapsulation being the technique. A software module hides information by encapsulating the information into a module or other construct which presents an interface.<sup class="reference" id="cite_ref-not_encaps_1-0"><a href="#cite_note-not_encaps-1">[1]</a></sup></p>
<p>A common use of information hiding is to hide the physical storage layout for data so that if it is changed, the change is restricted to a small subset of the total program. For example, if a three-dimensional point (<i>x</i>,<i>y</i>,<i>z</i>) is represented in a program with three <a href="/wiki/Floating_point" title="Floating point">floating point</a> <a class="mw-redirect" href="/wiki/Scalar_(computing)" title="Scalar (computing)">scalar</a> variables and later, the representation is changed to a single <a href="/wiki/Array_data_structure" title="Array data structure">array</a> variable of size three, a module designed with information hiding in mind would protect the remainder of the program from such a change.</p>
<p>In <a href="/wiki/Object-oriented_programming" title="Object-oriented programming">object-oriented programming</a>, information hiding (by way of <a href="/wiki/Nesting_(computing)" title="Nesting (computing)">nesting</a> of types) reduces software development risk by shifting the code's <a class="mw-redirect" href="/wiki/Dependency_(computer_science)" title="Dependency (computer science)">dependency</a> on an uncertain implementation (design decision) onto a well-defined <a class="mw-redirect" href="/wiki/Interface_(computer_science)" title="Interface (computer science)">interface</a>. Clients of the interface perform operations purely through it so if the implementation changes, the clients do not have to change.</p>
<h2><span class="mw-headline" id="Encapsulation">Encapsulation</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Information_hiding&amp;action=edit&amp;section=2" title="Edit section: Encapsulation">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<div class="hatnote" role="note">See also: <a class="mw-redirect" href="/wiki/Encapsulation_(object-oriented_programming)" title="Encapsulation (object-oriented programming)">Encapsulation (object-oriented programming)</a></div>
<p>In his book on <a href="/wiki/Object-oriented_design" title="Object-oriented design">object-oriented design</a>, <a href="/wiki/Grady_Booch" title="Grady Booch">Grady Booch</a> defined encapsulation as "the process of compartmentalizing the elements of an abstraction that constitute its structure and behavior; encapsulation serves to separate the contractual interface of an abstraction and its implementation."<sup class="reference" id="cite_ref-Booch_2-0"><a href="#cite_note-Booch-2">[2]</a></sup></p>
<p>The purpose is to achieve potential for change: the internal mechanisms of the component can be improved without impact on other components, or the component can be replaced with a different one that supports the same public interface. Encapsulation also protects the integrity of the component, by preventing users from setting the internal data of the component into an invalid or inconsistent state. Another benefit of encapsulation is that it reduces system complexity and thus increases robustness, by limiting the interdependencies between software components.<sup class="reference" id="cite_ref-Booch_2-1"><a href="#cite_note-Booch-2">[2]</a></sup></p>
<p>In this sense, the idea of encapsulation is more general than how it is applied in OOP: for example, a relational database is encapsulated in the sense that its only public interface is a <a href="/wiki/Query_language" title="Query language">Query language</a> (<a href="/wiki/SQL" title="SQL">SQL</a> for example), which hides all the internal machinery and data structures of the database management system. As such, encapsulation is a core principle of good software architecture, at every level of granularity.</p>
<p>Encapsulating software behind an interface allows the construction of objects that mimic the behavior and interactions of objects in the real world. For example, a simple digital alarm clock is a real-world object that a <a class="mw-redirect" href="/wiki/Lay_person" title="Lay person">lay person</a> can use and understand. They can understand what the alarm clock does, and how to use it through the provided interface (buttons and screen), without having to understand every part inside of the clock. Similarly, if the clock were replaced by a different model, the lay person could continue to use it in the same way, provided that the interface works the same.</p>
<p>In the more concrete setting of an object-oriented programming language, the notion is used to mean either an information hiding mechanism, a bundling mechanism, or the combination of the two. (See <a class="mw-redirect" href="/wiki/Encapsulation_(object-oriented_programming)" title="Encapsulation (object-oriented programming)">Encapsulation (object-oriented programming)</a> for details.)</p>
<h2><span class="mw-headline" id="History">History</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Information_hiding&amp;action=edit&amp;section=3" title="Edit section: History">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The concept of information hiding was first described by <a href="/wiki/David_Parnas" title="David Parnas">David Parnas</a> in <a href="#CITEREFParnas1972">Parnas (1972)</a>.<sup class="reference" id="cite_ref-FOOTNOTEScott2009173_3-0"><a href="#cite_note-FOOTNOTEScott2009173-3">[3]</a></sup> Before then, modularity was discussed by Richard Gauthier and Stephen Pont in their 1970 book <i>Designing Systems Programs</i> although <a href="/wiki/Modular_programming" title="Modular programming">modular programming</a> itself had been used at many commercial sites for many years previously – especially in <a class="mw-redirect" href="/wiki/I/O" title="I/O">I/O</a> <a class="mw-redirect" href="/wiki/Sub-system" title="Sub-system">sub-systems</a> and <a class="mw-redirect" href="/wiki/Software_library" title="Software library">software libraries</a> – without acquiring the 'information hiding' tag – but for similar reasons, as well as the more obvious <a href="/wiki/Code_reuse" title="Code reuse">code reuse</a> reason.</p>
<h2><span class="mw-headline" id="Example_of_information_hiding">Example of information hiding</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Information_hiding&amp;action=edit&amp;section=4" title="Edit section: Example of information hiding">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Information hiding serves as an effective criterion for dividing any piece of equipment, software or hardware, into modules of functionality. For instance a car is a complex piece of equipment. In order to make the design, manufacturing, and maintenance of a car reasonable, the complex piece of equipment is divided into <a class="mw-redirect" href="/wiki/Module_(programming)" title="Module (programming)">modules</a> with particular interfaces hiding design decisions. By designing a car in this fashion, a car manufacturer can also offer various options while still having a vehicle which is economical to manufacture.</p>
<p>For instance, a car manufacturer may have a luxury version of the car as well as a standard version. The luxury version comes with a more powerful engine than the standard version. The <a href="/wiki/Engineer" title="Engineer">engineers</a> designing the two different car engines, one for the luxury version and one for the standard version, provide the same interface for both engines. Both engines fit into the engine bay of the car which is the same between both versions. Both engines fit the same transmission, the same engine mounts, and the same controls. The differences in the engines are that the more powerful luxury version has a larger displacement with a fuel injection system that is programmed to provide the fuel air mixture that the larger displacement engine requires.</p>
<p>In addition to the more powerful engine, the luxury version may also offer other options such as a better radio with CD player, more comfortable seats, a better suspension system with wider tires, and different paint colors. With all of these changes, most of the car is the same between the standard version and the luxury version. The radio with CD player is a module which replaces the standard radio, also a module, in the luxury model. The more comfortable seats are installed into the same seat mounts as the standard types of seats. Whether the seats are leather or plastic, or offer lumbar support or not, doesn't matter.</p>
<p>The engineers design the car by dividing the task up into pieces of work which are assigned to teams. Each team then designs their <a class="mw-redirect" href="/wiki/Software_componentry" title="Software componentry">component</a> to a particular standard or interface which allows the team flexibility in the design of the component while at the same time ensuring that all of the components will fit together.</p>
<p>Motor vehicle manufacturers frequently use the same core structure for several different models, in part as a cost-control measure. Such a "<a class="mw-redirect" href="/wiki/Automobile_platform" title="Automobile platform">platform</a>" also provides an example of information hiding, since the floorpan can be built without knowing whether it is to be used in a sedan or a hatchback.</p>
<p>As can be seen by this example, information hiding provides flexibility. This flexibility allows a programmer to modify functionality of a computer program during normal evolution as the computer program is changed to better fit the needs of users. When a computer program is well designed decomposing the source code solution into modules using the principle of information hiding, evolutionary changes are much easier because the changes typically are local rather than global changes.</p>
<p>Cars provide another example of this in how they interface with drivers. They present a standard interface (pedals, wheel, shifter, signals, gauges, etc.) on which people are trained and licensed. Thus, people only have to learn to drive a car; they don't need to learn a completely different way of driving every time they drive a new model. (Granted, there are manual and automatic transmissions and other such differences, but on the whole cars maintain a unified interface.)</p>
