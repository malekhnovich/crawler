<p><b>Programming paradigms</b> are a way to classify <a href="/wiki/Programming_language" title="Programming language">programming languages</a> according to the style of <a href="/wiki/Computer_programming" title="Computer programming">computer programming</a>. Features of various programming languages determine which programming paradigms they belong to; as a result, some languages fall into only one paradigm, while others fall into multiple paradigms. Some paradigms are concerned primarily with implications for the <a href="/wiki/Execution_model" title="Execution model">execution model</a> of the language, such as allowing <a href="/wiki/Side_effect_(computer_science)" title="Side effect (computer science)">side effects</a>, or whether the sequence of operations is defined by the execution model. Other paradigms are concerned primarily with the way that code is organized, such as grouping code into units along with the state that is modified by the code. Yet others are concerned primarily with the style of syntax and grammar.</p>
<p>Common programming paradigms include <a href="/wiki/Imperative_programming" title="Imperative programming">imperative</a> which allows side effects, <a href="/wiki/Functional_programming" title="Functional programming">functional</a> which does not allow side effects, <a href="/wiki/Declarative_programming" title="Declarative programming">declarative</a> which does not state the order in which operations execute, <a href="/wiki/Object-oriented_programming" title="Object-oriented programming">object-oriented</a> which groups code together with the state the code modifies, <a href="/wiki/Procedural_programming" title="Procedural programming">procedural</a> which groups code into functions, <a href="/wiki/Logic_programming" title="Logic programming">logic</a> which has a particular style of execution model coupled to a particular style of syntax and grammar, and <a href="/wiki/Symbolic_programming" title="Symbolic programming">symbolic</a> programming which has a particular style of syntax and grammar.<sup class="reference" id="cite_ref-1"><a href="#cite_note-1">[1]</a></sup><sup class="reference" id="cite_ref-2"><a href="#cite_note-2">[2]</a></sup><sup class="reference" id="cite_ref-symbolic-programming-lisp_3-0"><a href="#cite_note-symbolic-programming-lisp-3">[3]</a></sup></p>
<p>For example, languages that fall into the imperative paradigm have two main features: they state the order in which operations take place, with constructs that explicitly control that order, and they allow side effects, in which state can be modified at one point in time, within one unit of code, and then later read at a different point in time inside a different unit of code. The communication between the units of code is not explicit. Meanwhile, in object-oriented programming, code is organized into <a class="mw-redirect" href="/wiki/Object_(programming)" title="Object (programming)">objects</a> that contain state that is only modified by the code that is part of the object. Most object oriented languages are also imperative languages. In contrast, languages that fit the declarative paradigm do not state the order in which to execute operations. Instead, they supply a number of operations that are available in the system, along with the conditions under which each is allowed to execute. The implementation of the language's execution model tracks which operations are free to execute and chooses the order on its own.</p>

</div>
<p>Just as <a href="/wiki/Software_engineering" title="Software engineering">software engineering</a> (as a process) is defined by differing <i>methodologies</i>, so the <a href="/wiki/Programming_language" title="Programming language">programming languages</a> (as models of computation) are defined by differing <i>paradigms</i>. Some languages are designed to support one paradigm (<a href="/wiki/Smalltalk" title="Smalltalk">Smalltalk</a> supports object-oriented programming, <a href="/wiki/Haskell_(programming_language)" title="Haskell (programming language)">Haskell</a> supports functional programming), while other programming languages support multiple paradigms (such as <a href="/wiki/Object_Pascal" title="Object Pascal">Object Pascal</a>, <a href="/wiki/C%2B%2B" title="C++">C++</a>, <a href="/wiki/Java_(programming_language)" title="Java (programming language)">Java</a>, <a href="/wiki/C_Sharp_(programming_language)" title="C Sharp (programming language)">C#</a>, <a href="/wiki/Scala_(programming_language)" title="Scala (programming language)">Scala</a>, <a href="/wiki/Visual_Basic" title="Visual Basic">Visual Basic</a>, <a href="/wiki/Common_Lisp" title="Common Lisp">Common Lisp</a>, <a href="/wiki/Scheme_(programming_language)" title="Scheme (programming language)">Scheme</a>, <a href="/wiki/Perl" title="Perl">Perl</a>, <a href="/wiki/Python_(programming_language)" title="Python (programming language)">Python</a>, <a href="/wiki/Ruby_(programming_language)" title="Ruby (programming language)">Ruby</a>, <a href="/wiki/Oz_(programming_language)" title="Oz (programming language)">Oz</a> and <a href="/wiki/F_Sharp_(programming_language)" title="F Sharp (programming language)">F#</a>). For example, programs written in C++ or Object Pascal can be purely <a href="/wiki/Procedural_programming" title="Procedural programming">procedural</a>, purely <a href="/wiki/Object-oriented_programming" title="Object-oriented programming">object-oriented</a>, or they can contain elements of both or other paradigms. Software designers and programmers decide how to use those paradigm elements.</p>
<p>In object-oriented programming, programs are treated as a collection of interacting objects. In <a href="/wiki/Functional_programming" title="Functional programming">functional programming</a>, programs are treated as a sequence of stateless function evaluations. When programming computers or systems with many processors, in <a href="/wiki/Process-oriented_programming" title="Process-oriented programming">process-oriented programming</a>, programs are treated as sets of concurrent processes acting on logically shared <a href="/wiki/Data_structure" title="Data structure">data structures</a>.</p>
<p>Many programming paradigms are as well known for the techniques they <i>forbid</i> as for those they <i>enable</i>. For instance, pure functional programming disallows use of <a class="mw-redirect" href="/wiki/Side-effect_(computer_science)" title="Side-effect (computer science)">side-effects</a>, while <a href="/wiki/Structured_programming" title="Structured programming">structured programming</a> disallows use of the <a href="/wiki/Goto" title="Goto">goto</a> statement. Partly for this reason, new paradigms are often regarded as doctrinaire or overly rigid by those accustomed to earlier styles.<sup class="reference" id="cite_ref-rubin87goto_5-0"><a href="#cite_note-rubin87goto-5">[5]</a></sup> Yet, avoiding certain techniques can make it easier to prove theorems about the correctness of programs, or simply to understand the program behavior.</p>
<p>Programming paradigms can also be compared with <i><a href="/wiki/Programming_model" title="Programming model">programming models</a></i> which allow invoking an external <a href="/wiki/Execution_model" title="Execution model">execution model</a> by using only an API. Programming models can also be classified into paradigms, based on features of the execution model.</p>
<p>For <a href="/wiki/Parallel_computing" title="Parallel computing">parallel computing</a>, using a programming model instead of a language is common. The reason is that details of the parallel hardware leak into the abstractions used to program the hardware. This causes the programmer to have to map patterns in the algorithm onto patterns in the execution model (which have been inserted due to leakage of hardware into the abstraction). As a consequence, no one parallel programming language fits well to all computation problems. It is thus more convenient to use a base sequential language and insert API calls to parallel execution models, via a programming model. Such parallel programming models can be classified according to abstractions that reflect the hardware, such as shared memory, distributed memory with message passing, notions of "place" visible in the code, and so forth. These can be considered flavors of programming paradigm that apply only to parallel languages and parallel programming models.</p>
<p>Some programming language researchers criticise the notion of paradigms as a classification of programming languages, e.g. Krishnamurthi.<sup class="reference" id="cite_ref-6"><a href="#cite_note-6">[6]</a></sup> They argue that many programming languages cannot be strictly classified into one paradigm, but rather include features from several paradigms.</p>
<h2><span class="mw-headline" id="History">History</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Programming_paradigm&amp;action=edit&amp;section=2" title="Edit section: History">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Different approaches to programming have developed over time, being identified as such either at the time or retrospectively. An early approach consciously identified as such is <a href="/wiki/Structured_programming" title="Structured programming">structured programming</a>, advocated since the mid 1960s. The concept of a "programming paradigm" as such dates at least to 1978, in the <a href="/wiki/Turing_Award" title="Turing Award">Turing Award</a> lecture of <a href="/wiki/Robert_W._Floyd" title="Robert W. Floyd">Robert W. Floyd</a>, entitled <i>The Paradigms of Programming</i>, which cites the notion of paradigm as used by <a href="/wiki/Thomas_Kuhn" title="Thomas Kuhn">Thomas Kuhn</a> in his <i><a href="/wiki/The_Structure_of_Scientific_Revolutions" title="The Structure of Scientific Revolutions">The Structure of Scientific Revolutions</a></i> (1962).<sup class="reference" id="cite_ref-7"><a href="#cite_note-7">[7]</a></sup></p>
<h3><span class="mw-headline" id="Machine_code">Machine code</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Programming_paradigm&amp;action=edit&amp;section=3" title="Edit section: Machine code">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>The <a href="/wiki/Low-level_programming_language" title="Low-level programming language">lowest level</a> programming paradigms are <a href="/wiki/Machine_code" title="Machine code">machine code</a>, which directly represents the <a href="/wiki/Instruction_set" title="Instruction set">instructions</a> (the contents of program memory) as a sequence of numbers, and <a href="/wiki/Assembly_language" title="Assembly language">assembly language</a> where the machine instructions are represented by mnemonics and <a href="/wiki/Memory_address" title="Memory address">memory addresses</a> can be given symbolic labels. These are sometimes called <a href="/wiki/First-generation_programming_language" title="First-generation programming language">first-</a> and <a href="/wiki/Second-generation_programming_language" title="Second-generation programming language">second-generation</a> languages.</p>
<p>In the 1960s, assembly languages were developed to support library COPY and quite sophisticated conditional macro generation and preprocessing abilities, CALL to (<a href="/wiki/Subroutine" title="Subroutine">subroutines</a>), external variables and common sections (globals), enabling significant code re-use and isolation from hardware specifics via use of logical operators such as READ/WRITE/GET/PUT. Assembly was, and still is, used for time critical systems and often in <a href="/wiki/Embedded_system" title="Embedded system">embedded systems</a> as it gives the most direct control of what the machine does.</p>
<h3><span class="mw-headline" id="Procedural_languages">Procedural languages</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Programming_paradigm&amp;action=edit&amp;section=4" title="Edit section: Procedural languages">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>The next advance was the development of <a class="mw-redirect" href="/wiki/Procedural_language" title="Procedural language">procedural languages</a>. These <a href="/wiki/Third-generation_programming_language" title="Third-generation programming language">third-generation</a> languages (the first described as <a href="/wiki/High-level_programming_language" title="High-level programming language">high-level languages</a>) use vocabulary related to the problem being solved. For example,</p>
<ul>
<li>COmmon Business Oriented Language (<a href="/wiki/COBOL" title="COBOL">COBOL</a>) –  uses terms like <a href="/wiki/Computer_file" title="Computer file">file</a>, <a href="/wiki/Move_(command)" title="Move (command)">move</a> and <a href="/wiki/Copy_(command)" title="Copy (command)">copy</a>.</li>
<li>FORmula TRANslation (<a class="mw-redirect" href="/wiki/FORTRAN" title="FORTRAN">FORTRAN</a>) –  using <a class="mw-redirect" href="/wiki/Mathematical" title="Mathematical">mathematical</a> language terminology, it was developed mainly for scientific and engineering problems.</li>
<li>ALGOrithmic Language (<a href="/wiki/ALGOL" title="ALGOL">ALGOL</a>) –  focused on being an appropriate language to define <a href="/wiki/Algorithm" title="Algorithm">algorithms</a>, while using mathematical language terminology and targeting scientific and engineering problems just like FORTRAN.</li>
<li>Programming Language One (<a href="/wiki/PL/I" title="PL/I">PL/I</a>) –  a hybrid commercial/scientific general purpose language supporting <a href="/wiki/Pointer_(computer_programming)" title="Pointer (computer programming)">pointers</a>.</li>
<li>Beginners All purpose Symbolic Instruction Code (<a href="/wiki/BASIC" title="BASIC">BASIC</a>) –  it was developed to enable more people to write programs.</li>
<li><a href="/wiki/C_(programming_language)" title="C (programming language)">C</a> –  a general-purpose programming language, initially developed by <a href="/wiki/Dennis_Ritchie" title="Dennis Ritchie">Dennis Ritchie</a> between 1969 and 1973 at <a class="mw-redirect" href="/wiki/AT%26T_Bell_Labs" title="AT&amp;T Bell Labs">AT&amp;T Bell Labs</a>.</li>
</ul>
<p>All these languages follow the procedural paradigm. That is, they describe, step by step, exactly the procedure that should, according to the particular programmer at least, be followed to solve a specific problem. The <a href="/wiki/Efficacy" title="Efficacy">efficacy</a> and <a href="/wiki/Algorithmic_efficiency" title="Algorithmic efficiency">efficiency</a> of any such solution are both therefore entirely subjective and highly dependent on that programmer's experience, inventiveness, and ability.</p>
<h3><span class="mw-headline" id="Object-oriented_programming">Object-oriented programming</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Programming_paradigm&amp;action=edit&amp;section=5" title="Edit section: Object-oriented programming">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="hatnote" role="note">Main article: <a href="/wiki/Object-oriented_programming" title="Object-oriented programming">Object-oriented programming</a></div>
<p>Following the widespread use of procedural languages, <a href="/wiki/Object-oriented_programming" title="Object-oriented programming">object-oriented programming</a> (OOP) languages were created, such as <a href="/wiki/Simula" title="Simula">Simula</a>, <a href="/wiki/Smalltalk" title="Smalltalk">Smalltalk</a>, <a href="/wiki/C%2B%2B" title="C++">C++</a>, <a href="/wiki/C_Sharp_(programming_language)" title="C Sharp (programming language)">C#</a>, <a href="/wiki/Eiffel_(programming_language)" title="Eiffel (programming language)">Eiffel</a>, and <a href="/wiki/Java_(programming_language)" title="Java (programming language)">Java</a>. In these languages, <a href="/wiki/Data" title="Data">data</a> and methods to manipulate it are kept as one unit called an <a href="/wiki/Object_(computer_science)" title="Object (computer science)">object</a>. The only way that another object or user can access the data is via the object's <i><a href="/wiki/Method_(computer_programming)" title="Method (computer programming)">methods</a></i>. Thus, the inner workings of an object may be changed without affecting any code that uses the object. There is still some <a href="/wiki/Object-oriented_programming#Criticism" title="Object-oriented programming">controversy</a> raised by <a href="/wiki/Alexander_Stepanov" title="Alexander Stepanov">Alexander Stepanov</a>, <a href="/wiki/Richard_Stallman" title="Richard Stallman">Richard Stallman</a><sup class="reference" id="cite_ref-8"><a href="#cite_note-8">[8]</a></sup> and other programmers, concerning the efficacy of the OOP paradigm versus the procedural paradigm. The need for every object to have associative methods leads some skeptics to associate OOP with <a href="/wiki/Software_bloat" title="Software bloat">software bloat</a>; an attempt to resolve this dilemma came through <a href="/wiki/Polymorphism_(computer_science)" title="Polymorphism (computer science)">polymorphism</a>.</p>
<p>Because object-oriented programming is considered a paradigm, not a language, it is possible to create even an object-oriented assembler language. <a href="/wiki/High_Level_Assembly" title="High Level Assembly">High Level Assembly</a> (HLA) is an example of this that fully supports advanced data types and object-oriented assembly language programming –  despite its early origins. Thus, differing programming paradigms can be seen rather like <i>motivational <a href="/wiki/Meme" title="Meme">memes</a></i> of their advocates, rather than necessarily representing progress from one level to the next. Precise comparisons of the efficacy of competing paradigms are frequently made more difficult because of new and differing terminology applied to similar entities and processes together with numerous implementation distinctions across languages.</p>
<h3><span class="mw-headline" id="Further_paradigms">Further paradigms</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Programming_paradigm&amp;action=edit&amp;section=6" title="Edit section: Further paradigms">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p><a href="/wiki/Literate_programming" title="Literate programming">Literate programming</a>, as a form of <a href="/wiki/Imperative_programming" title="Imperative programming">imperative programming</a>, structures programs as a human-centered web, as in a <a href="/wiki/Hypertext" title="Hypertext">hypertext</a> essay: documentation is integral to the program, and the program is structured following the logic of prose exposition, rather than compiler convenience.</p>
<p>Independent of the imperative branch, <a href="/wiki/Declarative_programming" title="Declarative programming">declarative programming</a> paradigms were developed. In these languages the computer is told what the problem is, not how to solve the problem –  the program is structured as a collection of properties to find in the expected result, not as a procedure to follow. Given a database or a set of rules, the computer tries to find a solution matching all the desired properties. An archetype of a declarative language is the <a href="/wiki/Fourth-generation_programming_language" title="Fourth-generation programming language">fourth generation language</a> <a href="/wiki/SQL" title="SQL">SQL</a>, and the family of functional languages and logic programming.</p>
<p><a href="/wiki/Functional_programming" title="Functional programming">Functional programming</a> is a subset of declarative programming. Programs written using this paradigm use <a href="/wiki/Subroutine" title="Subroutine">functions</a>, blocks of code intended to behave like <a href="/wiki/Function_(mathematics)" title="Function (mathematics)">mathematical functions</a>. Functional languages discourage changes in the value of variables through <a href="/wiki/Assignment_(computer_science)" title="Assignment (computer science)">assignment</a>, making a great deal of use of <a href="/wiki/Recursion_(computer_science)" title="Recursion (computer science)">recursion</a> instead.</p>
<p>The <a href="/wiki/Logic_programming" title="Logic programming">logic programming</a> paradigm views computation as <a href="/wiki/Automated_reasoning" title="Automated reasoning">automated reasoning</a> over a corpus of knowledge. Facts about the <a href="/wiki/Problem_domain" title="Problem domain">problem domain</a> are expressed as logic formulae, and programs are executed by applying <a class="mw-redirect" href="/wiki/Inference_rule" title="Inference rule">inference rules</a> over them until an answer to the problem is found, or the collection of formulae is proved inconsistent.</p>
<p><a href="/wiki/Symbolic_programming" title="Symbolic programming">Symbolic programming</a> is a paradigm that describes programs able to manipulate formulas and program components as data.<sup class="reference" id="cite_ref-symbolic-programming-lisp_3-1"><a href="#cite_note-symbolic-programming-lisp-3">[3]</a></sup> Programs can thus effectively modify themselves, and appear to "learn", making them suited for applications such as <a href="/wiki/Artificial_intelligence" title="Artificial intelligence">artificial intelligence</a>, <a class="mw-redirect" href="/wiki/Expert_systems" title="Expert systems">expert systems</a>, <a href="/wiki/Natural_language_processing" title="Natural language processing">natural language processing</a> and computer games. Languages that support this paradigm include <a href="/wiki/Lisp_(programming_language)" title="Lisp (programming language)">Lisp</a> and <a href="/wiki/Prolog" title="Prolog">Prolog</a>.<sup class="reference" id="cite_ref-9"><a href="#cite_note-9">[9]</a></sup></p>
<h2><span class="mw-headline" id="Multi-paradigm">Multi-paradigm</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Programming_paradigm&amp;action=edit&amp;section=7" title="Edit section: Multi-paradigm">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<div class="hatnote" role="note">See also: <a href="/wiki/Comparison_of_multi-paradigm_programming_languages" title="Comparison of multi-paradigm programming languages">Comparison of multi-paradigm programming languages</a></div>
<p>A <i>multi-paradigm programming language</i> is a <a href="/wiki/Programming_language" title="Programming language">programming language</a> that supports more than one programming paradigm.<sup class="reference" id="cite_ref-10"><a href="#cite_note-10">[10]</a></sup> The design goal of such languages is to allow programmers to use the most suitable programming style and associated language constructs for a given job, considering that no single paradigm solves all problems in the easiest or most efficient way.</p>
<p>One example is <a class="mw-redirect" href="/wiki/C_sharp_(programming_language)" title="C sharp (programming language)">C#</a>, which includes <a href="/wiki/Imperative_programming" title="Imperative programming">imperative</a> and <a href="/wiki/Object-oriented_programming" title="Object-oriented programming">object-oriented</a> paradigms, together with a certain level of support for <a href="/wiki/Functional_programming" title="Functional programming">functional programming</a> with features like <a href="/wiki/Delegate_(CLI)" title="Delegate (CLI)">delegates</a> (allowing functions to be treated as first-order objects), <a href="/wiki/Type_inference" title="Type inference">type inference</a>, <a href="/wiki/Anonymous_function" title="Anonymous function">anonymous functions</a> and <a href="/wiki/Language_Integrated_Query" title="Language Integrated Query">Language Integrated Query</a>. Other examples are <a class="mw-redirect" href="/wiki/F_sharp_(programming_language)" title="F sharp (programming language)">F#</a> and <a href="/wiki/Scala_(programming_language)" title="Scala (programming language)">Scala</a>, which provide similar functionality to C# but also include full support for <a href="/wiki/Functional_programming" title="Functional programming">functional programming</a> (including <a href="/wiki/Currying" title="Currying">currying</a>, <a href="/wiki/Pattern_matching" title="Pattern matching">pattern matching</a>, <a class="mw-redirect" href="/wiki/Algebraic_data_types" title="Algebraic data types">algebraic data types</a>, <a href="/wiki/Lazy_evaluation" title="Lazy evaluation">lazy evaluation</a>, <a class="mw-redirect" href="/wiki/Tail_recursion" title="Tail recursion">tail recursion</a>, <a href="/wiki/Immutable_object" title="Immutable object">immutability</a>, etc.). Perhaps the most extreme example is <a href="/wiki/Oz_(programming_language)" title="Oz (programming language)">Oz</a>, which has subsets that adhere to <a href="/wiki/Logic_programming" title="Logic programming">logic</a> (Oz descends from logic programming), <a href="/wiki/Functional_programming" title="Functional programming">functional</a>, <a class="mw-redirect" href="/wiki/Object-oriented_programming_language" title="Object-oriented programming language">object-oriented</a>, <a href="/wiki/Dataflow#Concurrency" title="Dataflow">dataflow concurrent</a>, and other paradigms. Oz was designed over a ten-year period to combine in a harmonious way concepts that are traditionally associated with different programming paradigms. <a href="/wiki/Lisp_(programming_language)" title="Lisp (programming language)">Lisp</a>, while often taught as a functional language, is known for its malleability and thus its ability to engulf many paradigms.</p>
