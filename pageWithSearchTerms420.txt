<p>In <a href="/wiki/Software_development" title="Software development">software development</a>, the <b>V-model</b><sup class="reference" id="cite_ref-2"><a href="#cite_note-2">[2]</a></sup> represents a <a href="/wiki/Software_development_process" title="Software development process">development process</a> that may be considered an extension of the <a href="/wiki/Waterfall_model" title="Waterfall model">waterfall model</a>, and is an example of the more <a class="mw-redirect" href="/wiki/V-model" title="V-model">general V-model</a>. Instead of moving down in a linear way, the process steps are bent upwards after the <a href="/wiki/Source_code" title="Source code">coding</a> phase, to form the typical V shape. The V-Model demonstrates the relationships between each phase of the development life cycle and its associated phase of <a href="/wiki/Software_testing" title="Software testing">testing</a>. The horizontal and vertical axes represents time or project completeness (left-to-right) and level of abstraction (coarsest-grain abstraction uppermost), respectively.</p>

<h2><span class="mw-headline" id="Verification_phases">Verification phases</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=V-Model_(software_development)&amp;action=edit&amp;section=1" title="Edit section: Verification phases">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<h3><span class="mw-headline" id="Requirements_analysis">Requirements analysis</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=V-Model_(software_development)&amp;action=edit&amp;section=2" title="Edit section: Requirements analysis">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>In the <a href="/wiki/Requirements_analysis" title="Requirements analysis">requirements analysis</a> phase, the first step in the verification process, the <a class="mw-redirect" href="/wiki/Requirements" title="Requirements">requirements</a> of the system are collected by analyzing the needs of the <a href="/wiki/User_(computing)" title="User (computing)">user(s)</a>. This phase is concerned with establishing what the ideal system has to perform. However it does not determine how the software will be designed or built. Usually, the users are interviewed and a document called the user requirements document is generated.</p>
<p>The user requirements document will typically describe the system's functional, interface, performance, data, security, etc. requirements as expected by the user. It is used by business analysts to communicate their understanding of the system to the users. The users carefully review this document as this document would serve as the guideline for the system designers in the system design phase. The user acceptance tests are designed in this phase. See also <a class="mw-redirect" href="/wiki/Functional_requirements" title="Functional requirements">Functional requirements</a>.</p>
<p>There are different methods for gathering requirements of both soft and hard methodologies including; interviews, questionnaires, document analysis, observation, throw-away prototypes, use cases and static and dynamic views with users.</p>
<h3><span class="mw-headline" id="System_design">System design</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=V-Model_(software_development)&amp;action=edit&amp;section=3" title="Edit section: System design">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p><a href="/wiki/Systems_design" title="Systems design">Systems design</a> is the phase where system engineers analyze and understand the business of the proposed system by studying the user requirements document. They figure out possibilities and techniques by which the user requirements can be implemented. If any of the requirements are not feasible, the user is informed of the issue. A resolution is found and the user requirement document is edited accordingly.</p>
<p>The software specification document which serves as a blueprint for the development phase is generated. This document contains the general system organization, menu structures, <a href="/wiki/Data_structure" title="Data structure">data structures</a> etc. It may also hold example business scenarios, sample windows, reports for the better understanding. Other technical documentation like entity diagrams, data dictionary will also be produced in this phase. The documents for system testing are prepared.</p>
<h3><span class="mw-headline" id="Architecture_design">Architecture design</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=V-Model_(software_development)&amp;action=edit&amp;section=4" title="Edit section: Architecture design">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>The phase of the design of <a href="/wiki/Computer_architecture" title="Computer architecture">computer architecture</a> and <a href="/wiki/Software_architecture" title="Software architecture">software architecture</a> can also be referred to as high-level design. The baseline in selecting the architecture is that it should realize all which typically consists of the list of modules, brief functionality of each module, their <a class="mw-redirect" href="/wiki/Interface_(computer_science)" title="Interface (computer science)">interface</a> relationships, <a class="mw-redirect" href="/wiki/Coupling_(computer_science)" title="Coupling (computer science)">dependencies</a>, <a href="/wiki/Database" title="Database">database</a> <a href="/wiki/Table_(database)" title="Table (database)">tables</a>, architecture diagrams, technology details etc. The integration testing design is carried out in the particular phase.<sup class="reference" id="cite_ref-3"><a href="#cite_note-3">[3]</a></sup></p>
<h3><span class="mw-headline" id="Module_design">Module design</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=V-Model_(software_development)&amp;action=edit&amp;section=5" title="Edit section: Module design">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>The <a class="mw-redirect" href="/wiki/Modularity_(programming)" title="Modularity (programming)">module design</a> phase can also be referred to as low-level design. The designed system is broken up into smaller units or modules and each of them is explained so that the programmer can start coding directly. The low level design document or program specifications will contain a detailed functional logic of the <a href="/wiki/Modular_programming" title="Modular programming">module</a>, in <a href="/wiki/Pseudocode" title="Pseudocode">pseudocode</a>:</p>
<ul>
<li>database tables, with all elements, including their type and size</li>
<li>all interface details with complete <a class="mw-redirect" href="/wiki/API" title="API">API</a> references</li>
<li>all dependency issues</li>
<li><a href="/wiki/Error_message" title="Error message">error message</a> listings</li>
<li>complete input and outputs for a module.</li>
</ul>
<p>The unit test design is developed in this stage.</p>
<h2><span class="mw-headline" id="Validation_phases">Validation phases</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=V-Model_(software_development)&amp;action=edit&amp;section=6" title="Edit section: Validation phases">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>In the V-model, each stage of verification phase has a corresponding stage in the validation phase.<sup class="reference" id="cite_ref-4"><a href="#cite_note-4">[4]</a></sup> The following are the typical phases of validation in the V-Model, though they may be known by other names.</p>
<h3><span class="mw-headline" id="Unit_testing">Unit testing</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=V-Model_(software_development)&amp;action=edit&amp;section=7" title="Edit section: Unit testing">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>In the V-Model, Unit Test Plans (UTPs) are developed during module design phase. These UTPs are executed to eliminate bugs at code level or unit level. A unit is the smallest entity which can independently exist, e.g. a program module. Unit testing verifies that the smallest entity can function correctly when isolated from the rest of the codes/units.</p>
<h3><span class="mw-headline" id="Integration_testing">Integration testing</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=V-Model_(software_development)&amp;action=edit&amp;section=8" title="Edit section: Integration testing">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Integration Test Plans are developed during the Architectural Design Phase. These tests verify that units created and tested independently can coexist and communicate among themselves. Test results are shared with customer's team.</p>
<h3><span class="mw-headline" id="System_testing">System testing</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=V-Model_(software_development)&amp;action=edit&amp;section=9" title="Edit section: System testing">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>System Tests Plans are developed during System Design Phase. Unlike Unit and Integration Test Plans, System Test Plans are composed by client's business team. System Test ensures that expectations from application developed are met. The whole application is tested for its functionality, interdependency and communication. System Testing verifies that functional and non-functional requirements have been met. Load and performance testing, stress testing, regression testing, etc., are subsets of system testing.</p>
<h3><span class="mw-headline" id="User_acceptance_testing">User acceptance testing</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=V-Model_(software_development)&amp;action=edit&amp;section=10" title="Edit section: User acceptance testing">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>User Acceptance Test (UAT) Plans are developed during the Requirements Analysis phase. Test Plans are composed by business users. UAT is performed in a user environment that resembles the production environment, using realistic data. UAT verifies that delivered system meets user's requirement and system is ready for use in real time.</p>
<h2><span class="mw-headline" id="Criticism">Criticism</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=V-Model_(software_development)&amp;action=edit&amp;section=11" title="Edit section: Criticism">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The V-Model has been criticized by <a href="/wiki/Agile_software_development" title="Agile software development">Agile</a> advocates and others as an inadequate model of software development for numerous reasons.<sup class="reference" id="cite_ref-5"><a href="#cite_note-5">[5]</a></sup><sup class="reference" id="cite_ref-6"><a href="#cite_note-6">[6]</a></sup><sup class="reference" id="cite_ref-7"><a href="#cite_note-7">[7]</a></sup> Criticisms include:</p>
<ol>
<li>It is too simple to accurately reflect the software development process, and can lead managers into a false sense of security. The V-Model reflects a project management view of software development and fits the needs of project managers, accountants and lawyers rather than software developers or users.</li>
<li>Although it is easily understood by novices, that early understanding is useful only if the novice goes on to acquire a deeper understanding of the development process and how the V-Model must be adapted and extended in practice. If practitioners persist with their naive view of the V-Model they will have great difficulty applying it successfully.</li>
<li>It is inflexible and encourages a rigid and linear view of software development and has no inherent ability to respond to change.</li>
<li>It provides only a slight variant on the <a href="/wiki/Waterfall_model" title="Waterfall model">waterfall model</a> and is therefore subject to the same criticisms as that model. It provides greater emphasis on testing, and particularly the importance of early test planning. However, a common practical criticism of the V-Model is that it leads to testing being squeezed into tight windows at the end of development when earlier stages have overrun but the implementation date remains fixed.</li>
<li>It is consistent with, and therefore implicitly encourages, inefficient and ineffective testing methodologies. It implicitly promotes writing test scripts in advance rather than exploratory testing; it encourages testers to look for what they expect to find, rather than discover what is truly there. It also encourages a rigid link between the equivalent levels of either leg (e.g. user acceptance test plans being derived from user requirements documents), rather than encouraging testers to select the most effective and efficient way to plan and execute testing.</li>
<li>It lacks coherence and precision. There is widespread confusion about what exactly the V-Model is. If one boils it down to those elements that most people would agree upon it becomes a trite and unhelpful representation of software development. Disagreement about the merits of the V-Model often reflects a lack of shared understanding of its definition.</li>
</ol>
<h2><span class="mw-headline" id="Current_state">Current state</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=V-Model_(software_development)&amp;action=edit&amp;section=12" title="Edit section: Current state">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Supporters of the V-Model argue that it has evolved over time and supports flexibility and agility throughout the development process.<sup class="reference" id="cite_ref-8"><a href="#cite_note-8">[8]</a></sup> They argue that in addition to being a highly disciplined approach, it promotes meticulous design, development, and documentation necessary to build stable software products. Lately, it is being adopted by the medical device industry.<sup class="reference" id="cite_ref-9"><a href="#cite_note-9">[9]</a></sup><sup class="reference" id="cite_ref-10"><a href="#cite_note-10">[10]</a></sup></p>
