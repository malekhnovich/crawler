<p>The <b>incremental build model</b> is a method of <a href="/wiki/Software_development" title="Software development">software development</a> where the product is <a href="/wiki/Software_design" title="Software design">designed</a>, implemented and <a href="/wiki/Software_testing" title="Software testing">tested</a> incrementally (a little more is added each time) until the product is finished. It involves both development and maintenance. The product is defined as finished when it satisfies all of its requirements. This model combines the elements of the <a href="/wiki/Waterfall_model" title="Waterfall model">waterfall model</a> with the iterative philosophy of <a class="mw-redirect" href="/wiki/Prototyping" title="Prototyping">prototyping</a>.</p>
<p>The product is decomposed into a number of components, each of which is designed and built separately (termed as builds). Each component is delivered to the client when it is complete. This allows partial utilization of the product and avoids a long development time. It also avoids a large initial capital outlay and subsequent long waiting period. This model of development also helps ease the traumatic effect of introducing a completely new system all at once. There are, however, several problems with this model.</p>

<h2><span class="mw-headline" id="Incremental_model">Incremental model</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Incremental_build_model&amp;action=edit&amp;section=1" title="Edit section: Incremental model">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The incremental model applies the <a href="/wiki/Waterfall_model" title="Waterfall model">waterfall model</a> incrementally.<sup class="reference" id="cite_ref-Pressman_1-0"><a href="#cite_note-Pressman-1">[1]</a></sup></p>
<p>The series of releases is referred to as “increments”, with each increment providing more functionality to the customers. After the first increment, a core product is delivered, which can already be used by the customer. Based on customer feedback, a plan is developed for the next increments, and modifications are made accordingly. This process continues, with increments being delivered until the complete product is delivered. The incremental philosophy is also used in the agile process model (see <a class="mw-redirect" href="/wiki/Agile_Modeling" title="Agile Modeling">Agile Modeling</a>).<sup class="reference" id="cite_ref-Pressman_1-1"><a href="#cite_note-Pressman-1">[1]</a></sup></p>
<p>The Incremental model is also being applied to <a href="/wiki/DevOps" title="DevOps">DevOps</a>. In DevOps it centers around the idea of minimizing risk and cost of a DevOps Adoption whilst building the necessary in-house skillset and momentum needed to have widespread – successful implementation across the enterprise.<sup class="reference" id="cite_ref-2"><a href="#cite_note-2">[2]</a></sup></p>
<p><b>Advantages</b><sup class="reference" id="cite_ref-3"><a href="#cite_note-3">[3]</a></sup><sup class="reference" id="cite_ref-4"><a href="#cite_note-4">[4]</a></sup></p>
<ol>
<li>After each iteration, regression testing should be conducted. During this testing, faulty elements of the software can be quickly identified because few changes are made within any single iteration.</li>
<li>It is generally easier to test and debug than other methods of software development because relatively smaller changes are made during each iteration. This allows for more targeted and rigorous testing of each element within the overall product.</li>
<li>Customer can respond to features and review the product for any needed or useful changes.</li>
<li>Initial product delivery is faster and costs lower.</li>
</ol>
<p><b>Disadvantages</b><sup class="reference" id="cite_ref-5"><a href="#cite_note-5">[5]</a></sup></p>
<ol>
<li>Resulting cost may exceed the cost of the organization.</li>
<li>As additional functionality is added to the product, problems may arise related to system architecture which were not evident in earlier prototypes.</li>
</ol>

<p>These tasks are common to all the models<sup class="reference" id="cite_ref-Pressman_1-2"><a href="#cite_note-Pressman-1">[1]</a></sup></p>
<ol>
<li>Communication: helps to understand the objective.</li>
<li>Planning: required as many people (software teams) work on the same project but different function at same time.</li>
<li>Modeling: involves business modeling, data modeling, and process modeling.</li>
<li>Construction: this involves the reuse software components and automatic code.</li>
<li>Deployment: integration of all the increments.</li>
</ol>
