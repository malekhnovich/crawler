<h1 class="firstHeading" id="firstHeading" lang="en">Multiton pattern</h1>
<div class="mw-body-content" id="bodyContent">
<div id="siteSub">From Wikipedia, the free encyclopedia</div>
<div id="contentSub"></div>
<div class="mw-jump" id="jump-to-nav">
					Jump to:					<a href="#mw-head">navigation</a>, 					<a href="#p-search">search</a>
</div>
<div class="mw-content-ltr" dir="ltr" id="mw-content-text" lang="en"><div class="thumb tright">
<div class="thumbinner" style="width:222px;"><a class="image" href="/wiki/File:Multiton.svg"><img alt="" class="thumbimage" data-file-height="106" data-file-width="230" height="101" src="//upload.wikimedia.org/wikipedia/commons/thumb/9/9f/Multiton.svg/220px-Multiton.svg.png" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/9/9f/Multiton.svg/330px-Multiton.svg.png 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/9/9f/Multiton.svg/440px-Multiton.svg.png 2x" width="220"/></a>
<div class="thumbcaption">
<div class="magnify"><a class="internal" href="/wiki/File:Multiton.svg" title="Enlarge"></a></div>
UML diagram of the multiton</div>
</div>

<p>In <a href="/wiki/Software_engineering" title="Software engineering">software engineering</a>, the <b>multiton pattern</b> is a <a class="mw-redirect" href="/wiki/Design_pattern_(computer_science)" title="Design pattern (computer science)">design pattern</a> similar to the <a href="/wiki/Singleton_pattern" title="Singleton pattern">singleton</a>, which allows only one instance of a class to be created. The multiton pattern expands on the singleton concept to manage a <a href="/wiki/Associative_array" title="Associative array">map</a> of named instances as key-value pairs.</p>
<p>Rather than having a single instance <i>per application</i> (e.g. the <code><a class="external text" href="//docs.oracle.com/javase/8/docs/api/java/lang/Runtime.html" rel="nofollow">java.lang.Runtime</a></code> object in the <a href="/wiki/Java_(programming_language)" title="Java (programming language)">Java programming language</a>) the multiton pattern instead ensures a single instance <i>per key</i>.</p>
<p>Most people and textbooks consider this a singleton pattern<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (April 2012)">citation needed</span></a></i>]</sup>. For example, multiton does not explicitly appear in the highly regarded <a href="/wiki/Object-oriented_programming" title="Object-oriented programming">object-oriented programming</a> textbook <i><a class="mw-redirect" href="/wiki/Design_Patterns_(book)" title="Design Patterns (book)">Design Patterns</a></i> (it appears as a more flexible approach named <b>registry of singletons</b>).</p>
<p></p>

<h2><span class="mw-headline" id="Description">Description</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Multiton_pattern&amp;action=edit&amp;section=1" title="Edit section: Description">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>While it may appear that the multiton is no more than a simple <a href="/wiki/Hash_table" title="Hash table">hash table</a> with synchronized access there are two important distinctions. First, the multiton does not allow clients to add mappings. Secondly, the multiton never returns a null or empty reference; instead, it creates and stores a multiton instance on the first request with the associated key. Subsequent requests with the same key return the original instance. A hash table is merely an implementation detail and not the only possible approach. The pattern simplifies retrieval of shared objects in an application.</p>
<p>Since the object pool is created only once, being a member associated with the class (instead of the instance), the multiton retains its flat behavior rather than evolving into a <a href="/wiki/Tree_(data_structure)" title="Tree (data structure)">tree structure</a>.</p>
<p>The multiton is unique in that it provides centralized access to a single directory (i.e. all keys are in the same namespace, <i>per se</i>) of multitons, where each multiton instance in the pool may exist having its own state. In this manner, the pattern advocates indexed storage of essential objects for the system (such as would be provided by an <a class="mw-redirect" href="/wiki/LDAP" title="LDAP">LDAP</a> system, for example). However, a multiton is limited to wide use by a single system rather than a myriad of distributed systems.</p>
<h2><span class="mw-headline" id="Drawbacks">Drawbacks</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Multiton_pattern&amp;action=edit&amp;section=2" title="Edit section: Drawbacks">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>This pattern, like the <a href="/wiki/Singleton_pattern" title="Singleton pattern">Singleton pattern</a>, makes <a href="/wiki/Unit_testing" title="Unit testing">unit testing</a> far more difficult,<sup class="reference" id="cite_ref-1"><a href="#cite_note-1">[1]</a></sup> as it introduces <a class="mw-redirect" href="/wiki/Global_variables" title="Global variables">global state</a> into an application.</p>
<p>With garbage collected languages it may become a source of memory leaks as it introduces global strong references to the objects.</p>
<h2><span class="mw-headline" id="Implementations">Implementations</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Multiton_pattern&amp;action=edit&amp;section=3" title="Edit section: Implementations">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>In Java, the multiton pattern can be implemented using an <a href="/wiki/Enumerated_type" title="Enumerated type">enumerated type</a>, with the values of the type corresponding to the instances. In the case of an enumerated type with a single value, this gives the singleton pattern.</p>
