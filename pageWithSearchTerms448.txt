<h1 class="firstHeading" id="firstHeading" lang="en">LePUS3</h1>
<div class="mw-body-content" id="bodyContent">

<p><b>LePUS3</b><sup class="reference" id="cite_ref-book_1-0"><a href="#cite_note-book-1">[1]</a></sup> is a language for modelling and visualizing object-oriented (<a href="/wiki/Java_(programming_language)" title="Java (programming language)">Java</a>, <a href="/wiki/C%2B%2B" title="C++">C++</a>, <a href="/wiki/C_Sharp_(programming_language)" title="C Sharp (programming language)">C#</a>) programs and <a href="/wiki/Design_pattern" title="Design pattern">design patterns</a>.<sup class="reference" id="cite_ref-companion_2-0"><a href="#cite_note-companion-2">[2]</a></sup> It is defined as a <a href="/wiki/Formal_specification" title="Formal specification">formal specification</a> language,<sup class="reference" id="cite_ref-3"><a href="#cite_note-3">[3]</a></sup><sup class="reference" id="cite_ref-4"><a href="#cite_note-4">[4]</a></sup> formulated as an axiomatized subset of <a class="mw-redirect" href="/wiki/First-order_predicate_logic" title="First-order predicate logic">First-order predicate logic</a>. A diagram in LePUS3 is also called a <b>Codechart</b>.<sup class="reference" id="cite_ref-5"><a href="#cite_note-5">[5]</a></sup> LePUS, the name of the first version of the language, is an abbreviation for <i>Language for Pattern Uniform Specification</i>.</p>
<p></p>

<h2><span class="mw-headline" id="Purpose">Purpose</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=LePUS3&amp;action=edit&amp;section=1" title="Edit section: Purpose">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>LePUS3 is tailored for the following purposes:</p>
<ul>
<li><i>Scalability</i>: To model large-scale programs using small Codecharts with only few symbols</li>
<li><i>Automated verifiability</i>: To allow programmers to check fully automatically conformance to design so as to keep the design in synch with the implementation</li>
<li><i>Program visualization</i>: To allow tools to reverse-engineer legible Codecharts from plain source code modeling their design</li>
<li><i>Pattern implementation</i>: To allow tools to determine automatically whether your program implements a design pattern</li>
<li><i>Design abstraction</i>: To specify unimplemented programs without committing prematurely to implementation minutiae</li>
<li><i>Generality</i>: To model a design pattern not as a specific implementation but as a design motif</li>
<li><i>Rigour</i>: To allow software designers to be sure exactly what Codecharts represent and reason rigorously about them</li>
</ul>
<h2><span class="mw-headline" id="Context">Context</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=LePUS3&amp;action=edit&amp;section=2" title="Edit section: Context">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>LePUS3 belongs to the following families of languages:</p>
<ul>
<li><i>Object-oriented software <a href="/wiki/Modeling_language" title="Modeling language">modeling languages</a></i> (e.g., <a href="/wiki/Unified_Modeling_Language" title="Unified Modeling Language">UML</a>): LePUS3 is a visual notation that is used to represent the building-blocks in the design of programs <a href="/wiki/Object-oriented_programming" title="Object-oriented programming">object-oriented programming</a> languages</li>
<li><i><a href="/wiki/Formal_specification" title="Formal specification">Formal specification</a> languages</i>: Like other Logic Visual Languages, Codecharts articulate sentences in mathematical logic. LePUS3 is axiomatized in and defined as a recursive (turing-decidable) subset of first-order predicate calculus. Its semantics are defined using finite <a href="/wiki/Structure_(mathematical_logic)" title="Structure (mathematical logic)">structure (mathematical logic)</a>.</li>
<li><i><a href="/wiki/Architecture_description_language" title="Architecture description language">Architecture description languages</a></i>: LePUS3 is a non-functional <a class="mw-redirect" href="/wiki/Specification" title="Specification">specification</a> language used to represent design decisions about programs in class-based object-oriented programming languages (such as <a href="/wiki/Java_(programming_language)" title="Java (programming language)">Java</a> and <a href="/wiki/C%2B%2B" title="C++">C++</a>).</li>
<li><i>Tool supported specification languages</i>: Verification of Codecharts (checking their consistency with a Java 1.4 program) can be established ('verified') by a click of a button, as demonstrated by the <a class="external text" href="http://ttp.essex.ac.uk/index.php?page=toolkit" rel="nofollow">Two-Tier Programming Toolkit</a>.</li>
<li><i><a href="/wiki/Software_visualization" title="Software visualization">Software visualization</a> notations</i> are notations which offer a graphical representation of the program, normally generated by reverse-engineering the source code of the program.</li>
</ul>
<h2><span class="mw-headline" id="Vocabulary">Vocabulary</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=LePUS3&amp;action=edit&amp;section=3" title="Edit section: Vocabulary">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>LePUS3 was designed to accommodate for parsimony and for economy of expression. Its vocabulary consists of only 15 visual tokens.</p>

<h2><span class="mw-headline" id="Tool_support">Tool support</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=LePUS3&amp;action=edit&amp;section=4" title="Edit section: Tool support">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The Two-Tier Programming Toolkit<sup class="reference" id="cite_ref-toolkit_6-0"><a href="#cite_note-toolkit-6">[6]</a></sup><sup class="reference" id="cite_ref-7"><a href="#cite_note-7">[7]</a></sup> can be used to</p>
<ul>
<li>create LePUS3 specifications (Codecharts)</li>
<li>verify automatically<sup class="reference" id="cite_ref-nasa09_8-0"><a href="#cite_note-nasa09-8">[8]</a></sup> the consistency of LePUS3 Codecharts with Java 1.4 programs; and</li>
<li><a class="external text" href="http://ttp.essex.ac.uk/index.php?page=navigation" rel="nofollow">reverse-engineer</a> Codecharts from Java source code.</li>
</ul>
<h2><span class="mw-headline" id="Design_patterns">Design patterns</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=LePUS3&amp;action=edit&amp;section=5" title="Edit section: Design patterns">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>LePUS3 was specifically designed to model, among others, the <a href="/wiki/Design_pattern" title="Design pattern">'Gang of Four' design patterns</a>, including <a href="/wiki/Abstract_factory_pattern" title="Abstract factory pattern">abstract factory</a>, <a href="/wiki/Factory_method_pattern" title="Factory method pattern">factory method</a>, <a href="/wiki/Adapter_pattern" title="Adapter pattern">adapter</a>, <a href="/wiki/Decorator_pattern" title="Decorator pattern">decorator</a>, <a href="/wiki/Composite_pattern" title="Composite pattern">composite</a>, <a href="/wiki/Proxy_pattern" title="Proxy pattern">proxy</a>, <a href="/wiki/Iterator_pattern" title="Iterator pattern">iterator</a>, <a href="/wiki/State_pattern" title="State pattern">state</a>, <a href="/wiki/Strategy_pattern" title="Strategy pattern">strategy</a>, <a href="/wiki/Template_method_pattern" title="Template method pattern">template method</a>, and <a href="/wiki/Visitor_pattern" title="Visitor pattern">visitor</a>. (See <a class="external text" href="http://www.lepus.org.uk/ref/companion/index.xml" rel="nofollow">"The 'Gang of Four' Companion")</a> <sup class="reference" id="cite_ref-companion_2-1"><a href="#cite_note-companion-2">[2]</a></sup> The abbreviation LePUS for "Language for Pattern Uniform Specification" is used because the precursor of this language was primarily concerned with design patterns. The implementation of design patterns specified in LePUS3 can be automatically verified by the TTP Toolkit.<sup class="reference" id="cite_ref-nasa09_8-1"><a href="#cite_note-nasa09-8">[8]</a></sup></p>
<h2><span class="mw-headline" id="Examples">Examples</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=LePUS3&amp;action=edit&amp;section=6" title="Edit section: Examples">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>LePUS3 is particularly suitable for modelling large programs, <a class="mw-redirect" href="/wiki/Design_patterns" title="Design patterns">design patterns</a>, and object-oriented <a href="/wiki/Application_framework" title="Application framework">application frameworks</a>. It is unsuitable for modelling non object-oriented programs, <a href="/wiki/Architectural_style" title="Architectural style">architectural styles</a>, and <a href="/wiki/Undecidable_problem" title="Undecidable problem">undecidable</a> and <a class="mw-redirect" href="/wiki/Semi-decidable" title="Semi-decidable">semi-decidable</a> properties.</p>
