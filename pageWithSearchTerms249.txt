<h1 class="firstHeading" id="firstHeading" lang="en">Software archaeology</h1>
<div class="mw-body-content" id="bodyContent">
<div id="siteSub">From Wikipedia, the free encyclopedia</div>
<div id="contentSub"></div>
<div class="mw-jump" id="jump-to-nav">
					Jump to:					<a href="#mw-head">navigation</a>, 					<a href="#p-search">search</a>
</div>
<div class="mw-content-ltr" dir="ltr" id="mw-content-text" lang="en"><p><b>Software archaeology</b> or <b>software archeology</b> is the study of poorly documented or undocumented <a href="/wiki/Legacy_system" title="Legacy system">legacy software</a> implementations, as part of <a href="/wiki/Software_maintenance" title="Software maintenance">software maintenance</a>.<sup class="reference" id="cite_ref-RGBH_1-0"><a href="#cite_note-RGBH-1">[1]</a></sup><sup class="reference" id="cite_ref-2"><a href="#cite_note-2">[2]</a></sup> Software archaeology, named by analogy with <a href="/wiki/Archaeology" title="Archaeology">archaeology</a>,<sup class="reference" id="cite_ref-3"><a href="#cite_note-3">[3]</a></sup> includes the <a href="/wiki/Reverse_engineering" title="Reverse engineering">reverse engineering</a> of software modules, and the application of a variety of tools and processes for extracting and understanding program structure and recovering design information.<sup class="reference" id="cite_ref-RGBH_1-1"><a href="#cite_note-RGBH-1">[1]</a></sup><sup class="reference" id="cite_ref-4"><a href="#cite_note-4">[4]</a></sup> Software archaeology may reveal dysfunctional team processes which have produced poorly designed or even unused software modules.<sup class="reference" id="cite_ref-5"><a href="#cite_note-5">[5]</a></sup> The term has been in use for decades,<sup class="reference" id="cite_ref-6"><a href="#cite_note-6">[6]</a></sup> and reflects a fairly natural metaphor: a programmer reading <a href="/wiki/Legacy_code" title="Legacy code">legacy code</a> may feel that he or she is in the same situation as an archaeologist exploring the rubble of an ancient civilization.<sup class="reference" id="cite_ref-AndyDave_7-0"><a href="#cite_note-AndyDave-7">[7]</a></sup></p>
<p></p>

<h2><span class="mw-headline" id="Techniques">Techniques</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Software_archaeology&amp;action=edit&amp;section=1" title="Edit section: Techniques">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>A workshop on Software Archaeology at the 2001 <a href="/wiki/OOPSLA" title="OOPSLA">OOPSLA</a> (Object-Oriented Programming, Systems, Languages &amp; Applications) conference identified the following software archaeology techniques, some of which are specific to <a href="/wiki/Object-oriented_programming" title="Object-oriented programming">object-oriented programming</a>:<sup class="reference" id="cite_ref-AndyDave_7-1"><a href="#cite_note-AndyDave-7">[7]</a></sup></p>
<ul>
<li><a href="/wiki/Scripting_language" title="Scripting language">Scripting languages</a> to build static reports and for filtering diagnostic output</li>
<li>Ongoing documentation in HTML pages or Wikis</li>
<li>Synoptic signature analysis, statistical analysis, and <a href="/wiki/Software_visualization" title="Software visualization">software visualization</a> tools</li>
<li>Reverse-engineering tools</li>
<li>Operating-system-level tracing via <a href="/wiki/Truss_(Unix)" title="Truss (Unix)">truss</a> or <a href="/wiki/Strace" title="Strace">strace</a></li>
<li>Search engines and tools to search for keywords in source files</li>
<li><a href="/wiki/Integrated_development_environment" title="Integrated development environment">IDE</a> file browsing</li>
<li><a href="/wiki/Unit_testing" title="Unit testing">Unit testing</a> frameworks such as <a href="/wiki/JUnit" title="JUnit">JUnit</a> and <a href="/wiki/CppUnit" title="CppUnit">CppUnit</a></li>
<li>API documentation generation using tools such as <a href="/wiki/Javadoc" title="Javadoc">Javadoc</a> and <a href="/wiki/Doxygen" title="Doxygen">doxygen</a></li>
<li><a href="/wiki/Debugger" title="Debugger">Debuggers</a></li>
</ul>
<p>More generally, <a href="/wiki/Andy_Hunt_(author)" title="Andy Hunt (author)">Andy Hunt</a> and <a href="/wiki/Dave_Thomas_(programmer)" title="Dave Thomas (programmer)">Dave Thomas</a> note the importance of <a class="mw-redirect" href="/wiki/Revision_control" title="Revision control">version control</a>, dependency management, text indexing tools such as <a href="/wiki/GLIMPSE" title="GLIMPSE">GLIMPSE</a> and <a href="/wiki/SWISH-E" title="SWISH-E">SWISH-E</a>, and "[drawing] a map as you begin exploring."<sup class="reference" id="cite_ref-AndyDave_7-2"><a href="#cite_note-AndyDave-7">[7]</a></sup></p>
<p>Like true archaeology, software archaeology involves investigative work to understand the thought processes of one's predecessors.<sup class="reference" id="cite_ref-AndyDave_7-3"><a href="#cite_note-AndyDave-7">[7]</a></sup> At the OOPSLA workshop, <a href="/wiki/Ward_Cunningham" title="Ward Cunningham">Ward Cunningham</a> suggested a synoptic signature analysis technique which gave an overall "feel" for a program by showing only punctuation, such as semicolons and <a href="/wiki/Block_(programming)" title="Block (programming)">curly braces</a>.<sup class="reference" id="cite_ref-8"><a href="#cite_note-8">[8]</a></sup> In the same vein, Cunningham has suggested viewing programs in 2 point font in order to understand the overall structure.<sup class="reference" id="cite_ref-9"><a href="#cite_note-9">[9]</a></sup> Another technique identified at the workshop was the use of <a href="/wiki/Aspect-oriented_programming" title="Aspect-oriented programming">aspect-oriented programming</a> tools such as <a href="/wiki/AspectJ" title="AspectJ">AspectJ</a> to systematically introduce <a href="/wiki/Tracing_(software)" title="Tracing (software)">tracing</a> code without directly editing the legacy program.<sup class="reference" id="cite_ref-AndyDave_7-4"><a href="#cite_note-AndyDave-7">[7]</a></sup></p>
<p>Network and temporal analysis techniques can reveal the patterns of collaborative activity by the developers of legacy software, which in turn may shed light on the strengths and weaknesses of the software artifacts produced.<sup class="reference" id="cite_ref-10"><a href="#cite_note-10">[10]</a></sup></p>
<p>Michael Rozlog of <a href="/wiki/Embarcadero_Technologies" title="Embarcadero Technologies">Embarcadero Technologies</a> has described software archaeology as a six-step process which enables programmers to answer questions such as "What have I just inherited?" and "Where are the scary sections of the code?"<sup class="reference" id="cite_ref-Rozlog_11-0"><a href="#cite_note-Rozlog-11">[11]</a></sup> These steps, similar to those identified by the OOPSLA workshop, include using visualization to obtain a visual representation of the program's design, using <a href="/wiki/Software_metric" title="Software metric">software metrics</a> to look for design and style violations, using <a href="/wiki/Unit_testing" title="Unit testing">unit testing</a> and <a href="/wiki/Profiling_(computer_programming)" title="Profiling (computer programming)">profiling</a> to look for bugs and performance bottlenecks, and assembling design information recovered by the process.<sup class="reference" id="cite_ref-Rozlog_11-1"><a href="#cite_note-Rozlog-11">[11]</a></sup> Software archaeology can also be a service provided to programmers by external consultants.<sup class="reference" id="cite_ref-12"><a href="#cite_note-12">[12]</a></sup></p>
<p><span id="DataArchaeology"></span> Mitch Rosenberg of InfoVentions.net, Inc. claims<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (June 2016)">citation needed</span></a></i>]</sup> that the first law of software archaeology (he calls it code or data archaeology) is:</p>
<p><i>Everything that is there is there for a reason, and there are 3 possible reasons:</i></p>
<ol>
<li><i>It used to need to be there but no longer does</i></li>
<li><i>It never needed to be there and the person that wrote the code had no clue</i></li>
<li><i>It STILL needs to be there and YOU have no clue</i></li>
</ol>
<p><i>The corollary to this "law" is that, until you know which was the reason, you should NOT modify the code (or data).</i></p>
<p>Software archaeology has continued to be a topic of discussion at more recent software engineering conferences.<sup class="reference" id="cite_ref-13"><a href="#cite_note-13">[13]</a></sup></p>
