<h1 class="firstHeading" id="firstHeading" lang="en">Executable UML</h1>
<div class="mw-body-content" id="bodyContent">
<div id="siteSub">From Wikipedia, the free encyclopedia</div>
<div id="contentSub"></div>
<div class="mw-jump" id="jump-to-nav">
					Jump to:					<a href="#mw-head">navigation</a>, 					<a href="#p-search">search</a>
</div>
<div class="mw-content-ltr" dir="ltr" id="mw-content-text" lang="en"><p><b>Executable UML</b> (<b>xtUML</b> or <b>xUML</b>) is both a software development method and a highly abstract software language. The language "combines a subset of the UML (<a href="/wiki/Unified_Modeling_Language" title="Unified Modeling Language">Unified Modeling Language</a>) graphical notation with executable semantics and timing rules."<sup class="reference" id="cite_ref-1"><a href="#cite_note-1">[1]</a></sup> The Executable UML method is the successor to the <a href="/wiki/Shlaer%E2%80%93Mellor_method" title="Shlaer–Mellor method">Shlaer–Mellor method</a>.<sup class="reference" id="cite_ref-2"><a href="#cite_note-2">[2]</a></sup></p>
<p>Executable UML models "can be run, <a href="/wiki/Software_testing" title="Software testing">tested</a>, debugged, and measured for performance.",<sup class="reference" id="cite_ref-3"><a href="#cite_note-3">[3]</a></sup> and can be <a href="/wiki/Compiler" title="Compiler">compiled</a> into a less abstract <a href="/wiki/Programming_language" title="Programming language">programming language</a> to target a specific <a href="/wiki/Implementation" title="Implementation">implementation</a>.<sup class="reference" id="cite_ref-4"><a href="#cite_note-4">[4]</a></sup> Executable UML supports <a href="/wiki/Model-driven_architecture" title="Model-driven architecture">model-driven architecture</a> (MDA) through specification of <a href="/wiki/Platform-independent_model" title="Platform-independent model">platform-independent models</a>, and the <a href="/wiki/Compiler" title="Compiler">compilation</a> of the <a href="/wiki/Platform-independent_model" title="Platform-independent model">platform-independent models</a> into <a href="/wiki/Platform-specific_model" title="Platform-specific model">platform-specific models</a>.<sup class="reference" id="cite_ref-5"><a href="#cite_note-5">[5]</a></sup><sup class="reference" id="cite_ref-6"><a href="#cite_note-6">[6]</a></sup></p>
<p></p>

<h2><span class="mw-headline" id="Overview">Overview</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Executable_UML&amp;action=edit&amp;section=1" title="Edit section: Overview">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Executable UML is a higher level of <a class="mw-redirect" href="/wiki/Abstraction_(computer_science)" title="Abstraction (computer science)">abstraction</a> than <a href="/wiki/Third-generation_programming_language" title="Third-generation programming language">third-generation programming languages</a>. This allows <a href="/wiki/Software_developer" title="Software developer">developers</a> to develop at the level of abstraction of the application.<sup class="reference" id="cite_ref-7"><a href="#cite_note-7">[7]</a></sup> The Executable UML aims for <a href="/wiki/Separation_of_concerns" title="Separation of concerns">separation of concerns</a>. This is supposed to increase ease of <a href="/wiki/Code_reuse" title="Code reuse">reuse</a> and to lower the cost of <a href="/wiki/Software_development" title="Software development">software development</a>. This also enables Executable UML domains to be <a href="/wiki/Cross-platform" title="Cross-platform">cross-platform</a>. That means it is not tied to any specific programming language, platform or technology.</p>
<p>Executable UML also allows for translation of <a href="/wiki/Platform-independent_model" title="Platform-independent model">platform-independent models</a> (PIM) into <a href="/wiki/Platform-specific_model" title="Platform-specific model">platform-specific models</a> (PSM). The Executable UML method enables valuing the model as <a href="/wiki/Intellectual_property" title="Intellectual property">intellectual property</a>, since the model is a fully executable solution for the problem space.</p>
<p>Actions are specified in <a href="/wiki/Action_(UML)" title="Action (UML)">action language</a>. This means that the automatic generation of implementation code from Executable UML models can be output in an optimized form.</p>
<p>Executable UML is intended to serve as executable code as well as documentation. The models are a graphical, executable specification of the problem space that is compiled into a target <a href="/wiki/Implementation" title="Implementation">implementation</a>. They are also intended to be <a class="mw-redirect" href="/wiki/Human-readable" title="Human-readable">human-readable</a>.</p>
<h2><span class="mw-headline" id="Executable_UML_building_blocks">Executable UML building blocks</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Executable_UML&amp;action=edit&amp;section=2" title="Edit section: Executable UML building blocks">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>A <a href="/wiki/Application_software" title="Application software">system</a> is composed of multiple subject matters, known as <i>domains</i> in Executable UML terms. Executable UML is used to model a domain at the level of <a class="mw-redirect" href="/wiki/Abstraction_(computer_science)" title="Abstraction (computer science)">abstraction</a> of its subject matter independent of implementation concerns. The resulting domain model is represented by the following elements:</p>
<ul>
<li>The <i>domain chart</i> provides a view of the <a href="/wiki/Aspect_(computer_programming)" title="Aspect (computer programming)">domain</a> being modeled, and the dependencies it has on other domains.</li>
<li>The <i>class diagram</i> defines the <a href="/wiki/Class_(computer_programming)" title="Class (computer programming)">classes</a> and class <a href="/wiki/Association_(object-oriented_programming)" title="Association (object-oriented programming)">associations</a> for the domain.</li>
<li>The <i>statechart diagram</i> defines the <a href="/wiki/Finite-state_machine" title="Finite-state machine">states</a>, events, and state transitions for a class or class instance.</li>
<li>The <i>action language</i> defines the actions or operations that perform processing on model elements.</li>
</ul>
<h3><span class="mw-headline" id="Domain_Chart">Domain Chart</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Executable_UML&amp;action=edit&amp;section=3" title="Edit section: Domain Chart">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Executable UML requires identification of the domains (also known as: <a href="/wiki/Aspect_(computer_programming)" title="Aspect (computer programming)">aspects</a><sup class="reference" id="cite_ref-8"><a href="#cite_note-8">[8]</a></sup> or <a href="/wiki/Concern_(computer_science)" title="Concern (computer science)">concerns</a>) of the system. "Each domain is an autonomous world inhabited by conceptual entities"<sup class="reference" id="cite_ref-9"><a href="#cite_note-9">[9]</a></sup> Each domain can be modeled independent of the other domains in the system, enabling a <a href="/wiki/Separation_of_concerns" title="Separation of concerns">separation of concerns</a>. As an example, domains for an automated teller system may include the following:</p>
<ul>
<li>The application domain model of the automated teller's <a href="/wiki/Business_logic" title="Business logic">business logic</a>.</li>
<li>The <a href="/wiki/Computer_security" title="Computer security">security</a> domain model of various issues regarding system security (such as <a href="/wiki/Authentication" title="Authentication">authentication</a> and <a href="/wiki/Encryption" title="Encryption">encryption</a>).</li>
<li>The data access domain model of methods for external <a href="/wiki/Data_(computing)" title="Data (computing)">data</a> usage.</li>
<li>The <a class="mw-redirect" href="/wiki/Data_logging" title="Data logging">logging</a> domain model of the various methods through which the system can or must log information.</li>
<li>The <a href="/wiki/User_interface" title="User interface">user interface</a> domain model of the user interactions with the system.</li>
<li>The <a href="/wiki/Computing_platform" title="Computing platform">architecture</a> domain model of the <a href="/wiki/Implementation" title="Implementation">implemented</a> of the Executable UML model on the system's <a href="/wiki/Computer_hardware" title="Computer hardware">hardware</a> and <a href="/wiki/Software" title="Software">software</a> platforms.</li>
</ul>
<p>The <i>separation of concerns</i> enables each domain to be developed and verified independently of the other domains in the system by the respective domain experts.</p>
<p>The connections between domains are called <i>bridges</i>. "A bridge is a layering dependency between domains".<sup class="reference" id="cite_ref-10"><a href="#cite_note-10">[10]</a></sup> This means that the domains can place requirements upon other domains. It is recommended that bridges are agreed upon by the different domain experts.</p>
<p>A domain can be marked as <i>realized</i> to indicate that the domain exists and does not need to be modeled. For example, a data access domain that uses a <a href="/wiki/MySQL" title="MySQL">MySQL</a> database would be marked as realized.</p>
<h3><span class="mw-headline" id="Class_diagram">Class diagram</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Executable_UML&amp;action=edit&amp;section=4" title="Edit section: Class diagram">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="hatnote" role="note">See also: <a href="/wiki/Class_diagram" title="Class diagram">Class diagram</a></div>
<p>Conceptual entities, such as tangible things, roles, incidents, interactions, and specifications, specific to the domain being modeled are abstracted into <i>classes</i>. Classes can have <i>attributes</i> and <i>operations</i>.</p>
<p>The relationships between these classes will be indicated with <i>associations</i> and <i>generalizations</i>. An association may require further abstraction as an <i>Association Class</i>.</p>
<p>Constraints on the class diagram can be written in both Action Language and <a href="/wiki/Object_Constraint_Language" title="Object Constraint Language">Object Constraint Language</a> (OCL).</p>
<p>The Executable UML method limits the UML elements that can be used in an Executable UML class diagram.</p>
<p>An Executable UML class diagram is meant to expose information about the domain. Too much complexity in the statechart diagrams is a good indicator that the class diagram should be reworked.</p>
<h3><span class="mw-headline" id="Statechart_Diagram">Statechart Diagram</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Executable_UML&amp;action=edit&amp;section=5" title="Edit section: Statechart Diagram">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="hatnote" role="note">See also: <a href="/wiki/Finite-state_machine" title="Finite-state machine">Finite-state machine</a> and <a href="/wiki/State_diagram" title="State diagram">State diagram</a></div>
<p>Classes have lifecycles which are modeled in Executable UML with a <a href="/wiki/State_diagram" title="State diagram">statechart diagram</a>. The statechart diagram defines the <i>states</i>, <i>transitions</i>, <i>events</i>, and <i>procedures</i> that define a class' behaviour.</p>
<p>Each state has only one procedure that is executed upon <a href="/wiki/Moore_machine" title="Moore machine">entry into that state</a>. A procedure is composed of actions, which are specified in an action language.</p>
<h3><span class="mw-headline" id="Action_Language">Action Language</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Executable_UML&amp;action=edit&amp;section=6" title="Edit section: Action Language">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>The class and state models by themselves can only provide a static view of the domain. In order to have an executable model, there must be a way to create class instances, establish associations, perform operations on attributes, call state events, etc. In Executable UML, this is done using an action language that conforms to the UML Action Semantics.</p>
<p>Action Semantics was added to the UML specification in 2001. The Action Semantics RFP was based on previous work in action languages supporting the <a href="/wiki/Shlaer%E2%80%93Mellor_method" title="Shlaer–Mellor method">Shlaer–Mellor method</a>. Existing action languages are Object Action Language (OAL), Shlaer–Mellor Action Language (SMALL), Action Specification Language (ASL), That Action Language (TALL), Starr's Concise Relational Action Language (SCRALL), Platform-independent Action Language (PAL) and PathMATE Action Language (PAL). SCRALL is the only one that is a graphical action language.</p>
<h3><span class="mw-headline" id="Model_testing_and_execution">Model testing and execution</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Executable_UML&amp;action=edit&amp;section=7" title="Edit section: Model testing and execution">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Once a domain is modeled, it can be <a href="/wiki/Software_testing" title="Software testing">tested</a> independent of the target <a href="/wiki/Implementation" title="Implementation">implementation</a> by executing the model. Each domain can be <a href="/wiki/Verification_and_validation" title="Verification and validation">verified and validated</a> independent of any other domain. This allows <a href="/wiki/Software_bug" title="Software bug">errors</a> detected to be associated with the domain and independent of other system concerns.</p>
<p><a class="mw-redirect" href="/wiki/Static_testing" title="Static testing">Verification</a> will involve such things as human <a href="/wiki/Peer_review" title="Peer review">review</a> of the models, performed by experts in the relevant domain, and automated checking of the Executable UML semantics. i.e., checking that the Executable UML model complies with the Executable UML <a href="/wiki/Meta" title="Meta">metamodel</a>.</p>
<p><a href="/wiki/Dynamic_testing" title="Dynamic testing">Validation</a> will typically involve use of an Executable UML tool to execute the model. The execution can occur either before or after model compilation.</p>
<h3><span class="mw-headline" id="Model_compilation">Model compilation</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Executable_UML&amp;action=edit&amp;section=8" title="Edit section: Model compilation">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>In order to support execution on the target <a href="/wiki/Implementation" title="Implementation">implementation</a>, the domain model must be translated into a less <a class="mw-redirect" href="/wiki/Abstraction_(computer_science)" title="Abstraction (computer science)">abstract</a> form. This translation process is called <i>model compilation</i>. Most model <a class="mw-redirect" href="/wiki/Compilation_(programming)" title="Compilation (programming)">compilers</a> target a known <a href="/wiki/Programming_language" title="Programming language">programming language</a>, because this allows reuse of existing <a href="/wiki/Compiler" title="Compiler">compiler</a> technologies.</p>
<p><a href="/wiki/Program_optimization" title="Program optimization">Optimizing</a> the domain models for target <a href="/wiki/Implementation" title="Implementation">implementation</a> reasons will reduce the level of <a class="mw-redirect" href="/wiki/Abstraction_(computer_science)" title="Abstraction (computer science)">abstraction</a>, adversely affect domain independence, and increase the cost of <a href="/wiki/Code_reuse" title="Code reuse">reuse</a>. In executable UML, <a class="mw-redirect" href="/wiki/Optimization_(computer_science)" title="Optimization (computer science)">optimizations</a> are done by the model <a class="mw-redirect" href="/wiki/Compilation_(programming)" title="Compilation (programming)">compiler</a> either automatically or through <i>marking</i>. Marking allows specific model elements to be targeted for specific lower-level <a href="/wiki/Implementation" title="Implementation">implementations</a>, and allows for broader architectural decisions, such as specifying that <a href="/wiki/Container_(abstract_data_type)" title="Container (abstract data type)">collections</a> of objects should be <a href="/wiki/Implementation" title="Implementation">implemented</a> as a <a href="/wiki/Doubly_linked_list" title="Doubly linked list">doubly linked list</a>.</p>
<p>In <a href="/wiki/Model-driven_architecture" title="Model-driven architecture">MDA</a> terms, the model <a class="mw-redirect" href="/wiki/Compilation_(programming)" title="Compilation (programming)">compiler</a> creates the <a href="/wiki/Platform-specific_model" title="Platform-specific model">PSM</a>. The separation between the <a href="/wiki/Platform-independent_model" title="Platform-independent model">PIM</a> and <a href="/wiki/Platform-specific_model" title="Platform-specific model">PSM</a> in Executable UML disables the ability to <a href="/wiki/Round-trip_engineering" title="Round-trip engineering">round-trip engineer</a> the model, and deters modifications to the <a href="/wiki/Platform-specific_model" title="Platform-specific model">PSM</a>.<sup class="reference" id="cite_ref-11"><a href="#cite_note-11">[11]</a></sup></p>
<h2><span class="mw-headline" id="Executable_UML_Key_Aspects">Executable UML Key Aspects</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Executable_UML&amp;action=edit&amp;section=9" title="Edit section: Executable UML Key Aspects">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Executable UML defines execution semantics for a subset of the UML. Key aspects of the Executable UML subset include the following:</p>
<ul>
<li>No support for implementation specific constructs, like aggregation and composition.<sup class="reference" id="cite_ref-12"><a href="#cite_note-12">[12]</a></sup></li>
<li>Generalizations are always notated as {complete, disjoint}.</li>
<li>Associations between classes are <i>always</i> named, have verb phrases on both ends specifying the roles, and have multiplicity specified on both ends.</li>
<li>Multiplicities on association ends are restricted to 0..1 (zero to one), * (zero to many), 1 (exactly one), or 1..* (one to many).</li>
<li><a href="/wiki/Data_type" title="Data type">Data types</a> are restricted to the following core data types: boolean, string, integer, real, date, timestamp, and arbitrary_id, or one of the following domain-specific data types: numeric, string, enumerated, and composite. Domain-specific numeric and string data types can represent subsets of the core data types. The domain-specific composite data type is to always be treated as a single unit within the domain. e.g., a <i>MailingAddress</i> composite data type could be declared, but city information couldn't be extracted from it.</li>
<li><a href="/wiki/Constraint_(information_theory)" title="Constraint (information theory)">Constraints</a> on the Executable UML models can either be represented as <a href="/wiki/Object_Constraint_Language" title="Object Constraint Language">Object Constraint Language</a> (OCL) or action language.</li>
</ul>
<h2><span class="mw-headline" id="fUML_and_ALF">fUML and ALF</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Executable_UML&amp;action=edit&amp;section=10" title="Edit section: fUML and ALF">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The <a href="/wiki/Object_Management_Group" title="Object Management Group">Object Management Group</a> has recently standardized the <a class="mw-redirect" href="/wiki/FUML" title="FUML">Foundational UML (fUML)</a>. fUML is strongly influenced by Executable UML, but <em><a class="mw-redirect" href="/wiki/FUML" title="FUML">Foundational UML (fUML)</a> is a not the Executable UML that is covered in this article!</em> See <a href="/wiki/Talk:Executable_UML#Is_fUML_Executable_UML.3F" title="Talk:Executable UML">Discussion</a>.</p>
<p>Action Language for Foundational UML (ALF), is a new standard action language proposed by the <a href="/wiki/Object_Management_Group" title="Object Management Group">Object Management Group</a>.</p>
