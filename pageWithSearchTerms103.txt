<p><b>General responsibility assignment software patterns</b> (or <b>principles</b>), abbreviated <b>GRASP</b>, consist of guidelines for assigning responsibility to classes and objects in <a href="/wiki/Object-oriented_design" title="Object-oriented design">object-oriented design</a>.</p>
<p>The different patterns and principles used in GRASP are: controller, creator, indirection, information expert, high <a href="/wiki/Cohesion_(computer_science)" title="Cohesion (computer science)">cohesion</a>, low <a class="mw-redirect" href="/wiki/Coupling_(computer_science)" title="Coupling (computer science)">coupling</a>, <a class="mw-redirect" href="/wiki/Polymorphism_(object-oriented_programming)" title="Polymorphism (object-oriented programming)">polymorphism</a>, protected variations, and pure fabrication. All these patterns answer some <a href="/wiki/Software" title="Software">software</a> problem, and these problems are common to almost every <a href="/wiki/Software_development" title="Software development">software development</a> project. These techniques have not been invented to create new ways of working, but to better document and standardize old, tried-and-tested <a href="/wiki/Computer_programming" title="Computer programming">programming</a> principles in object-oriented design.</p>
<p>Computer scientist <a href="/wiki/Craig_Larman" title="Craig Larman">Craig Larman</a> states that "the critical design tool for software development is a mind well educated in design principles. It is not the <a href="/wiki/Unified_Modeling_Language" title="Unified Modeling Language">UML</a> or any other technology."<sup class="reference" id="cite_ref-FOOTNOTELarman2005272_1-0"><a href="#cite_note-FOOTNOTELarman2005272-1">[1]</a></sup> Thus, GRASP are really a mental toolset, a learning aid to help in the design of object-oriented software.</p>

<h2><span class="mw-headline" id="Patterns">Patterns</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=GRASP_(object-oriented_design)&amp;action=edit&amp;section=1" title="Edit section: Patterns">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<h3><span class="mw-headline" id="Controller">Controller</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=GRASP_(object-oriented_design)&amp;action=edit&amp;section=2" title="Edit section: Controller">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>The <b>controller</b> pattern assigns the responsibility of dealing with system events to a non-<a class="mw-redirect" href="/wiki/User_Interface" title="User Interface">UI</a> class that represents the overall system or a <a href="/wiki/Use_case" title="Use case">use case</a> scenario. A controller object is a non-user interface object responsible for receiving or handling a system event.</p>
<p>A use case controller should be used to deal with <i>all</i> system events of a use case, and may be used for more than one use case (for instance, for use cases <i>Create User</i> and <i>Delete User</i>, one can have a single <i>UserController</i>, instead of two separate use case controllers).</p>
<p>It is defined as the first object beyond the UI layer that receives and coordinates ("controls") a system operation. The controller should delegate the work that needs to be done to other objects; it coordinates or controls the activity. It should not do much work itself. The GRASP Controller can be thought of as being a part of the application/service layer <sup class="reference" id="cite_ref-2"><a href="#cite_note-2">[2]</a></sup> (assuming that the application has made an explicit distinction between the application/service layer and the <a class="mw-redirect" href="/wiki/Domain_layer" title="Domain layer">domain layer</a>) in an object-oriented system with common layers in an information system logical architecture.</p>
<h3><span class="mw-headline" id="Creator">Creator</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=GRASP_(object-oriented_design)&amp;action=edit&amp;section=3" title="Edit section: Creator">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="hatnote" role="note">See also: <a class="mw-redirect" href="/wiki/Factory_pattern" title="Factory pattern">Factory pattern</a></div>
<p>Creation of objects is one of the most common activities in an object-oriented system. Which class is responsible for creating objects is a fundamental property of the relationship between objects of particular classes.</p>
<p>In general, a class <code>B</code> should be responsible for creating instances of class <code>A</code> if one, or preferably more, of the following apply:</p>
<ul>
<li>Instances of <code>B</code> contain or compositely aggregate instances of <code>A</code></li>
<li>Instances of <code>B</code> record instances of <code>A</code></li>
<li>Instances of <code>B</code> closely use instances of <code>A</code></li>
<li>Instances of <code>B</code> have the initializing information for instances of <code>A</code> and pass it on creation.</li>
</ul>
<h3><span class="mw-headline" id="High_cohesion">High cohesion</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=GRASP_(object-oriented_design)&amp;action=edit&amp;section=4" title="Edit section: High cohesion">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="hatnote" role="note">Main article: <a href="/wiki/Cohesion_(computer_science)" title="Cohesion (computer science)">Cohesion (computer science)</a></div>
<p><b>High cohesion</b> is an evaluative pattern that attempts to keep objects appropriately focused, manageable and understandable. High cohesion is generally used in support of low coupling. High cohesion means that the responsibilities of a given element are strongly related and highly focused. Breaking programs into classes and subsystems is an example of activities that increase the cohesive properties of a system. Alternatively, low cohesion is a situation in which a given element has too many unrelated responsibilities. Elements with low cohesion often suffer from being hard to comprehend, hard to reuse, hard to maintain and averse to change.<sup class="reference" id="cite_ref-larman314_3-0"><a href="#cite_note-larman314-3">[3]</a></sup></p>
<h3><span class="mw-headline" id="Indirection">Indirection</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=GRASP_(object-oriented_design)&amp;action=edit&amp;section=5" title="Edit section: Indirection">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="hatnote" role="note">See also: <a href="/wiki/Delegation_pattern" title="Delegation pattern">Delegation pattern</a></div>
<p>The <b>indirection</b> pattern supports low coupling (and reuse potential) between two elements by assigning the responsibility of mediation between them to an intermediate object. An example of this is the introduction of a controller component for mediation between data (model) and its representation (view) in the <a class="mw-redirect" href="/wiki/Model-view-controller" title="Model-view-controller">model-view-controller</a> pattern.</p>
<h3><span class="mw-headline" id="Information_expert">Information expert</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=GRASP_(object-oriented_design)&amp;action=edit&amp;section=6" title="Edit section: Information expert">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="hatnote" role="note">See also: <a href="/wiki/Information_hiding" title="Information hiding">Information hiding</a></div>
<p><b>Information expert</b> (also <b>expert</b> or the <b>expert principle</b>) is a principle used to determine where to delegate responsibilities. These responsibilities include methods, computed fields, and so on.</p>
<p>Using the principle of information expert, a general approach to assigning responsibilities is to look at a given responsibility, determine the information needed to fulfill it, and then determine where that information is stored.</p>
<p>Information expert will lead to placing the responsibility on the class with the most information required to fulfill it.<sup class="reference" id="cite_ref-4"><a href="#cite_note-4">[4]</a></sup></p>
<h3><span class="mw-headline" id="Low_coupling">Low coupling</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=GRASP_(object-oriented_design)&amp;action=edit&amp;section=7" title="Edit section: Low coupling">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="hatnote" role="note">Main article: <a href="/wiki/Loose_coupling" title="Loose coupling">Loose coupling</a></div>
<p>Coupling is a measure of how strongly one element is connected to, has knowledge of, or relies on other elements. <b>Low coupling</b> is an evaluative pattern that dictates how to assign responsibilities to support:</p>
<ul>
<li>lower dependency between the classes,</li>
<li>change in one class having lower impact on other classes,</li>
<li>higher reuse potential.</li>
</ul>
<h3><span class="mw-headline" id="Polymorphism">Polymorphism</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=GRASP_(object-oriented_design)&amp;action=edit&amp;section=8" title="Edit section: Polymorphism">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="hatnote" role="note">Main article: <a class="mw-redirect" href="/wiki/Polymorphism_in_object-oriented_programming" title="Polymorphism in object-oriented programming">Polymorphism in object-oriented programming</a></div>
<p>According to <b>polymorphism</b> principle, responsibility of defining the variation of behaviors based on type is assigned to the type for which this variation happens. This is achieved using <a href="/wiki/Polymorphism_(computer_science)" title="Polymorphism (computer science)">polymorphic</a> operations. The user of the type should use polymorphic operations instead of explicit branching based on type.</p>
<h3><span class="mw-headline" id="Protected_variations">Protected variations</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=GRASP_(object-oriented_design)&amp;action=edit&amp;section=9" title="Edit section: Protected variations">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="hatnote" role="note">See also: <a href="/wiki/Open/closed_principle" title="Open/closed principle">Open/closed principle</a></div>
<p>The <b>protected variations</b> pattern protects elements from the variations on other elements (objects, systems, subsystems) by wrapping the focus of instability with an <a class="mw-redirect" href="/wiki/Interface_(computer_science)" title="Interface (computer science)">interface</a> and using <a href="/wiki/Polymorphism_(computer_science)" title="Polymorphism (computer science)">polymorphism</a> to create various implementations of this interface.</p>
<h3><span class="mw-headline" id="Pure_fabrication">Pure fabrication</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=GRASP_(object-oriented_design)&amp;action=edit&amp;section=10" title="Edit section: Pure fabrication">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="hatnote" role="note">See also: <a href="/wiki/Service_(systems_architecture)" title="Service (systems architecture)">Service (systems architecture)</a></div>
<p>A <b>pure fabrication</b> is a class that does not represent a concept in the problem domain, specially made up to achieve low coupling, high cohesion, and the reuse potential thereof derived (when a solution presented by the <i>information expert</i> pattern does not). This kind of class is called a "service" in <a href="/wiki/Domain-driven_design" title="Domain-driven design">domain-driven design</a>.</p>
