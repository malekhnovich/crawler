<h1 class="firstHeading" id="firstHeading" lang="en">Code refactoring</h1>
<div class="mw-body-content" id="bodyContent">
<div id="siteSub">From Wikipedia, the free encyclopedia</div>
<div id="contentSub"><span class="mw-redirectedfrom">  (Redirected from <a class="mw-redirect" href="/w/index.php?title=Refactoring&amp;redirect=no" title="Refactoring">Refactoring</a>)</span></div>
<div class="mw-jump" id="jump-to-nav">
					Jump to:					<a href="#mw-head">navigation</a>, 					<a href="#p-search">search</a>
</div>
<div class="mw-content-ltr" dir="ltr" id="mw-content-text" lang="en"><div class="hatnote" role="note">
<div class="hatnote" role="note"><span class="plainlinks selfreference noprint">"Refactor" redirects here. For the use of "refactor" on Wikipedia, see <a href="/wiki/Wikipedia:Refactoring_talk_pages" title="Wikipedia:Refactoring talk pages">Wikipedia:Refactoring talk pages</a>.</span></div>
</div>
<p><b>Code refactoring</b> is the process of restructuring existing computer code—changing the <i><a href="/wiki/Decomposition_(computer_science)" title="Decomposition (computer science)">factoring</a></i>—without changing its external behavior. Refactoring improves <i><a href="/wiki/Non-functional_requirement" title="Non-functional requirement">nonfunctional</a></i> attributes of the <a href="/wiki/Software" title="Software">software</a>. Advantages include improved code <a href="/wiki/Readability" title="Readability">readability</a> and reduced <a href="/wiki/Cyclomatic_complexity" title="Cyclomatic complexity">complexity</a>; these can improve <a href="/wiki/Source_code" title="Source code">source-code</a> <a href="/wiki/Maintainability" title="Maintainability">maintainability</a> and create a more expressive internal <a href="/wiki/Software_architecture" title="Software architecture">architecture</a> or <a href="/wiki/Object_model" title="Object model">object model</a> to improve <a href="/wiki/Extensibility" title="Extensibility">extensibility</a>. Typically, refactoring applies a series of standardised basic <i>micro-refactorings</i>, each of which is (usually) a tiny change in a <a href="/wiki/Computer_program" title="Computer program">computer program</a>'s source code that either preserves the behaviour of the software, or at least does not modify its conformance to <a href="/wiki/Functional_requirement" title="Functional requirement">functional requirements</a>. Many <a class="mw-redirect" href="/wiki/Development_environment_(software_development_process)" title="Development environment (software development process)">development environments</a> provide automated support for performing the mechanical aspects of these basic refactorings. If done extremely well, code refactoring may also resolve hidden, dormant, or undiscovered <a class="mw-redirect" href="/wiki/Computer_bug" title="Computer bug">computer bugs</a> or <a href="/wiki/Vulnerability_(computing)" title="Vulnerability (computing)">vulnerabilities</a> in the system by simplifying the underlying logic and eliminating unnecessary levels of complexity. If done poorly it may fail the requirement that external functionality not be changed, introduce new bugs, or both.</p>
<blockquote class="templatequote">
<p>By continuously improving the design of code, we make it easier and easier to work with. This is in sharp contrast to what typically happens: little refactoring and a great deal of attention paid to expediently adding new features. If you get into the hygienic habit of refactoring continuously, you'll find that it is easier to extend and maintain code.</p>
<div class="templatequotecite"><cite>— Joshua Kerievsky, <i>Refactoring to Patterns</i><sup class="reference" id="cite_ref-kerievsky_1-0"><a href="#cite_note-kerievsky-1">[1]</a></sup></cite></div>
</blockquote>
<p></p>

<h2><span class="mw-headline" id="Overview">Overview</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Code_refactoring&amp;action=edit&amp;section=1" title="Edit section: Overview">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Refactoring is usually motivated by noticing a <a href="/wiki/Code_smell" title="Code smell">code smell</a>.<sup class="reference" id="cite_ref-fowler_2-0"><a href="#cite_note-fowler-2">[2]</a></sup> For example the method at hand may be very long, or it may be a near duplicate of another nearby method. Once recognized, such problems can be addressed by <i>refactoring</i> the source code, or transforming it into a new form that behaves the same as before but that no longer "smells". For a long routine, one or more smaller subroutines can be extracted; or for duplicate routines, the duplication can be removed and replaced with one shared function. Failure to perform refactoring can result in accumulating <a href="/wiki/Technical_debt" title="Technical debt">technical debt</a>; on the other hand, refactoring is one of the primary means of repaying technical debt.<sup class="reference" id="cite_ref-3"><a href="#cite_note-3">[3]</a></sup></p>
<p>There are two general categories of benefits to the activity of refactoring.</p>
<ol>
<li>Maintainability. It is easier to fix bugs because the source code is easy to read and the intent of its author is easy to grasp.<sup class="reference" id="cite_ref-martin_4-0"><a href="#cite_note-martin-4">[4]</a></sup> This might be achieved by reducing large monolithic routines into a set of individually concise, well-named, single-purpose methods. It might be achieved by moving a method to a more appropriate class, or by removing misleading comments.</li>
<li>Extensibility. It is easier to extend the capabilities of the application if it uses recognizable <a class="mw-redirect" href="/wiki/Design_patterns" title="Design patterns">design patterns</a>, and it provides some flexibility where none before may have existed.<sup class="reference" id="cite_ref-kerievsky_1-1"><a href="#cite_note-kerievsky-1">[1]</a></sup></li>
</ol>
<p>Before refactoring a section of code, a solid set of automatic <a href="/wiki/Unit_testing" title="Unit testing">unit tests</a> is needed. The tests are used to demonstrate that the behavior of the module is correct before the refactoring.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (July 2016)">citation needed</span></a></i>]</sup> If a test fails, it is generally best to fix the test first; if this is not done, it is hard to distinguish between failures introduced by refactoring and failures that were already present. After refactoring, the tests are run again to verify the refactoring did not break the tests. Unit tests cannot prove that there are no bugs, but the important point is that this process can be cost-effective: good unit tests can catch enough errors to make them worthwhile and to make refactoring safe enough.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (July 2016)">citation needed</span></a></i>]</sup></p>
<p>The process is then an iterative cycle of making a small <a href="/wiki/Program_transformation" title="Program transformation">program transformation</a>, testing it to ensure correctness, and making another small transformation. If at any point a test fails, the last small change is undone and repeated in a different way. Through many small steps the program moves from where it was to where you want it to be. For this very iterative process to be practical, the tests must run very quickly, or the programmer would have to spend a large fraction of his or her time waiting for the tests to finish. Proponents of <a href="/wiki/Extreme_programming" title="Extreme programming">extreme programming</a> and other <a href="/wiki/Agile_software_development" title="Agile software development">agile software development</a> describe this activity as an integral part of the <a href="/wiki/Software_development_process" title="Software development process">software development cycle</a>.</p>
<h2><span class="mw-headline" id="List_of_refactoring_techniques">List of refactoring techniques</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Code_refactoring&amp;action=edit&amp;section=2" title="Edit section: List of refactoring techniques">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Here are some examples of micro-refactorings; some of these may only apply to certain languages or language types. A longer list can be found in Fowler's refactoring book<sup class="reference" id="cite_ref-fowler_2-1"><a href="#cite_note-fowler-2">[2]</a></sup> and website.<sup class="reference" id="cite_ref-5"><a href="#cite_note-5">[5]</a></sup> Many development environments provide automated support for these micro-refactorings. For instance, a programmer could click on the name of a variable and then select the "Encapsulate field" refactoring from a context menu. The IDE would then prompt for additional details, typically with sensible defaults and a preview of the code changes. After confirmation by the programmer it would carry out the required changes throughout the code.</p>
<ul>
<li>Techniques that allow for more <a class="mw-redirect" href="/wiki/Abstraction_(computer_science)" title="Abstraction (computer science)">abstraction</a>
<ul>
<li><a href="/wiki/Field_encapsulation" title="Field encapsulation">Encapsulate Field</a> – force code to access the field with getter and setter methods</li>
<li><a href="/wiki/Type_generalization" title="Type generalization">Generalize Type</a> – create more general types to allow for more code sharing</li>
<li>Replace type-checking code with State/Strategy<sup class="reference" id="cite_ref-6"><a href="#cite_note-6">[6]</a></sup></li>
<li>Replace conditional with <a href="/wiki/Polymorphism_(computer_science)" title="Polymorphism (computer science)">polymorphism</a> <sup class="reference" id="cite_ref-7"><a href="#cite_note-7">[7]</a></sup></li>
</ul>
</li>
<li>Techniques for breaking code apart into more logical pieces
<ul>
<li>Componentization breaks code down into reusable semantic units that present clear, well-defined, simple-to-use interfaces.</li>
<li><a class="mw-redirect" href="/wiki/Extract_Class" title="Extract Class">Extract Class</a> moves part of the code from an existing class into a new class.</li>
<li>Extract Method, to turn part of a larger <a class="mw-redirect" href="/wiki/Method_(computer_science)" title="Method (computer science)">method</a> into a new method. By breaking down code in smaller pieces, it is more easily understandable. This is also applicable to <a class="mw-redirect" href="/wiki/Function_(programming)" title="Function (programming)">functions</a>.</li>
</ul>
</li>
<li>Techniques for improving names and location of code
<ul>
<li>Move Method or Move Field – move to a more appropriate <a class="mw-redirect" href="/wiki/Class_(computer_science)" title="Class (computer science)">Class</a> or source file</li>
<li><a class="mw-redirect" href="/wiki/Rename_Method" title="Rename Method">Rename Method</a> or Rename Field – changing the name into a new one that better reveals its purpose</li>
<li><a href="/wiki/Pull_Up_refactoring" title="Pull Up refactoring">Pull Up</a> – in <a href="/wiki/Object-oriented_programming" title="Object-oriented programming">OOP</a>, move to a <a class="mw-redirect" href="/wiki/Superclass_(computer_science)" title="Superclass (computer science)">superclass</a></li>
<li><a href="/wiki/Push_Down" title="Push Down">Push Down</a> – in <a href="/wiki/Object-oriented_programming" title="Object-oriented programming">OOP</a>, move to a <a class="mw-redirect" href="/wiki/Subclass_(computer_science)" title="Subclass (computer science)">subclass</a></li>
</ul>
</li>
</ul>
<h2><span class="mw-headline" id="Hardware_refactoring">Hardware refactoring</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Code_refactoring&amp;action=edit&amp;section=3" title="Edit section: Hardware refactoring">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>While the term <i>refactoring</i> originally referred exclusively to refactoring of software code, in recent years code written in <a href="/wiki/Hardware_description_language" title="Hardware description language">hardware description languages</a> (HDLs) has also been refactored. The term <i>hardware refactoring</i> is used as a shorthand term for refactoring of code in hardware description languages. Since HDLs are not considered to be <a href="/wiki/Programming_language" title="Programming language">programming languages</a> by most hardware engineers,<sup class="reference" id="cite_ref-8"><a href="#cite_note-8">[8]</a></sup> hardware refactoring is to be considered a separate field from traditional code refactoring.</p>
<p>Automated refactoring of analog hardware descriptions (in <a href="/wiki/VHDL-AMS" title="VHDL-AMS">VHDL-AMS</a>) has been proposed by Zeng and Huss.<sup class="reference" id="cite_ref-9"><a href="#cite_note-9">[9]</a></sup> In their approach, refactoring preserves the simulated behavior of a hardware design. The non-functional measurement that improves is that refactored code can be processed by standard synthesis tools, while the original code cannot. Refactoring of digital HDLs, albeit manual refactoring, has also been investigated by <a href="/wiki/Synopsys" title="Synopsys">Synopsys</a> <a href="/wiki/Fellow" title="Fellow">fellow</a> Mike Keating.<sup class="reference" id="cite_ref-10"><a href="#cite_note-10">[10]</a></sup><sup class="reference" id="cite_ref-11"><a href="#cite_note-11">[11]</a></sup> His target is to make complex systems easier to understand, which increases the designers' productivity.</p>
<h2><span class="mw-headline" id="History">History</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Code_refactoring&amp;action=edit&amp;section=4" title="Edit section: History">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Although refactoring code has been done informally for years, <a href="/wiki/Bill_Griswold" title="Bill Griswold">William Griswold</a>'s 1991 Ph.D. dissertation<sup class="reference" id="cite_ref-griswold-thesis_12-0"><a href="#cite_note-griswold-thesis-12">[12]</a></sup> is one of the first major academic works on refactoring functional and procedural programs, followed by <a href="/wiki/William_Opdyke" title="William Opdyke">William Opdyke</a>'s 1992 dissertation<sup class="reference" id="cite_ref-opdyke-thesis_13-0"><a href="#cite_note-opdyke-thesis-13">[13]</a></sup> on the refactoring of object-oriented programs,<sup class="reference" id="cite_ref-etymology_14-0"><a href="#cite_note-etymology-14">[14]</a></sup> although all the theory and machinery have long been available as <a href="/wiki/Program_transformation" title="Program transformation">program transformation</a> systems. All of these resources provide a catalog of common methods for refactoring; a refactoring method has a description of how to apply the <a href="/wiki/Scientific_method" title="Scientific method">method</a> and indicators for when you should (or should not) apply the method.</p>
<p><a href="/wiki/Martin_Fowler" title="Martin Fowler">Martin Fowler</a>'s book <i>Refactoring: Improving the Design of Existing Code</i><sup class="reference" id="cite_ref-fowler_2-2"><a href="#cite_note-fowler-2">[2]</a></sup> is the canonical reference.</p>
<p>The first known use of the term "refactoring" in the published literature was in a September, 1990 article by <a href="/wiki/William_Opdyke" title="William Opdyke">William Opdyke</a> and <a href="/wiki/Ralph_Johnson_(computer_scientist)" title="Ralph Johnson (computer scientist)">Ralph Johnson</a>.<sup class="reference" id="cite_ref-opdyke90_15-0"><a href="#cite_note-opdyke90-15">[15]</a></sup> Griswold's Ph.D. thesis,<sup class="reference" id="cite_ref-griswold-thesis_12-1"><a href="#cite_note-griswold-thesis-12">[12]</a></sup> Opdyke's Ph.D. thesis,<sup class="reference" id="cite_ref-opdyke-thesis_13-1"><a href="#cite_note-opdyke-thesis-13">[13]</a></sup> published in 1992, also used this term.<sup class="reference" id="cite_ref-etymology_14-1"><a href="#cite_note-etymology-14">[14]</a></sup></p>
<p>The term "factoring" has been used in the <a href="/wiki/Forth_(programming_language)" title="Forth (programming language)">Forth</a> community since at least the early 1980s. Chapter Six of <a class="new" href="/w/index.php?title=Leo_Brodie_(programmer)&amp;action=edit&amp;redlink=1" title="Leo Brodie (programmer) (page does not exist)">Leo Brodie</a>'s book <i><a class="new" href="/w/index.php?title=Thinking_Forth&amp;action=edit&amp;redlink=1" title="Thinking Forth (page does not exist)">Thinking Forth</a></i> (1984) is dedicated to the subject.</p>
<p>In extreme programming, the Extract Method refactoring technique has essentially the same meaning as factoring in Forth; to break down a "word" (or <a class="mw-redirect" href="/wiki/Function_(programming)" title="Function (programming)">function</a>) into smaller, more easily maintained functions.</p>
<p>Refactorings can also be reconstructed<sup class="reference" id="cite_ref-weissgerber2006_16-0"><a href="#cite_note-weissgerber2006-16">[16]</a></sup> posthoc to produce concise descriptions of complex software changes recorded in software repositories like CVS or SVN.</p>
<h2><span class="mw-headline" id="Automated_code_refactoring">Automated code refactoring</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Code_refactoring&amp;action=edit&amp;section=5" title="Edit section: Automated code refactoring">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Many software <a href="/wiki/Text_editor" title="Text editor">editors</a> and <a href="/wiki/Integrated_development_environment" title="Integrated development environment">IDEs</a> have automated refactoring support. Here is a list of a few of these editors, or so-called <a class="mw-redirect" href="/wiki/Refactoring_Browser" title="Refactoring Browser">refactoring browsers</a>.</p>
<ul>
<li><a href="/wiki/IntelliJ_IDEA" title="IntelliJ IDEA">IntelliJ IDEA</a> (for <a href="/wiki/Java_(programming_language)" title="Java (programming language)">Java</a>)</li>
