<p>A <b>functional specification</b> (also, <i>functional spec</i>, <i>specs</i>, <i>functional specifications document (FSD)</i>, <i>functional requirements specification</i>) in <a href="/wiki/Systems_engineering" title="Systems engineering">systems engineering</a> and <a href="/wiki/Software_development" title="Software development">software development</a> is a document that specifies the functions that a system or component must perform (often part of a requirements specification) (ISO/IEC/IEEE 24765-2010).<sup class="reference" id="cite_ref-2"><a href="#cite_note-2">[2]</a></sup></p>
<p>The documentation typically describes what is needed by the system user as well as requested properties of inputs and outputs (e.g. of the <a href="/wiki/Software" title="Software">software</a> system). A functional specification is the more technical response to a matching requirements document, e.g. the <a href="/wiki/Product_requirements_document" title="Product requirements document">Product Requirement Document</a> "PRD"<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (April 2016)">citation needed</span></a></i>]</sup>. Thus it picks up the results of the <a href="/wiki/Requirements_analysis" title="Requirements analysis">requirements analysis</a> stage. On more complex systems multiple levels of functional specifications will typically nest to each other, e.g. on the system level, on the module level and on the level of technical details.</p>

<h2><span class="mw-headline" id="Overview">Overview</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Functional_specification&amp;action=edit&amp;section=1" title="Edit section: Overview">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>A <a href="/wiki/Function_(engineering)" title="Function (engineering)">functional</a> specification does not define the inner workings of the proposed system; it does not include the specification of how the system function will be implemented. Instead, it focuses on what various outside agents (people using the program, computer peripherals, or other computers, for example) might "observe" when interacting with the system.</p>
<p>A functional requirement in a functional specification might state as follows:</p>
<dl>
<dd><i>When the user clicks the OK button, the dialog is closed and the focus is returned to the main window in the state it was in before this dialog was displayed.</i></dd>
</dl>
<p>Such a requirement describes an interaction between an external agent (the <a href="/wiki/User_(computing)" title="User (computing)">user</a>) and the software system. When the user provides input to the system by clicking the OK button, the program responds (or should respond) by closing the dialog window containing the OK button.</p>
<h2><span class="mw-headline" id="Functional_specification_topics">Functional specification topics</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Functional_specification&amp;action=edit&amp;section=2" title="Edit section: Functional specification topics">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<h3><span class="mw-headline" id="Purpose">Purpose</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Functional_specification&amp;action=edit&amp;section=3" title="Edit section: Purpose">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>There are many purposes for functional specifications. One of the primary purposes on team projects is to achieve some form of team consensus on what the program is to achieve before making the more time-consuming effort of writing <a href="/wiki/Source_code" title="Source code">source code</a> and <a href="/wiki/Test_case" title="Test case">test cases</a>, followed by a period of <a href="/wiki/Debugging" title="Debugging">debugging</a>. Typically, such consensus is reached after one or more reviews by the <a href="/wiki/Stakeholder_(corporate)" title="Stakeholder (corporate)">stakeholders</a> on the project at hand after having negotiated a cost-effective way to achieve the requirements the software needs to fulfill.</p>
<ol>
<li>To let the <a href="/wiki/Software_developer" title="Software developer">developers</a> know what to build.</li>
<li>To let the <a class="new" href="/w/index.php?title=Testers&amp;action=edit&amp;redlink=1" title="Testers (page does not exist)">testers</a> know what tests to run.</li>
<li>To let <a href="/wiki/Stakeholder_(corporate)" title="Stakeholder (corporate)">stakeholders</a> know what they are getting.</li>
</ol>
<h3><span class="mw-headline" id="Process">Process</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Functional_specification&amp;action=edit&amp;section=4" title="Edit section: Process">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>In the ordered industrial software engineering life-cycle (<a href="/wiki/Waterfall_model" title="Waterfall model">waterfall model</a>), functional specification describes <i>what</i> has to be implemented. The next, <a href="/wiki/Systems_architecture" title="Systems architecture">Systems architecture</a> document describes <i>how</i> the functions will be realized using a chosen software environment. In non industrial, prototypical systems development, functional specifications are typically written after or as part of <a href="/wiki/Requirements_analysis" title="Requirements analysis">requirements analysis</a>.</p>
<p>When the team agrees that functional specification consensus is reached, the functional spec is typically declared "complete" or "signed off". After this, typically the software development and testing team write source code and test cases using the functional specification as the reference. While testing is performed, the behavior of the program is compared against the expected behavior as defined in the functional specification.</p>
<h3><span class="mw-headline" id="Methods">Methods</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Functional_specification&amp;action=edit&amp;section=5" title="Edit section: Methods">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>One popular method of writing a functional specification document involves drawing or rendering either simple wireframes or accurate, graphically designed UI screenshots. After this has been completed, and the screen examples are approved by all stakeholders, graphical elements can be numbered and written instructions can be added for each number on the screen example. For example, a login screen can have the username field labeled '1' and password field labeled '2,' and then each number can be declared in writing, for use by software engineers and later for beta testing purposes to ensure that functionality is as intended. The benefit of this method is that countless additional details can be attached to the screen examples.</p>
