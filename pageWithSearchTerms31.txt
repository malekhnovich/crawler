<p>In <a href="/wiki/Object-oriented_programming" title="Object-oriented programming">object-oriented programming</a>, <b>inheritance</b> is when an <a href="/wiki/Object_(computer_science)" title="Object (computer science)">object</a> or <a href="/wiki/Class_(computer_programming)" title="Class (computer programming)">class</a> is based on another object (<a href="/wiki/Prototype-based_programming" title="Prototype-based programming">prototypal inheritance</a>) or class (<a href="/wiki/Class-based_programming" title="Class-based programming">class-based inheritance</a>), using the same implementation (inheriting from an object or class) specifying implementation to maintain the same behavior (realizing an interface; inheriting behavior; <a class="new" href="/w/index.php?title=Programming_by_difference&amp;action=edit&amp;redlink=1" title="Programming by difference (page does not exist)">programming by difference</a><sup class="reference" id="cite_ref-1"><a href="#cite_note-1">[1]</a></sup>). Such an inherited class is called a <b>subclass</b> of its parent class or super class. It is a mechanism for <a href="/wiki/Code_reuse" title="Code reuse">code reuse</a> and to allow independent extensions of the original software via public classes and interfaces. The relationships of objects or classes through inheritance give rise to a <a href="/wiki/Hierarchy" title="Hierarchy">hierarchy</a>. Inheritance was invented in 1967 for <a href="/wiki/Simula" title="Simula">Simula</a>.<sup class="reference" id="cite_ref-2"><a href="#cite_note-2">[2]</a></sup> The term "inheritance" is loosely used for both class-based and prototype-based programming, but in narrow use is reserved for class-based programming (one class <i>inherits from</i> another), with the corresponding technique in prototype-based programming being instead called <i><a href="/wiki/Delegation_(object-oriented_programming)" title="Delegation (object-oriented programming)">delegation</a></i> (one object <i>delegates to</i> another).</p>
<p>Inheritance <a href="#Inheritance_vs_subtyping">should not be confused</a> with <a href="/wiki/Subtyping" title="Subtyping">subtyping</a>.<sup class="reference" id="cite_ref-3"><a href="#cite_note-3">[3]</a></sup><sup class="reference" id="cite_ref-4"><a href="#cite_note-4">[4]</a></sup> In some languages inheritance and subtyping agree,<sup class="reference" id="cite_ref-5"><a href="#cite_note-5">[a]</a></sup> whereas in others they differ; in general, subtyping establishes an <a href="/wiki/Is-a" title="Is-a">is-a</a> relationship, whereas inheritance only reuses implementation and establishes a syntactic relationship, not necessarily a semantic relationship (inheritance does not ensure <a class="mw-redirect" href="/wiki/Behavioral_subtyping" title="Behavioral subtyping">behavioral subtyping</a>). To distinguish these concepts, subtyping is also known as <i>interface inheritance</i>, whereas inheritance as defined here is known as <i>implementation inheritance</i> or <i>code inheritance</i>.<sup class="reference" id="cite_ref-Mikhajlov_6-0"><a href="#cite_note-Mikhajlov-6">[5]</a></sup> Still, inheritance is a commonly used mechanism for establishing subtype relationships.<sup class="reference" id="cite_ref-7"><a href="#cite_note-7">[6]</a></sup></p>
<p>Inheritance is contrasted with <a href="/wiki/Object_composition" title="Object composition">object composition</a>, where one object <i>contains</i> another object (or objects of one class contain objects of another class); see <a href="/wiki/Composition_over_inheritance" title="Composition over inheritance">composition over inheritance</a>. Composition implements a <a href="/wiki/Has-a" title="Has-a">has-a</a> relationship, in contrast to the is-a relationship of subtyping.</p>

<h2><span class="mw-headline" id="Types_of_inheritance">Types of inheritance</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Inheritance_(object-oriented_programming)&amp;action=edit&amp;section=1" title="Edit section: Types of inheritance">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<div class="thumb tright">
<div class="thumbinner" style="width:172px;"><a class="image" href="/wiki/File:Single_Inheritance.jpg"><img alt="" class="thumbimage" data-file-height="300" data-file-width="300" height="170" src="//upload.wikimedia.org/wikipedia/en/thumb/3/32/Single_Inheritance.jpg/170px-Single_Inheritance.jpg" srcset="//upload.wikimedia.org/wikipedia/en/thumb/3/32/Single_Inheritance.jpg/255px-Single_Inheritance.jpg 1.5x, //upload.wikimedia.org/wikipedia/en/3/32/Single_Inheritance.jpg 2x" width="170"/></a>
<div class="thumbcaption">
<div class="magnify"><a class="internal" href="/wiki/File:Single_Inheritance.jpg" title="Enlarge"></a></div>
Single inheritance</div>
</div>
</div>
<div class="thumb tright">
<div class="thumbinner" style="width:172px;"><a class="image" href="/wiki/File:Multiple_Inheritance.jpg"><img alt="" class="thumbimage" data-file-height="300" data-file-width="300" height="170" src="//upload.wikimedia.org/wikipedia/en/thumb/6/69/Multiple_Inheritance.jpg/170px-Multiple_Inheritance.jpg" srcset="//upload.wikimedia.org/wikipedia/en/thumb/6/69/Multiple_Inheritance.jpg/255px-Multiple_Inheritance.jpg 1.5x, //upload.wikimedia.org/wikipedia/en/6/69/Multiple_Inheritance.jpg 2x" width="170"/></a>
<div class="thumbcaption">
<div class="magnify"><a class="internal" href="/wiki/File:Multiple_Inheritance.jpg" title="Enlarge"></a></div>
Multiple inheritance</div>
</div>
</div>
<p>There are various types of inheritance, based on paradigm and specific language.</p>
<dl>
<dt>Single inheritance</dt>
<dd>where subclasses inherit the features of one superclass. A class acquires the properties of another class.</dd>
<dt>Multiple inheritance</dt>
<dd>where one class can have more than one superclass and inherit features from all parent classes.</dd>
</dl>
<blockquote class="templatequote">
<p>"Multiple Inheritance (object-oriented programming) was widely supposed to be very difficult to implement efficiently. For example, in a summary of <a href="/wiki/C%2B%2B" title="C++">C++</a> in his book on <a href="/wiki/Objective-C" title="Objective-C">objective C</a> Brd.Cox actually claimed that adding Multiple inheritance to C++ was impossible. Thus, multiple inheritance seemed more of a challenge. Since I had considered multiple inheritance as early as 1982 and found a simple and efficient implementation technique in 1984. I couldn't resist the challenge. I suspect this to be the only case in which fashion affected the sequence of events."<sup class="reference" id="cite_ref-8"><a href="#cite_note-8">[7]</a></sup></p>
<div class="templatequotecite"><cite>— <a href="/wiki/Bjarne_Stroustrup" title="Bjarne Stroustrup">Bjarne Stroustrup</a></cite></div>
</blockquote>
<dl>
<dt>Multilevel inheritance</dt>
<dd>where a subclass is inherited from another subclass. It is not uncommon that a class is derived from another derived class as shown in the figure "Multilevel inheritance".
<div class="thumb tright">
<div class="thumbinner" style="width:172px;"><a class="image" href="/wiki/File:Multilevel_Inheritance.jpg"><img alt="" class="thumbimage" data-file-height="350" data-file-width="300" height="198" src="//upload.wikimedia.org/wikipedia/en/thumb/0/0e/Multilevel_Inheritance.jpg/170px-Multilevel_Inheritance.jpg" srcset="//upload.wikimedia.org/wikipedia/en/thumb/0/0e/Multilevel_Inheritance.jpg/255px-Multilevel_Inheritance.jpg 1.5x, //upload.wikimedia.org/wikipedia/en/0/0e/Multilevel_Inheritance.jpg 2x" width="170"/></a>
<div class="thumbcaption">
<div class="magnify"><a class="internal" href="/wiki/File:Multilevel_Inheritance.jpg" title="Enlarge"></a></div>
Multilevel inheritance</div>
</div>
</div>
The class <i>A</i> serves as a <i>base class</i> for the <i>derived class</i> <i>B</i>, which in turn serves as a <i>base class</i> for the <i>derived class</i> <i>C</i>. The class <i>B</i> is known as <i>intermediate</i> base class because it provides a link for the inheritance between <i>A</i> and <i>C</i>. The chain <i>ABC</i> is known as <i>inheritance path</i>.</dd>
</dl>
<p>A derived class with multilevel inheritance is declared as follows:</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="n">Class</span> <span class="nf">A</span><span class="p">(...);</span>      <span class="c1">//Base class</span>
<span class="n">Class</span> <span class="nl">B</span> <span class="p">:</span> <span class="k">public</span> <span class="n">A</span><span class="p">(...);</span>   <span class="c1">//B derived from A</span>
<span class="n">Class</span> <span class="nl">C</span> <span class="p">:</span> <span class="k">public</span> <span class="n">B</span><span class="p">(...);</span>   <span class="c1">//C derived from B</span>
</pre></div>
<p>This process can be extended to any number of levels.</p>
<dl>
<dt>Hierarchical inheritance</dt>
<dd>where one class serves as a superclass (base class) for more than one sub class.</dd>
<dt>Hybrid inheritance</dt>
<dd>a mix of two or more of the above types of inheritance.</dd>
</dl>
<h2><span class="mw-headline" id="Subclasses_and_superclasses">Subclasses and superclasses</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Inheritance_(object-oriented_programming)&amp;action=edit&amp;section=2" title="Edit section: Subclasses and superclasses">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>A <i>Subclass</i>, "derived class", <i>heir class</i>, or <i>child class</i> is a <a href="/wiki/Modular_programming" title="Modular programming">modular</a>, derivative class that inherits one or more <a href="/wiki/Programming_language" title="Programming language">language</a> entities from one or more other classes (called <i>superclasses</i>, <i>base classes</i>, or <i>parent classes</i>). The semantics of class inheritance vary from language to language, but commonly the subclass automatically inherits the <a href="/wiki/Instance_variable" title="Instance variable">instance variables</a> and <a class="mw-redirect" href="/wiki/Member_function" title="Member function">member functions</a> of its superclasses. The general form of defining a derived class is:</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="k">class</span> <span class="nc">derived</span><span class="o">-</span><span class="n">class</span><span class="o">-</span><span class="nl">name</span> <span class="p">:</span> <span class="n">visibility</span><span class="o">-</span><span class="n">mode</span> <span class="n">base</span><span class="o">-</span><span class="n">class</span><span class="o">-</span><span class="n">name</span>
<span class="p">{</span>
 <span class="p">.....</span><span class="c1">//</span>
 <span class="p">.....</span><span class="c1">// members of derived class</span>
 <span class="p">.....</span><span class="c1">//</span>
<span class="p">};</span>
</pre></div>
<p><sup class="reference" id="cite_ref-9"><a href="#cite_note-9">[8]</a></sup></p>
<ul>
<li>The colon indicates that the derived-class-name is derived from the base-class-name. The visibility-mode is optional and, if present, may be either <i>private</i> or <i>public</i>. The default visibility-mode is <i>private</i>. Visibility mode specifies whether the features of the base class are <i>privately derived</i> or <i>publicly derived</i>.</li>
</ul>
<p>Some languages support the <a class="mw-redirect" href="/wiki/Inheritance_(computer_science)" title="Inheritance (computer science)">inheritance</a> of other constructs also. For example, in <a href="/wiki/Eiffel_(programming_language)" title="Eiffel (programming language)">Eiffel</a>, <a class="mw-redirect" href="/wiki/Design_by_Contract" title="Design by Contract">contracts</a> that define the specification of a class are also inherited by heirs. The superclass establishes a common interface and foundational functionality, which specialized subclasses can inherit, modify, and supplement. The software inherited by a subclass is considered <a class="mw-redirect" href="/wiki/Reusable" title="Reusable">reused</a> in the subclass. A reference to an instance of a class may actually be referring to one of its subclasses. The actual class of the object being referenced is impossible to predict at <a class="mw-redirect" href="/wiki/Compile-time" title="Compile-time">compile-time</a>. A uniform interface is used to invoke the member functions of objects of a number of different classes. Subclass may replace superclass functions with entirely new functions that must share the same <a class="mw-redirect" href="/wiki/Method_signature" title="Method signature">method signature</a>.</p>
<h3><span class="mw-headline" id="Uninheritable_classes">Uninheritable classes</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Inheritance_(object-oriented_programming)&amp;action=edit&amp;section=3" title="Edit section: Uninheritable classes">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>In some languages a class may be declared as <a href="/wiki/Class_(computer_programming)#Non-subclassable" title="Class (computer programming)">uninheritable</a> by adding certain <a class="new" href="/w/index.php?title=Class_modifier&amp;action=edit&amp;redlink=1" title="Class modifier (page does not exist)">class modifiers</a> to the class declaration. Examples include the "final" keyword in <a href="/wiki/Final_(Java)" title="Final (Java)">Java</a> or the "sealed" keyword in <a href="/wiki/C_Sharp_(programming_language)" title="C Sharp (programming language)">C#</a>. Such modifiers are added to the class declaration before the "class" keyword and the class identifier declaration. Such sealed classes restrict <a href="/wiki/Reusability" title="Reusability">reusability</a>, particularly when developers only have access to precompiled <a href="/wiki/Binary_file" title="Binary file">binaries</a> and not <a href="/wiki/Source_code" title="Source code">source code</a>.</p>
<p>The sealed class has no subclasses, so it can be easily deduced at <a href="/wiki/Compile_time" title="Compile time">compile time</a> that references or pointers to objects of that class are actually referencing instances of that class and not instances of subclasses (they don't exist) or instances of superclasses (<a class="mw-redirect" href="/wiki/Upcasting" title="Upcasting">upcasting</a> a reference type violates the type system). <a class="mw-redirect" href="/wiki/Subtype_polymorphism" title="Subtype polymorphism">subtype polymorphism</a>. Because the exact type of the object being referenced is known before execution, <a class="mw-redirect" href="/wiki/Early_binding" title="Early binding">early binding</a> (or "<a href="/wiki/Static_dispatch" title="Static dispatch">static dispatch</a>") can be used instead of <a href="/wiki/Late_binding" title="Late binding">late binding</a> (also called "<a href="/wiki/Dynamic_dispatch" title="Dynamic dispatch">dynamic dispatch</a>" or "<a href="/wiki/Late_binding" title="Late binding">dynamic binding</a>"), which requires one or more <a href="/wiki/Virtual_method_table" title="Virtual method table">virtual method table</a> lookups depending on whether <a href="/wiki/Multiple_inheritance" title="Multiple inheritance">multiple inheritance</a> or only <a class="mw-redirect" href="/wiki/Single_inheritance" title="Single inheritance">single inheritance</a> are supported in the programming language that is being used.</p>
<h3><span class="mw-headline" id="Methods_that_cannot_be_overridden">Methods that cannot be overridden</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Inheritance_(object-oriented_programming)&amp;action=edit&amp;section=4" title="Edit section: Methods that cannot be overridden">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Just as classes may be sealed/finalized method declarations may contain method modifiers that prevent the method from being overridden (i.e. replaced with a new function with the same name and type signature in a subclass). A <a class="mw-redirect" href="/wiki/Encapsulation_(object-oriented_programming)#As_information_hiding_mechanism" title="Encapsulation (object-oriented programming)">private</a> method is unoverridable simply because it is not accessible by classes other than the class it is a member function of (this is not true for <a href="/wiki/C%2B%2B" title="C++">C++</a>, though). A <a href="/wiki/Final_(Java)#Final_methods" title="Final (Java)">"final" method</a> in <a href="/wiki/Java_(programming_language)" title="Java (programming language)">Java</a>, a "sealed" method in <a href="/wiki/C_Sharp_(programming_language)" title="C Sharp (programming language)">C#</a> or a frozen feature in <a href="/wiki/Eiffel_(programming_language)" title="Eiffel (programming language)">Eiffel</a> cannot be overridden.</p>
<h3><span class="mw-headline" id="Virtual_methods">Virtual methods</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Inheritance_(object-oriented_programming)&amp;action=edit&amp;section=5" title="Edit section: Virtual methods">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>If the superclass method is a <a class="mw-redirect" href="/wiki/Virtual_method" title="Virtual method">virtual method</a>, then invocations of the superclass method will be <a href="/wiki/Dynamic_dispatch" title="Dynamic dispatch">dynamically dispatched</a>. Some languages require methods to be specifically declared as virtual (e.g. <a href="/wiki/C%2B%2B" title="C++">C++</a>) and in others all methods are virtual (e.g. <a href="/wiki/Java_(programming_language)" title="Java (programming language)">Java</a>). An invocation of a non-virtual method will always be statically dispatched (i.e. the address of the function call is determined at compile-time). Static dispatch is faster than dynamic dispatch and allows optimisations such as <a href="/wiki/Inline_expansion" title="Inline expansion">inline expansion</a>.</p>

<h2><span class="mw-headline" id="Applications">Applications</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Inheritance_(object-oriented_programming)&amp;action=edit&amp;section=7" title="Edit section: Applications">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Inheritance is used to co-relate two or more classes to each other.</p>
<h3><span class="mw-headline" id="Overriding">Overriding</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Inheritance_(object-oriented_programming)&amp;action=edit&amp;section=8" title="Edit section: Overriding">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="thumb tright">
<div class="thumbinner" style="width:222px;"><a class="image" href="/wiki/File:Method_overriding_in_subclass.png"><img alt="" class="thumbimage" data-file-height="784" data-file-width="564" height="306" src="//upload.wikimedia.org/wikipedia/commons/thumb/a/a2/Method_overriding_in_subclass.png/220px-Method_overriding_in_subclass.png" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/a/a2/Method_overriding_in_subclass.png/330px-Method_overriding_in_subclass.png 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/a/a2/Method_overriding_in_subclass.png/440px-Method_overriding_in_subclass.png 2x" width="220"/></a>

<div class="hatnote" role="note">Main article: <a href="/wiki/Method_overriding" title="Method overriding">Method overriding</a></div>
<p>Many <a href="/wiki/List_of_object-oriented_programming_languages" title="List of object-oriented programming languages">object-oriented programming languages</a> permit a class or object to replace the implementation of an aspect—typically a behavior—that it has inherited. This process is usually called <i><a href="/wiki/Method_overriding" title="Method overriding">overriding</a></i>. Overriding introduces a complication: which version of the behavior does an instance of the inherited class use—the one that is part of its own class, or the one from the parent (base) class? The answer varies between programming languages, and some languages provide the ability to indicate that a particular behavior is not to be overridden and should behave as defined by the base class. For instance, in <a href="/wiki/C_Sharp_(programming_language)" title="C Sharp (programming language)">C#</a>, the base method or property can only be overridden in a subclass if it is marked with the virtual, abstract, or override modifier.<sup class="reference" id="cite_ref-11"><a href="#cite_note-11">[10]</a></sup> An alternative to overriding is <a class="mw-redirect" href="/wiki/Hiding_(programming)" title="Hiding (programming)">hiding</a> the inherited code.</p>
<h3><span class="mw-headline" id="Code_reuse">Code reuse</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Inheritance_(object-oriented_programming)&amp;action=edit&amp;section=9" title="Edit section: Code reuse">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Implementation inheritance is the mechanism whereby a subclass <a href="/wiki/Code_reuse" title="Code reuse">re-uses</a> code in a base class. By default the subclass retains all of the operations of the base class, but the subclass may <a href="/wiki/Method_overriding" title="Method overriding">override</a> some or all operations, replacing the base-class implementation with its own.</p>
<p>In the following Python example, subclasses <span style="font-family: monospace, monospace;">SquareSumComputer</span> and <span style="font-family: monospace, monospace;">CubeSumComputer</span> override the <span style="font-family: monospace, monospace;">transform()</span> method of the base class <span style="font-family: monospace, monospace;">SumComputer</span>. The base class comprises operations to compute the sum of the <a href="/wiki/Square_number" title="Square number">squares</a> between two integers. The subclass re-uses all of the functionality of the base class with the exception of the operation that transforms a number into its square, replacing it with an operation that transforms a number into its <a href="/wiki/Square_number" title="Square number">square</a> and <a class="mw-redirect" href="/wiki/Cubic_number" title="Cubic number">cube</a> respectively. The subclasses therefore compute the sum of the squares/cubes between two integers.</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>

    <span class="k">def</span> <span class="nf">inputs</span><span class="p">(</span><span class="bp">self</span><span class="p">):</span>

    <span class="k">def</span> <span class="nf">transform</span><span class="p">(</span><span class="bp">self</span><span class="p">,</span> <span class="n">x</span><span class="p">):</span>

<p>In most quarters, class inheritance for the sole purpose of code reuse has fallen out of favor.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (November 2009)">citation needed</span></a></i>]</sup> The primary concern is that implementation inheritance does not provide any assurance of <a class="mw-redirect" href="/wiki/Polymorphism_in_object-oriented_programming" title="Polymorphism in object-oriented programming">polymorphic</a> substitutability—an instance of the reusing class cannot necessarily be substituted for an instance of the inherited class. An alternative technique, explicit <a href="/wiki/Delegation_pattern" title="Delegation pattern">delegation</a>, requires more programming effort, but avoids the substitutability issue.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (May 2012)">citation needed</span></a></i>]</sup> In <a href="/wiki/C%2B%2B" title="C++">C++</a> private inheritance can be used as a form of <i>implementation inheritance</i> without substitutability. Whereas public inheritance represents an "<a href="/wiki/Is-a" title="Is-a">is-a</a>" relationship and delegation represents a "<a href="/wiki/Has-a" title="Has-a">has-a</a>" relationship, private (and protected) inheritance can be thought of as an "is implemented in terms of" relationship.<sup class="reference" id="cite_ref-12"><a href="#cite_note-12">[11]</a></sup></p>
<p>Another frequent use of inheritance is to guarantee that classes maintain a certain common interface; that is, they implement the same methods. The parent class can be a combination of implemented operations and operations that are to be implemented in the child classes. Often, there is no interface change between the supertype and subtype- the child implements the behavior described instead of its parent class.<sup class="reference" id="cite_ref-13"><a href="#cite_note-13">[12]</a></sup></p>
<h2><span class="mw-headline" id="Inheritance_vs_subtyping">Inheritance vs subtyping</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Inheritance_(object-oriented_programming)&amp;action=edit&amp;section=10" title="Edit section: Inheritance vs subtyping">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<div class="hatnote" role="note">Further information: <a href="/wiki/Subtyping" title="Subtyping">Subtyping</a></div>
<p>Inheritance is similar to but distinct from <a href="/wiki/Subtyping" title="Subtyping">subtyping</a>.<sup class="reference" id="cite_ref-FOOTNOTECookHillCanning1990_14-0"><a href="#cite_note-FOOTNOTECookHillCanning1990-14">[13]</a></sup> Subtyping enables a given type to be substituted for another type or abstraction, and is said to establish an <i><a href="/wiki/Is-a" title="Is-a">is-a</a></i> relationship between the subtype and some existing abstraction, either implicitly or explicitly, depending on language support. The relationship can be expressed explicitly via inheritance in languages that support inheritance as a subtyping mechanism. For example, the following C++ code establishes an explicit inheritance relationship between classes <i>B</i> and <i>A</i>, where <i>B</i> is both a subclass and a subtype of <i>A</i>, and can be used as an <i>A</i> wherever a <i>B</i> is specified (via a reference, a pointer or the object itself).</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="k">class</span> <span class="nc">A</span> 
<span class="p">{</span> <span class="k">public</span><span class="o">:</span>
   <span class="kt">void</span> <span class="n">DoSomethingALike</span><span class="p">()</span> <span class="k">const</span> <span class="p">{}</span>
<span class="p">};</span>

<span class="k">class</span> <span class="nc">B</span> <span class="o">:</span> <span class="k">public</span> <span class="n">A</span> 
<span class="p">{</span> <span class="k">public</span><span class="o">:</span>
   <span class="kt">void</span> <span class="n">DoSomethingBLike</span><span class="p">()</span> <span class="k">const</span> <span class="p">{}</span>
<span class="p">};</span>

<span class="kt">void</span> <span class="nf">UseAnA</span><span class="p">(</span><span class="n">A</span> <span class="k">const</span><span class="o">&amp;</span> <span class="n">some_A</span><span class="p">)</span>
<span class="p">{</span>
   <span class="n">some_A</span><span class="p">.</span><span class="n">DoSomethingALike</span><span class="p">();</span>
<span class="p">}</span>

<span class="kt">void</span> <span class="nf">SomeFunc</span><span class="p">()</span>
<span class="p">{</span>
   <span class="n">B</span> <span class="n">b</span><span class="p">;</span>
   <span class="n">UseAnA</span><span class="p">(</span><span class="n">b</span><span class="p">);</span> <span class="c1">// b can be substituted for an A.</span>
<span class="p">}</span>
</pre></div>
<p>In programming languages that do not support inheritance as a <a class="mw-redirect" href="/wiki/Subtype_polymorphism" title="Subtype polymorphism">subtyping mechanism</a>, the relationship between a base class and a derived class is only a relationship between implementations (a mechanism for code reuse), as compared to a relationship between <a href="/wiki/Data_type" title="Data type">types</a>. Inheritance, even in programming languages that support inheritance as a subtyping mechanism, does not necessarily entail <a class="mw-redirect" href="/wiki/Behavioral_subtyping" title="Behavioral subtyping">behavioral subtyping</a>. It is entirely possible to derive a class whose object will behave incorrectly when used in a context where the parent class is expected; see the <a href="/wiki/Liskov_substitution_principle" title="Liskov substitution principle">Liskov substitution principle</a>. <sup class="reference" id="cite_ref-Mitchell2002_15-0"><a href="#cite_note-Mitchell2002-15">[14]</a></sup> (Compare <a href="/wiki/Connotation_(semiotics)" title="Connotation (semiotics)">connotation/denotation</a>.) In some OOP languages, the notions of code reuse and subtyping coincide because the only way to declare a subtype is to define a new class that inherits the implementation of another.</p>
<h3><span class="mw-headline" id="Design_constraints">Design constraints</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Inheritance_(object-oriented_programming)&amp;action=edit&amp;section=11" title="Edit section: Design constraints">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Using inheritance extensively in designing a program imposes certain constraints.</p>
<p>For example, consider a class <span style="font-family: monospace, monospace;">Person</span> that contains a person's name, date of birth, address and phone number. We can define a subclass of <span style="font-family: monospace, monospace;">Person</span> called <span style="font-family: monospace, monospace;">Student</span> that contains the person's grade point average and classes taken, and another subclass of <span style="font-family: monospace, monospace;">Person</span> called <span style="font-family: monospace, monospace;">Employee</span> that contains the person's job-title, employer, and salary.</p>
<p>In defining this inheritance hierarchy we have already defined certain restrictions, not all of which are desirable:</p>
<dl>
<dt>Singleness</dt>
<dd>Using single inheritance, a subclass can inherit from only one superclass. Continuing the example given above, <span style="font-family: monospace, monospace;">Person</span> can be either a <span style="font-family: monospace, monospace;">Student</span> or an <span style="font-family: monospace, monospace;">Employee</span>, but not both. Using <a href="/wiki/Multiple_inheritance" title="Multiple inheritance">multiple inheritance</a> partially solves this problem, as one can then define a <span style="font-family: monospace, monospace;">StudentEmployee</span> class that inherits from both <span style="font-family: monospace, monospace;">Student</span> and <span style="font-family: monospace, monospace;">Employee</span>. However, in most implementations, it can still inherit from each superclass only once, and thus, does not support cases in which a student has two jobs or attends two institutions. The inheritance model available in <a href="/wiki/Eiffel_(programming_language)" title="Eiffel (programming language)">Eiffel</a> makes this possible through support for <a class="mw-redirect" href="/wiki/Diamond_problem" title="Diamond problem">repeated inheritance</a>.</dd>
</dl>
<dl>
<dt>Static</dt>
<dd>The inheritance hierarchy of an object is fixed at <a href="/wiki/Instance_(computer_science)" title="Instance (computer science)">instantiation</a> when the object's type is selected and does not change with time. For example, the inheritance graph does not allow a <span style="font-family: monospace, monospace;">Student</span> object to become a <span style="font-family: monospace, monospace;">Employee</span> object while retaining the state of its <span style="font-family: monospace, monospace;">Person</span> superclass. (This kind of behavior, however, can be achieved with the <a href="/wiki/Decorator_pattern" title="Decorator pattern">decorator pattern</a>.) Some have criticized inheritance, contending that it locks developers into their original design standards.<sup class="reference" id="cite_ref-holub_16-0"><a href="#cite_note-holub-16">[15]</a></sup></dd>
</dl>
<dl>
<dt>Visibility</dt>
<dd>Whenever client code has access to an object, it generally has access to all the object's superclass data. Even if the superclass has not been declared public, the client can still <a class="mw-redirect" href="/wiki/Typecast_(programming)" title="Typecast (programming)">cast</a> the object to its superclass type. For example, there is no way to give a function a pointer to a <span style="font-family: monospace, monospace;">Student</span>'s grade point average and transcript without also giving that function access to all of the personal data stored in the student's <span style="font-family: monospace, monospace;">Person</span> superclass. Many modern languages, including C++ and Java, provide a "protected" access modifier that allows subclasses to access the data, without allowing any code outside the chain of inheritance to access it.</dd>
</dl>
<p>The <a class="mw-redirect" href="/wiki/Composite_reuse_principle" title="Composite reuse principle">composite reuse principle</a> is an alternative to inheritance. This technique supports polymorphism and code reuse by separating behaviors from the primary class hierarchy and including specific behavior classes as required in any business domain class. This approach avoids the static nature of a class hierarchy by allowing behavior modifications at run time and allows one class to implement behaviors buffet-style, instead of being restricted to the behaviors of its ancestor classes.</p>
<h2><span class="mw-headline" id="Issues_and_alternatives">Issues and alternatives</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Inheritance_(object-oriented_programming)&amp;action=edit&amp;section=12" title="Edit section: Issues and alternatives">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Implementation inheritance is controversial among programmers and theoreticians of object-oriented programming since at least the 1990s. Among them are the authors of <i><a href="/wiki/Design_Patterns" title="Design Patterns">Design Patterns</a></i>, who advocate interface inheritance instead, and favor <a href="/wiki/Composition_over_inheritance" title="Composition over inheritance">composition over inheritance</a>. For example, the <a href="/wiki/Decorator_pattern" title="Decorator pattern">decorator pattern</a> (as mentioned <a href="#Design_constraints">above</a>) has been proposed to overcome the static nature of inheritance between classes. As a more fundamental solution to the same problem, <a href="/wiki/Role-oriented_programming" title="Role-oriented programming">role-oriented programming</a> introduces a distinct relationship, <i>played-by</i>, combining properties of inheritance and composition into a new concept.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (February 2016)">citation needed</span></a></i>]</sup></p>
<p>According to <a href="/wiki/Allen_Holub" title="Allen Holub">Allen Holub</a>, the main problem with implementation inheritance is that it introduces unnecessary <a href="/wiki/Coupling_(computer_programming)" title="Coupling (computer programming)">coupling</a> in the form of the "fragile base class problem":<sup class="reference" id="cite_ref-Mikhajlov_6-1"><a href="#cite_note-Mikhajlov-6">[5]</a></sup> modifications to the base class implementation can cause inadvertent behavioral changes in subclasses. Using interfaces avoids this problem because no implementation is shared, only the API.<sup class="reference" id="cite_ref-holub_16-1"><a href="#cite_note-holub-16">[15]</a></sup> Another way of stating this is that "inheritance breaks <a class="mw-redirect" href="/wiki/Encapsulation_(object-oriented_programming)" title="Encapsulation (object-oriented programming)">encapsulation</a>".<sup class="reference" id="cite_ref-17"><a href="#cite_note-17">[16]</a></sup> The problem surfaces clearly in open object-oriented systems such as <a href="/wiki/Software_framework" title="Software framework">frameworks</a>, where client code is expected to inherit from system-supplied classes and then substituted for the system's classes in its algorithms.<sup class="reference" id="cite_ref-Mikhajlov_6-2"><a href="#cite_note-Mikhajlov-6">[5]</a></sup></p>
<p>Reportedly, Java inventor <a href="/wiki/James_Gosling" title="James Gosling">James Gosling</a> has spoken against implementation inheritance, stating that he would not include it if he were to redesign Java.<sup class="reference" id="cite_ref-holub_16-2"><a href="#cite_note-holub-16">[15]</a></sup> Language designs that decouple inheritance from subtyping (interface inheritance) appeared as early as 1990;<sup class="reference" id="cite_ref-18"><a href="#cite_note-18">[17]</a></sup> a modern example of this is the <a href="/wiki/Go_(programming_language)" title="Go (programming language)">Go</a> programming language.</p>
<p>Complex inheritance, or inheritance used within an insufficiently mature design, may lead to the <a href="/wiki/Yo-yo_problem" title="Yo-yo problem">yo-yo problem</a>.</p>
<p>Another issue with inheritance is that subclasses must be defined in code, which means that program users cannot add new subclasses. Other design patterns<sup class="noprint Inline-Template" style="white-space:nowrap;">[<i><a class="mw-redirect" href="/wiki/Wikipedia:Avoid_weasel_words" title="Wikipedia:Avoid weasel words"><span title="The material near this tag possibly uses too vague attribution or weasel words. (July 2015)">which?</span></a></i>]</sup> allow program users to define variations of an entity on runtime.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (July 2015)">citation needed</span></a></i>]</sup></p>
