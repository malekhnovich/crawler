<h1 class="firstHeading" id="firstHeading" lang="en">Data access object</h1>
<div class="mw-body-content" id="bodyContent">
<div id="siteSub">From Wikipedia, the free encyclopedia</div>
<div id="contentSub"></div>
<div class="mw-jump" id="jump-to-nav">
					Jump to:					<a href="#mw-head">navigation</a>, 					<a href="#p-search">search</a>
</div>
<div class="mw-content-ltr" dir="ltr" id="mw-content-text" lang="en"><div class="hatnote" role="note">This article is about the software design pattern. For the Microsoft library, see <a href="/wiki/Jet_Data_Access_Objects" title="Jet Data Access Objects">Jet Data Access Objects</a>.</div>
<p>In <a class="mw-redirect" href="/wiki/Computer_software" title="Computer software">computer software</a>, a <b>data access object</b> (<b>DAO</b>) is an <a href="/wiki/Object_(computer_science)" title="Object (computer science)">object</a> that provides an abstract <a class="mw-redirect" href="/wiki/Interface_(computer_science)" title="Interface (computer science)">interface</a> to some type of <a href="/wiki/Database" title="Database">database</a> or other persistence mechanism. By mapping application calls to the persistence layer, DAO provide some specific data operations without exposing details of the database. This isolation supports the <a href="/wiki/Single_responsibility_principle" title="Single responsibility principle">Single responsibility principle</a>. It separates what data accesses the application needs, in terms of domain-specific objects and data types (the public interface of the DAO), from how these needs can be satisfied with a specific <a class="mw-redirect" href="/wiki/Database_Management_System" title="Database Management System">DBMS</a>, database schema, etc. (the implementation of the DAO).</p>
<p>Although this <a href="/wiki/Design_pattern" title="Design pattern">design pattern</a> is equally applicable to the following: (1- most programming languages; 2- most types of software with persistence needs; and 3- most types of databases) it is traditionally associated with <a href="/wiki/Java_Platform,_Enterprise_Edition" title="Java Platform, Enterprise Edition">Java EE</a> applications and with relational databases (accessed via the JDBC API because of its origin in Sun Microsystems' best practice guidelines<sup class="reference" id="cite_ref-1"><a href="#cite_note-1">[1]</a></sup> "Core J2EE Patterns" for that platform).</p>

<h2><span class="mw-headline" id="Advantages">Advantages</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Data_access_object&amp;action=edit&amp;section=1" title="Edit section: Advantages">edit</a><span class="mw-editsection-bracket">]</span></span></h2>

<p>The advantage of using data access objects is the relatively simple and rigorous separation between two important parts of an application that can but should not know anything of each other, and which can be expected to evolve frequently and independently. Changing business logic can rely on the same DAO interface, while changes to persistence logic do not affect DAO clients as long as the interface remains correctly implemented. All details of storage are hidden from the rest of the application (see <a href="/wiki/Information_hiding" title="Information hiding">Information hiding</a>). Thus, possible changes to the persistence mechanism can be implemented by just modifying one DAO implementation while the rest of the application isn't affected. DAOs act as an intermediary between the application and the database. They move data back and forth between objects and database records. <a href="/wiki/Unit_testing" title="Unit testing">Unit testing</a> the code is facilitated by substituting the DAO with a <a href="/wiki/Test_double" title="Test double">test double</a> in the test, thereby making the tests non-dependent on the persistence layer.</p>
<p>In the non specific context of the <a href="/wiki/Java_(programming_language)" title="Java (programming language)">Java</a> programming language, Data Access Objects as a design concept can be implemented in a number of ways. This can range from a fairly simple interface that separates the data access parts from the application logic, to frameworks and commercial products. DAO coding paradigms can require some skill. Technologies like <a href="/wiki/Java_Persistence_API" title="Java Persistence API">Java Persistence API</a> and <a href="/wiki/Enterprise_JavaBeans" title="Enterprise JavaBeans">Enterprise JavaBeans</a> come built into application servers and can be used in applications that use a JavaEE application server. Commercial products like <a href="/wiki/TopLink" title="TopLink">TopLink</a> are available based on <a href="/wiki/Object-relational_mapping" title="Object-relational mapping">Object-relational mapping</a> (ORM). Popular open source ORM products include <a href="/wiki/Doctrine_(PHP)" title="Doctrine (PHP)">Doctrine</a>, <a class="mw-redirect" href="/wiki/Hibernate_(Java)" title="Hibernate (Java)">Hibernate</a>, <a href="/wiki/IBATIS" title="IBATIS">iBATIS</a> and JPA implementations such as <a href="/wiki/Apache_OpenJPA" title="Apache OpenJPA">Apache OpenJPA</a>.</p>
<h2><span class="mw-headline" id="Disadvantages">Disadvantages</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Data_access_object&amp;action=edit&amp;section=2" title="Edit section: Disadvantages">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Potential disadvantages of using DAO include <a href="/wiki/Leaky_abstraction" title="Leaky abstraction">leaky abstraction</a>,<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="What kind of leak? (August 2014)">citation needed</span></a></i>]</sup> code duplication,<sup class="reference" id="cite_ref-2"><a href="#cite_note-2">[2]</a></sup> and <a href="/wiki/Abstraction_inversion" title="Abstraction inversion">abstraction inversion</a>. In particular, the abstraction of the DAO as a regular Java object can hide the high cost of each database access, and can also force developers to trigger multiple database queries to retrieve information that could otherwise be returned in a single operation with normal SQL set operations. Time consumption is moderate.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (December 2014)">citation needed</span></a></i>]</sup></p>
