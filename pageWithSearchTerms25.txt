<p><b>Component-based software engineering</b> (<b>CBSE</b>), also known as <b>component-based development</b> (<b>CBD</b>), is a branch of software engineering that emphasizes the <a href="/wiki/Separation_of_concerns" title="Separation of concerns">separation of concerns</a> in respect of the wide-ranging functionality available throughout a given <a href="/wiki/Software_system" title="Software system">software system</a>. It is a reuse-based approach to defining, implementing and composing loosely coupled independent components into systems. This practice aims to bring about an equally wide-ranging degree of benefits in both the short-term and the long-term for the software itself and for organizations that sponsor such software.</p>
<p>Software engineering practitioners regard components as part of the starting platform for <a href="/wiki/Service-orientation" title="Service-orientation">service-orientation</a>. Components play this role, for example, in <a href="/wiki/Web_service" title="Web service">web services</a>, and more recently, in <a href="/wiki/Service-oriented_architecture" title="Service-oriented architecture">service-oriented architectures</a> (SOA), whereby a component is converted by the web service into a <i>service</i> and subsequently inherits further characteristics beyond that of an ordinary component.</p>
<p>Components can produce or consume events and can be used for <a href="/wiki/Event-driven_architecture" title="Event-driven architecture">event-driven architectures</a> (EDA).</p>

<h2><span class="mw-headline" id="Definition_and_characteristics_of_components"><span id="Software_component"></span> Definition and characteristics of components</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Component-based_software_engineering&amp;action=edit&amp;section=1" title="Edit section: Definition and characteristics of components">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>An individual <b>software component</b> is a <a class="mw-redirect" href="/wiki/Package_(package_management_system)" title="Package (package management system)">software package</a>, a <a href="/wiki/Web_service" title="Web service">web service</a>, a <a href="/wiki/Web_resource" title="Web resource">web resource</a>, or a <a href="/wiki/Modular_programming" title="Modular programming">module</a> that encapsulates a set of related <a class="mw-redirect" href="/wiki/Function_(computer_science)" title="Function (computer science)">functions</a> (or data).</p>
<p>All system processes are placed into separate components so that all of the data and functions inside each component are semantically related (just as with the contents of classes). Because of this principle, it is often said that components are <i>modular</i> and <i>cohesive</i>.</p>
<p>With regard to system-wide co-ordination, components communicate with each other via <i>interfaces</i>. When a component offers services to the rest of the system, it adopts a <i>provided</i> interface that specifies the services that other components can utilize, and how they can do so. This interface can be seen as a signature of the component - the client does not need to know about the inner workings of the component (implementation) in order to make use of it. This principle results in components referred to as <i>encapsulated</i>. The UML illustrations within this article represent provided interfaces by a lollipop-symbol attached to the outer edge of the component.</p>
<p>However, when a component needs to use another component in order to function, it adopts a <i>used</i> interface that specifies the services that it needs. In the UML illustrations in this article, <i>used interfaces</i> are represented by an open socket symbol attached to the outer edge of the component.</p>

<p>Another important attribute of components is that they are <i>substitutable</i>, so that a component can replace another (at design time or run-time), if the successor component meets the requirements of the initial component (expressed via the interfaces). Consequently, components can be replaced with either an updated version or an alternative without breaking the system in which the component operates.</p>
<p>As a <a href="/wiki/Liskov_substitution_principle" title="Liskov substitution principle">rule of thumb</a> for engineers substituting components, component B can immediately replace component A, if component B provides at least what component A provided and uses no more than what component A used.</p>
<p>Software components often take the form of <a class="mw-redirect" href="/wiki/Object_(computing)" title="Object (computing)">objects</a> (not <a class="mw-redirect" href="/wiki/Class_(computer_science)" title="Class (computer science)">classes</a>) or collections of objects (from <a href="/wiki/Object-oriented_programming" title="Object-oriented programming">object-oriented programming</a>), in some binary or textual form, adhering to some <a href="/wiki/Interface_description_language" title="Interface description language">interface description language</a> (IDL) so that the component may exist autonomously from other components in a <a href="/wiki/Computer" title="Computer">computer</a>. In other words, a component acts without changing its source code. Although, the behavior of the component's source code may change based on the application's extensibility, provided by its writer.</p>
<p>When a component is to be accessed or shared across execution contexts or network links, techniques such as <a href="/wiki/Serialization" title="Serialization">serialization</a> or <a href="/wiki/Marshalling_(computer_science)" title="Marshalling (computer science)">marshalling</a> are often employed to deliver the component to its destination.</p>
<p><a href="/wiki/Reusability" title="Reusability">Reusability</a> is an important characteristic of a high-quality software component. Programmers should design and implement software components in such a way that many different programs can reuse them. Furthermore, <a class="mw-redirect" href="/wiki/Component-Based_Usability_Testing" title="Component-Based Usability Testing">component-based usability testing</a> should be considered when software components directly interact with users.</p>
<p>It takes significant effort and awareness to write a software component that is effectively reusable. The component needs to be:</p>
<ul>
<li>fully documented</li>
<li>thoroughly tested
<ul>
<li>robust - with comprehensive input-validity checking</li>
<li>able to pass back appropriate <a href="/wiki/Error_message" title="Error message">error messages</a> or return codes</li>
</ul>
</li>
<li>designed with an awareness that it <i>will</i> be put to unforeseen uses</li>
</ul>
<p>In the 1960s, programmers built scientific <a href="/wiki/Subroutine" title="Subroutine">subroutine</a> libraries that were reusable in a broad array of engineering and scientific applications. Though these subroutine libraries reused well-defined <a class="mw-redirect" href="/wiki/Algorithms" title="Algorithms">algorithms</a> in an effective manner, they had a limited domain of application. Commercial sites routinely created application programs from reusable modules written in <a href="/wiki/Assembly_language" title="Assembly language">Assembler</a>, <a href="/wiki/COBOL" title="COBOL">COBOL</a>, <a class="mw-redirect" href="/wiki/PL/1" title="PL/1">PL/1</a> and other <a class="mw-redirect" href="/wiki/Second_generation_language" title="Second generation language">second-</a> and <a class="mw-redirect" href="/wiki/Third_generation_language" title="Third generation language">third-generation languages</a> using both <a href="/wiki/Operating_system" title="Operating system">system</a> and user application libraries.</p>
<p>As of 2010<sup class="plainlinks noprint asof-tag update" style="display:none;"><a class="external text" href="//en.wikipedia.org/w/index.php?title=Component-based_software_engineering&amp;action=edit">[update]</a></sup>, modern reusable components encapsulate both data structures and the algorithms that are applied to the data structures. It<sup class="noprint Inline-Template" style="margin-left:0.1em; white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Please_clarify" title="Wikipedia:Please clarify"><span title="The text near this tag may need clarification or removal of jargon. (February 2010)">clarification needed</span></a></i>]</sup> builds on prior theories of <a class="mw-redirect" href="/wiki/Object_(object-oriented_programming)" title="Object (object-oriented programming)">software objects</a>, <a href="/wiki/Software_architecture" title="Software architecture">software architectures</a>, <a href="/wiki/Software_framework" title="Software framework">software frameworks</a> and <a href="/wiki/Software_design_pattern" title="Software design pattern">software design patterns</a>, and the extensive theory of <a href="/wiki/Object-oriented_programming" title="Object-oriented programming">object-oriented programming</a> and the <a class="mw-redirect" href="/wiki/Object_oriented_design" title="Object oriented design">object oriented design</a> of all these. It claims that software components, like the idea of hardware <a href="/wiki/Electronic_component" title="Electronic component">components</a>, used for example in telecommunications,<sup class="reference" id="cite_ref-1"><a href="#cite_note-1">[1]</a></sup> can ultimately be made interchangeable and reliable. On the other hand, it is argued that it is a mistake to focus on independent components rather than the framework (without which they would not exist).<sup class="reference" id="cite_ref-2"><a href="#cite_note-2">[2]</a></sup></p>
<h2><span class="mw-headline" id="History">History</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Component-based_software_engineering&amp;action=edit&amp;section=2" title="Edit section: History">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The idea that <a href="/wiki/Software" title="Software">software</a> should be componentized - built from prefabricated <i>components</i> - first became prominent with <a href="/wiki/Douglas_McIlroy" title="Douglas McIlroy">Douglas McIlroy</a>'s address at the <a href="/wiki/NATO" title="NATO">NATO</a> conference on <a href="/wiki/Software_engineering" title="Software engineering">software engineering</a> in <a href="/wiki/Garmisch-Partenkirchen" title="Garmisch-Partenkirchen">Garmisch</a>, <a href="/wiki/Germany" title="Germany">Germany</a>, 1968, titled <i>Mass Produced Software Components</i>.<sup class="reference" id="cite_ref-3"><a href="#cite_note-3">[3]</a></sup> The conference set out to counter the so-called <a href="/wiki/Software_crisis" title="Software crisis">software crisis</a>. McIlroy's subsequent inclusion of <a href="/wiki/Pipeline_(Unix)" title="Pipeline (Unix)">pipes and filters</a> into the <a href="/wiki/Unix" title="Unix">Unix</a> <a href="/wiki/Operating_system" title="Operating system">operating system</a> was the first implementation of an infrastructure for this idea.</p>
<p><a href="/wiki/Brad_Cox" title="Brad Cox">Brad Cox</a> of <a href="/wiki/Stepstone" title="Stepstone">Stepstone</a> largely defined the modern concept of a software component.<sup class="reference" id="cite_ref-4"><a href="#cite_note-4">[4]</a></sup> He called them <i>Software ICs</i> and set out to create an infrastructure and market for these components by inventing the <a href="/wiki/Objective-C" title="Objective-C">Objective-C</a> programming language. (He summarizes this view in his book <i>Object-Oriented Programming - An Evolutionary Approach</i> 1986.)</p>
<p>The software components are used in two different contexts and two kinds: (i) using components as parts to build a single executable, or (ii) each executable is treated as a component in a distributed environment, where components collaborate with each other using internet or intranet communication protocols for IPC (Inter Process Communications). The above belongs to former kind, while the below belongs to later kind.</p>
<p><a href="/wiki/IBM" title="IBM">IBM</a> led the path with their <a class="mw-redirect" href="/wiki/System_Object_Model" title="System Object Model">System Object Model (SOM)</a> in the early 1990s. As a reaction, <a href="/wiki/Microsoft" title="Microsoft">Microsoft</a> paved the way for actual deployment of component software with <a class="mw-redirect" href="/wiki/Object_linking_and_embedding" title="Object linking and embedding">OLE</a> and <a href="/wiki/Component_Object_Model" title="Component Object Model">COM</a>.<sup class="reference" id="cite_ref-claims_5-0"><a href="#cite_note-claims-5">[5]</a></sup> As of 2010<sup class="plainlinks noprint asof-tag update" style="display:none;"><a class="external text" href="//en.wikipedia.org/w/index.php?title=Component-based_software_engineering&amp;action=edit">[update]</a></sup> many successful software component models exist.</p>
<h2><span class="mw-headline" id="Differences_from_object-oriented_programming">Differences from object-oriented programming</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Component-based_software_engineering&amp;action=edit&amp;section=3" title="Edit section: Differences from object-oriented programming">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Proponents of <a href="/wiki/Object-oriented_programming" title="Object-oriented programming">object-oriented programming</a> (OOP) maintain that software should be written according to a mental model of the actual or imagined objects it represents. OOP and the related disciplines of <a class="mw-redirect" href="/wiki/Object-oriented_analysis" title="Object-oriented analysis">object-oriented analysis</a> and <a href="/wiki/Object-oriented_design" title="Object-oriented design">object-oriented design</a> focus on modeling real-world<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (January 2009)">citation needed</span></a></i>]</sup> interactions and attempting to create "nouns" and "verbs" that can be used in more human-readable ways, ideally by end users as well as by programmers coding for those end users.</p>
<p>Component-based software engineering, by contrast, makes no such assumptions, and instead states that developers should construct software by gluing together prefabricated components - much like in the fields of <a href="/wiki/Electronics" title="Electronics">electronics</a> or <a href="/wiki/Mechanics" title="Mechanics">mechanics</a>. Some peers<sup class="noprint Inline-Template" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Manual_of_Style/Words_to_watch#Unsupported_attributions" title="Wikipedia:Manual of Style/Words to watch"><span title="The material near this tag possibly uses too-vague attribution or weasel words. (January 2009)">who?</span></a></i>]</sup> will even talk of modularizing systems as software components as a new <a href="/wiki/Programming_paradigm" title="Programming paradigm">programming paradigm</a>. Example for possible paradigm: many experts feel adaptable to evolving needs is more important than reuse, since 80% of software engineering deals with maintaining or releasing new versions. So it is desirable to build complex system by assembling highly cohesive loosely coupled large components, where cost of redesigning each of such adoptable components (or replacing by a better component) must be minimized.</p>
<p>Some<sup class="noprint Inline-Template" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Manual_of_Style/Words_to_watch#Unsupported_attributions" title="Wikipedia:Manual of Style/Words to watch"><span title="The material near this tag possibly uses too-vague attribution or weasel words. (October 2009)">who?</span></a></i>]</sup> argue that earlier <a href="/wiki/Computer_scientist" title="Computer scientist">computer scientists</a> made this distinction, with <a href="/wiki/Donald_Knuth" title="Donald Knuth">Donald Knuth</a>'s theory of "<a href="/wiki/Literate_programming" title="Literate programming">literate programming</a>" optimistically assuming there was convergence between intuitive and formal models, and <a class="mw-redirect" href="/wiki/Edsger_Dijkstra" title="Edsger Dijkstra">Edsger Dijkstra</a>'s theory in the article <i><a class="mw-redirect" href="/wiki/The_Cruelty_of_Really_Teaching_Computer_Science" title="The Cruelty of Really Teaching Computer Science">The Cruelty of Really Teaching Computer Science</a></i>, which stated that programming was simply, and only, a branch of <a href="/wiki/Mathematics" title="Mathematics">mathematics</a>.<sup class="reference" id="cite_ref-6"><a href="#cite_note-6">[6]</a></sup><sup class="reference" id="cite_ref-7"><a href="#cite_note-7">[7]</a></sup></p>
<p>In both forms, this notion has led to many academic debates<sup class="noprint Inline-Template" style="margin-left:0.1em; white-space:nowrap;">[<i><a class="mw-redirect" href="/wiki/Wikipedia:Avoid_weasel_words" title="Wikipedia:Avoid weasel words"><span title="Who says this? (October 2009)">weasel words</span></a></i>]</sup> about the pros and cons of the two approaches and possible strategies for uniting the two. Some<sup class="noprint Inline-Template" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Manual_of_Style/Words_to_watch#Unsupported_attributions" title="Wikipedia:Manual of Style/Words to watch"><span title="The material near this tag possibly uses too-vague attribution or weasel words. (October 2009)">who?</span></a></i>]</sup> consider the different strategies not as competitors, but as descriptions of the same problem from different points of view.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (October 2009)">citation needed</span></a></i>]</sup></p>
<p>One approach to creating component-based software using object-oriented programming is <a href="/wiki/Interface-based_programming" title="Interface-based programming">interface-based programming</a>. However, interface-based programming does not inherently support <a class="mw-redirect" href="/wiki/Distributed_system" title="Distributed system">distributed systems</a>, and many computer systems are inherently distributed in the 21st century. Interface-based programming in the OOP sense may be extended to distributed systems with distributed component object models; however, many have argued in recent years that <a href="/wiki/Representational_state_transfer" title="Representational state transfer">REST</a> APIs or the <a href="/wiki/Actor_model" title="Actor model">actor model</a> are more suitable approaches.</p>
<h2><span class="mw-headline" id="Architecture">Architecture</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Component-based_software_engineering&amp;action=edit&amp;section=4" title="Edit section: Architecture">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>A computer running several software components is often called an <a href="/wiki/Application_server" title="Application server">application server</a>. This combination of application servers and software components is usually called <a href="/wiki/Distributed_computing" title="Distributed computing">distributed computing</a>. Typical real-world application of this is in, e.g., financial applications or business software.</p>
<h2><span class="mw-headline" id="Models">Models</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Component-based_software_engineering&amp;action=edit&amp;section=5" title="Edit section: Models">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>A component model is a definition of standards for component implementation, documentation and deployment. Examples of component models are: <a href="/wiki/Enterprise_JavaBeans" title="Enterprise JavaBeans">Enterprise JavaBeans</a> (EJB) model, <a href="/wiki/Component_Object_Model" title="Component Object Model">Component Object Model</a> (COM) model, <a href="/wiki/.NET_Framework" title=".NET Framework">.NET</a> model and <a href="/wiki/Common_Object_Request_Broker_Architecture" title="Common Object Request Broker Architecture">Common Object Request Broker Architecture</a> (CORBA) component Model. The component model specifies how interfaces should be defined and the elements that should be included in an interface definition.<sup class="reference" id="cite_ref-8"><a href="#cite_note-8">[8]</a></sup></p>
