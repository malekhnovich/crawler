<h1 class="firstHeading" id="firstHeading" lang="en">Function model</h1>
<div class="mw-body-content" id="bodyContent">
<div id="siteSub">From Wikipedia, the free encyclopedia</div>
<div id="contentSub"></div>
<div class="mw-jump" id="jump-to-nav">
					Jump to:					<a href="#mw-head">navigation</a>, 					<a href="#p-search">search</a>
</div>
<div class="mw-content-ltr" dir="ltr" id="mw-content-text" lang="en"><p>A <b>function model</b> or <i>functional model</i> in <a href="/wiki/Systems_engineering" title="Systems engineering">systems engineering</a> and <a href="/wiki/Software_engineering" title="Software engineering">software engineering</a> is a structured <a href="/wiki/Representation_(systemics)" title="Representation (systemics)">representation</a> of the <a href="/wiki/Function_(engineering)" title="Function (engineering)">functions</a> (<a href="/wiki/Activity_diagram" title="Activity diagram">activities</a>, <a href="/wiki/Task_analysis" title="Task analysis">actions</a>, <a class="extiw" href="https://en.wiktionary.org/wiki/process" title="wikt:process">processes</a>, <a href="/wiki/Operations_management" title="Operations management">operations</a>) within the modeled <a href="/wiki/System" title="System">system</a> or subject area.<sup class="reference" id="cite_ref-ITL93_1-0"><a href="#cite_note-ITL93-1">[1]</a></sup></p>
<div class="thumb tright">
<div class="thumbinner" style="width:362px;"><a class="image" href="/wiki/File:IDEF_Diagram_Example.jpg"><img alt="" class="thumbimage" data-file-height="600" data-file-width="750" height="288" src="//upload.wikimedia.org/wikipedia/commons/thumb/3/31/IDEF_Diagram_Example.jpg/360px-IDEF_Diagram_Example.jpg" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/3/31/IDEF_Diagram_Example.jpg/540px-IDEF_Diagram_Example.jpg 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/3/31/IDEF_Diagram_Example.jpg/720px-IDEF_Diagram_Example.jpg 2x" width="360"/></a>
<div class="thumbcaption">
<div class="magnify"><a class="internal" href="/wiki/File:IDEF_Diagram_Example.jpg" title="Enlarge"></a></div>
Example of a function model of the process of "Maintain Reparable Spares" in <a href="/wiki/IDEF0" title="IDEF0">IDEF0</a> notation.</div>
</div>
</div>
<p>A function model, similar with the <a href="/wiki/Activity_diagram" title="Activity diagram">activity model</a> or <a class="mw-redirect" href="/wiki/Process_model" title="Process model">process model</a>, is a graphical representation of an <a href="/wiki/Business" title="Business">enterprise</a>'s function within a defined scope. The purposes of the function model are to describe the functions and processes, assist with discovery of information needs, help identify opportunities, and establish a basis for determining product and service costs.<sup class="reference" id="cite_ref-rmsc-1995_2-0"><a href="#cite_note-rmsc-1995-2">[2]</a></sup></p>

<h2><span class="mw-headline" id="History">History</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Function_model&amp;action=edit&amp;section=1" title="Edit section: History">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The function model in the field of systems engineering and software engineering originates in the 1950s and 1960s, but the origin of functional modelling of organizational activity goes back to the late 19th century.</p>
<p>In the late 19th century the first diagrams appeared that pictured business activities, actions, processes, or operations, and in the first half of the 20th century the first structured methods for documenting business process activities emerged. One of those methods was the <a href="/wiki/Flow_process_chart" title="Flow process chart">flow process chart</a>, introduced by <a class="mw-redirect" href="/wiki/Frank_Gilbreth" title="Frank Gilbreth">Frank Gilbreth</a> to members of <a class="mw-redirect" href="/wiki/American_Society_of_Mechanical_Engineers" title="American Society of Mechanical Engineers">American Society of Mechanical Engineers</a> (ASME) in 1921 with the presentation, entitled “Process Charts—First Steps in Finding the One Best Way”.<sup class="reference" id="cite_ref-BBG02_3-0"><a href="#cite_note-BBG02-3">[3]</a></sup> Gilbreth's tools quickly found their way into <a href="/wiki/Industrial_engineering" title="Industrial engineering">industrial engineering</a> curricula.</p>
<p>The emerge of the field of <i>systems engineering</i> can be traced back to <a href="/wiki/Bell_Labs" title="Bell Labs">Bell Telephone Laboratories</a> in the 1940s.<sup class="reference" id="cite_ref-4"><a href="#cite_note-4">[4]</a></sup> The need to identify and manipulate the properties of a system as a whole, which in complex engineering projects may greatly differ from the sum of the parts' properties, motivated various industries to apply the discipline.<sup class="reference" id="cite_ref-5"><a href="#cite_note-5">[5]</a></sup> One of the first to define the function model in this field was the British engineer <a href="/wiki/William_Gosling_(engineer)" title="William Gosling (engineer)">William Gosling</a>. In his book <i>The design of engineering systems</i> (1962, p. 25) he stated:</p>
<dl>
<dd><i>A functional model must thus achieve two aims in order to be of use. It must furnish a throughput description mechanics capable of completely defining the first and last throughput states, and perhaps some of the intervening states. It must also offer some means by which any input, correctly described in terms of this mechanics, can be used to generate an output which is an equally correct description of the output which the actual system would have given for the input concerned. It may also be noted that there are two other things which a functional model may do, but which are not necessary to all functional models. Thus such a system may, but need not, describe the system throughputs other than at the input and output, and it may also contain a description of the operation which each element carries out on the throughput, but once again this is not.</i><sup class="reference" id="cite_ref-6"><a href="#cite_note-6">[6]</a></sup></dd>
</dl>
<p>One of the first well defined function models, was the <a class="mw-redirect" href="/wiki/Functional_Flow_Block_Diagram" title="Functional Flow Block Diagram">Functional Flow Block Diagram</a> (FFBD) developed by the defense-related <a href="/wiki/TRW_Inc." title="TRW Inc.">TRW</a> Incorporated in the 1950s.<sup class="reference" id="cite_ref-7"><a href="#cite_note-7">[7]</a></sup> In the 1960s it was exploited by the <a href="/wiki/NASA" title="NASA">NASA</a> to visualize the time sequence of events in a space systems and flight missions.<sup class="reference" id="cite_ref-8"><a href="#cite_note-8">[8]</a></sup> It is further widely used in classical <a href="/wiki/Systems_engineering" title="Systems engineering">systems engineering</a> to show the order of execution of system functions.<sup class="reference" id="cite_ref-TD03_9-0"><a href="#cite_note-TD03-9">[9]</a></sup></p>
<h2><span class="mw-headline" id="Functional_modeling_topics">Functional modeling topics</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Function_model&amp;action=edit&amp;section=2" title="Edit section: Functional modeling topics">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<h3><span class="mw-headline" id="Functional_perspective">Functional perspective</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Function_model&amp;action=edit&amp;section=3" title="Edit section: Functional perspective">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>In <a href="/wiki/Systems_engineering" title="Systems engineering">systems engineering</a> and <a href="/wiki/Software_engineering" title="Software engineering">software engineering</a> a function model is created with a functional <a class="mw-redirect" href="/wiki/Modeling_perspectives" title="Modeling perspectives">modeling perspective</a>. The functional perspective is one of the perspectives possible in <a class="mw-redirect" href="/wiki/Business_process_modelling" title="Business process modelling">business process modelling</a>, other perspectives are for example behavioural, organisational or informational.<sup class="reference" id="cite_ref-10"><a href="#cite_note-10">[10]</a></sup></p>
<p>A functional modeling perspective concentrates on describing the <a class="mw-redirect" href="/wiki/Dynamic_system" title="Dynamic system">dynamic process</a>. The main concept in this modeling perspective is the process, this could be a function, transformation, activity, action, task etc. A well-known example of a modeling language employing this perspective is <a href="/wiki/Data_flow_diagram" title="Data flow diagram">data flow diagrams</a>.</p>
<p>The perspective uses four symbols to describe a process, these being:</p>
<ul>
<li>Process: Illustrates transformation from input to output.</li>
<li>Store: Data-collection or some sort of material.</li>
<li>Flow: Movement of data or material in the process.</li>
<li>External Entity: External to the modeled system, but interacts with it.</li>
</ul>
<p>Now, with these symbols, a process can be represented as a network of these symbols. This decomposed process is a DFD, data flow diagram.</p>

<p>In <a class="mw-redirect" href="/wiki/Dynamic_Enterprise_Modeling" title="Dynamic Enterprise Modeling">Dynamic Enterprise Modeling</a> a division is made in the <a class="new" href="/w/index.php?title=Control_model&amp;action=edit&amp;redlink=1" title="Control model (page does not exist)">Control model</a>, Function Model, <a class="mw-redirect" href="/wiki/Process_model" title="Process model">Process model</a> and <a class="new" href="/w/index.php?title=Organizational_model&amp;action=edit&amp;redlink=1" title="Organizational model (page does not exist)">Organizational model</a>.</p>
<h3><span class="mw-headline" id="Functional_decomposition">Functional decomposition</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Function_model&amp;action=edit&amp;section=4" title="Edit section: Functional decomposition">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p><a href="/wiki/Functional_decomposition" title="Functional decomposition">Functional decomposition</a> refers broadly to the process of resolving a <a href="/wiki/Function_(mathematics)" title="Function (mathematics)">functional</a> relationship into its constituent parts in such a way that the original function can be reconstructed from those parts by <a href="/wiki/Function_composition" title="Function composition">function composition</a>. In general, this process of decomposition is undertaken either for the purpose of gaining insight into the identity of the constituent components, or for the purpose of obtaining a compressed representation of the global function, a task which is feasible only when the constituent processes possess a certain level of <i>modularity</i>.</p>
<p><a href="/wiki/Decomposition_(computer_science)" title="Decomposition (computer science)">Functional decomposition</a> has a prominent role in <a href="/wiki/Computer_programming" title="Computer programming">computer programming</a>, where a major goal is to <i>modularize</i> processes to the greatest extent possible. For example, a library management system may be broken up into an inventory module, a patron information module, and a fee assessment module. In the early decades of computer programming, this was manifested as the "art of subroutining," as it was called by some prominent practitioners.</p>
<p>Functional decomposition of engineering systems is a method for analyzing engineered systems. The basic idea is to try to divide a system in such a way that each block of the block diagram can be described without an "and" or "or" in the description.</p>
<p>This exercise forces each part of the system to have a pure <a href="/wiki/Role" title="Role">function</a>. When a system is composed of pure functions, they can be reused, or replaced. A usual side effect is that the interfaces between blocks become simple and generic. Since the interfaces usually become simple, it is easier to replace a pure function with a related, similar function.</p>
<h2><span class="mw-headline" id="Functional_modeling_methods">Functional modeling methods</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Function_model&amp;action=edit&amp;section=5" title="Edit section: Functional modeling methods">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The functional approach is extended in multiple diagrammic techniques and modeling notations. This section gives an overview of the important techniques in chronological order.</p>
<h3><span class="mw-headline" id="Function_block_diagram">Function block diagram</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Function_model&amp;action=edit&amp;section=6" title="Edit section: Function block diagram">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="thumb tright">
<div class="thumbinner" style="width:322px;"><a class="image" href="/wiki/File:Functional_block_diagram_of_the_attitude_control_and_maneuvering_electronics_system.jpg"><img alt="" class="thumbimage" data-file-height="308" data-file-width="600" height="164" src="//upload.wikimedia.org/wikipedia/commons/thumb/6/65/Functional_block_diagram_of_the_attitude_control_and_maneuvering_electronics_system.jpg/320px-Functional_block_diagram_of_the_attitude_control_and_maneuvering_electronics_system.jpg" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/6/65/Functional_block_diagram_of_the_attitude_control_and_maneuvering_electronics_system.jpg/480px-Functional_block_diagram_of_the_attitude_control_and_maneuvering_electronics_system.jpg 1.5x, //upload.wikimedia.org/wikipedia/commons/6/65/Functional_block_diagram_of_the_attitude_control_and_maneuvering_electronics_system.jpg 2x" width="320"/></a>
<div class="thumbcaption">

<p>A <a href="/wiki/Functional_block_diagram" title="Functional block diagram">functional block diagram</a> is a <a href="/wiki/Block_diagram" title="Block diagram">block diagram</a>, that describes the functions and interrelationships of a <a href="/wiki/System" title="System">system</a>. The functional block diagram can picture:<sup class="reference" id="cite_ref-11"><a href="#cite_note-11">[11]</a></sup></p>
<ul>
<li>Functions of a system pictured by blocks</li>
<li>Input and output elements of a block pictured with lines, and</li>
<li>Relationships between the functions</li>
<li>Functional sequences and paths for matter and or signals<sup class="reference" id="cite_ref-12"><a href="#cite_note-12">[12]</a></sup></li>
</ul>
<p>The block diagram can use additional schematic symbols to show particular properties.</p>
<p>Specific function block diagram are the classic <a class="mw-redirect" href="/wiki/Functional_Flow_Block_Diagram" title="Functional Flow Block Diagram">Functional Flow Block Diagram</a>, and the <a class="mw-redirect" href="/wiki/Function_Block_Diagram" title="Function Block Diagram">Function Block Diagram</a> (FBD) used in the design of <a href="/wiki/Programmable_logic_controller" title="Programmable logic controller">programmable logic controllers</a>.</p>
<h3><span class="mw-headline" id="Functional_Flow_Block_Diagram">Functional Flow Block Diagram</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Function_model&amp;action=edit&amp;section=7" title="Edit section: Functional Flow Block Diagram">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="thumb tright">
<div class="thumbinner" style="width:322px;"><a class="image" href="/wiki/File:Functional_Flow_Block_Diagram_Format.jpg"><img alt="" class="thumbimage" data-file-height="500" data-file-width="750" height="213" src="//upload.wikimedia.org/wikipedia/commons/thumb/e/e9/Functional_Flow_Block_Diagram_Format.jpg/320px-Functional_Flow_Block_Diagram_Format.jpg" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/e/e9/Functional_Flow_Block_Diagram_Format.jpg/480px-Functional_Flow_Block_Diagram_Format.jpg 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/e/e9/Functional_Flow_Block_Diagram_Format.jpg/640px-Functional_Flow_Block_Diagram_Format.jpg 2x" width="320"/></a>
<div class="thumbcaption">
<div class="magnify"><a class="internal" href="/wiki/File:Functional_Flow_Block_Diagram_Format.jpg" title="Enlarge"></a></div>
<a class="mw-redirect" href="/wiki/Functional_Flow_Block_Diagram" title="Functional Flow Block Diagram">Functional Flow Block Diagram</a> Format.<sup class="reference" id="cite_ref-SEF01_13-0"><a href="#cite_note-SEF01-13">[13]</a></sup></div>

<p>The <a href="/wiki/Functional_flow_block_diagram" title="Functional flow block diagram">Functional flow block diagram</a> (FFBD) is a multi-tier, time-sequenced, step-by-step <a href="/wiki/Flow_diagram" title="Flow diagram">flow diagram</a> of the <a href="/wiki/System" title="System">system</a>’s functional flow.<sup class="reference" id="cite_ref-faa.gov_14-0"><a href="#cite_note-faa.gov-14">[14]</a></sup> The diagram is developed in the 1950s and widely used in classical <a href="/wiki/Systems_engineering" title="Systems engineering">systems engineering</a>. The Functional Flow Block Diagram is also referred to as <i>Functional Flow Diagram</i>, <i>functional block diagram</i>, and <i>functional flow</i>.<sup class="reference" id="cite_ref-FAA08_15-0"><a href="#cite_note-FAA08-15">[15]</a></sup></p>
<p>Functional Flow Block Diagrams (FFBD) usually define the detailed, step-by-step operational and support sequences for <a href="/wiki/System" title="System">systems</a>, but they are also used effectively to define <a class="extiw" href="https://en.wiktionary.org/wiki/process" title="wikt:process">processes</a> in developing and producing systems. The <a href="/wiki/Software_development_process" title="Software development process">software development processes</a> also use FFBDs extensively. In the system context, the functional flow steps may include combinations of <a href="/wiki/Computer_hardware" title="Computer hardware">hardware</a>, <a href="/wiki/Software" title="Software">software</a>, <a class="mw-redirect" href="/wiki/Personnel" title="Personnel">personnel</a>, facilities, and/or procedures.</p>
<p>In the FFBD method, the functions are organized and depicted by their logical order of execution. Each function is shown with respect to its logical relationship to the execution and completion of other functions. A node labeled with the function name depicts each function. Arrows from left to right show the order of execution of the functions. Logic symbols represent sequential or parallel execution of functions.<sup class="reference" id="cite_ref-FAA06_16-0"><a href="#cite_note-FAA06-16">[16]</a></sup></p>
<h3><span class="mw-headline" id="HIPO_and_IPO">HIPO and IPO</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Function_model&amp;action=edit&amp;section=8" title="Edit section: HIPO and IPO">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="thumb tleft">

<p><a class="mw-redirect" href="/wiki/HIPO" title="HIPO">HIPO</a> for <i>hierarchical input process output</i> is a popular 1970s <a href="/wiki/Systems_analysis" title="Systems analysis">systems analysis</a> design aid and documentation technique<sup class="reference" id="cite_ref-17"><a href="#cite_note-17">[17]</a></sup> for representing the modules of a <a href="/wiki/System" title="System">system</a> as a <a href="/wiki/Hierarchy" title="Hierarchy">hierarchy</a> and for documenting each module.<sup class="reference" id="cite_ref-Sandia92_18-0"><a href="#cite_note-Sandia92-18">[18]</a></sup></p>
<p>It was used to develop requirements, construct the design, and support implementation of an expert system to demonstrate automated rendezvous. Verification was then conducted systematically because of the method of design and implementation.<sup class="reference" id="cite_ref-19"><a href="#cite_note-19">[19]</a></sup></p>
<p>The overall design of the system is documented using HIPO charts or <a href="/wiki/Structure_chart" title="Structure chart">structure charts</a>. The structure chart is similar in appearance to an organizational chart, but has been modified to show additional detail. Structure charts can be usedto display several types of information, but are used most commonly to diagram either <a href="/wiki/Data_structure" title="Data structure">data structures</a> or code structures.<sup class="reference" id="cite_ref-Sandia92_18-1"><a href="#cite_note-Sandia92-18">[18]</a></sup></p>
<h3><span class="mw-headline" id="N2_Chart">N2 Chart</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Function_model&amp;action=edit&amp;section=9" title="Edit section: N2 Chart">edit</a><span class="mw-editsection-bracket">]</span></span></h3>

<p>The <a class="mw-redirect" href="/wiki/N2_Chart" title="N2 Chart">N2 Chart</a> is a diagram in the shape of a <a href="/wiki/Matrix_(mathematics)" title="Matrix (mathematics)">matrix</a>, representing functional or physical interfaces between system elements. It is used to systematically identify, define, tabulate, design, and analyze functional and physical interfaces. It applies to system <a href="/wiki/Interface_(computing)" title="Interface (computing)">interfaces</a> and <a href="/wiki/Computer_hardware" title="Computer hardware">hardware</a> and/or <a href="/wiki/Software" title="Software">software</a> interfaces.<sup class="reference" id="cite_ref-faa.gov_14-1"><a href="#cite_note-faa.gov-14">[14]</a></sup></p>
<p>The N2 diagram has been used extensively to develop data interfaces, primarily in the <a href="/wiki/Software" title="Software">software</a> areas. However, it can also be used to develop hardware interfaces. The basic N2 chart is shown in Figure 2. The system functions are placed on the diagonal; the remainder of the squares in the N x N matrix represent the interface inputs and outputs. <sup class="reference" id="cite_ref-NASA95_20-1"><a href="#cite_note-NASA95-20">[20]</a></sup></p>
<h3><span class="mw-headline" id="Structured_Analysis_and_Design_Technique">Structured Analysis and Design Technique</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Function_model&amp;action=edit&amp;section=10" title="Edit section: Structured Analysis and Design Technique">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="thumb tleft">
<div class="thumbinner" style="width:182px;"><a class="image" href="/wiki/File:SADT.svg"><img alt="" class="thumbimage" data-file-height="550" data-file-width="780" height="127" src="//upload.wikimedia.org/wikipedia/commons/thumb/e/e3/SADT.svg/180px-SADT.svg.png" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/e/e3/SADT.svg/270px-SADT.svg.png 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/e/e3/SADT.svg/360px-SADT.svg.png 2x" width="180"/></a>
<div class="thumbcaption">
<div class="magnify"><a class="internal" href="/wiki/File:SADT.svg" title="Enlarge"></a></div>
SADT basis element.</div>

<p><a class="mw-redirect" href="/wiki/Structured_Analysis_and_Design_Technique" title="Structured Analysis and Design Technique">Structured Analysis and Design Technique</a> (SADT) is a <a class="mw-redirect" href="/wiki/Software_engineering_methodology" title="Software engineering methodology">software engineering methodology</a> for describing <a href="/wiki/System" title="System">systems</a> as a hierarchy of functions, a <a href="/wiki/Diagram" title="Diagram">diagrammatic</a> notation for constructing a sketch for a software application. It offers building blocks to represent entities and activities, and a variety of arrows to relate boxes. These boxes and arrows have an associated informal <a href="/wiki/Semantics" title="Semantics">semantics</a>.<sup class="reference" id="cite_ref-JM04_21-0"><a href="#cite_note-JM04-21">[21]</a></sup> SADT can be used as a functional analysis tool of a given process, using successive levels of details. The SADT method allows to define user needs for IT developments, which is used in industrial Information Systems, but also to explain and to present an activity’s manufacturing processes, procedures.<sup class="reference" id="cite_ref-FL_22-0"><a href="#cite_note-FL-22">[22]</a></sup></p>
<p>The SADT supplies a specific functional view of any enterprise by describing the functions and their relationships in a company. These functions fulfill the objectives of a company, such as sales, order planning, product design, part manufacturing, and human resource management. The SADT can depict simple functional relationships and can reflect data and control flow relationships between different functions. The <a href="/wiki/IDEF0" title="IDEF0">IDEF0</a> formalism is based on SADT, developed by <a href="/wiki/Douglas_T._Ross" title="Douglas T. Ross">Douglas T. Ross</a> in 1985.<sup class="reference" id="cite_ref-23"><a href="#cite_note-23">[23]</a></sup></p>
<h3><span class="mw-headline" id="IDEF0">IDEF0</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Function_model&amp;action=edit&amp;section=11" title="Edit section: IDEF0">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="thumb tright">
<div class="thumbinner" style="width:322px;"><a class="image" href="/wiki/File:IDEF_Diagram_Example.jpg"><img alt="" class="thumbimage" data-file-height="600" data-file-width="750" height="256" src="//upload.wikimedia.org/wikipedia/commons/thumb/3/31/IDEF_Diagram_Example.jpg/320px-IDEF_Diagram_Example.jpg" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/3/31/IDEF_Diagram_Example.jpg/480px-IDEF_Diagram_Example.jpg 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/3/31/IDEF_Diagram_Example.jpg/640px-IDEF_Diagram_Example.jpg 2x" width="320"/></a>
<div class="thumbcaption">
<div class="magnify"><a class="internal" href="/wiki/File:IDEF_Diagram_Example.jpg" title="Enlarge"></a></div>
<a href="/wiki/IDEF0" title="IDEF0">IDEF0</a> Diagram Example</div>
</div>

<p><a href="/wiki/IDEF0" title="IDEF0">IDEF0</a> is a <a class="mw-redirect" href="/wiki/Function_modeling" title="Function modeling">function modeling</a> methodology for describing <a href="/wiki/Manufacturing" title="Manufacturing">manufacturing</a> functions, which offers a functional <a href="/wiki/Modeling_language" title="Modeling language">modeling language</a> for the analysis, development, re-engineering, and integration of <a href="/wiki/Information_system" title="Information system">information systems</a>; business processes; or software engineering analysis.<sup class="reference" id="cite_ref-DAC01_24-0"><a href="#cite_note-DAC01-24">[24]</a></sup> It is part of the <a href="/wiki/IDEF" title="IDEF">IDEF</a> family of modeling languages in the field of <a href="/wiki/Software_engineering" title="Software engineering">software engineering</a>, and is built on the functional modeling language building <a class="mw-redirect" href="/wiki/Structured_Analysis_and_Design_Technique" title="Structured Analysis and Design Technique">SADT</a>.</p>
<p>The IDEF0 Functional Modeling method is designed to model the decisions, actions, and activities of an organization or system.<sup class="reference" id="cite_ref-VG00_25-0"><a href="#cite_note-VG00-25">[25]</a></sup> It was derived from the established graphic modeling language <a class="mw-redirect" href="/wiki/Structured_Analysis_and_Design_Technique" title="Structured Analysis and Design Technique">Structured Analysis and Design Technique</a> (SADT) developed by <a href="/wiki/Douglas_T._Ross" title="Douglas T. Ross">Douglas T. Ross</a> and <a href="/wiki/SofTech,_Inc." title="SofTech, Inc.">SofTech, Inc.</a>. In its original form, IDEF0 includes both a definition of a graphical modeling language (<a href="/wiki/Syntax" title="Syntax">syntax</a> and <a href="/wiki/Semantics" title="Semantics">semantics</a>) and a description of a comprehensive methodology for developing models.<sup class="reference" id="cite_ref-ITL93_1-1"><a href="#cite_note-ITL93-1">[1]</a></sup> The US Air Force commissioned the SADT developers to develop a function model method for analyzing and communicating the functional perspective of a system. IDEF0 should assist in organizing system analysis and promote effective communication between the analyst and the customer through simplified graphical devices.<sup class="reference" id="cite_ref-VG00_25-1"><a href="#cite_note-VG00-25">[25]</a></sup></p>
<h3><span class="mw-headline" id="Axiomatic_Design">Axiomatic Design</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Function_model&amp;action=edit&amp;section=12" title="Edit section: Axiomatic Design">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p><a href="/wiki/Axiomatic_design" title="Axiomatic design">Axiomatic design</a> is a top down hierarchical functional decomposition process used as a solution synthesis framework for the analysis, development, re-engineering, and integration of products, information systems, business processes or software engineering solutions.<sup class="reference" id="cite_ref-AD_26-0"><a href="#cite_note-AD-26">[26]</a></sup> Its structure is suited mathematically to analyze coupling between functions in order to optimize the architectural robustness of potential functional solution models.</p>
<h2><span class="mw-headline" id="Related_types_of_models">Related types of models</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Function_model&amp;action=edit&amp;section=13" title="Edit section: Related types of models">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>In the field of systems and software engineering numerous specific function and functional models and close related models have been defined. Here only a few general types will be explained.</p>
<h3><span class="mw-headline" id="Business_function_model">Business function model</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Function_model&amp;action=edit&amp;section=14" title="Edit section: Business function model">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>A <i>Business Function Model</i> (BFM) is a general description or category of operations performed routinely to carry out an organization's mission. They "provide a conceptual structure for the identification of general business functions".<sup class="reference" id="cite_ref-27"><a href="#cite_note-27">[27]</a></sup> It can show the critical <a href="/wiki/Business_process" title="Business process">business processes</a> in the context of the business area functions. The processes in the business function model must be consistent with the processes in the value chain models. Processes are a group of related business activities performed to produce an end product or to provide a service. Unlike business functions that are performed on a continual basis, processes are characterized by the fact that they have a specific beginning and an end point marked by the delivery of a desired output. The figure on the right depicts the relationship between the business processes, business functions, and the business area’s business reference model.<sup class="reference" id="cite_ref-28"><a href="#cite_note-28">[28]</a></sup></p>
<h3><span class="mw-headline" id="Business_Process_Model_and_Notation">Business Process Model and Notation</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Function_model&amp;action=edit&amp;section=15" title="Edit section: Business Process Model and Notation">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="thumb tright">

<p><a href="/wiki/Business_Process_Model_and_Notation" title="Business Process Model and Notation">Business Process Model and Notation</a> (BPMN) is a <a class="mw-redirect" href="/wiki/Information_visualisation" title="Information visualisation">graphical representation</a> for specifying <a href="/wiki/Business_process" title="Business process">business processes</a> in a <a href="/wiki/Workflow" title="Workflow">workflow</a>. BPMN was developed by <a class="mw-redirect" href="/wiki/Business_Process_Management_Initiative" title="Business Process Management Initiative">Business Process Management Initiative</a> (BPMI), and is currently maintained by the <a href="/wiki/Object_Management_Group" title="Object Management Group">Object Management Group</a> since the two organizations merged in 2005. The current version of BPMN is 2.0.<sup class="reference" id="cite_ref-bpmifaq_29-0"><a href="#cite_note-bpmifaq-29">[29]</a></sup></p>
<p>The Business Process Model and Notation (BPMN) specification provides a graphical notation for specifying <a href="/wiki/Business_process" title="Business process">business processes</a> in a <a class="new" href="/w/index.php?title=Business_Process_Diagram&amp;action=edit&amp;redlink=1" title="Business Process Diagram (page does not exist)">Business Process Diagram</a> (BPD).<sup class="reference" id="cite_ref-30"><a href="#cite_note-30">[30]</a></sup> The objective of BPMN is to support business process management for both technical users and business users by providing a notation that is intuitive to business users yet able to represent complex process semantics. The BPMN specification also provides a mapping between the graphics of the notation to the underlying constructs of execution languages, particularly <a class="mw-redirect" href="/wiki/BPEL4WS" title="BPEL4WS">BPEL4WS</a>.<sup class="reference" id="cite_ref-31"><a href="#cite_note-31">[31]</a></sup></p>
<h3><span class="mw-headline" id="Business_reference_model">Business reference model</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Function_model&amp;action=edit&amp;section=16" title="Edit section: Business reference model">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="thumb tright">

<div class="thumbcaption">
<div class="magnify"><a class="internal" href="/wiki/File:FEA_BRM_Hierachy.JPG" title="Enlarge"></a></div>
This FEA <a href="/wiki/Business_reference_model" title="Business reference model">Business reference model</a> depicts the relationship between the business processes, business functions, and the business area’s business reference model.</div>

<p>A <a href="/wiki/Business_reference_model" title="Business reference model">Business reference model</a> is a reference model, concentrating on the functional and organizational aspects of the <a href="/wiki/Core_business" title="Core business">core business</a> of an <a href="/wiki/Business" title="Business">enterprise</a>, <a href="/wiki/Tertiary_sector_of_the_economy" title="Tertiary sector of the economy">service organization</a> or <a href="/wiki/Government_agency" title="Government agency">government agency</a>. In enterprise engineering a business reference model is part of an <a class="mw-redirect" href="/wiki/Enterprise_Architecture_Framework" title="Enterprise Architecture Framework">Enterprise Architecture Framework</a> or <i>Architecture Framework</i>, which defines how to organize the structure and <a href="/wiki/View_model" title="View model">views</a> associated with an <a class="mw-redirect" href="/wiki/Enterprise_Architecture" title="Enterprise Architecture">Enterprise Architecture</a>.</p>
<p>A <a href="/wiki/Reference_model" title="Reference model">reference model</a> in general is a model of something that embodies the basic goal or idea of something and can then be looked at as a reference for various purposes. A business reference model is a means to describe the <a href="/wiki/Business_operations" title="Business operations">business operations</a> of an organization, independent of the <a href="/wiki/Organizational_structure" title="Organizational structure">organizational structure</a> that perform them. Other types of business reference model can also depict the relationship between the <a href="/wiki/Business_process" title="Business process">business processes</a>, business functions, and the <a class="mw-redirect" href="/wiki/Business_area" title="Business area">business area</a>’s business reference model. These reference model can be constructed in layers, and offer a foundation for the analysis of service components, technology, data, and performance.</p>
<h3><span class="mw-headline" id="Operator_function_model">Operator function model</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Function_model&amp;action=edit&amp;section=17" title="Edit section: Operator function model">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>The <i>Operator Function Model</i> (OFM) is proposed as an alternative to traditional <a href="/wiki/Task_analysis" title="Task analysis">task analysis</a> techniques used by <a class="mw-redirect" href="/wiki/Human_factors" title="Human factors">human factors</a> engineers. An operator function model attempts to represent in mathematical form how an operator might decompose a complex system into simpler parts and coordinate control actions and system configurations so that acceptable overall system performance is achieved. The model represents basic issues of knowledge representation, information flow, and decision making in complex systems. Miller (1985) suggests that the network structure can be thought of as a possible representation of an operator's <a href="/wiki/Mental_model" title="Mental model">internal model</a> of the system plus a control structure which specifies how the model is used to solve the decision problems that comprise operator control functions.<sup class="reference" id="cite_ref-32"><a href="#cite_note-32">[32]</a></sup>&gt;</p>
