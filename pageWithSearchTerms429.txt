<p><b>Domain-driven design</b> (<b>DDD</b>) is an approach to <a href="/wiki/Software_development" title="Software development">software development</a> for complex needs by connecting the <a href="/wiki/Implementation" title="Implementation">implementation</a> to an evolving model.<sup class="reference" id="cite_ref-definition_1-0"><a href="#cite_note-definition-1">[1]</a></sup> The premise of domain-driven design is the following:</p>
<ul>
<li>placing the project's primary focus on the core <a href="/wiki/Domain_(software_engineering)" title="Domain (software engineering)">domain</a> and domain logic;</li>
<li>basing complex designs on a model of the domain;</li>
<li>initiating a creative collaboration between technical and <a class="mw-redirect" href="/wiki/Domain_expert" title="Domain expert">domain experts</a> to iteratively refine a conceptual model that addresses particular domain problems.</li>
</ul>
<p>The term was coined by <a class="new" href="/w/index.php?title=Eric_Evans_(technologist)&amp;action=edit&amp;redlink=1" title="Eric Evans (technologist) (page does not exist)">Eric Evans</a> in his book of the same title.<sup class="reference" id="cite_ref-book.evans_2-0"><a href="#cite_note-book.evans-2">[2]</a></sup></p>

<h2><span class="mw-headline" id="Concepts">Concepts</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Domain-driven_design&amp;action=edit&amp;section=1" title="Edit section: Concepts">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Concepts of the model include:</p>
<dl>
<dt>Context</dt>
<dd>The setting in which a word or statement appears that determines its meaning;</dd>
<dt>Domain</dt>
<dd>A sphere of knowledge (<a href="/wiki/Ontology_(information_science)" title="Ontology (information science)">ontology</a>), influence, or activity. The subject area to which the user applies a program is the domain of the software;</dd>
<dt>Model</dt>
<dd>A system of abstractions that describes selected aspects of a domain and can be used to solve problems related to that domain;</dd>
<dt>Ubiquitous Language</dt>
<dd>A language structured around the <a href="/wiki/Domain_model" title="Domain model">domain model</a> and used by all team members to connect all the activities of the team with the software.</dd>
</dl>

<p>Ideally, it would be preferable to have a single, unified model. While this is a noble goal, in reality it typically fragments into multiple models. It is useful to recognize this fact of life and work with it.</p>
<p>Strategic Design is a set of principles for maintaining model integrity, distillation of the Domain Model and working with multiple models.</p>
<h3><span class="mw-headline" id="Bounded_context">Bounded context</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Domain-driven_design&amp;action=edit&amp;section=3" title="Edit section: Bounded context">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Multiple models are in play on any large project. Yet when code based on distinct models is combined, software becomes buggy, unreliable, and difficult to understand. Communication among team members becomes confusing. It is often unclear in what context a model should not be applied.</p>
<p><b>Therefore</b>: Explicitly define the context within which a model applies. Explicitly set boundaries in terms of team organization, usage within specific parts of the application, and physical manifestations such as code bases and database schemas. Keep the model strictly consistent within these bounds, but don’t be distracted or confused by issues outside.</p>
<h3><span class="mw-headline" id="Continuous_integration"><a href="/wiki/Continuous_integration" title="Continuous integration">Continuous integration</a></span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Domain-driven_design&amp;action=edit&amp;section=4" title="Edit section: Continuous integration">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>When a number of people are working in the same bounded context, there is a strong tendency for the model to fragment. The bigger the team, the bigger the problem, but as few as three or four people can encounter serious problems. Yet breaking down the system into ever-smaller contexts eventually loses a valuable level of integration and coherency.</p>
<p><b>Therefore</b>: Institute a process of merging all code and other implementation artifacts frequently, with automated tests to flag fragmentation quickly. Relentlessly exercise the ubiquitous language to hammer out a shared view of the model as the concepts evolve in different people’s heads.</p>
<h3><span class="mw-headline" id="Context_map">Context map</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Domain-driven_design&amp;action=edit&amp;section=5" title="Edit section: Context map">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>An individual bounded context leaves some problems in the absence of a global view. The context of other models may still be vague and in flux.</p>
<p>People on other teams won’t be very aware of the context bounds and will unknowingly make changes that blur the edges or complicate the interconnections. When connections must be made between different contexts, they tend to bleed into each other.</p>
<p><b>Therefore</b>: Identify each model in play on the project and define its bounded context. This includes the implicit models of non-object-oriented subsystems. Name each bounded context, and make the names part of the ubiquitous language. Describe the points of contact between the models, outlining explicit translation for any communication and highlighting any sharing. Map the existing terrain.</p>
<h2><span class="mw-headline" id="Building_blocks">Building blocks</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Domain-driven_design&amp;action=edit&amp;section=6" title="Edit section: Building blocks">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>In the book <i>Domain-Driven Design</i>,<sup class="reference" id="cite_ref-book.evans_2-1"><a href="#cite_note-book.evans-2">[2]</a></sup> a number of high-level concepts and practices are articulated, such as <i>ubiquitous language</i> meaning that the domain model should form a <i>common language</i> given by domain experts for describing system requirements, that works equally well for the business users or sponsors and for the software developers. The book is very focused on describing the <a class="mw-redirect" href="/wiki/Domain_layer" title="Domain layer">domain layer</a> as one of the <a class="mw-redirect" href="/wiki/Common_layers_in_an_information_system_logical_architecture" title="Common layers in an information system logical architecture">common layers</a> in an <a class="mw-redirect" href="/wiki/Object-oriented" title="Object-oriented">object-oriented</a> system with a <a href="/wiki/Multilayered_architecture" title="Multilayered architecture">multilayered architecture</a>. In DDD, there are artifacts to express, create, and retrieve domain models:</p>
<dl>
<dt><a href="/wiki/Entity" title="Entity">Entity</a></dt>
<dd>An object that is not defined by its attributes, but rather by a thread of continuity and its <a href="/wiki/Identity_(object-oriented_programming)" title="Identity (object-oriented programming)">identity</a>.</dd>
<dd><i>Example: Most airlines distinguish each seat uniquely on every flight. Each seat is an entity in this context. However, Southwest Airlines, EasyJet and Ryanair do not distinguish between every seat; all seats are the same. In this context, a seat is actually a value object.</i></dd>
<dt><a href="/wiki/Value_object" title="Value object">Value Object</a></dt>
<dd>An object that contains attributes but has no conceptual identity. They should be treated as <a href="/wiki/Immutable_object" title="Immutable object">immutable</a>.</dd>
<dd><i>Example: When people exchange business cards, they generally do not distinguish between each unique card; they only are concerned about the information printed on the card. In this context, business cards are value objects.</i></dd>
<dt>Aggregate</dt>
<dd>A collection of objects that are bound together by a root entity, otherwise known as an <span id="aggregate_root"><a class="new" href="/w/index.php?title=Aggregate_root&amp;action=edit&amp;redlink=1" title="Aggregate root (page does not exist)">aggregate root</a></span>. The aggregate root guarantees the consistency of changes being made within the aggregate by forbidding external objects from holding references to its members.</dd>
<dd><i>Example: When you drive a car, you do not have to worry about moving the wheels forward, making the engine combust with spark and fuel, etc.; you are simply driving the car. In this context, the car is an aggregate of several other objects and serves as the aggregate root to all of the other systems.</i></dd>
<dt>Domain Event</dt>
<dd>A domain object that defines an event (something that happens). A domain event is an event that domain experts care about.</dd>
<dt>Service</dt>
<dd>When an operation does not conceptually belong to any object. Following the natural contours of the problem, you can implement these operations in services. See also <a href="/wiki/Service_(systems_architecture)" title="Service (systems architecture)">Service (systems architecture)</a>.</dd>
<dt>Repository</dt>
<dd>Methods for retrieving domain objects should delegate to a specialized Repository object such that alternative storage implementations may be easily interchanged.</dd>
<dt>Factory</dt>
<dd>Methods for creating domain objects should delegate to a specialized <a href="/wiki/Factory_method_pattern" title="Factory method pattern">Factory</a> object such that alternative implementations may be easily interchanged.</dd>
</dl>
<h2><span class="mw-headline" id="Disadvantages">Disadvantages</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Domain-driven_design&amp;action=edit&amp;section=7" title="Edit section: Disadvantages">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>In order to help maintain the model as a pure and helpful language construct, the team must typically implement a great deal of isolation and encapsulation within the domain model. Consequently, a system based on Domain-driven Design can come at a relatively high cost. While Domain-driven Design provides many technical benefits, such as maintainability, Microsoft recommends that it be applied only to complex domains where the model and the linguistic processes provide clear benefits in the communication of complex information, and in the formulation of a common understanding of the domain.<sup class="reference" id="cite_ref-3"><a href="#cite_note-3">[3]</a></sup></p>
<h2><span class="mw-headline" id="Relationship_to_other_ideas">Relationship to other ideas</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Domain-driven_design&amp;action=edit&amp;section=8" title="Edit section: Relationship to other ideas">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<dl>
<dt><a href="/wiki/Object-oriented_analysis_and_design" title="Object-oriented analysis and design">Object-oriented analysis and design</a></dt>
<dd>Although, in theory, the general idea of DDD need not be restricted to object-oriented approaches, in practice DDD seeks to exploit the powerful advantages that object-oriented techniques make possible. These include entities/aggregate roots as receivers of commands/method invocations and the encapsulation of state within foremost aggregate roots and on a higher architectural level, bounded contexts. The reader should be aware that object orientation is not exclusive to OO-only languages, but can be a part of functional programming, also. Applying commands/method invocations to an entity or aggregate root can be seen as an application of a function to a data structure where the result of the function application is an identical data structure with different data and/or version (especially version if optimistic concurrency is used). In dynamic languages such as Ruby or Smalltalk, object instances can be queried on whether they support a method (by name and/or signature), which is similar to how a statically typed language might choose to use an inversion of control container (or a service bus, or a service locator) to support runtime lookup of the objects – services – that support a given protocol/method/command (see CQRS further down).</dd>
<dt><a href="/wiki/Model-driven_engineering" title="Model-driven engineering">Model-driven engineering</a> (MDE)</dt>
<dt><a href="/wiki/Model-driven_architecture" title="Model-driven architecture">Model-driven architecture</a> (MDA)</dt>
<dd>While DDD is compatible with MDA, the intent of the two concepts is somewhat different. MDA is concerned more with the means of translating a model into code for different technology platforms than with the practice of defining better domain models.</dd>
<dt><a href="/wiki/Plain_Old_Java_Object" title="Plain Old Java Object">POJOs</a> and <a href="/wiki/Plain_Old_CLR_Object" title="Plain Old CLR Object">POCOs</a></dt>
<dd>POJOs and POCOs are technical implementation concepts, specific to <a href="/wiki/Java_(programming_language)" title="Java (programming language)">Java</a> and the <a class="mw-redirect" href="/wiki/.NET_framework" title=".NET framework">.NET framework</a> respectively. However, the emergence of the terms POJO and POCO reflect a growing view that, within the context of either of those technical platforms, domain objects should be defined purely to implement the business behaviour of the corresponding domain concept, rather than be defined by the requirements of a more specific technology framework.</dd>
<dt>The <a href="/wiki/Naked_objects" title="Naked objects">naked objects</a> pattern</dt>
<dd>Based on the premise that if you have a good enough domain model, the user interface can simply be a reflection of this domain model; and that if you require the user interface to be a direct reflection of the domain model then this will force the design of a better domain model.<sup class="reference" id="cite_ref-book.haywood_4-0"><a href="#cite_note-book.haywood-4">[4]</a></sup></dd>
<dt><a href="/wiki/Domain-specific_modeling" title="Domain-specific modeling">Domain-specific modeling</a> (DSM)</dt>
<dd>DSM is DDD applied through the use of <a href="/wiki/Domain-specific_language" title="Domain-specific language">Domain-specific languages</a>.</dd>
<dt><a href="/wiki/Domain-specific_language" title="Domain-specific language">Domain-specific language</a> (DSL)</dt>
<dd>DDD does not specifically require the use of a DSL, though it could be used to help define a DSL and support methods like <a href="/wiki/Domain-specific_multimodeling" title="Domain-specific multimodeling">domain-specific multimodeling</a>.</dd>
<dt><a href="/wiki/Aspect-oriented_programming" title="Aspect-oriented programming">Aspect-oriented programming</a> (AOP)</dt>
<dd>AOP makes it easy to factor out technical concerns (such as security, transaction management, logging) from a domain model, and as such makes it easier to design and implement domain models that focus purely on the business logic.</dd>
<dt><a class="mw-redirect" href="/wiki/Command_Query_Responsibility_Segregation" title="Command Query Responsibility Segregation">Command Query Responsibility Segregation</a> (CQRS)</dt>
<dd>CQRS is an architectural pattern for separation of reads from writes, where the former is a Query and the latter is a Command. Commands mutate state and are hence approximately equivalent to method invocation on aggregate roots/entities. Queries query state but do not mutate it. CQRS is a derivative architectural pattern from the design pattern called Command and Query Separation (CQS) which was coined by Meyer. While CQRS does not require DDD, domain-driven design makes the distinction between commands and queries, explicit, around the concept of an aggregate root. The idea is that a given aggregate root has a method that corresponds to a command and a command handler invokes the method on the aggregate root. The aggregate root is responsible for performing the logic of the operation and yielding either a number of events or a failure (exception or execution result enumeration/number) response OR (if Event Sourcing (ES) is not used) just mutating its state for a persister implementation such as an ORM to write to a data store, while the command handler is responsible for pulling in infrastructure concerns related to the saving of the aggregate root's state or events and creating the needed contexts (e.g. transactions).</dd>
<dt><a class="new" href="/w/index.php?title=Event_Sourcing&amp;action=edit&amp;redlink=1" title="Event Sourcing (page does not exist)">Event Sourcing</a> (ES)</dt>
<dd>An architectural pattern which warrants that your entities (as per <a class="new" href="/w/index.php?title=Eric_Evans_(technologist)&amp;action=edit&amp;redlink=1" title="Eric Evans (technologist) (page does not exist)">Eric Evans’</a> definition) do not track their internal state by means of direct serialization or O/R mapping, but by means of reading and committing events to an <a href="/wiki/Event_store" title="Event store">event store</a>. Where ES is combined with CQRS and DDD, aggregate roots are responsible for thoroughly validating and applying commands (often by means having their instance methods invoked from a Command Handler), and then publishing a single or a set of events which is also the foundation upon which the aggregate roots base their logic for dealing with method invocations. Hence, the input is a command and the output is one or many events which are transactionally (single commit) saved to an event store, and then often published on a message broker for the benefit of those interested (often the views are interested; they are then queried using Query-messages). When modeling your aggregate roots to output events, you can isolate the internal state event further than would be possible when projecting read-data from your entities, as is done in standard n-tier data-passing architectures. One significant benefit from this is that tooling such as axiomatic theorem provers (e.g. Microsoft Contracts or CHESS) are easier to apply, as the aggregate root comprehensively hides its internal state. Events are often persisted based on the version of the aggregate root instance, which yields a domain model that synchronizes in distributed systems around the concept of optimistic concurrency.</dd>
</dl>
<h2><span class="mw-headline" id="Tools">Tools</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Domain-driven_design&amp;action=edit&amp;section=9" title="Edit section: Tools">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Practicing DDD does not depend upon the use of any particular software tool or framework. Nonetheless, there is a growing number of open-source tools and frameworks that provide support to the specific patterns advocated in Evans' book or the general approach of DDD. Among these are:</p>
<ul>
<li><a href="/wiki/Actifsource" title="Actifsource">Actifsource</a> is a plug-in for <a href="/wiki/Eclipse_(software)" title="Eclipse (software)">Eclipse</a> which enables software development combining DDD with <a href="/wiki/Model-driven_engineering" title="Model-driven engineering">model-driven engineering</a> and <a href="/wiki/Automatic_programming" title="Automatic programming">code generation</a>.</li>
<li>Apache Isis is a <a class="mw-redirect" href="/wiki/Java_SE" title="Java SE">Java</a> framework for developing domain-driven and <a href="/wiki/Representational_state_transfer" title="Representational state transfer">RESTful</a> applications using the <a href="/wiki/Naked_objects" title="Naked objects">Naked Objects</a> pattern.</li>
<li><a href="/wiki/ECO_(Domain_Driven_Design)" title="ECO (Domain Driven Design)">ECO (Domain Driven Design)</a>: Framework with database, class, code and state machine generation from UML diagrams by CapableObjects.</li>
<li><a href="/wiki/OpenMDX" title="OpenMDX">OpenMDX</a>: Open source, Java based, MDA Framework supporting <a class="mw-redirect" href="/wiki/Java_SE" title="Java SE">Java SE</a>, <a class="mw-redirect" href="/wiki/Java_EE" title="Java EE">Java EE</a>, and <a href="/wiki/.NET_Framework" title=".NET Framework">.NET</a>. OpenMDX differs from typical MDA frameworks in that <i>"use models to directly drive the runtime behavior of operational systems"</i>.</li>
<li><a href="/wiki/OpenXava" title="OpenXava">OpenXava</a>: Generates an <a class="mw-redirect" href="/wiki/AJAX" title="AJAX">AJAX</a> application from <a href="/wiki/Java_Persistence_API" title="Java Persistence API">JPA</a> entities. You only need to write the domain classes to obtain a ready to use application.</li>
<li><a href="/wiki/Restful_Objects" title="Restful Objects">Restful Objects</a> is a standard for a Restful API onto a domain object model (where the domain objects may represent entities, view models, or services). Two open source frameworks (one for Java, one for .NET) can create a Restful Objects API from a domain model automatically, using reflection.</li>
<li><a href="/wiki/CubicWeb" title="CubicWeb">CubicWeb</a> is an open source semantic web framework entirely driven by a data model. High-level directives allow to refine the data model iteratively, release after release. Defining the data model is enough to get a functioning web application. Further work is required to define how the data is displayed when the default views are not sufficient.</li>
<li>ENode: is a C# framework which support to develop DDD+CQRS+Event Sourcing architecture style applications. The project URL: <a class="external free" href="https://github.com/tangxuehua/enode" rel="nofollow">https://github.com/tangxuehua/enode</a></li>
</ul>
