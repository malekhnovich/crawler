<h1 class="firstHeading" id="firstHeading" lang="en">Data modeling</h1>
<div class="mw-body-content" id="bodyContent">

A <a href="/wiki/Conceptual_schema" title="Conceptual schema">conceptual data model</a> is developed based on the data <a href="/wiki/Requirement" title="Requirement">requirements</a> for the application that is being developed, perhaps in the context of an <a href="/wiki/Activity_diagram" title="Activity diagram">activity model</a>. <p><b>Data modeling</b> in <a href="/wiki/Software_engineering" title="Software engineering">software engineering</a> is the process of creating a <a href="/wiki/Data_model" title="Data model">data model</a> for an <a href="/wiki/Information_system" title="Information system">information system</a> by applying formal data modeling techniques.</p>

<h2><span class="mw-headline" id="Overview">Overview</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Data_modeling&amp;action=edit&amp;section=1" title="Edit section: Overview">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Data modeling is a <a href="/wiki/Software_development_process" title="Software development process">process</a> used to define and analyze data <a href="/wiki/Requirement" title="Requirement">requirements</a> needed to support the <a href="/wiki/Business_process" title="Business process">business processes</a> within the scope of corresponding information systems in organizations. Therefore, the process of data modeling involves professional data modelers working closely with business stakeholders, as well as potential users of the information system.</p>
<p>There are three different types of data models produced while progressing from requirements to the actual database to be used for the information system.<sup class="reference" id="cite_ref-RS001_2-0"><a href="#cite_note-RS001-2">[2]</a></sup> The data requirements are initially recorded as a <a href="/wiki/Conceptual_schema" title="Conceptual schema">conceptual data model</a> which is essentially a set of technology independent specifications about the data and is used to discuss initial requirements with the business stakeholders. The <a class="mw-redirect" href="/wiki/Conceptual_modeling" title="Conceptual modeling">conceptual model</a> is then translated into a <a href="/wiki/Logical_data_model" title="Logical data model">logical data model</a>, which documents structures of the data that can be implemented in databases. Implementation of one conceptual data model may require multiple logical data models. The last step in data modeling is transforming the logical data model to a <a href="/wiki/Physical_data_model" title="Physical data model">physical data model</a> that organizes the data into tables, and accounts for access, performance and storage details. Data modeling defines not just data elements, but also their structures and the relationships between them.<sup class="reference" id="cite_ref-3"><a href="#cite_note-3">[3]</a></sup></p>
<p>Data modeling techniques and methodologies are used to model data in a standard, consistent, predictable manner in order to manage it as a resource. The use of data modeling standards is strongly recommended for all projects requiring a standard means of defining and analyzing data within an organization, e.g., using data modeling:</p>
<ul>
<li>to assist business analysts, programmers, testers, manual writers, IT package selectors, engineers, managers, related organizations and clients to understand and use an agreed semi-formal model the concepts of the organization and how they relate to one another</li>
<li>to manage data as a resource</li>
<li>for the integration of information systems</li>
<li>for designing databases/data warehouses (aka data repositories)</li>
</ul>
<p>Data modeling may be performed during various types of projects and in multiple phases of projects. Data models are progressive; there is no such thing as the final data model for a business or application. Instead a data model should be considered a living document that will change in response to a changing business. The data models should ideally be stored in a repository so that they can be retrieved, expanded, and edited over time. <a href="/wiki/Jeffrey_L._Whitten" title="Jeffrey L. Whitten">Whitten</a> et al. (2004) determined two types of data modeling:<sup class="reference" id="cite_ref-WBD04_4-0"><a href="#cite_note-WBD04-4">[4]</a></sup></p>
<ul>
<li>Strategic data modeling: This is part of the creation of an information systems strategy, which defines an overall vision and architecture for information systems is defined. <a href="/wiki/Information_engineering" title="Information engineering">Information engineering</a> is a methodology that embraces this approach.</li>
<li>Data modeling during systems analysis: In <a href="/wiki/Systems_analysis" title="Systems analysis">systems analysis</a> logical data models are created as part of the development of new databases.</li>
</ul>
<p>Data modeling is also used as a technique for detailing business <a href="/wiki/Requirement" title="Requirement">requirements</a> for specific <a href="/wiki/Database" title="Database">databases</a>. It is sometimes called <i>database modeling</i> because a <a href="/wiki/Data_model" title="Data model">data model</a> is eventually implemented in a database.<sup class="reference" id="cite_ref-WBD04_4-1"><a href="#cite_note-WBD04-4">[4]</a></sup></p>
<h2><span class="mw-headline" id="Data_modeling_topics">Data modeling topics</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Data_modeling&amp;action=edit&amp;section=2" title="Edit section: Data modeling topics">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<h3><span class="mw-headline" id="Data_models">Data models</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Data_modeling&amp;action=edit&amp;section=3" title="Edit section: Data models">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="hatnote" role="note">Main article: <a href="/wiki/Data_model" title="Data model">Data model</a></div>
<div class="thumb tright">
<div class="thumbinner" style="width:322px;"><a class="image" href="/wiki/File:3-4_Data_model_roles.svg"><img alt="" class="thumbimage" data-file-height="477" data-file-width="741" height="206" src="//upload.wikimedia.org/wikipedia/commons/thumb/9/93/3-4_Data_model_roles.svg/320px-3-4_Data_model_roles.svg.png" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/9/93/3-4_Data_model_roles.svg/480px-3-4_Data_model_roles.svg.png 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/9/93/3-4_Data_model_roles.svg/640px-3-4_Data_model_roles.svg.png 2x" width="320"/></a>
<div class="thumbcaption">
<div class="magnify"><a class="internal" href="/wiki/File:3-4_Data_model_roles.svg" title="Enlarge"></a></div>
How data models deliver benefit.<sup class="reference" id="cite_ref-MW99_1-1"><a href="#cite_note-MW99-1">[1]</a></sup></div>
</div>
</div>
<p>Data models provide a structure for <a href="/wiki/Data" title="Data">data</a> used within <a href="/wiki/Information_system" title="Information system">information systems</a> by providing specific definition and format. If a data model is used consistently across systems then compatibility of data can be achieved. If the same data structures are used to store and access data then different applications can share data seamlessly. The results of this are indicated in the diagram. However, systems and interfaces are often expensive to build, operate, and maintain. They may also constrain the business rather than support it. This may occur when the quality of the data models implemented in systems and interfaces is poor.<sup class="reference" id="cite_ref-MW99_1-2"><a href="#cite_note-MW99-1">[1]</a></sup></p>
<ul>
<li>Business rules, specific to how things are done in a particular place, are often fixed in the structure of a data model. This means that small changes in the way business is conducted lead to large changes in computer systems and interfaces. So, business rules need to be implemented in a flexible way that does not result in complicated dependencies, rather the data model should be flexible enough so that changes in the business can be implemented within the data model in a relatively quick and efficient way.</li>
<li>Entity types are often not identified, or are identified incorrectly. This can lead to replication of data, data structure and functionality, together with the attendant costs of that duplication in development and maintenance.Therefore, data definitions should be made as explicit and easy to understand as possible to minimize misinterpretation and duplication.</li>
<li>Data models for different systems are arbitrarily different. The result of this is that complex interfaces are required between systems that share data. These interfaces can account for between 25-70% of the cost of current systems. Required interfaces should be considered inherently while designing a data model, as a data model on its own would not be usable without interfaces within different systems.</li>
<li>Data cannot be shared electronically with customers and suppliers, because the structure and meaning of data has not been standardised. To obtain optimal value from an implemented data model, it is very important to define standards that will ensure that data models will both meet business needs and be consistent.<sup class="reference" id="cite_ref-MW99_1-3"><a href="#cite_note-MW99-1">[1]</a></sup></li>
</ul>
<h3><span class="mw-headline" id="Conceptual.2C_logical_and_physical_schemas">Conceptual, logical and physical schemas</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Data_modeling&amp;action=edit&amp;section=4" title="Edit section: Conceptual, logical and physical schemas">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="thumb tright">
<div class="thumbinner" style="width:322px;"><a class="image" href="/wiki/File:4-2_ANSI-SPARC_three_level_architecture.svg"><img alt="" class="thumbimage" data-file-height="431" data-file-width="653" height="211" src="//upload.wikimedia.org/wikipedia/commons/thumb/8/84/4-2_ANSI-SPARC_three_level_architecture.svg/320px-4-2_ANSI-SPARC_three_level_architecture.svg.png" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/8/84/4-2_ANSI-SPARC_three_level_architecture.svg/480px-4-2_ANSI-SPARC_three_level_architecture.svg.png 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/8/84/4-2_ANSI-SPARC_three_level_architecture.svg/640px-4-2_ANSI-SPARC_three_level_architecture.svg.png 2x" width="320"/></a>
<div class="thumbcaption">
<div class="magnify"><a class="internal" href="/wiki/File:4-2_ANSI-SPARC_three_level_architecture.svg" title="Enlarge"></a></div>
The ANSI/SPARC three level architecture. This shows that a data model can be an external model (or view), a conceptual model, or a physical model. This is not the only way to look at data models, but it is a useful way, particularly when comparing models.<sup class="reference" id="cite_ref-MW99_1-4"><a href="#cite_note-MW99-1">[1]</a></sup></div>
</div>
</div>
<p>In 1975 <a href="/wiki/American_National_Standards_Institute" title="American National Standards Institute">ANSI</a> described three kinds of data-model <i>instance</i>:<sup class="reference" id="cite_ref-5"><a href="#cite_note-5">[5]</a></sup></p>
<ul>
<li><a href="/wiki/Conceptual_schema" title="Conceptual schema">Conceptual schema</a>: describes the semantics of a domain (the scope of the model). For example, it may be a model of the interest area of an organization or of an industry. This consists of entity classes, representing kinds of things of significance in the domain, and relationships assertions about associations between pairs of entity classes. A conceptual schema specifies the kinds of facts or propositions that can be expressed using the model. In that sense, it defines the allowed expressions in an artificial "language" with a scope that is limited by the scope of the model. Simply described, a conceptual schema is the first step in organizing the data requirements.</li>
<li><a class="mw-redirect" href="/wiki/Logical_schema" title="Logical schema">Logical schema</a>: describes the structure of some domain of information. This consists of descriptions of (for example) tables, columns, object-oriented classes, and XML tags. The logical schema and conceptual schema are sometimes implemented as one and the same.<sup class="reference" id="cite_ref-RS001_2-1"><a href="#cite_note-RS001-2">[2]</a></sup></li>
<li><a class="mw-redirect" href="/wiki/Physical_schema" title="Physical schema">Physical schema</a>: describes the physical means used to store data. This is concerned with partitions, CPUs, <a href="/wiki/Tablespace" title="Tablespace">tablespaces</a>, and the like.</li>
</ul>
<p>According to ANSI, this approach allows the three perspectives to be relatively independent of each other. Storage technology can change without affecting either the logical or the conceptual schema. The table/column structure can change without (necessarily) affecting the conceptual schema. In each case, of course, the structures must remain consistent across all schemas of the same data model.</p>
<h3><span class="mw-headline" id="Data_modeling_process">Data modeling process</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Data_modeling&amp;action=edit&amp;section=5" title="Edit section: Data modeling process">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="thumb tright">
<div class="thumbinner" style="width:362px;"><a class="image" href="/wiki/File:Data_modeling_context.svg"><img alt="" class="thumbimage" data-file-height="507" data-file-width="638" height="286" src="//upload.wikimedia.org/wikipedia/commons/thumb/2/2b/Data_modeling_context.svg/360px-Data_modeling_context.svg.png" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/2/2b/Data_modeling_context.svg/540px-Data_modeling_context.svg.png 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/2/2b/Data_modeling_context.svg/720px-Data_modeling_context.svg.png 2x" width="360"/></a>
<div class="thumbcaption">
<div class="magnify"><a class="internal" href="/wiki/File:Data_modeling_context.svg" title="Enlarge"></a></div>
Data modeling in the context of <a href="/wiki/Business_process" title="Business process">Business Process</a> Integration.<sup class="reference" id="cite_ref-SS93_6-0"><a href="#cite_note-SS93-6">[6]</a></sup></div>
</div>
</div>
<p>In the context of <a href="/wiki/Business_process_modeling#Business_process_integration" title="Business process modeling">business process integration</a> (see figure), data modeling complements <a href="/wiki/Business_process_modeling" title="Business process modeling">business process modeling</a>, and ultimately results in database generation.<sup class="reference" id="cite_ref-SS93_6-1"><a href="#cite_note-SS93-6">[6]</a></sup></p>
<p>The process of designing a database involves producing the previously described three types of schemas - conceptual, logical, and physical. The database design documented in these schemas are converted through a <a class="mw-redirect" href="/wiki/Data_Definition_Language" title="Data Definition Language">Data Definition Language</a>, which can then be used to generate a database. A fully attributed data model contains detailed attributes (descriptions) for every entity within it. The term "database design" can describe many different parts of the design of an overall <a class="mw-redirect" href="/wiki/Database_system" title="Database system">database system</a>. Principally, and most correctly, it can be thought of as the logical design of the base data structures used to store the data. In the <a href="/wiki/Relational_model" title="Relational model">relational model</a> these are the <a href="/wiki/Table_(database)" title="Table (database)">tables</a> and <a class="mw-redirect" href="/wiki/View_(database)" title="View (database)">views</a>. In an <a href="/wiki/Object_database" title="Object database">object database</a> the entities and relationships map directly to object classes and named relationships. However, the term "database design" could also be used to apply to the overall process of designing, not just the base data structures, but also the forms and queries used as part of the overall database application within the <a class="mw-redirect" href="/wiki/Database_management_system" title="Database management system">Database Management System</a> or DBMS.</p>
<p>In the process, system <a class="mw-redirect" href="/wiki/Interface_(computer_science)" title="Interface (computer science)">interfaces</a> account for 25% to 70% of the development and support costs of current systems. The primary reason for this cost is that these systems do not share a common data model. If data models are developed on a system by system basis, then not only is the same analysis repeated in overlapping areas, but further analysis must be performed to create the interfaces between them. Most systems within an organization contain the same basic data, redeveloped for a specific purpose. Therefore, an efficiently designed basic data model can minimize rework with minimal modifications for the purposes of different systems within the organization<sup class="reference" id="cite_ref-MW99_1-5"><a href="#cite_note-MW99-1">[1]</a></sup></p>
<h3><span class="mw-headline" id="Modeling_methodologies">Modeling methodologies</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Data_modeling&amp;action=edit&amp;section=6" title="Edit section: Modeling methodologies">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Data models represent information areas of interest. While there are many ways to create data models, according to <a class="new" href="/w/index.php?title=Len_Silverston&amp;action=edit&amp;redlink=1" title="Len Silverston (page does not exist)">Len Silverston</a> (1997)<sup class="reference" id="cite_ref-SIG97_7-0"><a href="#cite_note-SIG97-7">[7]</a></sup> only two modeling methodologies stand out, top-down and bottom-up:</p>
<ul>
<li>Bottom-up models or View Integration models are often the result of a <a class="mw-redirect" href="/wiki/Reengineering_(software)" title="Reengineering (software)">reengineering</a> effort. They usually start with existing data structures forms, fields on application screens, or reports. These models are usually physical, application-specific, and incomplete from an <a href="/wiki/Enterprise_architecture" title="Enterprise architecture">enterprise perspective</a>. They may not promote data sharing, especially if they are built without reference to other parts of the organization.<sup class="reference" id="cite_ref-SIG97_7-1"><a href="#cite_note-SIG97-7">[7]</a></sup></li>
<li>Top-down <a href="/wiki/Logical_data_model" title="Logical data model">logical data models</a>, on the other hand, are created in an abstract way by getting information from people who know the subject area. A system may not implement all the entities in a logical model, but the model serves as a reference point or template.<sup class="reference" id="cite_ref-SIG97_7-2"><a href="#cite_note-SIG97-7">[7]</a></sup></li>
</ul>
<p>Sometimes models are created in a mixture of the two methods: by considering the data needs and structure of an application and by consistently referencing a subject-area model. Unfortunately, in many environments the distinction between a logical data model and a physical data model is blurred. In addition, some <a href="/wiki/Computer-aided_software_engineering" title="Computer-aided software engineering">CASE</a> tools don’t make a distinction between logical and <a href="/wiki/Physical_data_model" title="Physical data model">physical data models</a>.<sup class="reference" id="cite_ref-SIG97_7-3"><a href="#cite_note-SIG97-7">[7]</a></sup></p>
<h3><span class="mw-headline" id="Entity_relationship_diagrams">Entity relationship diagrams</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Data_modeling&amp;action=edit&amp;section=7" title="Edit section: Entity relationship diagrams">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="hatnote" role="note">Main article: <a class="mw-redirect" href="/wiki/Entity-relationship_model" title="Entity-relationship model">Entity-relationship model</a></div>
<div class="thumb tright">

<p>There are several notations for data modeling. The actual model is frequently called "Entity relationship model", because it depicts data in terms of the entities and relationships described in the <a href="/wiki/Data" title="Data">data</a>.<sup class="reference" id="cite_ref-WBD04_4-2"><a href="#cite_note-WBD04-4">[4]</a></sup> An entity-relationship model (ERM) is an abstract conceptual representation of structured data. Entity-relationship modeling is a relational schema <a href="/wiki/Database_model" title="Database model">database modeling</a> method, used in <a href="/wiki/Software_engineering" title="Software engineering">software engineering</a> to produce a type of <a href="/wiki/Conceptual_schema" title="Conceptual schema">conceptual data model</a> (or <a href="/wiki/Semantic_data_model" title="Semantic data model">semantic data model</a>) of a system, often a <a href="/wiki/Relational_database" title="Relational database">relational database</a>, and its requirements in a <a class="mw-redirect" href="/wiki/Top-down" title="Top-down">top-down</a> fashion.</p>
<p>These models are being used in the first stage of <a href="/wiki/Information_system" title="Information system">information system</a> design during the <a href="/wiki/Requirements_analysis" title="Requirements analysis">requirements analysis</a> to describe information needs or the type of <a href="/wiki/Information" title="Information">information</a> that is to be stored in a <a href="/wiki/Database" title="Database">database</a>. The <a href="/wiki/Data_model" title="Data model">data modeling</a> technique can be used to describe any <a class="mw-redirect" href="/wiki/Ontology_(computer_science)" title="Ontology (computer science)">ontology</a> (i.e. an overview and classifications of used terms and their relationships) for a certain <a href="/wiki/Domain_of_discourse" title="Domain of discourse">universe of discourse</a> i.e. area of interest.</p>
<p>Several techniques have been developed for the design of data models. While these methodologies guide data modelers in their work, two different people using the same methodology will often come up with very different results. Most notable are:</p>

<h3><span class="mw-headline" id="Generic_data_modeling">Generic data modeling</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Data_modeling&amp;action=edit&amp;section=8" title="Edit section: Generic data modeling">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="hatnote" role="note">Main article: <a href="/wiki/Generic_data_model" title="Generic data model">Generic data model</a></div>
<div class="thumb tright">
<div class="thumbinner" style="width:322px;"><a class="image" href="/wiki/File:HL7_Reference_Information_Model.jpg"><img alt="" class="thumbimage" data-file-height="403" data-file-width="550" height="234" src="//upload.wikimedia.org/wikipedia/commons/thumb/f/f8/HL7_Reference_Information_Model.jpg/320px-HL7_Reference_Information_Model.jpg" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/f/f8/HL7_Reference_Information_Model.jpg/480px-HL7_Reference_Information_Model.jpg 1.5x, //upload.wikimedia.org/wikipedia/commons/f/f8/HL7_Reference_Information_Model.jpg 2x" width="320"/></a>
<div class="thumbcaption">
<div class="magnify"><a class="internal" href="/wiki/File:HL7_Reference_Information_Model.jpg" title="Enlarge"></a></div>
Example of a Generic data model.<sup class="reference" id="cite_ref-9"><a href="#cite_note-9">[9]</a></sup></div>
</div>
</div>
<p>Generic data models are generalizations of conventional <a href="/wiki/Data_model" title="Data model">data models</a>. They define standardized general relation types, together with the kinds of things that may be related by such a relation type. The definition of generic data model is similar to the definition of a natural language. For example, a generic data model may define relation types such as a 'classification relation', being a <a href="/wiki/Binary_relation" title="Binary relation">binary relation</a> between an individual thing and a kind of thing (a class) and a 'part-whole relation', being a binary relation between two things, one with the role of part, the other with the role of whole, regardless the kind of things that are related.</p>
<p>Given an extensible list of classes, this allows the classification of any individual thing and to specify part-whole relations for any individual object. By standardization of an extensible list of relation types, a generic data model enables the expression of an unlimited number of kinds of facts and will approach the capabilities of natural languages. Conventional data models, on the other hand, have a fixed and limited domain scope, because the instantiation (usage) of such a model only allows expressions of kinds of facts that are predefined in the model.</p>
<h3><span class="mw-headline" id="Semantic_data_modeling">Semantic data modeling</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Data_modeling&amp;action=edit&amp;section=9" title="Edit section: Semantic data modeling">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="hatnote" role="note">Main article: <a href="/wiki/Semantic_data_model" title="Semantic data model">Semantic data model</a></div>
<p>The logical data structure of a DBMS, whether hierarchical, network, or relational, cannot totally satisfy the requirements for a conceptual definition of data because it is limited in scope and biased toward the implementation strategy employed by the DBMS.</p>
<div class="thumb tright">
<div class="thumbinner" style="width:322px;"><a class="image" href="/wiki/File:A2_4_Semantic_Data_Models.jpg"><img alt="" class="thumbimage" data-file-height="333" data-file-width="608" height="175" src="//upload.wikimedia.org/wikipedia/commons/thumb/a/a7/A2_4_Semantic_Data_Models.jpg/320px-A2_4_Semantic_Data_Models.jpg" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/a/a7/A2_4_Semantic_Data_Models.jpg/480px-A2_4_Semantic_Data_Models.jpg 1.5x, //upload.wikimedia.org/wikipedia/commons/a/a7/A2_4_Semantic_Data_Models.jpg 2x" width="320"/></a>
<div class="thumbcaption">
<div class="magnify"><a class="internal" href="/wiki/File:A2_4_Semantic_Data_Models.jpg" title="Enlarge"></a></div>
Semantic data models.<sup class="reference" id="cite_ref-FIPS184_8-1"><a href="#cite_note-FIPS184-8">[8]</a></sup></div>
</div>
</div>
<p>Therefore, the need to define data from a conceptual view has led to the development of <a href="/wiki/Semantic_data_model" title="Semantic data model">semantic data modeling</a> techniques. That is, techniques to define the meaning of data within the context of its interrelationships with other data. As illustrated in the figure the real world, in terms of resources, ideas, events, etc., are symbolically defined within physical data stores. A semantic data model is an <a class="mw-redirect" href="/wiki/Abstraction_(computer_science)" title="Abstraction (computer science)">abstraction</a> which defines how the stored symbols relate to the real world. Thus, the model must be a true representation of the real world.<sup class="reference" id="cite_ref-FIPS184_8-2"><a href="#cite_note-FIPS184-8">[8]</a></sup></p>
<p>A semantic data model can be used to serve many purposes, such as:<sup class="reference" id="cite_ref-FIPS184_8-3"><a href="#cite_note-FIPS184-8">[8]</a></sup></p>
<ul>
<li>planning of data resources</li>
<li>building of shareable databases</li>
<li>evaluation of vendor software</li>
<li>integration of existing databases</li>
</ul>
<p>The overall goal of semantic data models is to capture more meaning of data by integrating relational concepts with more powerful <a class="mw-redirect" href="/wiki/Abstraction_(computer_science)" title="Abstraction (computer science)">abstraction</a> concepts known from the <a href="/wiki/Artificial_intelligence" title="Artificial intelligence">Artificial Intelligence</a> field. The idea is to provide high level modeling primitives as integral part of a data model in order to facilitate the representation of real world situations.<sup class="reference" id="cite_ref-10"><a href="#cite_note-10">[10]</a></sup></p>
