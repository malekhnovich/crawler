<h1 class="firstHeading" id="firstHeading" lang="en">Aspect-oriented software development</h1>
<div class="mw-body-content" id="bodyContent">

<p>In <a href="/wiki/Computing" title="Computing">computing</a>, <b>aspect-oriented software development</b> (AOSD) is a <a class="mw-redirect" href="/wiki/Software_development_methodology" title="Software development methodology">software development technology</a> that seeks new modularizations of <a href="/wiki/Software_system" title="Software system">software systems</a> in order to isolate secondary or supporting functions from the main program's <a href="/wiki/Business_logic" title="Business logic">business logic</a>. AOSD allows multiple <a href="/wiki/Concern_(computer_science)" title="Concern (computer science)">concerns</a> to be expressed separately and automatically unified into working systems.</p>
<p>Traditional software development focuses on decomposing systems into units of primary functionality, while recognizing that there are other issues of concern that do not fit well into the primary decomposition. The traditional development process leaves it to the programmers to code modules corresponding to the primary functionality and to make sure that all other issues of concern are addressed in the code wherever appropriate. Programmers need to keep in mind all the things that need to be done, how to deal with each issue, the problems associated with the possible interactions, and the execution of the right behavior at the right time. These concerns span multiple primary functional units within the application, and often result in serious problems faced during application development and maintenance. The distribution of the code for realizing a concern becomes especially critical as the requirements for that concern evolve – a system maintainer must find and correctly update a variety of situations.</p>
<p>Aspect-oriented software development focuses on the identification, specification and representation of <a href="/wiki/Cross-cutting_concern" title="Cross-cutting concern">cross-cutting concerns</a> and their modularization into separate functional units as well as their automated composition into a working system.</p>

<h2><span class="mw-headline" id="History">History</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Aspect-oriented_software_development&amp;action=edit&amp;section=1" title="Edit section: History">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Aspect-Oriented Software Development describes a number of approaches to software modularization and composition including, in order of publication, <a class="mw-redirect" href="/wiki/Reflection_(computer_science)" title="Reflection (computer science)">reflection</a> and <a href="/wiki/Metaobject" title="Metaobject">metaobject protocols</a>, <a class="mw-redirect" href="/wiki/Composition_Filters" title="Composition Filters">Composition Filters</a>,<sup class="reference" id="cite_ref-1"><a href="#cite_note-1">[1]</a></sup> developed at the <a href="/wiki/University_of_Twente" title="University of Twente">University of Twente</a> in the Netherlands, <a href="/wiki/Subject-oriented_programming" title="Subject-oriented programming">Subject-Oriented Programming</a><sup class="reference" id="cite_ref-2"><a href="#cite_note-2">[2]</a></sup> (later extended as <a class="mw-redirect" href="/wiki/Hyper/J" title="Hyper/J">Multidimensional Separation of Concerns</a>)<sup class="reference" id="cite_ref-3"><a href="#cite_note-3">[3]</a></sup> at IBM, <a class="mw-redirect" href="/wiki/Feature_Oriented_Programming" title="Feature Oriented Programming">Feature Oriented Programming</a><sup class="reference" id="cite_ref-4"><a href="#cite_note-4">[4]</a></sup> at University of Texas at Austin, <a class="new" href="/w/index.php?title=Adaptive_Programming&amp;action=edit&amp;redlink=1" title="Adaptive Programming (page does not exist)">Adaptive Programming</a><sup class="reference" id="cite_ref-5"><a href="#cite_note-5">[5]</a></sup> at <a href="/wiki/Northeastern_University" title="Northeastern University">Northeastern University</a>, USA, and <a class="mw-redirect" href="/wiki/Aspect-Oriented_Programming" title="Aspect-Oriented Programming">Aspect-Oriented Programming</a> (AOP)<sup class="reference" id="cite_ref-6"><a href="#cite_note-6">[6]</a></sup> at <a class="mw-redirect" href="/wiki/Palo_Alto_Research_Center" title="Palo Alto Research Center">Palo Alto Research Center</a>. The term <b>aspect-oriented</b> was introduced by <a href="/wiki/Gregor_Kiczales" title="Gregor Kiczales">Gregor Kiczales</a> and his team at <a class="mw-redirect" href="/wiki/Palo_Alto_Research_Center" title="Palo Alto Research Center">Palo Alto Research Center</a> who also first developed the explicit concept of AOP and the AOP language called <a href="/wiki/AspectJ" title="AspectJ">AspectJ</a> which has gained considerable acceptance and popularity within the Java developer community.</p>
<p>Currently, several <a class="mw-redirect" href="/wiki/Aspect-Oriented_Programming#Implementations" title="Aspect-Oriented Programming">aspect-oriented programming languages</a> are available for a variety of languages and platforms.</p>
<p>Just as <a href="/wiki/Object-oriented_programming" title="Object-oriented programming">object-oriented programming</a> led to the development of a large class of <a href="/wiki/Object-oriented_design" title="Object-oriented design">object-oriented development methodologies</a>, AOP has encouraged a nascent set of software engineering technologies, including methodologies for dealing with aspects, modeling techniques (often based on the ideas of the <a href="/wiki/Unified_Modeling_Language" title="Unified Modeling Language">Unified Modeling Language</a>, UML), and testing technology for assessing the effectiveness of aspect approaches. AOSD now refers to a wide range of software development techniques that support the modularization of <a href="/wiki/Cross-cutting_concern" title="Cross-cutting concern">crosscutting concerns</a> in a software system, from <a class="mw-redirect" href="/wiki/Requirement_engineering" title="Requirement engineering">requirement engineering</a> to <a href="/wiki/Business_process_management" title="Business process management">business process management</a>, <a href="/wiki/Object-oriented_analysis_and_design" title="Object-oriented analysis and design">analysis and design</a>, <a href="/wiki/Software_architecture" title="Software architecture">architecture</a>, programming and implementation techniques, <a href="/wiki/Software_testing" title="Software testing">testing</a> and <a href="/wiki/Software_maintenance" title="Software maintenance">software maintenance</a> techniques.</p>
<p>Aspect-oriented software development has constantly gained in popularity, and is the subject of an annual conference, the International Conference on Aspect-Oriented Software Development, held for the first time in 2002 in Enschede, The Netherlands. AOSD is a rapidly evolving area. It is a popular topic of <a href="/wiki/Software_engineering" title="Software engineering">Software Engineering</a> research, especially in Europe, where research activities on AOSD are coordinated by the <a class="external text" href="http://www.aosd-europe.net/" rel="nofollow">European Network of Excellence on Aspect-Oriented Software Development</a> (AOSD-Europe), funded by the European Commission.</p>
<h2><span class="mw-headline" id="Motivation">Motivation</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Aspect-oriented_software_development&amp;action=edit&amp;section=2" title="Edit section: Motivation">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<h3><span class="mw-headline" id="Crosscutting_concerns">Crosscutting concerns</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Aspect-oriented_software_development&amp;action=edit&amp;section=3" title="Edit section: Crosscutting concerns">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="thumb tright">
<div class="thumbinner" style="width:352px;"><a class="image" href="/wiki/File:UMLComponent_blur.gif"><img alt="" class="thumbimage" data-file-height="327" data-file-width="400" height="286" src="//upload.wikimedia.org/wikipedia/en/thumb/9/99/UMLComponent_blur.gif/350px-UMLComponent_blur.gif" srcset="//upload.wikimedia.org/wikipedia/en/9/99/UMLComponent_blur.gif 1.5x" width="350"/></a>
<div class="thumbcaption">
<div class="magnify"><a class="internal" href="/wiki/File:UMLComponent_blur.gif" title="Enlarge"></a></div>
<i>Figure 3</i> – An UML architecture diagram for a telecom component</div>
</div>
</div>
<p>The <a href="/wiki/Aspect-oriented_programming#Motivation_and_basic_concepts" title="Aspect-oriented programming">motivation for aspect-oriented programming</a> approaches stem from the problems caused by code scattering and tangling. The purpose of Aspect-Oriented Software Development is to provide systematic means to modularize crosscutting concerns.</p>
<p>The implementation of a concern is <b>scattered</b> if its code is spread out over multiple modules. The concern affects the implementation of multiple modules. Its implementation is not modular.</p>
<p>The implementation of a concern is <b>tangled</b> if its code is intermixed with code that implements other concerns. The module in which tangling occurs is not cohesive.</p>
<p>Scattering and tangling often go together, even though they are different concepts.</p>
<p>Aspect-oriented software development considers that code scattering and tangling are the <a href="/wiki/Code_smell" title="Code smell">symptoms</a> of crosscutting concerns. <b>Crosscutting concerns</b> cannot be modularized using the decomposition mechanisms of the language (object or procedures) because they inherently follow different decomposition rules. The implementation and integration of these concerns with the primary <a href="/wiki/Functional_decomposition" title="Functional decomposition">functional decomposition</a> of the system causes code tangling and scattering.</p>
<h4><span class="mw-headline" id="Example_1:_Logging_in_Apache_Tomcat">Example 1: Logging in Apache Tomcat</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Aspect-oriented_software_development&amp;action=edit&amp;section=4" title="Edit section: Example 1: Logging in Apache Tomcat">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>Classloading in Tomcat is a modular concern with respect to the system decomposition. Its implementation is contained in a small number of classes and is not intertwined with the implementation of other concerns.</p>
<p>Logging in Tomcat is a crosscutting concern. Its implementation spreads over many classes and packages and is intermixed with the implementation of many other concerns.</p>

<h3><span class="mw-headline" id="Problems_caused_by_scattering_and_tangling">Problems caused by scattering and tangling</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Aspect-oriented_software_development&amp;action=edit&amp;section=7" title="Edit section: Problems caused by scattering and tangling">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Scattering and tangling of behavior are the symptoms that the implementation of a concern is not well modularized. A concern that is not modularized does not exhibit a well-defined interface. The interactions between the implementation of the concern and the modules of the system are not explicitly declared. They are encoded implicitly through the dependencies and interactions between fragments of code that implement the concern and the implementation of other modules.</p>
<p>The lack of interfaces between the implementation of crosscutting concerns and the implementation of the modules of the system impedes the development, the evolution and the maintenance of the system.</p>
<h4><span class="mw-headline" id="System_development">System development</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Aspect-oriented_software_development&amp;action=edit&amp;section=8" title="Edit section: System development">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>A module is primarily a unit of independent development. It can be implemented to a large extent independently of other modules. Modularity is achieved through the definition of well-defined interfaces between segments of the system.<sup class="reference" id="cite_ref-parnas_7-0"><a href="#cite_note-parnas-7">[7]</a></sup></p>
<p>The lack of explicit interfaces between crosscutting concerns and the modules obtained through the functional decomposition of the system imply that the implementation of these concerns, as well as the responsibility with respect to the correct implementation of these concerns, cannot be assigned to independent development teams. This responsibility has to be shared among different developers that work on the implementation of different modules of the system and have to integrate the crosscutting concern with the module behavior.</p>
<p>Furthermore, modules whose implementation is tangled with crosscutting concerns are hard to reuse in different contexts. Crosscutting impedes reuse of components. The lack of interfaces between crosscutting concerns and other modules makes it hard to represent and reason about the overall architecture of a system. As the concern is not modularized, the interactions between the concern and the top-level components of the system are hard to represent explicitly. Hence, these concerns become hard to reason about because the dependencies between crosscutting concerns and components are not specified.</p>
<p>Finally, concerns that are not modularized are hard to test in isolation. The dependencies of the concern with respect to behavior of other modules are not declared explicitly. Hence, the implementation of unit test for such concerns requires knowledge about the implementation of many modules in the system.</p>
<h4><span class="mw-headline" id="System_maintenance_and_evolution">System maintenance and evolution</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Aspect-oriented_software_development&amp;action=edit&amp;section=9" title="Edit section: System maintenance and evolution">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>The lack of support for the modular implementation of crosscutting concerns is especially problematic when the implementation of this concern needs to be modified. The comprehension of the implementation of a crosscutting concern requires the inspection of the implementation of all the modules with which it interacts. Hence, modifications of the system that affect the implementation of crosscutting concern require a manual inspection of all the locations in the code that are relevant to the crosscutting concern. The system maintainer must find and correctly update a variety of poorly identified situations.</p>
<h2><span class="mw-headline" id="Overview">Overview</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Aspect-oriented_software_development&amp;action=edit&amp;section=10" title="Edit section: Overview">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<h3><span class="mw-headline" id="Nature_of_aspect-orientation">Nature of aspect-orientation</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Aspect-oriented_software_development&amp;action=edit&amp;section=11" title="Edit section: Nature of aspect-orientation">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>The focus of aspect-oriented software development is in the investigation and implementation of new structures for software modularity that provide support for explicit abstractions to modularize concerns. <a href="/wiki/Aspect-oriented_programming" title="Aspect-oriented programming">Aspect-oriented programming</a> approaches provide explicit abstractions for the modular implementation of concerns in design, code, documentation, or other artifacts developed during the software life-cycle. These modularized concerns are called <b><a class="mw-redirect" href="/wiki/Aspect_(computer_science)" title="Aspect (computer science)">aspects</a></b>, and aspect-oriented approaches provide methods to compose them. Some approaches denote a root concern as the <b>base</b>. Various approaches provide different flexibility with respect to composition of aspects.</p>
<h4><span class="mw-headline" id="Quantification_and_obliviousness">Quantification and obliviousness</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Aspect-oriented_software_development&amp;action=edit&amp;section=12" title="Edit section: Quantification and obliviousness">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>The best known definition of the nature of AOSD is due to Filman and Friedman, which characterized AOSD using the equation <i>aspect orientation = <a href="/wiki/Quantification_(science)" title="Quantification (science)">quantification</a> + obliviousness</i>.<sup class="reference" id="cite_ref-filman_8-0"><a href="#cite_note-filman-8">[8]</a></sup></p>
<blockquote>
<p>AOP can be understood as the desire to make quantified statements about the behavior of programs, and to have these quantifications hold over programs written by oblivious programmers.<sup class="reference" id="cite_ref-filman_8-1"><a href="#cite_note-filman-8">[8]</a></sup></p>
</blockquote>
<blockquote>
<p>AOP is the desire to make statements of the form: In program P, whenever condition C arises, perform action A over a conventionally coded program P.<sup class="reference" id="cite_ref-filman_8-2"><a href="#cite_note-filman-8">[8]</a></sup></p>
</blockquote>
<p><b>Obliviousness</b> implies that a program has no knowledge of which aspects modify it where or when, whereas quantification refers to the ability of aspects to affect multiple points in the program.</p>
<p>The notion of <b><a class="new" href="/w/index.php?title=Invasive_software&amp;action=edit&amp;redlink=1" title="Invasive software (page does not exist)">non-invasiveness</a></b> is often preferred to the term obliviousness. Non-invasiveness expresses that aspects can add behavior to a program without having to perform changes in that program, yet it does not assume that programs are not aware of the aspects.</p>
<p>Filman's definition of aspect-orientation is often considered too restrictive.<sup class="reference" id="cite_ref-rashid_9-0"><a href="#cite_note-rashid-9">[9]</a></sup> Many aspect-oriented approaches use <a href="/wiki/Java_annotation" title="Java annotation">annotations</a> to explicitly declare the locations in the system where aspects introduce behavior. These approaches require the manual inspection and modification of other modules in the system and are therefore invasive. Furthermore, aspect-orientation does not necessarily require quantification. Aspects can be used to isolate features whose implementation would otherwise be tangled with other features. Such aspects do not necessarily use quantification over multiple locations in the system.</p>
<p>The essential features of Aspect-Oriented Software Development are therefore better characterized in terms of the modularity of the implementation of crosscutting concerns, the abstractions provided by aspect-oriented languages to enable modularization and the expressiveness of the aspect-oriented composition <a class="mw-redirect" href="/wiki/Operator_(programming)" title="Operator (programming)">operators</a>.</p>
<h3><span class="mw-headline" id="Concepts_and_terminology">Concepts and terminology</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Aspect-oriented_software_development&amp;action=edit&amp;section=13" title="Edit section: Concepts and terminology">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Aspect-oriented approaches provide explicit support for localizing concerns into separated modules, called <b>aspects</b>. An aspect is a module that encapsulates a concern. Most aspect-oriented languages support the non-invasive introduction of behavior into a code base and quantification over points in the program where this behavior should be introduced. These points are called <a href="/wiki/Join_point" title="Join point">join points</a>.</p>
<h4><span class="mw-headline" id="Join_point_model">Join point model</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Aspect-oriented_software_development&amp;action=edit&amp;section=14" title="Edit section: Join point model">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>Join points are points in the execution of the system, such as method calls, where behavior supplied by aspects is combined. A join point is a point in the execution of the program, which is used to define the dynamic structure of a crosscutting concern.</p>
<p>The <b>join point model</b> of an aspect-oriented language defines the types of join points that are supported by the aspect-oriented language and the possible interaction points between aspects and base modules.</p>
<p>The dynamic interpretation of join points makes it possible to expose runtime information such as the caller or callee of a method from a join point to a matching <a href="/wiki/Pointcut" title="Pointcut">pointcut</a>. Nowadays, there are various join point models around and still more under development. They heavily depend on the underlying programming language and AO language.</p>
<p>Examples of join points are:</p>
<ul>
<li>method execution</li>
<li>method call</li>
<li>field read and write access</li>
<li>exception handler execution</li>
<li>static and dynamic initialization</li>
</ul>
<p>A method call join point covers the actions of an object receiving a method call. It includes all the actions that compose a method call, starting after all arguments are evaluated up to return.</p>
<p>Many AOP approaches implement aspect behavior by weaving hooks into <b>join point shadows</b>, which is the static projection of a join point onto the program code.</p>
<p>Figure 6 illustrates possible join points in the execution of a small object-oriented program. The highlighted join points include the execution of method <i>moveBy(int, int)</i> on a <i>Line</i> object, the calls to methods <i>moveBy(int, int)</i> on the <i>Point</i> objects in the context of the <i>Line</i> object, the execution of these methods in the context of the <i>Point</i> objects and the calls and execution of the <i>setX(int)</i> and <i>setY(int)</i> methods.</p>
<h4><span class="mw-headline" id="Pointcut_designators">Pointcut designators</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Aspect-oriented_software_development&amp;action=edit&amp;section=15" title="Edit section: Pointcut designators">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p><b>The quantification over join points is expressed at the language level</b>. This quantification may be implicit in the language structure or may be expressed using a query-like construct called a pointcut. Pointcuts are defined as a predicate over the syntax-tree of the program, and define an interface that constrains which elements of the base program are exposed by the pointcut. A pointcut picks out certain join points and values at those points. The syntactic formulation of a pointcut varies from approach to approach, but a pointcut can often be composed out of other pointcuts using the boolean operators AND, OR and NOT. Pointcut expressions can concisely capture a wide range of events of interests, using wildcards. For example, in <a href="/wiki/AspectJ" title="AspectJ">AspectJ</a> syntax, the move pointcut</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="k">pointcut</span> <span class="n">move</span><span class="o">:</span> <span class="k">call</span><span class="o">(</span><span class="kd">public</span> <span class="o">*</span> <span class="n">Figure</span><span class="o">.*</span> <span class="o">(..))</span>
</pre></div>
<p>picks out each call to Figure's public methods.</p>
<p>cflow poincuts identify join points based on whether they occur in the dynamic context of other join points. For example, in AspectJ syntax <code class="mw-highlight" dir="ltr" id="" style="">cflow(move())</code> picks out each join point that occurs in the dynamic context of the join points picked out by the move pointcut.</p>
<p>Pointcuts can be classified in two categories:</p>
<ul>
<li>Kinded pointcuts, such as the call pointcut, match one kind of join point using a signature.</li>
<li>Non-kinded pointcuts, such as the cflow pointcut match all kinds of join points using a variety of properties.</li>
</ul>
<h4><span class="mw-headline" id="Advice_bodies">Advice bodies</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Aspect-oriented_software_development&amp;action=edit&amp;section=16" title="Edit section: Advice bodies">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>An advice body is code that is executed when a join point is reached. Advice modularizes the functional details of a concern. The order in which the advice bodies contributed by aspects (and by the base) may be controlled in a variety of ways, including:</p>
<ul>
<li>as a join point is reached, before the execution proceeds with the base</li>
<li>after the base semantics for the join point. When the join point corresponds to the execution of a method, an after advice can be executed after the method returned or after raising an exception</li>
<li>as the join point is reached, with explicit control over whether the base semantics is executed. Around advice can modify the control flow of the program.</li>
</ul>
<p>More general ways to describe the ordering of advice bodies in terms of partial-order graphs have also been provided.<sup class="reference" id="cite_ref-10"><a href="#cite_note-10">[10]</a></sup></p>
<p>When the execution of a join point satisfies a pointcut expression, the base and advice code associated with the join point are executed. The advice may interact with the rest system through a join point instance containing reflective information on the context of the event that triggered the advice, such as the arguments of a method call or the target instance of a call.</p>
<h4><span class="mw-headline" id="Inter-type_declarations">Inter-type declarations</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Aspect-oriented_software_development&amp;action=edit&amp;section=17" title="Edit section: Inter-type declarations">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>Inter-type declarations allow the programmer to modify a program's static structure, such as class members and classes hierarchy. New members can be inserted and classes can be pushed down the class hierarchy.</p>
<h4><span class="mw-headline" id="Aspects">Aspects</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Aspect-oriented_software_development&amp;action=edit&amp;section=18" title="Edit section: Aspects">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>An aspect is a module that encapsulates a concern. An aspect is composed of pointcuts, advice bodies and inter-type declarations. In some approaches, an aspect may also contain classes and methods.</p>
<h4><span class="mw-headline" id="Aspect_weaving">Aspect weaving</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Aspect-oriented_software_development&amp;action=edit&amp;section=19" title="Edit section: Aspect weaving">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<div class="hatnote" role="note">Main article: <a href="/wiki/Aspect_weaver" title="Aspect weaver">Aspect weaver</a></div>
<p>Aspect weaving is a composition mechanism that coordinates aspects with the other modules of the system. It is performed by a specialized compiler, called an <a href="/wiki/Aspect_weaver" title="Aspect weaver">aspect weaver</a>.</p>

<h3><span class="mw-headline" id="Aspect-oriented_requirement_engineering">Aspect-oriented requirement engineering</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Aspect-oriented_software_development&amp;action=edit&amp;section=21" title="Edit section: Aspect-oriented requirement engineering">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Aspect-oriented <a class="mw-redirect" href="/wiki/Requirement_engineering" title="Requirement engineering">requirement engineering</a> (also referred to as "Early Aspects") focuses on the identification, specification and representation of crosscutting properties at the <a href="/wiki/Requirement" title="Requirement">requirement</a> level. Examples of such properties include <a class="mw-redirect" href="/wiki/Software_security" title="Software security">security</a>, mobility, availability and <a href="/wiki/Real-time_computing" title="Real-time computing">real-time</a> constraints. Crosscutting properties are requirements, <a href="/wiki/Use_case" title="Use case">use cases</a> or features that have a broadly scoped effect on other requirements or <a href="/wiki/Software_architecture" title="Software architecture">architecture</a> components.</p>
<p>Aspect-oriented requirements engineering approaches are techniques that explicitly recognise the importance of clearly addressing both <a class="mw-redirect" href="/wiki/Functional_requirements" title="Functional requirements">functional</a> and <a class="mw-redirect" href="/wiki/Non-functional_requirements" title="Non-functional requirements">non-functional</a> crosscutting concerns in addition to non-crosscutting ones. Therefore, these approaches focus on systematically and <a href="/wiki/Modular_programming" title="Modular programming">modularly</a> treating, reasoning about, <a href="/wiki/Function_composition_(computer_science)" title="Function composition (computer science)">composing</a> and subsequently tracing crosscutting functional and non-functional concerns via suitable <a href="/wiki/Abstraction" title="Abstraction">abstraction</a>, representation and composition mechanisms tailored to the requirements engineering domain.</p>
<p>Specific areas of excellence under the denominator of AO Requirements Analysis are:</p>
<ul>
<li>the aspect-oriented requirements process itself,</li>
<li>the aspect-oriented requirements notations,</li>
<li>aspect-oriented requirements tool support,</li>
<li>adoption and integration of aspect-oriented requirements engineering, and</li>
<li>assessment/evaluation of aspect-oriented requirements.</li>
</ul>
<h3><span class="mw-headline" id="Aspect_oriented_business_process_management_.28AOBPM.29">Aspect oriented business process management (AOBPM)</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Aspect-oriented_software_development&amp;action=edit&amp;section=22" title="Edit section: Aspect oriented business process management (AOBPM)">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Reducing complexity is an important issue in Business Process Management (BPM) area. One source of complexity is rooted in the variety of concerns that a business process addresses, such as security and privacy. Ideally, these concerns should be defined separately from the business processes, as they typically span several processes, and they can be subject for change on a general organisational level instead of specific process level. However, current Business Process Management Systems do not support this kind of modelling.</p>
<p>Aspect oriented business process management (AOBPM) tries to support separation of cross-cutting concerns from the core business concerns. It defines a set of requirements and a formal model. This model is designed using <a class="mw-redirect" href="/wiki/Coloured_Petri_Net" title="Coloured Petri Net">Coloured Petri Nets</a> (CPN).</p>
<p>The approach is implemented as a service in <a href="/wiki/YAWL" title="YAWL">YAWL</a> based on <a class="mw-redirect" href="/wiki/Service_Oriented_Architecture" title="Service Oriented Architecture">Service Oriented Architecture</a>.</p>
<p>The assessment result of current aspect oriented business process management approaches are defined based on five dimensions such as Signature Exposure, Rule Composition, Advice Relations, Transformation Patterns, and Phases Support. The result can be seen in the following figure.</p>
<p><a class="image" href="/wiki/File:AOBPMAssessment.jpg" title="The result of assessing AOBPM approaches"><img alt="The result of assessing AOBPM approaches" data-file-height="518" data-file-width="1346" height="308" src="//upload.wikimedia.org/wikipedia/commons/thumb/2/24/AOBPMAssessment.jpg/800px-AOBPMAssessment.jpg" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/2/24/AOBPMAssessment.jpg/1200px-AOBPMAssessment.jpg 1.5x, //upload.wikimedia.org/wikipedia/commons/2/24/AOBPMAssessment.jpg 2x" width="800"/></a></p>
<h3><span class="mw-headline" id="Aspect-oriented_system_architecture">Aspect-oriented system architecture</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Aspect-oriented_software_development&amp;action=edit&amp;section=23" title="Edit section: Aspect-oriented system architecture">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Aspect-oriented system architecture focuses on the localization and specification of crosscutting concerns in architectural designs. Crosscutting concerns that appear at the architectural level cannot be modularized by redefining the software architecture using conventional architectural abstractions. Aspect-oriented system architecture languages propose explicit mechanisms to identify, specify and evaluate aspects at the architecture design level.</p>
<p>Aspect-oriented architecture starts from the observation that we need to identify, specify and evaluate aspects explicitly at the architecture design level. Aspectual architecture approaches describe steps for identifying architectural aspects. This information is used to redesign a given architecture in which the architectural aspects are made explicit. In this regard, specific areas of excellence are:</p>
<ul>
<li>the aspect-oriented architecture process itself,</li>
<li>the aspect-oriented architecture notations,</li>
<li>aspect-oriented architecture tool support,</li>
<li>adoption and integration of aspect-oriented architecture, and</li>
<li>assessment/evaluation of aspect-oriented architecture.</li>
</ul>
<h3><span class="mw-headline" id="Aspect-oriented_modeling_and_design">Aspect-oriented modeling and design</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Aspect-oriented_software_development&amp;action=edit&amp;section=24" title="Edit section: Aspect-oriented modeling and design">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Aspect-oriented design has the same objectives as any software design activity, i.e. characterising and specifying the behavior and structure of the software system. Its unique contribution to software design lies in the fact that concerns that are necessarily scattered and tangled in more traditional approaches can be modularized. Typically, such an approach includes both a process and a language. The process takes as input requirements and produces a design model. The produced design model represents separate concerns and their relationships. The language provides constructs that can describe the elements to be represented in the design and the relationships that can exist between those elements. In particular, constructs are provided to support concern modularization and the specification of concern composition, with consideration for conflicts. Beyond that, the design of each individual modularized concern compares to standard software design.</p>
<p>Here, specific areas of excellence areas are:</p>
<ul>
<li>the aspect-oriented design process itself,</li>
<li>the aspect-oriented design notations,</li>
<li>aspect-oriented design tool support,</li>
<li>adoption and integration of aspect-oriented design, and</li>
<li>assessment/evaluation of aspect-oriented design.</li>
</ul>
<h3><span class="mw-headline" id="Aspect-oriented_programming_.28AOP.29">Aspect-oriented programming (AOP)</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Aspect-oriented_software_development&amp;action=edit&amp;section=25" title="Edit section: Aspect-oriented programming (AOP)">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="hatnote" role="note">Main article: <a href="/wiki/Aspect-oriented_programming" title="Aspect-oriented programming">Aspect-oriented programming</a></div>
<p>AOP includes programming techniques and tools that support the modularization of concerns at the level of the source code.</p>
<p>Just like any other programming language, an aspect-oriented language typically consists of two parts: a language specification and an implementation. Hence, there are two corresponding areas of concern: support for language developers and support for application developers.</p>
<p><b>Support for application developers</b></p>
<p>An aspect-oriented approach supports the implementation of concerns and how to compose those independently implemented concerns. While the specification of such a language is the primary manual for application developers, it provides obviously no guarantee that the application developer will produce high-quality aspect-oriented programs. Specific areas of excellence:</p>
<ul>
<li>the crucial concepts of aspect-oriented programming,</li>
<li>programming in aspect-oriented languages,</li>
<li>composing software components written in any language using aspect-oriented composition mechanisms, or</li>
<li>aspect-oriented programming environments.</li>
</ul>
<p><b>Support for language developers</b></p>
<p>Excellence in constructing aspect-oriented languages includes the following areas:</p>
<ul>
<li>constructing languages or DSLs for specific domains and/or platforms, and</li>
<li>transferring implementation principles from other aspect-oriented execution environments, including:
<ul>
<li>interpreters,</li>
<li>compilers, and</li>
<li>virtual machines.</li>
</ul>
</li>
</ul>
<h3><span class="mw-headline" id="Formal_method_support_for_aspect-orientation">Formal method support for aspect-orientation</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Aspect-oriented_software_development&amp;action=edit&amp;section=26" title="Edit section: Formal method support for aspect-orientation">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p><a href="/wiki/Formal_methods" title="Formal methods">Formal methods</a> can be used both to define aspects semantically and to analyze and verify aspect-oriented systems. Aspect-oriented programming extends programming notations with aspect modules that isolate the declaration of when the aspect should be applied (join points) and what actions should be taken when it is reached (advice). Expertise in formal semantic definitions of aspect constructs is useful for <a href="/wiki/Programming_language" title="Programming language">language designers</a> to provide a deep understanding of the differences among constructs. Aspects potentially can harm the reliability of a system to which they are woven, and could invalidate essential properties that already were true of the system without the aspect. It is also necessary to show that they actually do add intended crosscutting properties to the system. Hence, numerous questions of correctness and verification are raised by aspect languages. Among the kinds of expertise are:</p>
<ul>
<li>specially designed testing techniques to provide coverage for aspects,</li>
<li><a href="/wiki/Program_slicing" title="Program slicing">program slicing</a> and <a class="mw-redirect" href="/wiki/Program_analysis_(computer_science)" title="Program analysis (computer science)">code analysis</a> approaches to identify <a href="/wiki/Feature_interaction_problem" title="Feature interaction problem">interactions</a> among aspects and between aspects and underlying systems,</li>
<li><a href="/wiki/Model_checking" title="Model checking">model checking</a> techniques specialized for aspects, and</li>
<li>inductive techniques to verify aspect-oriented systems.</li>
</ul>
<p>Each of the above approaches can be used to:</p>
<ul>
<li>specify and analyze individual aspects relative to an existing system,</li>
<li>define conditions for composing multiple aspects correctly, and</li>
<li>detect and resolve potential interferences among aspects.</li>
</ul>
<p>Although some approaches are already used in aspect languages, others are still subject of research and are not ready for routine industrial application. Nevertheless, awareness of these issues is essential for language designers, and for effective use of aspects, especially in <a href="/wiki/Mission_critical" title="Mission critical">safety-critical</a> contexts.</p>
<h3><span class="mw-headline" id="Aspect-oriented_middleware">Aspect-oriented middleware</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Aspect-oriented_software_development&amp;action=edit&amp;section=27" title="Edit section: Aspect-oriented middleware">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p><a href="/wiki/Middleware" title="Middleware">Middleware</a> and AOSD strongly complement each other. In general, areas of excellence consist of:</p>
<ul>
<li>support for the application developer, which includes
<ul>
<li>the crucial concepts of aspect supporting middleware,</li>
<li>aspect-oriented software development using a specific middleware, involving the aspect programming model, aspect deployment model, platform infrastructure, and services of the middleware, and</li>
</ul>
</li>
<li><a class="mw-redirect" href="/wiki/Product_Family_Engineering" title="Product Family Engineering">Product Family Engineering</a> (methods, architectures, techniques) in <a href="/wiki/Distributed_computing" title="Distributed computing">distributed</a> and ambient computing, and</li>
<li>support for the middleware developer with respect to
<ul>
<li>host-infrastructure middleware,</li>
<li>distribution middleware,</li>
<li>common middleware services, and</li>
<li>domain-specific middleware services.</li>
</ul>
</li>
</ul>
<h2><span class="mw-headline" id="Adoption">Adoption</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Aspect-oriented_software_development&amp;action=edit&amp;section=28" title="Edit section: Adoption">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<ul>
<li>IBM <a class="mw-redirect" href="/wiki/WebSphere_Application_Server" title="WebSphere Application Server">WebSphere Application Server</a> (WAS) is a java application server that supports Java EE and Web Services. Websphere is distributed according to editions that support different features. Websphere uses AspectJ internally to isolate features of the different editions.</li>
<li><a class="mw-redirect" href="/wiki/JBoss_Application_Server" title="JBoss Application Server">JBoss Application Server</a> (JBoss AS) is a free, open-source java application server that supports Java EE. The core of JBoss AS is integrated with the JBoss AOP aspect-oriented programming language.<sup class="reference" id="cite_ref-11"><a href="#cite_note-11">[11]</a></sup> The application server uses JBoss AOP to deploy services such as security and transaction management.</li>
<li>Oracle <a href="/wiki/TopLink" title="TopLink">TopLink</a> is a Java object-to-relational persistence framework that is integrated with the Spring Application Server. TopLink achieves high levels of persistence transparency using Spring AOP.</li>
<li><a href="/wiki/SAP_SE" title="SAP SE">SAP</a></li>
<li>Sun Microsystems uses <a href="/wiki/AspectJ" title="AspectJ">AspectJ</a> to streamline mobile application development for the Java ME platform. Aspects are used to simplify the development of mobile applications for deployment to different operator decks and different mobile gaming community interfaces.</li>
<li>Siemens <a class="mw-redirect" href="/wiki/Soarian" title="Soarian">Soarian</a> is a health information management system that supports seamless access to patient medical records and the definition of workflows for health provider organizations. Soarian uses AspectJ to integrate crosscutting features such as tracing, auditing and performance monitoring in the context of an agile development process.</li>
<li>Motorola wi4 is a cellular infrastructure system that provides support for the WiMAX wireless broadband standard. The wi4 control software is developed using an aspect-oriented extension to the UML 2.0 standard called WEAVR. WEAVR is used during the development for debugging and testing purposes.</li>
<li>ASML is a provider of lithography systems for the semiconductor industry. ASML uses an aspect-oriented extension to C called Mirjam to modularize tracing and profiling concerns.</li>
<li>Glassbox is a troubleshooting agent for Java applications that automatically diagnoses common problems. The Glassbox inspector monitors the activity of the Java virtual machine using AspectJ.</li>
<li>.NET 3.5 supports Aspect Oriented concepts through the Unity container.</li>
</ul>
