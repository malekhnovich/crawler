<h1 class="firstHeading" id="firstHeading" lang="en">Asynchronous method invocation</h1>
<div class="mw-body-content" id="bodyContent">
<div id="siteSub">From Wikipedia, the free encyclopedia</div>
<div id="contentSub"></div>
<div class="mw-jump" id="jump-to-nav">
					Jump to:					<a href="#mw-head">navigation</a>, 					<a href="#p-search">search</a>
</div>
<div class="mw-content-ltr" dir="ltr" id="mw-content-text" lang="en"><p>In (<a class="mw-redirect" href="/wiki/Thread_(computer_science)" title="Thread (computer science)">multithreaded</a>) <a href="/wiki/Object-oriented_programming" title="Object-oriented programming">object-oriented programming</a>, <b>asynchronous method invocation</b> (<b>AMI</b>), also known as <b>asynchronous method calls</b> or <b>asynchronous pattern</b> is a client-side support that doesn't block the calling thread while waiting for a reply. Instead, the calling thread is notified when the reply arrives. Polling for a reply is an undesired option.</p>

<h2><span class="mw-headline" id="Background">Background</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Asynchronous_method_invocation&amp;action=edit&amp;section=1" title="Edit section: Background">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p><b>AMI</b> is a <a href="/wiki/Software_design_pattern" title="Software design pattern">design pattern</a> for <a href="/wiki/Asynchrony_(computer_programming)" title="Asynchrony (computer programming)">asynchronous</a> invocation of potentially long-running <a class="mw-redirect" href="/wiki/Method_(computer_science)" title="Method (computer science)">methods</a> of an <a href="/wiki/Object_(computer_science)" title="Object (computer science)">object</a>.<sup class="reference" id="cite_ref-Async.34.2.2371139_1-0"><a href="#cite_note-Async.34.2.2371139-1">[1]</a></sup> It is equivalent to the <a href="/wiki/IOU" title="IOU">IOU</a> pattern described in 1996 by Allan Vermeulen.<sup class="reference" id="cite_ref-2"><a href="#cite_note-2">[2]</a></sup><sup class="reference" id="cite_ref-3"><a href="#cite_note-3">[3]</a></sup></p>
<p>In most programming languages a called method is executed synchronously, i.e. in the <a class="mw-redirect" href="/wiki/Thread_(computer_science)" title="Thread (computer science)">thread of execution</a> from which it is invoked. If the method needs a long time to completion, e.g. because it is loading data over the internet, the calling thread is blocked until the method has finished. When this is not desired, it is possible to start a "worker thread" and invoke the method from there. In most programming environments this requires many lines of code, especially if care is taken to avoid the overhead that may be caused by creating many threads. AMI solves this problem in that it augments a potentially long-running ("synchronous") object method with an "asynchronous" variant that returns immediately, along with additional methods that make it easy to receive notification of completion, or to wait for completion at a later time.</p>
<p>One common use of AMI is in the <a href="/wiki/Active_object" title="Active object">active object</a> design pattern. Alternatives are synchronous method invocation and <a href="/wiki/Futures_and_promises" title="Futures and promises">future objects</a>.<sup class="reference" id="cite_ref-active_object_4-0"><a href="#cite_note-active_object-4">[4]</a></sup> An example for an application that may make use of AMI is a web browser that needs to display a web page even before all images are loaded.</p>
<h2><span class="mw-headline" id="Implementations">Implementations</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Asynchronous_method_invocation&amp;action=edit&amp;section=2" title="Edit section: Implementations">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<h3><span class="mw-headline" id="Java_class">Java class</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Asynchronous_method_invocation&amp;action=edit&amp;section=3" title="Edit section: Java class">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>FutureTask class<sup class="reference" id="cite_ref-5"><a href="#cite_note-5">[5]</a></sup> in <a href="/wiki/Java_(programming_language)" title="Java (programming language)">Java</a> use <a href="/wiki/Event_(synchronization_primitive)" title="Event (synchronization primitive)">events</a> to solve the same problem. This pattern is a variant of AMI whose implementation carries more overhead, but it is useful for objects representing <a href="/wiki/Component-based_software_engineering" title="Component-based software engineering">software components</a>.</p>
<h3><span class="mw-headline" id=".NET_Framework">.NET Framework</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Asynchronous_method_invocation&amp;action=edit&amp;section=4" title="Edit section: .NET Framework">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<ul>
<li>Asynchronous Programming Model (APM) pattern (used before .NET Framework 2.0)<sup class="reference" id="cite_ref-6"><a href="#cite_note-6">[6]</a></sup></li>
<li>Event-based Asynchronous Pattern (EAP) (used in .NET Framework 2.0)<sup class="reference" id="cite_ref-7"><a href="#cite_note-7">[7]</a></sup></li>
<li>Task-based Asynchronous Pattern (TAP) (used in .NET Framework 4.0)<sup class="reference" id="cite_ref-8"><a href="#cite_note-8">[8]</a></sup></li>
</ul>
<h4><span class="mw-headline" id="Example">Example</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Asynchronous_method_invocation&amp;action=edit&amp;section=5" title="Edit section: Example">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>The following example is loosely based on a standard AMI style used in the <a href="/wiki/.NET_Framework" title=".NET Framework">.NET Framework</a>.<sup class="reference" id="cite_ref-ms228969_9-0"><a href="#cite_note-ms228969-9">[9]</a></sup> Given a method <code>Accomplish</code>, one adds two new methods <code>BeginAccomplish</code> and <code>EndAccomplish</code>:</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
 <span class="k">class</span> <span class="nc">Example</span> 
 <span class="p">{</span>
   <span class="n">Result</span>       <span class="nf">Accomplish</span><span class="p">(</span><span class="n">args</span> <span class="err">…</span><span class="p">)</span>
   <span class="n">IAsyncResult</span> <span class="nf">BeginAccomplish</span><span class="p">(</span><span class="n">args</span> <span class="err">…</span><span class="p">)</span>
   <span class="n">Result</span>       <span class="nf">EndAccomplish</span><span class="p">(</span><span class="n">IAsyncResult</span> <span class="n">a</span><span class="p">)</span>
   <span class="err">…</span>
 <span class="p">}</span>
</pre></div>
<p>Upon calling <code>BeginAccomplish</code>, the client immediately receives an object of type <code>AsyncResult</code> (which implements the <code>IAsyncResult</code> interface), so it can continue the calling thread with unrelated work. In the simplest case, eventually there is no more such work, and the client calls <code>EndAccomplish</code> (passing the previously received object), which blocks until the method has completed and the result is available.<sup class="reference" id="cite_ref-ms228963_10-0"><a href="#cite_note-ms228963-10">[10]</a></sup> The <code>AsyncResult</code> object normally provides at least a method that allows the client to query whether the long-running method has already completed:</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
 <span class="k">interface</span> <span class="n">IAsyncResult</span> 
 <span class="p">{</span>
    <span class="kt">bool</span> <span class="nf">HasCompleted</span><span class="p">()</span>
   <span class="err">…</span>
 <span class="p">}</span>
</pre></div>
<p>One can also pass a callback method to <code>BeginAccomplish</code>, to be invoked when the long-running method completes. It typically calls <code>EndAccomplish</code> to obtain the return value of the long-running method. A problem with the callback mechanism is that the callback function is naturally executed in the worker thread (rather than in the original calling thread), which may cause race conditions.<sup class="reference" id="cite_ref-ms228972_11-0"><a href="#cite_note-ms228972-11">[11]</a></sup><sup class="reference" id="cite_ref-Async.34.3.2376161_12-0"><a href="#cite_note-Async.34.3.2376161-12">[12]</a></sup></p>
<p>In the .NET Framework documentation, the term event-based asynchronous pattern refers to an alternative API style (available since .NET 2.0) using a method named <code>AccomplishAsync</code> instead of <code>BeginAccomplish</code>.<sup class="reference" id="cite_ref-nageletal_13-0"><a href="#cite_note-nageletal-13">[13]</a></sup><sup class="reference" id="cite_ref-hkasytyf_14-0"><a href="#cite_note-hkasytyf-14">[14]</a></sup> A superficial difference is that in this style the return value of the long-running method is passed directly to the callback method. Much more importantly, the API uses a special mechanism to run the callback method (which resides in an event object of type <code>AccomplishCompleted</code>) in the same thread in which <code>BeginAccomplish</code> was called. This eliminates the danger of race conditions, making the API easier to use and suitable for software components; on the other hand this implementation of the pattern comes with additional object creation and synchronization overhead.<sup class="reference" id="cite_ref-ms228966_15-0"><a href="#cite_note-ms228966-15">[15]</a></sup></p>
