<h1 class="firstHeading" id="firstHeading" lang="en">Inversion of control</h1>
<div class="mw-body-content" id="bodyContent">
<div id="siteSub">From Wikipedia, the free encyclopedia</div>
<div id="contentSub"></div>
<div class="mw-jump" id="jump-to-nav">
					Jump to:					<a href="#mw-head">navigation</a>, 					<a href="#p-search">search</a>
</div>
<div class="mw-content-ltr" dir="ltr" id="mw-content-text" lang="en"><p>In <a href="/wiki/Software_engineering" title="Software engineering">software engineering</a>, <b>inversion of control</b> (<b>IoC</b>) is a design principle in which custom-written portions of a <a href="/wiki/Computer_program" title="Computer program">computer program</a> receive the <a href="/wiki/Control_flow" title="Control flow">flow of control</a> from a generic <a href="/wiki/Software_framework" title="Software framework">framework</a>. A <a href="/wiki/Software_architecture" title="Software architecture">software architecture</a> with this design inverts control as compared to traditional <a href="/wiki/Procedural_programming" title="Procedural programming">procedural programming</a>: in traditional programming, the custom code that expresses the purpose of the program <a class="mw-redirect" href="/wiki/Function_call#Main_concepts" title="Function call">calls</a> into reusable libraries to take care of generic tasks, but with inversion of control, it is the framework that calls into the custom, or task-specific, code.</p>
<p>Inversion of control is used to increase <a class="mw-redirect" href="/wiki/Modularity_(programming)" title="Modularity (programming)">modularity</a> of the program and make it <a href="/wiki/Extensible_programming" title="Extensible programming">extensible</a>,<sup class="reference" id="cite_ref-1"><a href="#cite_note-1">[1]</a></sup> and has applications in <a href="/wiki/Object-oriented_programming" title="Object-oriented programming">object-oriented programming</a> and other <a href="/wiki/Programming_paradigm" title="Programming paradigm">programming paradigms</a>. The term was popularized by <a href="/wiki/Robert_Cecil_Martin" title="Robert Cecil Martin">Robert C. Martin</a> and <a href="/wiki/Martin_Fowler" title="Martin Fowler">Martin Fowler</a>.</p>
<p>The term is related to, but different from, the <a href="/wiki/Dependency_inversion_principle" title="Dependency inversion principle">dependency inversion principle</a>, which concerns itself with <a class="mw-redirect" href="/wiki/Dependency_(computer_science)" title="Dependency (computer science)">decoupling dependencies</a> between <a href="/wiki/High-_and_low-level" title="High- and low-level">high-level and low-level</a> <a href="/wiki/Abstraction_layer" title="Abstraction layer">layers</a> through shared <a class="mw-redirect" href="/wiki/Abstraction_(computing)" title="Abstraction (computing)">abstractions</a>. The general concept is also related to <a href="/wiki/Event-driven_programming" title="Event-driven programming">event-driven programming</a> in that it is often implemented using IoC, so that the custom code is commonly only concerned with handling of events, whereas the <a href="/wiki/Event_loop" title="Event loop">event loop</a> and dispatch of events/messages is handled by the framework or the runtime environment.</p>

<h2><span class="mw-headline" id="Overview">Overview</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Inversion_of_control&amp;action=edit&amp;section=1" title="Edit section: Overview">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>As an example, with traditional programming, the <a class="mw-redirect" href="/wiki/Main_function" title="Main function">main function</a> of an application might make function calls into a menu library to display a list of available <a href="/wiki/Command_(computing)" title="Command (computing)">commands</a> and query the user to select one.<sup class="reference" id="cite_ref-FowlerDI_2-0"><a href="#cite_note-FowlerDI-2">[2]</a></sup> The library thus would return the chosen option as the value of the function call, and the main function uses this value to execute the associated command. This style was common in <a class="mw-redirect" href="/wiki/Text_based_interface" title="Text based interface">text based interfaces</a>. For example, an <a href="/wiki/Email_client" title="Email client">email client</a> may show a screen with commands to load new mails, answer the current mail, start a new mail, etc., and the program execution would block until the user presses a key to select a command.</p>
<p>With inversion of control, on the other hand, the program would be written using a <a href="/wiki/Software_framework" title="Software framework">software framework</a> that knows common behavioral and graphical elements, such as <a href="/wiki/Windowing_system" title="Windowing system">windowing systems</a>, menus, controlling the mouse, and so on. The custom code "fills in the blanks" for the framework, such as supplying a table of menu items and registering a code subroutine for each item, but it is the framework that monitors the user's actions and invokes the subroutine when a menu item is selected. In the mail client example, the framework could follow both the keyboard and mouse inputs and call the command invoked by the user by either means, and at the same time monitor the <a href="/wiki/Network_interface" title="Network interface">network interface</a> to find out if new messages arrive and refresh the screen when some network activity is detected. The same framework could be used as the skeleton for a spreadsheet program or a text editor. Conversely, the framework knows nothing about Web browsers, spreadsheets or text editors; implementing their functionality takes custom code.</p>
<p>Inversion of control carries the strong connotation that the reusable code and the problem-specific code are developed independently even though they operate together in an application. <a href="/wiki/Software_framework" title="Software framework">Software frameworks</a>, <a href="/wiki/Callback_(computer_programming)" title="Callback (computer programming)">callbacks</a>, <a href="/wiki/Scheduling_(computing)" title="Scheduling (computing)">schedulers</a>, <a href="/wiki/Event_loop" title="Event loop">event loops</a> and <a href="/wiki/Dependency_injection" title="Dependency injection">dependency injection</a> are examples of <a href="/wiki/Design_pattern" title="Design pattern">design patterns</a> that follow the inversion of control principle, although the term is most commonly used in the context of <a href="/wiki/Object-oriented_programming" title="Object-oriented programming">object-oriented programming</a>.</p>
<p>Inversion of control serves the following design purposes:</p>
<ul>
<li>To <a class="mw-redirect" href="/wiki/Object_decoupling" title="Object decoupling">decouple</a> the execution of a task from implementation.</li>
<li>To focus a module on the task it is designed for.</li>
<li>To free modules from assumptions about how other systems do what they do and instead rely on <a href="/wiki/Design_by_contract" title="Design by contract">contracts</a>.</li>
<li>To prevent <a href="/wiki/Side_effect_(computer_science)" title="Side effect (computer science)">side effects</a> when replacing a module.</li>
</ul>
<p>Inversion of control is sometimes facetiously referred to as the "Hollywood Principle: Don't call us, we'll call you".</p>
<h2><span class="mw-headline" id="Background">Background</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Inversion_of_control&amp;action=edit&amp;section=2" title="Edit section: Background">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Inversion of control is not a new term in computer science. <a href="/wiki/Martin_Fowler" title="Martin Fowler">Martin Fowler</a> traces the etymology of the phrase back to 1988.<sup class="reference" id="cite_ref-FowlerIoC_3-0"><a href="#cite_note-FowlerIoC-3">[3]</a></sup> <a href="/wiki/Dependency_injection" title="Dependency injection">Dependency injection</a> is a specific type of IoC using contextualized lookup.<sup class="reference" id="cite_ref-FowlerDI_2-1"><a href="#cite_note-FowlerDI-2">[2]</a></sup> A <a href="/wiki/Service_locator_pattern" title="Service locator pattern">service locator</a> such as the <a href="/wiki/Java_Naming_and_Directory_Interface" title="Java Naming and Directory Interface">Java Naming and Directory Interface</a> (JNDI) is similar. In an article by Loek Bergman,<sup class="reference" id="cite_ref-4"><a href="#cite_note-4">[4]</a></sup> it is presented as an architectural principle.</p>
<p>In an article by <a href="/wiki/Robert_Cecil_Martin" title="Robert Cecil Martin">Robert C. Martin</a>,<sup class="reference" id="cite_ref-robertmartin_5-0"><a href="#cite_note-robertmartin-5">[5]</a></sup> the dependency inversion principle and abstraction by layering come together. His reason to use the term "inversion" is in comparison with traditional software development methods. He describes the uncoupling of services by the abstraction of layers when he is talking about dependency inversion. The principle is used to find out where system borders are in the design of the abstraction layers.</p>
<h2><span class="mw-headline" id="Description">Description</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Inversion_of_control&amp;action=edit&amp;section=3" title="Edit section: Description">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>In traditional programming, the <a href="/wiki/Control_flow" title="Control flow">flow</a> of the <a href="/wiki/Business_logic" title="Business logic">business logic</a> is determined by objects that are <a class="mw-redirect" href="/wiki/Static_binding" title="Static binding">statically bound</a> to one another. With inversion of control, the flow depends on the object graph that is built up during program execution. Such a dynamic flow is made possible by object interactions being defined through abstractions. This <a href="/wiki/Late_binding" title="Late binding">run-time binding</a> is achieved by mechanisms such as <a href="/wiki/Dependency_injection" title="Dependency injection">dependency injection</a> or a <a href="/wiki/Service_locator_pattern" title="Service locator pattern">service locator</a>. In IoC, the code could also be linked statically during compilation, but finding the code to execute by reading its description from <a href="/wiki/Configuration_file" title="Configuration file">external configuration</a> instead of with a direct reference in the code itself.</p>
<p>In dependency injection, a dependent <a href="/wiki/Object-oriented_programming" title="Object-oriented programming">object</a> or module is coupled to the object it needs at <a href="/wiki/Run_time_(program_lifecycle_phase)" title="Run time (program lifecycle phase)">run time</a>. Which particular object will satisfy the dependency during program execution typically cannot be known at <a href="/wiki/Compile_time" title="Compile time">compile time</a> using <a class="mw-redirect" href="/wiki/Static_code_analysis" title="Static code analysis">static analysis</a>. While described in terms of object interaction here, the principle can apply to other programming methodologies besides <a href="/wiki/Object-oriented_programming" title="Object-oriented programming">object-oriented programming</a>.</p>
<p>In order for the running program to bind objects to one another, the objects must possess compatible <a class="mw-redirect" href="/wiki/Software_interface" title="Software interface">interfaces</a>. For example, class <code>A</code> may delegate behavior to interface <code>I</code> which is implemented by class <code>B</code>; the program instantiates <code>A</code> and <code>B</code>, and then injects <code>B</code> into <code>A</code>.</p>
<h2><span class="mw-headline" id="Implementation_techniques">Implementation techniques</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Inversion_of_control&amp;action=edit&amp;section=4" title="Edit section: Implementation techniques">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>In <a href="/wiki/Object-oriented_programming" title="Object-oriented programming">object-oriented programming</a>, there are several basic techniques to implement inversion of control. These are:</p>
<ul>
<li>Using a <a class="mw-redirect" href="/wiki/Factory_pattern" title="Factory pattern">factory pattern</a></li>
<li>Using a <a href="/wiki/Service_locator_pattern" title="Service locator pattern">service locator pattern</a></li>
<li>Using a <a href="/wiki/Dependency_injection" title="Dependency injection">dependency injection</a>, for example
<ul>
<li>A constructor injection</li>
<li>A parameter injection</li>
<li>A setter injection</li>
<li>An interface injection</li>
</ul>
</li>
<li>Using a contextualized lookup</li>
<li>Using <a class="mw-redirect" href="/wiki/Template_method_design_pattern" title="Template method design pattern">template method design pattern</a></li>
<li>Using <a class="mw-redirect" href="/wiki/Strategy_design_pattern" title="Strategy design pattern">strategy design pattern</a></li>
</ul>
<p>In an original article by Martin Fowler,<sup class="reference" id="cite_ref-6"><a href="#cite_note-6">[6]</a></sup> the first three different techniques are discussed. In a description about inversion of control types,<sup class="reference" id="cite_ref-7"><a href="#cite_note-7">[7]</a></sup> the last one is mentioned. Often the contextualized lookup will be accomplished using a service locator.</p>
<p>More important than the applied technique, however, is the optimization of the purposes.</p>
<h2><span class="mw-headline" id="Examples">Examples</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Inversion_of_control&amp;action=edit&amp;section=5" title="Edit section: Examples">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Most frameworks such as <a href="/wiki/.NET_Framework" title=".NET Framework">.NET</a> or <a href="/wiki/Enterprise_JavaBeans" title="Enterprise JavaBeans">Enterprise JavaBeans</a> display this pattern:</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="kd">public</span> <span class="kd">class</span> <span class="nc">ServerFacade</span> <span class="o">{</span>
   <span class="kd">public</span> <span class="o">&lt;</span><span class="n">K</span><span class="o">,</span> <span class="n">V</span><span class="o">&gt;</span> <span class="n">V</span> <span class="nf">respondToRequest</span><span class="o">(</span><span class="n">K</span> <span class="n">request</span><span class="o">)</span> <span class="o">{</span>
      <span class="k">if</span> <span class="o">(</span><span class="n">businessLayer</span><span class="o">.</span><span class="na">validateRequest</span><span class="o">(</span><span class="n">request</span><span class="o">))</span> <span class="o">{</span>
         <span class="n">DAO</span><span class="o">.</span><span class="na">getData</span><span class="o">(</span><span class="n">request</span><span class="o">);</span>
         <span class="k">return</span> <span class="n">Aspect</span><span class="o">.</span><span class="na">convertData</span><span class="o">(</span><span class="n">request</span><span class="o">);</span>
      <span class="o">}</span>
      <span class="k">return</span> <span class="kc">null</span><span class="o">;</span>
   <span class="o">}</span>
<span class="o">}</span>
</pre></div>
<p>This basic outline in Java gives an example of code following the IoC methodology. It is important, however, that in the <code class="mw-highlight" dir="ltr"><span class="n">ServerFacade</span></code> a lot of assumptions are made about the data returned by the <a href="/wiki/Data_access_object" title="Data access object">data access object</a> (DAO).</p>
<p>Although all these assumptions might be valid at some time, they couple the implementation of the <code class="mw-highlight" dir="ltr"><span class="n">ServerFacade</span></code> to the DAO implementation. Designing the application in the manner of inversion of control would hand over the control completely to the DAO object. The code would then become</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="kd">public</span> <span class="kd">class</span> <span class="nc">ServerFacade</span> <span class="o">{</span>
   <span class="kd">public</span> <span class="o">&lt;</span><span class="n">K</span><span class="o">,</span> <span class="n">V</span><span class="o">&gt;</span> <span class="n">V</span> <span class="nf">respondToRequest</span><span class="o">(</span><span class="n">K</span> <span class="n">request</span><span class="o">,</span> <span class="n">DAO</span> <span class="n">dao</span><span class="o">)</span> <span class="o">{</span>
      <span class="k">return</span> <span class="n">dao</span><span class="o">.</span><span class="na">getData</span><span class="o">(</span><span class="n">request</span><span class="o">);</span>
   <span class="o">}</span>
<span class="o">}</span>
</pre></div>
<p>The example shows that the way the method <code class="mw-highlight" dir="ltr"><span class="n">respondToRequest</span></code> is constructed determines if IoC is used. It is the way that parameters are used that define IoC. This resembles the <a class="mw-redirect" href="/wiki/Message_Passing" title="Message Passing">message-passing</a> style that some object-oriented programming languages use.</p>
