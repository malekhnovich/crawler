<p>In <a href="/wiki/Software_engineering" title="Software engineering">software engineering</a>, <b>behavior-driven development</b> (<b>BDD</b>) is a <a href="/wiki/Software_development_process" title="Software development process">software development process</a> that emerged from <a href="/wiki/Test-driven_development" title="Test-driven development">test-driven development</a> (TDD).<sup class="reference" id="cite_ref-BDD_Def_BehaviourDriven_1-0"><a href="#cite_note-BDD_Def_BehaviourDriven-1">[1]</a></sup><sup class="reference" id="cite_ref-BDD_JW_2-0"><a href="#cite_note-BDD_JW-2">[2]</a></sup><sup class="reference" id="cite_ref-3"><a href="#cite_note-3">[3]</a></sup><sup class="noprint Inline-Template" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Vagueness" title="Wikipedia:Vagueness"><span title="This information is too vague. (May 2015)">vague</span></a></i>]</sup> Behavior-driven development combines the general techniques and principles of TDD with ideas from <a href="/wiki/Domain-driven_design" title="Domain-driven design">domain-driven design</a> and <a href="/wiki/Object-oriented_analysis_and_design" title="Object-oriented analysis and design">object-oriented analysis and design</a> to provide software development and management teams with shared tools and a shared process to collaborate on software development.<sup class="reference" id="cite_ref-BDD_Def_BehaviourDriven_1-1"><a href="#cite_note-BDD_Def_BehaviourDriven-1">[1]</a></sup><sup class="reference" id="cite_ref-BDD_CodeMagazine_4-0"><a href="#cite_note-BDD_CodeMagazine-4">[4]</a></sup></p>
<p>Although BDD is principally an idea about how software development should be managed by both business interests and technical insight, the practice of BDD does assume the use of specialized software tools to support the development process.<sup class="reference" id="cite_ref-BDD_JW_2-1"><a href="#cite_note-BDD_JW-2">[2]</a></sup> Although these tools are often developed specifically for use in BDD projects, they can be seen as specialized forms of the tooling that supports test-driven development. The tools serve to add automation to the <a href="/wiki/Domain-driven_design#Building_blocks" title="Domain-driven design">ubiquitous language</a> that is a central theme of BDD.</p>
<p>BDD is largely facilitated through the use of a simple <a href="/wiki/Domain-specific_language" title="Domain-specific language">domain-specific language</a> (DSL) using natural language constructs (e.g., English-like sentences) that can express the behavior and the expected outcomes. Test scripts have long been a popular application of DSLs with varying degrees of sophistication. BDD is considered an effective technical practice especially when the "problem space" of the business problem to solve is complex.<sup class="reference" id="cite_ref-When_to_adopt_BDD_5-0"><a href="#cite_note-When_to_adopt_BDD-5">[5]</a></sup></p>
<p></p>

<h2><span class="mw-headline" id="History">History</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Behavior-driven_development&amp;action=edit&amp;section=1" title="Edit section: History">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Behavior-driven development is an extension of <a href="/wiki/Test-driven_development" title="Test-driven development">test-driven development</a>:<sup class="reference" id="cite_ref-BDD_Def_BehaviourDriven_1-2"><a href="#cite_note-BDD_Def_BehaviourDriven-1">[1]</a></sup> development that makes use of a simple, domain-specific scripting language. These DSLs convert structured natural language statements into executable tests. The result is a closer relationship to acceptance criteria for a given function and the tests used to validate that functionality. As such it is a natural extension of TDD testing in general.</p>
<p>BDD focuses on:</p>
<ul>
<li>Where to start in the process</li>
<li>What to test and what not to test</li>
<li>How much to test in one go</li>
<li>What to call the tests</li>
<li>How to understand why a test fails</li>
</ul>
<p>At the heart of BDD is a rethinking of the approach to the <a href="/wiki/Unit_testing" title="Unit testing">unit testing</a> and <a href="/wiki/Acceptance_testing" title="Acceptance testing">acceptance testing</a> that naturally arise with these issues. For example, BDD suggests that unit test names be whole sentences starting with a conditional verb ("should" in English for example) and should be written in order of business value. Acceptance tests should be written using the standard agile framework of a <a href="/wiki/User_story" title="User story">user story</a>: "As a [role] I want [feature] so that [benefit]". Acceptance criteria should be written in terms of scenarios and implemented as classes: <a href="/wiki/Given-When-Then" title="Given-When-Then">Given [initial context], when [event occurs], then [ensure some outcomes]</a> .<sup class="reference" id="cite_ref-BDD_Def_BehaviourDriven_1-3"><a href="#cite_note-BDD_Def_BehaviourDriven-1">[1]</a></sup></p>
<p>Starting from this point, many people developed BDD frameworks over a period of years, finally framing it as a communication and collaboration framework for developers, <a href="/wiki/Quality_assurance" title="Quality assurance">QA</a> and non-technical or business participants in a software project.<sup class="reference" id="cite_ref-6"><a href="#cite_note-6">[6]</a></sup><sup class="reference" id="cite_ref-BDD_comm_North_7-0"><a href="#cite_note-BDD_comm_North-7">[7]</a></sup> During the "Agile specifications, BDD and Testing eXchange" in November 2009 in London, Dan North<sup class="reference" id="cite_ref-8"><a href="#cite_note-8">[8]</a></sup> gave the following description of BDD:</p>
<blockquote>
<p>BDD is a second-generation, outside-in, pull-based, multiple-stakeholder, multiple-scale, high-automation, agile methodology. It describes a cycle of interactions with well-defined outputs, resulting in the delivery of working, tested software that matters.</p>
</blockquote>
<p>Dan North created a BDD framework, <a class="new" href="/w/index.php?title=JBehave_(software)&amp;action=edit&amp;redlink=1" title="JBehave (software) (page does not exist)">JBehave</a>,<sup class="reference" id="cite_ref-BDD_Def_BehaviourDriven_1-4"><a href="#cite_note-BDD_Def_BehaviourDriven-1">[1]</a></sup> followed by a story-level BDD framework for Ruby called RBehave<sup class="reference" id="cite_ref-rbehave_9-0"><a href="#cite_note-rbehave-9">[9]</a></sup> which was later integrated into the <a href="/wiki/RSpec" title="RSpec">RSpec</a> project.<sup class="reference" id="cite_ref-rbehave-integ_10-0"><a href="#cite_note-rbehave-integ-10">[10]</a></sup> He also worked with David Chelimsky, Aslak Hellesøy and others to develop RSpec and also to write "The RSpec Book: Behaviour Driven Development with RSpec, Cucumber, and Friends". The first story-based framework in RSpec was later replaced by <a href="/wiki/Cucumber_(software)" title="Cucumber (software)">Cucumber</a> mainly developed by <a class="new" href="/w/index.php?title=Aslak_Helles%C3%B8y&amp;action=edit&amp;redlink=1" title="Aslak Hellesøy (page does not exist)">Aslak Hellesøy</a>. <a href="/wiki/Capybara_(software)" title="Capybara (software)">Capybara (software)</a>, which is a part of the Cucumber testing framework is one such web-based test automation software.</p>
<h2><span class="mw-headline" id="Principles_of_BDD">Principles of BDD</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Behavior-driven_development&amp;action=edit&amp;section=2" title="Edit section: Principles of BDD">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Test-driven development is a software development methodology which essentially states that for each unit of software, a software developer must:</p>
<ul>
<li>define a test set for the unit <i>first</i>;</li>
<li>make the tests fail;</li>
<li>then implement the unit;</li>
<li>finally verify that the implementation of the unit makes the tests succeed.</li>
</ul>
<p>This definition is rather non-specific in that it allows tests in terms of high-level software requirements, low-level technical details or anything in between. One way of looking at BDD therefore, is that it is a continued development of TDD which makes more specific choices than TDD.</p>
<p>Behavior-driven development specifies that tests of any unit of software should be specified in terms of the desired behavior of the unit.<sup class="reference" id="cite_ref-BDD_JW_2-2"><a href="#cite_note-BDD_JW-2">[2]</a></sup><sup class="reference" id="cite_ref-BDD_CodeMagazine_4-1"><a href="#cite_note-BDD_CodeMagazine-4">[4]</a></sup><sup class="reference" id="cite_ref-IntroToBDD_DanNorth_11-0"><a href="#cite_note-IntroToBDD_DanNorth-11">[11]</a></sup> Borrowing from <a href="/wiki/Agile_software_development" title="Agile software development">agile software development</a> the "desired behavior" in this case consists of the requirements set by the business — that is, the desired behavior that has <a href="/wiki/Business_value" title="Business value">business value</a> for whatever entity commissioned the software unit under construction.<sup class="reference" id="cite_ref-BDD_JW_2-3"><a href="#cite_note-BDD_JW-2">[2]</a></sup><sup class="reference" id="cite_ref-IntroToBDD_DanNorth_11-1"><a href="#cite_note-IntroToBDD_DanNorth-11">[11]</a></sup> Within BDD practice, this is referred to as BDD being an "outside-in" activity.<sup class="reference" id="cite_ref-WhatStory_12-0"><a href="#cite_note-WhatStory-12">[12]</a></sup></p>
<h3><span class="mw-headline" id="Behavioral_specifications">Behavioral specifications</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Behavior-driven_development&amp;action=edit&amp;section=3" title="Edit section: Behavioral specifications">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Following this fundamental choice, a second choice made by BDD relates to <i>how</i> the desired behavior should be specified. In this area BDD chooses to use a semi-formal format for behavioral specification which is borrowed from user story specifications from the field of <a href="/wiki/Object-oriented_analysis_and_design" title="Object-oriented analysis and design">object-oriented analysis and design</a>. The scenario aspect of this format may be regarded as an application of <a href="/wiki/Hoare_logic" title="Hoare logic">Hoare logic</a> to behavioral specification of software units using the <a href="/wiki/Domain-specific_language" title="Domain-specific language">Domain Language</a> of the situation.</p>
<p>BDD specifies that business analysts and developers should collaborate in this area and should specify behavior in terms of user stories, which are each explicitly written down in a dedicated document.<sup class="reference" id="cite_ref-IntroToBDD_DanNorth_11-2"><a href="#cite_note-IntroToBDD_DanNorth-11">[11]</a></sup><sup class="reference" id="cite_ref-WhatStory_12-1"><a href="#cite_note-WhatStory-12">[12]</a></sup> Each user story should, in some way, follow the following structure:<sup class="reference" id="cite_ref-BDD_JW_2-4"><a href="#cite_note-BDD_JW-2">[2]</a></sup><sup class="reference" id="cite_ref-WhatStory_12-2"><a href="#cite_note-WhatStory-12">[12]</a></sup></p>
<dl>
<dt><b>Title: The story should have a clear, explicit title.</b></dt>
<dt>Narrative</dt>
<dd>A short, introductory section that specifies
<ul>
<li>who (which business or project role) is the driver or primary stakeholder of the story (the actor who derives business benefit from the story)</li>
<li>what effect the stakeholder wants the story to have</li>
<li>what business value the stakeholder will derive from this effect</li>
</ul>
</dd>
</dl>
<dl>
<dt>Acceptance criteria or scenarios</dt>
<dd>a description of each specific case of the narrative. Such a scenario has the following structure:
<ul>
<li>It starts by specifying the initial condition that is assumed to be true at the beginning of the scenario. This may consist of a single clause, or several.</li>
<li>It then states which event triggers the start of the scenario.</li>
<li>Finally, it states the expected outcome, in one or more clauses.</li>
</ul>
</dd>
</dl>
<p>BDD does not have any formal requirements for exactly <b>how</b> these user stories must be written down, but it does insist that each team using BDD come up with a simple, standardized format for writing down the user stories which includes the elements listed above.<sup class="reference" id="cite_ref-BDD_JW_2-5"><a href="#cite_note-BDD_JW-2">[2]</a></sup><sup class="reference" id="cite_ref-WhatStory_12-3"><a href="#cite_note-WhatStory-12">[12]</a></sup> However, in 2007 Dan North suggested a template for a textual format which has found wide following in different BDD software tools.<sup class="reference" id="cite_ref-WhatStory_12-4"><a href="#cite_note-WhatStory-12">[12]</a></sup> A very brief example of this format might look like this: <span id="template_for_a_textual_format_exmple"></span></p>
<pre>
 <b>Story</b>: Returns go to stock

<b>In order to</b> keep track of stock
<b>As a</b> store owner
<b>I want to</b> add items back to stock when they're returned.

<b>Scenario 1:</b> Refunded items should be returned to stock
<b>Given</b> that a customer previously bought a black sweater from me
<b>And</b> I have three black sweaters in stock.
<b>When</b> he returns the black sweater for a refund
<b>Then</b> I should have four black sweaters in stock.

<b>Scenario 2:</b> Replaced items should be returned to stock
<b>Given</b> that a customer previously bought a blue garment from me
<b>And</b> I have two blue garments in stock
<b>And</b> three black garments in stock.
<b>When</b> he returns the blue garment for a replacement in black
<b>Then</b> I should have three blue garments in stock
<b>And</b> two black garments in stock.
</pre>
<p>The scenarios are ideally phrased declaratively rather than imperatively — in the business language, with no reference to elements of the UI through which the interactions take place.<sup class="reference" id="cite_ref-declarative_13-0"><a href="#cite_note-declarative-13">[13]</a></sup></p>
<p>This format is referred to as the Gherkin language, which has a syntax similar to the above example. The term <i>Gherkin</i>, however, is specific to the Cucumber, JBehave and <a href="/wiki/Behat_(computer_science)" title="Behat (computer science)">Behat</a> software tools.<sup class="reference" id="cite_ref-Gherkin_14-0"><a href="#cite_note-Gherkin-14">[14]</a></sup><sup class="reference" id="cite_ref-JBehave_15-0"><a href="#cite_note-JBehave-15">[15]</a></sup><sup class="reference" id="cite_ref-Behat_16-0"><a href="#cite_note-Behat-16">[16]</a></sup></p>
<h3><span class="mw-headline" id="Specification_as_a_ubiquitous_language">Specification as a ubiquitous language</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Behavior-driven_development&amp;action=edit&amp;section=4" title="Edit section: Specification as a ubiquitous language">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Behavior-driven development borrows the concept of the <i>ubiquitous language</i> from <a class="mw-redirect" href="/wiki/Domain_driven_design" title="Domain driven design">domain driven design</a>.<sup class="reference" id="cite_ref-BDD_JW_2-6"><a href="#cite_note-BDD_JW-2">[2]</a></sup><sup class="reference" id="cite_ref-BDD_CodeMagazine_4-2"><a href="#cite_note-BDD_CodeMagazine-4">[4]</a></sup> A ubiquitous language is a (semi-)formal language that is shared by all members of a software development team — both software developers and non-technical personnel.<sup class="reference" id="cite_ref-DDD_Evans_17-0"><a href="#cite_note-DDD_Evans-17">[17]</a></sup> The language in question is both used and developed by all team members as a common means of discussing the domain of the software in question.<sup class="reference" id="cite_ref-DDD_Evans_17-1"><a href="#cite_note-DDD_Evans-17">[17]</a></sup> In this way BDD becomes a vehicle for communication between all the different roles in a software project.<sup class="reference" id="cite_ref-BDD_JW_2-7"><a href="#cite_note-BDD_JW-2">[2]</a></sup><sup class="reference" id="cite_ref-BDD_comm_North_7-1"><a href="#cite_note-BDD_comm_North-7">[7]</a></sup></p>
<p>A common risk with software development includes communication breakdowns between Developers and Business Stakeholders.<sup class="reference" id="cite_ref-BDD_comms_18-0"><a href="#cite_note-BDD_comms-18">[18]</a></sup> BDD uses the specification of desired behavior as a ubiquitous language for the project team members. This is the reason that BDD insists on a semi-formal language for behavioral specification: some formality is a requirement for being a ubiquitous language.<sup class="reference" id="cite_ref-BDD_JW_2-8"><a href="#cite_note-BDD_JW-2">[2]</a></sup> In addition, having such a ubiquitous language creates a domain model of specifications, so that specifications may be reasoned about formally.<sup class="reference" id="cite_ref-BDD_CodeProj_19-0"><a href="#cite_note-BDD_CodeProj-19">[19]</a></sup> This model is also the basis for the different BDD-supporting software tools that are available.</p>
<p>The example given above establishes a user story for a software system under development. This user story identifies a stakeholder, a business effect and a business value. It also describes several scenarios, each with a precondition, trigger and expected outcome. Each of these parts is exactly identified by the more formal part of the language (the term <b>Given</b> might be considered a <a class="mw-redirect" href="/wiki/Keyword_(computer_programming)" title="Keyword (computer programming)">keyword</a>, for example) and may therefore be processed in some way by a tool that understands the formal parts of the ubiquitous language.</p>
<h2><span class="mw-headline" id="Specialized_tooling_support">Specialized tooling support</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Behavior-driven_development&amp;action=edit&amp;section=5" title="Edit section: Specialized tooling support">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Much like test-driven design practice, behavior-driven development assumes the use of specialized support tooling in a project. Inasmuch as BDD is, in many respects, a more specific version of TDD, the tooling for BDD is similar to that for TDD, but makes more demands on the developer than basic TDD tooling.<sup class="noprint Inline-Template" style="margin-left:0.1em; white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Please_clarify" title="Wikipedia:Please clarify"><span title="The text near this tag may need clarification or removal of jargon. (September 2013)">clarification needed</span></a></i>]</sup><sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (September 2013)">citation needed</span></a></i>]</sup></p>
<h3><span class="mw-headline" id="Tooling_principles">Tooling principles</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Behavior-driven_development&amp;action=edit&amp;section=6" title="Edit section: Tooling principles">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>In principle a BDD support tool is a testing framework for software, much like the tools that support TDD. However, where TDD tools tend to be quite free-format in what is allowed for specifying tests, BDD tools are linked to the definition of the ubiquitous language discussed earlier.</p>
<p>As discussed, the ubiquitous language allows business analysts to write down behavioral requirements in a way that will also be understood by developers. The principle of BDD support tooling is to make these same requirements documents directly executable as a collection of tests. If this cannot be achieved because of reasons related to the technical tool that enables the execution of the specifications, then either the style of writing the behavioral requirements must be altered or the tool must be changed.<sup class="reference" id="cite_ref-BDD_Enterprise_20-0"><a href="#cite_note-BDD_Enterprise-20">[20]</a></sup> The exact implementation of behavioral requirements varies per tool, but agile practice has come up with the following general process:</p>
<ul>
<li>The tooling reads a specification document.</li>
<li>The tooling directly understands completely formal parts of the ubiquitous language (such as the <b>Given</b> keyword in the example above). Based on this, the tool breaks each scenario up into meaningful clauses.</li>
<li>Each individual clause in a scenario is transformed into some sort of parameter for a test for the user story. This part requires project-specific work by the software developers.</li>
<li>The framework then executes the test for each scenario, with the parameters from that scenario.</li>
</ul>
<p>Dan North has developed a number of frameworks that support BDD (including JBehave and <a class="new" href="/w/index.php?title=RBehave&amp;action=edit&amp;redlink=1" title="RBehave (page does not exist)">RBehave</a>), whose operation is based on the template that he suggested for recording user stories.<sup class="reference" id="cite_ref-BDD_JW_2-9"><a href="#cite_note-BDD_JW-2">[2]</a></sup> These tools use a textual description for use cases and several other tools (such as CBehave) have followed suit. However, this format is not required and so there are other tools that use other formats as well. For example, <a class="mw-redirect" href="/wiki/Fitnesse" title="Fitnesse">Fitnesse</a> (which is built around <a href="/wiki/Decision_table" title="Decision table">decision tables</a>), has also been used to roll out BDD.<sup class="reference" id="cite_ref-FitNesse_BDD_21-0"><a href="#cite_note-FitNesse_BDD-21">[21]</a></sup></p>
<h3><span class="mw-headline" id="Tooling_examples">Tooling examples</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Behavior-driven_development&amp;action=edit&amp;section=7" title="Edit section: Tooling examples">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>There are several different examples of BDD software tools in use in projects today, for different platforms and programming languages.</p>
<p>Possibly the most well-known is JBehave, which was developed by Dan North. The following is an example taken from that project:<sup class="reference" id="cite_ref-JBehave_15-1"><a href="#cite_note-JBehave-15">[15]</a></sup></p>
<p>Consider an implementation of the <a href="/wiki/Conway%27s_Game_of_Life" title="Conway's Game of Life">Game of Life</a>. A domain expert (or business analyst) might want to specify what should happen when someone is setting up a starting configuration of the game grid. To do this, he might want to give an example of a number of steps taken by a person who is toggling cells. Skipping over the narrative part, he might do this by writing up the following scenario into a plain text document (which is the type of input document that JBehave reads):</p>
<pre>
<b>Given</b> a 5 by 5 game
<b>When</b> I toggle the cell at (3, 2)
<b>Then</b> the grid should look like
.....
.....
.....
..X..
.....
<b>When</b> I toggle the cell at (3, 1)
<b>Then</b> the grid should look like
.....
.....
.....
..X..
..X..
<b>When</b> I toggle the cell at (3, 2)
<b>Then</b> the grid should look like
.....
.....
.....
.....
..X..
</pre>
<p>The bold print is not actually part of the input; it is included here to show which words are recognized as formal language. JBehave recognizes the terms <i>Given</i> (as a precondition which defines the start of a scenario), <i>When</i> (as an event trigger) and <i>Then</i> (as a postcondition which must be verified as the outcome of the action that follows the trigger). Based on this, JBehave is capable of reading the text file containing the scenario and <a href="/wiki/Parsing" title="Parsing">parsing</a> it into clauses (a set-up clause and then three event triggers with verifiable conditions). JBehave then takes these clauses and passes them on to code that is capable of setting a test, responding to the event triggers and verifying the outcome. This code must be written by the developers in the project team (in <a href="/wiki/Java_(programming_language)" title="Java (programming language)">Java</a>, because that is the platform JBehave is based on). In this case, the code might look like this:</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="kd">private</span> <span class="n">Game</span> <span class="n">game</span><span class="o">;</span>
<span class="kd">private</span> <span class="n">StringRenderer</span> <span class="n">renderer</span><span class="o">;</span>

<span class="nd">@Given</span><span class="o">(</span><span class="s">"a $width by $height game"</span><span class="o">)</span>
<span class="kd">public</span> <span class="kt">void</span> <span class="nf">theGameIsRunning</span><span class="o">(</span><span class="kt">int</span> <span class="n">width</span><span class="o">,</span> <span class="kt">int</span> <span class="n">height</span><span class="o">)</span> <span class="o">{</span>
    <span class="n">game</span> <span class="o">=</span> <span class="k">new</span> <span class="n">Game</span><span class="o">(</span><span class="n">width</span><span class="o">,</span> <span class="n">height</span><span class="o">);</span>
    <span class="n">renderer</span> <span class="o">=</span> <span class="k">new</span> <span class="n">StringRenderer</span><span class="o">();</span>
    <span class="n">game</span><span class="o">.</span><span class="na">setObserver</span><span class="o">(</span><span class="n">renderer</span><span class="o">);</span>
<span class="o">}</span>
    
<span class="nd">@When</span><span class="o">(</span><span class="s">"I toggle the cell at ($column, $row)"</span><span class="o">)</span>
<span class="kd">public</span> <span class="kt">void</span> <span class="nf">iToggleTheCellAt</span><span class="o">(</span><span class="kt">int</span> <span class="n">column</span><span class="o">,</span> <span class="kt">int</span> <span class="n">row</span><span class="o">)</span> <span class="o">{</span>
    <span class="n">game</span><span class="o">.</span><span class="na">toggleCellAt</span><span class="o">(</span><span class="n">column</span><span class="o">,</span> <span class="n">row</span><span class="o">);</span>
<span class="o">}</span>

<span class="nd">@Then</span><span class="o">(</span><span class="s">"the grid should look like $grid"</span><span class="o">)</span>
<span class="kd">public</span> <span class="kt">void</span> <span class="nf">theGridShouldLookLike</span><span class="o">(</span><span class="n">String</span> <span class="n">grid</span><span class="o">)</span> <span class="o">{</span>
    <span class="n">assertThat</span><span class="o">(</span><span class="n">renderer</span><span class="o">.</span><span class="na">asString</span><span class="o">(),</span> <span class="n">equalTo</span><span class="o">(</span><span class="n">grid</span><span class="o">));</span>
<span class="o">}</span>
</pre></div>
<p>The code has a method for every type of clause in a scenario. JBehave will identify which method goes with which clause through the use of <a href="/wiki/Annotation" title="Annotation">annotations</a> and will call each method in order while running through the scenario. The text in each clause in the scenario is expected to match the template text given in the code for that clause (for example, a <b>Given</b> in a scenario is expected to be followed by a clause of the form "a X by Y game"). JBehave supports the matching of actual clauses to templates and has built-in support for picking terms out of the template and passing them to methods in the test code as parameters. The test code provides an implementation for each clause type in a scenario which interacts with the code that is being tested and performs an actual test based on the scenario. In this case:</p>
<ul>
<li>The <code>theGameIsRunning</code> method reacts to a <b>Given</b> clause by setting up the initial game grid.</li>
<li>The <code>iToggleTheCellAt</code> method reacts to a <b>When</b> clause by firing off the toggle event described in the clause.</li>
<li>The <code>theGridShouldLookLike</code> method reacts to a <b>Then</b> clause by comparing the actual state of the game grid to the expected state from the scenario.</li>
</ul>
<p>The primary function of this code is to be a bridge between a text file with a story and the actual code being tested. Note that the test code has access to the code being tested (in this case an instance of <code>Game</code>) and is very simple in nature. The test code has to be simple, otherwise a developer would end up having to write tests for his tests.</p>
<p>Finally, in order to run the tests, JBehave requires some plumbing code that identifies the text files which contain scenarios and which inject dependencies (like instances of <code>Game</code>) into the test code. This plumbing code is not illustrated here, since it is a technical requirement of JBehave and does not relate directly to the principle of BDD-style testing.</p>
<h2><span class="mw-headline" id="Story_versus_specification">Story versus specification</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Behavior-driven_development&amp;action=edit&amp;section=8" title="Edit section: Story versus specification">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>A separate subcategory of behavior-driven development is formed by tools that use specifications as an input language rather than user stories. An example of this style is the <a href="/wiki/RSpec" title="RSpec">RSpec</a> tool that was also developed by Dan North. Specification tools don't use user stories as an input format for test scenarios but rather use functional specifications for units that are being tested. These specifications often have a more technical nature than user stories and are usually less convenient for communication with business personnel than are user stories.<sup class="reference" id="cite_ref-BDD_JW_2-10"><a href="#cite_note-BDD_JW-2">[2]</a></sup><sup class="reference" id="cite_ref-BDD_SPEC_OSHEROVE_22-0"><a href="#cite_note-BDD_SPEC_OSHEROVE-22">[22]</a></sup> An example of a specification for a <a href="/wiki/Stack_(abstract_data_type)" title="Stack (abstract data type)">stack</a> might look like this:</p>
<pre>
<b>Specification:</b> Stack

<b>When</b> a new stack is created
<b>Then</b> it is empty

<b>When</b> an element is added to the stack
<b>Then</b> that element is at the top of the stack

<b>When</b> a stack has N elements 
<b>And</b> element E is on top of the stack
<b>Then</b> a pop operation returns E
<b>And</b> the new size of the stack is N-1
</pre>
<p>Such a specification may exactly specify the behavior of the component being tested, but is less meaningful to a business user. As a result, specification-based testing is seen in BDD practice as a complement to story-based testing and operates at a lower level. Specification testing is often seen as a replacement for free-format <a href="/wiki/Unit_testing" title="Unit testing">unit testing</a>.<sup class="reference" id="cite_ref-BDD_SPEC_OSHEROVE_22-1"><a href="#cite_note-BDD_SPEC_OSHEROVE-22">[22]</a></sup></p>
<p>Specification testing tools like RSpec and JDave are somewhat different in nature from tools like JBehave. Since they are seen as alternatives to basic unit testing tools like <a href="/wiki/JUnit" title="JUnit">JUnit</a>, these tools tend to favor forgoing the separation of story and testing code and prefer embedding the specification directly in the test code instead. For example, an RSpec test for a <a class="mw-redirect" href="/wiki/Hashtable" title="Hashtable">hashtable</a> might look like this:<sup class="reference" id="cite_ref-RSpec_Hash_23-0"><a href="#cite_note-RSpec_Hash-23">[23]</a></sup></p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="n">describe</span> <span class="no">Hash</span> <span class="k">do</span>
  <span class="n">let</span><span class="p">(</span><span class="ss">:hash</span><span class="p">)</span> <span class="p">{</span> <span class="no">Hash</span><span class="o">[</span><span class="ss">:hello</span><span class="p">,</span> <span class="s1">'world'</span><span class="o">]</span> <span class="p">}</span>

  <span class="n">it</span> <span class="p">{</span> <span class="n">expect</span><span class="p">(</span><span class="no">Hash</span><span class="o">.</span><span class="n">new</span><span class="p">)</span><span class="o">.</span><span class="n">to</span> <span class="n">eq</span><span class="p">({})</span> <span class="p">}</span>

  <span class="n">it</span> <span class="s2">"hashes the correct information in a key"</span> <span class="k">do</span>
    <span class="n">expect</span><span class="p">(</span><span class="nb">hash</span><span class="o">[</span><span class="ss">:hello</span><span class="o">]</span><span class="p">)</span><span class="o">.</span><span class="n">to</span> <span class="n">eq</span><span class="p">(</span><span class="s1">'world'</span><span class="p">)</span>
  <span class="k">end</span>

  <span class="n">it</span> <span class="s1">'includes key'</span> <span class="k">do</span>
    <span class="nb">hash</span><span class="o">.</span><span class="n">keys</span><span class="o">.</span><span class="n">include?</span><span class="p">(</span><span class="ss">:hello</span><span class="p">)</span><span class="o">.</span><span class="n">should</span> <span class="n">be</span> <span class="kp">true</span>
  <span class="k">end</span>
<span class="k">end</span>
</pre></div>
<p>This example shows a specification in readable language embedded in executable code. In this case a choice of the tool is to formalize the specification language into the language of the test code by adding methods named <code>it</code> and <code>should</code>. Also there is the concept of a specification precondition – the <code>before</code> section establishes the preconditions that the specification is based on.</p>
<p>The result of test will be:</p>
<pre>
 Hash
   should eq {}
   includes key
   hashes the correct information in a key
