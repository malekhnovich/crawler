<h1 class="firstHeading" id="firstHeading" lang="en">Chain-of-responsibility pattern</h1>
<div class="mw-body-content" id="bodyContent">
<div id="siteSub">From Wikipedia, the free encyclopedia</div>
<div id="contentSub"><span class="mw-redirectedfrom">  (Redirected from <a class="mw-redirect" href="/w/index.php?title=Chain_of_responsibility_pattern&amp;redirect=no" title="Chain of responsibility pattern">Chain of responsibility pattern</a>)</span></div>
<div class="mw-jump" id="jump-to-nav">
					Jump to:					<a href="#mw-head">navigation</a>, 					<a href="#p-search">search</a>
</div>
<div class="mw-content-ltr" dir="ltr" id="mw-content-text" lang="en"><p>In <a href="/wiki/Object-oriented_design" title="Object-oriented design">object-oriented design</a>, the <b>chain-of-responsibility pattern</b> is a <a class="mw-redirect" href="/wiki/Design_pattern_(computer_science)" title="Design pattern (computer science)">design pattern</a> consisting of a source of <a href="/wiki/Command_pattern" title="Command pattern">command objects</a> and a series of <b>processing objects</b>.<sup class="reference" id="cite_ref-1"><a href="#cite_note-1">[1]</a></sup> Each processing object contains logic that defines the types of command objects that it can handle; the rest are passed to the next processing object in the chain. A mechanism also exists for adding new processing objects to the end of this chain.</p>
<p>In a variation of the standard chain-of-responsibility model, some handlers may act as <a href="/wiki/Dynamic_dispatch" title="Dynamic dispatch">dispatchers</a>, capable of sending commands out in a variety of directions, forming a <i>tree of responsibility</i>. In some cases, this can occur recursively, with processing objects calling higher-up processing objects with commands that attempt to solve some smaller part of the problem; in this case recursion continues until the command is processed, or the entire tree has been explored. An <a href="/wiki/XML" title="XML">XML</a> <a href="/wiki/Interpreter_(computing)" title="Interpreter (computing)">interpreter</a> might work in this manner.</p>
<p>This pattern promotes the idea of <a href="/wiki/Loose_coupling" title="Loose coupling">loose coupling</a>.</p>
<p></p>

<h3><span class="mw-headline" id="Java_example">Java example</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Chain-of-responsibility_pattern&amp;action=edit&amp;section=2" title="Edit section: Java example">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Below is an example of this pattern in Java. In this example we have different roles, each having a fixed purchasing limit and a successor. Every time a user in a role receives a purchase request that exceeds his or her limit, the request is passed to his or her successor.</p>
<p>The PurchasePower abstract class with the abstract method processRequest.</p>

<pre>

    <span class="kd">protected</span> <span class="n">PurchasePower</span> <span class="n">successor</span><span class="o">;</span>

    <span class="kd">abstract</span> <span class="kd">protected</span> <span class="kt">double</span> <span class="nf">getAllowable</span><span class="o">();</span>

    <span class="kd">public</span> <span class="kt">void</span> <span class="nf">setSuccessor</span><span class="o">(</span><span class="n">PurchasePower</span> <span class="n">successor</span><span class="o">)</span> <span class="o">{</span>

<p>In the following usage example, the successors are set as follows: Manager -&gt; Director -&gt; Vice President -&gt; President</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>

    <span class="kd">public</span> <span class="kd">static</span> <span class="kt">void</span> <span class="nf">main</span><span class="o">(</span><span class="n">String</span><span class="o">[]</span> <span class="n">args</span><span class="o">)</span> <span class="o">{</span>

        <span class="n">DirectorPPower</span> <span class="n">director</span> <span class="o">=</span> <span class="k">new</span> <span class="n">DirectorPPower</span><span class="o">();</span>

        <span class="n">PresidentPPower</span> <span class="n">president</span> <span class="o">=</span> <span class="k">new</span> <span class="n">PresidentPPower</span><span class="o">();</span>
        <span class="n">manager</span><span class="o">.</span><span class="na">setSuccessor</span><span class="o">(</span><span class="n">director</span><span class="o">);</span>

        <span class="n">vp</span><span class="o">.</span><span class="na">setSuccessor</span><span class="o">(</span><span class="n">president</span><span class="o">);</span>

        <span class="c1">// Press Ctrl+C to end.</span>
        <span class="k">try</span> <span class="o">{</span>

                <span class="n">System</span><span class="o">.</span><span class="na">out</span><span class="o">.</span><span class="na">println</span><span class="o">(</span><span class="s">"Enter the amount to check who should approve your expenditure."</span><span class="o">);</span>

                <span class="kt">double</span> <span class="n">d</span> <span class="o">=</span> <span class="n">Double</span><span class="o">.</span><span class="na">parseDouble</span><span class="o">(</span><span class="k">new</span> <span class="n">BufferedReader</span><span class="o">(</span><span class="k">new</span> <span class="n">InputStreamReader</span><span class="o">(</span><span class="n">System</span><span class="o">.</span><span class="na">in</span><span class="o">)).</span><span class="na">readLine</span><span class="o">());</span>
                <span class="n">manager</span><span class="o">.</span><span class="na">processRequest</span><span class="o">(</span><span class="k">new</span> <span class="n">PurchaseRequest</span><span class="o">(</span><span class="n">d</span><span class="o">,</span> <span class="s">"General"</span><span class="o">));</span>
           <span class="o">}</span>
        <span class="o">}</span> <span class="k">catch</span><span class="o">(</span><span class="n">Exception</span> <span class="n">e</span><span class="o">)</span> <span class="o">{</span>
            <span class="n">System</span><span class="o">.</span><span class="na">exit</span><span class="o">(</span><span class="mi">1</span><span class="o">);</span>
        <span class="o">}</span>
    <span class="o">}</span>
<span class="o">}</span>

<h3><span class="mw-headline" id="Cocoa_and_Cocoa_Touch">Cocoa and Cocoa Touch</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Chain-of-responsibility_pattern&amp;action=edit&amp;section=5" title="Edit section: Cocoa and Cocoa Touch">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>The <a href="/wiki/Cocoa_(API)" title="Cocoa (API)">Cocoa</a> and <a href="/wiki/Cocoa_Touch" title="Cocoa Touch">Cocoa Touch</a> frameworks, used for <a class="mw-redirect" href="/wiki/OS_X" title="OS X">OS X</a> and <a href="/wiki/IOS" title="IOS">iOS</a> applications respectively, actively use the chain-of-responsibility pattern for handling events. Objects that participate in the chain are called <i>responder</i> objects, inheriting from the <code>NSResponder</code> (OS X)/<code>UIResponder</code> (iOS) class. All view objects (<code>NSView</code>/<code>UIView</code>), view controller objects (<code>NSViewController</code>/<code>UIViewController</code>), window objects (<code>NSWindow</code>/<code>UIWindow</code>), and the application object (<code>NSApplication</code>/<code>UIApplication</code>) are responder objects.</p>
<p>Typically, when a view receives an event which it can't handle, it dispatches it to its superview until it reaches the view controller or window object. If the window can't handle the event, the event is dispatched to the application object, which is the last object in the chain. For example:</p>
<ul>
<li>On OS X, moving a textured window with the mouse can be done from any location (not just the title bar), unless on that location there's a view which handles dragging events, like slider controls. If no such view (or superview) is there, dragging events are sent up the chain to the window which does handle the dragging event.</li>
<li>On iOS, it's typical to handle view events in the view controller which manages the view hierarchy, instead of subclassing the view itself. Since a view controller lies in the responder chain after all of its managed subviews, it can intercept any view events and handle them.</li>
</ul>
