<h1 class="firstHeading" id="firstHeading" lang="en">Object (computer science)</h1>
<div class="mw-body-content" id="bodyContent">
<div id="siteSub">From Wikipedia, the free encyclopedia</div>
<div id="contentSub"></div>
<div class="mw-jump" id="jump-to-nav">
					Jump to:					<a href="#mw-head">navigation</a>, 					<a href="#p-search">search</a>
</div>
<div class="mw-content-ltr" dir="ltr" id="mw-content-text" lang="en"><p>In <a href="/wiki/Computer_science" title="Computer science">computer science</a>, an <b>object</b> can be a <a href="/wiki/Variable_(computer_science)" title="Variable (computer science)">variable</a>, a <a href="/wiki/Data_structure" title="Data structure">data structure</a>, or a <a href="/wiki/Subroutine" title="Subroutine">function</a> or a <a href="/wiki/Method_(computer_programming)" title="Method (computer programming)">method</a>, and as such, is a <a href="/wiki/Memory_address" title="Memory address">location in memory</a> having a <a href="/wiki/Value_(computer_science)" title="Value (computer science)">value</a> and possibly referenced by an <a class="mw-redirect" href="/wiki/Identifier_(computer_programming)" title="Identifier (computer programming)">identifier</a>.</p>
<p>In the <a href="/wiki/Class-based_programming" title="Class-based programming">class-based</a> <a href="/wiki/Object-oriented_programming" title="Object-oriented programming">object-oriented programming</a> paradigm, "object" refers to a particular <a href="/wiki/Instance_(computer_science)" title="Instance (computer science)">instance</a> of a <a class="mw-redirect" href="/wiki/Class_(computer_science)" title="Class (computer science)">class</a> where the object can be a combination of variables, functions, and data structures.</p>
<p>In <a href="/wiki/Relational_model" title="Relational model">relational</a> <a href="/wiki/Database" title="Database">database</a> management, an object can be a table or column, or an association between data and a database entity (such as relating a person's age to a specific person).<sup class="reference" id="cite_ref-Oppel_1-0"><a href="#cite_note-Oppel-1">[1]</a></sup></p>
<p></p>

<h2><span class="mw-headline" id="Object-based_languages">Object-based languages</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Object_(computer_science)&amp;action=edit&amp;section=1" title="Edit section: Object-based languages">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<div class="hatnote" role="note">Main article: <a class="mw-redirect" href="/wiki/Object-based_languages" title="Object-based languages">Object-based languages</a></div>
<p>An important distinction in programming languages is the difference between an object-oriented language and an object-based language. A language is usually considered object-based if it includes the basic capabilities for an object: identity, properties, and attributes. A language is considered object-oriented if it is object-based and also has the capability of <a href="/wiki/Polymorphism_(computer_science)" title="Polymorphism (computer science)">polymorphism</a> and inheritance. Polymorphism refers to the ability to overload the name of a function with multiple behaviors based on which object(s) are passed to it. Conventional message passing discriminates only on the first object and considers that to be "sending a message" to that object. However, some OOP languages such as <a href="/wiki/Flavors_(programming_language)" title="Flavors (programming language)">Flavors</a> and the <a href="/wiki/Common_Lisp_Object_System" title="Common Lisp Object System">Common Lisp Object System</a> (CLOS) enable discriminating on more than the first parameter of the function.<sup class="reference" id="cite_ref-2"><a href="#cite_note-2">[2]</a></sup> Inheritance is the ability to subclass an object class, to create a new class that is a subclass of an existing one and inherits all the data constraints and behaviors of its parents but also adds new and/or changes one or more of them.<sup class="reference" id="cite_ref-3"><a href="#cite_note-3">[3]</a></sup><sup class="reference" id="cite_ref-4"><a href="#cite_note-4">[4]</a></sup></p>
<h2><span class="mw-headline" id="Object-oriented_programming">Object-oriented programming</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Object_(computer_science)&amp;action=edit&amp;section=2" title="Edit section: Object-oriented programming">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<div class="hatnote" role="note">Main article: <a href="/wiki/Object-oriented_programming" title="Object-oriented programming">Object-oriented programming</a></div>
<p>Object-oriented programming is an approach to designing modular reusable software systems. The object-oriented approach is an evolution of good design practices that go back to the very beginning of computer programming. Object-orientation is simply the logical extension of older techniques such as <a href="/wiki/Structured_programming" title="Structured programming">structured programming</a> and <a class="mw-redirect" href="/wiki/Abstract_data_types" title="Abstract data types">abstract data types</a>. An object is an abstract data type with the addition of <a href="/wiki/Polymorphism_(computer_science)" title="Polymorphism (computer science)">polymorphism</a> and <a href="/wiki/Inheritance_(object-oriented_programming)" title="Inheritance (object-oriented programming)">inheritance</a>.</p>
<p>Rather than structure programs as code and data, an object-oriented system integrates the two using the concept of an "object". An object has state (data) and behavior (code). Objects can correspond to things found in the real world. So for example, a graphics program will have objects such as <i>circle</i>, <i>square</i>, <i>menu</i>. An online shopping system will have objects such as <i>shopping cart, customer, product</i>. The shopping system will support behaviors such as <i>place order</i>, <i>make payment</i>, and <i>offer discount</i>. The objects are designed as class hierarchies. So for example with the shopping system there might be high level classes such as <i>electronics product</i>, <i>kitchen product</i>, and <i>book</i>. There may be further refinements for example under <i>electronic products</i>: <i>CD Player, DVD player</i>, etc. These classes and subclasses correspond to sets and subsets in mathematical logic.<sup class="reference" id="cite_ref-5"><a href="#cite_note-5">[5]</a></sup><sup class="reference" id="cite_ref-6"><a href="#cite_note-6">[6]</a></sup></p>
<h2><span class="mw-headline" id="Specialized_objects">Specialized objects</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Object_(computer_science)&amp;action=edit&amp;section=3" title="Edit section: Specialized objects">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>An important concept for objects is the <a class="mw-redirect" href="/wiki/Design_pattern_(computer_science)" title="Design pattern (computer science)">design pattern</a>. A design pattern provides a reusable template to address a common problem. The following object descriptions are examples of some of the most common design patterns for objects.<sup class="reference" id="cite_ref-R.C.Martin_7-0"><a href="#cite_note-R.C.Martin-7">[7]</a></sup></p>
<ul>
<li><a href="/wiki/Function_object" title="Function object">Function object</a>: an object with a single method (in C++, this method would be the function operator, "operator()") that acts much like a function (like a C/C++ pointer to a function).</li>
<li><a href="/wiki/Immutable_object" title="Immutable object">Immutable object</a>: an object set up with a fixed state at creation time and which does not change afterward.</li>
<li><a class="mw-redirect" href="/wiki/First-class_object" title="First-class object">First-class object</a>: an object that can be used without restriction.</li>
<li><a class="mw-redirect" href="/wiki/Container_(data_structure)" title="Container (data structure)">Container object</a>: an object that can contain other objects.</li>
<li><a class="mw-redirect" href="/wiki/Factory_object" title="Factory object">Factory object</a>: an object whose purpose is to create other objects.</li>
<li><a href="/wiki/Metaobject" title="Metaobject">Metaobject</a>: an object from which other objects can be created (compare with a <a class="mw-redirect" href="/wiki/Class_(computer_science)" title="Class (computer science)">class</a>, which is not necessarily an object).</li>
<li><a href="/wiki/Prototype_pattern" title="Prototype pattern">Prototype object</a>: a specialized metaobject from which other objects can be created by copying</li>
<li><a href="/wiki/God_object" title="God object">God object</a>: an object that knows or does too much (it is an example of an <a href="/wiki/Anti-pattern" title="Anti-pattern">anti-pattern</a>).</li>
<li><a href="/wiki/Singleton_pattern" title="Singleton pattern">Singleton object</a>: an object that is the only instance of its class during the lifetime of the program.</li>
<li><a href="/wiki/Filter_object" title="Filter object">Filter object</a>.</li>
</ul>
<h2><span class="mw-headline" id="Distributed_objects">Distributed objects</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Object_(computer_science)&amp;action=edit&amp;section=4" title="Edit section: Distributed objects">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<div class="hatnote" role="note">Main article: <a href="/wiki/Distributed_object" title="Distributed object">Distributed object</a></div>
<p>The object-oriented approach is not just a programming model. It can be used equally well as an <a href="/wiki/Interface_description_language" title="Interface description language">interface definition language</a> for distributed systems. The objects in a distributed computing model tend to be larger grained, longer lasting, and more service-oriented than programming objects.</p>
<p>A standard method to package distributed objects is via an Interface Definition Language (IDL). An IDL shields the client of all of the details of the distributed server object. Details such as which computer the object resides on, what <a href="/wiki/Programming_language" title="Programming language">programming language</a> it uses, what <a href="/wiki/Operating_system" title="Operating system">operating system</a>, and other platform specific issues. The IDL is also usually part of a distributed environment that provides services such as transactions and persistence to all objects in a uniform manner. Two of the most popular standards for distributed objects are the <a href="/wiki/Object_Management_Group" title="Object Management Group">Object Management Group</a>'s <a class="mw-redirect" href="/wiki/CORBA" title="CORBA">CORBA</a> standard and <a href="/wiki/Microsoft" title="Microsoft">Microsoft</a>'s DCOM.<sup class="reference" id="cite_ref-8"><a href="#cite_note-8">[8]</a></sup></p>
<p>In addition to distributed objects, a number of other extensions to the basic concept of an object have been proposed to enable distributed computing:</p>
<ul>
<li><i>Protocol objects</i> are components of a <a href="/wiki/Protocol_stack" title="Protocol stack">protocol stack</a> that enclose network communication within an object-oriented interface.</li>
<li><i><a class="mw-redirect" href="/wiki/Replication_(computer_science)" title="Replication (computer science)">Replicated objects</a></i> are groups of distributed objects (called <i>replicas</i>) that run a distributed multi-party protocol to achieve high consistency between their internal states, and that respond to requests in a coordinated way. Examples include fault-tolerant <a class="mw-redirect" href="/wiki/CORBA" title="CORBA">CORBA</a> objects.</li>
<li><i><a href="/wiki/Live_distributed_object" title="Live distributed object">Live distributed objects</a></i> (or simply <i><a href="/wiki/Live_distributed_object" title="Live distributed object">live objects</a></i>)<sup class="reference" id="cite_ref-9"><a href="#cite_note-9">[9]</a></sup> generalize the <i>replicated object</i> concept to groups of replicas that might internally use any distributed protocol, perhaps resulting in only a weak consistency between their local states.</li>
</ul>
<p>Some of these extensions, such as <i>distributed objects</i> and <i>protocol objects</i>, are domain-specific terms for special types of "ordinary" objects used in a certain context (such as <a class="mw-redirect" href="/wiki/Remote_method_invocation" title="Remote method invocation">remote method invocation</a> or <a href="/wiki/Protocol_stack" title="Protocol stack">protocol composition</a>). Others, such as <i>replicated objects</i> and <i>live distributed objects</i>, are more non-standard, in that they abandon the usual case that an object resides in a single location at a time, and apply the concept to groups of entities (replicas) that might span across multiple locations, might have only weakly consistent state, and whose membership might dynamically change.</p>
<h2><span class="mw-headline" id="Objects_and_the_Semantic_Web">Objects and the Semantic Web</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Object_(computer_science)&amp;action=edit&amp;section=5" title="Edit section: Objects and the Semantic Web">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The <a href="/wiki/Semantic_Web" title="Semantic Web">Semantic Web</a> is essentially a distributed objects framework. Two key technologies in the Semantic Web are the <a href="/wiki/Web_Ontology_Language" title="Web Ontology Language">Web Ontology Language</a> (OWL) and the <a href="/wiki/Resource_Description_Framework" title="Resource Description Framework">Resource Description Framework</a> (RDF). RDF provides the capability to define basic objects—names, properties, attributes, relations—that are accessible via the Internet. OWL adds a richer object model, based on set theory, that provides additional modeling capabilities such as multiple inheritance.</p>
<p>OWL objects are not like standard large grained <a class="mw-redirect" href="/wiki/Distributed_objects" title="Distributed objects">distributed objects</a> accessed via an Interface Definition Language. Such an approach would not be appropriate for the Internet because the Internet is constantly evolving and standardization on one set of interfaces is difficult to achieve. OWL objects tend to be similar to the kind of objects used to define application domain models in programming languages such as Java and <a href="/wiki/C%2B%2B" title="C++">C++</a>.</p>
<p>However, there are important distinctions between OWL objects and traditional object-oriented programming objects. Where as traditional objects get compiled into static hierarchies usually with single inheritance, OWL objects are dynamic. An OWL object can change its structure at run time and can become an instance of new or different classes.</p>
<p>Another critical difference is the way the model treats information that is currently not in the system. Programming objects and most database systems use the "<a href="/wiki/Closed-world_assumption" title="Closed-world assumption">closed-world assumption</a>". If a fact is not known to the system that fact is assumed to be false. Semantic Web objects use the <a href="/wiki/Open-world_assumption" title="Open-world assumption">open-world assumption</a>, a statement is only considered false if there is actual relevant information that it is false, otherwise it is assumed to be unknown, neither true nor false.</p>
<p>OWL objects are actually most like objects in artificial intelligence <a href="/wiki/Frame_language" title="Frame language">frame languages</a> such as <a href="/wiki/KL-ONE" title="KL-ONE">KL-ONE</a> and Loom.</p>
<p>The following table contrasts traditional objects from Object-Oriented programming languages such as Java or C++ with Semantic Web Objects:<sup class="reference" id="cite_ref-10"><a href="#cite_note-10">[10]</a></sup><sup class="reference" id="cite_ref-11"><a href="#cite_note-11">[11]</a></sup></p>
<table class="wikitable">
<tr>
<th>OOP Objects</th>
<th>Semantic Web Objects</th>
</tr>
<tr>
<td>Classes are regarded as types for instances.</td>
<td>Classes are regarded as sets of individuals.</td>
</tr>
<tr>
<td>Instances can not change their type at runtime.</td>
<td>Class membership may change at runtime.</td>
</tr>
<tr>
<td>The list of classes is fully known at compile-time and cannot change after that.</td>
<td>Classes can be created and changed at runtime.</td>
</tr>
<tr>
<td>Compilers are used at build-time. Compile-time errors indicate problems.</td>
<td>Reasoners can be used for classification and consistency checking at runtime or build-time.</td>
</tr>
<tr>
<td>Classes encode much of their meaning and behavior through imperative functions and methods.</td>
<td>Classes make their meaning explicit in terms of OWL statements. No imperative code can be attached.</td>
</tr>
<tr>
<td>Instances are anonymous insofar that they cannot easily be addressed from outside of an executing program.</td>
<td>All named RDF and OWL resources have a unique URI under which they can be referenced.</td>
</tr>
<tr>
<td>Closed world: If there is not enough information to prove a statement true, then it is assumed to be false.</td>
<td>Open world: If there is not enough information to prove a statement true, then it may be true or false.<sup class="reference" id="cite_ref-12"><a href="#cite_note-12">[12]</a></sup></td>
</tr>
</table>
