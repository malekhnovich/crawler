<p>In <a href="/wiki/Software_engineering" title="Software engineering">software engineering</a>, <b>software configuration management</b> (<b>SCM</b> or <b>S/W CM</b>)<sup class="reference" id="cite_ref-1"><a href="#cite_note-1">[1]</a></sup> is the task of tracking and controlling changes in the software, part of the larger cross-disciplinary field of <a href="/wiki/Configuration_management" title="Configuration management">configuration management</a>.<sup class="reference" id="cite_ref-2"><a href="#cite_note-2">[2]</a></sup> SCM practices include <a class="mw-redirect" href="/wiki/Revision_control" title="Revision control">revision control</a> and the establishment of <a href="/wiki/Baseline_(configuration_management)" title="Baseline (configuration management)">baselines</a>. If something goes wrong, SCM can determine what was changed and who changed it. If a configuration is working well, SCM can determine how to replicate it across many hosts.</p>
<p>The acronym "SCM" is also expanded as <b>source configuration management process</b> and <b>software change and configuration management</b>.<sup class="reference" id="cite_ref-3"><a href="#cite_note-3">[3]</a></sup> However, "configuration" is generally understood to cover changes typically made by a <a href="/wiki/System_administrator" title="System administrator">system administrator</a>.</p>

<p>The goals of SCM are generally:<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (March 2008)">citation needed</span></a></i>]</sup></p>
<ul>
<li>Configuration identification - Identifying configurations, <a href="/wiki/Configuration_item" title="Configuration item">configuration items</a> and <a href="/wiki/Baseline_(configuration_management)" title="Baseline (configuration management)">baselines</a>.</li>
<li>Configuration control - Implementing a <a href="/wiki/Change_control" title="Change control">controlled change</a> process. This is usually achieved by setting up a <a href="/wiki/Change_control_board" title="Change control board">change control board</a> whose primary function is to approve or reject all change requests that are sent against any baseline.</li>
<li>Configuration status accounting - Recording and reporting all the necessary information on the status of the development process.</li>
<li>Configuration auditing - Ensuring that configurations contain all their intended parts and are sound with respect to their specifying documents, including requirements, architectural specifications and user manuals.</li>
<li><a class="mw-redirect" href="/wiki/Build_management" title="Build management">Build management</a> - Managing the process and tools used for builds.</li>
<li><a href="/wiki/Process_management" title="Process management">Process management</a> - Ensuring adherence to the organization's development process.</li>
<li>Environment management - Managing the software and hardware that host the system.</li>
<li><a href="/wiki/Teamwork" title="Teamwork">Teamwork</a> - Facilitate team interactions related to the process.</li>
<li>Defect tracking - Making sure every defect has traceability back to the source.</li>
</ul>
<p>With the introduction of <a href="/wiki/Cloud_computing" title="Cloud computing">cloud computing</a> the purposes of SCM tools have become merged in some cases. The SCM tools themselves have become virtual appliances that can be instantiated as virtual machines and saved with state and version. The tools can model and manage cloud-based virtual resources, including virtual appliances, storage units, and software bundles. The roles and responsibilities of the actors have become merged as well with developers now being able to dynamically instantiate virtual servers and related resources.<sup class="reference" id="cite_ref-4"><a href="#cite_note-4">[4]</a></sup></p>
<h2><span class="mw-headline" id="History">History</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Software_configuration_management&amp;action=edit&amp;section=2" title="Edit section: History">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The history of software configuration management (SCM) in computing can be traced back as early as the 1950s, when CM (for Configuration Management), originally for hardware development and production control, was being applied to software development. Early software had a physical footprint, such as <a href="/wiki/Punched_card" title="Punched card">cards</a>, <a href="/wiki/Punched_tape" title="Punched tape">tapes</a>, and other media. The first software configuration management was a manual operation. With the advances in language and complexity, <a href="/wiki/Software_engineering" title="Software engineering">software engineering</a>, involving configuration management and other methods, became a major concern due to issues like schedule, budget, and quality. Practical lessons, over the years, had led to the definition, and establishment, of procedures and tools. Eventually, the tools became systems to manage software changes.<sup class="reference" id="cite_ref-5"><a href="#cite_note-5">[5]</a></sup> Industry-wide practices were offered as solutions, either in an open or proprietary manner (such as <a href="/wiki/Revision_Control_System" title="Revision Control System">Revision Control System</a>). With the growing use of computers, systems emerged that handled a broader scope, including <a href="/wiki/Requirements_management" title="Requirements management">requirements management</a>, design alternatives, quality control, and more; later tools followed the guidelines of organizations, such as the Capability Maturity Model of the <a href="/wiki/Software_Engineering_Institute" title="Software Engineering Institute">Software Engineering Institute</a>.</p>
