<h1 class="firstHeading" id="firstHeading" lang="en">Systems Modeling Language</h1>
<div class="mw-body-content" id="bodyContent">

<p>The <b>Systems Modeling Language</b> (<b>SysML</b>)<sup class="reference" id="cite_ref-1"><a href="#cite_note-1">[1]</a></sup> is a <a href="/wiki/General-purpose_modeling" title="General-purpose modeling">general-purpose modeling</a> language for <a href="/wiki/Systems_engineering" title="Systems engineering">systems engineering</a> applications. It supports the specification, <a href="/wiki/Analysis" title="Analysis">analysis</a>, <a href="/wiki/Design" title="Design">design</a>, <a href="/wiki/Verification_and_validation" title="Verification and validation">verification and validation</a> of a broad range of systems and <a href="/wiki/System_of_systems" title="System of systems">systems-of-systems</a>.</p>
<p>SysML was originally developed by an <a class="mw-redirect" href="/wiki/Open_source" title="Open source">open source</a> specification project, and includes an open source license for distribution and use.<sup class="reference" id="cite_ref-faq_2-0"><a href="#cite_note-faq-2">[2]</a></sup> SysML is defined as an extension of a subset of the <a href="/wiki/Unified_Modeling_Language" title="Unified Modeling Language">Unified Modeling Language</a> (UML) using <a href="/wiki/Profile_(UML)" title="Profile (UML)">UML's profile mechanism</a>.</p>

<h2><span class="mw-headline" id="Overview">Overview</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Systems_Modeling_Language&amp;action=edit&amp;section=1" title="Edit section: Overview">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>SysML offers systems engineers several noteworthy improvements over UML, which tends to be software-centric. These improvements include the following:<sup class="reference" id="cite_ref-faq_2-1"><a href="#cite_note-faq-2">[2]</a></sup></p>
<ul>
<li>SysML's semantics are more flexible and expressive. SysML reduces UML's software-centric restrictions and adds two new diagram types, requirement and parametric diagrams. The former can be used for <a href="/wiki/Requirements_engineering" title="Requirements engineering">requirements engineering</a>; the latter can be used for <a href="/wiki/Profiling_(computer_programming)" title="Profiling (computer programming)">performance analysis</a> and <a href="/wiki/Statistics" title="Statistics">quantitative analysis</a>. Consequent to these enhancements, SysML is able to model a wide range of systems, which may include hardware, software, information, processes, personnel, and facilities.</li>
<li>SysML is a comparatively little language that is easier to learn and apply.<sup class="reference" id="cite_ref-3"><a href="#cite_note-3">[3]</a></sup> Since SysML removes many of UML's software-centric constructs, the overall language measures smaller both in diagram types and total constructs.</li>
<li>SysML allocation tables support common kinds of allocations. Whereas UML provides only limited support for tabular notations, SysML furnishes flexible allocation tables that support requirements allocation, functional allocation, and structural allocation. This capability facilitates automated <a href="/wiki/Verification_and_validation" title="Verification and validation">verification and validation</a> (V&amp;V) and gap analysis.</li>
<li>SysML model management constructs support models, views, and viewpoints. These constructs extend UML's capabilities and are architecturally aligned with <a class="mw-redirect" href="/wiki/IEEE-Std-1471-2000" title="IEEE-Std-1471-2000">IEEE-Std-1471-2000</a> (IEEE Recommended Practice for Architectural Description of Software Intensive Systems).</li>
</ul>
<p>SysML reuses seven of UML 2's fourteen diagrams, and adds two diagrams (requirement and parametric diagrams) for a total of nine diagram types. SysML also supports allocation tables, a tabular format that can be dynamically derived from SysML allocation relationships. A table which compares SysML and UML 2 diagrams is available in the SysML FAQ.<sup class="reference" id="cite_ref-4"><a href="#cite_note-4">[4]</a></sup></p>
<p>The advantages of SysML over UML for systems engineering become obvious if you consider a concrete example, like modeling an automotive system. With SysML you can use Requirement diagrams to efficiently capture functional, performance, and interface requirements, whereas with UML you are subject to the limitations of <a href="/wiki/Use_Case_Diagram" title="Use Case Diagram">Use Case Diagram</a> to define high-level functional requirements. Likewise, with SysML you can use Parametric diagrams to precisely define performance and quantitative constraints like maximum <a href="/wiki/Acceleration" title="Acceleration">acceleration</a>, minimum <a href="/wiki/Curb_weight" title="Curb weight">curb weight</a>, and total <a href="/wiki/Air_conditioning" title="Air conditioning">air conditioning</a> capacity. UML provides no straightforward mechanism to capture this sort of essential performance and quantitative information.</p>
<p>Concerning the rest of the automotive system, enhanced <a href="/wiki/Activity_diagram" title="Activity diagram">activity diagrams</a> and <a href="/wiki/State_diagram#UML_state_diagram" title="State diagram">state machine diagrams</a> can be used to specify the <a href="/wiki/Embedded_software" title="Embedded software">embedded software</a> control logic and information flows for the on-board automotive computers. Other SysML structural and behavioral diagrams can be used to model factories that build the automobiles, as well as the interfaces between the organizations that work in the factories.</p>
<h2><span class="mw-headline" id="History">History</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Systems_Modeling_Language&amp;action=edit&amp;section=2" title="Edit section: History">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The SysML initiative originated in a January 2001 decision by the <a class="mw-redirect" href="/wiki/INCOSE" title="INCOSE">International Council on Systems Engineering</a> (INCOSE) Model Driven Systems Design workgroup to customize the UML for systems engineering applications. Following this decision, INCOSE and the <a href="/wiki/Object_Management_Group" title="Object Management Group">Object Management Group</a> (OMG), which maintains the UML specification, jointly chartered the OMG Systems Engineering Domain Special Interest Group (SE DSIG) in July 2001. The SE DSIG, with support from INCOSE and the <a href="/wiki/ISO_10303" title="ISO 10303">ISO AP 233</a> workgroup, developed the requirements for the modeling language, which were subsequently issued by the OMG parting in the <i>UML for Systems Engineering Request for Proposal</i> (UML for SE RFP; OMG document ad/03-03-41) in March 2003.<sup class="reference" id="cite_ref-rfp_5-0"><a href="#cite_note-rfp-5">[5]</a></sup></p>
<p>In 2003 <a href="/wiki/Cris_Kobryn" title="Cris Kobryn">Cris Kobryn</a> and Sanford Friedenthal organized and co-chaired the <a href="/wiki/SysML_Partners" title="SysML Partners">SysML Partners</a>, an informal association of industry leaders and tool vendors,<sup class="reference" id="cite_ref-6"><a href="#cite_note-6">[6]</a></sup> which initiated an open source specification project to develop the SysML in response to the UML for Systems Engineering RFP.<sup class="reference" id="cite_ref-sysml_faq_7-0"><a href="#cite_note-sysml_faq-7">[7]</a></sup> The original technical contributors and co-authors of the SysML 1.0a specification were Laurent Balmelli, Conrad Bock, Rick Steiner, Alan Moore and Roger Burkhart. The SysML Partners distributed their first open source SysML specification drafts in 2004, and submitted SysML 1.0a to the OMG for technology adoption in November 2005.</p>
<h3><span class="mw-headline" id="OMG_SysML">OMG SysML</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Systems_Modeling_Language&amp;action=edit&amp;section=3" title="Edit section: OMG SysML">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>After a series of competing SysML specification proposals, a SysML Merge Team was proposed to the OMG in April 2006.<sup class="reference" id="cite_ref-8"><a href="#cite_note-8">[8]</a></sup> This proposal was voted upon and adopted by the OMG in July 2006 as OMG SysML, to differentiate it from the original open source specification from which it was derived. Because OMG SysML is derived from open source SysML, it also includes an open source license for distribution and use.</p>
<p>The OMG SysML v. 1.0 specification was issued by the OMG as an Available Specification in September 2007.<sup class="reference" id="cite_ref-9"><a href="#cite_note-9">[9]</a></sup> The current version of OMG SysML is v1.4, which was issued by the OMG in September 2015.<sup class="reference" id="cite_ref-10"><a href="#cite_note-10">[10]</a></sup></p>
<p><br/></p>

<h2><span class="mw-headline" id="Limitations_and_Criticisms">Limitations and Criticisms</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Systems_Modeling_Language&amp;action=edit&amp;section=5" title="Edit section: Limitations and Criticisms">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p><sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="The entire Limitations and Criticisms section is missing references. (August 2015)">citation needed</span></a></i>]</sup></p>
<p>SysML takes time and effort to learn. It is intended to be a conventional language for conducting Model Based System Engineering but it has some important limitations that reduce its efficiency and limit its acceptance. This limited usage is, in itself, a limitation as the SysML language can be considered a hermetic language that non-initiated find sometimes difficult to exploit.</p>
<p>SysML has been criticized for being incomplete although there is little to stop a practitioner from adding their own content. In SysML, there is no provision for several diagrams and graphically-oriented tools that are commonly used in system engineering. Some of these diagrams can be built, for example, by extending the intended use of block definition diagrams but the results are not always adequate and often not in concordance with the rules of the SysML language. Missing elements include <a href="/wiki/Functional_block_diagram" title="Functional block diagram">functional block diagram</a>, <a href="/wiki/N2_chart" title="N2 chart">N2 chart</a> , <a href="/wiki/House_of_Quality" title="House of Quality">House of Quality</a>, <a href="/wiki/Ishikawa_diagram" title="Ishikawa diagram">Ishikawa diagram</a> (fishbone), parameter diagram and others. The language is continually improving and other diagram types are being considered for addition in future updates, should they be deemed sufficiently useful for inclusion.</p>
<p>The diagrams generated by SysML are complicated and some are difficult to understand by people that are unfamiliar with the language. Some elements are slightly counter-intuitive and this could lead to confusion and errors. This is occasionally aggravated by SysML users believing SysML should be a widespread convention. This can be overcome by including explanatory notes and legends in SysML diagrams to ease their interpretation. As system-engineering diagrams are primarily intended for other members of an embedded team and people outside the team that, more often than not, are not system engineers and less likely to know SysML.</p>
<p>When drawn in a software tool, the diagrams that respect the rules of SysML often include redundant pieces of model information that can impair their interpretation.</p>
<h2><span class="mw-headline" id="Tools">Tools</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Systems_Modeling_Language&amp;action=edit&amp;section=6" title="Edit section: Tools">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>There are several modeling tool vendors already offering SysML support, or are in the process of updating their tools to comply with the OMG SysML specification. Lists of tool vendors who support, or have announced support of, SysML or OMG SysML can be found on the SysML Forum<sup class="reference" id="cite_ref-11"><a href="#cite_note-11">[11]</a></sup> or SysML<sup class="reference" id="cite_ref-12"><a href="#cite_note-12">[12]</a></sup> websites, respectively.</p>
<h3><span class="mw-headline" id="Model_Exchange">Model Exchange</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Systems_Modeling_Language&amp;action=edit&amp;section=7" title="Edit section: Model Exchange">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>As an <a href="/wiki/Object_Management_Group" title="Object Management Group">OMG</a> <a href="/wiki/Unified_Modeling_Language" title="Unified Modeling Language">UML 2.0</a> <a href="/wiki/Profile_(UML)" title="Profile (UML)">profile</a>, SysML models are designed to be exchanged using the <a href="/wiki/XML_Metadata_Interchange" title="XML Metadata Interchange">XML Metadata Interchange</a> (XMI) standard. In addition, architectural alignment work is underway to support the <a href="/wiki/ISO_10303" title="ISO 10303">ISO 10303</a> (also known as STEP, the Standard for the Exchange of Product model data) AP-233 standard for exchanging and sharing information between <a href="/wiki/Systems_engineering" title="Systems engineering">systems engineering</a> software applications and tools.</p>
